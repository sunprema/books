Code That Writes Code, for Agents

There's a second, sneakier place metaprogramming meets AI: not the code an agent calls, but the code an agent writes. When an LLM edits your project, you want it changing the syntax tree — not blindly find-replacing text.

A coding agent that patches files with regular expressions is one stray match away from corrupting your source. The Elixir ecosystem's answer is Igniter — a code-generation and project-patching framework that operates on the AST, using Sourceror to read, modify, and re-print code while preserving comments and formatting.

an Igniter task patches the tree, not the text# mix igniter.install ash
# under the hood, a task composes AST-level changes:

igniter
|> Igniter.Project.Deps.add_dep({:ash, "~> 3.0"})
|> Igniter.Project.Formatter.import_dep(:ash)
|> Igniter.Project.Module.create_module(
     MyApp.Support.Ticket,
     """
     use Ash.Resource
     """
   )

Each step is a semantic edit: "add this dependency," "make this module exist." Run it twice and it won't duplicate anything, because Igniter reasons about what's already in the tree. That idempotence is what makes it safe for an agent to drive.

🔑
Why AST-patching wins for agents

Text edits are fragile and order-dependent; AST edits are precise and composable. Igniter underpins the installers for Ash, Phoenix tooling, Spark, and more — so "let the agent run mix igniter.install" is a reliable instruction, not a gamble.

Giving the agent the rules

An agent writes better Elixir when it knows each library's conventions. usage_rules is a package that collects the usage-rules.md shipped by your dependencies and consolidates them into your AGENTS.md, so the model reads authoritative, version-correct guidance instead of guessing:

one command, agent-ready context# gather every dependency's usage rules into AGENTS.md
$ mix usage_rules.sync AGENTS.md --all

Letting the agent see runtime truth

Tidewave connects an AI assistant to your running app over the Model Context Protocol (MCP). Instead of reasoning about code in the abstract, the agent can evaluate code in the live app, inspect Ecto queries, and read logs — grounding its edits in what the system actually does.

The agent edits the tree, reads the rules, and watches the live system — all on a metaprogramming substrate.
🤖
The full circle

Macros let your program describe itself; Spark/Ash turn that into introspectable DSLs; Igniter lets an agent edit those declarations safely; usage_rules and Tidewave feed the agent correct context. Metaprogramming isn't a side feature of Elixir's AI story — it's the foundation the whole toolchain stands on.
