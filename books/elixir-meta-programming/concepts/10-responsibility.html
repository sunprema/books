Concept 10 · Judgment

With Great Power

You can now bend the compiler. The final, hardest skill is knowing when not to. Most code that could be a macro is better as a function — and Elixir's own documentation says so, in a list of named anti-patterns.

There's a saying in the community: "the first rule of macro club is, don't write macros." It's only half a joke. Macros add a layer of indirection between what's written and what runs; every reader of your code now has to expand it in their head. Pay that tax only when the power is genuinely needed.

When a macro is justified

  • You need an argument left unevaluated (a block, a condition) — a function can't do this.
  • You're building a declarative DSL whose readability is the whole point.
  • You want to generate code at compile time from data for match-speed dispatch.
  • You need compile-time guarantees — validating a declaration before the program ever runs.
🔑
The test

Ask: "Could a plain function, a higher-order function, or a behaviour do this?" If yes, do that instead. Reach for a macro only when the answer is a clear no.

The official anti-patterns

Elixir ships a Meta-programming anti-patterns reference. Keep these four on a sticky note:

Anti-patternDo instead
Unnecessary macrosReplace with named functions. A macro should be essential, not convenient.
Large code generationKeep quote blocks small; delegate the real work to a regular function the macro calls.
use instead of importPrefer alias/import. If you must offer __using__, document its effects like a nutrition label.
Compile-time dependenciesAvoid macros that pull modules into compile-time deps unnecessarily — they trigger recompilation cascades.
⚠️
The hidden cost

Macros that generate a lot of code slow compilation and bloat your .beam files — Elixir "may have to expand, compile, and execute the code multiple times." And broad use propagates hidden dependencies that make a module hard to reason about. Power has a maintenance bill.

🧑‍🏫
Where we landed

Metaprogramming is a precision tool: code as data, quoted and unquoted, generated and introspected. Used sparingly it gives Elixir its best DSLs and — now — a toolchain LLM agents can read, call, and safely edit. Used carelessly it gives you a codebase no one wants to touch. Write the function first. Reach for the macro when, and only when, the function can't reach far enough.
