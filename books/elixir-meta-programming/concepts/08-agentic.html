Metaprogramming Meets the LLM

An LLM agent needs a precise, machine-readable description of what it may do and what shape its answer must take. That description is data about your code — which is exactly what metaprogramming and introspection produce.

Think about what tool-calling actually demands: a list of tools, each with a name, a human description, and a typed argument schema, so the model can pick one and fill in valid arguments. Now recall the last two chapters — declarations recorded as data, read back out by introspection. The shapes line up almost too neatly.

A tool is a declaration

In Elixir's LangChain library, a tool the model can call is just a struct you declare — name, description, parameter schema, and a function to run:

LangChain.Functionalias LangChain.Function

locate =
  Function.new!(%{
    name: "locate",
    description: "Return the location of a requested item.",
    parameters_schema: %{
      type: "object",
      properties: %{
        thing: %{type: "string",
                 description: "the item to locate"}
      },
      required: ["thing"]
    },
    function: fn %{"thing" => thing}, context ->
      {:ok, context[thing]}
    end
  })

That parameters_schema is JSON Schema — the lingua franca of tool-calling. You wrote it by hand here, but nothing stops a macro from generating it from a typed declaration, which is exactly what the frameworks do.

Declare the tools, let introspection emit the schema

Ash AI takes the declarative route to its conclusion. You expose existing resource actions to a model by naming them in a DSL block; Ash introspects the action's typed arguments and generates the tool schema for you — authorization and types preserved:

Ash AI — tools as a DSLtools do
  tool :read_tickets, MyApp.Support.Ticket, :read
  tool :open_ticket, MyApp.Support.Ticket, :create
end

No glue code translates the action into a tool definition — the declaration is the source of truth, read back out as the model's catalog.

Declare → introspect → emit schema → the model calls → your code runs.
🔑
The deeper point

LLMs are happiest when a system can describe itself in structured terms. That self-description is precisely what metaprogramming produces. Elixir didn't pivot to chase AI — the agentic world arrived at a problem the language was already shaped for.

Schemas as output contracts

The other half of agentic work is getting structured data out of a model. Instructor (and InstructorLite) reuse Ecto's schema DSL — itself pure metaprogramming — as the contract the model's answer must satisfy:

Instructor — Ecto schema as the specdefmodule SpamPrediction do
  use Ecto.Schema
  use Instructor

  @llm_doc "Classify whether a message is spam."
  embedded_schema do
    field :spam?, :boolean
    field :confidence, :float
  end
end

Instructor.chat_completion(
  model: "gpt-4o",
  response_model: SpamPrediction,
  messages: [%{role: "user", content: text}]
)
# => {:ok, %SpamPrediction{spam?: true, confidence: 0.97}}

The schema does triple duty: it generates the JSON Schema sent to the model, validates the reply with Ecto changesets, and — on a bad response — feeds the errors back for a retry. One declaration, the whole contract.

Why the BEAM, too

  • Schemas are guardrails. The same schema that validates a human bounds the model's arguments and its output — fewer malformed calls, fewer hallucinated fields.
  • Concurrency is native. Thousands of supervised, crash-isolated agent processes is the runtime's home turf — frameworks like Jido lean on exactly that.
  • One uniform shape. Tools, prompts, schemas, and policies all read as declarations, so an agent system stays legible as it grows.
