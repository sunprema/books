Generating Functions from Data

Here's where metaprogramming earns real money: read a table of data at compile time and stamp out a fast, pattern-matched function clause for every row. No runtime lookup, no dictionary — just compiled code.

The trick is that def is itself a macro, and you can call it inside a for comprehension at the top level of a module. Each iteration defines a clause. Combine that with unquote and the comprehension variable becomes part of the generated function head — a technique called an unquote fragment.

lib/mime.exdefmodule MIME do
  @types %{
    "text/html"  => ["html", "htm"],
    "image/png"  => ["png"],
    "image/jpeg" => ["jpg", "jpeg"]
  }

  # at COMPILE time, walk the map and emit clauses
  for {type, exts} <- @types do
    def exts_for(unquote(type)), do: unquote(exts)

    for ext <- exts do
      def type_for(unquote(ext)), do: unquote(type)
    end
  end

  def type_for(_unknown), do: nil
end

After compilation, MIME contains real, separate function clauses — as if you'd hand-written each one:

iexiex> MIME.type_for("png")
"image/png"
iex> MIME.exts_for("text/html")
["html", "htm"]

Why this is fast

Dispatch is by the BEAM's pattern matcher — effectively a jump table, not a hash lookup you wrote. The data was paid for once, at compile time, and disappears from the runtime entirely.

One row of data → one compiled clause. The table is gone by runtime.

From external data

The data needn't be inline. Read a file at compile time and generate from it — the file is consulted during mix compile, never at runtime:

compile-time file read@external_resource "priv/mimes.txt"

@types "priv/mimes.txt"
       |> File.read!()
       |> String.split("\n", trim: true)

for line <- @types do
  [type, ext] = String.split(line, "\t")
  def type_for(unquote(ext)), do: unquote(type)
end
🔑
@external_resource

Tagging the file with @external_resource tells Mix to recompile this module whenever that file changes — so your generated code stays in sync with its source data. This is the backbone of i18n libraries, Unicode tables, and codec generators.

⚠️
Keep it bounded

Generating thousands of clauses bloats the BEAM file and slows compilation — the official large code generation anti-pattern. If the table is huge, prefer a runtime map. Generate code when you want match-speed dispatch over a modest, known set.
