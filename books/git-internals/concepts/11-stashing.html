Chapter 11 of 17 · The side pocket

Stashing: Temporary Storage for In-Progress Work

Chapter 10 closed with a promise: this chapter is about the opposite problem — not undoing work, but pausing it mid-flight. Here's the scenario it was pointing at: you're elbow-deep in a feature, half the file is broken on purpose while you're mid-refactor, and then — Slack ping — a teammate needs you on a different branch right now. You're not ready to commit this mess. You don't want to lose it either. git stash is Git's answer to exactly that moment: a side pocket where in-progress work waits, untouched, while you go deal with something else.

The Problem: Switch Tasks, Keep the Work

Try to git switch or git checkout to another branch while you have uncommitted edits, and Git will often refuse outright — "your local changes would be overwritten by checkout." Even when it lets you through, those edits ride along into the new branch with you, which is rarely what you want. You're left with three bad options: commit half-finished, broken code just to clear the deck (and now your history has a "wip please ignore" commit in it forever); throw the work away and redo it later; or just... not switch, and leave the teammate waiting. None of those are good. Stashing is the fourth option — the one that actually works.

git stash push: Off to the Side, Not Gone

git stash push takes every change in your working directory and index that differs from HEAD, tucks it away in a safe place, and then resets both your working directory and your index back to match HEAD exactly — a clean slate, as if you'd never touched anything. Nothing is deleted. It's more like a coat check than a trash can: hand over the mess, get a claim ticket, walk away with empty hands. Give it a -m message the same way you'd write a commit message — future-you, staring at a list of stashes next week, will thank present-you for not just calling it "stuff": 

Terminal — an edit, mid-thought$ mkdir stash-quest && cd stash-quest && git init -q -b main
$ printf "Welcome to Staging Quest!\nHP: 100\n" > player.txt
$ git add player.txt && git commit -q -m "Start the story"
$ printf "Welcome to Staging Quest!\nHP: 100\nInventory: sword\n" > player.txt
$ git status
On branch main
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
	modified:   player.txt

no changes added to commit (use "git add" and/or "git commit -a")

Half a thought, sitting unstaged. Stash it, and watch both the status and the file itself snap back to a clean HEAD: 

Terminal — git stash push -m, then a clean slate$ git stash push -m "WIP: sword inventory"
Saved working directory and index state On main: WIP: sword inventory
$ git status
On branch main
nothing to commit, working tree clean
$ cat player.txt
Welcome to Staging Quest!
HP: 100

"Nothing to commit, working tree clean" — the exact phrase Chapter 10 taught you to treat as a warning sign after reset --hard. Here it means something completely different: your edit isn't gone, it's filed. (Bare git stash, no subcommand, is shorthand for git stash push — you'll see both spellings in the wild.)

Warning

By default, git stash push only picks up changes to files Git is already tracking. A brand-new file you haven't git add-ed yet is invisible to a plain git stash — it'll still be sitting in your working directory, unstashed, when you switch branches. If you want untracked files swept up too, add -u (or --include-untracked): git stash push -u -m "…". Easy to forget, and the classic way a stash "loses" a file that was never in it to begin with.

git stash list: A Stack, Not a Slot

You're not limited to one stash at a time. Every git stash push adds a new entry to a LIFO stack — last in, first out, same discipline as a stack of trays in a cafeteria. Ask what's waiting with git stash list: 

Terminal — one entry on the stack$ git stash list
stash@{0}: On main: WIP: sword inventory

stash@{0} is always the most recent entry — the top of the stack. Stash something else without touching this one first, and it becomes stash@{1}, pushed one slot down, while the new one claims stash@{0}. Most of the commands below default to stash@{0} when you don't name one explicitly, which is exactly the LIFO behavior you'd expect: "give me back the thing I most recently set aside."

Under the Hood: A Stash Is Just a Commit

Here's the payoff for having read Chapter 3 first: stashing isn't a separate storage mechanism bolted onto Git. A stash is a commit object — the exact same kind of object every git commit in this book has created — just one that lives under a different name, refs/stash, instead of refs/heads/<branch>. Print it with the same git cat-file -p plumbing Chapter 3 introduced, and it looks almost like an ordinary commit, with one telling difference: 

Terminal — the stash, pretty-printed as a commit$ cat .git/refs/stash
68534364af04eb6e55353710af019e3672aa0ac9
$ git cat-file -p 68534364af04eb6e55353710af019e3672aa0ac9
tree 9fc66ff9e51835c758ceeeb1e090da681b9a7c1c
parent 957bd8bce473171daa2810d5e9e830b379ae6ad5
parent 921548a24c018d07be50162de520a6ddd58598dd
author Ava <ava@example.com> 1752400060 +0000
committer Ava <ava@example.com> 1752400060 +0000

On main: WIP: sword inventory

Two parent lines. Every commit you've seen since Chapter 3 has had exactly one (or zero, for the first commit ever) — a stash commit is a small merge commit with two: the first parent is whatever HEAD pointed at the moment you stashed, and the second is a commit built from your index at that same instant. (Stash with -u and you'll get a third parent, holding the untracked files too.) That's the whole trick: git stash push packages "here's HEAD" and "here's what was staged and unstaged" into one object, points refs/stash at it, and only then resets your working directory to look clean. Nothing new needed inventing — it's blobs, trees, and commits, arranged in a slightly unusual shape.

Key Idea

git stash list isn't reading some separate stash database — it's printing the reflog of refs/stash, the exact mechanism Chapter 9 covered for HEAD. Every push appends a line to .git/logs/refs/stash recording the old tip, the new tip, and your -m message; that log is the numbered stash@{n} stack. Run git log -g --oneline refs/stash and you'll see the identical entries git stash list shows you, just spelled the reflog way — two chapters, one underlying idea.

git stash pop: Bring It Back, One Time

Back to the interruption. You stashed, switched context, and handled the urgent thing — here, a one-line fix in a completely different file, committed normally: 

Terminal — the urgent detour, unrelated to the stash$ printf "URGENT: fix typo in intro\n" > bugfix.txt
$ git add bugfix.txt && git commit -q -m "Fix urgent typo"
$ git log --oneline
1f3f99e Fix urgent typo
957bd8b Start the story

Detour handled. Now reclaim the shelved work with git stash pop, which reapplies stash@{0}'s changes on top of your current working directory and, if that applies cleanly, removes the entry from the stack — a bundled "apply, then drop": 

Terminal — git stash pop, work resumes$ git stash pop
On branch main
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
	modified:   player.txt

no changes added to commit (use "git add" and/or "git commit -a")
Dropped refs/stash@{0} (68534364af04eb6e55353710af019e3672aa0ac9)
$ cat player.txt
Welcome to Staging Quest!
HP: 100
Inventory: sword
$ git stash list

The unstaged edit is back exactly as you left it, the "Dropped" line confirms the commit object was removed from the stack, and git stash list prints nothing — the pocket's empty again. The commit itself isn't destroyed the instant it's popped, by the way — same rule as Chapter 9's deleted branches — it just becomes unreachable and eventually gets swept up by git gc, so a just-popped stash is still technically rescuable from the reflog for a little while if you immediately realize you needed it.

git stash apply vs git stash pop: Keep or Remove

pop is the common case, but sometimes you want to try a stash on without committing to giving it up — maybe you're not sure this is even the right stash, or you want to apply the same fix to two different branches. That's what git stash apply is for: same restoration, but the entry stays on the stack afterward, so you can apply it again, or apply it somewhere else, or just decide later you're done with it. 

Terminal — stash something new, then apply instead of pop$ git add player.txt && git commit -q -m "Add sword to inventory"
$ printf "Welcome to Staging Quest!\nHP: 100\nInventory: sword\nPotion: healing (WIP)\n" > player.txt
$ git stash push -m "WIP: healing potion"
Saved working directory and index state On main: WIP: healing potion
$ git stash apply
On branch main
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
	modified:   player.txt

no changes added to commit (use "git add" and/or "git commit -a")
$ git stash list
stash@{0}: On main: WIP: healing potion

Look closely: the "Potion: healing (WIP)" line is back in player.txt, exactly like pop would've done — but git stash list still shows stash@{0} sitting right there, untouched. Nothing was dropped. Once you're actually done with an applied stash, remove it explicitly with git stash drop (with no argument it drops stash@{0}, same default as everything else): 

Terminal — done with it now, drop it for real$ git stash drop
Dropped refs/stash@{0} (8ba875f9f2c71ad54f1aa99a18531b0257b5b3a0)
$ git stash list

Put plainly: pop = apply + drop, run as one step. Reach for pop when you're confident you're done with the stash the moment it's back; reach for apply (plus a manual drop later) when you want a safety net while you check the result first.

Pro Tip

Always name your stashes with -m "…". A bare git stash push labels the entry with just the branch name and the last commit's subject — useless once you've got three stashes and can't remember which is which. And if popping or applying ever produces a merge conflict (it can, just like any other merge), don't panic: Git leaves the stash entry on the stack until you resolve it and either commit or manually git stash drop it — a failed pop never silently deletes your work.

Warning

Like Chapter 9's reflog, the stash stack is local only — it lives under refs/stash and .git/logs/refs/stash on your machine, and it is never pushed, fetched, or cloned. A stash is not a backup you can hand to a teammate or recover on a different computer; if this laptop dies before you pop it, that work is gone with it. Treat stashing as a same-session, "I'll be right back" tool — not a substitute for actually committing (even to a scratch branch) anything you want to survive past today.

Where Stashing Actually Earns Its Keep

  • The urgent interrupt. Mid-feature, a hotfix request lands. Stash, switch, fix, push, switch back, pop — five commands, zero lost context.
  • A clean tree before pulling. git pull (and especially a rebase-based pull) can refuse to run, or produce confusing conflicts, with uncommitted local edits in the way. Stash first, pull, pop after — a much cleaner story than a tangled merge conflict mixing your WIP with incoming history.
  • "Does this actually reproduce on a clean tree?" Stash your in-progress debugging edits to check whether a bug exists on plain HEAD, confirm it, then pop your changes right back and keep working.
  • Trying an idea without committing to it. Stash gives you a free "undo everything since my last commit" you can walk back from — safer than reset --hard because nothing is discarded, just set aside.
Try It Yourself

Live the whole interrupt-and-resume story yourself. Fresh throwaway repo, one starting commit: 

Terminal — set the scene$ mkdir stash-quest && cd stash-quest && git init -q -b main
$ printf "Welcome to Staging Quest!\nHP: 100\n" > player.txt
$ git add player.txt && git commit -q -m "Start the story"

Start some in-progress work, mid-thought, then check the working tree is genuinely dirty: 

Terminal — the WIP edit$ printf "Welcome to Staging Quest!\nHP: 100\nInventory: sword\n" > player.txt
$ git status --short
 M player.txt

Stash it with a real message, and confirm the working tree goes fully clean — the edit isn't in the diff anymore, but it's not gone either: 

Terminal — push, then verify a clean slate$ git stash push -m "WIP: sword inventory"
Saved working directory and index state On main: WIP: sword inventory
$ git status
On branch main
nothing to commit, working tree clean
$ git stash list
stash@{0}: On main: WIP: sword inventory

Now go do something else entirely — the "urgent interrupt," touching a totally different file so there's no chance of overlap: 

Terminal — the interrupt, handled and committed$ printf "URGENT: fix typo in intro\n" > bugfix.txt
$ git add bugfix.txt && git commit -q -m "Fix urgent typo"
$ git log --oneline
1f3f99e Fix urgent typo
957bd8b Start the story

Interrupt handled. Bring the original work back with pop, and watch it disappear from the stack the moment it lands: 

Terminal — pop, and the stack empties$ git stash pop
Dropped refs/stash@{0} (…)
$ cat player.txt
Welcome to Staging Quest!
HP: 100
Inventory: sword
$ git stash list

Empty stack, and your sword edit is right back where you left off. Last step — commit that, then create one more stash and this time compare apply against pop directly: 

Terminal — apply keeps the entry, drop removes it explicitly$ git add player.txt && git commit -q -m "Add sword to inventory"
$ printf "Welcome to Staging Quest!\nHP: 100\nInventory: sword\nPotion: healing (WIP)\n" > player.txt
$ git stash push -m "WIP: healing potion"
$ git stash apply
$ git stash list
stash@{0}: On main: WIP: healing potion
$ git stash drop
Dropped refs/stash@{0} (…)
$ git stash list

Notice apply left the entry sitting in git stash list right after restoring the file — it took drop, run separately and on purpose, to actually clear it. That's the whole difference between the two commands, felt with your own hands instead of just read about.

Zoom out and stashing turns out to be the least mysterious feature in this book once you've read Chapter 3: no bespoke storage format, no magic — just a two-parent commit under refs/stash, a stack that's really the reflog Chapter 9 already explained, and two familiar verbs, apply and drop, that pop simply bundles together. It's the tool for pausing work, not marking it — for that, Git has a different mechanism entirely.
