Security and Integrity: Trusting Your History

Chapter 15 closed on a question it deliberately left hanging: git fsck can tell you every object in your repo is intact — no corruption, no broken links — but intact isn't the same question as trustworthy. A perfectly healthy object store can still be full of commits that lie about who wrote them. This chapter is about the difference between those two guarantees, and the one tool Git gives you to close the gap: cryptographic signatures.

Integrity Is Already Solved. Authenticity Isn't.

Chapter 3 showed you that Git is built from just three kinds of object — blob tree commit — each one named by a hash of its own content. Chapter 4 showed you exactly what gets fed into that hash, and why the avalanche effect makes two similar-but-different objects land on completely unrelated fingerprints. Chapter 6 showed you that a commit doesn't just point at a tree; it also points at its parent commit's hash, which is how three snapshots become one connected graph instead of three loose photographs. Put those three chapters together and you get a structure with a name: a Merkle DAG — a directed acyclic graph where every node's name is a hash that depends on the hashes of everything it points to.

That structure has a property this chapter has been building toward the whole book: you cannot change one bit anywhere in history without changing the hash of every object downstream of it. Not "it's hard to hide." Not "someone would probably notice." The math makes it structurally impossible to tamper quietly. That's integrity, and Git has had it since Chapter 3's very first git hash-object. What it doesn't give you for free is authenticity — proof of who created an object in the first place. Those are genuinely different guarantees, and mixing them up is exactly the mistake that makes unsigned history dangerous. The rest of this chapter proves the first claim with real hashes from a real repo, then spends the second half closing the gap on the second one.

Key Idea

A hash proves an object wasn't altered after it was named. It says nothing about who created it. git fsck (Chapter 15) will happily certify a repository as perfectly intact even if every single commit in it has a fabricated author line — integrity and authenticity are separate promises, and only one of them comes free with content-addressing.

The Weak Spot: Nobody Checks the Author Line

Here's the uncomfortable part. Every commit carries an author line — a name and an email address — and Git will write whatever you tell it to, no questions asked. That name doesn't have to match who's typing. It doesn't have to match any account on any hosting platform. It doesn't have to be a real person at all. Git's job is recording history faithfully, not policing who's allowed to claim credit for it — those are, once again, two different jobs, and this book has been showing you Git only signed up for the first one.

Warning

The author name and email on a commit are not verified or authenticated by Git in any way. Anyone with a copy of Git can set user.name and user.email to literally anything — including your name, or a maintainer's name, or a name that doesn't exist — and every plain commit will accept it silently. There is no password, no identity check, no gatekeeping at the object-format level. Unsigned authorship is a claim, not a proof. Everything below this warning exists to close that gap.

Don't take that on faith — it's one command. This really ran, in a throwaway repo, with nothing exotic: 

Terminal — impersonating a famous maintainer, no special access required$ git commit --allow-empty -m "Legit-looking fix" \
    --author="Linus Torvalds <torvalds@linux-foundation.org>"
$ git log -1 --format="Author: %an <%ae>%nSigned:  %G? (%GS)"
Author: Linus Torvalds <torvalds@linux-foundation.org>
Signed:  N ()

Git accepted the commit without complaint — correctly, by its own rules, since nothing in the object format claims otherwise. The %G? column reads N: no signature was even attempted, so there was never any authenticity claim to check. That's the exact gap the rest of this chapter closes.

GPG Signing: Proving It Was Really You

Git's answer is the same one email, software downloads, and legal documents landed on decades ago: a digital signature. You generate a GPG (GNU Privacy Guard) key pair — a private key you never share, and a public key you publish. Sign a commit with your private key, and anyone holding your public key can mathematically confirm two things at once: the commit's content matches exactly what you signed (tamper-evidence again, this time on the signature itself), and only someone holding your private key could have produced that signature. Tell Git which key is yours, and add one flag to any commit or tag: 

Terminal — one-time setup: tell Git which key is yours$ gpg --list-secret-keys --keyid-format=long
$ git config --global user.signingkey <your-key-id>
$ git config --global commit.gpgsign true   # sign every commit by default

With that in place, a single flag signs one commit on demand — and this is real, live output, not a transcript pulled from a manual: 

Terminal — a real GPG-signed commit, verified on the spot$ git commit -S -m "Add sword to inventory"
$ git log --show-signature -1
commit f931c9fcab7caeb4f2e88f922e589eb0643819da
gpg: Signature made Mon Jul 13 01:55:52 2026 UTC
gpg:                using EDDSA key 4707A397E239C5893871FE48A37DBA7D0A533877
gpg: Good signature from "Ava TestQuest <ava@example.com>" [ultimate]
Author: Ava TestQuest <ava@example.com>
Date:   Mon Jul 13 12:00:00 2026 +0000

    Add sword to inventory

That "Good signature" line is GPG itself telling you the math checked out — the commit's content hasn't changed since Ava's private key signed it, and nobody without that key could have produced this exact signature. Notice the signature isn't stored anywhere fancy: it's just another field inside the commit object, right alongside author and tree — sitting in the same plain-text format Chapter 3 first showed you with cat-file -p.

Pro Tip

Two lowercase-vs-uppercase flags on git commit get mixed up constantly, and they do completely different things: -S (capital) cryptographically signs the commit with GPG, the whole subject of this chapter. -s (lowercase) just appends a plain-text Signed-off-by: Your Name <you@example.com> line to the message — a convention some projects use to record "I agree to this project's contribution terms," with zero cryptography involved. You can use both together (git commit -sS), but only the capital one gives you an actual, unforgeable proof of identity.

Verified vs. Unverified: Checking Signatures Later

A signature is only useful if someone actually checks it later. git log --show-signature (above) is the interactive way; for scripts and CI, git verify-commit gives you a plain pass/fail, and the %G? placeholder in --pretty=format lets you scan a whole history at a glance. Run it across this chapter's four demo commits and the difference is immediate: 

Terminal — git verify-commit, and %G? across a real history$ git verify-commit f931c9f
gpg: Signature made Mon Jul 13 01:55:52 2026 UTC
gpg:                using EDDSA key 4707A397E239C5893871FE48A37DBA7D0A533877
gpg: Good signature from "Ava TestQuest <ava@example.com>" [ultimate]

$ git log --pretty="format:%h  %G?  %an  %s" --all
51b4dc0  N  Linus Torvalds  Legit-looking fix
a2f8c35  N  Ava TestQuest    Add shield to inventory
f931c9f  G  Ava TestQuest    Add sword to inventory
410aa16  N  Ava TestQuest    Start the story

G means good signature; N means no signature was attempted at all. Other codes exist too — B for a signature that's outright bad, U for good-but-from-an-untrusted key, E when Git couldn't even check (missing key, GPG error) — but the two you'll see 95% of the time are these: signed-and-good, or not signed at all. Notice that a2f8c35 and the spoofed 51b4dc0 land in the exact same bucket, N. Git makes no distinction between "an honest commit nobody bothered to sign" and "a commit actively impersonating someone" — both simply have no authenticity claim to evaluate. That symmetry is precisely why the next section matters.

Pro Tip

Hosting platforms build on exactly this mechanism for the little badges you've probably already seen next to commits. Once you upload your public GPG (or SSH, or S/MIME) key to your account, GitHub marks each commit or tag Verified, Partially verified, or Unverified based on the same signature check git log --show-signature runs locally — it's not a separate, platform-specific trust system, just a friendlier UI on top of the GPG math. Turn on "vigilant mode" in your GitHub settings and it'll flag every commit claiming to be you that isn't signed, which is the platform-level version of the warning above.

Signed Tags: git tag -s

Chapter 12 mentioned this in passing and promised a return trip: annotated tags can optionally be GPG-signed too, with the same idea applied one level up — not "this commit is really mine" but "this release is really mine." Swap -a for -s and Git signs the tag object itself: 

Terminal — a real signed release tag, then verified$ git tag -s v1.0.0 -m "Release 1.0.0: sword and shield update"
$ git cat-file -t v1.0.0
tag
$ git tag -v v1.0.0
gpg: Signature made Mon Jul 13 01:55:59 2026 UTC
gpg:                using EDDSA key 4707A397E239C5893871FE48A37DBA7D0A533877
gpg: Good signature from "Ava TestQuest <ava@example.com>" [ultimate]
object a2f8c35f34d8a18ec5fb76e6c11fcdad17a35ce4
type commit
tag v1.0.0
tagger Ava TestQuest <ava@example.com> 1783944300 +0000

Release 1.0.0: sword and shield update

That's the exact tag-object shape Chapter 12 showed you — object, type, tag, tagger, message — with one addition sitting inside the object alongside them: a signature block, checked the same way git verify-commit checks a commit. For an open-source project, this is the single highest-leverage signature to get right: thousands of people may pip install or npm install whatever v1.0.0 points to without ever reading a line of it — the tag is the trust boundary.

Fun Fact

You don't strictly need GPG anymore. Since Git 2.34 (2021), you can sign commits and tags with a plain SSH key instead — the same kind you may already have for pushing over ssh:// — by setting git config gpg.format ssh and pointing user.signingkey at your SSH public key. GitHub, GitLab, and friends will happily mark SSH-signed commits "Verified" the same as GPG ones. The upside is skipping GPG key management entirely if you already maintain SSH keys; the tradeoff is that Git has no built-in trust store for SSH keys the way GPG has its web of trust, so verification needs an explicit gpg.ssh.allowedSignersFile mapping emails to public keys before --show-signature can confirm anything.

Why This Actually Matters: Supply-Chain Trust

This isn't a hypothetical concern dreamed up for a security chapter. In February 2021, researchers submitted a series of intentionally buggy patches to the Linux kernel — one of the most heavily reviewed codebases on Earth — as part of a published study on whether malicious-but-plausible-looking commits could slip past code review under fabricated identities. Maintainers caught it, the University of Minnesota was banned from contributing to the kernel, and the incident became a widely cited case study in exactly the failure mode this chapter is about: a commit's claimed author is worth nothing on its own, no matter how legitimate the surrounding pull request looks.

Scale that up to the entire open-source ecosystem and the stakes get concrete fast. Package managers — npm, PyPI, crates.io, and dozens more — pull source straight from Git repositories and tags, often with no human reading every line before it reaches production. A single convincingly-authored commit slipped into a popular dependency doesn't just fool one reviewer; it potentially ships to every project downstream that installs that package. That's the "supply chain" in supply-chain security: trust doesn't stay contained to the repo it was planted in, it propagates outward through every dependency edge, the same way a tampered blob's hash propagates forward through every commit after it.

Signing doesn't make bad code impossible — a legitimate maintainer with a valid key can still sign something malicious or simply buggy, and code review still has to do its actual job. What signing removes is the cheapest attack: pretending to be someone you aren't. Once maintainers require signed commits and signed release tags — a setting many serious projects now enforce with a branch-protection rule — "this patch came from Linus Torvalds" stops being a free claim anyone can type into --author, and starts being a mathematical fact you can check yourself, offline, with nothing but a public key.

Try It Yourself

You don't need a real GPG key to see the tamper-evidence half of this chapter — it's pure plumbing, the same commands Chapter 3 already taught you. Fresh throwaway repo: 

Terminal — set the scene: one commit, one blob$ mkdir trust-quest && cd trust-quest && git init -q -b main
$ printf "Welcome to Trust Quest!\nHP: 100\n" > player.txt
$ git add player.txt && git commit -q -m "Start the story"
$ ROOT=$(git rev-parse HEAD)
$ TREE=$(git rev-parse HEAD^{tree})
$ echo "$ROOT $TREE"

Now tamper: hash a changed version of the same file without touching the working tree, splice it into a rebuilt tree, and rebuild the commit — all with plumbing, none of it touching HEAD: 

Terminal — rebuild blob, tree, and commit by hand$ NEWBLOB=$(printf "Welcome to Trust Quest!\nHP: 9999\n" | git hash-object -w --stdin)
$ NEWTREE=$(printf "100644 blob %s\tplayer.txt\n" "$NEWBLOB" | git mktree)
$ NEWROOT=$(printf "Start the story" | git commit-tree "$NEWTREE")
$ echo "original: $ROOT  tampered: $NEWROOT"

Two completely different commit hashes, from a one-word content edit — and if you had a second and third commit built on top of the original (the way this chapter's diagram does), you'd rebuild those too with git commit-tree <same tree> -p $NEWROOT and watch their hashes change as well, even though you never touched their trees. That's the whole diagram above, reproduced with your own hands.

If you do have GPG installed (gpg --version), the signing half is worth trying for real too: 

Terminal — a real signature, if you have a GPG key$ gpg --quick-generate-key "Test Key <test@example.com>" default default 1d
$ git config user.signingkey "Test Key"
$ git commit --allow-empty -S -m "a signed commit"
$ git log --show-signature -1

You should see gpg: Good signature from "Test Key <test@example.com>" — the exact same confirmation this chapter's own demo produced. (That key's only good for a day on purpose — the 1d expiry — so there's nothing left lying around afterward worth cleaning up.)

Zoom out, and this chapter has really been about upgrading one word: Chapter 4 taught you a hash makes an object's content provable; this chapter taught you a signature makes an object's author provable, too. Neither one is optional plumbing tucked away for security specialists — hashing is running under every command you've typed since Chapter 3, and signing is one flag away whenever the question "did this really come from who it claims?" actually matters. Sixteen chapters in, you've now seen every load-bearing idea underneath the git binary you use every day. The last chapter turns the telescope around: where does all of this go from here?
