Git Objects: Blobs, Trees, and Commits

Chapter 2 ended with a promise: crack open objects/ and meet the three residents living inside. Time to keep it. Everything Git has ever tracked — every file you've saved, every folder you've organized, every commit message you've typed — boils down to just three kinds of object: blob, tree, and commit. Learn to tell them apart and the rest of Git's "magic" turns out to be three simple shapes, wired together in a graph.

The Blob: Content, and Only Content

Start with the simplest resident. A blob (short for "binary large object") is the raw contents of a single file — nothing else. Not the filename. Not which folder it lives in. Not the permissions, not the timestamp, not even whether it's currently part of any commit. Just the bytes, zlib-compressed and saved under a name that's the SHA-1 hash of those bytes.

That's a stranger idea than it sounds. Most tools think in terms of files: a path, a name, some content. Git's storage layer doesn't know what a "file" is at all — it only knows content. The filename "hello.txt" is metadata that gets attached one level up, by a tree. As far as the blob is concerned, it could be called anything, live anywhere, or live in fifty places at once.

Key Idea

Git is content-addressed: an object's name is a hash of its own content. That means two files with byte-for-byte identical content — anywhere in your repo, in any commit, in any branch — are stored as the exact same blob. Rename a file, duplicate it, move it to a different folder: if the bytes don't change, Git doesn't create a new blob. This is a huge part of why Git repos don't balloon in size the way you'd expect.

The Tree: A Packing List for a Folder

A blob has no idea it's called hello.txt or that it lives inside src/. Something has to record that — that's the tree's entire job. Think of a tree object as a folder's packing list: one line per entry, and each line says a mode, a type, a hash, and a name.

ModePoints toMeaning
100644blobA regular, non-executable file.
100755blobAn executable file (the run permission is stored here, not in the blob).
120000blobA symbolic link — the blob's content is the link's target path.
040000treeA subdirectory — itself another tree object.

That last row is the important one: a tree can point at other trees, not just blobs. Here's a real one, taken straight from a terminal, for a tiny project with hello.txt, README.md, and a src/ folder containing app.py: 

Terminal — the project's root tree, pretty-printed$ git cat-file -p d5a1e31448758f6dd49b702b5407432149091568
100644 blob 0805455a24b6c68fbc38d0fa5d121f735984285d	README.md
100644 blob 8d0e41234f24b6da002d962a26c2495ea16a425f	hello.txt
040000 tree 755d89e1c086583a7bef11c39dbdd6859858a3f6	src

That src row is a tree pointing at another tree. Open that one, and it's the exact same shape all over again: 

Terminal — the src/ subtree$ git cat-file -p 755d89e1c086583a7bef11c39dbdd6859858a3f6
100644 blob 9f1b437537a2acdadafd3174f6f0af9c1a04f5e4	app.py

Trees pointing at trees pointing at blobs, all the way down. That recursion is the whole trick: one single root tree hash can describe an entire directory structure of any depth, and Git can walk it to rebuild your whole project, folder by folder, file by file.

The Commit: A Snapshot With a Name and a Memory

A tree captures what a project looked like. It doesn't say who made it look that way, when, or why — and it has no memory of what came before it. That's the commit's job: it's a small object that points at exactly one tree (the project's root at that moment), lists zero or more parent commits (zero for the very first commit, one for a normal commit, two-plus for a merge), and records who made the snapshot, when, and why: 

Terminal — the first commit (no parent)$ git cat-file -p ffdd52830151ab2e73369039549eaa4f4d8c15a4
tree d5a1e31448758f6dd49b702b5407432149091568
author Ava <ava@example.com> 1752400000 +0000
committer Ava <ava@example.com> 1752400000 +0000

Add hello.txt, README.md, and src/app.py

Notice author and committer are separate lines — usually the same person, but not always: a rebase or a cherry-pick keeps the original author while updating the committer to whoever replayed the commit. Now watch what happens on a second commit, after editing src/app.py and leaving hello.txt and README.md untouched: 

Terminal — the second commit (with a parent)$ git cat-file -p e535ddc30f77a15a15d0f46a82e88d26e52204fd
tree 5db2445422dec397ad6bfcd24babb0d0220a35b3
parent ffdd52830151ab2e73369039549eaa4f4d8c15a4
author Ava <ava@example.com> 1752403600 +0000
committer Ava <ava@example.com> 1752403600 +0000

Tweak src/app.py

That parent line is the entire mechanism behind Git history — each commit points back at the one before it, forming the "graph of snapshots" from Chapter 1's Key Idea. And here's the part worth pausing on: the new root tree (5db24454…) still lists the exact same blob hashes for hello.txt and README.md as before. They didn't change, so Git didn't create new blobs for them — it just reused the old ones inside a new tree. Only src/ and app.py got fresh objects. That's content-addressing paying off in real storage savings, not just as a neat trivia fact.

Pro Tip

git cat-file -p <hash> pretty-prints any object — it figures out on its own whether it's looking at a blob, a tree, or a commit. If you just want to confirm the type first, use git cat-file -t <hash>, and for the size in bytes, git cat-file -s <hash>. And git hash-object <file> without -w is a safe dry run — it prints the hash the content would get without writing anything, handy for checking before you commit to storing it.

Warning

A blob you create by hand with git hash-object -w isn't automatically part of your project — no tree points at it, no commit points at that tree, and no branch points at that commit. Object jargon for this is dangling or unreachable. It's completely fine for the experiment below, but don't rely on a hand-created object as real storage: Git's garbage collector (git gc, or its lower-level cousin git prune) is allowed to sweep away loose objects that nothing reachable points to.

Try It Yourself

This one actually writes something — safely. Open a terminal in any Git repo (or a fresh git init test-quest && cd test-quest) and create a blob directly from a string, with no file involved at all:

printf 'Hello, Git objects!' | git hash-object -w --stdin

You should see this exact hash print back: 8331099cb7a9c6b8990e4af9f1db9ad2a69d5dd3 — the same one, on your machine, as on every machine that has ever run that exact command. That's content-addressing proving itself: the hash depends only on the bytes, never on who typed them or when. Now inspect what you just made:

git cat-file -p 8331099cb7a9c6b8990e4af9f1db9ad2a69d5dd3
git cat-file -t 8331099cb7a9c6b8990e4af9f1db9ad2a69d5dd3
ls .git/objects/83/

The first prints Hello, Git objects! back out, the second prints blob, and the third shows the file Git actually wrote to disk — 31099cb7…, the remaining 38 characters of the hash, sharded under a two-character folder exactly like Chapter 2 described. One object, one file, zero surprises.

Three shapes, one recursive idea: blobs hold content, trees hold structure, commits hold history. Every branch, every checkout, every merge you'll ever run is just Git walking this graph of objects and figuring out which hash to point somewhere next. There's just one loose thread left: where do these forty-character hashes actually come from, and why is a hash collision something you basically never have to think about? That's next.
