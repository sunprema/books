SHA-1 Hashes: Git's Fingerprint System

Chapter 3 left a thread dangling on purpose: every blob, tree, and commit gets a forty-character hex name, and that name is somehow squeezed out of the object's own content. No filename, no counter, no central server handing out IDs — just math. Time to open up the machine that makes those names and see exactly what it's chewing on.

The Recipe: What Actually Gets Hashed

It's tempting to assume Git just runs SHA-1 over the raw bytes of a file. Close, but not quite — and the difference matters. Before hashing, Git glues a small header onto the front of the content: the object's type, a space, the content's length in decimal bytes, and a single NUL byte (\0). Only then does SHA-1 run over the whole thing: 

The exact recipe (same shape for blob, tree, commit, tag)sha1( "<type> <byte-length>\0" + <content> )

# worked example, straight from Chapter 3's lab:
#   content: "Hello, Git objects!"  (19 bytes)
#   header:  "blob 19\0"
$ printf 'blob 19\0Hello, Git objects!' | sha1sum
8331099cb7a9c6b8990e4af9f1db9ad2a69d5dd3  -

That's not a coincidence — it's the exact hash git hash-object --stdin printed back in Chapter 3's lab for that same string. This is the whole trick laid bare: pipe the header plus the content into plain old SHA-1, and you get the object's name. No secret sauce, no salt, no server round-trip — just a well-known algorithm and a tiny, predictable prefix.

Key Idea

The header is part of the content being hashed, which is why a blob and a tag can never accidentally collide even if their raw bytes happen to match — one hashes as "blob <n>\0...", the other as "tag <n>\0...", so the inputs to SHA-1 are different strings even when the payload looks identical. Type is baked into identity.

The Avalanche Effect: One Byte, a Whole New Fingerprint

Cryptographic hash functions have a property called the avalanche effect: flip a single bit anywhere in the input, and the output doesn't shift a little — it scrambles completely, unrecognizably. That's by design. If similar inputs produced similar-looking hashes, you could guess things about content just by eyeballing its fingerprint, and content-addressing would leak information it's supposed to hide. This is exactly why Git can trust a 40-character hash as a stand-in for an entire file, folder tree, or commit history: two meaningfully different objects essentially never end up with the same fingerprint by accident, and the fingerprint gives no hint about how close two objects' contents actually are. "Looks similar" and "hashes similarly" have nothing to do with each other.

Pro Tip

You'll rarely type a full 40-character hash by hand — Git lets you abbreviate one to the shortest prefix that's still unambiguous in your repo (commonly 7–12 characters, controlled by core.abbrev, which defaults to auto and grows the prefix length automatically as the repo grows). Try git rev-parse --short <hash> to see the shortest safe abbreviation for any object you have locally.

Why Not Just Use a Counter?

A simple incrementing ID (commit-1, commit-2, …) would be easier to read — so why didn't Git do that? Because a counter needs a single authority to hand out the next number, and Git has no central authority. Every clone is a full, independent copy of the repository. Content-addressing sidesteps the whole problem: any two people, on any two machines, with no network connection between them, who create the exact same object will independently compute the exact same hash for it. No coordination required, no collisions between clones, and (as Chapter 3 showed) free deduplication as a side effect.

It buys integrity checking too. An object's name is a hash of its own content, so if a single bit flips — disk corruption, a bad transfer, a tampered file — the content no longer matches its own name. Git can detect that mismatch (that's part of what git fsck checks), which is a property a plain filename or counter could never give you.

Warning

Abbreviated hashes are only unambiguous within one repository at one point in time — the same short prefix can point at completely different objects in someone else's fork, or even in your own repo after enough new objects land. Never assume a 7-character hash you copied from a chat message or a script means the same thing on another machine; resolve it with git rev-parse or paste the full hash when it actually matters.

Fun Fact

In February 2017, researchers at CWI Amsterdam and Google (the "SHAttered" attack) published two different PDF files that hash to the identical SHA-1 digest — proof that SHA-1 collisions had gone from theoretical to practically achievable, at a cost of real compute time. It wasn't an attack on any actual Git repository, but Git took it seriously: since Git 2.13 (2017), Git ships with collision-detecting SHA-1 (sha1dc) on by default, which recognizes the mathematical fingerprints of known collision attacks and refuses to hash that content quietly. The longer-term fix is a full migration to SHA-256 object names — you can already try it with git init --object-format=sha256 — but as of today it's still an opt-in, early-stage transition: SHA-1 stays the default, SHA-1 and SHA-256 repositories can't yet interoperate, and major hosts like GitHub don't host SHA-256 repos yet. Slow and careful, exactly how you want a change to your entire object model to go.

Try It Yourself

Reproduce the avalanche-effect diagram above yourself, with nothing but a terminal. First, hash the original string exactly like Chapter 3 did:

printf 'Hello, Git objects!' | git hash-object --stdin

You should get 8331099cb7a9c6b8990e4af9f1db9ad2a69d5dd3 again. Now change one trailing character and hash it again:

printf 'Hello, Git objects?' | git hash-object --stdin

That prints 71bf49369b3f412ed4f63523afaca6df9581347d — a totally different fingerprint from a one-character edit. Want to see the header trick with your own eyes, no Git involved at all? Both strings are 19 bytes, so build the exact bytes Git hashes and feed them straight to a generic SHA-1 tool:

printf 'blob 19\0Hello, Git objects!' | sha1sum

Same hash comes back — 8331099c… — proving there's no Git-specific magic in the algorithm itself, only in the tiny header Git glues on first.

A fingerprint is precise, but nobody wants to type 71bf49369b3f412ed4f63523afaca6df9581347d to check out yesterday's work, and Git itself needs some way to know which commit hash is "current" without you telling it every single time. That's the job of the next chapter's stars: branches and HEAD — human-friendly pointers that ride on top of this hash graph instead of replacing it.
