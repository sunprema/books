Merging and Rebasing: Combining Histories

Chapter 6 closed on a loose thread: every commit chain you've built so far has been a straight line, one parent after another. Real projects don't stay that tidy — you branch off to try something, a teammate branches off too, and sooner or later those two independent snapshot chains have to become one history again. Git gives you exactly two tools for that job, merge and rebase, and they produce genuinely different-shaped graphs. Neither is "the right one" — they're different answers to the same question, and this chapter shows you precisely what each one does to the objects underneath.

Same Goal, Two Different Graphs

Say you branched feature off main a while back, and both branches picked up new commits since. Merging takes both histories exactly as they happened and stitches them together with a brand-new commit that has two parents — the graph keeps every branch point and every commit's original hash intact, forever. Rebasing takes the opposite approach: it picks up your branch's commits, sets them aside, and replays them one at a time on top of wherever the other branch now stands — the result looks like a straight line that never branched at all, but every replayed commit is a genuinely new object with a new hash. Same end goal — one combined history — two very different object graphs left behind.

The Easy Case: Fast-Forward Merge

Start with the simplest possible situation: you branch feature off main, commit once on feature, and never touch main again in the meantime. When you merge feature back into main, Git notices something convenient — main's current commit is already a direct ancestor of feature's tip. There's nothing to combine. Git just slides main's ref file forward to point at the same commit feature already points at. No new commit object gets created at all — this is called a fast-forward merge, and it's exactly the "move a 41-byte pointer" trick from Chapter 5, just triggered by git merge instead of by hand.

Run it for real and Git tells you exactly what it did, in plain words: 

Terminal — a fast-forward merge$ git switch main
Switched to branch 'main'
$ git merge feature
Updating c825f27..88b4339
Fast-forward
 file.txt | 1 +
 1 file changed, 1 insertion(+)
Key Idea

Fast-forward is only possible when the branch you're merging into hasn't moved since you branched off — in other words, its tip is still an ancestor of the tip you're merging in. The instant both sides have picked up commits of their own, Git can no longer just slide a pointer forward, and it falls back to the next section's strategy instead.

When Histories Diverge: The Three-Way Merge

Now the more common case: both branches picked up commits since they split, so neither tip is an ancestor of the other. Git can't just move a pointer — it has to actually combine the two histories. It does this by comparing three trees at once (hence "three-way"): the common ancestor both branches share, your current branch's tip, and the other branch's tip. For every file, if only one side changed it since the ancestor, Git takes that side's version. If both sides changed the same lines, that's a conflict — more on that in a moment.

Whatever the outcome, a successful three-way merge always ends the same way: Git writes one new commit object, and — this is the detail that makes it different from every commit you've seen so far — that object has two parent lines instead of one. Chapter 3 showed you a commit's anatomy has a repeatable parent <hash> line; nothing stops Git from writing that line twice. A merge commit is a completely ordinary commit object that just happens to point at two ancestors instead of one.

Key Idea

A regular commit has zero or one parent line. A merge commit has two (or, for an unusual multi-branch "octopus merge", more). That's the entire difference — there's no separate "merge object" type in Git's four object kinds from Chapter 3. It's a commit like any other, just with more than one thread of history stitched to it.

When Git Can't Decide: Merge Conflicts

Git's three-way comparison only auto-resolves a file when just one side changed it since the common ancestor. If both sides edited the same lines differently, Git has no principled way to pick a winner — so it doesn't try. It stops mid-merge, writes both versions directly into the file separated by conflict markers, and hands the decision to you: 

story.txt — mid-conflict, real outputWelcome to Merge Quest!
<<<<<<< HEAD
Status: villain appears
=======
Status: hero arrives
>>>>>>> hero

Read the markers left to right: everything between <<<<<<< HEAD and ======= is what your current branch has there right now; everything between ======= and >>>>>>> hero is what the branch you're merging in has instead. Nothing is lost — both full versions are sitting right there in the file. Your job is to edit it down to what the file should say, delete all three marker lines, then git add the file and git commit to finish the merge — that commit is the two-parent merge commit from the diagram above.

Warning

A conflicted merge leaves your working directory in a genuinely half-finished state — git status will call the file "both modified" until you resolve it. If it ever looks worse than you expected, you are not stuck: git merge --abort throws the whole attempt away and puts you back exactly where you were before you typed git merge, no harm done. Only commit once every marker is gone — a leftover <<<<<<< line committed by accident is a classic (and embarrassing) mistake.

Rebase: Rewriting History as a Replay

Rebase solves the same "combine two histories" problem with a completely different move. Instead of creating a two-parent commit that remembers both branches diverged, git rebase main (run while on your branch) does this: find every commit your branch has that main doesn't, temporarily set them aside, fast-forward your branch to match main's current tip, then replay each set-aside commit on top, one at a time, as if you'd written it starting from main's new tip all along.

Here's the detail that trips people up, and it's a direct payoff of Chapter 4's lesson that a commit's hash is computed from its content — including its parent line: replaying a commit onto a new parent means its content changed, so its hash must change too. Rebase doesn't move or relabel your old commits — it builds entirely new commit objects, with the same tree changes and message but a different parent and therefore a different hash. The originals don't vanish immediately (they're still sitting in objects/, unreachable, until garbage collection eventually sweeps them — Chapter 9's reflog is how you'd rescue one), but your branch ref now points at the new copies. 

Terminal — a clean rebase, real hashes$ git log --oneline --all --graph
* 270ad0d Main: add README
| * b8b05b8 Topic: add a line
|/
* b13d926 Add notes.txt
$ git switch topic
$ git rebase main
Successfully rebased and updated refs/heads/topic.
$ git log --oneline --all --graph
* 0c3400b Topic: add a line
* 270ad0d Main: add README
* b13d926 Add notes.txt

Notice the graph shape change: before, it forked and never rejoined in this excerpt; after, it's one straight line, as if topic had been written starting from 270ad0d the whole time. That straight-line readability is rebase's whole selling point — and its whole risk, because it's achieved by quietly discarding the original commits and minting new ones.

Rebase can hit conflicts too, for the exact same reason a merge can: if a commit being replayed touches a line the new base already changed, Git pauses mid-replay with the same conflict markers you just saw, except the fix is git rebase --continue instead of a commit: 

Terminal — a rebase conflict, real output$ git rebase main
Rebasing (1/2)
Auto-merging notes.txt
CONFLICT (content): Merge conflict in notes.txt
error: could not apply 084333b... Topic: change A
hint: Resolve all conflicts manually, mark them as resolved with
hint: "git add/rm <conflicted_files>", then run "git rebase --continue".
hint: You can instead skip this commit: run "git rebase --skip".
hint: To abort and get back to the state before "git rebase", run "git rebase --abort".

Same drill as a merge conflict — edit out the markers, git add the file — but because a rebase replays commits one by one, you might hit this more than once in a row, and you finish with git rebase --continue instead of a normal commit. git rebase --abort is your undo button here too, same as git merge --abort.

Pro Tip

The golden rule, straight from Git's own docs: never rebase commits that other people have already pulled. Since rebase mints new hashes for every replayed commit, rewriting a branch someone else has already based work on forces them into a painful manual reconciliation. Rebase freely on your own local, not-yet-pushed branches — it's a great way to tidy up before opening a pull request — but once a branch is shared, reach for merge instead.

Try It Yourself

Time to cause a real conflict on purpose, so it stops being scary. Fresh throwaway repo, one starting file: 

Terminal — set the scene$ mkdir merge-quest && cd merge-quest && git init -q -b main
$ printf "Welcome to Merge Quest!\nStatus: draft\n" > story.txt
$ git add story.txt && git commit -q -m "Start the story"
$ git branch hero

Now edit the same line of story.txt two different ways on two branches — that's the setup that guarantees a conflict instead of a clean auto-merge: 

Terminal — diverge on purpose$ git switch hero
$ sed -i 's/Status: draft/Status: hero arrives/' story.txt
$ git commit -qam "Hero arrives in the story"
$ git switch main
$ sed -i 's/Status: draft/Status: villain appears/' story.txt
$ git commit -qam "Villain appears in the story"

Both branches changed line 2 of the same file, starting from the same ancestor — Git has no way to auto-pick a winner. Merge hero into main and watch it stop mid-merge: 

Terminal — trigger the conflict$ git merge hero
Auto-merging story.txt
CONFLICT (content): Merge conflict in story.txt
Automatic merge failed; fix conflicts and then commit the result.
$ cat story.txt
Welcome to Merge Quest!
<<<<<<< HEAD
Status: villain appears
=======
Status: hero arrives
>>>>>>> hero

Resolve it like a human would — keep the meaning of both sides, delete every marker line, then finish the merge: 

Terminal — resolve and commit$ printf "Welcome to Merge Quest!\nStatus: villain appears, then hero arrives\n" > story.txt
$ git add story.txt
$ git commit -m "Merge branch 'hero'"
$ git log --oneline --graph
*   70e8541 Merge branch 'hero'
|\
| * 4b5f799 Hero arrives in the story
* | b4c5e1b Villain appears in the story
|/
* b59e7e4 Start the story
$ git cat-file -p HEAD | head -3
tree 6198b11bb9691f610e8a05c04d8cd8e67b28ecfe
parent b4c5e1b226a9fd7f9629292c5d1cbe6d27a410e8
parent 4b5f799d7ace29ef768491a36db68a146a1483b4

There it is, straight from cat-file: two parent lines, exactly matching this chapter's diagram. You didn't just read about a three-way merge — you built one, broke it on purpose, and fixed it with your own hands.

Merge and rebase both end with the same combined content — the difference is entirely in the shape of the graph they leave behind, and you now know exactly which commits get created, reused, or replayed either way.
