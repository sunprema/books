Tags and Releases: Marking Important Moments

Chapter 11 closed with a promise: Git has a mechanism for marking a commit you want to remember forever, as opposed to just pausing work mid-flight. This is that mechanism. A tag is a name you plant on one specific commit and leave there — permanently, on purpose. Not "wherever main happens to be today" like a branch, which keeps sliding forward with every commit. A tag says: this commit, right here, is v1.0.0, and it always will be.

The Problem: Hashes Don't Mean Anything to Humans

Chapter 5 solved "how do I avoid typing a 40-character hash to find my work" with branches — but a branch is a moving target by design; that's the whole point of it. Releases need the opposite property. When you ship version 2.4.0 of a project, you need a name that points at exactly the commit that got tested, packaged, and shipped — today, next month, and five years from now, even after hundreds more commits have landed on main. A branch can't promise that; the moment someone commits again, main means something new. Git's answer is a second kind of ref, one built to sit still: the tag.

Lightweight Tags: A Ref That Just Doesn't Move

The simplest tag is almost embarrassingly familiar, if you've read Chapter 5: git tag <name> writes a 41-byte file under .git/refs/tags/ containing the hash of the current commit — the exact same trick .git/refs/heads/ uses for branches. Pick up right where the sword-and-shield story from Chapter 11 left off: 

Terminal — three commits, then a lightweight tag$ mkdir tag-quest && cd tag-quest && git init -q -b main
$ printf "Welcome to Staging Quest!\nHP: 100\n" > player.txt
$ git add player.txt && git commit -q -m "Start the story"
$ printf "Welcome to Staging Quest!\nHP: 100\nInventory: sword\n" > player.txt
$ git add player.txt && git commit -q -m "Add sword to inventory"
$ printf "Welcome to Staging Quest!\nHP: 100\nInventory: sword, shield\n" > player.txt
$ git add player.txt && git commit -q -m "Add shield to inventory"
$ git tag v0.9-lw
$ git cat-file -t v0.9-lw
commit

git cat-file -t — the same plumbing command Chapter 3 used to inspect blobs and trees — says the object type behind v0.9-lw is commit. Not "tag." Just commit, plain as day, because a lightweight tag isn't its own object at all. It's a ref file pointing straight at the commit, indistinguishable in principle from a branch — the only difference is that nothing ever updates it automatically. Print it and you get an ordinary commit, tree and parent and all: 

Terminal — cat-file -p on a lightweight tag: just a commit$ git cat-file -p v0.9-lw
tree 719a0c81b3e985458a3386ce5f1405e93bd0c695
parent 66d309e8b491332f884d575d1298893477111353
author Ava <ava@example.com> 1783936800 +0000
committer Ava <ava@example.com> 1783936800 +0000

Add shield to inventory

Annotated Tags: A Tag Object With Its Own Story

An annotated tag is a genuinely different animal. Add -a, and Git creates a brand-new object — a fourth kind, alongside the blob/tree/commit trio from Chapter 3 — that carries its own tagger name, email, timestamp, and message, and can optionally be GPG-signed. That object, not the commit, is what the tag name actually points at: 

Terminal — git tag -a -m, then cat-file -t$ git tag -a v1.0.0 -m "Release 1.0.0: sword and shield update"
$ git cat-file -t v1.0.0
tag

Read that again — the type isn't commit this time, it's tag. That's a real, distinct Git object type with its own entry in .git/objects/, exactly the way a blob or a tree is. Print it with the same cat-file -p and the shape is completely different from anything Chapter 3 showed you: 

Terminal — cat-file -p on an annotated tag: a distinct object$ git cat-file -p v1.0.0
object a44937d407b7d667d4f3e28fa812b7610683204b
type commit
tag v1.0.0
tagger Ava <ava@example.com> 1783937100 +0000

Release 1.0.0: sword and shield update

No tree, no parent — because this isn't a commit. It's a small record with exactly one job: hold metadata about the act of tagging, and point at the commit it's tagging via that object line. (Sign it with git tag -s instead of -a and this object also carries a PGP signature block, verifiable later with git tag -v — the mechanism Chapter 16 revisits when it covers trusting your history.)

Key Idea

Git only has four object types, ever: blob (file content), tree (a directory listing), commit (a snapshot + metadata), and tag (a signed, dated pointer to another object — almost always a commit). A lightweight tag doesn't use that fourth type at all; it's just a ref, a plain text pointer, the same mechanism as a branch. Only an annotated tag actually creates one.

Proving It: Two Different Hashes for "the Same" Tag

The cleanest proof that these are genuinely different structures: ask Git for each tag's own hash with git rev-parse. A lightweight tag's hash is the commit's hash — there's nothing else it could be. An annotated tag's hash is the tag object's own checksum, which is different from the commit it describes: 

Terminal — rev-parse: one hash matches the commit, one doesn't$ git rev-parse v0.9-lw
a44937d407b7d667d4f3e28fa812b7610683204b
$ git rev-parse v1.0.0
7aa8a92882271b1a4c90a65677f5f15b544ab394
$ git rev-parse v1.0.0^{commit}
a44937d407b7d667d4f3e28fa812b7610683204b

v0.9-lw and the commit share one hash — because they're the same object. v1.0.0 resolves to 7aa8a92…, the tag object's own checksum, which is nowhere else in this book's history — and only the ^{commit} "peel it" suffix, or a command that automatically peels tags for you, gets you back to the familiar a44937d…. Every day-to-day Git command (checkout, log, diff) peels this automatically without you noticing; it's only visible when you go looking for it with plumbing like rev-parse or cat-file.

Semantic Versioning: The Naming Convention for Releases

Git doesn't care what you name a tag — v1.0.0, release-day, and final-final-2 are all equally valid ref names to it. But the ecosystem overwhelmingly settled on one convention for release tags: Semantic Versioning (semver), written vMAJOR.MINOR.PATCH — the leading v is a near-universal habit, not a Git requirement. Each number means something specific:

  • MAJOR (the first number) bumps when you ship a breaking change — something that could break code depending on you. v1.x.x → v2.0.0.
  • MINOR (the middle number) bumps when you add functionality that's backward-compatible — new features, nothing removed or changed underneath existing users. v1.3.x → v1.4.0.
  • PATCH (the last number) bumps for backward-compatible bug fixes only — nothing new, nothing broken, just corrected. v1.4.2 → v1.4.3.

The payoff is that anyone — a teammate, a package manager, a CI pipeline — can glance at v2.1.0 vs. v1.9.4 and immediately know whether upgrading is safe or demands a careful read of the changelog, purely from the numbers, before reading a single line of release notes. Reach for an annotated tag for every real release — you want the tagger, date, and message on record for exactly this commit, forever. Save lightweight tags for throwaway, personal bookmarks you don't intend to publish, like marking "the commit right before I tried that risky refactor."

Listing, Inspecting, and Removing Tags

A few commands round out day-to-day tag management — all of them read naturally once you know a tag is just a specially-shaped ref: 

Terminal — list, annotate-aware listing, and decorated log$ git tag -l
v0.9-lw
v1.0.0
$ git tag -n
v0.9-lw         Add shield to inventory
v1.0.0          Release 1.0.0: sword and shield update
$ git log --oneline --decorate -1
a44937d (HEAD -> main, tag: v1.0.0, tag: v0.9-lw) Add shield to inventory

git tag -l (or bare git tag) lists every tag name, alphabetically, with no way to tell lightweight from annotated at a glance. git tag -n is the fix — it prints each tag's first message line right alongside it, pulled from the commit message for lightweight tags and the tag object's own message for annotated ones. git show <tag> shows the full tag details (annotation included, for annotated tags) plus the commit's diff, in one command. And a tag you no longer want is one flag away from gone: git tag -d v0.9-lw deletes it locally — the commit itself is untouched, only the pointer goes away.

Warning

Once you've published a tag — pushed it, told people it's "the v1.0.0 release" — don't move it. Retagging the same name onto a different commit (git tag -f v1.0.0 <other-commit>) rewrites what that name means on your machine, but anyone who already fetched the old v1.0.0 now has a different commit under the same name than you do — a quiet, confusing kind of history rewrite, in the same family as the force-pushes Chapter 10 warned about. If you tagged the wrong commit, delete the tag, tell anyone who pulled it, and cut a new version number instead of reusing the old one.

Sharing a Tag: Pushing It to a Remote

Here's a detail that trips people up on their first release: git push does not send tags by default. Same rule as branches — nothing leaves your machine until you say so explicitly. Chapter 13 covers remotes properly, but the release story isn't complete without at least seeing the two ways to hand a tag over: 

Terminal — pushing a tag, two ways$ git push origin v1.0.0
To …
 * [new tag]         v1.0.0 -> v1.0.0
$ git push origin --tags
To …
 * [new tag]         v0.9-lw -> v0.9-lw

git push origin <tagname> pushes exactly one named tag — the precise, deliberate move for "ship this release." git push origin --tags pushes every tag you haven't pushed yet, in one go, which is convenient but also less careful — it'll happily send along any scratch lightweight tags you never meant to publish. (There's a middle ground, too: git push --follow-tags pushes only the annotated tags that point at commits you're already pushing — handy once you're comfortable with remotes.) Once pushed, anyone who clones or fetches the repository gets the tag along with the history it points at, and can check out v1.0.0 to see exactly what shipped — byte for byte, no matter how far main has moved on since.

Pro Tip

Default habit: annotated tags for anything you'll ever share — releases, milestones, anything with your name on it — and reserve lightweight tags for private, throwaway bookmarks on your own machine. Most hosting platforms (GitHub included) build their "Releases" feature directly on top of annotated tags, using the tagger/message metadata to populate the release page — another reason the extra -a -m "…" keystrokes are worth it every time.

Try It Yourself

Ship your own "release," start to finish. Fresh throwaway repo, same story as before: 

Terminal — set the scene$ mkdir tag-quest && cd tag-quest && git init -q -b main
$ printf "Welcome to Staging Quest!\nHP: 100\n" > player.txt
$ git add player.txt && git commit -q -m "Start the story"
$ printf "Welcome to Staging Quest!\nHP: 100\nInventory: sword\n" > player.txt
$ git add player.txt && git commit -q -m "Add sword to inventory"
$ printf "Welcome to Staging Quest!\nHP: 100\nInventory: sword, shield\n" > player.txt
$ git add player.txt && git commit -q -m "Add shield to inventory"

Plant a lightweight tag first, and confirm with cat-file -t that it's nothing but a commit in disguise: 

Terminal — a lightweight tag is just a ref$ git tag v0.9-lw
$ git cat-file -t v0.9-lw
commit
$ git rev-parse v0.9-lw HEAD

Those two hashes on the last line should be identical — the tag and HEAD name the same object. Now cut the real release with an annotated tag, and watch the type change: 

Terminal — an annotated tag is its own object$ git tag -a v1.0.0 -m "Release 1.0.0: sword and shield update"
$ git cat-file -t v1.0.0
tag
$ git cat-file -p v1.0.0
$ git rev-parse v1.0.0
$ git rev-parse v1.0.0^{commit}

Confirm for yourself: the plain git rev-parse v1.0.0 prints a hash that appears nowhere else in your git log — that's the tag object's own checksum — while the ^{commit}-suffixed version peels through it back to the familiar commit hash. Run git tag -n and check both messages show up correctly. Last step: stand up a throwaway "remote" (a real remote is next chapter's topic, but a tag isn't shared until it's pushed somewhere) and publish the release: 

Terminal — a bare repo standing in for a real remote, and the push$ git init -q --bare ../tag-quest-remote.git
$ git remote add origin ../tag-quest-remote.git
$ git push -q origin main
$ git push origin v1.0.0
To …/tag-quest-remote.git
 * [new tag]         v1.0.0 -> v1.0.0
$ git ls-remote --tags origin

Look closely at the ls-remote output: the annotated tag shows up twice — once as the tag object's hash, and once more suffixed ^{} with the commit hash it peels to. That's the two-hop structure from this chapter's diagram, confirmed one more time, straight from a remote you control end-to-end.

Zoom out and tags turn out to be the smallest idea in this book with the biggest real-world job: a lightweight one is just a ref that's agreed to hold still, and an annotated one adds a single extra object — a signed, dated note explaining why this commit matters — sitting between the name and the snapshot. Everything else, from semantic versioning to "Releases" pages on GitHub, is built on that one small primitive. But a tag sitting only on your laptop isn't a release anyone else can see — you saw that the moment git push quietly skipped it. Next chapter opens up what origin actually is, and how push and pull move commits, branches, and tags between repositories for real: remote repositories.
