The .git Folder: Git's Secret Hideout

Last chapter ended with a dare: run ls -la .git in a real repo and just look. If you actually did it, good — you already met the cast of this chapter. If you skipped it, no judgment, open a terminal now. We're about to walk into the one folder that quietly is your entire project's history, and figure out what every name inside it means.

The Repository's Brain, Hiding in Plain Sight

Here's a fact that surprises almost everyone the first time they hear it: your source files — the ones you actually edit in your editor — are not what makes a folder a Git repository. They're just the working copy, a convenient, human-readable view of one snapshot. The actual repository, the thing Git means when it says "repo," is a single hidden folder sitting right next to them: .git.

Run git init in any folder and Git creates exactly one thing: a .git subfolder. That's it. That's the whole trick. Every commit you've ever made, every branch, every tag, every bit of history — all of it lives inside that one folder, as plain files on disk. Delete your working files and keep .git, and git checkout rebuilds everything from scratch. Delete .git and keep your working files, and Git amnesia sets in instantly — you're left with a folder of code and zero memory of how it got that way.

Key Idea

.git is not a cache, a log file, or a convenience — it is the repository. Copy that one folder to another machine (that's basically all git clone does) and you've copied the entire project's history, every branch, every commit, everything. Nothing outside it is required for Git to know your past.

A Quick Tour of the Neighborhood

Pop .git open and you'll find a small, unglamorous set of files and folders — no database engine, no server, nothing exotic. Here's the map of who lives where:

EntryKindWhat it actually does
objects/folder The database. Every blob, tree, and commit you've ever created, addressed by SHA-1 hash. Chapter 3's whole neighborhood.
refs/folder Sticky-note pointers — branches (refs/heads/), remote-tracking branches (refs/remotes/), tags (refs/tags/).
HEADfile "You are here" — which branch (or commit) is currently checked out.
configfile This repo's own settings: remotes, branch tracking, core options.
logs/folder The reflog — a running history of everywhere each ref has pointed. Your future undo button (Chapter 9).
indexfile The staging area, as a binary snapshot — what git add actually edits (Chapter 8).
hooks/folder Sample and custom scripts Git runs on events like commit or push (Chapter 14).
descriptionfile Cosmetic only — used by the old gitweb viewer. Almost nobody touches this.
info/excludefile A local, un-committed .gitignore — ignore rules just for you, on this one clone.

objects/ — Where Everything Actually Lives

This is the one that matters most, and we'll spend all of Chapter 3 there — but the short version: objects/ is a content-addressable store. Every file's contents, every folder listing, and every commit gets hashed with SHA-1 and saved as a compressed file named after its own hash. To keep one folder from holding millions of files, Git shards them: the first two hex characters of the hash become a subfolder, and the remaining thirty-eight become the filename inside it.

Three kinds of object live here, and you'll meet each one properly next chapter: blob (raw file contents), tree (a folder listing), and commit (a snapshot pointer with a message and a parent). Once a repo has a lot of history, Git also compresses many loose objects together into objects/pack/ — a story for the performance chapter later on.

refs/ — Sticky Notes, Not Boxes

A "reference" in Git is almost comically simple: it's a plain text file whose entire content is a commit hash. refs/heads/main is your main branch — open it and you'll see forty characters of hex and nothing else. refs/tags/ works the same way for tags, and refs/remotes/origin/ holds your local repo's memory of where a remote's branches last pointed.

That's genuinely the entire mechanism behind a branch — a named file holding one hash, that Git moves forward automatically every time you commit. We'll come back and prove it in Chapter 5. (Once a repo collects hundreds of branches and tags, Git may compress them into a single packed-refs file for speed — same idea, tidier storage.)

HEAD — You Are Here

Remember the compass from Chapter 1? HEAD is that compass, and it's a single small text file. Almost always it holds a symbolic reference — not a hash, but a pointer to a ref: 

Terminal — a normal HEAD$ cat .git/HEAD
ref: refs/heads/main

Translation: "HEAD is wherever the main branch currently points." Commit, and Git updates refs/heads/main to the new commit; HEAD itself doesn't have to change, because it's just pointing at the pointer. If you ever check out a specific commit hash directly instead of a branch, HEAD holds that raw hash instead — that's the famous detached HEAD state, and now you know exactly what "detached" means: the compass is pointing at a commit directly, with no named branch attached to carry it forward.

config — This Repo's Rulebook

.git/config is plain, readable INI-style text holding settings that apply only to this repository — layered on top of your global ~/.gitconfig and Git's system-wide defaults. A typical one looks like this: 

Terminal — a typical .git/config$ cat .git/config
[core]
	repositoryformatversion = 0
	filemode = true
	bare = false
[remote "origin"]
	url = https://github.com/you/your-project.git
	fetch = +refs/heads/*:refs/remotes/origin/*
[branch "main"]
	remote = origin
	merge = refs/heads/main

That [remote "origin"] block is how git push and git pull know where "home base" is, and the [branch "main"] block is how Git knows which remote branch main tracks. Run git config --list and you're seeing this file merged with your global settings — a handy way to check without opening anything.

logs/ — Your Undo History, Already Waiting

Every time a ref moves — a new commit, a branch switch, a reset, even a rebase — Git appends a line to a log file in logs/. logs/HEAD tracks everywhere HEAD has ever pointed, and logs/refs/heads/main tracks main specifically. This quiet little audit trail is called the reflog, and it's the exact mechanism behind almost every "I thought I lost my work" recovery story — it gets a full chapter of its own later (Chapter 9), because it deserves one.

Pro Tip

Everything in .git except objects/ and index is plain, boring text — totally safe to open with cat or a text editor and just read. Looking around never changes anything. objects/ is zlib-compressed binary, so cat-ing one directly just prints garbage; we'll open those properly with git cat-file next chapter.

Warning

Looking is safe. Editing by hand usually isn't. Hand-editing refs/, HEAD, or the index without knowing exactly what you're doing can leave your repo in a confusing, half-broken state. And never, ever delete the whole .git folder to "start fresh" unless you've genuinely backed up or already pushed everything — it doesn't tidy up your project, it erases its memory completely, with no reflog left to save you.

Try It Yourself

Same rule as last chapter: everything here is read-only, nothing gets touched. Open a terminal inside any Git repo (your own project, or a throwaway one from git init test-quest) and run these three, one at a time:

ls -la .git
cat .git/HEAD
cat .git/config

Match what you see against the tour above: find objects and refs in the listing, confirm HEAD says ref: refs/heads/<your branch>, and spot the [remote "origin"] block in config if your repo has one. If something doesn't match — say, HEAD holds a raw hash instead of a ref: line — that's not broken, that's a detached HEAD, and you already know what that means now.

That's the whole floor plan: objects/ stores everything, refs/ points at it, HEAD says which point you're standing on, and config/logs//index handle the bookkeeping around them. Nothing in this folder is magic — it's compressed files, text files, and a handful of pointers. Next chapter, we open objects/ itself and meet its three residents up close: blobs, trees, and commits.
