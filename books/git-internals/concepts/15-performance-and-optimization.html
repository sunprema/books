Performance and Optimization: Git Under Load

Chapter 14 closed with a promise: this book has mostly been about correctness — what Git stores and how — and now it's time to talk about what happens when a repository stops being a tidy little demo and starts being a five-year-old monorepo with forty thousand commits, a design team's folder of 200MB Photoshop files, and three hundred contributors all pushing at once. Git doesn't panic. It has a whole toolkit for staying fast under that kind of load — some of it running quietly in the background without you ever asking, some of it one command away when you need to reach for it yourself.

Why Big Repos Actually Get Slow

"Big" isn't one problem, it's three, and they have completely different fixes — which is exactly why Git ships more than one maintenance tool instead of just one big "make it fast" button.

Deep history is commit count: tens of thousands of commits piling up over years. Chapter 6 already told you this one is mostly fine — every commit is a self-contained snapshot, so git checkout and git log don't get slower just because history got longer, they just have a bigger graph to walk. The real cost of deep history shows up in disk space and network transfer: every commit, tree, and blob object Git has ever created is still sitting in .git/objects/ unless something cleans it up.

Wide trees are file count at any single snapshot — a repo with 500,000 files means every git status has to stat 500,000 files against the index, even if you only touched one of them. That's a working-copy problem more than an object-store one; it's the reason genuinely enormous monorepos (think: an entire OS, or Windows' own source tree) lean on extra tooling on top of plain Git, which the "fun fact" callout below touches on.

Huge binary files are the odd one out, and the one Git's core design genuinely struggles with — we'll get to exactly why, and the standard fix, later in this chapter.

Key Idea

Chapter 6 showed you that packing — deduping and delta-compressing loose objects into a .pack file — is what keeps Git's storage layer small. This chapter is about the commands that trigger and manage that process on purpose: git gc, git prune, and git fsck. Same underlying machinery, now with your hands on the controls.

git gc: Git's Own Cleanup Crew

gc stands for garbage collect, and it's genuinely the closest thing Git has to a "tidy my whole repo" button. One command runs a whole checklist: it packs your loose objects into compact .pack files, repacks any existing packs that have gotten fragmented, compresses refs/ into a single packed-refs file instead of thousands of tiny files, and — carefully — prunes loose objects that are both unreachable and old enough that nothing should still need them.

You almost never have to run it by hand, because Git already is: most porcelain commands (commit, merge, pull, and friends) call git gc --auto at the end, which checks two thresholds before doing any real work — roughly 6,700 loose objects (gc.auto), or more than 50 packs sitting around unconsolidated (gc.autoPackLimit). Below those numbers, --auto is a no-op that costs you nothing.

Pro Tip

Want to watch it happen instead of taking the diagram's word for it? git count-objects -v is the live readout — count is loose objects, in-pack is objects already folded into a pack, packs is how many pack files exist. Run it, run git gc, run it again. Chapter 6 showed you this same trick; this chapter's lab below runs it for real, on purpose, around a repo cleanup instead of a plain commit.

git prune: The Blunt Instrument Underneath

git gc doesn't invent object deletion — it delegates that specific job to git prune, a smaller, sharper tool you can also run directly. prune does exactly one thing: it finds loose objects that are unreachable — nothing in refs/, no reflog entry, nothing points at them, directly or transitively — and deletes them from .git/objects/ for good.

The safety net is a grace period: by default, prune only deletes unreachable loose objects older than two weeks (gc.pruneExpire). That's deliberate slack for exactly the kind of thing Chapter 9 covered — a commit you just detached from with reset --hard is unreachable by the strict definition, but it's still sitting safely in your reflog, and two weeks is plenty of time to notice and rescue it before prune (or the prune that gc runs internally) ever touches it.

Warning

Two flags turn off the safety net, and Chapter 6 already warned you about the first: git gc --prune=now (equivalently, git prune --expire=now) skips the two-week grace period and deletes eligible objects immediately. The word "eligible" is doing real work in that sentence, though — an object the reflog still remembers isn't eligible at all, grace period or not, exactly like Chapter 9 explained. To actually force a specific dangling commit gone, you'd need to expire the reflog entries pointing at it first (git reflog expire --expire=now --expire-unreachable=now --all), then prune. That's two deliberate, separate "yes, really" steps — treat needing both of them together as your last warning before something becomes properly, permanently gone.

git fsck: The Repo's Health Inspector

fsck — "file system check," a name Git borrowed straight from Unix — doesn't clean anything up. It looks. It walks every object it can reach, verifies the object store isn't corrupted, and reports anything unusual: broken links, objects with the wrong hash for their content, or objects that exist but nothing points to. Run it plain and, on a healthy repo, it should print nothing at all — silence is the good outcome.

Two flags make it genuinely useful for detective work rather than just corruption-checking:

  • git fsck --unreachable lists every object nothing points to — the raw material prune would consider deleting.
  • git fsck --dangling is the more useful subset: objects that are unreachable and aren't referenced by any other unreachable object either — in practice, almost always the tip commit of some abandoned work, exactly the thing you'd want to recover.

There's a wrinkle worth knowing before you try this yourself, and it's the exact same reflog awareness from the warning above: by default, fsck treats anything your reflogs still remember as reachable, so a commit you just orphaned with reset --hard won't show up as unreachable or dangling yet — the reflog is quietly keeping it alive. Add --no-reflogs to see the object store's raw reachability, ignoring that safety net. The lab below shows you exactly what that looks like, both ways.

Try It Yourself

Build a small repo, orphan a commit on purpose, and watch the whole cast — fsck, count-objects, and gc — work together. Fresh throwaway repo, same as always: 

Terminal — a small guild roster, two commits$ mkdir perf-quest && cd perf-quest && git init -q -b main
$ printf "Guild Roster\nMembers: 1\n" > roster.txt
$ git add roster.txt && git commit -q -m "Found the guild"
$ echo "Members: 2 (+rogue)" >> roster.txt
$ git add roster.txt && git commit -q -m "Recruit a rogue"
$ git log --oneline
b754f30 Recruit a rogue
a52058d Found the guild

Recruit one more member, then decide — loudly, and with reset --hard — that it was a mistake: 

Terminal — a bad recruit, then reset --hard to abandon it$ echo "Members: 3 (+wizard, unstable build)" >> roster.txt
$ git add roster.txt && git commit -q -m "Recruit a wizard (buggy respec)"
$ git log --oneline
0ffe6e2 Recruit a wizard (buggy respec)
b754f30 Recruit a rogue
a52058d Found the guild
$ git reset -q --hard HEAD~1
$ git log --oneline
b754f30 Recruit a rogue
a52058d Found the guild

The wizard's commit, 0ffe6e2, is gone from git log — but you already know from Chapter 9 that reset --hard only moves a pointer. Ask fsck plainly, and it stays quiet: 

Terminal — fsck finds nothing... at first$ git fsck --unreachable
$

Nothing printed — because your reflog still remembers 0ffe6e2 as somewhere HEAD recently stood, so by default fsck counts it as reachable. Tell it to ignore the reflog and look at raw object-store reachability instead: 

Terminal — the real, un-safety-netted picture$ git fsck --unreachable --no-reflogs
unreachable commit 0ffe6e2f560d54091925974edb06031bd45984f1
unreachable tree f437b92b92f51e22a03c1f40bb29cad464d1022e
unreachable blob 0c8ca5bbfb96de06266b57a5bbb207d129fbe25e
$ git fsck --dangling --no-reflogs
dangling commit 0ffe6e2f560d54091925974edb06031bd45984f1

There's the whole abandoned snapshot: the commit, its tree, and the one blob it introduced. --dangling narrows it to just the commit — exactly the hash you'd hand to git checkout -b if you'd actually wanted that wizard recruit back. Now watch count-objects before and after a cleanup: 

Terminal — count-objects, then git gc, then count-objects again$ git count-objects -v
count: 9
size: 36
in-pack: 0
packs: 0
$ git gc
$ git count-objects -v
count: 0
in-pack: 9
packs: 1

All nine loose objects — six from your two real commits, plus the wizard's abandoned commit, tree, and blob — got folded into one packfile. Notice the wizard's commit didn't get deleted: it's well inside the two-week grace period and the reflog still vouches for it, so gc packed it right alongside everything else instead of pruning it. That's the safety net from the warning above, working exactly as designed — tidy, but not trigger-happy.

Where Git's Model Breaks: Huge Binary Files

Everything above assumes Git's delta compression can do its job, and that job depends on a quiet assumption: that two versions of a file are usually similar, so a delta — "start from this base, apply these small byte-level edits" — comes out tiny. That's true for source code, almost by definition. It's often false for binary formats. Re-save a Photoshop file, re-export a video, or rebuild a compiled asset, and the bytes on disk can scramble almost completely even though the change "feels" small to a human — most binary formats aren't laid out for byte-level diffing the way plain text is. Git still tries, but the delta often ends up nearly as big as the file itself, so every version effectively gets stored close to in full. Multiply that by a growing folder of 200MB design files and a few years of history, and you get a repository where git clone takes an actual coffee break.

Pro Tip

The standard fix isn't "compress harder" — it's Git LFS (Large File Storage), and the trick is refreshingly simple: git lfs track "*.psd" tells Git that matching files shouldn't live in the repo at all. Instead, Git commits a tiny text pointer file (just an object ID and a size) in their place, while a clean filter uploads the real bytes to separate LFS storage on push, and a smudge filter downloads them back on checkout. Your history stays full of small, fast-to-clone commits; the big binaries live somewhere built to hold them. It's not built into core Git — it's a companion tool most hosts (GitHub, GitLab, Bitbucket) support natively — but it's the answer nine times out of ten when someone says "Git is bad with huge binary files."

Fun Fact

Ever wonder how GitHub or GitLab keep this snappy at a scale where "big repo" means millions of objects and thousands of pushes a minute? A few of the same ideas from this chapter, industrialized. GitHub replicates every repository across multiple file servers (a system called Spokes) and runs git gc as a scheduled background job instead of waiting on your local --auto threshold; GitLab does something similar with a dedicated Git RPC service called Gitaly, sharding repositories across nodes. Both also lean on newer Git features built for exactly this problem: protocol v2 (2018) lets the server filter which refs it even advertises during a fetch, instead of listing every branch and tag up front — a huge win for repositories with hundreds of thousands of refs. And when the problem is history depth or working-tree size rather than object count, clients can ask for less in the first place: a shallow clone (git clone --depth=1) skips old history entirely, while a partial clone (git clone --filter=blob:none, since Git 2.19) fetches every commit and tree up front but defers downloading file contents until you actually check them out. None of it changes the object model this book has spent fourteen chapters on — it's the same blobs, trees, and commits, just served smarter.

Zoom out and this chapter has really been about one idea repeated three ways: Git's object model (Chapters 2–6) is deliberately simple and honest, and performance is what you get by layering smart storage tricks — packing, pruning, filtering — on top of that simple model instead of complicating the model itself. A commit is still just a pointer to a tree, whether it's sitting loose, packed, replicated across three servers, or fetched partially over a slow connection. That same "keep the model simple, layer trust and safety on top" instinct is exactly where the book heads next — Chapter 16 asks a different question about all these objects: not just "are they intact," which fsck already checks, but "can I actually trust that this history is who it says it is?"
