Chapter 13 of 17 · Talking to origin

Remote Repositories: How Push and Pull Work

Chapter 12 closed on an awkward truth: a tag sitting only on your laptop isn't a release anyone else can see — git push quietly skipped it, no error, no warning. That's your cue that everything so far in this book, every blob, tree, commit, branch, and tag, has happened in exactly one .git folder: yours. This chapter opens up the second folder — the one on GitHub, GitLab, a teammate's laptop, wherever — and the handful of commands that move objects and refs between the two without either repository trusting the other blindly.

A Remote Is Just Another .git Folder, Somewhere Else

Strip away the GitHub web UI, the green "Code" button, the pull request reviews — underneath, a remote is nothing more exotic than another repository with its own objects/ and refs/, reachable over a URL instead of sitting on your local disk. Most remotes you'll touch day to day are bare repositories — no working directory, just the .git internals this whole book has been dissecting, because a server has no use for checked-out files, only the history. GitHub and GitLab are, structurally, enormous racks of bare repositories with a UI bolted on top.

Git never assumes there's exactly one "true" copy. Every clone is a full, independent repository — complete object database, its own refs, its own HEAD. What connects them is a name and a URL your local .git/config remembers for you, so you never have to retype it: 

Terminal — naming a remote, and listing it back$ git remote add origin https://github.com/you/quest.git
$ git remote -v
origin  https://github.com/you/quest.git (fetch)
origin  https://github.com/you/quest.git (push)

origin is a convention, not a keyword — it's just the name Git hands your first remote by default when you git clone something, exactly the way main is a convention for your default branch, not a reserved word. Nothing stops you from adding a second remote named upstream or backup pointing somewhere else entirely; most solo projects just never bother.

Push: Handing Your Commits Over

git push is the simplest of the three commands this chapter covers, once you say it plainly: copy any objects the remote is missing, then ask it to move one of its ref files to point at your latest commit. That's the entire operation — the same blob/tree/commit objects from Chapter 3, transferred byte-for-byte, and a 41-byte ref file updated to match, this time on a machine you don't have a terminal open on. 

Terminal — the first push of a new branch$ git push -u origin main
To ../quest-remote.git
 * [new branch]      main -> main
branch 'main' set up to track 'origin/main'.

The -u (or --set-upstream) flag only matters the first time you push a branch: it tells Git "remember that my local main corresponds to origin's main," so every push and pull after this one can drop the origin main arguments entirely and just be git push / git pull. Skip -u and pushing still works fine — you'll just have to spell out the remote and branch by name each time, or set the tracking link up later with git branch -u origin/main.

Remote-Tracking Branches: origin/main Is Also Just a Ref

Here's the detail that trips people up first, because the name origin/main looks like a branch living on the remote. It isn't. It's a bookmark that lives entirely on your machine, recording where origin's main was the last time you talked to it. Chapter 5 already gave you the exact mechanism this reuses: a branch is a 41-byte text file under .git/refs/heads/ holding a commit hash. A remote-tracking branch is precisely the same trick, filed under a different folder: 

Terminal — same mechanism, different folder$ cat .git/refs/heads/main
571b9730cf3baa8e184d5f4c6b0b6f0341312209
$ cat .git/refs/remotes/origin/main
571b9730cf3baa8e184d5f4c6b0b6f0341312209
$ find .git/refs -type f
.git/refs/heads/main
.git/refs/remotes/origin/main
Key Idea

refs/heads/main is a branch you move, one commit at a time, by committing on it. refs/remotes/origin/main is a branch only Git itself moves, and only during network operations — fetch, pull, push — to record what it just saw on the other end. You can git checkout origin/main to look around (you'll land in detached HEAD, Chapter 5's other big idea), but you can't commit onto it the way you commit onto main. It's a read-only snapshot of somebody else's ref, not a workspace of your own.

Fetch: Look, Don't Touch

git fetch downloads whatever commits and objects the remote has that you don't, and updates your remote-tracking branches to match — full stop. It never touches refs/heads/main, never touches your working directory, never merges anything. It's the safest command in this whole chapter, which is exactly why experienced Git users reach for it constantly: there's no way to lose work by fetching. 

Terminal — before and after a fetch, side by side$ git branch -vv
* main 2f82b5d [origin/main] Start the story

$ git fetch
From ../quest-remote
   2f82b5d..571b973  main       -> origin/main

$ git branch -vv
* main 2f82b5d [origin/main: behind 1] Start the story

$ git status
On branch main
Your branch is behind 'origin/main' by 1 commit, and can be fast-forwarded.
  (use "git pull" to update your local branch)

nothing to commit, working tree clean

Read those two git branch -vv lines closely — that's the whole chapter in one diff. Before the fetch, main and origin/main agree, both at 2f82b5d. After it, origin/main jumped to 571b973 — a teammate's commit Git just learned about — while main sat exactly still at 2f82b5d. Your working directory hasn't changed one byte. You now simply know something you didn't before: origin has a commit you don't. What you do with that knowledge is up to you.

Pull: Fetch, Then Merge (or Rebase), Back to Back

git pull is not a third, separate mechanism — it's a shorthand for running two commands you already know in sequence: git fetch, immediately followed by git merge origin/main (assuming your current branch tracks origin/main, which -u set up back in the push section). Fetch does the download and moves origin/main, exactly as above; the merge step is Chapter 7's material, applied automatically to close the gap between your main and the origin/main that fetch just updated. 

Terminal — continuing on from the fetch above$ git pull
Updating 2f82b5d..571b973
Fast-forward
 player.txt | 1 +
 1 file changed, 1 insertion(+)

$ git branch -vv
* main 571b973 [origin/main] Add sword to inventory

Now both ref files agree again — 571b973 in each — because this particular merge was a fast-forward: your main had made zero commits of its own since the last sync, so Git just slid its pointer forward to match origin/main, no new merge commit needed, same mechanism Chapter 5 described for any fast-forward. If you'd committed locally too in the meantime, pull would fall back to a real three-way merge (Chapter 7 again) and might ask you to resolve a conflict — git fetch first, on its own, is exactly how you preview whether that's coming before it happens.

Prefer a straight line in your history instead of a merge commit every time you sync? git pull --rebase swaps the second half of the operation — instead of merging origin/main into your branch, it replays your local commits on top of it, the same rebase mechanics Chapter 7 walked through, just triggered automatically as pull's second step. Plenty of teams set this as their default with git config --global pull.rebase true so every git pull behaves this way without the flag.

Warning

git push only succeeds when it's a fast-forward for the remote too — if someone else pushed since your last fetch, a plain push is rejected with "updates were rejected because the remote contains work that you do not have." That rejection is a safety feature, not a bug: it stops you from silently erasing a teammate's commits. Never reach for git push --force to make the error go away on a shared branch — it overwrites origin's ref to match yours, discarding whatever commits it pointed at before, the same history-rewrite danger Chapter 10 warned about with reset --hard, now inflicted on everyone who already fetched the old history. Fetch, merge or rebase properly, resolve anything that conflicts, then push.

Pro Tip

git branch -vv is the single most useful command in this chapter for a reason — it prints every local branch next to the remote-tracking branch it's linked to, and how many commits you're ahead or behind by, all in one line, with zero network calls (it's reading refs you've already fetched, not asking the server anything fresh). Get in the habit of a bare git fetch before you start reading diffs or planning a merge — it costs nothing, touches nothing, and tells you immediately whether you're about to work from stale information.

Try It Yourself

Play both sides of a team on one machine: "you" and a "teammate," each with their own clone of a shared fake remote. Start with a bare repo standing in for GitHub — the same trick Chapter 12 used, this time with -b main so every clone agrees on the default branch from the very first commit: 

Terminal — a bare "remote," and your own clone of it$ git init -q --bare -b main quest-remote.git
$ git clone -q quest-remote.git quest-you
$ cd quest-you
$ printf "Welcome to Remote Quest!\nHP: 100\n" > player.txt
$ git add player.txt && git commit -q -m "Start the story"
$ git push -u origin main
To …/quest-remote.git
 * [new branch]      main -> main
branch 'main' set up to track 'origin/main'.

Now open a second clone in a different folder to play your teammate, and have them add to the story and publish it — from your side, this is indistinguishable from a real coworker pushing to GitHub while you're not looking: 

Terminal — a teammate's clone, making their own commit$ cd .. && git clone -q quest-remote.git quest-teammate
$ cd quest-teammate
$ printf "Welcome to Remote Quest!\nHP: 100\nInventory: sword\n" > player.txt
$ git commit -q -am "Add sword to inventory"
$ git push origin main
To …/quest-remote.git
   2f82b5d..571b973  main -> main

Back in quest-you — the one you never touched during that last step — check where things stand, then fetch and watch only one of the two ref files move: 

Terminal — cd ../quest-you, then fetch$ cd ../quest-you
$ git branch -vv
* main 2f82b5d [origin/main] Start the story
$ git fetch
From …/quest-remote
   2f82b5d..571b973  main       -> origin/main
$ git branch -vv
* main 2f82b5d [origin/main: behind 1] Start the story
$ cat player.txt

Your working file still says HP: 100 with no inventory line — the fetch changed what Git knows, not what's on disk. origin/main now points at 571b973; your own main is still sitting at 2f82b5d, exactly where you left it. Confirm it in the raw ref files if you want zero doubt left: 

Terminal — the two ref files, disagreeing on purpose$ cat .git/refs/heads/main
2f82b5df016ff720d4aa2715a763c447b36d1d6d
$ cat .git/refs/remotes/origin/main
571b9730cf3baa8e184d5f4c6b0b6f0341312209

Last move: close the gap. git pull runs the fetch again (nothing new this time) and then merges — and since you made zero local commits in between, it's a clean fast-forward, no merge commit required: 

Terminal — git pull closes the gap$ git pull
Updating 2f82b5d..571b973
Fast-forward
 player.txt | 1 +
 1 file changed, 1 insertion(+)
$ git branch -vv
* main 571b973 [origin/main] Add sword to inventory
$ cat player.txt
Welcome to Remote Quest!
HP: 100
Inventory: sword

Both ref files now hold 571b973…, your working copy shows the sword, and you never once had to think about merge conflicts — because you fetched first and looked before you leaped. That habit is 90% of working with remotes well.

Zoom out and the whole chapter is really one idea wearing three names: push sends your refs and objects out, fetch brings the remote's refs and objects in without touching your work, and pull is fetch plus the merge (or rebase) needed to fold that new information into the branch you're standing on. Every remote-tracking branch like origin/main is, underneath, the exact same 41-byte ref file Chapter 5 opened up — Git never stopped using its one trick, it just started using it to talk to other repositories too. None of this, though, stops a teammate — or you, at 2am — from pushing something malformed or forgetting to run the test suite first. Chapter 14 covers the mechanism Git gives you to catch that automatically, before or after it happens: hooks.
