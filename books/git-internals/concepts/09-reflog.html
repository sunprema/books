Chapter 9 of 17 · The panic button

Reflog: Git's Time Travel Log

"I deleted my branch. Can I get it back?" If you've never typed that sentence into a search bar at 2am, heart sinking, you will one day — and when you do, the answer is almost always yes. Chapter 8 ended with a promise: a second, quieter safety net has been recording every move you've made in this book without you ever asking it to. This chapter pulls back the curtain on the reflog, and by the end you'll have deleted a branch on purpose, felt the little jolt of "wait, is it actually gone?", and pulled it right back out of the log where Git was keeping it the whole time.

The Panic, in Real Time

Here's the scenario, and it's not exaggerated — ask any developer with a few years in and they'll have a version of this story. You finish a feature on a branch, merge it (or think you did), and clean up after yourself: 

Terminal — the moment of panic$ git branch -D feature-boss-fight
Deleted branch feature-boss-fight (was 28ce70b).
$ git branch -v
* main 2653c06 Start the story
$ git log --oneline --all
2653c06 Start the story

And there it is — the sinking feeling. git branch -v shows only main. git log --all, which walks every ref Git knows about, shows exactly one commit. Two whole commits of work — "Add boss room" and "Defeat the boss" — have vanished from every command that lists history. Before you panic further: they haven't gone anywhere. git branch -D only ever deletes the 41-byte pointer file in refs/heads/ (Chapter 5's whole lesson). The commit objects themselves are still sitting in objects/, completely intact — they're just unreachable now, because nothing points at them anymore. Or rather, nothing you've checked yet.

Key Idea

Deleting a branch never deletes commits — it deletes a pointer to them. A commit only truly disappears once nothing reachable references it and enough time passes for Git's garbage collector to sweep it away. In between those two moments, there's a window where the commit is "lost" from the normal views but still fully recoverable — and the reflog is exactly the map that tells you where to look.

What the Reflog Actually Is

Chapter 2 already showed you where it lives: .git/logs/. The reflog (short for "reference log") is Git quietly keeping a diary for every ref that matters, especially HEAD. Every single time HEAD starts pointing at a different commit — because you committed, checked out a branch, merged, rebased, reset, or even just detached — Git appends one line to logs/HEAD: the old hash, the new hash, who did it, when, and a short note about which command moved it. Individual branches get their own diary too: logs/refs/heads/main tracks only main's journey, separately from HEAD's.

You don't read those files directly — you ask for them with git reflog, which (per its own docs) is shorthand for git reflog show HEAD: it defaults to showing HEAD's log specifically, newest entry first. Each line has an address of its own, HEAD@{n} — "where HEAD was n moves ago" — and that address works anywhere Git expects a commit: git show HEAD@{2}, git diff HEAD@{5} HEAD@{0}, all of it.

Reading the Trail Back to the Lost Tip

Back to the panic. Instead of despairing at git log, ask the reflog what it remembers: 

Terminal — git reflog, right after the delete$ git reflog
2653c06 HEAD@{0}: checkout: moving from feature-boss-fight to main
28ce70b HEAD@{1}: commit: Defeat the boss
ba4132c HEAD@{2}: commit: Add boss room
2653c06 HEAD@{3}: checkout: moving from main to feature-boss-fight
2653c06 HEAD@{4}: commit (initial): Start the story

Read it like a diary, top to bottom, newest to oldest: "HEAD just checked out from feature-boss-fight back to main. Before that, HEAD was at a commit called 'Defeat the boss.' Before that, 'Add boss room.'" That second line, HEAD@{1}, is exactly the commit the deleted branch used to point at — the tip of the work you thought you'd lost, sitting right there with its hash spelled out: 28ce70b. The commit was never in danger. Only your name for it was gone.

Recovery is one command — hand that hash to git checkout -b (or the equivalent git switch -c) to create a brand new branch pointing right at it, exactly the way Chapter 5's hand-rolled refs/heads/feature trick worked: 

Terminal — resurrecting the branch$ git checkout -b feature-boss-fight 28ce70b
Switched to a new branch 'feature-boss-fight'
$ git log --oneline
28ce70b Defeat the boss
ba4132c Add boss room
2653c06 Start the story
$ git branch -v
* feature-boss-fight 28ce70b Defeat the boss
  main               2653c06 Start the story

Full history, same hash, same file contents, same branch name if you want it back exactly — because you rebuilt the pointer that pointed at real, never-deleted commit objects the whole time. This is not a special trick reserved for reflog experts; it's the single most common real-world use of the reflog, and Pro Git's own "Data Recovery" chapter walks through this exact move.

Pro Tip

The reflog rescues more than deleted branches. A botched git rebase, an over-eager git reset --hard, even accidentally amending the wrong commit — all of them just move where a ref points, and the reflog remembers every stop along the way. git reflog (or its longer twin, git log -g, which shows full commit messages per entry) is worth trying before you conclude anything is really gone. If you know roughly when, git reflog also accepts time-based lookups like HEAD@{2.hours.ago} or HEAD@{yesterday}.

What the Reflog Won't Do

Two limits worth knowing before you lean on it as a permanent backup plan. First: the reflog only records history for refs this repository has actually seen move. If a commit was never checked out, committed, merged into, or otherwise made current here, there's no reflog entry pointing at it — git fsck --full (which scans every object directly) is the fallback for that rarer case. Second, and this is the one that trips people up the most:

Warning

The reflog is local only — it is never pushed, pulled, fetched, or cloned, and it doesn't live forever. It's a file under .git/logs/ on your machine, describing what your HEAD and branches did on your clone. Your teammate's reflog knows nothing about your deleted branch, and pushing or cloning never transfers it — only commits, trees, blobs, and refs travel over the network. It also isn't permanent: Git's garbage collector (git gc, which runs automatically in the background as git gc --auto) expires old entries — by default, reflog entries still reachable from a branch expire after 90 days (gc.reflogExpire), and entries that have become unreachable, like our deleted branch's old tip once nothing else points at it, expire after just 30 days (gc.reflogExpireUnreachable). Delete a branch and mean to recover it — do it soon, on the same machine, before gc has a chance to sweep the entry away.

Try It Yourself

Live the whole panic-and-rescue story yourself, start to finish. Fresh throwaway repo: 

Terminal — set the scene$ mkdir reflog-quest && cd reflog-quest && git init -q -b main
$ printf "Welcome to Staging Quest!\nHP: 100\n" > player.txt
$ git add player.txt && git commit -q -m "Start the story"
$ git checkout -q -b feature-boss-fight
$ echo "Boss room unlocked" >> player.txt
$ git add player.txt && git commit -q -m "Add boss room"
$ echo "Boss defeated" >> player.txt
$ git add player.txt && git commit -q -m "Defeat the boss"
$ git checkout -q main

Now do the scary part on purpose — delete the branch you just built, the way a real "cleanup" slip could actually happen: 

Terminal — the delete, and the panic check$ git branch -D feature-boss-fight
Deleted branch feature-boss-fight (was 28ce70b).
$ git branch -v
* main 2653c06 Start the story
$ git log --oneline --all
2653c06 Start the story

Two commits, gone from every listing command you know. This is the exact moment to reach for the reflog instead of panicking: 

Terminal — find it$ git reflog
2653c06 HEAD@{0}: checkout: moving from feature-boss-fight to main
28ce70b HEAD@{1}: commit: Defeat the boss
ba4132c HEAD@{2}: commit: Add boss room
2653c06 HEAD@{3}: checkout: moving from main to feature-boss-fight
2653c06 HEAD@{4}: commit (initial): Start the story

Copy the hash next to HEAD@{1} — that's your deleted branch's tip — and hand it straight to git checkout -b: 

Terminal — resurrection$ git checkout -b feature-boss-fight 28ce70b
Switched to a new branch 'feature-boss-fight'
$ cat player.txt
Welcome to Staging Quest!
HP: 100
Boss room unlocked
Boss defeated
$ git branch -v
* feature-boss-fight 28ce70b Defeat the boss
  main               2653c06 Start the story

There it is — both commits, the exact same hashes, the file exactly as you left it. You didn't restore a backup or undo anything magical; you just told Git to point a new sticky note at a commit it never actually threw away. That's the entire trick, and now you've done it with your own hands instead of just reading about it at 2am.

Zoom out and the reflog is really just Chapter 5's lesson turned into a safety net: branches are cheap pointers, commits are the durable thing underneath, and the reflog is the log of every place a pointer has ever stood. It won't save you from a bad merge conflict or a typo in your code — but "I deleted the wrong branch," "I reset too hard," and "I rebased and now something's missing" are all, deep down, the same problem: a pointer moved and you want to know where it used to be. Next chapter takes that same idea and points it the other direction — deliberately moving pointers on purpose, safely, with git reset and git revert.
