Chapter 8 of 17 · Before the snapshot

The Staging Area (Index): Git's Waiting Room

Chapter 2 pointed at one line in the .git floor plan and told you to come back later: index, "the staging area, as a binary snapshot — what git add actually edits." Chapter 7 leaned on it too, quietly, every time it told you to git add a resolved conflict before committing. Later is now. Every commit you've made since Chapter 1 secretly passed through this one file first — it's time to open it up and watch it work.

Three Copies of Your Project, Live at Once

Here's the idea that makes the rest of this chapter click: at any moment, Git is tracking three different versions of your project simultaneously, and they can all disagree with each other. Your working directory is whatever's on disk right now — the file you're mid-edit in your editor. Your staging area (also called the index) is the version you've deliberately marked as "this is what I want in the next commit." And HEAD — the tip of your current branch — is the last version you actually committed, already sealed into history. git add and git commit are just the two moves that shuffle content between these three, one step at a time.

That's why the index deserves the nickname "waiting room": nothing reaches the repository directly from your editor. Every change queues up in the index first, and git commit only ever looks at what's staged — never at whatever unstaged edits happen to be sitting in your working directory at the time. Forget to git add a change, and git commit will happily commit without it, no warning given.

Meet .git/index: A Snapshot Under Construction

The staging area isn't a metaphor or a UI concept — it's one real file, .git/index, sitting right there among the entries Chapter 2 mapped out. It's binary, not text, so cat just prints garbage at you (same rule as objects/). Structurally it's a flat list of entries, one per tracked file, and each entry holds the file's path, its file mode, and — this is the important part — the SHA-1 hash of a blob object, exactly the kind of object Chapter 3 introduced. git add doesn't just make a note that "this file changed." It immediately computes the file's blob hash, writes that blob into objects/ if it isn't already there, and points the index entry at it. The object exists the moment you stage it — long before you ever type git commit.

You can read the index yourself with the plumbing command git ls-files --stage, which prints exactly those fields: 

Terminal — reading the index directly$ git ls-files --stage
100644 e360e42923f59636bba0ec059c74701ce6a7f2f0 0	player.txt

Left to right: 100644 is the file mode (a normal, non-executable file), the long hex string is the staged blob's SHA-1 — the exact hash git add just computed — the 0 is the stage number, and player.txt is the path. Stage 0 means "normal, no conflict." That number only becomes interesting during an unresolved merge, and this is where Chapter 7's conflicts stop being spooky: while <<<<<<< markers sit in your file, the index is quietly holding three versions of it at once — stage 1 for the common ancestor, stage 2 for your side (HEAD), stage 3 for theirs: 

Terminal — the index mid-conflict, real output$ git ls-files --stage
100644 e5c5c5583f49a34e86ce622b59363df99e09d4c6 1	f.txt
100644 c3f770ebb024c04d4c7909c55d8aefd6d6e1b4a5 2	f.txt
100644 c90e8dae6fb411bcbf6105d0ea4e8bd558715963 3	f.txt

That's not three files — it's one path, f.txt, listed three times because Git can't collapse it to a single answer yet. The moment you edit the file, delete the markers, and run plain git add f.txt, all three stage entries collapse back down to a single stage 0 — your resolution — and that's precisely the signal Git uses to know the conflict is settled.

Key Idea

git diff with no arguments compares your working directory against the index — "what have I changed that isn't staged yet?" git diff --staged (or its older spelling, git diff --cached) compares the index against HEAD — "what will actually go into the next commit?" Two different questions, two different diffs, and mixing them up is a classic way to be baffled by an empty git diff right before committing something you swore you hadn't reviewed yet.

git commit Only Ever Reads the Index

This is the payoff of everything above: when you finally run git commit, Git does not look at your working directory at all. It takes whatever is currently sitting in the index, converts that flat list of path→blob entries into a tree object (nesting sub-trees for sub-directories, exactly as Chapter 3 described), and wraps that tree in a new commit object pointing at your current HEAD as its parent. The index, at the instant you commit, becomes the new snapshot — word for word the mechanism Chapter 6 called "commits are snapshots, not diffs." The staging area is where that snapshot gets built, one git add at a time, before it's sealed.

That's also why staging is selective by design, not all-or-nothing. You're free to edit five files and git add only two of them — the other three stay unstaged, excluded from the next commit, still sitting untouched in your working directory for later. Go finer still with git add -p, which walks you through a file change hunk by hunk and lets you stage half a file's edits and leave the rest for a separate commit. The index doesn't care whether a change came from one file or one line — it just holds whatever you told it to hold.

Pro Tip

git restore --staged <file> is the modern (Git 2.23+) way to unstage — it copies the file's entry back out of HEAD into the index and leaves your working directory completely alone. If you're reading an older script or a Stack Overflow answer from before 2019, you'll see the equivalent written as git reset HEAD -- <file>. Same effect, older spelling — both just move content backward one step, from repository to index.

Warning

git restore --staged <file> and plain git restore <file> look like the same command with an optional flag. They are not. Without --staged, git restore overwrites your working directory file with whatever's in the index — which means any edit you haven't staged yet is gone, no confirmation asked, no reflog entry to rescue it (working-directory changes were never committed anywhere for the reflog to remember). Read the flag before you hit enter. If in doubt, git status already tells you the exact right command to type — it's printed right there under "Changes to be committed" and "Changes not staged."

Try It Yourself

Time to watch the index change with your own eyes instead of taking the book's word for it. Fresh throwaway repo, one starting file: 

Terminal — set the scene$ mkdir staging-quest && cd staging-quest && git init -q -b main
$ printf "Welcome to Staging Quest!\nHP: 100\n" > player.txt
$ git add player.txt && git commit -q -m "Start the story"

Now take some damage — edit the file, and check git status before touching the index at all: 

Terminal — a working-directory edit, unstaged$ printf "Welcome to Staging Quest!\nHP: 80\n" > player.txt
$ git status
On branch main
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
	modified:   player.txt

no changes added to commit (use "git add" and/or "git commit -a")

Git already sees the change — it's just not queued up yet. Stage it, and watch the same command change its story completely: 

Terminal — git add, then git status again$ git add player.txt
$ git status
On branch main
Changes to be committed:
  (use "git restore --staged <file>..." to unstage)
	modified:   player.txt

Same one-line edit, completely different section of the output — "not staged" became "to be committed." Now look at what git add actually did to the index: 

Terminal — the staged blob, in the index$ git ls-files --stage
100644 e360e42923f59636bba0ec059c74701ce6a7f2f0 0	player.txt
$ git hash-object player.txt
e360e42923f59636bba0ec059c74701ce6a7f2f0

That's not a coincidence — git hash-object just recomputes the same SHA-1 git add already wrote into the index a moment ago. The blob for "HP: 80" genuinely exists in objects/ right now, staged and ready, even though you haven't committed anything. Change your mind and back it out: 

Terminal — unstage with git restore --staged$ git restore --staged player.txt
$ git status
On branch main
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
	modified:   player.txt

no changes added to commit (use "git add" and/or "git commit -a")
$ git ls-files --stage
100644 af0893a73b352784c1f208d5d40a79016c12769c 0	player.txt
$ cat player.txt
Welcome to Staging Quest!
HP: 80

Read that last block closely — it's the whole chapter in one screenful. git status is back to "not staged," exactly where it started. git ls-files --stage now shows a different hash than before — af0893a7…, the original "HP: 100" blob from your first commit — because the index snapped back to matching HEAD. And player.txt on disk still says "HP: 80," untouched, because you used --staged. Three areas, three different contents, all live at once, precisely as promised.

You've now watched a single edit travel every stop on the line: working directory, staged in the index as a real blob, unstaged again with the working copy left intact. That's the whole mechanism behind every git add you've typed since Chapter 1 — and behind every conflict resolution in Chapter 7, which was really just you manually collapsing three index stages back down to one. There's a second, quieter safety net running underneath everything you've done in this book, one that's been recording every move without your ever needing to ask — every commit, every reset, every branch switch. Chapter 9 pulls the curtain back on the reflog, Git's built-in time machine.
