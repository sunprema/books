Chapter 6 of 17 · Settling an old argument

Commits Are Snapshots, Not Diffs

Chapter 5 ended on a promise, and it's time to cash it in: a commit is a full snapshot of your project, not a diff. If you've used Subversion, CVS, or Perforce before — or you've just heard someone describe version control as "a big list of changes" — that instinct is worth actively unlearning here, because Git was built from day one to reject it.

Where the Diff Delusion Comes From

It's an honest mistake. Plenty of older, centralized version control systems really do store commits as diffs — a base file plus a chain of "apply these line changes" records, because disk space used to be brutally expensive and that's the most space-frugal way to describe a small edit. Git showed up in 2005 and made the opposite bet on purpose: every commit gets its own complete, self-contained picture of the entire project, full stop.

You've actually already seen the proof, back in Chapter 3 — you just might not have clocked what it meant yet. A commit object has exactly four kinds of line: tree, zero-or-more parent, author/committer, and a message. Look again: there's no diff line, no changes field, nowhere for a list of edits to even live. tree <hash> points at one fully-formed root tree, recursively describing every file in the project at that exact moment. That's the entire mechanism. A commit doesn't know what changed — it just knows what is.

Key Idea

Every single commit, including the ten-thousandth one in a five-year-old repo, points at a tree that describes the entire project — not just the files that changed. Want to see exactly what your code looked like on day one? Check out the first commit. Want day 4,000? Check out that one. Git never has to "rebuild" a snapshot by replaying history from the start — every commit is already a finished, ready-to-read picture.

The Photo Album, Not the Edit List

Here's the mental picture that sticks: think of a diff-based history as a notebook of touch-up instructions — "brighten by 10%, crop 2px off the left, fix the red-eye." To see what photo #47 actually looked like, you'd need photo #1 and then every single instruction from #2 through #47, applied in order, one after another. Skip one page of the notebook and the picture comes out wrong.

Git's history is a photo album instead. Every page is a complete, printed photograph. Want to see day 47? Flip straight to page 47. No replaying, no notebook, no risk of a torn page corrupting everything after it. That's not just a tidier metaphor — it's the reason git checkout, git bisect, and jumping to any arbitrary commit are all roughly the same speed whether that commit is one day old or ten years old. There's nothing to replay.

So How Does It Stay Small?

Fair question — "just store everything, every time" sounds like it should balloon a repo's size fast. Two separate mechanisms keep it from happening, and it's worth being precise about which is which, because they operate at completely different layers.

The first is the one the diagram above is really about, and you already met it in Chapter 3: content addressing. Because a blob's name is a hash of its bytes, an unchanged file gets the exact same blob in every commit it appears in — Git never stores it twice. That's not a background optimization; it's baked directly into the object model itself.

The second is a genuine storage-layer trick, and it's worth naming correctly: on disk, each object Git writes starts out loose — one file per object, individually zlib-deflated, sharded into those two-character folders Chapter 2 showed you. That's simple and fast to write, but deflating one tiny object at a time can't spot similarity between objects — two nearly-identical 20KB files still cost roughly 20KB and 20KB, compressed separately. So once loose objects pile up — you run git gc, or push to a remote, or just accumulate enough of them — Git packs them: it finds objects that are similarly sized and named across your whole object store, keeps one of them in full, and stores the rest as tiny delta records — byte-level "start from that base object, then apply these edits" instructions — all bundled into one .pack file plus an .idx index for fast lookup. Pro Git's own worked example packs two versions of a 22KB source file down to roughly 7KB total, because the second version compresses to just a handful of delta bytes against the first.

Key Idea

Delta compression is real, and it's genuinely how Git keeps large repos small on disk — but it lives entirely in the storage layer, underneath the object model, not inside it. git cat-file -p always hands back the full, un-delta'd content no matter whether the object is loose or buried in a pack, and every commit still conceptually points at exactly one complete tree. Packing is Git being clever about how it shelves the photo album — it is not Git secretly switching to instruction-list photos when you're not looking. If someone tells you "well actually, Git does store diffs, in packfiles" — they're half right about the mechanism and fully wrong about the model.

Pro Tip

git count-objects -v shows you the split live — count is loose objects sitting around uncompressed against each other, in-pack is how many have already been folded into a packfile. Run git gc and watch count drop to (close to) zero while in-pack and packs pick up the slack. Packing also picks its base object smartly: within a delta chain, the most recent version of a file is usually the one kept in full, with older versions stored as deltas against it — since you check out recent history far more often than ancient history, that keeps the common case fast.

Warning

Chapter 3 warned you that a dangling object — one nothing reachable points at — is fair game for cleanup. Here's the sharp edge: plain git gc gives unreachable loose objects a grace period (roughly two weeks by default) before pruning them, in case a reflog entry or an in-progress operation still needs them. But git gc --prune=now (or git prune run directly) skips that grace period entirely and deletes eligible objects immediately. That's fine for objects you genuinely don't need — but if you're mid-rescue on a commit you just detached from, don't reach for --prune=now until you're sure a branch or tag points at it.

Try It Yourself

Prove the dedup with your own eyes instead of taking the diagram's word for it. Fresh throwaway repo, same as always: 

Terminal — commit 1: two files$ mkdir snapshot-quest && cd snapshot-quest && git init -q
$ printf "Hello, Git objects!\n" > hello.txt
$ printf "# Snapshot Quest\n" > README.md
$ git add hello.txt README.md
$ git commit -q -m "First commit: hello.txt and README.md"
$ git cat-file -p HEAD^{tree}
100644 blob c44af9f35795fc5beab4538c7627ee05cf70cf9d	README.md
100644 blob b2a2cb66744c59559a3d7728b5cfeff1cb817fcf	hello.txt

Now change only hello.txt and commit again, leaving README.md completely untouched: 

Terminal — commit 2: edit hello.txt only$ printf "Hello, Git objects!\nOne more line.\n" > hello.txt
$ git add hello.txt
$ git commit -q -m "Second commit: tweak hello.txt only"
$ git cat-file -p HEAD^{tree}
100644 blob c44af9f35795fc5beab4538c7627ee05cf70cf9d	README.md
100644 blob 5e77b2bb3b0434f4641bc1e5c7b5adbaf278afbe	hello.txt

Put the two cat-file -p outputs side by side. hello.txt's hash changed — b2a2cb6… became 5e77b2b… — because its content changed. But README.md lists c44af9f3… in both trees, character for character. That's not a coincidence and it's not Git being lazy about updating a field — it's the exact same blob object, reused, because Git hashed README.md's current content, found that hash already sitting in objects/, and pointed straight at it instead of writing a duplicate. Two full, independent snapshots; one shared file stored exactly once.

One loose thread left, and it's a big one: every snapshot chain so far in this book has been a straight line, one commit after another. Real projects branch, and eventually those branches need to become one history again — sometimes cleanly, sometimes not. That's exactly where Chapter 7 picks up: merging and rebasing, and what actually happens to the commit graph when two independent snapshot chains collide.
