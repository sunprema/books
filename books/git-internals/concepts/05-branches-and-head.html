Branches and HEAD: Pointers, Not Copies

Chapter 4 ended on a promise: nobody wants to type 71bf49369b3f412ed4f63523afaca6df9581347d just to switch back to yesterday's work. This chapter delivers on it. Branches and HEAD are the human-friendly layer Git bolts on top of the raw hash graph — and Chapter 2 already spoiled most of the trick when it peeked inside refs/. Time to open that folder for real and prove there's nothing else going on underneath.

A Branch Is Just a File With a Hash In It

Here's the single fact this whole chapter hangs on: a branch is not a copy of your project, a separate folder, or anything resembling the "save slot" you'd get in most other tools. It's a 41-byte text file — forty hex characters plus a trailing newline — sitting in .git/refs/heads/, and its entire content is the hash of one commit. That's it. That's the whole feature.

Run git branch feature and Git does exactly one thing: it writes the hash currently under HEAD into a new file, .git/refs/heads/feature. No files are copied, no new objects/ entries get created, nothing about your working directory changes yet. That's why creating a branch in Git is instantaneous even in a repository with a million commits — older version control systems that copy the whole tree to branch can take seconds or minutes; Git just writes 41 bytes.

Key Idea

A branch doesn't contain commits — it points at one, and that one commit points at its parent, which points at its parent, and so on back through history (Chapter 3's whole story). "Being on a branch" just means HEAD is following that one pointer. Delete refs/heads/feature and every commit reachable from somewhere else survives untouched; only the sticky note is gone, never the commits themselves — unless nothing else points at them, in which case Git's garbage collector eventually cleans them up (a story for a later chapter).

HEAD: You Are Here, One Level Up

Chapter 2 already showed you HEAD's contents: a single line, ref: refs/heads/main. Now the payoff: HEAD is itself a pointer — but it almost never points at a commit hash directly. It points at a branch, which points at a commit. Two hops, not one. That extra layer of indirection is the entire mechanism behind "switching branches":

  • git checkout feature rewrites .git/HEAD to say ref: refs/heads/feature — one line changes, and your working directory updates to match whatever feature points at.
  • Make a new commit while on feature, and Git writes the new commit object, then updates refs/heads/feature to the new hash. HEAD itself doesn't change at all — it's still just saying "follow feature."
  • That's why committing on a branch feels like the branch "moves forward": the ref file's content changes, but which file HEAD points at usually doesn't.

In the diagram above, that's the top arrow: HEAD points at the box labeled main, and main points at the commit. If you were on feature instead, that top arrow would simply swing left to point at the feature box — same mechanism, different target.

Detached HEAD: When the Compass Skips a Step

So what happens if you check out a raw commit hash instead of a branch name — git checkout d8afa6d, say? Git can't write ref: refs/heads/d8afa6d into HEAD, because there's no branch by that name. So it does the only other thing it can: it writes the hash itself straight into HEAD, skipping the branch layer entirely. That's the whole definition of detached HEAD — HEAD pointing directly at a commit, with no named branch riding along to carry it forward.

Detached HEAD isn't broken or dangerous by itself — it's exactly what happens whenever you check out a tag, a commit from git log, or someone else's PR by hash to poke around. Looking, running the code, even building it: all completely safe. The one thing to watch is committing while detached: those new commits are real objects in objects/, but no ref file points at the newest one, so if you switch to a branch afterward with nothing pointing at that tip, it becomes effectively invisible — Chapter 9's reflog is the safety net that gets it back.

Warning

If you make commits in detached HEAD and want to keep them, create a branch before you switch away: git switch -c rescue-branch (or git branch rescue-branch) while still detached, right where Git's own advice text points you. Switch straight to main first and those commits have no branch holding them up — not gone forever (the reflog remembers), but no longer one git checkout away either.

Pro Tip

Modern Git split checkout's two jobs — "switch branches" and "restore files" — into two clearer commands: git switch <branch> and git restore <file>. git switch still detaches HEAD the same way when you hand it a raw hash (git switch --detach <hash>), it's just harder to trigger by accident than the old all-purpose checkout. Either command edits the exact same one-line HEAD file underneath — new verbs, same plumbing.

Try It Yourself

This one edits files by hand — use a throwaway repo, not a real project: mkdir branch-quest && cd branch-quest && git init -q, then make one commit so there's a hash to point at:

echo "hello git internals" > file.txt
git add file.txt && git commit -m "First commit"

Grab that commit's hash and write it into a brand new branch file yourself — no git branch command, just a text redirect, exactly like Git would: 

Terminal — hand-rolling a branch$ git rev-parse HEAD
04bc1e6164b18fdbdd880de81b1f224c7265bbf4
$ echo 04bc1e6164b18fdbdd880de81b1f224c7265bbf4 > .git/refs/heads/feature
$ git branch
  feature
* main

Git sees feature and lists it right alongside main — because as far as Git's concerned, there's no difference between a branch it created and one you hand-wrote. Now commit again on main to make the two branches diverge, and hop over to your hand-made one: 

Terminal — diverge, then switch$ echo "more work" >> file.txt
$ git add file.txt && git commit -m "Second commit"
$ git branch -v
  feature 04bc1e6 First commit
* main    d8afa6d Second commit
$ git checkout feature
Switched to branch 'feature'

Last move: detach on purpose. Ask main for its hash and check that out directly instead of by name: 

Terminal — going detached$ git checkout $(git rev-parse main)
Note: switching to 'd8afa6dfed1c3c278bec2f2465fa2f6372f81dda'.

You are in 'detached HEAD' state. You can look around, make experimental
changes and commit them, and you can discard any commits you make in this
state without impacting any branches by switching back to a branch.
…
HEAD is now at d8afa6d Second commit
$ cat .git/HEAD
d8afa6dfed1c3c278bec2f2465fa2f6372f81dda

Read that last line twice — HEAD now holds the raw hash, no ref: prefix, no branch name in sight. That's the entire detached HEAD mechanism laid bare, with your own eyes on the file. git checkout main whenever you're ready to reattach.

Zoom out and the whole chapter compresses to one sentence: branches are cheap, disposable pointers, and HEAD is a pointer to whichever pointer you're currently riding. Neither one stores a single byte of your project — that job belongs entirely to the commit graph underneath, which is exactly where we're headed next. Chapter 6 opens up a commit object itself and settles a common misconception for good: a commit is a full snapshot, not a diff.
