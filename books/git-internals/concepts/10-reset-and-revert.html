Chapter 10 of 17 · Moving pointers on purpose

Reset and Revert: Undoing Mistakes

Chapter 9 ended on a promise: this chapter takes the same idea — "a pointer moved, and you want it to point somewhere else" — and turns it around. Instead of recovering from an accidental pointer move, you're about to make one on purpose. Two tools do this: git reset, which rewinds your current branch to an earlier commit, and git revert, which leaves history alone and adds a brand-new commit that cancels one out. They solve the same problem — "undo this" — in opposite ways, and picking the wrong one on the wrong branch is how "undo my mistake" becomes a bigger mistake. By the end of this chapter you'll know exactly which one to reach for, and you'll have the reflog standing by in case you don't.

git reset: The Three-Tree Model, Rewound

Chapter 8 gave you the mental model this whole chapter builds on: Git is always juggling three trees at once — your working directory, the index (staging area), and HEAD (the tip of your current branch, sealed history). git add moves content forward from working directory into the index; git commit moves it forward again from index into HEAD. git reset is the command that moves things backward — and its three flags, --soft, --mixed, and --hard, are really just three answers to one question: how far backward do you want the damage to reach?

Per Git's own docs, reset always works in the same order, and each flag just decides where it stops:

  1. Move HEAD (and the branch it points to) to the target commit. Every mode does this.
  2. Update the index to match that target commit. --soft stops before this step.
  3. Update the working directory to match the index. Only --hard goes this far.

Put in plain English: --soft just relabels which commit your branch points to — everything you'd committed is now sitting staged, ready to be re-committed differently. --mixed does that and also unstages it, dumping it back into your working directory as plain unstaged edits — nothing is lost, it's just no longer queued for commit. --hard does both of those and then overwrites the actual files on disk to match — whatever your edits were, gone from view the instant the command returns.

Key Idea

A useful shortcut for remembering the order: soft moves the least, hard moves the most, and each step is a superset of the one before it. If you're ever unsure which flag you need, reach for the gentler one first — you can always run git reset --hard a second time to go further, but you can't un---hard your working directory back from nothing.

Watching it happen makes the three stopping points click faster than any description. Same starting point every time — two commits, c1 then c2 — three different endings: 

Terminal — git reset --soft HEAD~1$ git log --oneline
ca1b9f4 c2
cdb6f12 c1
$ git reset --soft HEAD~1
$ git log --oneline
cdb6f12 c1
$ git status
On branch main
Changes to be committed:
  (use "git restore --staged <file>..." to unstage)
	modified:   f.txt

HEAD moved back to c1, but c2's content didn't vanish — it's sitting in the index, staged, exactly as if you'd just run git add a moment ago. Reset that same starting point with --mixed (the default — no flag needed) instead, and the difference from --soft shows up immediately in git status: 

Terminal — git reset HEAD~1 (--mixed, the default)$ git reset HEAD~1
Unstaged changes after reset:
M	f.txt
$ git status
On branch main
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
	modified:   f.txt

"Changes to be committed" became "Changes not staged" — the index snapped back to match the new HEAD, same as Chapter 8's git restore --staged, except reset also dragged HEAD itself backward. The file on disk still has c2's content either way; only the index's opinion of it changed. Now the one that earns its warning color: 

Terminal — git reset --hard HEAD~1$ git reset --hard HEAD~1
HEAD is now at cdb6f12 c1
$ git status
On branch main
nothing to commit, working tree clean
$ cat f.txt
line1

"Nothing to commit, working tree clean" — but that cleanliness is the warning sign, not the reassurance. c2's change to f.txt is gone from the file entirely, with zero confirmation prompt. The commit object for c2 itself still exists in objects/ (nothing that was ever committed is deleted this easily), and Chapter 9's reflog still remembers HEAD's trip through it — but any edit that was only ever on disk, never staged or committed, had no reflog entry to begin with, and --hard just overwrote it with no way back.

Warning

git reset --hard is the most dangerous common command in this book. It overwrites your working directory immediately, with no confirmation, no "are you sure?" — and unlike almost every other mistake in Git, an uncommitted edit it destroys was never an object, so there's no blob for the reflog or git fsck to hand back to you. Two habits keep this command safe: run git status first and actually read it (anything under "Changes not staged" is about to die), and if you only need to look at an old commit without disturbing your branch, reach for git checkout <commit> or git stash (Chapter 11) instead of reset. If you do run it by accident on a commit that was already committed, Chapter 9's reflog is exactly the safety net you built there — git reflog, find the hash, git reset --hard <hash> right back to where you were.

git revert: Undo Without Rewriting History

Every reset mode does the same fundamental thing: it moves your branch's pointer to point at an older commit. As far as git log is concerned, the commits after that point stop being part of the branch's history at all. That's exactly the behavior you want on a branch only you have ever seen. It's exactly the behavior you don't want on a branch you've already pushed and a teammate has already pulled — because their clone still has those commits, and the next time histories meet, Git sees two branches that disagree about the past.

git revert solves the same problem — "undo what this commit did" — without ever moving HEAD backward. Instead, it looks at the target commit's changes, computes the exact opposite diff, and commits that as a brand-new commit on top of your history. Nothing is deleted, nothing is rewritten — the mistake and its correction both stay on the record, forever, like a ledger entry marked "reversed" instead of torn out of the book.

Mechanically, git revert <commit> figures out what that commit changed, applies the reverse of that change to your working directory and index, and opens a commit message for you — pre-filled as Revert "<original subject>" with a body noting This reverts commit <hash>. so the link back to the original is permanent and searchable. Pass --no-edit to accept that message as-is instead of opening an editor: 

Terminal — reverting a commit already on main$ git log --oneline
525a5c6 Equip cursed amulet (oops, shared with team)
c237a45 Add potion slot
70b6d77 Start the story
$ git revert --no-edit HEAD
[main cddb29d] Revert "Equip cursed amulet (oops, shared with team)"
 1 file changed, 1 deletion(-)
$ git log --oneline
cddb29d Revert "Equip cursed amulet (oops, shared with team)"
525a5c6 Equip cursed amulet (oops, shared with team)
c237a45 Add potion slot
70b6d77 Start the story

Four commits now, not three — the "mistake" commit is still right there in the log, permanently, with a new commit directly above it canceling its effect out. Anyone who already pulled 525a5c6 loses nothing when they pull again: they just receive one new commit, same as any other change, no history conflict possible.

Pro Tip

The rule of thumb every team eventually adopts: if it's only ever been on your machine, reset is fine; the moment it's been pushed anywhere someone else might have pulled from, revert instead. Reset a commit that a teammate already has, and the next git push gets rejected as a non-fast-forward — the only way past that is git push --force, which can silently drop your teammate's work if they'd built anything on top of the commit you erased. Revert never has this problem, because it never removes anything the remote already saw.

Scenario: You Committed to the Wrong Branch

Here's the situation that makes reset --soft genuinely useful day to day, not just a trivia answer. You meant to be on a feature branch. You weren't — maybe you forgot to git switch after your last merge, maybe a terminal tab lied to you about where you were. Either way, you just committed real work directly onto main: 

Terminal — the mistake, made in good faith$ git branch
* main
$ git add player.txt
$ git commit -q -m "Add inventory system"
$ git log --oneline
1bdc88c Add inventory system
f0b337f Start the story

Whatever you do next, don't reach for --hard — that would throw the inventory-system work away entirely, and it's good work, it's just committed to the wrong pointer. What you actually want is to un-commit it while keeping every byte exactly as it is, so you can hand it to the right branch instead. That's precisely what --soft is for: 

Terminal — back it out with reset --soft$ git reset --soft HEAD~1
$ git log --oneline
f0b337f Start the story
$ git status
On branch main
Changes to be committed:
  (use "git restore --staged <file>..." to unstage)
	modified:   player.txt

main is back to a clean state — the commit never happened, as far as history is concerned — but the change itself is still fully staged, sitting in the index exactly as it was a second before you typed commit. Now switch to where it actually belongs. A staged change travels with you across git switch, so nothing needs re-adding: 

Terminal — switch branches, staged change comes along$ git switch -c feature-inventory
Switched to a new branch 'feature-inventory'
$ git status
On branch feature-inventory
Changes to be committed:
  (use "git restore --staged <file>..." to unstage)
	modified:   player.txt
$ git commit -q -m "Add inventory system"
$ git log --oneline
3696ff3 Add inventory system
f0b337f Start the story
$ git switch main
$ git log --oneline
f0b337f Start the story

Same commit message, same file content, brand-new hash — because it's a genuinely new commit object, just built on the branch you meant all along. main never saw it. This is the move worth memorizing: reset --soft, switch, commit again. Three commands, zero lost work, and it fixes the single most common "wait, what branch am I on?!" moment you'll hit in real projects.

Try It Yourself

Live the wrong-branch rescue yourself, start to finish. Fresh throwaway repo: 

Terminal — set the scene$ mkdir reset-quest && cd reset-quest && git init -q -b main
$ printf "Welcome to Staging Quest!\nHP: 100\n" > player.txt
$ git add player.txt && git commit -q -m "Start the story"

Now make the mistake on purpose — stay on main and commit feature work that should have gone on its own branch: 

Terminal — commit to the wrong branch$ printf "Welcome to Staging Quest!\nHP: 100\nInventory: sword, shield\n" > player.txt
$ git add player.txt && git commit -q -m "Add inventory system"
$ git branch -v
* main 1bdc88c Add inventory system

Caught it. Back the commit out with --soft so the work stays staged instead of vanishing: 

Terminal — reset --soft, then check the index$ git reset --soft HEAD~1
$ git log --oneline
f0b337f Start the story
$ git diff --staged --stat
 player.txt | 1 +
 1 file changed, 1 insertion(+)

main is clean, and the change is still right there, staged. Move it to a proper branch and finish the commit: 

Terminal — the rescue, completed$ git switch -c feature-inventory
Switched to a new branch 'feature-inventory'
$ git commit -q -m "Add inventory system"
$ git log --oneline
3696ff3 Add inventory system
f0b337f Start the story
$ git switch -q main && git log --oneline
f0b337f Start the story

main never carried the extra commit; the work now lives cleanly on feature-inventory, right where it belonged from the start. That's the entire fix — no force push, no drama, no lost inventory system.

Try It Yourself

Now the other side — undoing a commit that's already shared, where reset is off the table. Same repo, keep going, or start fresh with a couple more commits: 

Terminal — build a small history, including a bad idea$ printf "Welcome to Staging Quest!\nHP: 100\nPotion: none\n" > player.txt
$ git add player.txt && git commit -q -m "Add potion slot"
$ printf "Welcome to Staging Quest!\nHP: 100\nPotion: none\nCursed amulet: equipped\n" > player.txt
$ git add player.txt && git commit -q -m "Equip cursed amulet (oops, shared with team)"
$ git log --oneline
525a5c6 Equip cursed amulet (oops, shared with team)
c237a45 Add potion slot
f0b337f Start the story

Pretend that last commit already went out in a git push — a teammate may already have it. Resetting it away would set up a rejected push later; revert it instead: 

Terminal — revert, not reset$ git revert --no-edit HEAD
[main cddb29d] Revert "Equip cursed amulet (oops, shared with team)"
 1 file changed, 1 deletion(-)
$ cat player.txt
Welcome to Staging Quest!
HP: 100
Potion: none
$ git log --oneline
cddb29d Revert "Equip cursed amulet (oops, shared with team)"
525a5c6 Equip cursed amulet (oops, shared with team)
c237a45 Add potion slot
f0b337f Start the story

The cursed amulet is gone from player.txt again — same practical result as a hard reset — but 525a5c6 is still sitting in the log, untouched, with a new commit on top explaining exactly what got undone and why. A teammate pulling right now gets one new commit and no conflict, no force push, no confusion about where their local copy diverged.

Zoom out and reset and revert are two tools built for the same job — "make Git forget this happened" — aimed at two different audiences. Reset rewrites your branch's story, which is perfectly safe as long as you're the only reader of that story so far; it's the right call for the wrong-branch fumble, the "let me try that commit message again" cleanup, the local experiment that didn't pan out. Revert writes a correction into the public record instead of erasing a page, which is the only safe move once other people have already turned to that page. Keep Chapter 9's reflog running underneath either one as your net, and there's genuinely no "undo" mistake left in Git that can't be walked back. Next chapter is about the opposite problem — not undoing work, but pausing it mid-flight, with git stash.
