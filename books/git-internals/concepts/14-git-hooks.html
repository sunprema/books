Git Hooks: Automating with Scripts

Chapter 13 ended on a warning: nothing stops a teammate — or you, at 2am — from pushing something malformed, or forgetting to run the test suite before committing. Git can't read your mind, but it can run your code at exactly the right moment to catch that stuff automatically. Those moments are called hooks, and you've technically already met their home: Chapter 2's floor plan of .git flagged a folder named hooks/ and promised we'd come back for it. We're back.

A Hook Is Just a Script With the Right Filename

Strip away the mystique and a Git hook is almost insultingly simple: it's an executable script, dropped in .git/hooks/, named exactly after the event that should trigger it — no config file, no registration step, no special syntax. Git checks that folder before and after nearly every major operation, and if a file matching the event's name exists and is executable, Git runs it. That's the entire mechanism. 

Terminal — a fresh repo already ships sample hooks, just disabled$ git init -q quest && cd quest
$ ls .git/hooks | head -5
applypatch-msg.sample
commit-msg.sample
fsmonitor-watchman.sample
post-update.sample
pre-applypatch.sample

Every one of those ends in .sample — Git ships example scripts for nearly every hook it supports, but ignores them until you strip the .sample suffix and make the file executable. Miss the chmod +x step and your hook silently does nothing, which is, no exaggeration, the single most common "why isn't my hook running?!" bug beginners hit.

HookRuns whenCan it block the operation?
pre-commitBefore the commit message editor even opens Yes — nonzero exit aborts the commit
prepare-commit-msgRight after pre-commit, before the editor opens; can pre-fill the message Rarely used to block, but nonzero exit does abort
commit-msgAfter you've written a message, before the commit is finalized Yes — nonzero exit aborts the commit
post-commitRight after the commit object exists No — informational only
post-mergeAfter a merge completes successfully No — informational only
pre-pushBefore objects upload to a remote Yes — nonzero exit aborts the push

That's the short list of client-side hooks — the ones that run on your machine, the only kind this chapter covers. There's a parallel set of server-side hooks (pre-receive, update, post-receive) that fire on whatever machine hosts origin when it receives your push — that's how a company can reject a force-push to main before it ever lands, a thread we'll pick back up in Chapter 16.

Key Idea

Every blocking hook follows the same tiny contract: exit 0 and Git proceeds; exit anything else and Git stops, printing whatever your script wrote to stderr. You don't call any special Git API to "reject" a commit — you just exit 1 like any other broken shell script, and Git treats that as a veto. It's the same convention as every Unix command line tool you've ever run.

Writing a commit-msg Hook: Enforcing a Message Format

commit-msg is the hook to reach for when you care about what the commit says, not what it changes. Git calls it with exactly one argument — $1, the path to a temporary file already holding the message you just wrote — after the editor closes (or immediately, if you used -m), but before the commit is finalized. You can read that file to validate the message, or even rewrite it in place; exit nonzero and Git throws the whole commit back at you, message and all, so nothing is lost.

Here's a hook that enforces a Conventional Commits style prefix — feat:, fix:, docs:, and a handful of others — the same convention a lot of real teams use to auto-generate changelogs: 

.git/hooks/commit-msg#!/bin/sh
# .git/hooks/commit-msg — reject commits that don't start with a
# conventional-commit-style prefix (feat:, fix:, docs:, chore:, ...)

msg_file="$1"
first_line=$(head -n1 "$msg_file")

if ! echo "$first_line" | grep -qE '^(feat|fix|docs|style|refactor|test|chore)(\(.+\))?: .+'; then
  echo "commit-msg: rejected — message must start with a type prefix," >&2
  echo "  e.g. 'feat: add inventory system' or 'fix: correct HP bug'." >&2
  echo "  Got: \"$first_line\"" >&2
  exit 1
fi

exit 0

Nothing exotic — a regular POSIX shell script, reading its one argument, running grep -qE against a pattern, and exiting 0 or 1. That's genuinely the whole API surface a hook has to learn.

Try It Yourself

Save that script as .git/hooks/commit-msg in any throwaway repo (or reuse one from an earlier chapter's lab), make it executable, and try to break the rule on purpose before following it: 

Terminal — chmod, then a bad message on purpose$ chmod +x .git/hooks/commit-msg
$ printf "hello quest\n" > player.txt && git add player.txt
$ git commit -m "started the game"
commit-msg: rejected — message must start with a type prefix,
  e.g. 'feat: add inventory system' or 'fix: correct HP bug'.
  Got: "started the game"
$ echo $?
1

Git left the staged file exactly as it was — nothing lost, commit just refused to happen. Now give it a message the hook actually accepts: 

Terminal — a conventional-commit-style message gets through$ git commit -m "feat: start the story"
[main (root-commit) 8f95cdc] feat: start the story
 1 file changed, 1 insertion(+)
 create mode 100644 player.txt

Same staged content both times — the only thing that changed was whether the first line of the message matched the pattern. That's the entire job of a commit-msg hook: gatekeeping words, not code.

Writing a pre-commit Hook: Running Tests Before Every Commit

pre-commit gets no arguments at all — it just runs, with your staged changes sitting in the index, and Git waits to see whether it exits 0. That makes it the natural home for anything that inspects code rather than the commit message: linters, formatters, secret-scanners, or — the classic example — your test suite. Nobody enjoys a "fix broken build" commit three minutes after the one that broke it; a pre-commit hook can make that commit simply never happen.

Here's a deliberately tiny stand-in for a real test suite — it just checks that a game character's HP never went negative — plus the hook that wires it in: 

run-tests.sh — a trivial "test suite"#!/bin/sh
# run-tests.sh — stand-in for a real test suite. Fails if hp.txt
# contains a negative HP value, succeeds otherwise.
if grep -q '^HP: -' hp.txt 2>/dev/null; then
  echo "FAIL: HP went negative — that's a bug." >&2
  exit 1
fi
echo "PASS: HP looks sane."
exit 0
.git/hooks/pre-commit#!/bin/sh
# .git/hooks/pre-commit — block the commit if the test suite fails.
echo "Running tests before commit..."
./run-tests.sh
status=$?
if [ $status -ne 0 ]; then
  echo "pre-commit: tests failed, aborting commit." >&2
  exit 1
fi
exit 0
Try It Yourself · The Challenge

This is the chapter's real challenge: wire up both files above, chmod +x the hook (and the test script, since the hook calls it directly), then try to commit a bug on purpose: 

Terminal — a failing "test" blocks the commit$ chmod +x .git/hooks/pre-commit run-tests.sh
$ echo "HP: -10" > hp.txt && git add hp.txt
$ git commit -m "fix: update hp"
Running tests before commit...
FAIL: HP went negative — that's a bug.
pre-commit: tests failed, aborting commit.
$ echo $?
1

Fix the actual bug — not the hook — and the same command sails through: 

Terminal — fix the bug, same command now succeeds$ echo "HP: 90" > hp.txt && git add hp.txt
$ git commit -m "fix: update hp"
Running tests before commit...
PASS: HP looks sane.
[main 0c38295] fix: update hp
 1 file changed, 1 insertion(+)

Swap run-tests.sh for whatever your actual project uses — npm test, pytest, go test ./... — and you've got a real, working "no broken commits" tripwire, running in under a second before anything ships.

Warning

.git/hooks/ lives inside .git — and everything in .git is, by definition, the one thing git itself never version-controls. Commit your beautiful pre-commit hook, push it, and your teammate's git clone gets none of it — hooks stay purely local, forever, until someone writes them by hand. That's exactly why real teams don't rely on raw .git/hooks/ files: tools like Husky (JavaScript ecosystem), the pre-commit framework (Python, but language-agnostic in practice), and Lefthook store hook config in a tracked file inside the repo and install the real scripts into .git/hooks/ automatically after every clone. Same underlying mechanism this chapter just showed you — they're just solving the "how do I share this with my team" problem on top of it.

Pro Tip

Genuinely stuck and need to get a commit through right now, hook be damned? git commit --no-verify (short: -n) skips pre-commit and commit-msg entirely, and git push --no-verify does the same for pre-push. It's a legitimate escape hatch for a genuine emergency — just don't make it a habit, or you've quietly rebuilt the world where nothing ever got checked in the first place.

Zoom out and hooks are really just Git handing you the microphone at specific moments: about to commit, about to finalize this message, just merged, about to push. Two of those moments — pre-commit and commit-msg — happen early enough to say no; the rest only get to react. None of it requires learning a plugin API: it's executable files, an exit code, and a filename Git already knows to look for.
