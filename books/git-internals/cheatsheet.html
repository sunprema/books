Seventeen chapters, boiled down. This page doesn't teach — it reminds. Keep it open in a tab, or print it, and use the "gotchas" list before you run anything with --hard or --force in it.

The 30-Second Summary

Everything in .git is one idea wearing different hats: content-addressed objects. A blob is file content, a tree is a folder listing of blobs and trees, a commit is a snapshot (a tree plus parent pointers, author, and message) — and every one of them is named by the SHA-1 hash of its own bytes, so the ID is a tamper-evident fingerprint, not an arbitrary label. Branches, HEAD, and tags are just sticky notes — 41-byte text files holding one hash each — not copies of your code, which is why creating and switching branches is instant. Between a commit and your files sit three trees: the committed HEAD snapshot, the staging area / index (the snapshot you're building), and your working directory (what you're actually editing) — add/restore move changes between them, and commit only ever reads the index. And when something goes wrong, the reflog is your safety net: Git rarely deletes an object immediately, so almost anything "lost" to a bad reset --hard or a rebase gone sideways is still sitting in .git/objects, one git reflog away from being found again — usually for about 90 days, until git gc finally sweeps it.

The Three Object Types

blobRaw file content only — no filename, no permissions
treeA folder listing: mode + type + hash + name, per entry
commitOne tree hash + parent hash(es) + author + message
tagAn annotated tag's own object: message + tagger + signature

Hash = SHA-1 over "<type> <size>\0<content>". Same content, same hash, every time, anywhere — that's content addressing.

⚠ Gotchas — Read Before You Run These

  • ⚠git reset --hardThrows away uncommitted working-dir changes with no prompt. The reflog can rescue a lost commit, but never uncommitted edits — they were never an object.
  • ⚠git push --forceOverwrites the remote branch outright, silently erasing anyone else's commits on it. Prefer --force-with-lease, which refuses if the remote moved since you last fetched.
  • ⚠Rebasing shared historyRebase rewrites commit hashes. Rebasing a branch others have already pulled forces everyone else into a painful, confusing history reconciliation. Rule of thumb: only rebase branches that are still yours alone.
  • ⚠git gc --prune=now / aggressive pruneImmediately sweeps unreachable objects that would otherwise sit as a ~90-day safety margin. Run only once you're sure the reflog has nothing you still need.
  • ⚠Deleting .gitDeletes all history, objects, and the reflog in one shot — your working files survive, but every commit, branch, and stash does not. There's no undo once it's gone.
  • ⚠git stash drop / clearing the stashA dropped stash is just an unreachable commit — recoverable for a while via git fsck --unreachable, but not through any normal stash command.
