Chapter 03 · The Generic Server

GenServer

The loop from Chapter 01, written once, correctly, forever.

Remember the hand-rolled Counter loop — spawn, receive, recurse with new state? Every stateful process wants that skeleton, plus a dozen fiddly details: replying to the right caller, timeouts, clean shutdown, code upgrades, debugging hooks. Writing that by hand every time is how bugs are born. GenServer is that skeleton, written once by people who ran telecoms on it, leaving you to fill in only the interesting clauses.

The deal is a clean split. You write two kinds of code: a client API (plain functions your callers use) and a set of callbacks (how the server reacts). GenServer owns the loop, the mailbox, and the message plumbing between them.

The one idea

A GenServer is a process whose loop is provided for you. You implement callbacks — init/1 sets the starting state, handle_call/3 answers synchronous requests, handle_cast/2 takes fire-and-forget ones, handle_info/2 catches everything else — and each returns a tuple telling GenServer the new state and what to do next.

call vs. cast, on the wire

Implementation: a Stack, end to end

Let's build the canonical example. Read it top to bottom: the client API is what the outside world calls; the callbacks are how the server responds. The two halves talk only through GenServer.call and cast.

lib/stack.exdefmodule Stack do
  use GenServer

  # ---------- Client API (runs in the CALLER's process) ----------
  def start_link(initial \\ []) do
    GenServer.start_link(__MODULE__, initial, name: __MODULE__)
  end

  def push(value), do: GenServer.cast(__MODULE__, {:push, value})
  def pop,         do: GenServer.call(__MODULE__, :pop)

  # ---------- Server callbacks (run in the SERVER's process) ----------
  @impl true
  def init(initial), do: {:ok, initial}   # state starts as `initial`

  @impl true
  def handle_call(:pop, _from, [top | rest]) do
    {:reply, top, rest}                    # reply `top`, new state `rest`
  end
  def handle_call(:pop, _from, []) do
    {:reply, {:error, :empty}, []}
  end

  @impl true
  def handle_cast({:push, value}, state) do
    {:noreply, [value | state]}            # no reply, just new state
  end
end
iex — using it{:ok, _pid} = Stack.start_link(["a", "b"])
Stack.push("c")   #=> :ok        (async, returns instantly)
Stack.pop()        #=> "c"        (blocks for the reply)
Stack.pop()        #=> "a"
Read the return tuples

The whole callback contract is in the tuples. {:reply, value, new_state} answers a call. {:noreply, new_state} updates state silently. {:stop, reason, new_state} ends the server. There's even {:noreply, state, {:continue, arg}} to hand off heavy work to handle_continue/2 right after — handy for doing slow setup after init/1 returns so startup stays snappy.

handle_info and handle_continue

Two more callbacks earn their keep. handle_info/2 catches messages that didn't come through call/cast — raw send/2 messages, monitor :DOWN notices, and timer ticks from Process.send_after/3. It's how a GenServer runs periodic work. handle_continue/2 lets init/1 return fast and then finish expensive initialization as the very next thing the server does — so a slow dependency doesn't block your whole supervision tree from booting.

a self-ticking server  @impl true
  def init(state) do
    Process.send_after(self(), :tick, 1_000)
    {:ok, state}
  end

  @impl true
  def handle_info(:tick, state) do
    IO.puts("tick")
    Process.send_after(self(), :tick, 1_000)  # schedule the next one
    {:noreply, state}
  end
The server is single-threaded — respect it

A GenServer handles one message at a time. That's the gift (no locks) and the trap: a slow handle_call blocks every other caller behind it. Keep callbacks fast; push heavy or slow work to a Task (Chapter 07) so the server stays responsive. And prefer call over cast when senders could outpace the server — the caller waiting is your back-pressure.

So what did we actually gain? We stopped writing loops and started writing behaviour. The use GenServer line quietly gave us a correct receive loop, a child_spec/1 so a supervisor can start us (Chapter 04), timeout handling, and debug tooling. All we supplied were the clauses that make this server different from every other. That's the OTP bargain in miniature.

Challenges

  1. Add a peek/0 that returns the top without removing it. Which callback — call or cast — and why?
  2. Make push/1 a call instead of a cast. When would that actually be the better choice? (Think about a fast producer and a slow disk.)
  3. Add handle_call(:pop, _from, []) that instead returns {:stop, :empty, {:error, :empty}, []}. Watch the server terminate. Now put it under a supervisor and watch it come back.
