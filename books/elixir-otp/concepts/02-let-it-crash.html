Chapter 02 · Failure as a Feature

Let It Crash

The bravest error-handling strategy is to not handle the error.

Coming from most languages, "let it crash" sounds reckless — even negligent. But it's the single most important cultural idea in OTP, and once it clicks, half your defensive try/rescue code evaporates. The claim is this: a process that hits an unexpected state should die immediately and cleanly, and something else — a supervisor — should start a fresh one. We don't nurse a sick process back to health; we replace it.

This only works because processes are isolated (Chapter 01). One process crashing can't corrupt another's memory, because they share none. So a crash stays local — a blast radius of exactly one process — unless we deliberately wire processes together. Wiring them together is what this chapter is about. There are exactly two wires: links and monitors.

The one idea

A link is a two-way tie: if either linked process dies abnormally, the other is sent an exit signal and, by default, dies too. A monitor is a one-way watch: the watcher gets a friendly :DOWN message in its mailbox when the target dies, and carries on. Supervisors are built on links; request/reply code and Task.await are built on monitors.

The two wires, side by side

Links: shared fate

spawn_link/1 spawns a process already linked to you. If it crashes, the exit signal travels the link and takes you down too — which sounds bad until you realize that's precisely what you want inside a supervision tree: a broken subtree should collapse so the supervisor can rebuild it whole.

iex — a link propagates the crashparent = self()
spawn_link(fn -> raise "boom" end)
# ** the linked child raises, the signal propagates, and *we* exit too.

Trapping exits: turning a signal into a message

Sometimes you don't want to die with your child — you want to hear about it and react. Set trap_exit and the lethal exit signal is converted into an ordinary {:EXIT, pid, reason} message in your mailbox. This is the mechanism a supervisor uses internally to notice a child died and restart it.

trapping exitsProcess.flag(:trap_exit, true)
spawn_link(fn -> exit(:kaboom) end)

receive do
  {:EXIT, from, reason} ->
    IO.puts("child #{inspect(from)} died: #{inspect(reason)}")
end
:normal doesn't kill

A process exiting with reason :normal (it simply finished) does not take its links down — only abnormal exits propagate. So linking is safe for the common case; it only bites when something genuinely went wrong, which is exactly when you want it to.

Monitors: watch without dying

A monitor is the tool when you want information, not shared fate. Process.monitor/1 returns a unique reference; when the target dies you get a :DOWN message tagged with that reference. It's one-directional and it never kills the watcher.

iex — monitoringref = Process.monitor(pid)

receive do
  {:DOWN, ^ref, :process, ^pid, reason} ->
    IO.puts("it's gone: #{inspect(reason)}")
end

The ^ref pin means "match only the message carrying this reference" — so even if you monitor several processes, each :DOWN is unambiguous. Under the hood, GenServer.call/3 monitors the server it's calling, which is how it can raise a clean error the instant the server dies instead of hanging until timeout.

So what did we actually gain? A vocabulary for failure relationships. We no longer sprinkle try/rescue hoping to catch every possible fault inside a process. Instead we let a faulty process die, and we decide, per relationship, who should share its fate (link) and who should merely be told (monitor). Everything in the next chapters — supervisors, DynamicSupervisor, Task.await — is just links and monitors with good packaging.

Challenges

  1. Spawn a linked child that exits :normal. Confirm you survive. Now make it raise. Confirm you don't (without trapping).
  2. Write a process that monitors three others and prints which one died first.
  3. Why does a supervisor set trap_exit? Answer it in one sentence using this chapter's vocabulary.
