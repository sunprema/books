Chapter 01 · The Actor Model

Processes

Everything in OTP is a process. Let's earn that sentence.

Here's the whole trick of Elixir concurrency in one line: instead of many threads sharing one blob of memory and fighting over locks, you get millions of tiny, cheap, completely isolated processes that never share a single byte — they only send each other messages. Get comfortable with that one idea and OTP stops looking like magic and starts looking inevitable.

These are not operating-system processes. A BEAM process (the BEAM is the Erlang virtual machine Elixir runs on) starts at a few hundred bytes and spawns in microseconds. A busy system runs hundreds of thousands of them at once, each with its own tiny heap and its own mailbox. The scheduler slices them across your CPU cores for you. You will make a lot of these.

The one idea

A process has private state and a mailbox. It cannot reach into another process's memory — the only way to affect it is to send it a message, which lands in its mailbox. No shared memory means no data races, no mutexes, no torn reads. This is the actor model, and it's the ground floor everything else in this book stands on.

Watch it happen

Below is a tiny BEAM: five isolated processes, each drawn with its own mailbox slots underneath it. Watch messages fly between them and queue up on arrival — a call is a message whose sender waits for a reply; a cast is fire-and-forget. Nobody touches anyone else's state. They just mail letters.

Design: three verbs

The entire primitive layer is three functions. spawn starts a process running a function and hands you back its PID (process identifier). send drops a message into a PID's mailbox. receive pulls the next matching message out of your own mailbox, blocking until one arrives.

iex — the three primitives# spawn returns a PID immediately; the function runs concurrently.
pid = spawn(fn ->
  receive do
    {:hello, sender} -> send(sender, :hi_back)
    other                -> IO.puts("got: #{inspect(other)}")
  end
end)

send(pid, {:hello, self()})   # self() is *our* PID, so it can reply to us

receive do
  :hi_back -> IO.puts("it replied!")
after
  1_000 -> IO.puts("gave up waiting")  # receive can time out
end

Notice the shape already: we send our own self() along inside the message so the other process knows where to reply. That "who do I answer?" bookkeeping is so common that GenServer (Chapter 03) will do it for us. But it's just this, underneath.

Why a mailbox, and not a shared variable?

Because a mailbox serializes access for free. A process handles one message at a time, top to bottom, so its state is only ever touched by one line of code at one instant. You get the safety of a lock without ever writing a lock. The cost is that talking to another process means a message hop — but on the BEAM those hops are astonishingly cheap, and buying correctness with them is one of the great trades in systems programming.

Selective receive

receive scans the mailbox for the first message that matches a pattern, leaving the rest in place. That means you can wait for one specific reply while other messages pile up harmlessly behind it — a small superpower we'll lean on constantly.

Implementation: a process that remembers

A process keeps state by looping and passing the new state to itself on each turn. Here's a counter with no library at all — just recursion and a mailbox:

a hand-rolled stateful processdefmodule Counter do
  def start(n), do: spawn(fn -> loop(n) end)

  defp loop(n) do
    receive do
      {:inc, by}      -> loop(n + by)          # new state = the next loop
      {:get, from}    -> send(from, n); loop(n)
      :stop            -> :ok                    # stop looping = process ends
    end
  end
end

Look at what state is here: it's just the argument n to the next call of loop/1. There's no mutable variable anywhere. The "memory" of the process is a tail-recursive loop that keeps handing the updated value forward. Every stateful server you write for the rest of this book is a dressed-up version of exactly this loop — and in Chapter 03 we'll let GenServer write the loop so we only fill in the interesting parts.

So what did we actually gain? A unit of concurrency so cheap we stop rationing it. Instead of asking "can I afford another thread?", we model each connection, each job, each user session as its own process — isolated, crash-contained, and scheduled for us. The rest of OTP is just good manners layered on top of this.
Mind the mailbox

The one thing a process can't do is keep up forever with more messages than it can handle — an unbounded mailbox is a memory leak with a slow fuse. When we reach GenServer we'll talk about back-pressure: preferring call (which makes the sender wait) over cast when the sender is outrunning the server.

Challenges

  1. Extend Counter with a {:dec, by} message. What happens to messages that match no clause? (Hint: they stay in the mailbox — forever.)
  2. Spawn 100 000 counters in a loop and time it. Notice you can. Try that with OS threads.
  3. Add an after 5_000 -> loop(n) clause to loop/1 so an idle counter prints a heartbeat. You've just built a timeout — a feature GenServer gives you as an option.
