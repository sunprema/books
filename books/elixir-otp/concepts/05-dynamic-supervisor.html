Chapter 05 · Children on Demand

DynamicSupervisor

When you don't know your children until the system is running.

The supervisor in Chapter 04 knows its children up front — you list them in init/1 and they start in order at boot. But so much of real work is unpredictable: one process per connected user, per game room, per uploaded file, per background job. You can't list those; they don't exist until someone shows up. That's exactly the gap DynamicSupervisor fills.

The one idea

A DynamicSupervisor starts with no children and grows them at runtime with start_child/2. It still gives you supervision — crash isolation, restarts, shutdown — but for a fluctuating population of identical-shaped workers. Its only strategy is :one_for_one, because dynamic children are independent by definition.

Set it up, then feed it

You start one DynamicSupervisor (usually in your app's static tree) and from then on just ask it to start children whenever you need one.

in your application's supervision treechildren = [
  {DynamicSupervisor, name: MyApp.WorkerSup, strategy: :one_for_one}
]
Supervisor.start_link(children, strategy: :one_for_one)
start a worker whenever a user connectsdef connect(user_id) do
  spec = {MyApp.Session, user_id}
  DynamicSupervisor.start_child(MyApp.WorkerSup, spec)
  #=> {:ok, #PID<0.312.0>}
end

Each started child is a fully supervised process. If a Session crashes, the DynamicSupervisor restarts just that one — the other thousands of sessions never feel it. When a user disconnects, you terminate that child and it's gone. The population breathes with your load.

Prefer :temporary for dynamic children

For per-connection or per-job workers you often want restart: :temporary in the child's spec — if a session dies, it usually shouldn't silently respawn without the user reconnecting. Reach for :permanent only when the worker truly must always exist. This is a spec on the child, not the supervisor.

A module-based DynamicSupervisor

As with everything in OTP, you can wrap it in a module to give it a home and a clean API:

lib/my_app/worker_sup.exdefmodule MyApp.WorkerSup do
  use DynamicSupervisor

  def start_link(init_arg) do
    DynamicSupervisor.start_link(__MODULE__, init_arg, name: __MODULE__)
  end

  def start_session(user_id) do
    DynamicSupervisor.start_child(__MODULE__, {MyApp.Session, user_id})
  end

  @impl true
  def init(_init_arg) do
    DynamicSupervisor.init(strategy: :one_for_one, max_children: 10_000)
  end
end
Cap the population

An unbounded dynamic supervisor is an unbounded resource commitment. Set max_children so a flood of connections can't exhaust the node — beyond the cap, start_child/2 returns {:error, :max_children} and you can shed load gracefully instead of falling over. For very high throughput, shard across several dynamic supervisors with PartitionSupervisor so one process isn't the sole gatekeeper.

So what did we actually gain? Supervision that scales with reality. Static supervisors model the fixed spine of your system; DynamicSupervisor models the parts that ebb and flow. Together they let a single node run millions of independently-supervised little machines — which is exactly how Elixir systems hold up so many concurrent users on so little hardware.

There's one loose end. We can start a session for user 42, but if a message for user 42 arrives later, how do we find that specific PID among thousands? We need a phone book. That's the next chapter.

Challenges

  1. Start 5 sessions, then DynamicSupervisor.count_children/1. Terminate one with terminate_child/2 and count again.
  2. Give Session a restart: :temporary spec and make it crash. Confirm it stays dead. Switch to :permanent and watch it respawn.
  3. Set max_children: 3 and try to start a fourth. Handle the {:error, :max_children} return like a real load-shedding path would.
