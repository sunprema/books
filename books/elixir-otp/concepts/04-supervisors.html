Chapter 04 · Supervision Trees

Supervisors

The "something else" that restarts the process you let crash.

Chapter 02 promised that when a process dies, something restarts it. This is that something. A supervisor is a process whose only job is to start a set of child processes, link to them, and — when one dies — restart it according to a strategy you choose. Supervisors supervise workers, and supervisors can supervise other supervisors, so your whole application becomes a tree: durable branches near the root, disposable leaves that come and go. That tree is your system's fault-tolerance model, made explicit.

The one idea

A supervisor holds a list of child specs and a restart strategy. When a child crashes, the strategy decides who gets restarted. Because restarting means starting a fresh process with fresh state, "turn it off and on again" becomes a first-class, automatic recovery mechanism — not an embarrassment.

See it heal

Here's a live supervision tree: one root supervisor, two sub-supervisors, six workers. Every couple of seconds a worker crashes (watch it flash and vanish) and the supervisor immediately starts a fresh one in its place. Drag to orbit the tree; scroll to zoom. Nothing you do stops the healing — that's the point.

Restart strategies

When a child dies, which children get restarted? That's the strategy. There are three, and choosing correctly is mostly about how tightly your children depend on each other.

Implementation: a module-based supervisor

You almost always write a supervisor as its own module using the Supervisor behaviour. The children are listed as child specs — and the friendly shorthand {Module, arg} expands, via the child_spec/1 that use GenServer/use Supervisor generated for you, into the full map.

lib/my_app/supervisor.exdefmodule MyApp.Supervisor do
  use Supervisor

  def start_link(init_arg) do
    Supervisor.start_link(__MODULE__, init_arg, name: __MODULE__)
  end

  @impl true
  def init(_init_arg) do
    children = [
      {Stack, ["a", "b"]},          # shorthand → Stack.child_spec(["a","b"])
      {MyApp.Cache, :name},         # another worker
      MyApp.Metrics                   # bare module → Metrics.child_spec([])
    ]

    Supervisor.init(children, strategy: :one_for_one)
  end
end

The child spec, spelled out

The shorthand is nice, but it helps to know what it expands to. A child spec is just a map:

the full child spec map%{
  id:       Stack,                        # unique within this supervisor
  start:    {Stack, :start_link, [["a", "b"]]}, # {module, fun, args}
  restart:  :permanent,                  # :permanent | :transient | :temporary
  shutdown: 5_000,                       # ms to wait, or :brutal_kill / :infinity
  type:     :worker                      # :worker | :supervisor
}
restartmeaning
:permanentalways restart (the default) — for long-lived services
:transientrestart only if it died abnormally; a :normal exit is left alone
:temporarynever restart — for disposable, best-effort work
The restart limit — a circuit breaker

What if a child crashes because the world is broken (a database is down)? Restarting it in a tight loop helps nobody. So a supervisor gives up if it sees more than max_restarts (default 3) within max_seconds (default 5). When it gives up, the supervisor itself crashes — handing the problem up to its supervisor. Failure escalates up the tree until someone at the right altitude can deal with it. That escalation is the whole design, not an accident.

So what did we actually gain? A place to write down our recovery policy as data. The list of children, the strategy, the restart limits — that's an executable spec of "what this part of the system needs to be alive, and what to do when it isn't." Reading a project's supervision tree tells you more about how it behaves under stress than reading any single module. We'll grow this tree at runtime next.

Challenges

  1. Two workers where the second reads state the first owns. Which strategy keeps them consistent after a crash — and why not :one_for_one?
  2. Set max_restarts: 2, max_seconds: 10 on a child that crashes on a loop. Time how long until the supervisor itself gives up.
  3. Change a worker's restart: to :transient and make it exit :normal. Confirm it is not restarted.
