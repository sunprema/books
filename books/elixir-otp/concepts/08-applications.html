Chapter 08 · Where It All Starts

Applications & the Boot Tree

Every process we've built needs a root. This is the root.

We've built processes, taught them to crash well, wrapped them in GenServers, and organized them under supervisors both static and dynamic. One question remains: when you type mix run or start a release, who starts the first supervisor? The answer is the Application — the OTP unit of "a thing you start and stop as a whole." It's the trunk from which your entire supervision tree grows.

The one idea

An Application has a single callback, start/2, whose job is to start your top-level supervisor and return its PID. The BEAM calls it for you at boot and calls stop/1 at shutdown. Your whole system is then one supervision tree hanging off that one root — which is why a graceful shutdown, a clean restart, and an orderly boot all come for free.

The two pieces

First, mix.exs names the module the BEAM should call at startup:

mix.exsdef application do
  [
    extra_applications: [:logger],
    mod: {MyApp.Application, []}   # ← start MyApp.Application.start/2 at boot
  ]
end

Then the application module itself, which starts the top supervisor listing every long-lived process in the system — pulling together everything from the previous chapters:

lib/my_app/application.exdefmodule MyApp.Application do
  use Application

  @impl true
  def start(_type, _args) do
    children = [
      # a phone book for dynamic processes (Ch. 06)
      {Registry, keys: :unique, name: MyApp.Registry},
      # a couple of always-on services (Ch. 03)
      MyApp.Cache,
      {MyApp.Stack, ["a", "b"]},
      # supervised background jobs (Ch. 07)
      {Task.Supervisor, name: MyApp.TaskSup},
      # room to grow processes at runtime (Ch. 05)
      {DynamicSupervisor, name: MyApp.WorkerSup, strategy: :one_for_one}
    ]

    # the root supervisor — the trunk of the whole tree
    Supervisor.start_link(children, strategy: :one_for_one, name: MyApp.Supervisor)
  end
end

The whole system, one picture

Config & order

Children start in listed order and stop in reverse — so list a dependency before the things that need it (Registry before the sessions that register in it). Read runtime settings with Application.get_env/3, and prefer config/runtime.exs for anything that must be read from the environment at boot (like a database URL in a release).

So what did we actually gain? A system, not a pile of processes. The Application gives your tree a single on/off switch and a deterministic boot order. Stack the eight chapters up: isolated processes that crash cleanly, linked into supervised trees, named through a registry, sized with the right tool, all rooted in one Application. That is OTP — not a framework you call, but a shape you grow your system into. Now go build something that refuses to stay down.

Challenges

  1. Generate a fresh app with mix new my_app --sup and find the Application module Mix wrote for you. It's exactly this.
  2. Add a deliberately-crashing worker to the tree, start the app, and watch the supervisor restart it in the logs.
  3. Reorder two dependent children the wrong way and observe the boot failure. Then fix the order. Order is a contract.
