Chapter 07 · The Lighter Tools

Task & Agent

Not every job deserves a whole GenServer.

A GenServer is wonderfully general, and generality has a cost: for simple jobs it's ceremony. If all you want is to run some work concurrently, reach for Task. If all you want is to hold a bit of shared state, reach for Agent. Both are GenServers underneath — same isolation, same supervision — with a tiny API shaped to one job. Knowing when to drop down to them keeps your code honest.

The one idea

Task = concurrency without state: spin off work, optionally wait for its result. Agent = state without behaviour: a value other processes can read and update safely. Reach for a full GenServer only when you need both — stateful logic that reacts to many kinds of messages over time.

Task: run it alongside

The classic use is fan-out: kick off several slow things at once and gather the results. Each Task.async runs the function in its own process; Task.await blocks until it's done (using a monitor, so a crash surfaces as a clean error, not a hang).

fan out, then gather# fire three requests concurrently...
tasks =
  ["/a", "/b", "/c"]
  |> Enum.map(fn path -> Task.async(fn -> fetch(path) end) end)

# ...then wait for all of them (total time ≈ the slowest one, not the sum)
results = Enum.map(tasks, &Task.await(&1, 10_000))

For processing a big collection concurrently with a bounded number of workers, there's a purpose-built tool — Task.async_stream — which caps concurrency for you so you don't launch a million tasks at once:

bounded concurrency over a collectionurls
|> Task.async_stream(&fetch/1, max_concurrency: 20, timeout: 10_000)
|> Enum.to_list()
#=> [ok: result1, ok: result2, exit: :timeout, ...]
Supervise your fire-and-forget work

For background work whose result you'll never await, don't just Task.start and forget — run it under a Task.Supervisor so a crash is caught and logged rather than silently lost, and so the BEAM shuts the work down cleanly. Add {Task.Supervisor, name: MyApp.TaskSup} to your tree and use Task.Supervisor.start_child/2.

Agent: shared state, no boilerplate

An Agent wraps a single value behind a process. You hand it an anonymous function to read or update that value; the Agent runs your function in its own process, so concurrent access is serialized and safe — same guarantee as a GenServer's mailbox, none of the callbacks.

a counter, the easy way{:ok, _} = Agent.start_link(fn -> 0 end, name: Counter)

Agent.update(Counter, fn n -> n + 1 end)   # mutate
Agent.get(Counter, fn n -> n end)         #=> 1  (read)
Agent.get_and_update(Counter, fn n -> {n, n + 1} end) #=> 1, now holds 2
Keep the fun small — it runs in the Agent

The function you pass to an Agent executes inside the Agent's process, one at a time. Do heavy computation in there and you serialize every other reader and writer behind it. Compute in the caller; use the Agent only to store and swap the finished value. When the logic grows past "get / update," that's your signal to graduate to a real GenServer.

Choosing between the three

You want to…Reach for
Run work concurrently, maybe await a resultTask
Process a big collection with capped concurrencyTask.async_stream
Hold a value many processes read/updateAgent
React to many message types, keep evolving state, run timersGenServer
Start/stop a fluctuating set of the aboveDynamicSupervisor
So what did we actually gain? The right size of tool. OTP isn't "everything is a GenServer" — it's a small kit where each tool is a GenServer specialized down to one job. Using Task for concurrency and Agent for state keeps your intent obvious at a glance, and leaves the full GenServer for when a component genuinely has a life of its own.
