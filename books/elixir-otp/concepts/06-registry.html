Chapter 06 · Finding a Process

Registry & Naming

A PID is a phone number. A Registry is the phone book.

A PID identifies a process, but it's opaque and — worse — it changes. When a supervisor restarts a crashed worker, the replacement has a brand-new PID. So any PID you stashed away is now pointing at a corpse. We need to refer to a process by a stable name — something meaningful like "room:lobby" or {:user, 42} — and have the lookup always resolve to the currently-alive process behind that name.

For a single, always-there process, you already saw the simplest answer in Chapter 03: name: __MODULE__. But that only works for one process per name, known at compile time. For the fluctuating populations of Chapter 05 — one process per user, keyed by a runtime value — we need Registry.

The one idea

Registry is a fast, concurrent, in-memory key→PID map that lives right in the BEAM. Register a process under any term as a key; look it up later to get the live PID. Best of all, entries are automatically removed when the process dies — the phone book never lists a disconnected number.

The :via tuple — self-registering GenServers

The cleanest pattern is to let each GenServer register itself under a business key as it starts, using a :via tuple as its name. Then call and cast can address the process by that key directly — no PID bookkeeping anywhere.

lib/my_app/session.exdefmodule MyApp.Session do
  use GenServer

  def start_link(user_id) do
    GenServer.start_link(__MODULE__, user_id, name: via(user_id))
  end

  # the :via tuple: "register me in MyApp.Registry under this key"
  defp via(user_id), do: {:via, Registry, {MyApp.Registry, {:session, user_id}}}

  # address the process by user_id — resolves to the live PID for you
  def whoami(user_id), do: GenServer.call(via(user_id), :whoami)

  @impl true
  def init(user_id), do: {:ok, user_id}

  @impl true
  def handle_call(:whoami, _from, user_id), do: {:reply, user_id, user_id}
end

Wire the Registry into the tree

A Registry is itself a process, so it goes in your supervision tree, and it pairs naturally with the DynamicSupervisor from last chapter:

application childrenchildren = [
  {Registry, keys: :unique, name: MyApp.Registry},
  {DynamicSupervisor, name: MyApp.WorkerSup, strategy: :one_for_one}
]
start on demand, then reach by keyDynamicSupervisor.start_child(MyApp.WorkerSup, {MyApp.Session, 42})

MyApp.Session.whoami(42)   #=> 42   (found the live PID via the key)

# or look the PID up yourself:
Registry.lookup(MyApp.Registry, {:session, 42})
#=> [{#PID<0.312.0>, nil}]   (empty list [] if nobody's registered)
:unique vs :duplicate

A :unique registry allows one process per key — perfect for "the session for user 42." A :duplicate registry allows many processes under one key — perfect for pub/sub, where every subscriber to "topic:news" registers under that key and you dispatch to all of them at once. Same tool, two shapes of problem.

Why not just keep the PID?

Because restarts. The instant a supervised process crashes and is replaced, its PID is stale. The Registry entry, by contrast, is cleaned up on death and re-created when the fresh process registers itself on start — so a lookup by key always yields the current process, or an honest empty list. Name things by what they mean, never by their current PID.

So what did we actually gain? Stable identity in a world of disposable processes. DynamicSupervisor lets processes come and go; Registry lets the rest of the system keep talking to "user 42's session" without ever caring which incarnation is currently answering. Crash-and-restart becomes invisible to callers — which is the entire promise of fault tolerance, delivered.

Challenges

  1. Call whoami/1 for a user who was never started. What error do you get, and why is failing loudly the right behaviour here?
  2. Start a session, crash it, and (with a supervisor restarting it) confirm a later lookup still resolves. Inspect the PID before and after.
  3. Build a tiny pub/sub with a :duplicate registry: subscribers register under a topic, a publisher uses Registry.dispatch/3 to message them all.
