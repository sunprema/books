Chapter 07 · Who May Do What

7Authorization & Policies

Authorization in Ash isn't scattered through controllers — it's declared on the resource, next to the actions it guards.

First, two words you'll see everywhere:

  • Actor — who is doing the thing (usually the current user). You pass it in with actor:.
  • Policy — a rule that decides whether that actor may run an action.

Turn policies on for a resource by adding the authorizer, then write the rules in a policies block:

on the resourceuse Ash.Resource,
  domain: Helpdesk.Support,
  authorizers: [Ash.Policy.Authorizer]

policies do
  # admins can do anything
  bypass actor_attribute_equals(:admin, true) do
    authorize_if always()
  end

  # otherwise: you may only read tickets assigned to you
  policy action_type(:read) do
    authorize_if expr(representative_id == ^actor(:id))
  end
end

Read it as sentences:

  • policy action_type(:read) — “this rule applies to read actions.” That's the condition.
  • authorize_if expr(...) — “allow it if the ticket's representative_id equals the actor's id.” That's the check.
  • bypass — a short-circuit: if the actor is an admin, allow and stop checking.
The rule of decisions Checks are tried in order, and the first check that reaches a decision wins. You have authorize_if / forbid_if / authorize_unless / forbid_unless to express it whichever way reads clearest.

Filtering, not just forbidding

Here's the elegant part. For read actions, a policy doesn't just throw “forbidden” — by default it filters the query. Ask for all tickets as a non-admin, and Ash quietly returns only the ones you're allowed to see. Authorization becomes part of the WHERE clause.

How you trigger it Policies run when a call is authorized. Pass the actor through the code interface — Support.list_tickets!(actor: current_user) — or set authorize?: true. Forget the actor and the checks can't reference one, so wire it in at your app's boundary once.
Don't hand-roll auth around Ash The temptation is to add if user.admin? checks in your LiveView. Resist it. Put the rule in the policies block so every path — web, GraphQL, JSON:API, background jobs — is guarded by the same declaration. One rule, enforced everywhere.
