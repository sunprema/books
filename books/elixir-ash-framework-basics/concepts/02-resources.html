Chapter 02 · The Noun

2Resources

A resource is a static definition of one “thing” in your system — and it becomes the single source of truth about that thing.

Let's write one. Here is a Ticket for our helpdesk. Read it top to bottom — it's almost English:

lib/helpdesk/support/ticket.exdefmodule Helpdesk.Support.Ticket do
  use Ash.Resource,
    domain: Helpdesk.Support,
    data_layer: Ash.DataLayer.Ets

  attributes do
    uuid_primary_key :id

    attribute :subject, :string do
      allow_nil? false
      public? true
    end

    attribute :status, :atom do
      constraints one_of: [:open, :closed]
      default :open
      allow_nil? false
    end
  end
end

Every piece is doing real work:

  • use Ash.Resource — this module is a resource. It names its domain (chapter 4) and its data_layer (chapter 6).
  • attributes — the fields. Each declares its type and its rules right there: allow_nil?, default, and constraints like one_of:.
  • public? — whether the field is exposed through derived APIs. Fields are private by default; you opt in.
Why this matters That one_of: [:open, :closed] constraint isn't just for the database. It becomes a validation, part of the GraphQL enum, part of the JSON:API schema, and something the admin UI understands — because they all read this one definition.

The anatomy of a resource

A resource is organized into a handful of named blocks. You've seen attributes; the others slot in the same way. Here's the whole shape at a glance — we'll meet each block as the book goes on.

Foot-gun to remember A brand-new attribute is not public. If a field mysteriously won't appear in your GraphQL or JSON:API output, you almost certainly forgot public? true. This default is deliberate — it keeps you from leaking internal fields by accident.

Notice what's missing: there's no separate schema file, no ORM model, no migration written by hand (AshPostgres can generate those), and no DTO. The resource is all of those things at once. Next we give it verbs.
