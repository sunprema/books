Chapter 04 · Grouping & Calling

4Domains & Code Interfaces

A domain groups related resources. A code interface turns their actions into plain, friendly functions you call from anywhere.

The domain: a table of contents for resources

Resources don't float loose — they live inside a domain. A domain lists its resources and holds domain-wide configuration. Ours is tiny:

lib/helpdesk/support.exdefmodule Helpdesk.Support do
  use Ash.Domain

  resources do
    resource Helpdesk.Support.Ticket
    resource Helpdesk.Support.Representative
  end
end

Think of a domain as a bounded context: “Support”, “Billing”, “Accounts”. Grouping resources this way keeps a large app legible, and lets extensions configure a whole slice of the system at once.

Naming that reads well Notice Helpdesk.Support.Ticket — the domain (Support) is right there in the module path. When someone new opens your codebase, the folder structure already tells them the shape of the business.

The pipeline is powerful… but verbose

Building a changeset and piping it into Ash.create! every time gets wordy. For everyday calls you want something that reads like a normal function. That's the code interface.

Code interfaces: name the call

You can define the interface right on the resource…

on the resourcecode_interface do
  define :open, args: [:subject]
end

…and now you call it as a function, no pipeline in sight:

Helpdesk.Support.Ticket.open!("My mouse won't click!")

Or you can define it on the domain, which reads even more like an API for the whole context:

on the domainresources do
  resource Ticket do
    define :open_ticket, args: [:subject], action: :open
  end
end

# call it:
Helpdesk.Support.open_ticket!("My mouse won't click!")

A very common one is a “get by” lookup:

define :get_ticket_by_id, action: :read, get_by: [:id]

# then:
Helpdesk.Support.get_ticket_by_id!(id, load: [:representative])
This is the layer you'll live in Every definition generates both a raising (open_ticket!) and a non-raising (open_ticket) version, and they all accept an actor: option for authorization (chapter 7) and options like load:. Your controllers and LiveViews mostly just call these.
