Chapter 06Lifetimes in Function Signatures

Functions are where lifetime annotations earn their keep. One famous ten-line example — longest — contains everything there is to know. We'll build it, break it, and read the compiler's mind at a call site.

The problem: which input does the output borrow from?

Let's write a function that returns the longer of two string slices. First attempt, no annotations:

fn longest(x: &str, y: &str) -> &str {
    if x.len() > y.len() { x } else { y }
}
error[E0106]: missing lifetime specifier
  |
1 | fn longest(x: &str, y: &str) -> &str {
  |               ----     ----     ^ expected named lifetime parameter
  |
  = help: this function's return type contains a borrowed value,
    but the signature does not say whether it is borrowed from `x` or `y`

Pause on the help line — it states the real issue perfectly. The returned reference borrows from an input (where else could it point?), but a caller reading only the signature can't tell which one. And the caller must know, because the caller's borrow checker needs to figure out how long the returned reference stays valid. Rust checks each function against its signature alone — it never peeks across function boundaries — so the signature has to carry this information.

The fix: name the relationship

fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() { x } else { y }
}

Read the signature as a sentence: “For some region 'a: given two string slices both valid throughout 'a, I return a slice valid throughout 'a.” We haven't said which input comes back — we've said something weaker and sufficient: the result lives no longer than the region where both inputs are valid. Since the answer might be either input, that's exactly the honest promise.

The Key Idea

At each call site, the compiler picks the concrete region for 'a: the largest region that fits inside both arguments' lifetimes — their overlap. The returned reference is then usable only within that overlap. Different call sites get different 'as; the annotation is a rule, not a value.

Watching a call site get checked

Here's the Rust Book's demonstration pair. First, a call that works:

let string1 = String::from("long string is long");
{
    let string2 = String::from("xyz");
    let result = longest(string1.as_str(), string2.as_str());
    println!("The longest string is {result}");   // ✓ inside the overlap
}

And one that doesn't:

let string1 = String::from("long string is long");
let result;
{
    let string2 = String::from("xyz");
    result = longest(string1.as_str(), string2.as_str());
}                                                    // string2 dies here
println!("The longest string is {result}");         // ✗ used past the overlap
error[E0597]: `string2` does not live long enough

We humans can see the result would be string1 — “long string is long” wins. Doesn't matter. The signature promised only the overlap, string2's inner block, and result is used after it. The checker verifies promises, not runs. If you need the long-way-around proof that this strictness is right: imagine the strings were read from a file at run time and string2 won.

Annotations follow the data

The corollary: only relationships that exist need names. If the result can only come from one input, only that input shares the output's lifetime:

fn first_or_default<'a>(x: &'a str, y: &str) -> &'a str {
    if !x.is_empty() { x } else { "default" }
}

y gets no annotation tying it to the return — callers may pass a y that dies immediately after the call, and the result stays good as long as x. Precision here is kindness to your callers: every unnecessary lifetime link is a restriction they pay for. Annotate the flows that exist; leave the rest free.

Common Trap

You cannot conjure a reference from nothing. This fails no matter what you annotate:

fn longest<'a>() -> &'a str {
    let result = String::from("really long string");
    result.as_str()      // ✗ E0515: references data owned by
}                            //   the current function

The local dies at the brace. The honest signature returns the owned String and moves it out. When a lifetime error resists annotation, the flow itself is wrong — change who owns, not what's written on the slip.

A worked translation exercise

Practice reading signatures as promises. Cover the right column and try each one:

SignatureThe promise, in English
fn f<'a>(s: &'a str) -> &'a str “My result borrows from s; it's good as long as s is.”
fn f<'a,'b>(x: &'a str, y: &'b str) -> &'a str “The result tracks x only; y can die right after the call.”
fn f<'a>(x: &'a str, y: &'a str) -> &'a str “The result is only good while both inputs are alive (their overlap).”
fn f(s: &str) -> String “No strings attached — I return an owned value; borrow ends at my return.”

So what did we actually gain?

Function lifetime annotations answer one question for callers: which inputs does the output borrow from? The compiler solves each call by intersecting the named inputs' regions and confining the result to that overlap. Annotate real flows, keep unrelated inputs free, and when no annotation helps, return ownership instead. But wait — you've written a hundred functions taking &str with no 'a in sight. Why did those compile? Because the compiler was annotating for you. Next chapter: the elision rules.

Challenges

  1. Change longest to always return x. Now loosen the signature so the failing call site above compiles. (Hint: y needs its own lifetime — or none.)
  2. Write the signature for fn split_once(s: &str, sep: &str) -> (&str, &str) with explicit lifetimes. Which input do the outputs borrow from?
  3. Why does Rust check functions against signatures instead of inferring lifetimes from bodies? What would break — for compile times, for semver, for error messages — if bodies leaked through?
