Chapter 10The Borrow Checker's Mind: NLL and Beyond

We've spent nine chapters obeying the borrow checker. In this last one, we sit down at its bench: how it reasoned in 2015, the upgrade that made it humane (non-lexical lifetimes), the successor being forged (Polonius), and the advanced corners — HRTBs and the 2024-edition capture rules — where its reasoning shows through the syntax.

Act I — lexical lifetimes: the brace was the law

Rust 1.0's checker used lexical lifetimes: a borrow lasted from creation to the end of its enclosing scope — the closing brace — whether or not you ever touched it again. Simple to implement, miserable to live with:

let mut map = HashMap::new();
match map.get_mut(&key) {         // borrow began here…
    Some(v) => *v += 1,
    None => { map.insert(key, 1); }  // ✗ 2015: map still “borrowed”
}                                   // …lexically, until here

Every seasoned Rustacean of that era learned the workarounds: extra blocks to force borrows to end, contorted control flow, the occasional sacrificial clone(). The checker wasn't wrong, exactly — it was coarse. It measured loans by the brace, not by use.

Act II — NLL: loans end at last use

The 2018 edition shipped non-lexical lifetimes (fully default for all editions by Rust 1.63, August 2022). The insight: stop computing lifetimes on the source code's block structure, and compute them on the control-flow graph — the compiler's map of every path execution can take. A borrow's region became: the set of program points from creation through its last use along each path.

let mut s = String::from("hello");
let r1 = &s;
let r2 = &s;
println!("{r1} {r2}");    // last use — r1, r2's loans END here
let r3 = &mut s;          // ✓ fine since 1.31 — no live overlap
r3.push_str(" world");

Under lexical rules this was three overlapping brace-long loans — an error. Under NLL, r1 and r2 are already dead when r3 is born. The rules of chapter 4 didn't change; the measurement of a loan's span got honest. That one refinement retired most of the folk-workarounds overnight.

The Key Idea

The modern checker asks, at every program point, for every loan: “is this borrow live — could it still be used from here?” Conflicts only matter between live loans. When you read a borrow error today, find the “borrow later used here” note: that use is what's keeping the loan alive. Delete or move the use and the loan shrinks.

Act III — Polonius: from “where valid” to “who's owed what”

Even NLL has famous blind spots. The best-known is “problem case #3,” returning a borrow from a loop-with-fallback:

fn get_or_insert<'m>(map: &'m mut Map) -> &'m Value {
    if let Some(v) = map.get(&key) {
        return v;              // borrow escapes on THIS path only…
    }
    map.insert(key, Value::new());  // ✗ NLL: “map still borrowed”
    map.get(&key).unwrap()
}

Any human can see the first borrow only matters if we returned. NLL can't: because the borrow flows into the return value, its region gets stretched to cover the whole call — including the path where it was never returned at all. The workaround (restructure to look up twice, or use the entry API) works, but the rejection is a false alarm.

Polonius is the next-generation analysis built to fix exactly this class. It flips the question. NLL computes, top-down, “where is each lifetime valid?” Polonius instead tracks origins — for each reference, which loans might it contain? — and only flags an action if it invalidates a loan some still-live reference actually depends on, on that path. As of mid-2026 it remains experimental (-Z polonius on nightly; the team's stated goal is an initial stabilization of the NLL-compatible subset), so treat it as a lens, not a tool: thinking in loans-and-origins is already the most accurate mental model of what the checker owes you.

Note

Practical takeaway while Polonius bakes: when you hit a false alarm like the one above, reach (in order) for restructuring the control flow, the entry-style API that does the branch inside the library, or an early-computed value. These aren't defeats — they're the same proof, written where today's checker can read it.

The advanced corners, briefly

HRTBs: “for any lifetime you like”

Sometimes a bound must hold not for one caller-chosen region but for every possible one. That's a higher-ranked trait bound, written with for<'a>:

fn apply_to_lines<F>(text: &str, f: F)
where
    F: for<'a> Fn(&'a str) -> &'a str,
{ … }

Read for<'a> as “for all regions 'a”: the closure must handle a borrow of any span — including ones that don't exist yet when the closure is created, like each loop iteration's fresh slice. You've used HRTBs unknowingly all along: elided fn(&str) -> &str pointer types and closure bounds desugar to them.

The 2024 edition: what does impl Trait capture?

One genuinely new rule to file. A function returning impl Trait hides a concrete type — but if that type borrows from a parameter, the caller's checker must know. Which lifetimes does the opaque type capture? In the 2024 edition, the answer became: all in-scope generic and lifetime parameters, by default (earlier editions captured lifetimes only when explicitly mentioned). When that default is too grabby — the return doesn't actually borrow from everything — you release lifetimes with the use<…> syntax:

// 2024: captures 'a by default — the caller must assume borrowing:
fn nums<'a>(xs: &'a [i32]) -> impl Iterator<Item = i32> { … }

// explicitly capture nothing — result is independent of xs:
fn nums<'a>(xs: &'a [i32]) -> impl Iterator<Item = i32> + use<> {
    xs.to_vec().into_iter()   // owns its data, borrows nothing
}

If a borrow error points at an impl Trait return and you never wrote a lifetime — this is where to look.

So what did we actually gain — from the whole book?

Let's close the shop. Memory is a workbench and a storeroom, and Rust's answer to “who frees?” is the owner's closing brace (chapters 2–3). Borrows are loans under two rules — shared XOR mutable, and never outlive the data (chapter 4). A lifetime is a region of code; annotations describe relationships and change nothing (chapter 5). Functions declare which inputs their outputs borrow from; elision writes the obvious cases for you (chapters 6–7). Types that hold borrows are borrows (chapter 8). 'static means self-contained more often than it means forever (chapter 9). And the checker enforcing all of it measures loans by liveness on the control-flow graph, getting sharper with every generation (this chapter).

The next time the borrow checker says no, you won't hear a riddle. You'll hear a colleague pointing at two bars on a timeline that don't fit — and you'll know which one to move. That's the whole craft: not fighting the checker, but seeing what it sees, a half-second sooner.
