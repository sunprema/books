Chapter 03Ownership, Moves, and Drop

One owner per value. Ownership can move, but never split. When the owner's scope ends, the value is destroyed. Three sentences that replace an entire garbage collector — let's earn them.

The rules of the shop

Rust's ownership system is small enough to write on a shop wall:

The Ownership Rules

1. Each value in Rust has exactly one owner.
2. Ownership can be moved — but then the old owner is done.
3. When the owner goes out of scope, the value is dropped.

Why so strict? Remember the question from last chapter: who frees the heap buffer? If two variables both “owned” the same buffer, both would free it at their closing braces — a double-free. If zero variables owned it, nobody would free it — a leak. Exactly one is the only count that makes the closing brace a safe place to put the cleanup. The whole design falls out of that arithmetic.

Moves: handing over the tool

So what happens when you assign a heap-owning value to another variable? Rust moves it: the new variable becomes the owner, and the old one is dead — not zeroed, not dangling, just statically forbidden.

let s1 = String::from("hello");
let s2 = s1;                 // ownership MOVES to s2
println!("{s1}");            // ✗ error: borrow of moved value

And the compiler's verdict, which is worth reading closely because it teaches the model:

error[E0382]: borrow of moved value: `s1`
 --> src/main.rs:3:11
  |
1 | let s1 = String::from("hello");
  |     -- move occurs because `s1` has type `String`,
  |        which does not implement the `Copy` trait
2 | let s2 = s1;
  |          -- value moved here
3 | println!("{s1}");
  |           ^^^^ value borrowed here after move

Mechanically, the move is cheap: the three stack words (pointer, length, capacity) are copied to s2, and the heap bytes never move at all. What changed is bookkeeping: the compiler now knows s2 is responsible for the buffer, and marks s1 as uninitialized. No shallow-copy-plus-shared-mutation surprise like Python lists; no silent deep copy like C++ copy construction. A move is a transfer of responsibility.

Moves happen everywhere assignment-like things happen: passing a value to a function moves it in; returning a value moves it out; storing it in a struct or pushing it into a Vec moves it into the container. Each time, ownership — and with it, the duty to free — relocates.

fn takes(s: String) { println!("{s}"); }   // s dropped here

let s = String::from("borrowed? no — gone");
takes(s);            // moved into the function…
// println!("{s}");  // ✗ …so s is unusable here

If you genuinely want two independent copies of the heap data, say so — s.clone() duplicates the buffer, explicitly and visibly. Rust never deep-copies behind your back; expensive things look expensive in the source.

Drop: the closing brace is the destructor

The third rule is where memory management actually happens. When an owner goes out of scope, Rust calls Drop::drop on it — freeing buffers, closing files, releasing locks, whatever that type cleans up. You can watch it happen:

struct Noisy(&'static str);

impl Drop for Noisy {
    fn drop(&mut self) { println!("dropping {}", self.0); }
}

fn main() {
    let _a = Noisy("a");
    {
        let _b = Noisy("b");
    }                            // prints: dropping b
    println!("end of main");
}                                // prints: dropping a

Deterministic, ordered (locals drop in reverse declaration order), and free of runtime machinery. C++ programmers will recognize RAII; the Rust twist is that the move rules make it safe — you can't accidentally keep a second handle to a value that's about to be destroyed, because ownership never split in the first place.

Note

Need shared ownership anyway — a value with several rightful owners, freed when the last one leaves? That's what Rc<T> (and thread-safe Arc<T>) provide: a reference count paid at run time. It's the escape hatch, not the default — and even then, the count is what's shared; the single-owner rule still governs the Rc handles themselves.

Why this matters for lifetimes

Here's the connection to our real subject. Ownership pins down, for every value, the exact program point where it dies: the owner's closing brace (or earlier, if moved away). That certainty is what makes lifetime checking possible at all. A reference is safe precisely when its use ends before the owner's death point — and because death points are static, the compiler can check that with no runtime help. Garbage-collected languages can't make this check because values have no fixed death point; C doesn't make it because it never tracked ownership. Rust can, because rule 3 turned “when does this die?” into a compile-time fact.

So what did we actually gain?

Three rules, one consequence: every value has a statically known destruction point. Moves transfer the duty to free; clone duplicates it visibly; Drop executes it at the brace. Memory management in Rust isn't a runtime service — it's a property of the program's shape. Now we can finally afford to lend things out without giving them away: borrowing.
