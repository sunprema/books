Chapter 08Lifetimes in Structs and Methods

Sooner or later you'll want a type that holds a reference — a parser over its input, a view into a buffer. The moment a struct contains a borrow, the struct itself becomes a borrow, and everything we know about loan periods applies to whole values.

A struct that borrows

The Rust Book's example is small and perfect. Suppose we want to keep the first sentence of a novel around, without copying it:

struct ImportantExcerpt<'a> {
    part: &'a str,
}

fn main() {
    let novel = String::from("Call me Ishmael. Some years ago...");
    let first_sentence = novel.split('.').next().unwrap();
    let i = ImportantExcerpt { part: first_sentence };
}

Why is <'a> mandatory here when functions got away with elision? Because a struct definition is a contract about storage, not a local computation. Whoever holds an ImportantExcerpt holds, transitively, a borrow of some string — and the type system needs that fact to travel with the value wherever it goes: into vectors, across function calls, into other structs. The lifetime parameter is how the fact travels.

The Key Idea

Read ImportantExcerpt<'a> as: “a value that must not outlive region 'a, because it contains a reference only valid there.” A struct holding a borrow is a borrow — bigger and better dressed, but bound by the same loan period as its most short-lived field.

The compiler enforces exactly what it enforced for plain references — the excerpt can't outlive the novel:

let i;
{
    let novel = String::from("Call me Ishmael...");
    let first_sentence = novel.split('.').next().unwrap();
    i = ImportantExcerpt { part: first_sentence };
}                          // ✗ novel dropped while `i.part` still borrows it
println!("{}", i.part);    //   E0597: `novel` does not live long enough

Borrow or own? The design decision

Before reaching for a lifetime parameter, ask the workshop question: should this type borrow the data or own it? Both are one line:

struct Borrowing<'a> { part: &'a str }   // zero-copy, but tethered
struct Owning            { part: String }   // independent, but copies
Borrowing (&'a str)Owning (String)
CostFree — points into existing dataAllocates and copies
MobilityTethered to the source's scopeGoes anywhere, lives freely
Best forShort-lived views: parsers, iterators, search resultsLong-lived data: configs, caches, anything stored
API impactLifetime parameter spreads to every containing typeNone

That last row is the one experience teaches. A lifetime parameter is infectious: put Borrowing<'a> inside another struct, and that struct needs <'a> too, and so on up the tree. For hot paths and view types, that's a price worth paying — zero-copy parsing is a Rust superpower. For a type you'll stash in a HashMap and forget about, own the data and keep your signatures clean. Neither answer is “more Rust”; matching the tool to the job is.

Methods on borrowing types

Writing an impl for a lifetime-carrying type means declaring the parameter after impl and using it after the type name — it's a generic parameter, same as T:

impl<'a> ImportantExcerpt<'a> {
    fn announce_and_return_part(&self, announcement: &str) -> &str {
        println!("Attention please: {announcement}");
        self.part
    }
}

Look at that method: two reference parameters, no annotations, compiles fine. You can name the reason now — elision Rule 3. self's lifetime flows to the output. And here's a subtlety worth savoring: the rule tied the return to &self's borrow, but self.part is actually good for 'a — the excerpt's whole region, which outlives any particular &self borrow. The elided signature is correct but conservative. If a caller ever needs the looser truth, say it explicitly:

impl<'a> ImportantExcerpt<'a> {
    // the part outlives this borrow of self — return the 'a truth:
    fn part(&self) -> &'a str {
        self.part
    }
}

With the elided version, this fails; with -> &'a str, it works:

let part;
{
    let i2 = ImportantExcerpt { part: first_sentence };
    part = i2.part();
}                        // i2 (the struct) is gone…
println!("{part}");      // ✓ …but the novel isn't, and part borrows the NOVEL

This is the chapter's deepest lesson: the struct is a middleman borrow. The data's true owner is the novel; 'a tracks that original loan, and a method can hand out the original slip rather than a copy tied to the middleman's own borrow.

Common Trap

The self-referential struct: “I'll store the String and a &str slice into it, in the same struct.” Safe Rust forbids it — moving the struct would move the String's stack handle and invalidate any interior pointer's provenance story, and no lifetime name can describe “borrows from a sibling field.” Store indices (usize ranges) instead and slice on demand, or keep owner and view in separate types. (Crates like ouroboros exist for the rare genuine need; reach for them last.)

So what did we actually gain?

Types can borrow, and when they do, the lifetime parameter carries the loan period through the type system: struct-holding-a-borrow is a borrow; the parameter is infectious; own-vs-borrow is a design decision about mobility versus copies. Methods get Rule 3's conservative default, and you can return the looser 'a truth when callers need it. One lifetime keeps appearing in error messages without ever being declared. Time to meet 'static.
