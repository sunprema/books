Chapter 07Elision: The Rules You've Been Using All Along

Most Rust functions that juggle references carry no lifetime annotations at all. That's not because lifetimes are absent — it's because the compiler fills them in by three fixed, purely syntactic rules. Learn to run the rules by hand and half the mystery of lifetime errors evaporates.

A bit of history

In pre-1.0 Rust, every reference in every signature needed an explicit lifetime. Code was wallpapered with <'a>. The team studied the codebase and found that a tiny number of patterns covered the overwhelming majority of signatures — so those patterns became defaults (RFC 141, “lifetime elision”), and the annotations disappeared from everyday code. That's why this compiles bare:

fn first_word(s: &str) -> &str {
    s.split_whitespace().next().unwrap_or("")
}

Elision is abbreviation, not inference. The compiler doesn't look at your function body and deduce clever lifetimes; it applies three mechanical rewrite rules to the signature. If the rules produce an answer, that answer is law. If they don't, you get E0106 and write the lifetimes yourself. Nothing in between.

The three rules

The Elision Rules

Rule 1. Each elided lifetime in the parameters becomes its own fresh lifetime parameter.
Rule 2. If there is exactly one input lifetime, it is assigned to every elided output lifetime.
Rule 3. If there are multiple input lifetimes but one of them is &self or &mut self, the lifetime of self is assigned to every elided output lifetime.

Let's run them by hand, the way the compiler does.

Worked example 1: one input

// you write:
fn first_word(s: &str) -> &str

// Rule 1 — each input gets a fresh lifetime:
fn first_word<'a>(s: &'a str) -> &str

// Rule 2 — exactly one input lifetime → it flows to the output:
fn first_word<'a>(s: &'a str) -> &'a str   // ✓ done

Sensible, isn't it? If a function takes one reference and returns one, where else could the output borrow from? The default encodes the only honest answer.

Worked example 2: two inputs — the rules give up

// you write:
fn longest(x: &str, y: &str) -> &str

// Rule 1:
fn longest<'a, 'b>(x: &'a str, y: &'b str) -> &str

// Rule 2: two input lifetimes — doesn't apply.
// Rule 3: no self — doesn't apply.
// Output lifetime still unknown → E0106, over to you.

Now you know exactly why chapter 6's longest demanded annotations while first_word sailed through. The rules aren't smart; ambiguity stops them cold, on purpose. A guess here would silently encode the wrong contract.

Worked example 3: methods, where Rule 3 shines

// you write:
impl Parser {
    fn peek(&self, offset: usize) -> &str { … }
}

// Rules 1+3 — self's lifetime flows to the output:
fn peek<'a>(&'a self, offset: usize) -> &'a str

Rule 3 is why method-heavy code almost never writes lifetimes: “the return borrows from self” is nearly always the truth. When it isn't — say a method that returns a slice of its argument rather than of self — the default is wrong, and you must annotate to overrule it:

impl Highlighter {
    // the return borrows from `text`, NOT from self:
    fn trim<'t>(&self, text: &'t str) -> &'t str {
        text.trim()
    }
}

Without the annotation, Rule 3 would tie the result to &self — and callers would mysteriously find their trimmed strings imprisoned by the highlighter's borrow. When a lifetime error makes no sense, de-elide by hand: write out what the rules produced, and the phantom in the error message will step into the light.

The supporting cast

A few more places lifetimes get filled in without your help:

const and static declarations default reference lifetimes to 'static: const GREETING: &str = "hello"; means &'static str. Makes sense — what else could a global borrow from?

Trait objects get a default object lifetime bound: Box<dyn Draw> outside an expression means Box<dyn Draw + 'static>, while &'a dyn Draw defaults to &'a (dyn Draw + 'a) — the container's lifetime rubs off. File this away for chapter 9; it explains the sudden 'static demands around boxed closures.

The anonymous lifetime '_ opts a single position into the normal rules while making the borrow visible: fn make(s: &str) -> Excerpt<'_>. Rust lints (elided_lifetimes_in_paths) nudge you to write it exactly so readers can see, at a glance, that Excerpt carries a borrow.

Note

Elision applies to function signatures. Struct and enum definitions never elide — a type holding a reference must declare its lifetime parameter explicitly, always. That's next chapter's business.

So what did we actually gain?

The invisible ink is readable now. Every unannotated & in a signature is elision — three syntactic rules: fresh per input; single input flows out; self flows out. When the rules can't decide, you decide. When an error confuses you, de-elide and the signature will confess. One habit to keep: read every elided signature as if the lifetimes were written, because to the compiler, they are.

Challenges

  1. De-elide by hand: fn debug(lvl: usize, s: &str), fn substr(s: &str, until: usize) -> &str, and fn get_mut(&mut self) -> &mut T. Check against the rules table in the Reference.
  2. Why is fn frob(s: &str, t: &str) -> &str an error while fn frob(s: &str, t: &str) is fine?
  3. Write a method where Rule 3's default is wrong and produces a real borrow-checker error at a call site; then fix it with one explicit lifetime.
