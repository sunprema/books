Chapter 04Borrowing: References Without Ownership

Moving a value every time someone needs to look at it would make Rust unbearable. Borrowing lets you lend access while keeping ownership — under two rules that make data races and dangling pointers impossible before the program ever runs.

Lending, not giving

At the end of last chapter you invented a need: use a value in a function without losing it. The answer is a reference — written &value — which lets you borrow the value. The owner stays put; the borrower gets an address and a promise.

fn len_of(s: &String) -> usize {
    s.len()
}                                // s (the reference) goes away;
                                 // the String it points to does NOT

let s = String::from("hello");
let n = len_of(&s);          // lend it…
println!("{s} is {n} bytes"); // ✓ …and s is still ours

In the workshop: &s is a borrow slip. len_of may use the tool, but the slip expires at the end of the loan, and the tool was never theirs to put away. Dropping a reference frees nothing — references don't own.

The two borrow rules

Rust distinguishes shared references (&T — look, don't touch) from mutable references (&mut T — exclusive access, may modify). And it enforces exactly two rules about them:

The Borrow Rules

1. At any point, a value has either any number of shared borrows, or exactly one mutable borrow — never both.
2. Every borrow must end before the owner's data goes away.

Rule 2 is the lifetime story — the whole rest of the book. Rule 1 is the aliasing story, and it deserves a moment, because it's the rule that surprises newcomers most. Why can't I have two mutable references? Why can't I read while someone else writes?

Because aliasing plus mutation is where nightmares live. If a shared reference points into a Vec while a mutable reference pushes onto it, the push may reallocate the buffer — and the shared reference now dangles, in a program with no unsafe in sight. Across threads, the same combination is a data race. Rust's rule — shared XOR mutable — eliminates the whole category, and it's why people say Rust gives you “fearless concurrency”: the borrow rules you learn single-threaded are the same ones that keep threads honest.

let mut v = vec![1, 2, 3];
let first = &v[0];       // shared borrow into the buffer
v.push(4);               // ✗ needs &mut v — might reallocate!
println!("{first}");     // shared borrow still alive here
error[E0502]: cannot borrow `v` as mutable because it is
              also borrowed as immutable
  |
2 | let first = &v[0];
  |              - immutable borrow occurs here
3 | v.push(4);
  |   ^^^^^^^ mutable borrow occurs here
4 | println!("{first}");
  |           ------- immutable borrow later used here

Read that last annotation carefully: “immutable borrow later used here.” The conflict isn't that two borrows were created — it's that their useful lives overlap. That word “used” is doing a lot of work, and it's our first sighting of how the modern borrow checker thinks (chapter 10 makes it precise).

Rule 2, meet your enforcement

The second rule — borrows can't outlive the data — is where the classic error lives. Here's the canonical example from the Rust Book:

fn main() {
    let r;
    {
        let x = 5;
        r = &x;              // borrow x…
    }                        // …x is dropped here
    println!("r: {r}");      // ✗ …but the borrow is still in use
}
error[E0597]: `x` does not live long enough
  |
5 |         let x = 5;
  |             - binding `x` declared here
6 |         r = &x;
  |             ^^ borrowed value does not live long enough
7 |     }
  |     - `x` dropped here while still borrowed
8 |     println!("r: {r}");
  |              --- borrow later used here

How did the compiler check this? It compared two extents: the region of code where x is alive, and the region where the borrow &x is in use. The borrow's region pokes out past x's. Those regions are lifetimes. We've been discussing them all along; the next chapter just gives them their name and notation.

Common Trap

“I'll just return a reference to my local variable.” The local dies at the function's closing brace; the reference would dangle. Rust rejects it (E0515: returns a value referencing data owned by the current function). The fix is almost always to return the owned value — let ownership move out — rather than a reference into a corpse.

So what did we actually gain?

Borrowing completes Rust's memory model: owners hold, borrowers visit. Shared XOR mutable kills data races and iterator invalidation at compile time; borrows-can't-outlive-data kills dangling pointers. And we saw the checker's true currency — regions of code where things are alive or in use. Time to give those regions their proper name.
