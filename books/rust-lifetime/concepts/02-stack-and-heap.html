Chapter 02Stack, Heap, and the Shape of Memory

Before we can talk about how long data lives, we need to know where it lives. This chapter is the memory-management foundation: two regions, two sets of rules, and the question every language must answer — who cleans up?

Two rooms in the workshop

A running program gets (among other things) two regions of memory with very different personalities.

The stack is the workbench: a strip of memory used in strict last-in, first-out order. When a function is called, a frame is pushed onto the bench — space for its parameters and local variables. When the function returns, the whole frame is popped off in one motion. Allocation is a single pointer bump; deallocation is free. The price of that speed: everything on the stack must have a size known at compile time, and nothing on the stack can outlive its frame.

The heap is the storeroom: a big, unordered space where you can request a shelf of any size, at any time, and keep it as long as you like. The allocator finds you a spot and hands back its address — a pointer. Flexibility is the point; the price is that allocation takes real work, and — crucially — someone must remember to give the shelf back.

The question every language must answer

That storeroom shelf has to be returned eventually, or the program slowly eats the machine. Returning it twice, or using it after returning it, is worse — that's undefined behavior, the source of a large share of the security bugs in systems software. So: who cleans up, and when? Historically there have been two answers, and Rust adds a third.

StrategyWho frees?CostFailure mode
Manual (C) You, by calling free Your vigilance Leaks, double-frees, use-after-free
Garbage collection (Java, Go, Python) A runtime, later Memory overhead, pauses, a runtime Unpredictable timing; leaks still possible via live references
Ownership (Rust) The compiler inserts the free at a known point Thinking at compile time (this book!) Programs that don't compile until the proof works

Rust's answer is the interesting one: every heap allocation has exactly one owner, and when the owner goes out of scope, the compiler inserts the cleanup call right there — deterministically, at compile time, with no runtime watching over your shoulder. It's as fast as the C answer and as safe as the garbage-collected one. The mechanism that makes it work is ownership, and it's the subject of the next chapter.

The Key Idea

Rust's memory management is its scoping rules. “When is this freed?” always has a static answer: at the end of the owner's scope. Lifetimes exist to extend that guarantee to references — pointers that must never outlive the owned data they point into.

Seeing it in Rust

Let's make the picture concrete. Here's a function with both kinds of data:

fn workbench() {
    let x = 42;                          // i32: 4 bytes, lives on the stack
    let s = String::from("hello");     // handle on the stack, bytes on the heap
    let y = x;                           // copy: two independent 42s
    println!("{x} {y} {s}");
}                                        // ← frame pops: x, y vanish; s's heap
                                         //   buffer is freed here, automatically

Notice what happens at the closing brace. The stack frame pops — that part is universal. But Rust also knows s owns a heap buffer, so it inserts a call to s's destructor (its Drop implementation) right at that brace. No free to forget, no collector to wait for. The brace is the memory management.

Note

Types like i32, bool, char, and shared references implement Copy: assignment duplicates them, because copying a few bytes on the stack is trivially cheap and there's no heap resource to double-free. Heap-owning types like String and Vec<T> are not Copy — and what assignment does to them instead is the subject of chapter 3.

Where do references fit?

A reference — &s — is just an address, a single machine word on the stack pointing at data that lives somewhere else. It doesn't own anything; dropping a reference frees nothing. That's exactly what makes it dangerous in other systems languages: the address can outlive the thing it points to. The workbench frame that held the data pops, or the storeroom shelf is returned, and the address now points at ghosts. C calls this a dangling pointer and lets it happen. Rust's entire lifetime apparatus exists to make it impossible.

From the Bench

A good habit, starting now: whenever you see a variable, ask “stack or heap? owner or borrower?” Every lifetime puzzle in this book unravels once you can answer those two questions for each name in the code.

So what did we actually gain?

We now know the terrain. Stack: fast, fixed-size, frame-scoped. Heap: flexible, but someone must free it. Rust's answer to “who frees?” is the owner's closing brace — a compile-time answer where other languages give a runtime one. References are non-owning addresses into either region, and keeping them honest is the borrow checker's whole job. Next: ownership, the rule that makes the closing brace trustworthy.

Challenges

  1. A Vec<i32> with 1,000 elements: how many bytes does it occupy on the stack? (Hint: the same as an empty one.)
  2. Why can't a value whose size is only known at run time — like the string contents typed by a user — live directly on the stack?
  3. In C, free(p); free(p); is catastrophic. Explain, in workshop terms, why “exactly one owner” makes a double-free structurally impossible rather than merely discouraged.
