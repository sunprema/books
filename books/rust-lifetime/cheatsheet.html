Quick ReferenceThe Lifetime Cheatsheet

The 30-second version

A lifetime is a region of code where a borrow is valid. The rule: the data's region must cover the borrow's region. Annotations ('a) describe relationships between regions — they never extend anything. Borrows end at their last use (NLL), not at the closing brace. When stuck: change who owns, not what's annotated.

Ownership & borrow rules

① One owner per value. ② Assignment/passing moves ownership (Copy types copy instead). ③ Owner's scope end = Drop. ④ Any number of &T xor exactly one &mut T. ⑤ No borrow outlives its data.

Reading signatures

fn f<'a>(x: &'a str) -> &'a str          // result borrows from x
fn f<'a>(x: &'a str, y: &str) -> &'a str // …from x only; y is free
fn f<'a>(x: &'a str, y: &'a str) -> &'a str // …good only in x∩y overlap
fn f(x: &str) -> String                  // no strings attached (owned)

'b: 'a = “'b outlives 'a” (region containment). T: 'a = “T holds no borrow shorter than 'a”.

Elision — the three rules (functions only)

#Rule
1Each elided input lifetime → fresh parameter
2Exactly one input lifetime → it flows to all outputs
3&self/&mut self present → self's lifetime flows to outputs

No rule applies → E0106, annotate by hand. Structs/enums never elide. const/static refs default to 'static. Box<dyn Trait> defaults to + 'static; &'a dyn Trait to + 'a. '_ = “elide here, visibly.”

Structs that borrow

struct Excerpt<'a> { part: &'a str }   // can't outlive the source
impl<'a> Excerpt<'a> {
    fn part(&self) -> &'a str { self.part }  // outlives &self borrow
}

A struct holding a borrow is a borrow. Lifetime params are infectious upward. Borrow for zero-copy views; own for stored data. No self-referential structs in safe Rust — store indices instead.

'static — two meanings

FormMeansTypical
&'static T Reference valid for the whole program String literals, static items, Box::leak
T: 'static T contains no non-'static borrows (self-contained) thread::spawn, async tasks — fix with move/Arc/scoped threads

Pasting 'static over a lifetime error treats the alarm, not the fire.

Errors decoded

CodeSaysUsual fix
E0106Missing lifetime specifierName which input the output borrows from
E0597Value does not live long enoughShorten the borrow's use, or move ownership out
E0515Returns reference to localReturn the owned value instead
E0502Mutable + shared borrows overlapReorder so the shared borrow's last use comes first
E0382Use after moveBorrow instead of move, or clone() deliberately

“Borrow later used here” marks what keeps the loan alive — move or delete that use and the loan shrinks (NLL: loans end at last use).

Advanced corners

HRTB: for<'a> Fn(&'a str) -> &'a str = must work for every region.
2024 edition: -> impl Trait captures all in-scope lifetimes by default; release with + use<>.
NLL since 1.63 everywhere; Polonius (origins/loans model) still experimental — when it false-alarms, restructure or use entry-style APIs.

Decision guide — “the checker said no”

① Find the “later used here” — can the use move earlier? ② Should this be owned instead of borrowed (return String, not &str)? ③ Is elision's guess wrong — de-elide and check. ④ Crossing threads/tasks — move, Arc, or thread::scope. ⑤ Two mutable paths through one value — split the struct, or take the fields apart first. ⑥ Only then consider Rc/RefCell — runtime counting is the escape hatch, not the default.
