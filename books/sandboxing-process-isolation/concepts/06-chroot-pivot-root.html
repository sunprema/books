Chapter 06
Rebuilding the world

chroot & pivot_root: Rebuilding the Prisoner's World

We've walled off what the inmate can see and confiscated their keys. Now we rebuild the ground under their feet: the filesystem. The inmate should open / and find only their world — their libraries, their files — with no path leading back to the host's real root. The old tool for this is chroot(). It is also, I must warn you, one of the most misunderstood locks in the building. People think it's a cell. It's a painted line on the floor.

Why chroot is not a cell

chroot() changes what a process calls /. That's all it does. Two things it deliberately does not do, and both are escape hatches:

  • It does not change your current working directory. If you were standing outside the new root when you called it, you're still standing there.
  • It does not close file descriptors opened before the call. An open handle to a directory outside the jail is a rope over the wall.
Case file · classic technique · not a CVE

The fchdir walk-out

A process holding CAP_SYS_CHROOT saves a file descriptor to a directory, chroots into a subdirectory, then uses that saved fd to fchdir() back outside the new root — and simply walks ../../.. up to the real filesystem root. The jail never closed the door behind it.

int saved = open("/", O_RDONLY | O_DIRECTORY); // a rope, kept from before
mkdir("jail", 0755);
chroot("jail");        // "/" now means .../jail — but cwd didn't move
fchdir(saved);          // step back outside using the old handle
for (int i = 0; i < 64; i++) chdir(".."); // climb to the real root
chroot(".");           // re-root at the top: escaped
This is the lesson that humbles every new guard: chroot was never designed to hold a hostile inmate. It was built in the 1970s to give a build or a service a tidy view of the world, back when the process wasn't your enemy. A program that wants to jail itself honestly can — drop CAP_SYS_CHROOT, chdir into the new root, close stray fds — but you cannot use it to hold code that's actively trying to leave.

pivot_root: swap the ground, then remove it

This is why real containers don't rely on chroot. They use pivot_root(2) inside a private mount namespace (Chapter 2's mount wall). Instead of merely renaming /, pivot_root swaps the process's entire root mount for the container's rootfs and moves the old root aside — so you can then umount it and leave no mount, and no path, back to the host tree. There's no rope because the old world is unmounted entirely.

Key idea pivot_root plus a mount namespace gives a filesystem boundary that actually holds — but it is only the filesystem. Network, IPC, syscalls, and privilege still need their own walls from the previous chapters. Rebuilding the prisoner's world is one job of many, which is exactly why the next chapter assembles all of them into a single cell.
