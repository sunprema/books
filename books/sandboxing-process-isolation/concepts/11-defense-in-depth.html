Defense in Depth: Choosing Your Layers

We've walked the whole facility — the walls, the rations, the confiscated keys, the guard at the window, the rebuilt ground, the assembled cell, the escapes that worked, and the stronger cells with their own floors. Now the warden's real job: deciding which of these to build for a given inmate. Not every process needs a microVM, and not every microVM is enough. This last chapter is the decision, not another mechanism.

The layers, stacked

Read from the inside out, each layer stops a different threat, and each has a different TCB:

LayerStopsDoesn't stop
NamespacesSeeing other processes, mounts, network, hostsOver-privilege inside the cell
CgroupsResource exhaustion / riots (DoS)Escape of any kind
CapabilitiesUsing root's dangerous powersAnything a kept capability allows
SeccompReaching most of the syscall surfacePath-based policy; kept syscalls
pivot_root + mount nsReaching the host filesystem treeNon-filesystem escapes
microVM / gVisorA host-kernel bug being an escapeHypervisor / CPU side-channel bugs
WASM + capabilitiesAny authority you didn't explicitly grantHost-runtime bugs; over-broad grants
Notice there's no single row that stops everything. That's not a gap in the table — it's the entire thesis. Every layer has a "doesn't stop" column, and defense in depth is the art of arranging things so that one layer's blind spot sits behind another layer's wall. A kernel bug walks through your seccomp filter — and lands against a hypervisor boundary. That's depth.

A warden's decision guide

Match the cell to the inmate. Roughly, in rising order of distrust:

The inmateBuild them
First-party code you wrote and trustA hardened container: namespaces + cgroups + dropped caps + seccomp + rootless. Depth is cheap; use it anyway.
Third-party code, semi-trustedAll of the above, tightened — minimal capability allowlist, strict seccomp profile, read-only rootfs, no host mounts.
Genuinely untrusted / AI-agent-generated code that needs a real OSA separate floor: gVisor or a Firecracker microVM per task. Don't bet the host kernel on it.
Untrusted code that only needs to compute over dataA WASM + capability sandbox: grant nothing by default, hand over only the exact resources needed.
Key idea Two rules decide everything. Least privilege: hand over the minimum authority — an empty capability set, a preopened directory, nothing more. Defense in depth: never trust one boundary, because history (Chapter 8) shows every single kind has been bypassed. Stack independent layers whose TCBs don't overlap, so no one bug is fatal.
Watch out Depth without understanding is a costume. Adding layers you can't explain gives false confidence — the most dangerous kind. For every boundary in your design, be able to say in one sentence what threat it stops and what its TCB is. If you can't, you don't have defense in depth; you have a pile of walls and a hope.
That's the tour. Keep the escape files on your shelf, keep your runtimes patched, and treat every inmate as if they're smarter and more patient than you — because eventually one will be. Build the cell empty, admit the inmate last, and never trust a single wall. The facility holds because it's many walls, each watching a different way out. Now go run a tight block.

Sources: gVisor Security Model; OCI Runtime Spec v1.1; Firecracker (NSDI '20); WASI Overview.
