The visiting rules

Seccomp-BPF: The Visiting Rules

Every request an inmate makes of the outside world goes through one window: the system call. Open a file, send a packet, mount a disk, fork a child — all of it is a syscall, the process asking the kernel to act on its behalf. seccomp (secure computing) is the guard at that window, holding the rulebook. It inspects each syscall as it's made and decides: allowed, denied, or worse. These are the visiting rules, and unlike capabilities they can filter by the exact request, not just the class of key.

Two rulebooks: strict and filter

seccomp comes in two modes:

ModeThe rule
STRICTSolitary confinement. Only read, write, _exit, and sigreturn are permitted. Any other syscall → instant SIGKILL.
FILTERThe real workhorse. You supply a classic BPF program that inspects each syscall and returns a verdict. This is "seccomp-BPF."

In FILTER mode the guard's rulebook is a little BPF program that reads a struct seccomp_data — the syscall number and up to six argument registers — and returns an action. The actions, in priority order, are:

SCMP_ACT_*What the guard does
KILLKill the thread or the whole process on sight
TRAPRaise SIGSYS — deny, but let the process catch it
ERRNOFail the call with a chosen error (e.g. EPERM) — the inmate thinks the door is just locked
NOTIFYWake a supervisor to decide — the powerful new one, below
ALLOWLet it through
There's an art to choosing KILL versus ERRNO. Kill is honest and loud — the inmate slams into the wall and dies, and you get a clean log entry. Errno is quieter: the syscall just fails as if permission were denied, and well-behaved software degrades gracefully instead of crashing. Docker's default profile leans on errno for exactly that reason. Loud walls catch attackers; quiet walls keep friends running.

Try this

The rulebook is code. Here's a filter that defaults to "deny with EPERM" and allows only write — the guard reading the syscall number:

/* classic BPF: inspect seccomp_data.nr, allow __NR_write, else EPERM */
struct sock_filter filter[] = {
  BPF_STMT(BPF_LD | BPF_W | BPF_ABS,
           offsetof(struct seccomp_data, nr)),
  BPF_JUMP(BPF_JMP | BPF_JEQ | BPF_K, __NR_write, 0, 1),
  BPF_STMT(BPF_RET | BPF_K, SECCOMP_RET_ALLOW),
  BPF_STMT(BPF_RET | BPF_K, SECCOMP_RET_ERRNO | (EPERM & 0xffff)),
};

Real code uses libseccomp so you write seccomp_rule_add(ctx, SCMP_ACT_ALLOW, SCMP_SYS(write), 0) instead of hand-assembling BPF — but this is what it compiles down to.

The supervisor at the window: user notification

The cleverest rule is SCMP_ACT_NOTIFY, the user notification mechanism (since Linux 5.0). When a filtered syscall fires, the inmate is frozen in the kernel and a separate supervisor process is woken up, handed the details, and asked to decide — it can even perform the action on the inmate's behalf and return the result. This is how a rootless container lets an unprivileged inmate "mount" something: the inmate isn't trusted to do it, so a trusted supervisor at the window does it for them, under supervision.

Watch out seccomp filters on the syscall number and register values — it cannot safely dereference a pointer argument, so it can't read the path string in an open(). A filter that tried would race the inmate swapping the string underneath it (a TOCTOU bug). So seccomp reduces the syscall surface; it does not do path-based policy. Pair it with the other walls. Docker's default profile, for reference, blocks roughly 44 of 300+ syscalls this way — keyctl, kexec_load, bpf, and other rarely-needed doors.

Sources: seccomp(2); seccomp_unotify(2); Docker seccomp profiles.
