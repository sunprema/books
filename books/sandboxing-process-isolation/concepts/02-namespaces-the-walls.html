Chapter 02
The walls

Namespaces: The Walls of the Facility

Walk with me down the main corridor. Each cell looks, from the inside, like the whole world — the inmate sees a process list, a filesystem, a network, a hostname, and believes that's everything there is. That illusion is a namespace: the kernel partitions one global resource so the processes inside a namespace see only their own private instance of it. Namespaces are the walls. They don't take power away from the inmate; they just shrink what the inmate can see to the size of a cell.

Eight walls, eight things to hide

There are eight namespace types today. Each hides exactly one kind of global resource:

NamespaceCLONE flagWhat it walls off
MountCLONE_NEWNSThe set of mount points — the filesystem tree the process can see
PIDCLONE_NEWPIDThe process-ID number space; inside, your init is PID 1
NetworkCLONE_NEWNETNetwork devices, IP stacks, ports, routing tables, firewall rules
IPCCLONE_NEWIPCSystem V IPC objects and POSIX message queues
UTSCLONE_NEWUTSThe hostname and NIS domain name
UserCLONE_NEWUSERUID/GID mappings — and capabilities (this one is special)
CgroupCLONE_NEWCGROUPThe cgroup root directory the process sees
TimeCLONE_NEWTIMEThe boot and monotonic clocks
The mount namespace was the first wall we ever built — Linux 2.4.19, back in 2002. That's why the flag is just NEWNS, no letter after it: when it was the only namespace, it didn't need a surname. The others arrived over two decades: user got finished in 3.8, cgroup in 4.6, and time — the newest recruit — in 5.6.

Three tools build every wall

You only need three syscalls to run this whole cell block:

clone(2)Fork a new process directly into fresh namespaces, one per CLONE_NEW* flag. This is how a runtime builds a cell before the inmate arrives.
unshare(2)Move the calling process into new namespaces without forking — "wall off where I'm standing."
setns(2)Join an existing namespace via a file descriptor (e.g. /proc/[pid]/ns/net) — how a guard steps into a cell to check on things.

Try this

Build yourself a cell and look around. Inside, you're PID 1 with your own hostname and an empty network:

$ unshare --pid --net --uts --ipc --mount --fork --user --map-root-user bash
# hostname jail && hostname
jail
# ps -e
  PID TTY          TIME CMD
    1 pts/0    00:00:00 bash
# ip link
1: lo: <LOOPBACK> mtu 65536 ... state DOWN

One process. One hostname you set. A network with nothing but a downed loopback. The rest of the machine still exists — you just can't see it from in here.

The user namespace is the master wall

One namespace changes the game: the user namespace. Inside it, a process can be root — UID 0 with a full capability set — while being an unprivileged nobody on the host. The kernel maps the inner UID 0 to some harmless outer UID like 100000. This is the foundation of rootless containers: the inmate feels omnipotent inside the cell, yet holds no real authority in the yard.

Watch out A wall hides a view; it does not remove power. A process can still be dangerously over-privileged inside its own namespaces — able to load kernel modules, mount filesystems, or trace neighbors — if you haven't also stripped its capabilities and syscalls. Namespaces are necessary, never sufficient. The next three chapters are the locks that go on these walls.

Sources: namespaces(7); unshare(2); user_namespaces(7).
