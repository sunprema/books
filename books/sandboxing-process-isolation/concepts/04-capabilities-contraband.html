Contraband

Capabilities: Confiscating root's Master Key

In the old days there was one master key — root, UID 0 — and it opened every door in the facility: mount drives, load kernel modules, rewrite the network, read any file, trace any process. Hand an inmate that key and containment is a joke. Linux capabilities are the locksmith's answer: they cut that single master key into 41 small keys, each opening one class of door. Now you can confiscate the dangerous ones and let the inmate keep only the harmless few. Capabilities are the contraband we search for at every gate.

41 keys, numbered 0 to 40

A capability is one unit of what used to be root's power. The highest one defined today is CAP_CHECKPOINT_RESTORE, number 40 (added in Linux 5.9), so CAP_LAST_CAP is 40 and there are 41 in total (0 through 40). A handful of them are what a wise warden treats as pure contraband:

CapabilityThe door it opens — why it's dangerous
CAP_SYS_ADMINThe new master key. A catch-all: mount, namespace operations, and dozens more. So overloaded it is effectively root. never grant
CAP_SYS_PTRACETrace and inspect any process — read and write another's memory. A window into every other cell.
CAP_NET_ADMINFull control of network configuration — interfaces, routes, firewall rules.
CAP_DAC_OVERRIDEBypass all file read/write/execute permission checks. The skeleton key to the filesystem.
CAP_SYS_CHROOTCall chroot() — and, as Chapter 6 shows, a poorly done chroot plus this key is an escape.
If you remember one rule from this chapter, make it this: CAP_SYS_ADMIN is not a capability, it's a confession. Grant it to an untrusted workload and you've handed back the master key you spent all this effort cutting up. Any config that gives it away deserves a second look and usually a firm no.

Five pockets to search

The kernel doesn't track one set of keys per process — it tracks five, and confiscation only works if you empty the right ones. Search all the pockets:

SetWhat it means
PermittedThe keys the thread is allowed to pick up and use
EffectiveThe keys currently in hand — what the kernel checks right now
InheritableKeys that can survive across execve() into a new program
BoundingThe ceiling — a key never in this set can never be gained, no matter what
AmbientInheritable keys for unprivileged programs across execve()
Watch out Dropping a key from the effective set but leaving it in the permitted or inheritable set is like taking the knife out of an inmate's hand while leaving it in their pocket — they can pick it back up. Real confiscation means dropping it from the bounding set, which puts a hard ceiling on what can ever be regained.

Try this

Inspect a process's current keys, then run a payload with the dangerous ones stripped from the bounding set:

$ grep Cap /proc/self/status
CapPrm: 0000000000000000
CapEff: 0000000000000000
CapBnd: 000001ffffffffff   # all 41 bits set = full ceiling

$ capsh --print | grep 'Bounding'      # decode them by name
# capsh --drop=cap_sys_admin,cap_sys_ptrace,cap_net_admin \
        --drop=cap_dac_override,cap_sys_chroot -- -c 'id; mount -t tmpfs t /mnt'
uid=0(root) ...
mount: permission denied           # CAP_SYS_ADMIN is gone — good

Root by UID, but neutered by capabilities. That gap — "root that can't actually do the dangerous things" — is exactly what you want an inmate to hold.

Key idea Least privilege, made concrete: start from an empty capability set and add back only the specific keys the job proves it needs — never start from full root and try to remember everything to remove. Containers do exactly this; a well-run runtime drops the bounding set to a short allowlist before execve.

Sources: capabilities(7).
