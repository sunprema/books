Intake: Who Are You Locking Up?

Welcome to the facility. Before I hand you a ring of keys and walk you past the cell blocks, we sit down at the intake desk and answer one question, because every wall we pour after this depends on it: who, exactly, are we locking up, and what are they trying to do? A sandbox with no threat model is just decoration — an expensive fence with a gate somebody left open.

In this facility the inmates are processes: a Python script an AI agent just wrote, a build step from a pull request you've never read, a customer's uploaded plugin. You did not vet them. You must assume at least one is clever, patient, and actively looking for a way out. That assumption is not paranoia; it is the job.

I've run this block a long time, and here's what I tell every new guard on their first day: the inmate only has to find one unlocked door. You have to lock all of them. That asymmetry is the whole business. It's why we don't rely on a single wall — we rely on many, each watching a different way out.

The three intake questions

A threat model is just three honest answers. Skip them and you'll harden the wrong wall while the real door stands open.

QuestionIn prison terms
What does the attacker control?What can the inmate do from inside the cell? Run arbitrary code? Only submit data? Do they arrive already holding a knife (a known kernel bug)?
What do they want?Escape to the host? Read the cell next door (another tenant's secrets)? Just cause a riot (denial of service)?
Which boundary must hold?The one thing that, if it breaks, the game is over. That's your perimeter wall — everything else buys you time and depth.

Your Trusted Computing Base is the guard staff

Every isolation scheme trusts something to enforce it. That set of components — the code and hardware that must be correct for the wall to hold — is the Trusted Computing Base (TCB). The smaller it is, the fewer things can betray you.

Key idea For an ordinary Linux container, the TCB includes the entire shared host kernel. Millions of lines. So any kernel local-privilege-escalation bug — like Dirty COW, which we'll visit in Chapter 8 — is a potential escape route, no matter how tidy your namespaces are. Shrinking that TCB is the whole reason microVMs and WebAssembly exist (Chapters 9–10).

Two principles, carved over the gate

Least privilege. An inmate gets exactly the authority the job requires and not one capability more. A process that only needs to read one directory and compute a number should not be able to open a socket, mount a filesystem, or trace another process.

Defense in depth. Never trust one wall. Stack independent boundaries so that defeating one lands the attacker in another, not in the yard. When a bug breaks your seccomp filter, the user namespace should still be standing; when the kernel itself falls, the hardware boundary should still hold.

Threat model Throughout this book, assume the worst realistic inmate: arbitrary code execution inside the sandbox, actively probing for escape, possibly armed with a public exploit for an unpatched kernel. If your design survives that, the friendlier cases come free.
