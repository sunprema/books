Chapter 10
The other kind of cell

WebAssembly & Capability-Based Security

Everything so far has been about containing an inmate who arrives with the full run of the machine and must be walled back. WebAssembly inverts the whole philosophy. A WASM inmate is born into a padded room with no doors at all — no memory outside its own, no syscalls, no filesystem, no network. It cannot ask for anything the host didn't already hand it. This isn't a cell we lock; it's a cell with nothing in it until we choose to add something. That inversion is the deepest idea in this book.

The linear-memory sandbox

A WebAssembly module runs against a single linear memory: one bounds-checked array of bytes. It has no ambient access to host memory, no raw pointers into the process, and no way to issue a syscall. The only things it can call are functions the host explicitly imports into it. If the host imports nothing, the module can compute and return values — and nothing else. The wall isn't enforced by the kernel at all; it's enforced by the shape of the bytecode and the runtime that checks every memory access.

Think about how different this is from a container. With a container I start with an inmate who can do everything and I spend five chapters taking things away — and one forgotten capability is a breakout. With WASM I start with an inmate who can do nothing and I add back exactly what the job needs. A mistake in the first model grants power by accident. A mistake in the second only ever withholds it. I know which side of that I'd rather guard.

WASI: authority you can only be handed

Of course, most useful code needs some access — to read a file, say. That's WASI, the WebAssembly System Interface, and its defining rule is no ambient authority. A WASI module cannot open an arbitrary path or dial an arbitrary socket. The host must explicitly grant each capability at instantiation — most classically a preopened directory. Hand the module a preopened handle to /work and it may operate inside /work and nowhere else. There is no open("/etc/passwd") that could ever succeed, because the module was never given a handle from which that path is reachable.

Try this

Authority is a command-line flag. The module gets only the directory you preopen:

$ wasmtime run --dir=./work untrusted.wasm
# inside the module, "/work/out.txt" works;
# "/etc/passwd" fails — no capability was ever granted for it.
$ wasmtime run untrusted.wasm      # no --dir at all
# the module has NO filesystem access whatsoever. The default is nothing.

Compare that to a container, where the default is "root inside a full OS" and safety is subtraction. Here the default is emptiness and safety is addition.

The object-capability model

This is a concrete instance of an old, elegant idea: object-capability security. Authority is conveyed only by possessing an unforgeable reference — a capability — rather than by who you are plus an ambient permission check. In the Unix model, any process can name /etc/passwd and the kernel then checks whether it's allowed; the authority is ambient, floating in the environment. In the capability model, if you don't hold a reference to a thing, it does not exist for you — there is nothing to name and nothing to check. A component holds no more authority than it was handed.

Key idea WASI Preview 1 handed capabilities as a flat set of preopened directory fds; Preview 2 rebuilds WASI on the Component Model, where every external resource is a typed, unforgeable handle in the type system — finer-grained, and harder to grant by accident. As of 2026 Preview 2 (WASI 0.2) is stable and Preview 3 is adding native async. Treat exact version/tooling maturity as volatile and check current docs before you build.
Watch out WASM sandboxes the module's memory and imports — but a bug in the host runtime, or a too-generous grant (preopening / "just to make it work"), reopens the hole. The model is only as good as the smallest set of capabilities you can bear to hand over. Grant like a miser.
