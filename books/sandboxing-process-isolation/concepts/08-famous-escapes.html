Chapter 08
The escape files

Notable Escapes: A History of Breakouts

Every warden keeps a shelf of case files — the escapes that worked. Not to admire the escapees, exactly, but because each breakout teaches a lesson no design document ever could: this is where the boundary actually was, and here is how it failed. Read these with respect. The people who found them were clever, and the fixes that followed are why your cells are stronger today. Notice the pattern: almost none of these are namespace bugs. They exploit a trust assumption — the shared kernel, a host binary, a mount race.

Case file · runc breakout

The one where the inmate overwrote the guard CVE-2019-5736

A malicious container pointed one of its entry binaries at #!/proc/self/exe — a handle to the running runc itself. When runc re-executed it, the attacker wrote through /proc/self/exe and clobbered the host's runc binary, gaining root on the host the next time any container started. The inmate didn't climb the wall; they rewrote the guard who opens the gate.

Case file · cgroups v1

The forgotten trapdoor in the quartermaster's office CVE-2022-0492

The cgroups v1 release_agent mechanism runs a host program when a cgroup empties — and the kernel's cgroup_release_agent_write() forgot to check for CAP_SYS_ADMIN. A container that made a new user + cgroup namespace, mounted cgroupfs, and set notify_on_release + release_agent could make the host kernel run an attacker binary as root. Fixed around 5.17; this trapdoor simply doesn't exist in cgroups v2 (Chapter 3) — a good reason to be on v2.

Case file · kernel LPE · the famous one

Dirty COW CVE-2016-5195

A race condition in copy-on-write handling of private read-only memory mappings let an unprivileged local user gain write access to memory that should have been read-only, and escalate to root. It lived in the kernel from 2.6.22 (2007) until it was fixed in 2016. Because every container shares that one kernel, a shared-kernel bug like this is a container escape by definition — the exact TCB problem Chapter 1 warned about.

Dirty COW is the case I make new guards study first. There was nothing wrong with your walls, your rations, your confiscated keys. The floor everyone was standing on — the shared kernel — had a crack in it for nine years. No arrangement of container primitives can fix a hole in the thing enforcing all of them. That single realization is what the last two chapters of this book are about.
Case file · runc · "Leaky Vessels"

The rope left dangling CVE-2024-21626

An internal file-descriptor leak in runc let a container's working directory (WORKDIR) resolve into the host filesystem namespace — a leaked handle pointing outside the cell, the modern cousin of Chapter 6's chroot rope. It enabled escape and host-file overwrite. Affected runc ≤ 1.1.11; fixed in runc 1.1.12. Keep your runtime patched.

Case file · NVIDIA Container Toolkit · relevant to AI workloads

The GPU gate that mounted too much CVE-2024-0132

A TOCTOU race in how the NVIDIA Container Toolkit mounts files let a crafted container image mount host directories into itself and escape to the host (CVSS 9.0). This one matters especially if you run untrusted AI workloads that need a GPU — the very glue that gives a cell a GPU became the way out. Affected the toolkit ≤ 1.16.1; fixed in 1.16.2.

Key idea Read the shelf and a pattern emerges: escapes rarely defeat a namespace head-on. They exploit a shared component you trusted — the host kernel (Dirty COW), a host binary (runc), a privileged helper (NVIDIA toolkit), or a legacy mechanism (cgroup-v1 release_agent). The defense isn't a better wall of the same kind; it's a different kind of boundary that doesn't share that trust. That is exactly the leap to microVMs and WebAssembly.

Sources: CVE-2019-5736; CVE-2022-0492; CVE-2016-5195; CVE-2024-21626; CVE-2024-0132 (Wiz).
