06 Delegation

Delegating to Subagents

The Agent tool spawns a subagent in its own context window. It works through a task on its own, then returns a single text result. The parent never sees the subagent's intermediate steps — just that final answer. It runs free.

Why delegate? Two reasons. A noisy search ("read 40 files, tell me where auth lives") would flood the main context — a subagent absorbs the dumps and hands back only the conclusion. And independent work can run in parallel, several subagents at once.

▣
Cap the work

To bound how long a subagent runs, set maxTurns in the subagent definition. The parent sees only the result, so a runaway subagent is invisible until it returns.

Forked subagents

The same tool also launches a fork when fork mode is on. Unlike a named subagent that starts fresh, a fork inherits the full parent conversation, always runs in the background, and still surfaces permission prompts in your terminal. Use it when the task needs everything the main thread already knows.

What tools can a subagent use?

It depends on two frontmatter fields, tools and disallowedTools:

Fields setSubagent gets…
neitherevery tool the parent has
tools onlyonly the listed tools
disallowedTools onlyall parent tools except those
bothdisallowedTools wins — a tool in both is removed
★
Launching is free; the work isn't

Spawning a subagent never prompts on its own. Each tool call the subagent makes is checked against your permission rules as it runs. To sandbox one, narrow its tools, leave Bash off the list, or set deny rules.

Permission prompts, foreground vs. background

Foreground

Shows the same prompts you'd see in the main conversation, at the moment each call happens.

Background

As of v2.1.186, prompts surface in your main session, named by subagent; Esc denies that one call without stopping it.

⚠
Before v2.1.186

Older background subagents auto-denied any call that would have prompted and carried on without that tool — so a silent permission wall, not a question.
