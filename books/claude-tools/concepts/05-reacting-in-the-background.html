05 Watch & React

Reacting in the Background

Most tools answer a question once. Monitor is different: it watches something over time and pokes Claude when it changes — without pausing your conversation. Think of it as Claude's peripheral vision. It asks first.

Point it at almost anything that produces a stream of events:

  • Tail a log file and flag errors as they appear
  • Poll a PR or CI job and report when its status changes
  • Watch a directory for file changes
  • Track output from any long-running script
  • Connect to a WebSocket feed and react to each message

For most watches, Claude writes a small script, runs it in the background, and receives each output line as it arrives. You keep working in the same session; Claude interjects when an event fires. Stop a monitor by asking Claude to cancel it, or by ending the session.

★
This is what powers a self-paced loop

When you ask Claude to "let me know when CI goes green," a persistent Monitor is the wake signal — events arrive as notifications and resume the work immediately, instead of Claude blindly polling on a timer.

A command watch streams lines; a WebSocket watch streams messages.

The WebSocket source

When a server already pushes events, Claude skips the polling script and connects directly (v2.1.195+). It takes a ws input instead of command:

  • Text messages → one event each
  • Binary → a placeholder like [binary frame, 512 bytes]
  • Over 1 MiB → the watch ends (subscribe to a filtered feed)
  • Socket close → the watch ends with the close code

The URL must be ws:// or wss://; opening it prompts for approval, and Claude Code denies private, link-local, or cloud-metadata addresses.

⚠
Where Monitor isn't available

Requires v2.1.98+. Not on Amazon Bedrock, Google Vertex AI, or Microsoft Foundry — and disabled when DISABLE_TELEMETRY or CLAUDE_CODE_DISABLE_NONESSENTIAL_TRAFFIC is set. Command watches obey your Bash permission rules.
