Concept 01 · The Why

The Big Idea

Most code does something. A DSL lets you describe something, and then machinery turns the description into the doing. Spark is the machinery.

Let's start with a feeling you already have. When you write a recipe, you don't write the chemistry of caramelization — you write "sauté the onions until golden." That sentence is a tiny domain-specific language: a vocabulary tuned so tightly to its domain that saying what you want is the work. The cook fills in the how. A software DSL is exactly this trick. You invent a small vocabulary — field, tool, route — and the surrounding program reads your description and does the heavy lifting.

Here's the thing worth pausing on: in a DSL you write nouns and adjectives, not verbs and loops. You declare what is true about your validator, your API, your agent — and you let the framework figure out the control flow. That's what "declarative" means, and it's why a good DSL reads like a spec sheet instead of a script.

A 20-second taste

This is a real Spark DSL (we build it for real in Chapter 6). Read it like English:

describe, don't dodefmodule MyApp.PersonValidator do
  use MyLibrary.Validator

  fields do
    required [:name]
    field :name, :string

    field :email, :string do
      check &String.contains?(&1, "@")
      transform &String.trim/1
    end
  end
end

Nowhere did we write a parser, a loop, or an if. We declared two fields and a rule. Calling MyApp.PersonValidator.validate(...) just works — because Spark turned that declaration into code at compile time.

Imperative vs. declarative

Hand-rolled

You write the validation loop, the error collection, the trimming — and re-write it slightly differently in every project. The intent is buried in mechanism.

DSL

You state the fields and rules. The mechanism is written once, in the library, and is shared. Intent is the whole file.

So what is Spark?

One sentence: Spark is a framework for building declarative DSLs in Elixir from plain data structures, with first-class tooling baked in. You describe your language as a few structs — "here are my entities, here's how they nest" — and in return Spark hands you, for free:

  • Autocomplete & inline docs in the editor, for your DSL, with zero extra work.
  • Schema validation with friendly compile-time error messages when someone misuses it.
  • Generated documentation for every option and entity.
  • Introspection — clean functions to read a module's DSL back out at runtime (the hero of Chapter 5).
  • Extensibility — anyone can write an extension that adds to your DSL.
⚡
Where you've already met it

If you've seen the Ash Framework, you've seen Spark — every Ash resource, every attributes do … end block, is a Spark DSL. Spark is the engine; Ash is its most famous passenger. It's battle-tested across thousands of real DSL definitions.

🤖
Why this matters for agents

Hold one idea in your head for the whole book: because a Spark DSL is data you can read back out, a program — or a language model — can ask a module "what tools do you offer? what are their arguments?" and get a precise, machine-readable answer. That is exactly the shape of problem LLM agents have. We cash this in at Chapter 7.
