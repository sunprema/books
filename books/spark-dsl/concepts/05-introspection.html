Introspection & InfoGenerator

We've put data into the DSL. The whole payoff is getting clean, typed data back out — at runtime, fast, and without poking at internals. This is the chapter the agentic stuff stands on.

A declaration is only useful if something can read it. Spark gives you two ways. The low-level way is Spark.Dsl.Extension.get_entities/2 — handy, but raw. The civilized way is an Info module: you declare one with Spark.InfoGenerator and it writes a tidy set of accessor functions for you, complete with docs and caching.

define an info moduledefmodule MyLibrary.Validator.Info do
  use Spark.InfoGenerator,
    extension: MyLibrary.Validator.Dsl,
    sections: [:fields]
end

From those four lines, Spark generates functions named after your sections and options:

iexiex> Info.fields(MyApp.PersonValidator)
[%Field{name: :name, type: :string, …},
 %Field{name: :email, type: :string, …}]

# {:ok, value} / :error form
iex> Info.fields_required(MyApp.PersonValidator)
{:ok, [:name]}

# bang form for options you know are set
iex> Info.fields_required!(MyApp.PersonValidator)
[:name]

The naming pattern

Once you see the rule, you can predict every generated function:

GeneratedReturns
<section>(mod)the section's entities
<section>_<opt>(mod){:ok, v} or :error
<section>_<opt>!(mod)v or raises
⚡
It accepts state too

A lovely detail: these accessors take either a compiled module or the raw dsl_state map. That's why verifiers (Chapter 4) can call Info.fields(dsl_state) mid-compile, before the module exists. One API, both worlds.

Why generate instead of just reading?

  • Type safety: named functions, not stringly-typed lookups, so typos fail at compile time.
  • Performance: the data is cached and optimized for runtime reads — the DSL traversal already happened at compile time.
  • Consistency: every Spark library exposes the same shape of API, so the Ash ecosystem feels uniform.
  • Docs: the accessors are documented automatically from your schema.
🤖
The hinge of the whole book

Stop and feel this. Info.fields(MyApp.PersonValidator) returns a precise, structured list describing what a module declared. Swap "fields" for "tools" and you have an agent asking a module "what can you do, and with what arguments?" — and getting a typed answer it can hand straight to an LLM. Introspection is the bridge from a declarative DSL to a callable tool catalog. That's Chapters 7 and 8.

🔎
The raw escape hatch

When you need it: Spark.Dsl.Extension.get_entities(mod, [:section]) and get_opt(mod, [:section], :opt, default) read the DSL directly. Prefer the Info module for everyday code; reach for these when writing the framework itself.
