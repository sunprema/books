Concept 02 · Anatomy

Entities, Sections & Extensions

Every Spark DSL — no matter how fancy — is built from exactly three nested kinds of brick. Learn these and you can read any of them.

Think of it like grammar. An entity is a word — a single thing you can say, like field :name, :string. A section is a clause that groups related words, like the fields do … end block. And an extension is the dictionary: the package that declares which sections and words exist at all. That's the whole hierarchy.

Extensions hold sections, sections hold entities, and entities can hold more entities.

1 · The Entity — a single constructor

An entity is one thing the user can declare. It has a name (the function they'll call), a target struct to build, positional args, and a schema of options. Here's the modern builder API from the official tutorial:

elixir · builder APIalias Spark.Builder.{Entity, Field, Section}

@field Entity.new(:field, Field,
  describe: "A field accepted by the validator",
  args: [:name, :type],
  schema: [
    Field.new(:name, :atom, required: true),
    Field.new(:type, {:one_of, [:integer, :string]},
      required: true),
    Field.new(:check, {:fun, 1}),
    Field.new(:transform, {:fun, 1})
  ]
)
|> Entity.build!()

Two dialects, same idea

You'll also see the older struct form, %Spark.Dsl.Entity{name: :field, target: Field, …}, all over the Ash codebase and older guides. The builder API (Entity.new |> Entity.build!) is the current recommended style; both compile to the same thing. Don't be thrown when you meet both.

2 · The Section — a named block

A section groups entities under a block name, and can carry its own block-level options via a schema. It can also nest other sections.

elixir@fields Section.new(:fields,
  describe: "Configure supported & required fields",
  schema: [
    Field.new(:required, {:list, :atom})
  ],
  entities: [@field]
)
|> Section.build!()

3 · The Extension — the package

An extension bundles your sections (and later, the transformers and verifiers from Chapter 4) into something reusable. use Spark.Dsl.Extension is what makes the words real:

elixirdefmodule MyLibrary.Validator.Dsl do
  # … @field and @fields defined above …
  use Spark.Dsl.Extension, sections: [@fields]
end

Then a thin DSL module turns the extension into something users can use:

elixirdefmodule MyLibrary.Validator do
  use Spark.Dsl,
    default_extensions: [extensions: [MyLibrary.Validator.Dsl]]
end
🔑
The mental model

Entity ⊂ Section ⊂ Extension ⊂ DSL module. Users use the DSL module; they type entities inside sections; Spark validates it all against your schemas. Everything else in this book is detail hung on this skeleton.
