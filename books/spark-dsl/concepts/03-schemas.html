Schemas & Spark.Options

A schema is a tiny type language. It's what turns "you typed the DSL wrong" into a friendly compile-time error instead of a 2 a.m. runtime surprise — and it writes your docs as a side effect.

Every entity and section carries a schema: a keyword list where each option declares its type, whether it's required, a default, and a doc string. Spark validates user input against it with Spark.Options (a close cousin of Elixir's NimbleOptions) at compile time. Get it wrong and the compiler points at exactly the offending line.

a schema, annotatedschema: [
  name: [
    type: :atom,
    required: true,
    doc: "The name of the field"
  ],
  type: [
    type: {:one_of, [:integer, :string]},
    required: true
  ],
  default: [
    type: {:or, [:string, :integer, :boolean]}
  ],
  check: [
    type: {:fun, 1},
    doc: "1-arity validator run after type check"
  ]
]
📚
Docs for free

Those doc: strings aren't decoration — Spark feeds them into generated documentation and editor tooltips. Write the schema well and you've written the manual. This is the "exceptional developer experience" the README keeps promising.

The type vocabulary

A working subset you'll reach for constantly:

TypeAccepts
:atom :string :integer :booleanthe obvious primitives
:pos_integer :non_neg_integerbounded integers
{:list, inner}a list of inner (e.g. {:list, :atom})
{:one_of, [..]}an enum — one of a fixed set
{:fun, arity}a function of that arity (captures & anon fns)
{:or, [t1, t2]}any of several types
{:struct, Mod}a struct of module Mod
:keyword_list {:map, k, v}nested config shapes
{:behaviour, Mod}a module implementing a behaviour

Per-option keys

KeyMeaning
type:one of the types above
required:true ⇒ compile error if omitted
default:value used when not supplied
doc:documentation & tooltip text
🛡️
Why functions-as-types matter

That {:fun, 1} type is sneakily powerful. It lets your DSL accept behaviour — a validation lambda, a transform — not just data. Spark does some quiet magic to let a user write check &String.contains?(&1, "@") right inside the DSL and have it captured correctly. Hand-rolled DSLs find this genuinely hard.

🤖
A schema is a machine-readable contract

Keep this in your back pocket for Chapter 7: a schema is precisely the kind of structured, typed description an LLM tool-calling API wants. A Spark schema can be translated almost mechanically into a JSON-Schema tool definition. Your validation rules become the model's argument spec.
