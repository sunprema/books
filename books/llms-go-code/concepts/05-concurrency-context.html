Chapter 5

Goroutines, Channels & Context: Auditing What Agents Get Wrong Most

Every other chapter so far has been about reading code that sits still on the page. This one is about code that moves. Goroutines run at the same time as each other, and the instant you have two things happening at once, you've traded a problem you can reason about linearly for one you have to reason about concurrently. That trade is exactly where agent-written Go tends to fall apart — not because concurrency is exotic, but because it's the one place where code can look completely correct, compile cleanly, pass a happy-path test, and still be broken.

The Big Picture: Cheap Threads, Not Free Threads

A goroutine is a function that runs concurrently with the rest of your program. You start one by putting go in front of a function call:

go fetchUser(id)

That's it. No thread pool to configure, no future to wrap. The Go runtime multiplexes goroutines onto a small number of OS threads, and each one starts with a stack of only a couple of kilobytes that grows as needed. That's why people say goroutines are “cheap” — you can genuinely spin up tens of thousands of them without blinking.

But cheap to create doesn't mean free to manage. Every goroutine you start is a commitment: something, somewhere, has to know when it's done and be able to make it stop. A goroutine with no owner and no exit path isn't lightweight — it's a slow leak. We'll come back to that, because it's the single most common concurrency mistake an agent makes.

Channels: How Goroutines Talk Without Sharing Memory

Go's proverb is “don't communicate by sharing memory; share memory by communicating.” A channel is a typed pipe that one goroutine can send values into and another can receive values from:

ch := make(chan int)   // unbuffered
go func() { ch <- 42 }()
result := <-ch                // blocks until someone sends

An unbuffered channel like the one above has no storage: a send blocks until a receive is ready for it, and vice versa. That rendezvous is a synchronization point — it's what makes unbuffered channels so useful for handing off work, and also what makes them dangerous when nobody shows up on the other end.

A buffered channel, make(chan int, 4), holds up to four values before a send blocks. Buffering decouples producer and consumer speed, but it doesn't remove the need for someone to eventually drain it — it just delays the moment the mistake becomes visible, which is exactly why agents like to reach for a buffer size as a band-aid instead of fixing the underlying ownership problem.

select lets a goroutine wait on multiple channel operations at once, proceeding with whichever is ready first:

select {
case v := <-work:
    process(v)
case <-ctx.Done():
    return ctx.Err()
}

That second case is the pattern to look for in every blocking select you review. Without an escape hatch, a select with only one channel case is just a blocking receive wearing a disguise.

context.Context: A Signal, Not a Kill Switch

Here's a fact worth sitting with: Go has no way to forcibly stop a goroutine from the outside. There's no goroutine.Kill(). Every goroutine has to check in on its own and decide to leave. context.Context is the standard mechanism for asking it to.

A context carries a cancellation signal (and optionally a deadline and request-scoped values) down a call tree. You derive one from a parent:

ctx, cancel := context.WithTimeout(parent, 5*time.Second)
defer cancel()

context.WithCancel, WithTimeout, and WithDeadline all return a derived context and a cancel function. Calling cancel closes an internal channel that ctx.Done() exposes; anything selecting on <-ctx.Done() wakes up and can check ctx.Err() to see whether it was cancellation or a timeout. That defer cancel() on the very next line is non-negotiable — skip it and the context (and its internal timer, if it's a timeout context) leaks until the deadline eventually fires on its own, or forever in the WithCancel case.

Two conventions matter enough that the Go team put them in the standard library's own style guide: context is always the first parameter, named ctx, never stored inside a struct; and a context only ever flows downward, from caller to callee, never sideways or back up. If you see an agent stash a context on a struct field so it can reuse it later, that's a tell — it means the agent doesn't understand that a context's lifetime is supposed to match a single call, not an object.

Key idea Cancellation in Go is cooperative. Calling cancel() doesn't stop anything by itself — it just closes a channel. The goroutine still has to be written to notice. A goroutine blocked on a plain channel receive with no ctx.Done() case will simply never notice, no matter how many times you cancel its context.

The Three Bugs Agents Write Most Often

Ask an agent to “fetch these ten URLs concurrently and combine the results,” and it will almost always produce something that compiles, runs, and even prints the right answer on your machine, today. That's the trap. Concurrency bugs are timing-dependent, so a single successful run tells you almost nothing. There are three failure modes worth knowing cold, because they cover the overwhelming majority of what shows up in agent-generated Go.

1. Data races

A data race happens when two goroutines access the same memory location concurrently and at least one of those accesses is a write, with no synchronization between them. The classic example is an agent adding a counter to a worker pool:

var total int
for _, job := range jobs {
    go func() {
        total++          // read-modify-write from N goroutines, no lock
    }()
}

This compiles. It often even produces the right number, because on your laptop the goroutines rarely collide mid-increment. The bug is invisible by inspection and invisible on most runs — it needs a tool, not a read-through, which is exactly why we cover the tooling below.

2. Deadlocks

A deadlock is two or more goroutines each waiting on something the other holds, so none of them can ever proceed. The simplest version an agent writes is a channel send with nobody receiving:

ch := make(chan int)  // unbuffered, and this goroutine is the only one that touches it
ch <- 1                    // blocks forever: nobody is on the other end

The Go runtime can actually catch the special case where every goroutine in the program is asleep — it panics with fatal error: all goroutines are asleep - deadlock! But that only fires when the whole program is stuck. A deadlock between two goroutines out of a hundred running ones produces no crash at all; the rest of the program looks perfectly healthy while a piece of it quietly stops making progress forever.

3. Goroutine leaks

A goroutine leak is a goroutine that blocks permanently on a channel, mutex, or condition variable that will never become ready, and is never going to be cleaned up. It's a deadlock's quieter cousin: instead of freezing the whole program, it just accumulates, one stuck goroutine at a time, silently eating stack memory until something falls over hours or days later. This is the bug agents write the most, because it usually comes from an innocent-looking shortcut: spawning a goroutine to do work and assuming someone will always be there to collect the result.

A Worked Example: The Leak an Agent Wrote

Say we prompted an agent for “a function that fetches a price quote, but gives up after 2 seconds.” A very plausible first draft looks like this:

func GetQuote(sym string) (float64, error) {
    ch := make(chan float64)  // unbuffered
    go func() {
        price := slowLookup(sym)  // may take a while
        ch <- price               // leak: only sent if someone is still listening
    }()

    select {
    case p := <-ch:
        return p, nil
    case <-time.After(2*time.Second):
        return 0, errors.New("quote timed out")
    }
}

It reads fine. It even handles the timeout — there's a select with two cases, which looks like exactly the pattern we asked for earlier. But walk through what happens when slowLookup takes 5 seconds: the time.After case fires, GetQuote returns an error, and the calling goroutine moves on. Three seconds later, the inner goroutine finishes slowLookup and tries ch <- price — but ch is unbuffered and nobody is receiving anymore. That send blocks forever. The goroutine, and the memory it's holding onto, never gets reclaimed. Call GetQuote in a hot path and you leak one goroutine per slow lookup, indefinitely.

The fix is to give the goroutine a way to notice that nobody's listening anymore — which is precisely what context.Context is for:

func GetQuote(ctx context.Context, sym string) (float64, error) {
    ctx, cancel := context.WithTimeout(ctx, 2*time.Second)
    defer cancel()

    ch := make(chan float64, 1)  // buffered: the send below can never block
    go func() {
        ch <- slowLookup(sym)
    }()

    select {
    case p := <-ch:
        return p, nil
    case <-ctx.Done():
        return 0, ctx.Err()
    }
}

Two small, deliberate changes. First, the channel is now buffered with capacity 1, so the inner goroutine's send always succeeds immediately, whether or not anyone is still around to receive — the goroutine finishes and exits cleanly either way, and the buffered value is simply garbage collected if nobody ever reads it. Second, the timeout is expressed as a real context.Context passed in from the caller instead of a bare time.After, so the caller can also cancel early, and the deadline composes correctly if GetQuote is itself called from something with its own deadline. Neither change is dramatic. That's the point — the fix for a leak is almost never a rewrite, it's making sure every goroutine has an exit that doesn't depend on someone else's good behavior.

Tooling: You Cannot Review Concurrency By Reading It

This is the chapter where “read the diff carefully” stops being enough advice. Races and leaks are scheduling-dependent; a correct-looking function can fail one time in a thousand runs. You need tools that watch the program execute, not just the source.

ToolCommandCatches
Race detectorgo test -race ./...Data races, observed at runtime on executed code paths
Race detectorgo build -race / go run -raceSame instrumentation, for manual or load-test runs
Goroutine leak profileGOEXPERIMENT=goroutineleakprofileGoroutines blocked on an unreachable channel/mutex/cond
go vetgo vet ./...Suspicious lock copies, unreachable Println-style misuse

The race detector works by instrumenting every memory access when you build with -race: the runtime watches reads and writes to shared variables and reports a race the moment it observes one, with a stack trace for each side of the conflicting access. It only finds races on code paths that actually execute during that run, so it's not a proof of absence — it's a proof of presence when it fires, and it needs your test suite to actually exercise the concurrent paths to have anything to observe. It's also expensive: expect memory use to climb roughly 5–10× and runtime 2–20×, which is exactly why it belongs in CI rather than every local go build.

Goroutine leaks are newer territory for tooling. Go 1.26 shipped an experimental goroutineleak profile in runtime/pprof, enabled at build time with GOEXPERIMENT=goroutineleakprofile (and exposed live at /debug/pprof/goroutineleak when the experiment is on). It works by leaning on the garbage collector: if a goroutine is blocked on a channel or mutex, and that primitive is unreachable from anything that's still runnable, the runtime knows it can never be unblocked and flags it. That's a genuinely good fit for auditing agent output, because it doesn't need you to already suspect where the leak is — point it at a running program and it tells you.

The Agent Angle This is, empirically, where coding agents fail the most. An agent optimizes for “the code I generated ran and produced the right answer once,” and concurrency bugs are specifically the class of bug that a single successful run cannot reveal. An agent will confidently hand you a goroutine with no shutdown path, a select with no ctx.Done() escape, or a shared counter with no mutex — and every one of those will look completely normal in review unless you know to look for the specific shape of the mistake. Treat any agent-generated code that spawns a goroutine as a draft, not a deliverable, until it's passed -race under real concurrent load.
Watch for A dead giveaway of agent-written concurrency is a goroutine launched with no corresponding sync.WaitGroup, no channel it's expected to report back on, and no context threaded through it at all — “fire and forget” code that reads as fine because nothing crashes today. Ask: if the caller returns early, does this goroutine know? If the answer is no, you're looking at a future leak.

Reflection. None of this makes concurrency in Go complicated — goroutines, channels, and context really are a small, coherent set of primitives, and that's still true with an agent in the loop. What changes is where your attention has to go. You're no longer reviewing whether the code looks right; you're reviewing whether every goroutine has an owner, every context has a deferred cancel, and every blocking operation has an escape. That's a checklist, not a mystery, and it's the one place in this book where running the tool genuinely tells you more than reading the diff ever will.

Audit Checklist

  • Every go func() has a clear owner and a visible shutdown path — someone waits on it, cancels it, or explains in a comment why it's meant to outlive the caller.
  • Every context.WithCancel, WithTimeout, or WithDeadline is followed immediately by defer cancel().
  • context.Context is the first parameter, named ctx, and is never stored on a struct or passed backward up the call stack.
  • No blocking channel send or receive exists without a ctx.Done() escape in a select, or a buffer sized so the send can never block.
  • go test -race ./... actually runs in CI, not just locally on a good day.
  • Shared state written from more than one goroutine is behind a mutex or a channel — never a bare variable incremented from multiple go statements.
