Test-Driven Agent Development: Red, Green, Refactor — with an LLM

Here's a question worth asking before you type a single prompt: what's the fastest way to tell an agent exactly what “done” means, without writing a page of prose it can misread? The answer, it turns out, is the same one Go testers have leaned on for over a decade — a table of inputs and expected outputs, executed as real, running code. Give an agent a failing test instead of a paragraph of requirements, and you've replaced an ambiguous instruction with an unambiguous contract. It either makes the test pass, or it doesn't, and there's no room for a confident-sounding guess in between.

Why the Test Comes First, Not the Code

Classic test-driven development asks you to write a failing test, then write just enough code to make it pass, then clean up. That discipline is useful with a human at the keyboard. It's close to essential with an agent at the keyboard, for a reason that has nothing to do with process purity: an agent will happily implement the version of your feature that's easiest to write, unless something concrete stops it. Ask for “a function that calculates a cart subtotal” in prose, and you'll likely get a function that handles the happy path beautifully and quietly ignores negative quantities, unknown SKUs, and empty carts — not out of malice, but because prose requirements are a fuzzy target and code that compiles is a strong signal of “good enough,” even when it isn't.

A test table closes that gap. When the edge cases are already sitting in the test file before the agent writes a line of implementation, there's no version of “good enough” that skips them — go test will simply keep failing until every row is satisfied. You've turned a matter of judgment into a matter of fact.

The Go Idiom: a Table, Not a Pile of Functions

If you've written Go tests before, you've almost certainly seen this shape, and if you haven't, you're about to adopt it for life. Instead of one function per scenario — TestParse_Empty, TestParse_Negative, TestParse_TooLong, each duplicating the same setup and assertion logic — Go's idiom is a single slice of small structs, one struct per case, run through one shared loop. The Go Wiki's TableDrivenTests page gives the canonical minimal shape: a slice of anonymous structs with an input field and a want field, walked in a loop that calls the function under test and compares the result.

var tests = []struct {
    name  string
    input int
    want  string
}{
    {name: "zero", input: 0, want: "zero"},
    {name: "one", input: 1, want: "one"},
    {name: "negative", input: -1, want: "minus one"},
}

That's it — no framework, no assertion DSL, no annotations. It's just a Go value. Which is exactly the point: a test table is data, and data is the easiest thing in the world to review quickly, diff cleanly, and extend by adding one more line. Dave Cheney's widely cited “Prefer table driven tests” pushes the idea one step further: use a map[string]struct{...} instead of a slice when the case order genuinely shouldn't matter, because Go's undefined map iteration order will shuffle your tests on every run — a cheap, free way to catch a test (or an implementation) that was secretly depending on execution order.

Subtests via t.Run: Isolation and Names You Can Grep

A bare loop over a table has one weakness: if case three panics or fails an assertion, you may never learn whether cases four and five would have passed too. Since Go 1.7, the testing package has solved this with subtests. Wrap the body of your loop in t.Run(tt.name, func(t *testing.T) {...}) and each table row becomes its own named, independently reported test — one failure doesn't take down its neighbors, and each gets its own pass/fail line in the output.

func TestParse(t *testing.T) {
    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            got := Parse(tt.input)
            if got != tt.want {
                t.Errorf("Parse(%d) = %q, want %q", tt.input, got, tt.want)
            }
        })
    }
}

Two small habits pay off here. First, name every case with a phrase that describes the scenario, not the input — "negative quantity is rejected" reads as a specification when it shows up in a failing test log; "case 4" does not. Second, remember that named subtests are individually addressable: go test -run TestParse/negative runs just that one, which turns “the agent broke something” into “the agent broke exactly this one row” in a single command.

The Loop-Variable Trap

Here's a subtlety worth getting precisely right, because agents get it wrong in both directions. For years, Go's for loops declared a single loop variable that was reused across every iteration. If you captured that variable in a closure — exactly what t.Run's callback does — and the loop had already moved on by the time the closure ran, every closure could see the same, final value of tt. This bit hardest when subtests ran in parallel: add t.Parallel() inside the closure so cases run concurrently, and without a fix, every parallel subtest could silently race to see the last row's data instead of its own. The standard workaround, seen in nearly every pre-2024 Go codebase, was to shadow the variable once per iteration:

for _, tt := range tests {
    tt := tt // pre-Go 1.22 idiom: give this iteration its own copy
    t.Run(tt.name, func(t *testing.T) {
        t.Parallel()
        // ... uses tt safely, even after the outer loop has moved on
    })
}

Go 1.22 (February 2024) changed this at the language level. Both classic for loops and for…range loops now give each iteration its own fresh copy of the loop variable, so the closure captured inside t.Run sees exactly the row it was handed — no aliasing, no race, and the tt := tt line is no longer doing anything. Crucially, the new behavior is opt-in per module: it only applies to packages in a module whose go.mod declares go 1.22 or later, so a module still pinned to an older go directive keeps the old, reused-variable semantics even when built with a modern toolchain.

The Agent Angle Watch for this in both directions. Agents trained on a huge corpus of older Go will cargo-cult tt := tt into brand-new, Go-1.22+ code out of habit — harmless, if a little noisy, so don't fight it hard in review. The dangerous direction is the opposite: an agent that omits the shadow line in a codebase whose go.mod still declares an older Go version, reasoning (correctly, for modern Go, incorrectly for yours) that “Go fixed this already.” That's a real, silent data race hiding behind a green build. Before you trust an agent's judgment on this one, check the go line in go.mod yourself.

Mocking Without a Database: Small Interfaces as Test Doubles

Back in Chapter 3 we built the case for implicit interfaces as sandboxes for agent-generated code — define the boundary, and let the agent build safely on either side of it without ever touching the other. Test-driven agent development is where that pays its biggest dividend. If the function you want tested depends on a real database, a real HTTP client, or a real clock, you either need to stand up that infrastructure for every test run, or you need a boundary you can swap out. Go's structural typing makes the second option nearly free: define a small interface that describes only the handful of methods your code actually calls, and hand the agent a trivial fake that satisfies it in memory.

The interface should be as narrow as the code needs — not a mirror of your entire *sql.DB, just the one or two methods this function actually calls. A narrow interface is also a narrower thing for an agent to accidentally misuse.

The Worked Example

Let's put all three ideas — a table, subtests, and a mock behind a small interface — into one compact, complete test. Say we need a function that totals a shopping cart. It needs prices, but it shouldn't need a database to be tested, so it depends on a narrow PriceLookup interface instead of a concrete store:

type CartLine struct {
    SKU      string
    Quantity int
}

type PriceLookup interface {
    Price(sku string) (cents int64, err error)
}

func Subtotal(prices PriceLookup, lines []CartLine) (int64, error) {
    var total int64
    for _, l := range lines {
        if l.Quantity < 0 {
            return 0, fmt.Errorf("negative quantity for %s", l.SKU)
        }
        cents, err := prices.Price(l.SKU)
        if err != nil {
            return 0, fmt.Errorf("price lookup for %s: %w", l.SKU, err)
        }
        total += cents * int64(l.Quantity)
    }
    return total, nil
}

The fake that satisfies PriceLookup in tests is a single line of behavior on top of a map — no mocking framework required, because Go's implicit interfaces mean any type with a matching Price method qualifies:

type fakeCatalog map[string]int64

func (f fakeCatalog) Price(sku string) (int64, error) {
    cents, ok := f[sku]
    if !ok {
        return 0, fmt.Errorf("unknown sku %q", sku)
    }
    return cents, nil
}

Now the table — and notice this is where the human's job actually lives. The happy path is one line. The other four are the ones an agent, left to its own devices, tends to skip: a zero-quantity line that should be valid and free, a negative quantity that should be rejected outright, and a SKU the catalog has never heard of, which should come back as a clearly wrapped error rather than a panic.

func TestSubtotal(t *testing.T) {
    tests := []struct {
        name    string
        catalog fakeCatalog
        lines   []CartLine
        want    int64
        wantErr bool
    }{
        {
            name:    "single line",
            catalog: fakeCatalog{"WIDGET": 500},
            lines:   []CartLine{{"WIDGET", 2}},
            want:    1000,
        },
        {
            name:    "multiple lines sum correctly",
            catalog: fakeCatalog{"WIDGET": 500, "GADGET": 1200},
            lines:   []CartLine{{"WIDGET", 1}, {"GADGET", 3}},
            want:    4100,
        },
        {
            name:    "zero quantity line is valid and free",
            catalog: fakeCatalog{"WIDGET": 500},
            lines:   []CartLine{{"WIDGET", 0}},
            want:    0,
        },
        {
            name:    "negative quantity is rejected",
            catalog: fakeCatalog{"WIDGET": 500},
            lines:   []CartLine{{"WIDGET", -1}},
            wantErr: true,
        },
        {
            name:    "unknown sku surfaces a wrapped error",
            catalog: fakeCatalog{"WIDGET": 500},
            lines:   []CartLine{{"GHOST", 1}},
            wantErr: true,
        },
    }

    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            t.Parallel()
            got, err := Subtotal(tt.catalog, tt.lines)
            if (err != nil) != tt.wantErr {
                t.Fatalf("Subtotal() error = %v, wantErr %v", err, tt.wantErr)
            }
            if err == nil && got != tt.want {
                t.Errorf("Subtotal() = %d, want %d", got, tt.want)
            }
        })
    }
}

Notice what's absent from that loop: no tt := tt shadow line. This file's go.mod declares go 1.22 or later, so each iteration already owns its own tt, and t.Parallel() is safe as written.

Handing It to the Agent

With that test file finished and committed — still red, since Subtotal and fakeCatalog don't exist yet in this telling — the prompt to the agent becomes almost embarrassingly short: “Here is cart_test.go. Implement Subtotal and the PriceLookup interface it needs so that go test passes. Don't modify the test file.” That last sentence matters as much as the rest of the prompt combined; without it, an agent under pressure to turn a test green will sometimes take the easy road and loosen the assertion instead of fixing the code. The test file is the spec. It stays locked.

From here the loop is genuinely mechanical: the agent edits, runs go test, reads the failure, edits again, until every subtest reports PASS. You don't need to watch every intermediate attempt — that's the whole benefit of having turned the spec into an executable check instead of a conversation. Once it's green, you read the diff for the refactor step: is the implementation doing more than the tests demand? Did the agent introduce an interface for something that only ever has one caller? Ask for a cleanup pass with the tests still passing, and you've completed one full red-green-refactor cycle without writing the implementation yourself.

Edge Cases in the Practice Itself

A couple of failure modes are worth naming, because they show up even when every individual technique above is used correctly. First, a test table is only as good as the cases in it — if the human writing the table doesn't think of the negative-quantity case, the agent has no reason to handle it either, and a green build will quietly ship the gap. Writing the tricky cases first, before the happy path even, is a good habit for exactly this reason. Second, watch for an agent that makes a test pass by special-casing the test's own inputs — an if sku == "GHOST" { return errors.New(...) } hiding inside otherwise generic code is a green build that proves nothing. The fix is simple and cheap: add one more row with a different unknown SKU and confirm it still fails the same way for the same general reason.

Reflection

Table-driven tests were never really about saving keystrokes, though they do that too. They're about turning “what should this function do” into something you can hand to another party — a teammate, your future self, or an agent — without a conversation. Writing the table first forces you to think through the tricky cases while you still have the full context in your head, and it gives the agent an unambiguous target it cannot argue its way around. The agent still writes the code. You still own the definition of correct.

Challenges

  1. Take a function an agent already wrote for you and retrofit a table-driven test around it before reading its implementation closely. How many rows pass on the first try, and did writing the table surface a case the original code never handled?
  2. Convert one of your test tables from a slice to a map[string]struct{...} as Dave Cheney suggests, and run go test a few times. Did the shuffled order expose any hidden ordering dependency?
  3. Find a table-driven test in your codebase that still has tt := tt inside its loop. Check the go directive in go.mod and decide, deliberately, whether the line is a harmless habit or load-bearing.
The Agent Angle A test-first prompt does double duty as a leash. “Implement against this test file; don't modify it” keeps an agent from quietly redefining success when the real implementation turns out to be harder than the happy path suggested. If you ever do want the agent's opinion on whether a test case itself is wrong, ask for that explicitly, in words, separate from the implementation turn — never let an agent change the specification and the code that must satisfy it in the same breath.

Audit Checklist

  • Edge and negative cases exist before the agent sees the table. A table with only happy-path rows will get you a happy-path-only implementation.
  • Subtests use t.Run with descriptive names. Names should read as a scenario (“negative quantity is rejected”), not an index, so a failure log doubles as documentation.
  • Mocking goes through a small interface, not a real dependency. If a test needs a live database or network call to pass, the boundary interface is missing.
  • If t.Parallel() is used, loop-variable handling matches the module's declared Go version. Check the go line in go.mod: 1.22+ needs no tt := tt shadow; anything older still does.
  • The agent didn't edit the test file to make it pass. Diff the test file separately from the implementation; a loosened assertion is a red flag, not a green build.
  • A refactor pass happened after green. Confirm the human reviewed the implementation for unnecessary complexity once the tests passed, not just that the tests passed.
