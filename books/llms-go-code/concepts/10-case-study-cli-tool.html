Case Study: Prompting a CLI Tool Into Existence

A CLI tool sounds like the easiest thing you could ask an agent to build. No database, no HTTP handlers, no auth middleware — just flags in, text out. That simplicity is exactly why it's a good closing case study: with the moving parts stripped down this far, the mistakes an agent makes aren't hidden under architectural noise. They're sitting right there in forty lines of main.go, and once you know where to look, you'll spot them in about the time it takes to read the diff.

The Brief We Hand the Agent

Let's build statusck, a small utility that checks the HTTP status of a list of URLs and reports which ones are failing — the kind of tool you'd reach for in a deploy script or a cron job. The prompt we give the agent is short, but it's doing real work as an executable specification (Chapter 7's idea, back in action):

The prompt “Write a Go CLI called statusck. It takes one or more URLs as arguments, or reads them from stdin if none are given. Flags: -timeout (duration, default 5s), -quiet (bool, only print failures), -o (string, write the report to a file instead of stdout). Support -h with a clear usage message. Exit 0 if every URL returns a status under 400, exit 1 if any fail, exit 2 on a bad invocation. Put the real logic in a run function separate from main so it's unit-testable without touching the real network or the real stdout.”

Notice how much of that paragraph is really about the shell of the program — flags, streams, exit codes — and how little is about checking an HTTP status, which is three lines of net/http. That ratio is not an accident. It's also exactly backwards from where most engineers, and most agents, focus their attention. The status check is the part everyone gets right on the first try. The shell is where the bugs live.

Flags, the Standard Way

Go's standard library ships flag parsing in the box — flag — and as of Go 1.26 it's still the right default choice for a tool this size; reach for a third-party flag library only once you need subcommands or POSIX-style short-flag clustering. The package-level functions like flag.String and flag.Parse() are the ones every tutorial shows first, and they work — but they parse into a global, process-wide FlagSet that exits the process on a bad flag. That's fine for a five-line script. It's a trap for anything you want to unit test, which is exactly what our brief asked for. So we build our own FlagSet instead:

type config struct {
    timeout time.Duration
    quiet   bool
    outFile string
}

func newFlagSet(stderr io.Writer) (*flag.FlagSet, *config) {
    fs := flag.NewFlagSet("statusck", flag.ContinueOnError)
    fs.SetOutput(stderr)

    cfg := &config{}
    fs.DurationVar(&cfg.timeout, "timeout", 5*time.Second, "per-request timeout")
    fs.BoolVar(&cfg.quiet, "quiet", false, "only print failing URLs")
    fs.StringVar(&cfg.outFile, "o", "", "write the report here instead of stdout")

    fs.Usage = func() {
        fmt.Fprintf(stderr, "Usage: statusck [flags] <url> [url...]\n\n"+
            "Checks the HTTP status of one or more URLs.\n"+
            "Reads URLs from stdin if none are given as arguments.\n\nFlags:\n")
        fs.PrintDefaults()
    }
    return fs, cfg
}

flag.ContinueOnError is the quiet hero of this snippet. With it, fs.Parse returns an error instead of calling os.Exit on your behalf — which matters twice over. First, it means a test can feed run a bad flag and assert on the returned error instead of watching the test binary vanish. Second, it keeps every call to os.Exit in exactly one place in the program, which is the whole point of the next section. The package also wires up -h and -help for you automatically: if either is passed, Parse calls your Usage function and returns the sentinel error flag.ErrHelp. You don't have to special-case -h yourself — you just have to not swallow the error it hands back, which, as we'll see, is precisely the step an agent likes to skip.

Splitting the Shell from the Engine

Here's the design idea that makes the rest of the file testable: main should do almost nothing. All it does is gather the real operating-system inputs — os.Args, os.Stdin, os.Stdout, os.Stderr — hand them to a plain function, and translate whatever comes back into an exit code. That plain function, run, takes its inputs and outputs as ordinary parameters instead of reaching into global state, which means a test can call it with a fake stdin and a bytes.Buffer standing in for stdout, with no process boundary anywhere in sight. This isn't a Go-specific trick — it's the same shell-versus-engine split good CLI writers in every language reach for — but Go's small, explicit function signatures make it almost free to set up.

Reading Input, Writing a Report

The rest of run is where the brief's remaining requirements live: fall back to stdin when no URLs are given on the command line, and optionally send the report to a file instead of stdout. Both are small, and both are places an agent's error handling tends to get sloppy, so let's write them carefully.

func run(args []string, stdin io.Reader, stdout, stderr io.Writer) (err error) {
    fs, cfg := newFlagSet(stderr)
    if perr := fs.Parse(args); perr != nil {
        return perr // ErrHelp on -h; fs.Usage already printed to stderr
    }

    urls := fs.Args()
    if len(urls) == 0 {
        if urls, err = readLines(stdin); err != nil {
            return fmt.Errorf("reading stdin: %w", err)
        }
    }
    if len(urls) == 0 {
        fs.Usage()
        return errors.New("no URLs given")
    }

    out := stdout
    if cfg.outFile != "" {
        f, ferr := os.Create(cfg.outFile)
        if ferr != nil {
            return fmt.Errorf("opening report file: %w", ferr)
        }
        defer func() {
            if cerr := f.Close(); err == nil {
                err = cerr // surface a failed flush/close, don't swallow it
            }
        }()
        out = f
    }

    client := &http.Client{Timeout: cfg.timeout}
    failures := 0
    for _, u := range urls {
        code, cerr := check(client, u)
        switch {
        case cerr != nil:
            failures++
            fmt.Fprintf(out, "FAIL  %s  (%v)\n", u, cerr)
        case code >= 400:
            failures++
            fmt.Fprintf(out, "FAIL  %s  %d\n", u, code)
        case !cfg.quiet:
            fmt.Fprintf(out, "OK    %s  %d\n", u, code)
        }
    }
    if failures > 0 {
        return fmt.Errorf("%d of %d checks failed", failures, len(urls))
    }
    return nil
}

func check(client *http.Client, url string) (int, error) {
    resp, err := client.Get(url)
    if err != nil {
        return 0, err
    }
    defer resp.Body.Close()
    return resp.StatusCode, nil
}

func readLines(r io.Reader) ([]string, error) {
    var lines []string
    sc := bufio.NewScanner(r)
    for sc.Scan() {
        if line := sc.Text(); line != "" {
            lines = append(lines, line)
        }
    }
    return lines, sc.Err()
}

Two small things in there are worth calling out because they're easy to miss and even easier for an agent to skip. First, run uses a named return value, err, purely so the deferred close can report its own failure. f.Close() can fail — a full disk won't necessarily surface until the final flush — and if you throw that error away, you've told the caller a report was written successfully when it might be truncated garbage. Second, check defers resp.Body.Close() immediately after confirming the request succeeded, which is the single most common resource leak in AI-generated Go HTTP code (it came up back in the book's very first audit checklist, and it'll keep coming up — that's how you know it's a real pattern and not a one-off tip).

The thin shell that ties it together is almost anticlimactic after all that, and that's exactly the point:

func main() {
    err := run(os.Args[1:], os.Stdin, os.Stdout, os.Stderr)
    if errors.Is(err, flag.ErrHelp) {
        os.Exit(0) // -h is not a failure
    }
    if err != nil {
        fmt.Fprintln(os.Stderr, "statusck:", err)
        os.Exit(1)
    }
}

Eight lines, one call to run, and every os.Exit in the entire program. That last part is the load-bearing detail.

The Agent Angle os.Exit terminates the process immediately — the Go documentation is explicit that deferred functions do not run. Every defer in this file, including the one closing the report file and the one closing each response body, depends on nothing ever calling os.Exit before it has a chance to unwind back through main. An agent that reaches for log.Fatal inside check or readLines — a very natural-looking shortcut, since log.Fatal is one line and reads like a normal error return — is quietly calling os.Exit(1) from three stack frames deep, skipping every deferred cleanup between there and the top. The fix costs nothing: return err instead of log.Fatal(err), everywhere except inside main itself.

Where Agents Go Wrong in Exactly This Scenario

Once you've reviewed a few dozen agent-written CLIs, the mistakes stop being surprising — they cluster into the same handful of shapes every time.

Swallowing the error Parse hands back. The single most common bug is code that calls fs.Parse(args) and ignores the return value, or worse, uses the package-level flag.Parse() inside a function that isn't main. The package-level flag set defaults to flag.ExitOnError, which calls os.Exit(2) for a bad flag and os.Exit(0) for -h — from inside the flag package itself. That's an exit call buried two layers below your own code, invisible in a diff, and it turns your carefully separated run function untestable the moment a test passes it a flag it doesn't recognize.

Writing to the wrong stream. The Unix convention, older than most languages in active use today, is that normal output goes to stdout and diagnostics go to stderr — standard file descriptors 1 and 2, respectively, kept separate precisely so a shell pipeline can redirect the data (statusck url... > report.txt) while error messages still land on the terminal instead of corrupting the file. An agent that reaches for a bare fmt.Println("error:", err) sends that message to stdout by default, where it silently pollutes whatever's consuming your output. Our main is careful to route the error message through os.Stderr explicitly; it's one keystroke of discipline that a generated-in-a-hurry version is prone to drop.

Skipping usage discovery. Users reach for -h reflexively, before they read anything you wrote. If an agent leaves fs.Usage unset, they get the flag package's generic default — a bare list of flag names with no explanation of what the tool does or what a “URL” argument even means here. Setting Usage explicitly, as we did above, costs four lines and turns -h into actual documentation instead of a shrug.

Treating exit codes as an afterthought. The brief asked for three distinct codes — 0 for success, 1 for “ran fine but found failures,” 2 for “you used the tool wrong.” An agent that returns a plain error for both cases and lets main flatten everything to os.Exit(1) has thrown away information a caller in a script needs — “retry this” and “fix your invocation” are very different responses. Here it falls out almost for free: a bad flag returns before we ever reach the URL loop, and flag.ExitOnError's own convention of reserving 2 for usage errors is worth matching rather than fighting.

Watch for this Grep any agent-generated main.go for log.Fatal, log.Panic, and bare os.Exit calls outside func main. Every hit is a deferred cleanup that silently never runs, and a function that looked testable on the page but exits the whole test binary the moment you call it.

Reflection

None of what we just built is exotic. A FlagSet, a run function with explicit parameters instead of ambient globals, a main that does nothing but translate an error into an exit code — every piece is ordinary standard-library Go. That's exactly why a CLI tool is such a good closing case study for this book. There's no framework to hide behind and no clever abstraction to distract from the fundamentals: explicit errors, one owner for os.Exit, deferred cleanup that only works if nothing short-circuits it, and streams used for what they're for. Every concept from the earlier chapters — errors as values, small explicit surfaces, the toolchain as a feedback loop — shows up here in miniature, which makes this exactly the shape of program worth handing to an agent first, before you trust it with something bigger.

Challenges

  1. Write a table-driven test for run that feeds it a fake stdin via strings.NewReader, captures stdout in a bytes.Buffer, and asserts on both the printed report and the returned error — no real HTTP request required once you inject a fake http.Client.
  2. Prompt an agent to add a -json flag that switches the report to newline-delimited JSON. Check whether it threads the new flag through config and newFlagSet, or bolts a second, parallel parsing path onto the side.
  3. Delete the defer that reports f.Close()'s error, force a write failure in a test, and confirm the bug you just reintroduced is exactly the one this chapter warned about.
The Agent Angle When you hand off a CLI spec, state the shell/engine split as a hard constraint, not a suggestion: “all logic lives in run(args, stdin, stdout, stderr) error; main contains only the call to run and the exit-code translation.” It's one sentence, it's easy for an agent to satisfy, and it gives you a single, obvious place to check for the exact class of bug this chapter is about — if os.Exit or log.Fatal shows up anywhere else in the diff, that's the whole review, right there.

Audit Checklist

  • Every os.Exit, log.Fatal, and log.Panic lives in main. Grep the rest of the package for all three; any hit outside main is a skipped defer waiting to happen.
  • Flag parsing uses ContinueOnError, not the package-level flag.Parse(). The latter's default ExitOnError makes the parsing path untestable and hides an os.Exit call inside the standard library itself.
  • fs.Usage is set explicitly and -h actually explains the tool. Run it and read what a first-time user would see.
  • Normal output goes to stdout; diagnostics and errors go to stderr. Redirect stdout to /dev/null and confirm error messages still show up on the terminal.
  • Every opened file and response body has a checked defer ...Close(). For files you write to, the close error is surfaced, not discarded — a full disk should be a reported failure, not a silent one.
  • Exit codes are distinct and documented. 0 for success, a non-zero code for “ran correctly but found a problem,” and a different one for “you called this wrong” — verify the tool's own -h text says which is which.

That's the last worked example in the book.
