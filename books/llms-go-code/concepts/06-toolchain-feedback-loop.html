Chapter 6

The Go Toolchain as an Automated Feedback Loop for Agents

Here's a question worth sitting with: when an agent writes you a hundred lines of Go at two in the morning, who checks it before you wake up? The honest answer, for most of us, is nobody — unless we've built something that does. This chapter is about that something. Go ships with an opinionated, batteries-included toolchain, and it turns out that toolchain is almost perfectly shaped to be an agent's tireless, incorruptible reviewer.

The cheapest bug you'll ever get

Let's start with a comforting fact about how LLMs fail at Go specifically. When a model hallucinates — invents a function that doesn't exist, imports a package that was never published, misremembers a standard-library signature — Go's strict compiler and small standard library turn that hallucination into the loudest, cheapest possible signal: a go build failure. Not a silent runtime panic three weeks from now. Not a subtly wrong number in a report nobody double-checks. A red exit code, on your machine, before a single byte of the change reaches a colleague or a customer.

Compare that to a dynamically typed language, where a typo'd attribute name or a made-up keyword argument can sail through review and only explode when a rare code path finally executes in production. Go's compiler doesn't let hallucinations hide. That's the whole thesis of this chapter: Go's toolchain isn't just a set of developer conveniences, it's a sequence of increasingly expensive gates, and every one of them can be pointed at an agent instead of (or in addition to) a human.

Formattinggofmt -lStyle disagreements, diff noiseMilliseconds
Static analysisgo vetReal, provable bugsUnder a second
Compilationgo buildHallucinated symbols, type errorsSeconds

gofmt — no bikeshedding, ever

Every team that has ever argued about tabs versus spaces, or where the brace goes, has wasted hours it will never get back. Go's answer was blunt: there is one canonical formatting, gofmt produces it, and the argument is over before it starts. This matters more than it sounds like it should, because it removes an entire category of noise from code review — human or agent.

For our purposes, the useful trick is gofmt -l, which lists any file that isn't already formatted instead of silently reformatting it. That distinction matters in an automated loop: you don't want your pipeline quietly rewriting an agent's output (that hides information), you want it to fail loudly when the output doesn't match the canonical form, so the failure becomes an instruction the agent can act on.

$ gofmt -l ./...
internal/pricing/discount.go
$ echo $?
1

A non-empty file list plus a non-zero exit code is all you need. Feed that filename back to the agent with the instruction "run gofmt -w on this file and resubmit," and you've closed the cheapest loop in the whole chapter — often before the agent even gets to explain itself.

go vet — a linter for bugs, not taste

It's tempting to lump go vet in with style linters, but that undersells it. go vet doesn't care about your naming conventions; it looks for constructs that compile fine but are almost certainly wrong — a Printf call whose format verbs don't match its arguments, a struct copied by value that contains a sync.Mutex, an unreachable branch, a loop variable captured incorrectly. These are exactly the kinds of small, confident mistakes an LLM makes when it's pattern-matching on "code that looks like this usually works."

$ go vet ./...
# github.com/acme/inventory/internal/pricing
internal/pricing/discount.go:42:2: Printf call needs 1 arg but has 2 args

Notice the shape of that message: file, line, column, and a plain-English description of exactly what's wrong. That's not an accident — it's designed to be actionable by a human skimming a terminal, and it turns out to be equally actionable pasted verbatim into an agent's next turn. No summarizing, no interpretation required.

go build and go test -race — ground truth

Formatting and vetting are cheap sanity checks. go build and go test are where the rubber meets the road, because they're the only two commands in this chapter that actually execute the agent's reasoning rather than just reading it. Consider an agent asked to add a rounding helper to a pricing package. It writes the call site, but forgets — or hallucinates — the helper itself:

package pricing

func ApplyDiscount(price float64, pct int) float64 {
    discounted := price - (price * float64(pct) / 100)
    return RoundToCents(discounted) // never defined anywhere
}
$ go build ./...
# github.com/acme/inventory/internal/pricing
internal/pricing/discount.go:6:9: undefined: RoundToCents

This is the "cheapest bug" from the top of the chapter, made concrete. A model confidently reached for a function name that felt plausible — probably because it exists in some other codebase it was trained on — and Go's compiler refused to let it slide. No test run needed, no staging deploy, no 3am page. Just an exit code and a line number.

Once the code compiles, go test -race is your next and most important gate, especially for anything touching goroutines (which, as the last chapter argued, is exactly where agents stumble most). The race detector instruments memory accesses at build time and watches for two goroutines touching the same location without synchronization — something no amount of code review will reliably catch by eye.

$ go test -race ./internal/pricing/...
==================
WARNING: DATA RACE
Write at 0x00c0000a4010 by goroutine 8:
  internal/pricing.(*Cache).Set()
      /internal/pricing/cache.go:34 +0x64

Previous read at 0x00c0000a4010 by goroutine 7:
  internal/pricing.(*Cache).Get()
      /internal/pricing/cache.go:22 +0x50

Goroutine 8 (running) created at:
  internal/pricing.(*Cache).warm()
      /internal/pricing/cache.go:58 +0x8c
==================
--- FAIL: TestCacheConcurrentAccess (0.02s)
FAIL

That trace names the exact write, the exact conflicting read, and the goroutine that created the offending one. An agent doesn't need you to translate "there's a race condition somewhere in the cache" — it needs precisely this: three stack traces and two line numbers. Feed the block back verbatim and the agent has everything a senior engineer would ask for.

go.mod — reproducible dependencies an agent can't wiggle out of

Agents love to reach for a library. Sometimes that's the right call; often it's a shortcut around code the agent should have just written. Go's module system doesn't stop an agent from adding a dependency, but it does make the decision visible and reviewable. Every dependency an agent pulls in lands as an explicit, diffable line in go.mod and a pinned, hash-verified entry in go.sum:

module github.com/acme/inventory

go 1.26

require (
    github.com/google/uuid v1.6.0
)

Treat that diff exactly like you'd treat a human teammate quietly adding a new dependency in a pull request: read it. Does the package exist? Is it maintained? Did the agent actually need it, or did it reach for a library to avoid writing twelve lines of arithmetic? Because go.mod pins exact versions and go.sum records cryptographic checksums, the build is reproducible on every machine and every CI run — there's no "works on my agent's sandbox" ambiguity to untangle later.

The Agent Angle Ask your agent to run go mod tidy as part of its own workflow, not as an afterthought you run yourself. An agent that tidies its own module file after every change surfaces unused imports and stray dependencies immediately — while the context of why it added something is still fresh in the conversation, not three commits later when nobody remembers.

pprof — profiling what the agent shipped

Correctness gates catch bugs; they don't catch an agent that wrote something correct but needlessly slow — an O(n²) loop where a map lookup would do, or a hot path that allocates a fresh slice on every call. That's what pprof is for. Add the standard import for a quick local profile, or wire up net/http/pprof for a running service, and you get CPU, heap, and (as of the newest toolchain) goroutine-health profiles you can hand an agent as concretely as a stack trace:

$ go test -run none -bench BenchmarkApplyDiscount -cpuprofile cpu.out ./internal/pricing
$ go tool pprof -top cpu.out
Showing nodes accounting for 1.81s, 94.3% of 1.92s total
      flat  flat%   sum%        cum   cum%
     1.12s 58.33% 58.33%      1.12s 58.33%  internal/pricing.roundHalfEven
      0.41s 21.35% 79.68%      0.41s 21.35%  runtime.mallocgc

The rule of thumb here isn't "profile everything" — that's a good way to drown in noise. It's "profile anything you're about to trust with real load." An agent's code should earn its way into a hot path the same way a human's does: with a benchmark and a profile, not a vibe.

Wiring it into a loop

Here's the idea this whole chapter has been building toward. Individually, these commands are just tools you run by hand. Wired together with a little scripting, they become something more interesting: a closed feedback loop that lets an agent self-correct without a human in the middle of every iteration.

The mechanics are almost embarrassingly simple. A script — a shell script, a pre-commit hook, a CI job, or the harness driving your coding agent directly — runs each gate in order and stops at the first failure:

#!/usr/bin/env bash
set -euo pipefail

unformatted=$(gofmt -l .)
if [ -n "$unformatted" ]; then
  # feed $unformatted back to the agent as the next instruction
  exit 1
fi

go vet ./...   # feed stderr back verbatim on failure
go build ./... # feed stderr back verbatim on failure
go test -race ./... # feed stdout+stderr back verbatim on failure

The crucial design choice is the word verbatim. Don't summarize the failure, don't paraphrase it, don't tell the agent "the build failed, please fix it." Paste the exact compiler or test output back as the next turn. Go's error messages are already written for a reader with no other context — file, line, column, plain description — which is precisely what a language model needs to make a targeted, minimal fix instead of rewriting half the package out of guesswork.

The Agent Angle Resist the urge to let an agent "fix" a failure by deleting the failing test or adding a //go:build ignore tag to a file that won't compile. Both make the loop go green without making the code correct. Your gate script should treat a shrinking test count as suspicious as a red one — diff the test files, not just the exit code.

Edge cases: when the loop lies to you

No feedback loop is perfect, and it's worth naming where this one can quietly mislead you. go vet is deliberately conservative — it only flags patterns it can prove are wrong, so an agent can still write logically broken code that vets clean. go test -race only catches races that actually occur during the run; a race guarded by a timing coincidence in your test data can hide for months. And a loop that only checks "does it compile and pass" says nothing about whether the code does the right thing — that's still a job for the tests you write, which is exactly the subject of the test-driven chapter ahead. Treat every green run as "no known problems," never as "provably correct."

Reflection

What makes this loop worth building isn't any single command — it's that Go handed us all five gates already sharpened, already fast, and already opinionated enough that there's little to configure. A team using a language with a looser compiler or a fragmented tooling ecosystem has to build most of this scaffolding from scratch before an agent can be trusted to iterate unsupervised. In Go, you mostly just have to wire together commands that already ship in the box. The agent gets faster, cheaper iterations; you get to spend your attention on the one thing none of these tools can check — whether the code is solving the right problem in the first place.

Key takeaway Every gate in this chapter answers a different question — gofmt: is it readable, vet: is it obviously wrong, build: does it exist, test -race: does it behave, pprof: is it fast enough. Run them in that order, cheapest first, and feed every failure back to the agent exactly as the toolchain wrote it.

Audit Checklist

  • CI runs gofmt -l, go vet, go build, and go test -race on every agent-authored change, in that order, and fails the pipeline on the first non-zero exit.
  • Failure output is captured and replayed to the agent verbatim — not summarized, not paraphrased — so line numbers and messages stay actionable.
  • A shrinking test count or a new //go:build ignore / skip tag is treated as a red flag, not a passing loop.
  • go.mod and go.sum are committed and reviewed like any other diff, especially new require entries an agent added on its own.
  • pprof is used to sanity-check anything performance-sensitive — a benchmark plus a profile, not a vibe — before agent-authored code ships to a hot path.
  • The toolchain version is pinned in go.mod's go directive and kept current with security patch releases, not just major versions.
