Errors as Values: Catching an Agent That Cuts Corners

Quick question: what does a Go function do when something goes wrong? It doesn't throw. It doesn't raise. It doesn't unwind the stack looking for a handler three call frames up. It just returns. An error in Go is a value — the same way an int or a string is a value — and that one design choice changes almost everything about how you read, write, and audit Go code.

Motivation: no exceptions, no magic

Coming from Python, Java, or JavaScript, this feels like a downgrade at first. Where's try? Where's catch? Go's answer is: nowhere, on purpose. Instead, any function that can fail returns two things — the result you wanted, and an error describing what went wrong, or nil if nothing did:

func ReadConfig(path string) (*Config, error) {
    data, err := os.ReadFile(path)
    if err != nil {
        return nil, err
    }
    var cfg Config
    if err := json.Unmarshal(data, &cfg); err != nil {
        return nil, err
    }
    return &cfg, nil
}

There's no hidden control-flow jump here. Every place a call can fail is a visible if err != nil a few lines below it. That's tedious to type, and it's exactly why people joke about Go being "if err != nil the language." But tedium is the price of honesty: nothing is invisible, nothing is inferred, and nothing can silently blow past three layers of your call stack the way an uncaught exception can. When you're reading code — especially code you didn't write yourself, like an agent's output — that honesty is worth a lot more than it costs.

Design: the error interface

Under the hood, error is almost insultingly simple. It's a one-method interface defined in the language's universe block:

type error interface {
    Error() string
}

Anything with an Error() string method satisfies it — remember from the last chapter, that's implicit, structural typing at work again. That minimalism is deliberate: an error just needs to be able to describe itself. Go gives you three common ways to produce one, and knowing all three — and when an agent should reach for which — is the real skill in this chapter.

1. Sentinel errors

A sentinel error is a package-level value, usually built with errors.New, that callers compare against to detect a specific, well-known failure:

var ErrNotFound = errors.New("item not found")

func Lookup(id string) (*Item, error) {
    item, ok := store[id]
    if !ok {
        return nil, ErrNotFound
    }
    return item, nil
}

The standard library does this too — io.EOF and sql.ErrNoRows are both sentinels. The catch: errors.New returns a distinct value every time it's called, even with identical text, so sentinels only work as a shared var a caller can reference. And crucially, callers should compare with errors.Is(err, ErrNotFound), never == — more on why in a moment.

2. Custom error types

Sometimes a string isn't enough — you want structured data alongside the failure: which field was invalid, what the retry-after duration was, which HTTP status came back. For that, define a type and give it an Error() string method:

type ValidationError struct {
    Field string
    Msg   string
}

func (e *ValidationError) Error() string {
    return fmt.Sprintf("field %q: %s", e.Field, e.Msg)
}

Now a caller who wants the structured detail can pull it back out with errors.As, which we'll get to in a page. This is the pattern to reach for whenever "something went wrong" isn't descriptive enough for the code that has to react to it.

3. Wrapping: %w versus %v

Real systems have layers. A repository calls a driver, a service calls the repository, a handler calls the service — and an error from the bottom needs to travel up through all of them without losing its identity, while still picking up context at each hop ("failed to save user" is more useful than a bare "connection refused"). Since Go 1.13, fmt.Errorf has a verb for exactly this: %w.

func SaveUser(u User) error {
    if err := db.Insert(u); err != nil {
        return fmt.Errorf("save user %s: %w", u.ID, err)
    }
    return nil
}

When fmt.Errorf sees %w, the error it returns grows an Unwrap() error method that hands back the original error underneath. That one method is the entire mechanism — there's no other magic. It's what lets errors.Is and errors.As walk back down through every layer of wrapping to find a match, using a pre-order, depth-first walk of the error tree (the Go blog's own description of the algorithm). Swap that %w for %v, and you still get a nice printable message — but the resulting error has no Unwrap method. The original error is baked into a string and thrown away as a distinct value. errors.Is and errors.As simply can't see past it anymore.

The signatures worth memorizing, straight from the errors package:

func New(text string) error
func Is(err, target error) bool
func As(err error, target any) bool
func Unwrap(err error) error
func Join(errs …error) error

errors.Is is for sentinels — "is this, or does it wrap, that specific value?" errors.As is for custom types — "does this chain contain an error of this type, and if so, give it to me." Both only work as far down the chain as the Unwrap methods actually go. As of Go 1.26 there's also a generic convenience, errors.AsType[E], which does the same walk as As but returns the typed value directly instead of writing through a pointer — handy, but it doesn't change the underlying contract: wrapping still has to be real for it to find anything.

Key idea %w isn't a formatting nicety — it's the one thing that keeps an error's identity attached to its message. Lose it, and you can still print the error; you just can't programmatically react to it anymore.

Implementation: what "checking" actually means

Notice that everything above still boils down to the same three-line ritual: call something, check err != nil, decide what to do. The interesting part — the part that separates good Go from Go-shaped noise — is what goes inside that branch. Your options, roughly in order of how often you should use them:

SituationDo this
Caller needs to know the operation failed, nothing morereturn err
Caller needs context about where it failedreturn fmt.Errorf("...: %w", err)
Caller needs to branch on a known failure kinderrors.Is / errors.As
This really is the end of the line, nothing more to dolog.Printf / handle it, don't discard it

What's not on that table: silently swallowing the error and moving on as if nothing happened. That's always wrong, and it's the single most common way agent-written Go quietly breaks in production.

Edge cases: where this bites

A few sharp corners worth knowing before you go looking for them in agent output:

  • Comparing sentinels with ==. It works right up until someone wraps the error one layer further up the stack, at which point the identity comparison silently starts failing. errors.Is doesn't have this problem — use it by default, even when you're "sure" nothing wraps it today.
  • Double-wrapping the same context. If the repository layer already says "save user u123:" and the service layer wraps again with "save user u123:", the caller gets a redundant, confusing message. Each layer should add new information, not repeat what's already there.
  • Wrapping a sentinel changes its message but not its identity. fmt.Errorf("query users: %w", sql.ErrNoRows) still satisfies errors.Is(err, sql.ErrNoRows) — the printed text is cosmetic, the chain underneath is what matters.
  • errors.As needs a pointer to the target type, e.g. var ve *ValidationError; errors.As(err, &ve). Passing the wrong shape panics at runtime instead of failing to compile — a rare case where Go's strictness doesn't catch a mistake until execution.

The agent angle: three ways an agent cuts corners

Here's the thing about coding agents and error handling: an agent under token or instruction pressure will produce code that compiles, looks idiomatic at a glance, and passes a shallow review — while quietly discarding the information a caller three layers up actually needs. Three patterns show up constantly. Watch for all of them.

The Agent Angle Agents love the blank identifier. _, _ = f() compiles cleanly, satisfies go vet in the common case, and makes the diff look tidy — but it throws the error away before anyone ever sees it. If you see _ = or _, _ = anywhere near a call that can fail, stop and ask: was that deliberate, or did the agent just want the line count to shrink?

Here's a "spot the bug" pair. Both compile. Only one is safe to ship.

// --- agent-written, broken ---
func ProcessOrder(id string) error {
    order, _ := repo.FindOrder(id)              // (1) error silently dropped

    if err := billing.Charge(order); err != nil {
        return nil                                    // (2) checked, then thrown away
    }

    if err := ship.Dispatch(order); err != nil {
        return fmt.Errorf("dispatch failed: %v", err)  // (3) %v hides ErrOutOfStock
    }
    return nil
}
// --- corrected ---
func ProcessOrder(id string) error {
    order, err := repo.FindOrder(id)
    if err != nil {
        return fmt.Errorf("process order %s: %w", id, err)  // (1) fixed: checked and wrapped
    }

    if err := billing.Charge(order); err != nil {
        return fmt.Errorf("charge order %s: %w", id, err)   // (2) fixed: real failure, real return
    }

    if err := ship.Dispatch(order); err != nil {
        return fmt.Errorf("dispatch order %s: %w", id, err)  // (3) fixed: %w preserves ErrOutOfStock
    }
    return nil
}

Walk through why each fix matters:

  • (1) If FindOrder fails, the broken version happily proceeds with a zero-value order and charges a customer for nothing. The blank identifier didn't just lose an error message — it let the function barrel past a failure as if it succeeded.
  • (2) This is the sneakiest one, because it looks like careful error handling — there's an if err != nil right there. But the branch body does nothing useful; it reports success (nil) after a real charge failure. A caller checking if err != nil on ProcessOrder will never know billing failed.
  • (3) Suppose ship.Dispatch returns a sentinel, ErrOutOfStock, when the warehouse is empty, and the caller wants to retry later specifically for that case with errors.Is(err, ship.ErrOutOfStock). With %v, that check always fails — the sentinel's identity was flattened into a string the moment it was formatted. Swapping in %w is a one-character fix that restores the caller's ability to make that decision at all.
Warning None of these three bugs fail a build. go build and even go vet are happy with all of them. This is exactly the kind of defect that slips through an agent's own self-check ("does it compile? does it pass the test I wrote?") and only shows up later, in production, as a customer who got charged for an order that was never dispatched.

Reflection

Go didn't give you exceptions because exceptions make it too easy to let a failure travel somewhere you didn't intend, silently, until it surfaces as a stack trace nobody wrote down context for. Errors as values force every failure path into the open — which is exactly the property you want when the code writing those paths is an agent instead of you. The syntax is boring on purpose: if err != nil is small enough to read in a glance, which means it's small enough to audit in a glance, too. Your job isn't to eliminate that boilerplate. It's to make sure every single occurrence of it does something real.

Challenges

  1. Grep a real (or agent-generated) Go file for _ = and _, err := followed by an empty or no-op branch. For each hit, decide: deliberate, or corner-cut?
  2. Take the corrected ProcessOrder above and add a *ValidationError custom type for a bad order ID, then write the errors.As call a caller would use to detect it specifically.
  3. Write a one-sentence prompt instruction you'd add to an agent's system prompt that would have prevented all three bugs in the broken snippet. Test it by asking an agent to write a similar three-step pipeline and see if the instruction actually holds.

Audit Checklist

  • Grep the diff for _ = or _, _ = next to any call that returns an error — every hit needs a reason.
  • Every err returned by a called function is either checked or explicitly, deliberately ignored with a comment saying why.
  • Every non-empty if err != nil branch actually does something with the error — return it, wrap it, log it, or retry. Not just return nil.
  • %w is used (not %v) anywhere a caller might reasonably need errors.Is or errors.As to work on that error later.
  • No layer repeats context a caller below it already added — each wrap should add new information, not restate the last one.
  • Sentinel errors are compared with errors.Is, and custom error types are extracted with errors.As — never a bare == or type assertion on something that might be wrapped.
