Chapter 7

Executable Specifications: Prompts, Interfaces & Schema-First Design

Everything up to this point has trained your eye. You can read a struct and tell whether it models the domain honestly. You can look at an interface and know whether it's a real seam or a leaky one. You can catch a swallowed error, spot a leaked goroutine, and wire the toolchain into a loop that keeps an agent honest after the fact. That's Part I: reading and auditing. Part II turns the camera around. Instead of asking “is this code good?” we're asking a harder question up front: “how do I tell the agent what good even means, in a form it can't quietly misread?”

The problem with prose

Here's a spec, the kind you'd type into a chat box without thinking twice about it:

“Write a function that looks up a user by their email address.”

It sounds complete. It is not. What does it return if no user matches — a zero value, a nil pointer, an error? Is the lookup case-sensitive? Does it trim whitespace? Does it hit a cache, a database, both? Is it safe to call concurrently? None of that is in the sentence, so the agent has to invent answers, and it will invent something — confidently, plausibly, and possibly differently every time you ask. A paragraph of English is a weak specification because nothing checks it. You read the code, it looks reasonable, you move on, and the gap between what you meant and what got built survives to production.

Compare that to this:

// UserByEmail looks up a user by email address. The match is
// case-insensitive; leading and trailing whitespace is trimmed
// before comparison. It returns ErrNotFound if no user has that
// email, and it is safe for concurrent use by multiple goroutines.
type UserFinder interface {
    UserByEmail(ctx context.Context, email string) (User, error)
}

That's still English — the doc comment is prose too — but it's prose sitting on top of a func signature, and the signature is a contract the compiler enforces whether the agent read the comment carefully or not. Wrong argument count: compile error. Wrong return type: compile error. Forgot the error return and tried to panic instead: compile error, because nothing else in your codebase can satisfy the interface without it. The English still carries the nuance — case-insensitivity, trimming, the sentinel error — but now it has teeth, because you're about to pair it with a test that checks whether the agent actually read it.

This is the central claim of the chapter: a Go interface, a doc comment, and a failing test are a strong specification, because the compiler and the test runner can mechanically check whether the agent satisfied it. You don't have to trust the agent's summary of what it did. You run go build and go test and you get a verdict. We call this class of artifact an executable specification — not because it runs the business logic itself, but because it runs the check for you.

Interface first, implementation second

Chapters 2 and 3 already built the muscle for this: model the data as a struct before the agent writes a method, and describe behavior as a small interface before the agent writes an implementation. Executable specifications are just that discipline applied to the moment you hand work to an agent. The order matters:

  1. Define the types and the interface. Decide what the caller-facing shape looks like — the struct fields, the method signatures, the error types — before any business logic exists.
  2. Write a doc comment on every exported symbol that states the contract: what valid input looks like, what the zero-value or error case is, whether there are ordering or concurrency requirements. Go's own convention — doc comments start with the symbol's name and read as a full sentence — already nudges you toward a testable claim rather than a vague description.
  3. Write one stub, failing test that calls the not-yet-implemented function and asserts on the contract you just wrote down. It should fail to compile or fail to pass, on purpose — that failure is the target the agent is aiming at.
  4. Hand all three to the agent as the spec, and ask it to make the test pass without changing the interface or the test's assertions.

Notice what you're not doing: you're not describing the algorithm. You don't care whether the agent uses a map or a binary search or a database index, as long as UserByEmail returns the right thing for the right reasons within the constraints you wrote. That's the trade-off of this approach — it buys you a mechanically checkable contract at the cost of some up-front design work that used to happen implicitly, in whatever code the agent felt like writing. In our experience that trade is almost always worth it, because the design work has to happen somewhere, and doing it before generation is far cheaper than doing it as an audit afterward.

Key idea A spec is only as strong as what checks it. English is checked by a human skimming the diff, which is unreliable and doesn't scale. A type signature is checked by the compiler, every single time, for free. A test is checked by go test, every single time, for free. Push as much of your intent as you can from the first bucket into the second and third.

Small packages, small blast radius

There's a second, quieter failure mode that executable specifications also happen to fix: cross-package hallucination. Drop an agent into a single sprawling package — the classic internal/ junk drawer with forty files and no clear boundaries — and it has to guess which of the two hundred exported names it's allowed to touch, which ones are load-bearing for some other subsystem, and which struct field is safe to add without breaking an invariant three files away that it never read. It will guess wrong sometimes. Not out of malice — it genuinely can't hold the whole package in its head any better than you can.

The fix is architectural, not prompt-engineering: a modular monolith. Keep it one deployable binary if that's the right call for your system, but split the code into a handful of small, purpose-named packages, each with a narrow public surface and its own doc comment describing what it owns:

billing/      // invoices, charges, refunds — owns money movement
catalog/      // products, pricing — owns what's for sale
identity/     // users, sessions, auth — owns who's asking
notify/       // email, webhooks — owns telling people things

When you ask an agent to add a refund reason code, you hand it the billing package's interfaces and tests — not the whole module. It literally cannot import identity's unexported helpers by accident, because Go's package-level visibility won't let it, and it has no reason to even open catalog's files because nothing in its task touches them. The interface between packages becomes the same kind of checkable contract as the interface between your code and the agent: small, explicit, and enforced by the compiler rather than by the agent's memory of a sprawling codebase.

Schema-first design for services

Interfaces work beautifully inside a Go binary, where the compiler is watching. The moment your contract crosses a process boundary — an HTTP API, a gRPC service, a database — the compiler can't help you anymore, and that's exactly where an unsupervised agent tends to invent field names, get a type slightly wrong (a price as a float64 instead of an integer cents value, say), or drift the client and server apart one small “helpful” change at a time.

The fix is the same idea one level up: make the schema itself the executable specification, and generate the boilerplate from it rather than asking the agent to type it out by hand.

  • HTTP APIs — OpenAPI. Write the OpenAPI document first (paths, request/response schemas, error shapes), then run a generator such as oapi-codegen to produce the Go structs, router bindings, and client stubs. The agent's job becomes filling in the handler bodies against generated interfaces, not guessing at JSON field names.
  • gRPC services — Protocol Buffers. Write the .proto file, then run protoc with the protoc-gen-go and protoc-gen-go-grpc plugins to generate the message types and service interfaces, per the official Go Generated Code Guide. The agent implements the generated server interface; it never hand-writes a wire type.
  • SQL-backed services — the schema itself. Write the CREATE TABLE statements and the queries, then run sqlc to generate type-safe Go functions for each query, columns and all. The agent never writes a raw db.Query call with a hand-typed scan target that can silently drift from the actual column list.

In every case the shape of the trade-off is identical to the interface-and-test pattern: you spend effort up front describing the contract precisely, in a format a tool can parse, and in exchange you get boilerplate that is generated — and therefore correct by construction — instead of hand-typed and therefore one autocomplete away from a typo that only shows up at 2 a.m. The rule of thumb worth tattooing on your prompt template: if a human would find generating it by hand tedious and mechanical, an agent will find it tedious too, and tedious is exactly where hallucination creeps in. Let the generator own the tedious part; save the agent's judgment for the part that actually needs judgment.

A worked example: the spec you'd paste into the context window

Let's make this concrete. Say we're adding a coupon-redemption check to the billing package from earlier. Here is literally what we'd put in front of an agent — interface, doc comment, and one failing table-driven test stub — before writing a single line of the implementation ourselves.

// coupon.go
package billing

import "context"

// ErrCouponExpired is returned by Redeem when the coupon's expiry
// date has passed. ErrCouponUnknown is returned when no coupon with
// that code exists. Both are safe to check with errors.Is.
var (
    ErrCouponExpired = errors.New("billing: coupon expired")
    ErrCouponUnknown = errors.New("billing: coupon unknown")
)

// CouponRedeemer applies a coupon code to an order total.
//
// Redeem looks up code (case-sensitive, no whitespace trimming —
// codes are generated uppercase-only) and, if valid and unexpired,
// returns the discounted total in cents. It never returns a
// negative total: a coupon that would discount below zero clamps
// to zero. Redeem does not mutate the coupon's use count; call
// MarkUsed separately once the order is confirmed. Safe for
// concurrent use.
type CouponRedeemer interface {
    Redeem(ctx context.Context, code string, totalCents int) (int, error)
}
// coupon_test.go
package billing

import (
    "context"
    "errors"
    "testing"
)

func TestRedeem(t *testing.T) {
    cases := []struct {
        name       string
        code       string
        totalCents int
        wantCents  int
        wantErr    error
    }{
        {"valid 20% off", "SAVE20", 1000, 800, nil},
        {"discount clamps at zero", "FREEBIE", 50, 0, nil},
        {"unknown code", "NOPE", 1000, 0, ErrCouponUnknown},
        {"expired code", "OLD10", 1000, 0, ErrCouponExpired},
    }
    for _, tc := range cases {
        t.Run(tc.name, func(t *testing.T) {
            var r CouponRedeemer = newTestRedeemer(t) // t.Fatal: not implemented
            got, err := r.Redeem(context.Background(), tc.code, tc.totalCents)
            if !errors.Is(err, tc.wantErr) {
                t.Fatalf("err = %v, want %v", err, tc.wantErr)
            }
            if got != tc.wantCents {
                t.Errorf("got %d cents, want %d", got, tc.wantCents)
            }
        })
    }
}

That's the spec, pasted whole into the context window. Here's the “and now your task” prompt that would go with it — short, on purpose, because everything load-bearing already lives in the code above:

Example prompt Implement newTestRedeemer and a concrete type satisfying CouponRedeemer in coupon.go, backed by the in-memory couponStore already in this package. Don't change the interface signature, the error variables, or any assertion in coupon_test.go. Run go test ./billing/... and don't stop until it's green.

Notice how little the prompt has to say. It doesn't explain what “expired” means, what rounding rule to use, or what to return on an unknown code — that's all already nailed down, mechanically, in the doc comment and the test table. The prompt's only job is to point at the target and describe the one implementation detail that isn't inferable from the spec (which store to use). Everything else is checked, not requested.

The Agent Angle Agents are eager to please, and prose specs give them room to please you with something other than what you meant. A failing test closes that room: there is exactly one way to make it pass, which is to satisfy the contract. If you notice yourself writing three paragraphs describing expected behavior in a chat prompt, stop and ask whether that paragraph could instead be a doc comment plus a test case. It almost always can, and the agent will treat “make this red test green without touching its assertions” as a far more disciplined goal than “please handle the coupon logic.”

Edge cases the format doesn't save you from

A word of caution before you start treating interfaces as magic: executable specs constrain the shape of the answer, not its quality. An agent can satisfy CouponRedeemer's signature and pass every case in the table above while still writing an O(n) linear scan over every coupon ever issued, or holding a lock longer than it needs to, or leaking the store's internal mutex through a subtle aliasing bug. The test table only covers the cases you thought to write down; a clever-but-wrong implementation can satisfy all of them and still be wrong on a case you didn't enumerate. That's not a flaw in the approach — it's a reminder that executable specs raise the floor, they don't replace the audit skills from Part I. Read the diff. Run go vet and the race detector from Chapter 6's feedback loop. The spec tells you the agent didn't cheat on the contract; it doesn't tell you the contract was complete.

The same caution applies to generated boilerplate: generation only helps if the schema is actually the source of truth. If someone hand-edits a .pb.go file, or the OpenAPI document drifts out of sync with a route someone added by hand, you've quietly reintroduced the exact problem schema-first design was supposed to solve — now with the added confusion of a generator that thinks it still owns a file it doesn't. Regenerate on every schema change, commit the generated output so drift is visible in the diff, and treat a hand-edit inside a generated file as a bug in your process, not a shortcut.

Reflection

The theme underneath all of this is one you've now seen from several angles across the book: Go gives you cheap, compiler-enforced checkpoints — a type signature, a package boundary, a doc comment convention — and an agent workflow gets dramatically more reliable the more of your intent you can push into one of those checkpoints instead of leaving it as prose the agent has to infer. Writing the interface first isn't extra ceremony bolted onto “real” development; it is the design work, done at the moment it's cheapest to do it — before code exists to disagree with it. The next chapter takes this one step further: instead of one failing test as a starting point, we'll build a whole red-green-refactor loop around the agent, table by table.

Audit Checklist

  • Did the agent get an interface and a failing test, not just a paragraph of prose describing the desired behavior?
  • Does every exported symbol in the spec have a doc comment that states pre-conditions, post-conditions, and the exact error/zero-value returned on failure?
  • Is the package boundary small enough that the agent's task didn't require it to read or guess at unrelated packages?
  • For any service boundary (HTTP/gRPC/SQL): is there an actual schema file (OpenAPI doc, .proto, CREATE TABLE) checked in, and was the boilerplate regenerated from it rather than hand-typed?
  • Does the generated code match the schema right now — no hand-edits inside a generated file, no route or field that exists in one but not the other?
  • Did passing the test table actually verify the contract, or only the handful of cases someone thought to write down — and does the diff still deserve a real read (go vet, race detector) on top of a green test run?
