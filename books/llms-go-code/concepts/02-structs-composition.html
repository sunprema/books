Chapter 2

Structs & Composition: Modeling Data Before the Agent Writes a Line

Here's a question worth sitting with before you write a single line of Go: if a colleague handed you a completely empty file and asked you to build a Service from scratch, where would you even start? Most of us don't start with behavior. We start by asking what do I need to know to do this job. That's a struct. Everything else — the methods, the interfaces, the goroutines — is just stuff that happens around the data. Get the data shape wrong, and no amount of clever code fixes it. Get it right, and the code almost writes itself.

No classes, no inheritance, no problem

If you're coming from Java, C++, Python, or TypeScript, the first thing that feels strange about Go is what's missing. There's no class keyword. There's no extends. There's no inheritance hierarchy to draw on a whiteboard. Go gives you exactly one tool for grouping data: the struct.

type Book struct {
    Title    string
    Author   string
    Pages    int
    Checked  bool
}

That's it. A struct is a named bag of fields, nothing more. You build values of it with struct literals:

b1 := Book{Title: "The Pragmatic Programmer", Author: "Hunt & Thomas", Pages: 352}

// positional form works too, but skips field names — fragile if the struct changes
b2 := Book{"Crafting Interpreters", "Nystrom", 352, false}

// zero-value struct: every field gets its type's zero value
var b3 Book // Title: "", Author: "", Pages: 0, Checked: false

Notice that b1's literal names its fields. That's the idiomatic form, and it's not just style — it's insurance. Add a field to Book next year and the positional literal in b2 silently shifts every value one slot to the left. The named form just picks up the new field's zero value and keeps going. Small detail, but it's exactly the kind of thing that separates code that survives refactors from code that doesn't.

So if Go has no inheritance, how do you reuse behavior across types? You don't model “is-a” relationships at all. You model “has-a” relationships, through composition. And Go has one elegant trick that makes composition feel almost as convenient as inheritance: embedding.

Composition, and the trick that makes it painless

Suppose every service in your codebase needs to log things. The naive approach is a named field:

type Service struct {
    logger Logger
    // ... other fields
}

func (s *Service) DoWork() {
    s.logger.Log("starting work") // have to go through the field name
}

That works fine, and it's honest about the relationship: a Service has a logger. But now drop the field name and embed the type instead — just declare the type with no field name of its own:

type Logger struct {
    prefix string
}

func (l *Logger) Log(msg string) {
    fmt.Printf("[%s] %s\n", l.prefix, msg)
}

type Service struct {
    *Logger          // embedded — no field name, just the type
    name    string
    timeout time.Duration
}

Now something interesting happens. Service doesn't just gain a Logger value tucked away inside it — it gains Logger's methods, promoted onto itself:

svc := &Service{Logger: &Logger{prefix: "svc"}, name: "billing"}
svc.Log("starting work") // no ".Logger." needed — the method was promoted

This is method promotion, and it's the entire reason embedding feels magical the first time you see it. The compiler is doing something quite mechanical: when it can't find a Log method directly on Service, it looks at the embedded field's type and checks there. One level of embedding, one level of lookup. There's no dynamic dispatch, no vtable, none of the machinery inheritance usually drags along. It's syntactic sugar for “forward this call to my embedded field” — nothing more.

That last sentence matters, so let's draw it.

One nuance worth internalizing right away: promotion follows the type you embed. Embed Logger by value and only methods with a value receiver on Logger get promoted onto a value Service (methods with a pointer receiver still get promoted, but only when the outer value is addressable). Embed *Logger, as we did above, and both value- and pointer-receiver methods come along, because a non-nil pointer can always reach either. It's a small mechanical rule, but it's exactly the kind of thing an agent will get subtly wrong if you don't specify it — more on that in a moment.

Value receivers vs. pointer receivers — the nuance everyone trips on

Every method in Go is attached to a type through a receiver — the thing before the method name in the func declaration:

func (b Book) String() string {          // value receiver — gets a copy of b
    return b.Title
}

func (b *Book) MarkChecked() {          // pointer receiver — gets a pointer to the original
    b.Checked = true
}

This looks like a cosmetic choice — a star or no star — but it changes real semantics, and it's one of the most commonly mis-applied nuances in the language, by humans and agents alike. Here's the working rule, straight from Go's own style guidance: mutate the receiver, or the struct holds something that shouldn't be copied (a sync.Mutex, for instance), or the struct is large enough that copying it is wasteful — use a pointer receiver. Otherwise a value receiver is fine. And then the rule that matters more than either of those individually: pick one and stay consistent across the whole type. Google's Go style guide puts it plainly — “prefer to make the methods for a type either all pointer methods or all value methods” — and when you're genuinely unsure, its advice is “when in doubt, use a pointer receiver.”

Why does mixing receiver types bite people? Because a value receiver method silently operates on a copy. Watch this go wrong:

func (b Book) MarkChecked() {   // value receiver — oops
    b.Checked = true       // mutates the copy, not the original
}

b := Book{Title: "Go in Action"}
b.MarkChecked()
fmt.Println(b.Checked) // false — the mutation never left the method

The call compiles. It runs without error. It just quietly does nothing, because b inside the method is a separate copy that gets thrown away when the function returns. This is the sneaky part: a mismatched receiver type doesn't crash, it just silently fails to do what you meant. That's exactly the kind of bug that's expensive precisely because nothing points at it.

The embedding trap: reuse is not the same as “has-a”

Embedding is seductive because it's free method promotion — a few keystrokes and your type suddenly has a dozen new methods. That's exactly what makes it dangerous. The question to ask before you embed something is never “does this save me from writing delegate methods?” It's “is this relationship actually true?” Does a Service genuinely have a Logger as part of its identity, or are you just borrowing its methods because writing func (s *Service) Log(msg string) { s.logger.Log(msg) } felt like too much typing?

Uber's widely-used Go style guide calls this out directly, and it's worth quoting because it's blunter than most style advice: embedded types “leak implementation details, inhibit type evolution, and obscure documentation.” An embedded type is, in their words, “rarely necessary — it is a convenience that helps you avoid writing tedious delegate methods.” Every embedded field you add is a field you can't freely remove or swap later without breaking every caller who (knowingly or not) relies on the promoted methods it brought along. Delegation — a named field plus a couple of explicit forwarding methods — costs you a few extra lines today and buys you the freedom to change your mind next quarter.

None of this means “never embed.” It means: embed when the has-a relationship is real and central to the type's identity (a http.Handler wrapping another http.Handler, a bytes.Buffer inside a type that is fundamentally a buffered writer), and reach for a plain named field plus explicit methods everywhere else. Embedding should read like a sentence: “a RetryingClient is built around an http.Client” — not “I wanted these five methods without typing them.”

The Agent Angle

Here's the practical payoff of everything above. When you hand an agent an empty problem — “build me a billing service” — you've handed it an open design space, and an LLM will cheerfully fill that space with invented fields, invented helper types, and invented relationships between them, because it has to produce something plausible. That's where hallucination and scope creep actually come from: not bad code generation, but an underspecified data model.

So flip the order. Design the struct first, yourself, and hand the agent a stub:

// You write this:
type Invoice struct {
    ID       string
    Cents    int64
    Currency string
    PaidAt   *time.Time // nil until paid
}

// TODO(agent): implement MarkPaid. Must be idempotent — calling it
// twice on an already-paid invoice should return an error, not panic.
func (inv *Invoice) MarkPaid(at time.Time) error {
    panic("not implemented")
}

Notice what you've already decided for the agent: Invoice uses cents as an int64 (no float-money bugs to invent), a pointer *time.Time to distinguish “unpaid” from “paid at the zero time” (no sentinel-date hack to invent), and a pointer receiver already declared on the stub (no receiver-consistency call to make). The agent's whole job shrinks to “write a method body that satisfies this exact signature and this one sentence of spec.” That's a task an LLM is very good at. “Invent a data model for a billing system” is a task it is not.

A worked example: stub in, method out

Let's make that concrete with something slightly bigger, combining embedding and receiver discipline in one shot. Say you want every “service” type in your codebase to log consistently, and you've decided (correctly, per the design test above) that a Service genuinely is built around a logger as part of its identity:

type Logger struct {
    prefix string
}

func (l *Logger) Log(msg string) {
    fmt.Printf("[%s] %s\n", l.prefix, msg)
}

type Service struct {
    *Logger
    name     string
    inflight int
}

// TODO(agent): implement Handle. It must:
//   1. increment inflight before doing work
//   2. decrement inflight when done, even on error (use defer)
//   3. call s.Log with a one-line summary of what happened
func (s *Service) Handle(req string) error {
    panic("not implemented")
}

An agent filling in Handle doesn't need to guess whether Service has a logger, what it's called, or how to reach it — s.Log(...) just works, because you already decided the shape. A reasonable fill-in looks like this:

func (s *Service) Handle(req string) error {
    s.inflight++
    defer func() { s.inflight-- }()

    if req == "" {
        s.Log("rejected empty request")
        return fmt.Errorf("empty request")
    }
    s.Log("handled: " + req)
    return nil
}

That's the whole trick. You didn't prompt the agent with paragraphs of prose about logging conventions — you prompted it with a type. The struct and the stub signature are the specification. We'll push this idea much further in Part II, when we look at interfaces and executable specifications as prompts in their own right. But it starts here, with getting the data shape right before anyone — human or agent — writes a method body.

Watch for An agent that embeds a type purely to inherit its methods, when the has-a relationship isn't real. A tell-tale sign: an embedded field whose promoted methods never get called through the receiver's own logic, only forwarded outward — that's delegation wearing an embedding costume, and it should be a named field with explicit methods instead.

Reflection

Structs feel almost too simple when you first meet them — just fields, no magic. But that simplicity is the point. Go refuses to give you inheritance because inheritance hides relationships behind a hierarchy you have to trace to understand. Composition keeps every relationship visible as a field. Embedding keeps the convenient parts of inheritance — not having to write forwarding boilerplate — without the parts that make large codebases hard to change. And once you start treating “design the struct first” as a discipline rather than an afterthought, you'll notice it does double duty: it makes your own code easier to reason about, and it gives an agent a fence to build inside instead of an open field to wander.

Challenges

  1. Take the Book struct from the top of this chapter and add a pointer-receiver Renew(days int) method. Write a value-receiver version by mistake on purpose, call it, and confirm the field doesn't change — then fix it.
  2. Design a Cache struct that embeds a sync.Mutex. Look up why Go's own style guidance singles out mutexes as a type that should never be embedded in an exported struct, and rewrite it as a named field instead.
  3. Write a stub-and-TODO prompt (like the ones in this chapter) for a RateLimiter struct with an Allow() bool method, hand it to an agent, and audit whether it added any fields you didn't ask for.

Audit Checklist

  • Unspecified fields. Diff the struct the agent produced against the one you specified — flag any exported field you didn't ask for, especially ones that smell like scope creep (a Cache map[string]any nobody requested).
  • Receiver consistency. Grep func (x Type) vs. func (x *Type) across every method on a type — a mix is either a deliberate, documented exception or a bug waiting to silently no-op.
  • Copy-and-mutate bugs. Any value-receiver method that assigns to a field and expects the caller to see the change is broken; go vet won't always catch this, so read the method body.
  • Embedding used as a shortcut. For every embedded type, ask “is this a real has-a relationship, or did the agent just want the methods?” If it's the latter, push back for a named field and explicit delegation.
  • Embedded mutexes or other non-copyable types exposed on an exported struct — these should never be embedded (and rarely even exported) because callers can copy the struct and silently duplicate the lock.
  • Struct literals gone positional. Agent-written struct literals should use named fields; positional literals silently break the moment the struct gains or reorders a field.
