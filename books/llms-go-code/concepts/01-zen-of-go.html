Part I · The Go Blueprint

Chapter 1

The Zen of Go: Why Boring Is a Superpower

Here's a question worth sitting with before you write a single line of Go: why does a language with no generics-heavy metaprogramming, no operator overloading, no class hierarchies, and a grand total of twenty-five keywords keep winning fans among people who build large, long-lived systems? The short answer is that Go was designed to be boring — deliberately, stubbornly boring. And it turns out that boring is exactly the property you want most when the thing writing your code isn't you anymore, but an agent.

A Small Vocabulary, On Purpose

Open the Go spec and you can read the entire keyword list on one hand’s worth of fingers, twice over: func, if, for, go, chan, defer, select, and a small handful more. There's one loop construct. One way to format a file — gofmt doesn't negotiate. There's no operator overloading, so + always means the same thing it meant last time you saw it. There's no implementation inheritance, so you can't accidentally call a method three superclasses up that quietly reshapes behavior underfoot. As one of the community's favorite Go Proverbs puts it, “gofmt's style is no one's favorite, yet gofmt is everyone's favorite” — a small, admitted sacrifice of personal taste in exchange for a codebase where every file looks like it was written by the same careful person.

Compare that to a language built around expressive power: C++ gives you templates, multiple inheritance, operator overloading, and increasingly elaborate metaprogramming facilities; Scala gives you implicits, higher-kinded types, and a type system flexible enough to encode entire DSLs. None of that is a knock on either language — those features exist because they solve real problems for real teams. But every one of them is also a place where two equally reasonable engineers can write functionally identical code that looks completely different, and where the “correct” idiom depends on which framework, which era, and which mentor you learned from. Go was designed, from Rob Pike's original “Less is exponentially more” talk onward, to minimize exactly that kind of surface area. Fewer ways to say the same thing means fewer things to guess at — for a human reviewer, and, as it turns out, for a language model too.

The Real Argument: Fewer Abstractions, Fewer Hallucinations

Here's the core claim of this chapter, and honestly of this whole book: the fewer "magic" abstractions a language offers, the less room a coding agent has to hallucinate. An LLM writing code is, at its core, predicting plausible-looking tokens based on patterns it has seen. Give it a language with a dozen ways to inject behavior implicitly — implicit conversions, macro expansion, deep inheritance chains, ambient context threaded through decorators — and you've given it a dozen plausible-looking but potentially wrong paths to wander down. Give it Go, where almost everything is an explicit function call passing explicit values, and there's a lot less surface for a confident-sounding fabrication to hide in.

This isn't just a vibe. A 2024 study, “We Have a Package for You!”, analyzed 576,000 code samples from sixteen LLMs and found they hallucinated non-existent packages in roughly 5.2% of Python samples and a striking 21.7% of JavaScript samples — over 205,000 unique hallucinated package names across the study. That's an agent confidently import-ing a library that was never real, because somewhere in its training data a package with a plausible-sounding name could have existed. Go isn't immune to this — an agent can still invent a function that doesn't exist in net/http — but Go stacks two defenses on top of the base rate: a comparatively tiny, extremely well-documented standard library (there just aren't that many packages to confuse each other with), and a compiler that refuses to build anything referencing a name it can't resolve. In a dynamically typed language, a hallucinated function call can sail through code review and blow up three weeks later in production. In Go, it dies at go build, in about four hundred milliseconds, with a file name and a line number. That's not a small difference in degree — it's a difference in when the failure becomes visible, and cheap failures that happen early are worth an enormous amount.

Key idea A hallucination isn't dangerous because it's wrong — humans write wrong code too. It's dangerous because of where it's caught. Go's strict compiler converts most hallucinated identifiers, wrong argument counts, and type mismatches into an immediate, unmissable build failure instead of a silent runtime surprise. The agent, and you, find out in seconds, not in an incident review.

The second half of the evidence is about what happens even when the code does compile and run. A January 2026 Stack Overflow analysis, “Are bugs and incidents inevitable with AI coding agents?”, scanned hundreds of open-source repositories and found AI-authored pull requests carried roughly 1.7× as many bugs as human-authored ones, with the gap worst in exactly the category you'd expect: logic and correctness errors, which showed up about 75% more often in AI code — 194 incidents per hundred PRs. A companion arXiv survey, “A Survey of Bugs in AI-Generated Code”, builds a taxonomy across eight major bug categories and finds functional and syntax bugs are the most common failure modes overall. Put the two findings together and a pattern emerges: agents are decent at producing code that looks plausible and often compiles, but weaker at the deeper logical reasoning that keeps it correct. That's precisely the gap a strict compiler, a small standard library, and a language with few implicit behaviors is built to narrow. It won't catch a logic bug for you — nothing catches all of those — but it will strip away the whole category of bugs caused by the agent guessing at how a feature behaves.

Spot the Smell: When an Agent Over-Engineers Go

Let's make this concrete. Ask an agent for “a user store,” and if its training skews toward Java or C#, you'll often get something like this — a generic interface built to wrap a single concrete type that will, in this codebase, only ever have one implementation:

// smells: interface + factory function for one implementation
type UserStore interface {
    GetUser(ctx context.Context, id string) (*User, error)
}

type userStoreImpl struct {
    db *sql.DB
}

func NewUserStore(db *sql.DB) UserStore {
    return &userStoreImpl{db: db}
}

func (s *userStoreImpl) GetUser(ctx context.Context, id string) (*User, error) {
    row := s.db.QueryRowContext(ctx, "SELECT id, name FROM users WHERE id = ?", id)
    var u User
    if err := row.Scan(&u.ID, &u.Name); err != nil {
        return nil, err
    }
    return &u, nil
}

Nothing here is broken. It compiles, it works, go vet is happy. But it's fighting the grain of the language. There's an interface with exactly one implementation, a constructor that returns the interface instead of the concrete type (hiding useful methods from callers and from your IDE), and a name, userStoreImpl, that exists purely to distinguish itself from an interface that didn't need to exist yet. The flat, idiomatic Go rewrite drops all of that ceremony:

type UserStore struct {
    db *sql.DB
}

func NewUserStore(db *sql.DB) *UserStore {
    return &UserStore{db: db}
}

func (s *UserStore) GetUser(ctx context.Context, id string) (*User, error) {
    row := s.db.QueryRowContext(ctx, "SELECT id, name FROM users WHERE id = ?", id)
    var u User
    if err := row.Scan(&u.ID, &u.Name); err != nil {
        return nil, err
    }
    return &u, nil
}

Same behavior, one fewer type, one fewer indirection to hold in your head. If you ever genuinely need a second implementation — a mock for tests, say — that's the moment to introduce the interface, and Go's implicit interfaces (more on those in the next chapter) mean you can do it later without touching this file at all.

The Agent Angle The interface-for-one-impl pattern is a small tell, but it's a reliable one: it means the agent reached for a habit from another ecosystem instead of reasoning about this codebase's actual needs. That matters for review speed, not just style. A flat struct has one call graph to trace when you're checking an agent's diff: caller → method → database. The wrapped version has three types to open across two files before you can answer “does this actually do what the PR description says?” Every unnecessary layer is a place a bug — or a second hallucinated method the agent invented on the interface but never wired to the impl — can hide in the gap between two files that are supposed to agree with each other. When you're reviewing at agent speed, flat code isn't a style preference. It's a safety property.

The same instinct shows up as deeply nested closures instead of named helper functions — an agent trying to be clever with a chain of func() func() func(){...} where three short, named top-level functions would read linearly and show up cleanly in a stack trace. The fix is the same fix every time: ask for the version with fewer named things wrapping other named things, and let each function say, in its name and its signature, exactly one thing about what it does.

Edge Cases: Boring Isn't the Same as Safe

It's worth being honest about the limits of this argument. A small language surface reduces one category of failure — syntactic hallucination, invented APIs, accidental misuse of an implicit feature — but it does not, by itself, prevent an agent from writing code that compiles cleanly and does the wrong thing. The Stack Overflow analysis cited above found AI-authored code carries more concurrency and dependency-correctness mistakes than human code, not fewer, and Go hands you two of the sharpest concurrency tools in the industry — goroutines and channels — which an agent can absolutely misuse into a deadlock or a data race that go build will never catch. (We'll spend all of Chapter 5 on exactly that.) Nor does a small stdlib make an agent immune to inventing a function that sounds like it should exist — strings.Contains and strings.HasSubstring are close enough in shape that an agent can confuse them, and Go's compiler will happily catch that one for you at build time, which is exactly the point: it turns a hallucination into a five-second fix instead of a five-day mystery.

Reflection

None of this makes Go a language where agents can't make mistakes. It makes Go a language where the cheap mistakes get caught immediately and for free, so the time you spend reviewing an agent's diff goes toward the mistakes that actually require your judgment — the logic error, the missed edge case, the race condition. Every chapter from here on is really about that same idea from a different angle: give the agent a small, explicit, well-lit room to work in, and spend your own attention on the load-bearing decisions the compiler can't make for you.

Challenges

  1. Take a Go file an agent generated for you recently. Count the number of interfaces that have exactly one implementation in the codebase. For each one, ask whether it earns its keep yet.
  2. Prompt an agent to write the same small feature (say, a rate limiter) once with no constraints, and once with the instruction “prefer the flattest possible design; introduce an interface only if you need two implementations.” Diff the results.
  3. Find a spot where an agent introduced a wrapper type purely to satisfy an interface. Try deleting the interface and see how much code actually breaks.
The Agent Angle When you're writing the prompt or the system instructions for a Go coding agent, it's worth stating the bias explicitly: “prefer concrete types; introduce an interface only when a second implementation exists.” It costs you one sentence and removes an entire category of needless indirection before it ever reaches your review queue.

Audit Checklist

  • Interfaces with one implementation. Grep for type X interface and check whether more than one concrete type actually satisfies it. If not, ask why it exists.
  • Constructors that return an interface. func New...() SomeInterface hides the concrete type's own methods from callers — usually a sign the interface arrived too early.
  • Deeply nested closures. More than one level of func() func(){} nesting is almost always clearer as named top-level functions.
  • Fresh imports and stdlib calls. Run go build and go vet immediately after any agent turn — a hallucinated function or package should die here, in seconds, not in code review.
  • Unfamiliar package names. Cross-check any newly imported third-party module against its real repository before trusting it; package-hallucination rates for JS/TS agents run into double digits.
  • “Clever” diffs. If a diff needs a paragraph of explanation to justify its structure, ask the agent for the flat version and compare.
