Chapter 3

Implicit Interfaces: Building Sandboxes for Agent-Generated Code

Here's a question worth sitting with for a second: when you hand an AI agent a task, how much of your codebase does it actually need to see? In most languages, the honest answer is “more than you'd like.” In Go, thanks to one quiet design decision, the answer can be nothing at all.

Motivation: the coupling problem

Say you're building a service that needs to notify someone when a job finishes — maybe a Slack message, maybe an email, maybe both eventually. You don't want to write the notification logic yourself; you want to hand that off to an agent. But handing off code usually means handing off trust: the agent needs to see your types, your internal helpers, maybe your database layer, just to figure out how its new code should plug in.

In a language like Java or C#, you'd typically declare a class as implements Notifier, which means the class and the interface have to agree on that relationship at compile time, by name, usually inside a chain of imports that ties them together. Coupling isn't a decision here so much as a consequence of the mechanism itself.

Go throws that mechanism out entirely. And once you feel the weight come off, it's hard to go back.

Design: satisfied, not declared

Go's official tour puts it about as plainly as language documentation ever does:

From the Go documentation “A type implements an interface by implementing its methods. There is no explicit declaration of intent, no ‘implements’ keyword.”

That's the whole mechanism. An interface in Go is a method set — a list of method signatures. Any concrete type whose methods happen to cover that list satisfies the interface, automatically, with no annotation anywhere. The Go language specification frames it the same way at a more formal level: a type is a member of an interface's type set purely because its method set is a superset of the interface's method set. No linkage, no registration, no shared import.

The tour goes on to name the consequence directly, and it's the consequence this whole chapter is built on:

From the Go documentation “Implicit interfaces decouple the definition of an interface from its implementation, which could then appear in any package without prearrangement.”

Without prearrangement. Read that phrase twice. It means the package that defines the interface and the package that satisfies it never have to agree on anything except a method shape. They can be written by different people, on different days, without either one importing the other. One of them can even be written by an agent that has never seen the other package's source code at all.

Implementation: the Notifier sandbox

Let's make this concrete. Suppose your core package — the one you, the human architect, own and review carefully — defines a small interface for sending a message somewhere:

// package core
package core

import (
	"context"
)

// Notifier delivers a message through some external channel.
// Core code depends only on this interface — never on a concrete sender.
type Notifier interface {
	Notify(ctx context.Context, msg string) error
}

type JobRunner struct {
	notifier Notifier
}

func NewJobRunner(n Notifier) *JobRunner {
	return &JobRunner{notifier: n}
}

func (r *JobRunner) Finish(ctx context.Context, jobID string) error {
	if err := r.doWork(ctx, jobID); err != nil {
		return err
	}
	return r.notifier.Notify(ctx, "job "+jobID+" finished")
}

That's the entire contract. One method, three arguments, one return value. Now here's the sandbox: you can prompt an agent with nothing but this snippet and the sentence “write a package that satisfies core.Notifier by posting to Slack.” The agent doesn't need read access to package core at all — it just needs the interface's shape.

// package slacknotify — written entirely by the agent
package slacknotify

import (
	"bytes"
	"context"
	"encoding/json"
	"fmt"
	"net/http"
)

type Sender struct {
	webhookURL string
	client     *http.Client
}

func New(webhookURL string, client *http.Client) *Sender {
	if client == nil {
		client = http.DefaultClient
	}
	return &Sender{webhookURL: webhookURL, client: client}
}

// Notify satisfies core.Notifier — implicitly. Nothing here imports
// or even mentions the core package.
func (s *Sender) Notify(ctx context.Context, msg string) error {
	body, err := json.Marshal(map[string]string{"text": msg})
	if err != nil {
		return fmt.Errorf("slacknotify: marshal payload: %w", err)
	}

	req, err := http.NewRequestWithContext(ctx, http.MethodPost, s.webhookURL, bytes.NewReader(body))
	if err != nil {
		return fmt.Errorf("slacknotify: build request: %w", err)
	}
	req.Header.Set("Content-Type", "application/json")

	resp, err := s.client.Do(req)
	if err != nil {
		return fmt.Errorf("slacknotify: send: %w", err)
	}
	defer resp.Body.Close()

	if resp.StatusCode >= 300 {
		return fmt.Errorf("slacknotify: webhook returned status %d", resp.StatusCode)
	}
	return nil
}

Wiring the two together happens in a third place entirely — typically your main.go or a small composition-root file, which is the only code in the whole program that imports both core and slacknotify:

func main() {
	sender := slacknotify.New(os.Getenv("SLACK_WEBHOOK_URL"), nil)
	runner := core.NewJobRunner(sender) // *slacknotify.Sender satisfies core.Notifier
	// ...
}

Notice what never happened: slacknotify never imported core. The agent that wrote it never needed the source, the tests, or even the design intent behind JobRunner. It needed one method signature. If tomorrow you want an emailnotify package too — maybe written by a second, entirely independent agent run — it plugs in exactly the same way, with exactly the same zero-trust boundary.

The Agent Angle This is the sandboxing move: define the interface yourself, in the package you trust, and hand the agent only that signature plus a one-line description of the behavior you want. The agent's output lives in its own package, compiles independently, and can be reviewed, tested, and even deleted without touching a single line of your core logic. You never had to paste your business logic into a prompt, and the agent never had to be trusted with it. If the agent's output is wrong, the blast radius is exactly one package.

Design discipline: keep the sandbox small

This trick has one failure mode, and it's not subtle once you see it: a bloated interface. If Notifier grew to twelve methods — Notify, NotifyBatch, Configure, HealthCheck, RetryPolicy, and so on — you'd be handing the agent a much bigger surface to guess wrong about, and you'd be quietly leaking implementation assumptions from core into the interface itself. The sandbox walls only stay thin if the contract inside them stays small.

Go's own community has a proverb for exactly this, from Rob Pike's Go Proverbs talk: “The bigger the interface, the weaker the abstraction.” The standard library backs this up by example rather than lecture — io.Writer has exactly one method, and that's precisely why hundreds of unrelated types across the ecosystem can satisfy it without ever coordinating.

Effective Go's naming convention nudges toward the same discipline: “one-method interfaces are named by the method name plus an -er suffix or similar modification to construct an agent noun: Reader, Writer, Formatter, CloseNotifier.” If you can't name your interface that way, that's often a sign it's trying to do too much.

There's a companion proverb worth pinning next to your editor while you're writing prompts for agents: “accept interfaces, return structs.” Functions that take a Notifier parameter stay flexible about who's calling them; functions that return a concrete struct keep the caller's code simple, since the caller doesn't have to type-assert its way back to whatever extra methods that concrete type offers. Notice that NewJobRunner above follows exactly this shape: it accepts a Notifier interface, and it returns a concrete *JobRunner.

Edge cases

A few things worth knowing before you rely on this pattern in a real review:

  • Pointer vs. value receivers matter. If Notify is defined with a pointer receiver (func (s *Sender) Notify(...)), then only *Sender satisfies Notifier — a bare Sender value does not. Agents sometimes mix receiver types across a single struct's methods, which silently breaks satisfaction. The compiler will catch it the moment you try to assign the value to the interface, but only at that assignment site, not where the method was written.
  • Compile-time verification is cheap insurance. Because satisfaction is implicit, nothing forces the compiler to check it until something actually assigns the type to the interface variable. It's common — and worth asking an agent to include — to add a blank-identifier assertion near the type definition: var _ core.Notifier = (*Sender)(nil). That one line fails to compile the instant the method set drifts, long before any runtime wiring would have caught it.
  • Implicit doesn't mean invisible. Reviewing agent output still means checking that the method signature matches exactly — including the context.Context parameter and the error return. An agent that writes Notify(msg string) error without the context parameter has written a method that looks right and satisfies nothing.

Reflection

It's worth noticing how much of this chapter was really about trust boundaries, not syntax. Go's implicit interfaces let you draw a wall around exactly the amount of code an agent needs to see — a method signature and nothing more — and the compiler enforces that wall for you. You're not writing a design document explaining the contract; the interface is the contract, machine-checked. The discipline you bring is keeping that contract small enough that it stays a real boundary instead of a leaky one.

Audit Checklist

  • No reverse imports. Confirm the agent's adapter package doesn't import your core package (or anything upstream of it) — the whole safety property depends on the dependency arrow pointing one way.
  • Method set matches exactly. Check parameter types, order, and the error return against the interface definition line by line; a near-miss signature compiles fine right up until someone tries to wire it in.
  • Receiver type is consistent. Make sure the agent didn't mix pointer and value receivers across the type's methods in a way that breaks interface satisfaction.
  • No exported surface creep. Watch for the agent exporting extra public methods, fields, or a whole second constructor “just in case” — each exported symbol is a bit of implementation detail now leaking past the sandbox wall.
  • Interface itself hasn't grown. If the agent proposed adding a method to Notifier to make its own implementation easier, treat that as a design decision for you to make, not something to accept silently in a diff.
  • Compile-time assertion present. Look for (or add) a var _ core.Notifier = (*Sender)(nil) line so a future signature drift fails the build instead of failing silently at runtime.
