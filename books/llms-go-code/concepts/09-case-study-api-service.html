Chapter 9

Case Study: Building an API Service with a Human Architect

Eight chapters ago we started with plain structs. Along the way we picked up implicit interfaces, honest errors, context-aware concurrency, a toolchain that argues back, and a way to turn all of that into a specification an agent can't wiggle out of. This chapter is Part III's opening move — “Built With an Agent” — where none of those ideas stay theoretical anymore. We're going to build one small, completely real REST resource: a bookmarks service. You, the human, will design the data, the route, and the interface. Then you'll hand an agent a failing test and watch it write the rest.

Motivation: who draws the property line?

Here's the question this whole case study answers: when a human and an agent build an API endpoint together, who decides what, exactly? Get this division wrong and you end up in one of two bad places. Either you're back to typing every line yourself, agent along for the ride doing nothing, or you're pasting a one-sentence prompt into a chat window and hoping the shape of the JSON matches what your frontend expects. Neither is the point of this book.

The division that actually works, and the one this chapter walks through end to end, is the one we've been building toward since Chapter 2: the human owns the contract, the agent owns the fill-in. Structs and interfaces (Chapters 2 and 3) give us a contract with a compiler behind it. Errors as values (Chapter 4) and context (Chapter 5) tell us what that contract has to look like to be trustworthy. Executable specifications (Chapter 7) and test-driven agent development (Chapter 8) give us a way to hand that contract to an agent without writing a paragraph of prose that it can quietly misread. This chapter is what happens when you run all of that on one ticket: “add an endpoint to save a bookmark.”

Design: the contract the human writes

Before any agent sees a prompt, we do the part that stays ours: decide what a bookmark is, and what a caller can do to it. That's a data-modeling exercise, straight out of Chapter 2 — small, explicit structs, no premature abstraction.

// package bookmarks
package bookmarks

import (
	"context"
	"time"
)

// Bookmark is the resource we expose over the API and persist in a Store.
type Bookmark struct {
	ID        string    `json:"id"`
	URL       string    `json:"url"`
	Title     string    `json:"title"`
	CreatedAt time.Time `json:"created_at"`
}

// CreateBookmarkRequest is the wire shape a client sends us.
// It is deliberately narrower than Bookmark — callers don't get to set
// an ID or a CreatedAt; the server decides those.
type CreateBookmarkRequest struct {
	URL   string `json:"url"`
	Title string `json:"title"`
}

Next, the interface. This is the single most important artifact the human produces this chapter — it's the sandbox wall from Chapter 3, drawn around exactly the storage operations a handler needs and nothing else:

// Store persists bookmarks. Core handler code depends only on this
// interface — never on a concrete database.
type Store interface {
	Get(ctx context.Context, id string) (Bookmark, error)
	Create(ctx context.Context, b Bookmark) (Bookmark, error)
	List(ctx context.Context) ([]Bookmark, error)
}

// ErrNotFound is the sentinel a Store returns when Get can't find the id.
var ErrNotFound = errors.New("bookmarks: not found")

Notice the fingerprints of earlier chapters all over these eleven lines. context.Context comes first in every method, per Chapter 5 — a Store call that talks to a database or the network has no business ignoring cancellation. Errors come back as values, not exceptions, and ErrNotFound is a sentinel the handler can check with errors.Is, per Chapter 4. And the interface itself is small — three methods, each nameable, each obviously necessary — because a bloated interface is a bloated sandbox.

Finally we design the route, in a one-line spec any agent (or teammate) can read without ambiguity: POST /bookmarks accepts a CreateBookmarkRequest body and returns 201 Created with the saved Bookmark, or 400 Bad Request if the URL is missing or the body doesn't parse.

Design: the failing test is the handoff

Chapter 8 argued that a prose description is a lossy way to brief an agent, and a failing test is a lossless one. This is where that pays off. Instead of writing three paragraphs about status codes, we write the test we want to see turn green, using a hand-rolled stub that satisfies our own Store interface — no mocking framework required, just a struct with a function field:

type stubStore struct {
	createFn func(ctx context.Context, b Bookmark) (Bookmark, error)
}

func (s *stubStore) Get(ctx context.Context, id string) (Bookmark, error) {
	return Bookmark{}, ErrNotFound
}
func (s *stubStore) Create(ctx context.Context, b Bookmark) (Bookmark, error) {
	return s.createFn(ctx, b)
}
func (s *stubStore) List(ctx context.Context) ([]Bookmark, error) {
	return nil, nil
}

func TestCreateBookmarkHandler(t *testing.T) {
	tests := []struct {
		name       string
		body       string
		store      *stubStore
		wantStatus int
		wantBody   string
	}{
		{
			name: "valid bookmark is created",
			body: `{"url":"https://go.dev","title":"Go"}`,
			store: &stubStore{createFn: func(ctx context.Context, b Bookmark) (Bookmark, error) {
				b.ID = "bm_1"
				return b, nil
			}},
			wantStatus: http.StatusCreated,
			wantBody:   `"id":"bm_1"`,
		},
		{
			name:       "missing url is rejected",
			body:       `{"title":"No URL"}`,
			store:      &stubStore{},
			wantStatus: http.StatusBadRequest,
		},
		{
			name:       "malformed json is rejected",
			body:       `{"url": `,
			store:      &stubStore{},
			wantStatus: http.StatusBadRequest,
		},
	}

	for _, tt := range tests {
		t.Run(tt.name, func(t *testing.T) {
			req := httptest.NewRequest(http.MethodPost, "/bookmarks", strings.NewReader(tt.body))
			rec := httptest.NewRecorder()

			NewCreateBookmarkHandler(tt.store).ServeHTTP(rec, req)

			if rec.Code != tt.wantStatus {
				t.Fatalf("status = %d, want %d", rec.Code, tt.wantStatus)
			}
			if tt.wantBody != "" && !strings.Contains(rec.Body.String(), tt.wantBody) {
				t.Errorf("body = %s, want substring %q", rec.Body.String(), tt.wantBody)
			}
		})
	}
}

That's the whole handoff. No architecture doc, no chat transcript — just bookmark.go (structs and interface), handler_test.go (the test above, currently red because NewCreateBookmarkHandler doesn't exist yet), and the instruction: “make this pass.” The test pins down status codes, JSON shape, and validation behavior all at once — three things a prose prompt would need several careful sentences to nail down, and would still leave room to misread.

Implementation: the agent writes the handler

Handed that interface and that test, here's a reasonable, idiomatic handler an agent can produce — and did, in our own dry run of this exact prompt:

func NewCreateBookmarkHandler(store Store) http.Handler {
	return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
		var req CreateBookmarkRequest
		if err := json.NewDecoder(r.Body).Decode(&req); err != nil {
			http.Error(w, "invalid JSON body", http.StatusBadRequest)
			return
		}
		if req.URL == "" {
			http.Error(w, "url is required", http.StatusBadRequest)
			return
		}

		b := Bookmark{URL: req.URL, Title: req.Title, CreatedAt: time.Now()}
		saved, err := store.Create(r.Context(), b)
		if err != nil {
			http.Error(w, "could not save bookmark", http.StatusInternalServerError)
			return
		}

		w.Header().Set("Content-Type", "application/json")
		w.WriteHeader(http.StatusCreated)
		json.NewEncoder(w).Encode(saved)
	})
}

Run go test ./... and the three cases go green. Notice the three habits the test forced into existence: the decode error is checked and turned into a 400, the empty-URL case is checked before anything touches the store, and r.Context() — not context.Background() — is what gets threaded into store.Create. None of that was spelled out in a design doc. It fell out of a test the human already had reason to write.

Implementation: the agent writes the Store

The same prompt, extended with “now implement Store in memory, safe for concurrent use,” produces something like this:

type MemoryStore struct {
	mu   sync.Mutex
	data map[string]Bookmark
	next int
}

func NewMemoryStore() *MemoryStore {
	return &MemoryStore{data: make(map[string]Bookmark)}
}

func (s *MemoryStore) Create(ctx context.Context, b Bookmark) (Bookmark, error) {
	s.mu.Lock()
	defer s.mu.Unlock()
	s.next++
	b.ID = fmt.Sprintf("bm_%d", s.next)
	s.data[b.ID] = b
	return b, nil
}

For a real deployment, the SQL-backed twin implements the same three methods against a database. Get and Create touch one row, so they're natural fits for QueryRowContext. List is the one that returns multiple rows, and it's worth showing in full because its cleanup is exactly the kind of thing an agent gets right when it's copying an idiom it's seen a thousand times, and wrong when it isn't paying attention:

func (s *SQLStore) List(ctx context.Context) ([]Bookmark, error) {
	rows, err := s.db.QueryContext(ctx, `SELECT id, url, title, created_at FROM bookmarks`)
	if err != nil {
		return nil, fmt.Errorf("bookmarks: list: %w", err)
	}
	defer rows.Close()

	var out []Bookmark
	for rows.Next() {
		var b Bookmark
		if err := rows.Scan(&b.ID, &b.URL, &b.Title, &b.CreatedAt); err != nil {
			return nil, fmt.Errorf("bookmarks: scan: %w", err)
		}
		out = append(out, b)
	}
	return out, rows.Err()
}

Three lines are doing quiet, load-bearing work here: defer rows.Close() right after the error check, so a mid-loop return still releases the connection back to the pool; and rows.Err() at the very end, because rows.Next() returning false means either “done” or “something broke,” and only Err() tells you which.

Edge cases: where this exact scenario goes wrong

A green test suite for the happy path is not the same thing as a safe handler. In practice, when we've watched agents build handlers like this one, the same five mistakes show up again and again — and they're worth naming precisely, because a table-driven test rarely catches them unless someone thought to write a case for them.

MistakeWhat it looks likeWhy the happy-path test misses it
Ignoring request cancellationA slow handler calls context.Background() instead of r.Context(), or never checks ctx.Err() before expensive workThe test client never disconnects mid-request
Unclosed response bodyThe handler calls another service with http.Get or a bare client.Do and never defer resp.Body.Close()One test run doesn't leak enough connections to notice
Unclosed sql.RowsA List-style method calls QueryContext but skips defer rows.Close()An in-memory stub store never touches a real connection pool
Swallowed decode errorsjson.NewDecoder(r.Body).Decode(&v) with the error thrown away or logged and ignoredTest bodies are always well-formed unless someone wrote a malformed-JSON case
Missing validation before the store callAn empty or malformed field reaches store.Create, which either panics or silently persists garbageTable-driven tests only cover the inputs someone thought to list

The first two deserve a closer look, because they compound: imagine we extend the handler to enrich a bookmark by fetching its page title from the URL the caller submitted. That means the handler is now itself an HTTP client, calling someone else's server, and both mistakes above can hide in the same four lines.

Spot the bug. Here's a first draft an agent might hand back:

func (h *Handler) fetchTitle(ctx context.Context, url string) (string, error) {
	resp, err := http.Get(url) // bug: ignores ctx entirely
	if err != nil {
		return "", err
	}
	body, err := io.ReadAll(resp.Body) // bug: resp.Body is never closed
	if err != nil {
		return "", err
	}
	return parseTitle(body), nil
}

It compiles, it passes a happy-path test against a local test server, and it will slowly leak a file descriptor and an idle connection for every bookmark saved — and it will never notice when the parent request is canceled, because http.Get has no idea a context even exists. Here's the fix, and it's smaller than the bug:

func (h *Handler) fetchTitle(ctx context.Context, url string) (string, error) {
	req, err := http.NewRequestWithContext(ctx, http.MethodGet, url, nil)
	if err != nil {
		return "", err
	}
	resp, err := h.client.Do(req)
	if err != nil {
		return "", err
	}
	defer resp.Body.Close()

	body, err := io.ReadAll(resp.Body)
	if err != nil {
		return "", err
	}
	return parseTitle(body), nil
}

http.NewRequestWithContext ties the outbound call's lifetime to the inbound request's; if the original caller hangs up, this call gets canceled instead of running to completion for no one. And defer resp.Body.Close(), placed immediately after the error check that guarantees resp is non-nil, closes the connection whether the function returns cleanly or through an early error path below it. Neither line changes what the function returns on the happy path — which is exactly why a happy-path test can't tell the two versions apart.

The Agent Angle Notice that none of these five mistakes are exotic. They're all places where the “obviously correct” code and the “actually correct” code produce identical output on the request the agent tested against, and only diverge under conditions — a canceled request, a malformed body, a thousandth call in a row — that a quick manual check doesn't create. That's precisely why this book leans so hard on tests and audits instead of trusting a glance at the diff. Ask the agent directly: “show me the test case where the client disconnects mid-request” or “show me the test where the JSON body is truncated.” If it can't produce one, that's your cue to write it yourself before merging.

Reflection

Step back and look at what actually happened in this chapter. We didn't invent a new methodology. We took the struct discipline from Chapter 2, the sandboxed interface from Chapter 3, the sentinel errors from Chapter 4, the context-first signatures from Chapter 5, and the red-test handoff from Chapters 7 and 8, and pointed all of it at one ordinary endpoint. The agent wrote real code — a working handler, a working store, a passing test suite — and it did so without ever seeing the rest of our codebase, because the interface was the only door we opened. The mistakes it could still make weren't mistakes of syntax; they were mistakes of omission, in exactly the corners a green test suite doesn't shine a light into. That's the shape of the job now: design the contract narrowly enough that an agent can't misunderstand it, and audit narrowly enough that you're checking the corners it's most likely to skip.

Audit Checklist

  • Context flows through. Every Store call inside the handler uses r.Context(), not context.Background() or context.TODO() — and any outbound call the handler makes uses http.NewRequestWithContext with that same context.
  • Every response body is closed. If the handler is itself a client of another service, grep for .Do( or http.Get( and confirm a defer resp.Body.Close() sits immediately after the nil-error check.
  • Every sql.Rows is closed and checked. A defer rows.Close() right after the query, and a rows.Err() check after the loop — not just a rows.Scan() error check inside it.
  • Decode errors are handled, not discarded. json.NewDecoder(r.Body).Decode(&v)'s error return must reach an http.Error call, never a blank identifier or a bare log line the caller never sees.
  • Validation happens before the Store call. Required fields are checked and rejected with 400 before store.Create or store.Get is ever reached — the store should never be the thing catching bad input.
  • The test suite has an unhappy path for each of the above. A canceled context, a truncated body, a missing field — if the table-driven test doesn't cover it, assume the agent didn't either.
