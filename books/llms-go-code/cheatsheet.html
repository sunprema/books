Quick Reference

The One-Page Audit

Every checklist from every chapter, condensed. Pin this next to the diff when you're reviewing what an agent handed you.

The toolchain loop, first Run this before you read a single line by eye: 
gofmt -l . && go vet ./... && go build ./... && go test -race ./...
A hallucinated import or function fails at go build — cheap and loud, not a silent runtime surprise.

1 · The Zen of Go

Audit Checklist

  • No needless generics/wrapper-interfaces around a single implementation.
  • No deeply nested closures where a flat sequence of statements would do.
  • gofmt -l is clean — formatting is not a matter of taste here.

2 · Structs & Composition

Audit Checklist

  • No exported fields the spec didn't ask for.
  • Embedding used for genuine “has-a”, not just to reuse a method for free.
  • Receiver type (value vs. pointer) is consistent across a type's method set.

3 · Implicit Interfaces

Audit Checklist

  • The interface stayed small — the agent didn't widen it to fit its implementation.
  • The adapter package doesn't import back into core.
  • No unrelated exported methods leaking implementation detail through the interface.

4 · Errors as Values

VerbEffect
%wwraps — visible to errors.Is / errors.As
%vflattens to text — caller can't unwrap it

Audit Checklist

  • Grep for _ = hiding a returned error.
  • Every err != nil branch does something meaningful.
  • %w used wherever a caller needs to unwrap; sentinel errors compared with errors.Is, not ==.

5 · Goroutines, Channels & Context

Audit Checklist

  • Every goroutine has a clear owner and shutdown path.
  • Every context.WithCancel/WithTimeout has its cancel deferred.
  • go test -race actually runs in CI — not just locally, once.
  • No blocking channel op inside a select without a ctx.Done() escape.

6 · The Go Toolchain

Audit Checklist

  • gofmt -l / go vet / go build / go test -race gate every agent-authored change.
  • pprof sanity-checked before anything performance-sensitive ships.
  • go.mod is committed and reviewed like any other diff.

7 · Executable Specifications

Audit Checklist

  • The agent got an interface + a failing test, not just a paragraph of prose.
  • Package boundary is small — the agent didn't need to guess at unrelated packages.
  • Generated code actually matches the schema (OpenAPI/proto/SQL) it was generated from.

8 · Test-Driven Agent Development

Audit Checklist

  • The test table includes negative/edge cases — written before the agent saw the task.
  • Subtests use t.Run with descriptive names.
  • Mocking goes through a small interface, not a real DB/API dependency.
  • Loop-variable capture is correct for the target Go version if t.Parallel() is used.

9 · Case Study: an API Service

Audit Checklist

  • Handler respects request cancellation via r.Context().
  • Response bodies, rows, and files are closed — check every defer ... Close().
  • JSON decode errors aren't swallowed; input is validated before it reaches the store.

10 · Case Study: a CLI Tool

Audit Checklist

  • Real logic lives in a testable run(args []string) error, not scattered through main.
  • os.Exit only ever called from main — never inside library code (deferred cleanup won't run otherwise).
  • Errors go to stderr, normal output to stdout; -h/usage behaves the way users expect.
