Chapter 09 · The Grown-Up Chapter

Trust & Guardrails: Where the Liability Lives

Everything that makes Pi a great opportunity — four powerful tools, full system-access extensions, reach into real machines — is also what can get you sued, breached, or shut down. An opportunist who skips this chapter isn't bold; they're a future incident report. Here is the single most important fact to internalize, stated without softening: Pi has no built-in permission system for restricting filesystem, process, network, or credential access. The guardrails are yours to build.

Read this twice

By default, a Pi agent with Bash and full-access extensions can do anything the process it runs as can do — read any file, run any command, reach any network the host can. There is no sandbox turned on for you. Trust in a Pi deployment is constructed, not inherited.

Sandboxing is a deployment choice: three patterns

Because the permission system isn't in the harness, containment lives in how you deploy it. Three documented patterns, in rough order of how much you're carrying:

PatternWhat it gives youReach for it when
GondolinA Pi extension built for sandboxing the agentYou want containment expressed in Pi's own terms
DockerStandard container isolation — filesystem, network, resource limitsYour team already lives in containers
OpenShellA constrained execution boundary for the shellYou need to bound what Bash can touch

Supply-chain hardening is already in the box

Not everything is left to you. Pi's own security posture includes supply-chain hardening you inherit: pinned dependencies, an npm shrinkwrap to lock the full tree, and that --ignore-scripts flag on install (recall npm install -g --ignore-scripts @earendil-works/pi-coding-agent) which blocks arbitrary install-time script execution — a common supply-chain attack vector. It's a meaningful signal about how the project thinks, and a baseline you should match in your own package.

The don't-get-burned checklist

  • Sandbox by default. Never run an agent with real reach outside a container boundary you control (Gondolin / Docker / OpenShell).
  • Gate every write. Reads can be liberal; anything that mutates state, posts, restarts, or spends money passes a permission gate with human approval until proven safe.
  • Audit everything. Every tool call logged, attributable, replayable. In regulated domains the audit trail is the product.
  • Scope the data. The agent sees only what its job needs — minimum-necessary, enforced by the container and the retrieval tools, not by good intentions.
  • Own the keys. BYOK keeps the data path yours; treat the keys as the crown jewels and scope them per-deployment.
  • Match the supply-chain hygiene. Pin your deps, shrinkwrap, --ignore-scripts, and review what your extensions pull in — they run with full access.
The honest close

Pi's minimalism is a deal: it gives you maximum reach and hands you the responsibility for containing it. That's the right trade for a serious builder, but only if you take the second half as seriously as the first. The opportunities in this book are real. So is the liability. The builders who win are the ones who make the guardrails part of the product, not an afterthought. Now go find your vertical.
