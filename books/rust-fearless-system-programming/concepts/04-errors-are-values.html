Concept 04 · Failure

Errors Are Values, Not Surprises

Most languages treat failure as a trapdoor that can open under any line. Rust treats it as a value you're holding — and can't pretend you aren't.

Think about how exceptions actually feel. You call a function, and somewhere three layers down it throws, and control teleports up your call stack to whatever catch happens to be waiting — maybe one you wrote, maybe none at all. The failure is invisible in the function's signature. You cannot tell, by reading readFile(path), whether it might explode. You just have to know, or find out in production.

Rust throws that whole model out. A function that can fail says so, right in its return type, by handing back a Result: either Ok(value) if it worked, or Err(problem) if it didn't. Failure isn't a hidden trapdoor — it's a package the function returns to you, and the type system won't let you get at the good stuff inside without acknowledging the bad case exists.

🔑
The core idea

A fallible function returns Result<T, E>. The error is a normal value that flows through normal code — no invisible control flow, no surprise teleports. If a signature returns Result, you know it can fail, just by looking.

The ? operator: honest and ergonomic

"Okay," you say, "but if I have to hand-check every single error, my code will be a swamp of if statements." Right you are — and that's exactly the trap that makes people cheat and ignore errors in other languages. Rust's fix is one of the great little pieces of language design: the ? operator.

rust · "? " = if this failed, return the error; else unwrap ituse std::fs;

fn read_config() -> Result<String, std::io::Error> {
    let raw = fs::read_to_string("config.toml")?;
    let trimmed = raw.trim().to_string();
    Ok(trimmed)
}

That little ? means: "if this was an Err, stop and return it up to my caller; otherwise, give me the value inside and carry on." You get the tidy, happy-path reading flow of exceptions, but with a crucial difference — the ? is visible in the code, and the Result is visible in the signature. Nothing is hidden. You handled the error and your function still reads top-to-bottom.

🤖
Why this is gold for agent workflows

An LLM's classic sin in other languages is the empty catch {} that swallows a failure and moves on, leaving you to debug a ghost. In Rust it cannot quietly ignore an error — the unused Result lights up a warning, and it can't reach the value without addressing the failure. The language nudges even a careless generator toward honest error handling.

Two kinds of wrong: recoverable vs. "the world is broken"

Rust draws a clean line that other languages smudge. There are problems you expect and should handle — a file's missing, the network blipped, the user typed nonsense. Those are Result. Then there are problems that mean a core assumption of your program is violated — an index way out of bounds, an invariant you swore could never break. Those panic: the program stops loudly rather than limping on with corrupted logic.

SituationToolMeaning
File not found, bad input, timeoutResultexpected; handle it
"This can't happen" happenedpanic!a bug; stop now
Value might be absentOptionno error, just maybe-nothing

The point isn't the vocabulary — it's the discipline. By separating "failures I planned for" from "my program is now lying to itself," Rust keeps you from the twin sins of crashing on routine hiccups and soldiering on through genuine corruption.

Rich errors, when you want them

Real programs stitch together many failure types — a network error here, a parse error there. The ecosystem has two beloved crates for this: thiserror to define your own tidy error enums for a library, and anyhow to just say "anything could go wrong here, give me a good message and a backtrace" for an application. You don't need them to start; know they exist for when you do.

🧭
The concept to carry, not the crates

What matters isn't anyhow vs thiserror — the agent can wire those up. What matters is that you decide, per function: is this a failure the caller should handle (Result), or a broken assumption that should halt the world (panic!)? That judgment call is yours; the plumbing is the agent's.

✅
The quiet superpower

Because errors are ordinary values, they compose with everything: you can collect them, map them, retry on them, log them, turn a whole batch's worth into one summary. No special exception machinery — just data flowing through your program like any other data.
