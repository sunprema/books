Concept 08 · The Payoff

Vibe-Coding Rust with Agents

Here's the thesis of the whole book, finally out loud: Rust and coding agents are unusually good partners, because the language's strictness is exactly the guardrail an agent needs.

There's a worry people have about letting an AI write Rust: "isn't Rust the hard language? Won't the model just flail against the borrow checker?" It's a fair question with a surprising answer. Yes, Rust is strict — and that strictness is precisely why the pairing works. A loose language lets an agent's mistakes through, to be discovered later by your users. Rust catches them at the door, in specific language, in seconds.

Think about what a coding agent fundamentally is: a very fast writer that learns from feedback. Its output is only as good as the feedback it gets. And Rust provides the richest, strictest, most machine-readable feedback of any mainstream language — a compiler that catches whole bug classes, a linter that suggests idioms, a test runner, all through one uniform tool. You're not fighting the compiler with the agent. You're aiming the compiler at the agent's work.

🔑
The core insight

An agent's weakness is that it can be confidently wrong. Rust's superpower is refusing to compile confident wrongness of the memory-, type-, and concurrency- kind. Put them together and the machine polices the machine — while you stay up at the level of design.

The division of labor

The winning mental model: you are the architect, the agent is the builder, and the compiler is the inspector. Each does what it's best at.

Give to the agent

  • Boilerplate & scaffolding
  • Syntax you'd have to look up
  • Wiring crates together
  • Translating a clear spec to code
  • Fixing compiler & clippy complaints
  • Writing the first draft of tests

Keep for yourself

  • How the data is modeled
  • What the error cases are
  • Whether Rust is even right (ch. 9)
  • Security & correctness boundaries
  • Reading & approving the result
  • The unsafe decision (see right)
🤖
The concepts in this book ARE your steering wheel

Every chapter you've read is a lever you can pull on the agent: "model this as an enum" (ch. 3), "return a Result, don't panic here" (ch. 4), "make this a newtype so the units can't mix" (ch. 3), "parallelize this — the compiler will keep it race-free" (ch. 5). You don't need the syntax. You need the instinct for which instruction to give.

A recipe that actually works

Concretely, here's a loop I'd trust with a real feature. Notice how little of it is about typing Rust:

  1. Describe the shape, not the code. "A CLI that reads a folder of JSON logs, counts errors per service, prints a table. Model a LogEntry so a malformed line can't become a valid entry."
  2. Let the agent scaffold with Cargo — cargo new, add serde and clap, write the types and the logic.
  3. Make it run the loop: "Run cargo build, cargo clippy, and cargo test, and fix everything until all three are clean." This is where Rust shines — the agent iterates against a strict, honest oracle.
  4. You review the design, not the semicolons. Are the types honest? Are errors handled or swallowed? Is anything unsafe? Those are the questions from Chapters 2–5.
  5. Ask for tests of the nasty cases you thought of that the agent didn't — the empty folder, the giant file, the Unicode filename.
✅
Why the loop converges

Because each iteration's feedback is specific and correct, the agent tends to converge rather than thrash. "Cannot borrow as mutable" has one right fix, and the compiler nearly names it. Contrast a dynamically-typed language, where the same bug surfaces as a mystery at runtime, if at all.

The two things to watch like a hawk

⛔
1 · The unsafe escape hatch

Rust has an unsafe keyword that switches off the safety guarantees for a block — needed for a few low-level things, but it's where all of Rust's promises can be voided. If an agent reaches for unsafe to make a borrow-checker complaint "go away," that's a red flag, not a solution. Insist it solve the problem in safe Rust unless there's a real, explained reason. Grep your agent's output for unsafe and interrogate every hit.

⚠️
2 · "It compiles" ≠ "it's correct"

The compiler proves your program is memory- and type-safe. It does not prove your logic is right — a race-free, leak-free program can still compute the wrong average. So the compiler retires a huge class of bugs and frees you to spend your review on the one that's left: does this actually do the right thing? That's still a human's call. Tests and your judgment cover the gap.

That's the honest picture. Rust doesn't make agents infallible; it makes their failures cheap and loud instead of expensive and silent. You trade "debug a corruption in production" for "read a compiler error and try again." For serious software, that is a spectacular trade — and it's why learning the concepts, not the syntax, is the highest-leverage thing you can do.
