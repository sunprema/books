Concept 03 · Types

Make Illegal States Unrepresentable

The best bug is the one you can't write down. Rust's type system is a machine for making whole categories of mistake un-typeable.

Here's a game I want you to play whenever you design anything in Rust. Look at your data and ask: can I arrange the types so that a wrong combination simply cannot be expressed? Not "will be caught," not "will throw" — cannot be written at all. When you get good at this, enormous classes of bug vanish, and your code review gets boring in the best way.

Start with the most famous mistake in computing. Tony Hoare invented the null reference in 1965 and later called it his "billion-dollar mistake." The problem: in most languages, any value might secretly be null, so every access is a gamble. You forget one check, and months later a NullPointerException takes down production.

🔑
Rust's answer: there is no null

Rust simply doesn't have null. If a value might be absent, you say so in its type with Option<T> — it's either Some(value) or None — and the compiler won't let you use it until you've dealt with the None case. The bug isn't caught at runtime. It's unrepresentable.

Enums that actually mean something

An Option is just an ordinary enum, and enums are where Rust's type system gets fun. In a lot of languages an "enum" is a glorified list of integers. In Rust an enum is "this value is exactly one of these shapes, and each shape can carry its own data." That's a superpower for modeling reality.

rust · one value, several honest shapesenum Payment {
    Cash,
    Card { number: String, expiry: String },
    Crypto(String),          // wallet address
}

A Payment is cash, or a card with its fields, or crypto with an address — and never a nonsensical half-cash-half-card blob. You cannot construct the illegal thing. Compare that to a struct with a nullable cardNumber, a nullable walletAddress, and a type string that's supposed to keep them consistent but never quite does.

🤖
Steering the agent with types

This is the single highest-leverage instruction you can give a coding agent: "model this with an enum so the invalid combinations can't exist." The agent writes the types; the type system then forces every later line — including lines the agent writes next week — to respect them. You're encoding your judgment into a shape the compiler defends.

Pattern matching: handle every case, guaranteed

The partner to enums is match, and it comes with a quietly heroic feature: it's exhaustive. If you match on a Payment and forget the Crypto case, the program doesn't compile. Add a new payment type six months later, and the compiler marches you to every single place that now needs updating. It's a to-do list that can't be ignored.

rust · the compiler makes you handle absencefn greet(name: Option<String>) -> String {
    match name {
        Some(n) => format!("Hey, {n}!"),
        None    => "Hey, you!".to_string(),
    }
    // forget the None arm → it won't compile. No forgotten null check.
}

Notice what just happened. The "did you remember to check for missing?" question — the one that causes a billion dollars of crashes elsewhere — got moved from your discipline to the compiler's job. That's the recurring melody of Rust: take a thing humans reliably forget, and make forgetting impossible.

The newtype trick: even a number can be honest

You can push this further than you'd think. A u64 of seconds and a u64 of milliseconds look identical to a compiler — until you wrap each in its own tiny type. Now mixing them is a type error, not a production incident. NASA-grade paranoia, one line of code.

rust · a "newtype" makes units un-mixablestruct Seconds(u64);
struct Millis(u64);

// timeout(Millis(500)) compiles;
// timeout(Seconds(500)) does NOT. The mistake is now impossible.
🔑
The design instinct to keep

Before writing behavior, shape the data so wrong states can't be built: use an enum for "one of these," Option for "maybe," and a newtype for "don't confuse this with that." Do this and half your would-be bugs never get a chance to be born.

⚠️
Don't over-do it

You can drown a program in ceremony if you wrap everything. The art is spending this power where a mix-up would actually hurt — money, units, security, state machines — and staying relaxed everywhere else. Feynman's razor: make it as simple as possible, but no simpler.
