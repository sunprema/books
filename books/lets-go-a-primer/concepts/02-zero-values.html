Zero Values: Nothing Is Ever Undefined

Here's a question that sounds trivial but isn't: when you declare a variable and don't give it a value, what's inside it? In a lot of languages the honest answer is "garbage — whatever bits happened to be sitting in that memory." Go refuses to answer that way.

This is one of those small decisions that quietly prevents an entire category of bugs, and it's worth understanding before anything else, because you'll lean on it constantly without noticing.

The rule: every type has one honest default

In Go, declaring a variable with var and no initial value doesn't leave it undefined — it sets it to that type's zero value, a well-defined, documented default:

Numbers zero out, booleans go false, strings become the empty string, and anything that's fundamentally "a reference to something else" — pointers, slices, maps, channels, functions, interfaces — becomes nil, Go's "points at nothing" value.

Structs get it for free, recursively

This is where it gets genuinely elegant. A struct's zero value is simply every field set to its own zero value, all the way down:

type Account struct {
    Owner   string
    Balance int
    Tags    []string
}

var a Account
// a is Account{Owner: "", Balance: 0, Tags: nil} — no crash, no garbage
fmt.Println(a.Balance) // 0, safely

Compare that to languages where a freshly-declared object reference is a trap waiting to throw a null-pointer exception the moment you touch it. In Go, var a Account is already a usable, sane value — you can read every field immediately.

Design goal, not accident The Go authors call this "make the zero value useful." A sync.Mutex's zero value is already an unlocked, ready-to-use mutex. A bytes.Buffer's zero value is already an empty, ready-to-write buffer. You almost never need a constructor just to get a valid starting state.

The nil slice gotcha (and why it's not really a gotcha)

A nil slice looks alarming coming from languages where null means "touch this and crash." In Go, a nil slice behaves like an empty one for almost everything:

var names []string       // nil, but perfectly safe
fmt.Println(len(names))    // 0 — reading length never panics
names = append(names, "Ada") // works fine — append handles nil

The one thing you truly can't do to a nil slice or map is index-assign into a nil map (m[k] = v panics if m is nil) — because a map, unlike a slice, needs internal bookkeeping structures that make allocates. We'll meet slices properly, backing array and all, on the very next page.

Why this matters: "the zero value is always usable" is a promise Go's own standard library leans on everywhere, and it's a promise you should design your own types around too — a struct whose zero value is broken will surprise every caller who forgets to call a special "constructor" first.
