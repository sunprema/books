Arrays vs. Slices: The Secret Header

Almost nobody uses a Go array directly. Everybody uses a slice. So why teach arrays at all? Because a slice is built from an array, and the moment you can picture that relationship, half of Go's "weird" slice behavior stops being weird.

An array is a fixed-size box of boxes

An array's length is part of its type. [8]int and [10]int are simply different types — you can't resize one into the other, and passing an array to a function copies the whole thing, value by value:

var scores [8]int   // exactly 8 ints, zero-valued: [0 0 0 0 0 0 0 0]
scores[0] = 10
tweak(scores)         // tweak gets its OWN COPY of all 8 ints

That "whole array copied by value" behavior is rarely what you want for growing collections — which is exactly why Go gives you something better for everyday use.

A slice is a tiny header, not the data

A slice is a small struct with exactly three fields: a pointer to some backing array, a length (how many elements you can see), and a capacity (how far you could grow before Go needs a bigger array). Slicing an array — arr[2:5] — doesn't copy anything; it just builds one of these tiny headers pointing partway into the array you already had.

A slice is a header (pointer, length, capacity) — never the data itself.

Two consequences fall straight out of that picture, and both trip up beginners coming from other languages:

1. Slices of the same array can alias each other If two slices share a backing array, writing through one is visible through the other — they're windows onto the same memory, not independent copies.
2. append can silently switch you to a new array As long as there's spare capacity, append writes in place. The moment you outgrow capacity, Go allocates a bigger array, copies everything across, and hands back a slice pointing at the new one — which is exactly why you must always keep append's return value: s = append(s, x), never just append(s, x) and discard it.

Building slices directly

You don't need an array lying around first — make builds a slice (and its hidden backing array) in one step, and a slice literal does too:

s1 := make([]int, 3)         // len 3, cap 3: [0 0 0]
s2 := make([]int, 0, 10)     // len 0, cap 10 — room to grow before reallocating
s3 := []string{"a", "b", "c"}   // a literal: len 3, cap 3

Why this matters: once "slice = header onto a backing array" is second nature, you'll immediately understand why copy() exists (to actually duplicate data instead of aliasing it), why re-slicing is nearly free, and why a function that appends to a slice parameter might — or might not — mutate the caller's data, depending on whether it had spare capacity. That last one is a genuinely common Go bug, and now you know exactly why it happens.
