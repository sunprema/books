Chapter 10 · Putting It Together

Packages, Modules & the Go Toolchain

We've spent nine chapters on how Go thinks. This last one is about how a Go project is actually organized and shipped — kept deliberately short, because just like everywhere else in Go, there's really only one way to do it.

Every file belongs to exactly one package

The first non-comment line of every Go file declares its package:

package geometry   // this file is part of the "geometry" package

A package is Go's unit of code organization — usually one directory equals one package, and every .go file in that directory shares the same package name. One package is special: package main with a func main() marks a directory as producing an executable rather than a reusable library.

Capitalization is the entire visibility system

Go has no public/private keywords. Instead, a single, simple rule: an identifier (a function, type, variable, struct field...) is exported — visible outside its package — if and only if its name starts with a capital letter.

func Parse(s string) (Result, error) { ... }  // exported — other packages can call it
func normalize(s string) string { ... }        // unexported — this package only

No annotation to remember, no keyword to type — you can tell a name's visibility just by glancing at its first letter, from any call site, without needing to go look up its declaration.

Modules: go.mod names your project and pins its dependencies

A module is a collection of packages versioned and distributed together, described by a go.mod file at the project root:

// go.mod
module github.com/ada/geometry

go 1.26

require (
    github.com/some/dependency v1.4.0
)

go mod init github.com/ada/geometry creates this file; go mod tidy keeps its require list in sync with what your code actually imports (adding what's missing, removing what's unused); a matching go.sum file records exact cryptographic checksums of every dependency, so builds stay reproducible.

One CLI, one command per job

Where many ecosystems stitch together a separate compiler, formatter, test runner, linter, and dependency manager, Go bundles all of it behind a single go command:

CommandWhat it does
go run file.goCompile and immediately execute — good for quick iteration
go buildCompile the module into a standalone binary (no runtime to install on the target machine)
go test ./...Run every _test.go file's tests, recursively
go fmt ./...Rewrite every file into Go's one canonical formatting — no debate, no config
go vet ./...Static analysis: flag suspicious constructs a compiler wouldn't reject but a human would regret
go mod tidyReconcile go.mod / go.sum with what your code actually imports
go doc pkg.NamePrint documentation straight from source comments — no separate doc site needed
The payoff: one static binary go build produces a single, statically-linked executable for your target OS and architecture — no interpreter, no VM, no "install the right runtime version on the server first." Cross-compiling for another platform is often as simple as setting two environment variables: GOOS=linux GOARCH=arm64 go build.

Why this matters: this chapter has been the least "philosophical" of the book on purpose — because by this point, the toolchain's simplicity should feel like the natural conclusion of everything before it. A language built around one formatting standard, one visibility rule, and small, explicit interfaces was always going to end up with one tool that does everything, instead of a pile of configuration files arguing with each other.

Where to go from here Open the cheatsheet next for the syntax details we skipped on purpose — control flow, generics syntax, the full built-in function list. Then go write something real: a small CLI, a tiny HTTP server with net/http, a script that processes some files. Go rewards building over reading faster than most languages do.
