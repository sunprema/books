Chapter 06 · The Big Idea

Errors Are Just Values

Here's a sentence that sounds almost too simple to matter: in Go, an error is not a special control-flow event that unwinds your program. It's just a value, sitting in a variable, exactly like an int or a string. That one sentence explains a huge amount of what Go code looks like.

No exceptions. Multiple return values instead.

Go functions that can fail simply return an extra value alongside their normal result:

func Divide(a, b float64) (float64, error) {
    if b == 0 {
        return 0, errors.New("divide: division by zero")
    }
    return a / b, nil
}

result, err := Divide(10, 0)
if err != nil {
    // handle it right here, right now
    log.Println("could not divide:", err)
    return
}
// past this point, result is guaranteed usable

That if err != nil line shows up constantly in Go code, and new Go programmers sometimes find it repetitive coming from a try/catch world. But it's the entire point: the possibility of failure is written directly into the function signature, and you deal with it exactly where it happens, not three stack frames away in a catch block that has to guess what went wrong and where.

error is just an interface — one method

Remember the last chapter? error is a built-in interface with exactly one method:

type error interface {
    Error() string
}

Any type with an Error() string method is an error — which means you can build your own error types that carry structured data (an HTTP status code, a field name that failed validation, a wrapped underlying cause), not just a flat string message.

Wrapping errors: keep the chain, add context

When an error bubbles up through several layers, you usually want to add context at each layer without losing the original cause. fmt.Errorf with the %w verb does exactly that:

func loadConfig(path string) error {
    _, err := os.Open(path)
    if err != nil {
        return fmt.Errorf("loadConfig: %w", err) // wraps, doesn't hide, the original
    }
    return nil
}

// later, a caller can still ask "was this specifically a file-not-found?"
if errors.Is(err, os.ErrNotExist) {
    // ...
}

errors.Is walks the whole wrapped chain looking for a match against a known "sentinel" error value; errors.As does the same but pulls out a matching error of a specific concrete type so you can read its fields. Together they let you add human-readable context at every layer while still letting code further up the chain make precise decisions about the root cause.

Idiomatic habits worth adopting immediately Check every error — never _ = someCall() a fallible call away out of laziness. Handle it, return it, or (rarely, at real program boundaries) panic — never silently swallow it. Add context on the way up (%w), but log the final error only once, at the top, not at every layer it passes through.
panic is not "Go's exception" panic exists for truly unrecoverable situations — a programming bug like an out-of-bounds index, not "the file wasn't found." If you find yourself reaching for panic for an ordinary, expected failure, that's a sign you want an error return instead. We'll meet panic and its partner recover properly next chapter, right alongside defer.

Why this matters: making failure an ordinary return value, not a special jump, means you can always see — right there in a function's signature — whether it can fail, and the compiler will nag you if you ever declare an error variable and then never check it. Nothing is hidden in a call stack you can't see from the call site.
