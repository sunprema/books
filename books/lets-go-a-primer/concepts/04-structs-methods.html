Structs & Methods: Data With Behavior Attached

Go has no class keyword. That's not laziness — it's a deliberate split. A struct is only data, grouped fields with names. A method is a completely ordinary function that happens to be attached to a type. Keeping those two ideas visibly separate turns out to simplify almost everything downstream.

A struct is just a labeled bundle of fields

type Circle struct {
    X, Y   float64
    Radius float64
}

c := Circle{X: 0, Y: 0, Radius: 5}

No inheritance, no access modifiers on individual fields (visibility works at the package level instead — capital letter, exported; lowercase, not — we'll get there in the last chapter). Just data.

A method is a function with one extra thing: a receiver

func (c Circle) Area() float64 {
    return math.Pi * c.Radius * c.Radius
}

fmt.Println(c.Area()) // 78.53...

That (c Circle) between func and the method name is the receiver — it's just syntax for "this function's first, implicit argument is a Circle, and you can call it with dot notation." Under the hood, c.Area() really is just Area(c) wearing a costume.

Value receiver vs. pointer receiver — the one decision that matters

This is the question every Go beginner asks eventually: func (c Circle) or func (c *Circle)? The answer comes straight from what a struct is: plain data, copied by value like any other Go value (remember arrays, last chapter?).

The rule of thumb Use a pointer receiver (*Circle) whenever the method needs to mutate the struct, or the struct is large enough that copying it on every call would be wasteful. Use a value receiver (Circle) for small, read-only, "just compute something" methods. Go even lets you call a pointer-receiver method on an addressable value directly — c.Scale(2) — and it silently takes &c for you.
One consistency rule If any method on a type needs a pointer receiver, it's idiomatic to make all its methods pointer receivers — mixing the two on the same type is a common source of "why didn't my change stick" confusion.

Why this matters: in languages built around classes, "does calling this method change my object?" is usually assumed (everything is a reference). In Go it's an explicit, visible choice at the top of every method — one glance at the receiver tells you whether that call can possibly mutate what you passed in.

Sources: go.dev/ref/spec — "Method declarations"; Effective Go, "Pointers vs. Values".
