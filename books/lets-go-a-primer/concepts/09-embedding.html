Embedding: Composition, Not Inheritance

We said back in Chapter 4 that Go has no class keyword and no inheritance. So how do you reuse a chunk of behavior across several types without copying and pasting it everywhere? Go's answer is embedding — and it's a deliberately more limited tool than inheritance, which turns out to be its whole appeal.

Put a type inside a struct, with no field name

type Person struct {
    Name string
    Age  int
}
func (p Person) Greet() string { return "Hi, I'm " + p.Name }

type Employee struct {
    Person          // embedded — no field name, just the type
    EmployeeID string
}

e := Employee{Person: Person{Name: "Ada", Age: 30}, EmployeeID: "E42"}
fmt.Println(e.Name)     // "Ada" — Person's field, promoted onto Employee
fmt.Println(e.Greet())  // Person's method, promoted too — "Hi, I'm Ada"

Because Person has no explicit field name, its fields and methods are promoted — they appear to be Employee's own, reachable with a single dot. But look closely at what actually happened: Employee literally contains a whole Person value as one of its fields. This is composition ("has-a"), wearing syntax that reads almost like inheritance ("is-a"). The distinction matters.

Overriding, without polymorphism

If Employee defines its own Greet, that one wins for Employee values — the outer type's method "shadows" the embedded one. But there's a crucial difference from real inheritance: if Person.Greet itself calls some other method, it always calls Person's version, never a version overridden on the outer Employee. There is no dynamic dispatch back up the chain, because there is no chain — just one struct value sitting inside another.

The gotcha coming from OOP languages In Java or Python, overriding a method changes behavior everywhere that method is called polymorphically, even from inside the base class. In Go, embedding gives you field/method promotion for convenience, but not virtual dispatch. If you need "the right version gets called no matter who's asking," that's exactly what an interface (Chapter 5) is for — embedding and interfaces solve two different problems that inheritance conflates into one.

Interfaces embed too

The same idea works one level up, composing small interfaces into bigger ones — this is how the standard library builds up its most useful interfaces from tiny one-method pieces:

type Reader interface { Read(p []byte) (int, error) }
type Writer interface { Write(p []byte) (int, error) }

type ReadWriter interface {
    Reader   // embeds Reader's method set
    Writer   // embeds Writer's method set
}

Why this matters: Go deliberately splits what other languages bundle into one "inheritance" feature into two separate, smaller tools — embedding for reusing an implementation's fields and methods, and interfaces for "this can be treated as that." Each tool is simpler alone, and together they cover what inheritance covers without its most common failure mode: deep, fragile hierarchies where changing a base class ripples unpredictably through everything beneath it.
