Defer, Panic & Recover: Go's Safety Net

You've now seen that Go doesn't use exceptions for ordinary failure — errors are values you check. So what handles cleanup, and what happens on the rare occasion something truly unrecoverable goes wrong? Three small, related keywords: defer, panic, and recover.

defer: "run this later, when I leave, no matter how"

defer schedules a function call to run right before its surrounding function returns — whether it returns normally, or via panic. It's Go's answer to "always close the file, always unlock the mutex, no matter which exit path I take":

func readConfig(path string) error {
    f, err := os.Open(path)
    if err != nil {
        return err
    }
    defer f.Close() // runs no matter which return below fires

    // ...read the file, maybe return an error partway through...
    return nil
}

You write the cleanup right next to the thing that needs cleaning up, instead of duplicating f.Close() before every single return statement in the function. That locality is the whole appeal.

Multiple defers unwind last-in, first-out

If a function defers several calls, they run in the reverse of the order you wrote them — exactly like a stack of plates: the last one you put down is the first one you take back off.

Defers always run last-declared-first, whether the function returns normally or panics.

panic: when something is truly unrecoverable

panic stops normal execution immediately and starts running deferred calls on its way out — of this function, then its caller, then its caller — in reverse order, just like a normal return, except it keeps going upward until either a deferred call stops it, or it reaches the top of the program and crashes it with a stack trace.

recover: catching a panic mid-unwind

recover is oddly specific: it only does anything when called directly inside a deferred function, during an active panic. It stops the unwinding and returns the value passed to panic:

func safeDivide(a, b int) (result int, err error) {
    defer func() {
        if r := recover(); r != nil {
            err = fmt.Errorf("safeDivide: recovered from: %v", r)
        }
    }()
    return a / b, nil // division by zero panics here if b == 0
}

This pattern — a deferred closure that calls recover and turns the panic into an ordinary error — is exactly how Go converts an unrecoverable-looking crash back into the normal error-value world at a well-defined boundary (a web server famously does this per-request, so one handler panicking doesn't take the whole server down).

Don't reach for panic/recover as everyday control flow Idiomatic Go reserves panic for programmer errors and truly unrecoverable situations, and uses it to implement recover-based safety nets at specific boundaries — not as a general substitute for returning an error. If you're tempted to panic for an expected failure, go back to the previous chapter.

Why this matters: once you can picture defer as a LIFO stack that panic walks down looking for a recover, three things that feel unrelated on the page — resource cleanup, crash safety nets, and "why did my defers run in that order" — turn out to be one single mechanism viewed from three angles.
