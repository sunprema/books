Chapter 08 · The Big Idea

Goroutines & Channels: Concurrency You Can Picture

This is the feature people usually mean when they say "Go is good at concurrency." Not because Go has some exotic threading model — because it gives you two simple, orthogonal tools, and a slogan that tells you how to combine them: "Don't communicate by sharing memory; share memory by communicating."

A goroutine is a function that runs alongside, not instead of

Put go in front of any function call, and it starts running concurrently with the rest of the program, instead of blocking until it finishes:

go backup(db)   // starts running, doesn't wait — control returns immediately
fmt.Println("backup started")

A goroutine is not an operating-system thread. It starts with a tiny (a few kilobytes) stack that grows as needed, and the Go runtime multiplexes potentially hundreds of thousands of goroutines onto a small handful of real OS threads for you. That's why it's completely normal in Go to spin up a goroutine per incoming request, per item in a list, per anything — the cost is close to nothing, where an OS thread would be expensive.

A channel is a typed pipe between goroutines

Starting a goroutine is easy; the hard part in any concurrent system is getting results back safely, without two goroutines stepping on the same piece of memory at the same time. Go's answer is the channel: a typed conduit you send values into and receive values out of.

ch := make(chan int)   // unbuffered channel of ints

go func() {
    ch <- 42          // send: blocks until someone receives
}()

v := <-ch          // receive: blocks until someone sends
fmt.Println(v)      // 42

An unbuffered channel is a rendezvous point: both sides must show up together.
Unbuffered vs. buffered An unbuffered channel (make(chan int)) has no storage at all — a send blocks until a receiver is ready, right that instant, like a handshake. A buffered channel (make(chan int, 2)) can hold a few values in flight — a send only blocks once the buffer is full, and a receive only blocks once it's empty.

select: waiting on several channels at once

When a goroutine needs to react to whichever of several channels is ready first — a result, a cancellation signal, a timeout — select is a switch for channel operations:

select {
case v := <-resultCh:
    fmt.Println("got:", v)
case <-time.After(2 * time.Second):
    fmt.Println("timed out waiting")
case <-ctx.Done():
    fmt.Println("cancelled")
}

Whichever case becomes ready first runs; if several are ready simultaneously, Go picks one at random (so no single channel starves the others). This one construct is how Go expresses timeouts, cancellation, and "first result wins" fan-in, all with the same building block.

sync.WaitGroup: "wait until everyone's done"

Not every concurrent task needs to send a value back — sometimes you just need to know when a batch of goroutines has finished. A sync.WaitGroup counts them down:

var wg sync.WaitGroup
for _, url := range urls {
    wg.Add(1)
    go func(u string) {
        defer wg.Done()  // note: defer, from two chapters ago
        fetch(u)
    }(url)
}
wg.Wait() // blocks here until every Done() has fired
When you truly must share memory: sync.Mutex, and go run -race Channels cover most concurrent designs, but sometimes a plain shared counter or map really is simplest guarded by a sync.Mutex (mu.Lock() ... defer mu.Unlock()). Whichever approach you use, run your tests with go run -race / go test -race — Go's built-in race detector instruments memory accesses and reliably catches two goroutines touching the same data without synchronization, which is otherwise one of the hardest classes of bug to reproduce.

Why this matters: a channel isn't just a queue — it's a synchronization point with clear, predictable blocking rules. Once you can picture "unbuffered = must meet at the same instant, buffered = a mailbox with N slots," goroutine code stops feeling like magic and starts feeling like ordinary data flow with well-defined waiting points.
