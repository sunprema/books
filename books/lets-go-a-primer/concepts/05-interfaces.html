Chapter 05 · The Big Idea

Interfaces: Shape, Not Ancestry

If I had to bet on the single Go feature that most changes how you think about program design, I'd bet on this one. It's quiet, it's easy to miss on a first read, and it's the reason Go code composes so well without a class hierarchy in sight.

You never say "implements." Go just checks the shape.

In many languages, a type must explicitly declare which interfaces it implements — class Duck implements Swimmer. Go throws that ceremony out entirely. A type satisfies an interface simply by having the right methods. Nothing is declared, nothing is imported for the purpose of "implementing" it:

type Swimmer interface {
    Swim() string
}

type Duck struct{ Name string }
func (d Duck) Swim() string { return d.Name + " paddles." }

// Duck never mentions Swimmer anywhere. It satisfies the interface just by
// having a Swim() string method. That's the whole rule.
var s Swimmer = Duck{Name: "Waddles"}

This is called structural typing (people also say "duck typing, but checked at compile time" — if it swims like a Swimmer, it's a Swimmer). The Go compiler verifies the method exists with the exact right signature; you never had to wire the two types together by hand.

Idiomatic Go interfaces are tiny

Because satisfying an interface costs nothing (no registration, no keyword), Go culture leans toward very small interfaces — often exactly one method. The standard library's most-used interface is a perfect example:

type Writer interface {
    Write(p []byte) (n int, err error)
}

Files, network connections, in-memory buffers, gzip compressors, HTTP response bodies — dozens of unrelated types across the standard library all happen to have a Write([]byte) (int, error) method, and so they all "are" an io.Writer, without ever coordinating with each other.

The idiom: "accept interfaces, return concrete types" Write functions that take the smallest interface that does the job (so callers can pass in anything that fits), but have them return a concrete struct type (so callers get the full, specific API back, not a watered-down one). This single habit is behind most well-designed Go APIs.

The empty interface, briefly

interface{} (or its modern alias, any) describes zero methods — which every single type trivially satisfies. It's Go's escape hatch for "a value of any type," used sparingly (things like fmt.Println(v ...any)) — reach for real generics (added in Go 1.18) or a small custom interface before reaching for any as your default tool.

Why this matters: implicit satisfaction is what lets Go code stay decoupled without a framework enforcing it. A package can define exactly the interface it needs, and every existing type in the universe that happens to already have the right methods just... works, with zero coordination. It's a strange kind of compile-time duck typing, and once you feel it click, going back to explicit "implements" lists feels like unnecessary paperwork.
