Anatomy of a Request: Socket to Reply

Skip redis-cli for a moment and talk to Redis the way it actually hears you — over a raw socket, in its wire protocol, RESP: 

Terminal — raw RESP over a bare TCP connection$ printf '*3\r\n$3\r\nSET\r\n$3\r\nfoo\r\n$3\r\nbar\r\n' | nc localhost 6379
+OK\r\n

That *3\r\n$3\r\nSET\r\n... is a RESP multibulk request: an array of three bulk strings, SET, foo, bar. Every client library you've ever used — redis-cli included — is just building strings that look like this. This chapter follows those bytes from the moment they land on the socket to the moment a reply goes back out.

The Lifecycle, Stage by Stage

  1. Connection setup. accept() fires as a file event on the listening socket; Redis allocates a client struct, sets it non-blocking, and registers a new file event for reads on the new connection.
  2. Input buffer handling. When bytes arrive, readQueryFromClient() reads them into the client's query buffer — a growable string, not one read-per-command, so a pipeline of many commands in one TCP packet is handled efficiently.
  3. Protocol parsing. processInputBuffer() splits the buffer into complete commands, understanding both the multibulk form above and a simpler inline form (space-separated, newline-terminated — mostly for humans typing directly into telnet).
  4. Command lookup. The first bulk string (SET) is looked up in the command table — the subject of Chapter 7.
  5. Execution. The command's handler function runs, reading or mutating the keyspace directly — no queue, no indirection, just a function call.
  6. Reply buffering. The handler doesn't write to the socket directly; it calls addReply() and friends, which append the RESP-encoded reply to the client's output buffer.
  7. Flushing. Just before the event loop goes back to sleep waiting for more events (beforeSleep()), every client with pending output gets it written to its socket, batching the syscalls instead of doing one write per reply.

What Actually Blocks the Loop

"Single-threaded" means the loop can only be doing one thing. The danger is any command whose execution step takes a long time in wall-clock terms: a full keyspace scan (KEYS *), a huge SORT or LRANGE over millions of elements, an unbounded Lua script. None of these are illegal — they're just commands whose cost scales with data size instead of staying constant, and on a single thread, that cost is paid by every other client as wait time.

Pro Tip

This is exactly why SCAN exists next to KEYS. KEYS * does all its O(N) work in one command, blocking the loop for the whole scan. SCAN does the same traversal but hands you a cursor, doing a small bounded slice of work per call — the same "break a big job into small increments" trick you'll see again for rehashing (Ch. 6) and expiration (Ch. 10). Prefer SCAN in any code path that runs against production.

The Clever Part: "Blocking" Commands Don't Block the Loop

BLPOP looks, from a client's point of view, like it blocks: the connection just sits there until another client pushes to the list. But it does not block the event loop for anyone else. When a client issues BLPOP against an empty list, Redis marks that one client as "blocked waiting on this key," parks it on a per-key waiting list, and immediately returns control to the loop to serve other clients. The moment some other client runs an RPUSH on that key, Redis notices the key now has waiters, wakes the blocked client, and serves its reply — all still one command at a time, never actually pausing the thread to wait.

Key Idea

"Blocking" in Redis is a client-visible illusion built entirely out of bookkeeping — a waiting list per key and a wake-up check after every write — not an actual blocked OS thread. It's a clean example of the recurring habit (Chapter 12 names it directly): solve a concurrency-shaped problem with a small data structure instead of an actual concurrency primitive.

Warning

A truly long-running piece of execution — not waiting, actual CPU-bound work, like a Lua script stuck in a loop — genuinely does stall the whole server, because step 5 above really is one uninterruptible function call. Redis's answer isn't to make execution preemptible (that would reopen every concurrency problem Chapter 2 avoided) — it's lua-time-limit / script timeouts and operational discipline: keep any single command's work bounded.
