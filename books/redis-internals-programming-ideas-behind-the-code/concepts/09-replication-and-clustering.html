The Handshake: Full Sync, Then a Stream

A fresh replica has to get a full copy of the dataset before it can start applying live updates. The handshake, built around the PSYNC command, does exactly that: the replica connects, exchanges a PING and some REPLCONF capability negotiation, then sends PSYNC <replid> <offset>. If the primary doesn't recognize that replication ID or offset (a brand-new replica sends ? and -1), the primary triggers a full resynchronization: it forks exactly the way BGSAVE does (Chapter 8) to produce a consistent RDB snapshot, streams that to the replica, and the replica loads it wholesale before switching to live updates.

The Replication Backlog: Avoiding a Full Sync on Every Blip

A full resync means forking and re-transferring the entire dataset — expensive, and unnecessary for something as ordinary as a replica's network blipping for a few seconds. The primary keeps a replication backlog, a fixed-size ring buffer of the most recent write stream. When a replica reconnects, if its last known offset still falls inside that backlog's window, the primary does a partial resynchronization instead: it just replays the missing slice of the backlog, and the replica is caught up in proportion to how much it missed, not how large the whole dataset is.

Failover and Topology

Replication by itself is passive — a primary doesn't promote a replica on its own if it dies. Redis Sentinel (a separate monitoring process, run alongside your Redis instances) watches primaries and replicas, and when a quorum of Sentinels agree a primary is genuinely down (not just having a network hiccup one Sentinel saw alone), it picks a replica, promotes it with REPLICAOF NO ONE, and reconfigures the remaining replicas to follow the new primary. Well-behaved clients subscribe to Sentinel for the current primary's address rather than hardcoding it, so they can redirect automatically when a promotion happens.

Cluster: Sharding by Hash Slot

Replication solves availability and read scaling — every replica holds the whole dataset. Redis Cluster solves the different problem of a dataset too large for one primary to hold at all, by splitting the keyspace into exactly 16384 hash slots. Every key maps to a slot with CRC16(key) mod 16384 (or, if the key contains a {hash tag}, only the part inside the braces is hashed — the standard way to force related keys onto the same slot so they can be operated on together). Each slot is owned by exactly one primary node; 16384 was chosen deliberately small enough that a full slot ownership bitmap still fits comfortably in a cluster gossip heartbeat message, while staying large enough to divide sensibly across hundreds of nodes.

Pro Tip

A cluster-aware client keeps a local map of slot → node and only pays the redirect cost once, right after the map goes stale (e.g. after a resharding). If you're calling Redis Cluster directly with redis-cli -c, the -c flag tells it to follow MOVED/ASK redirects for you automatically — without it you'll just see the raw redirect error.

What Cluster Deliberately Doesn't Solve

Redis Cluster keeps its coordination model simple on purpose: node state and slot ownership are propagated via a gossip protocol between nodes rather than a strongly-consistent external coordinator, and replication to a slot's replicas is asynchronous. That means a failover can, in rare cases, lose a handful of the most recent writes that hadn't yet reached a replica — Redis accepts this as a deliberate trade-off for a simpler, faster, self-contained cluster rather than paying the latency and complexity cost of a strongly consistent commit protocol across nodes. There is also no cross-slot multi-key transaction guarantee: an operation touching keys in two different slots is simply rejected unless you've forced them into the same slot with a hash tag.

Key Idea

Both replication and clustering repeat Chapter 1's founding trade at a bigger scale: Redis picks simplicity and predictable behavior (async replication, gossip-based slot ownership, no distributed transactions) over the strongest possible consistency guarantee, because the stronger guarantee would cost latency and operational complexity that most workloads using Redis don't actually need.

Replication and clustering are about keeping data available and distributed. The next problem is the opposite instinct: actively forgetting data, on purpose, on a schedule — expiring keys and evicting them under memory pressure. That's next.
