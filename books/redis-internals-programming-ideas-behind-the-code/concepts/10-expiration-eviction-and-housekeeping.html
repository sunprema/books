Expiration, Eviction, and the Housekeeping Cron

Set a key with a one-second TTL, wait, and read it back: 

Terminal — a key that expires without anyone asking$ redis-cli SET session:9 "active" EX 1
OK
$ redis-cli TTL session:9
(integer) 1
$ sleep 2 && redis-cli GET session:9
(nil)

Nobody read that key in the gap — and yet it's gone. Redis doesn't set a timer per key (millions of concurrent timers would be its own scaling nightmare); instead it combines two much cheaper mechanisms that only cost anything when they actually need to.

Passive Expiration: Check on the Way Past

Every key with a TTL is tracked in a separate expires dictionary (key → expiry timestamp), sitting alongside the main keyspace dict from Chapter 6. Whenever any command is about to read or write a key, expireIfNeeded() checks that dictionary first: if the key's TTL has passed, it's deleted right then, before the command ever sees it. This costs nothing extra for keys that are never accessed again after expiring — which is exactly the common case for things like session tokens and rate-limit counters.

Active Expiration: Sweep the Rest in Small Batches

Passive expiration alone would leave dead keys sitting in memory forever if nobody happened to touch them again. So a time event, run from serverCron, periodically samples a small number of random keys from the expires dictionary (20 by default), deletes whichever ones have actually expired, and — only if a large fraction of that sample (more than roughly 10%) turned out to be expired — immediately samples another batch, on the theory that a dictionary with lots of stale keys probably has more nearby. If the sample comes back mostly alive, the cycle backs off and waits for its next scheduled turn instead of scanning further.

Key Idea

Passive expiration guarantees correctness (an expired key is never returned to a client, ever, even if the active cycle hasn't reached it yet); active expiration is what actually reclaims the memory in the background, cheaply, using sampling instead of an exhaustive scan. Correctness and cleanup are solved by two different, narrowly scoped mechanisms — not one mechanism trying to do both.

Eviction: When Memory Runs Out

Expiration is voluntary — keys you told Redis had a TTL. Eviction is what happens when maxmemory is set and the dataset grows past it, and it applies even to keys with no TTL, depending on the configured maxmemory-policy (noeviction, allkeys-lru, volatile-lru, allkeys-lfu, volatile-lfu, allkeys-random, volatile-random, volatile-ttl). Redis doesn't track a perfectly-ordered LRU or LFU list — that itself would be bookkeeping overhead on every single access. Instead it approximates:

  • LRU is approximated with a 24-bit clock stored directly in each object's robj header, updated on access. When eviction is needed, Redis samples a small number of random keys (maxmemory-samples, default 5) and evicts whichever sampled key has the oldest clock value — not the single oldest key in the whole dataset, just the oldest among a small random sample, refined across calls by keeping a small pool of good eviction candidates from previous rounds.
  • LFU is approximated with a tiny probabilistic counter (a Morris counter) packed into just a few bits per object, which increments less and less often as it grows (so a wildly popular key doesn't need an ever-larger counter) and decays slowly over time so old popularity fades.
Pro Tip

OBJECT FREQ key (with an LFU policy active) and OBJECT IDLETIME key (with an LRU policy active) let you read these approximated values directly — handy for understanding why a specific key did or didn't get evicted.

Warning

noeviction is the factory default: with no maxmemory-policy configured for eviction, once maxmemory is hit, Redis starts rejecting writes with an OOM error rather than silently dropping data. If you're using Redis as a pure cache, you almost certainly want an explicit allkeys-lru or allkeys-lfu instead — check this before you're surprised by write errors in production.

serverCron: Everything Else That Runs in the Background

Active expiration is just one of many jobs serverCron handles — the single recurring time event this whole chapter has been describing (and the same time event Chapter 2's reactor loop showed alongside the client sockets). By default it fires hz times per second (10, and dynamic-hz yes lets it speed up automatically under load). In each tick it nudges forward every background job that must never be allowed to run unbounded in one shot: incremental dict rehashing (Chapter 6), the active expire cycle above, AOF fsync scheduling (Chapter 8), replication pings and backlog housekeeping (Chapter 9), client idle-timeout checks, and statistics updates.

Key Idea

Notice the pattern repeating across the whole chapter — and really across the whole book: rehashing, expiration, and eviction are three completely different problems that all get solved the same way, in small time-boxed or randomly-sampled increments, driven from the same recurring cron, specifically so none of them can ever cause the one big pause Chapter 1 set out to avoid.

That repeating pattern is worth naming explicitly, and worth asking where else it shows up. Before that, though: how do you actually watch all of this happening in a live server, and how do you know why Redis is fast rather than just taking it on faith? That's next.
