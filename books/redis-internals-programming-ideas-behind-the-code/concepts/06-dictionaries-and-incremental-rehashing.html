Dictionaries and Incremental Rehashing

Load a few million keys and watch latency while it happens: 

Terminal — loading 2 million keys while watching latency$ redis-cli --latency -i 1 &
$ redis-benchmark -n 2000000 -r 2000000 -t set -q
min: 0, max: 2, avg: 0.19 (no sustained spike, even mid-load)

That flat latency line is worth stopping on. The main keyspace — every key name mapped to its value — is itself one giant hash table. Hash tables that fill up need to grow, and growing a hash table normally means allocating a bigger array and re-inserting every single entry into it: an O(n) pause proportional to how much data you have. For a keyspace with millions of entries, that pause would be exactly the kind of latency spike Chapter 1 said Redis refuses to accept. So Redis doesn't do it that way.

The dict: Two Tables, Not One

The hash table implementation, dict.c, keeps two hash tables per dictionary, ht[0] and ht[1], plus one integer, rehashidx. Normally rehashidx is -1: the dict isn't rehashing, ht[0] is the only table in use, and ht[1] is empty. When ht[0]'s load factor crosses its threshold, Redis allocates ht[1] at roughly double the size and sets rehashidx to 0 — and that's the moment everything gets interesting.

Rehashing a Little Bit at a Time

Instead of migrating every entry from ht[0] to ht[1] immediately, Redis migrates a few buckets at a time, riggered opportunistically:

  • Every read or write on the dictionary, while rehashidx ≥ 0, first migrates a small, bounded number of buckets from ht[0][rehashidx] into ht[1], then advances rehashidx, before doing the actual lookup or insert the caller asked for.
  • A background time event (part of serverCron, Chapter 10) also nudges rehashing forward in small time-boxed steps even if the keyspace is momentarily quiet, so a dict doesn't get stuck half-rehashed forever on an idle server.
  • While rehashing is in progress, reads check both tables — look in ht[0] first, then ht[1] — and all new writes go straight into ht[1], so the dict never needs to migrate an entry it just added.
  • When ht[0] finally empties out, Redis frees it, promotes ht[1] to be the new ht[0], and resets rehashidx back to -1. Business as usual, until the next resize.
Key Idea

Incremental rehashing turns one big, unpredictable pause into many tiny, bounded ones, each riding along on a command that was going to touch the dict anyway. It's the single clearest example in the whole codebase of Chapter 1's core trade: a little bit of extra bookkeeping (two tables, one index, a branch in every lookup) in exchange for eliminating a latency spike entirely.

Pro Tip

This exact same dict structure backs more than just your keyspace — it's also what a large hash-encoded Hash, Set, or the expires table (Chapter 10) rehashes into once they outgrow their compact encoding (Chapter 5). Learn the two-table trick once here and you've understood how every big dictionary in Redis grows.

Warning

Incremental rehashing amortizes cost, it doesn't eliminate it — a dict actively rehashing briefly uses roughly 1.5× its steady-state memory (both tables live at once), and iterating a dict mid-rehash (as SCAN does) has to account for entries living in either table. That complexity is exactly why SCAN's cursor guarantee is "every key present for the whole scan is returned at least once," not "returned exactly once" — a subtlety worth knowing before you rely on SCAN for exact-once processing.

Two tables and one index solve "how do you grow a hash table without a pause." The next problem is different: given that exactly one command runs at a time (Chapter 2), what does that buy you for free when it comes to correctness under concurrent access? That's atomicity, and it's the whole subject of Chapter 7.
