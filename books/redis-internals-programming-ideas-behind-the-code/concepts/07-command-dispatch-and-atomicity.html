Command Dispatch and Atomicity Without Locks

Fire the same increment at Redis from fifty concurrent clients and count the result: 

Terminal — 50 clients racing to INCR the same key$ redis-cli SET counter 0
OK
$ redis-benchmark -c 50 -n 5000 -t incr -q -r 0
INCR: ... requests per second
$ redis-cli GET counter
"5000"

Exactly 5000, every time, no matter how many concurrent clients raced to get there. In a database with real internal concurrency, that would need a lock around the counter to avoid a classic read-modify-write race. Redis has no such lock — and doesn't need one, because of a guarantee this chapter makes precise: exactly one command's execution step ever runs at a time, start to finish, with nothing else touching the dataset in between.

The Command Table

Every command Redis understands is registered once, in a table keyed by name, as a struct redisCommand: the command's name, a function pointer to its handler, its arity (the exact argument count, or a negative number meaning "at least N"), a set of flags (is it a write command? read-only? does it require enough free memory, deny-oom? is it an admin-only command?), and metadata describing which of its arguments are actual keys (so cluster routing and MULTI conflict-checking can find them generically). lookupCommand() resolves the first word of a request against this table with a single dictionary lookup — the same dict structure from Chapter 6, put to a different use.

Pro Tip

Run COMMAND INFO get set expire from any live server to see this metadata firsthand — arity, flags, and first/last/step key positions, exactly as the dispatcher reads it. It's the same introspection Redis Cluster clients use to figure out which key(s) a command touches without hardcoding a list per command.

From Lookup to call()

Once the handler is found, the dispatcher's call() function invokes it directly: cmd->proc(c). There is no task queue, no scheduling decision, no yield point in the middle — it's an ordinary, synchronous C function call, and the function doesn't return until the command is completely finished mutating the keyspace and has queued its reply. That's the whole mechanism. Atomicity isn't implemented — it falls out for free from "the event loop only ever does one thing at a time" (Chapter 2) plus "executing a command is one uninterrupted function call" (Chapter 3).

Key Idea

Almost every single Redis command is atomic simply by virtue of being one command — INCR, LPUSH, HSET, even compound-sounding ones like SETNX (set-if-not-exists) or GETSET. What needs actual thought is when your application logic spans multiple commands and the state in between matters — that's what MULTI/EXEC and Lua scripting exist for.

Transactions: MULTI/EXEC

MULTI switches a connection into queuing mode: every command after it is parsed and validated but not executed, just appended to a per-client queue. EXEC then runs every queued command back-to-back, as one atomic block — no other client's command can be interleaved in the middle, because, again, there's only one execution lane and this block has claimed it entirely for its duration. The catch: MULTI/EXEC queues commands blindly. It can't look at the result of the second queued command to decide what the third one should be — there's no branching once you're inside the block. 

Terminal — a queued, atomic transaction$ redis-cli
127.0.0.1:6379> MULTI
OK
127.0.0.1:6379> INCR visits
QUEUED
127.0.0.1:6379> EXPIRE visits 60
QUEUED
127.0.0.1:6379> EXEC
1) (integer) 1
2) (integer) 1

Scripting: Lua and Determinism

Lua scripting (EVAL, and its cousin FUNCTION) solves exactly the limitation MULTI/EXEC has: a script can read a value, branch on it, and decide what to write next, all inside one atomic unit, because the whole script runs to completion as a single unit of execution — no other command can run in the middle. That power comes with a rule: scripts must be deterministic, because their effects are replicated to replicas and persisted to the AOF (Chapter 8) as if they were the commands the script itself ran. Redis actively blocks a script from calling a random or clock-dependent command (like SPOP or TIME) and then following it with a write, specifically to keep every replica's data path deterministic.

Warning

A script that runs long doesn't get preempted — the same "one uninterrupted execution" guarantee that makes it atomic also means a runaway loop inside a script stalls the entire server exactly like the dangerous commands from Chapter 3. The busy-reply-threshold (formerly lua-time-limit) exists to let you notice a stuck script and intervene — not to make the script safe to run unbounded in the first place.

Modules: Extending Redis Without Breaking the Contract

Modules (loaded via MODULE LOAD) let you register whole new commands, written in C or any language with a native binding, using RedisModule_CreateCommand(). The API is layered deliberately: a high-level layer that looks like calling any other Redis command, and a low-level layer with near-native access to core data structures for performance-critical modules. Either way, a module command is expected to respect the same execution contract every built-in command does — run to completion without silently blocking the loop — because the module boundary is a place to add new capability, not a place to opt out of the guarantees the rest of the engine depends on.

Atomic execution is what makes the data path trustworthy moment to moment. But none of it survives a restart unless it's written somewhere durable — and writing to disk is exactly the kind of slow operation Chapter 1 said the main thread must never wait on. How Redis squares that circle is next.
