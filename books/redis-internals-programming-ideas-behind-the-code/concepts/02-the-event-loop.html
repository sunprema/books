The Event Loop: One Thread, Thousands of Clients

Open two terminals against the same Redis. In the first, run a command that takes a while on a large keyspace; in the second, time a plain GET: 

Terminal A — a deliberately expensive scan$ redis-cli KEYS '*'   # O(N) over the whole keyspace — never do this in production

Terminal B — running at the same moment$ redis-cli GET some:key
(waits until KEYS finishes)

That pause in Terminal B isn't a bug — it's the single most important fact about how Redis works, made visible. Every command Redis executes runs to completion on one thread, inside one loop, before the next command even begins. This chapter is about that loop: the tiny, custom-built library that drives it, and why Redis chose to write its own instead of reaching for an off-the-shelf one.

The Asynchronous Event Library: ae

Redis's event loop lives in ae.c / ae.h — a self-contained, roughly-700-line C library Redis maintains itself rather than depending on libevent or libuv. It wraps whichever OS multiplexing syscall is available (epoll on Linux, kqueue on BSD/macOS, a portable select() fallback) behind one small API, and it understands exactly two kinds of things to wait for:

  • File events — a client socket becoming readable (bytes have arrived) or writable (there's room to send a reply). Registered with aeCreateFileEvent().
  • Time events — "call this function again in N milliseconds," used for periodic housekeeping. Registered with aeCreateTimeEvent().

The main loop itself, aeMain(), is almost embarrassingly short: it repeatedly calls aeProcessEvents(), which figures out how long it can safely block waiting for socket activity (bounded by the nearest scheduled time event), waits, then fires whatever file and time events are now ready. There is no thread pool here executing your commands concurrently — there's one thread taking turns.

Why One Main Loop?

It would be easy to read "single-threaded" as a limitation Redis is stuck with. It's closer to the opposite: it's a limitation Redis chose, because of what it buys back. If two commands could run at the same instant against the same dictionary, every data structure in the codebase would need locking — mutexes around hash table buckets, careful ordering to avoid deadlock, memory barriers to avoid torn reads. All of that is expensive, both in raw CPU cycles and, worse, in engineer time spent finding and fixing races. By committing to "exactly one command executes at a time," Redis's own C code for lists, hashes, and sorted sets can stay refreshingly simple: no lock ever needs to be taken to mutate the keyspace, because nothing else is ever running concurrently with it.

Key Idea

Single-threaded execution isn't the absence of concurrency handling — it's a different, cheaper way of getting the same safety guarantee a lock would give you, paid for once at the architecture level instead of thousands of times throughout the code. Chapter 7 pushes this all the way to its most useful consequence: commands become atomic for free.

A Nuance: I/O Threads Don't Break This

Modern Redis (6.0+) can use extra threads for reading raw bytes off the network and writing raw bytes back — the plain socket I/O, not command execution. That's a throughput optimization for very high-connection-count workloads, and it's opt-in via io-threads. The command itself — parsing what it means, looking it up, mutating the dataset — still happens on the single main thread, one at a time. The "exactly one command runs against the data at once" guarantee this chapter is built on is unaffected.

Warning

Because there's exactly one thread doing the work, a slow command really does stall every other client — there's no other thread to pick up the slack. KEYS * on a large keyspace, a huge SORT, or a Lua script with an accidental infinite loop are the classic ways to learn this the hard way in production. Chapter 3 covers exactly which operations are dangerous and why, and Chapter 11 covers how to catch them (SLOWLOG) before your users do.

Try It Yourself

Start redis-server locally and load a few hundred thousand keys with redis-benchmark -n 300000 -r 300000 -t set -q. Then, in one terminal run redis-cli --bigkeys (it scans the whole keyspace) while in a second terminal you run redis-cli --latency. Watch the latency samples spike for the duration of the scan, then return to baseline the instant it finishes — that's the single event loop, visible from the outside.

The event loop tells you when a socket has something worth reading. It doesn't yet tell you what happens to those bytes once they're read — that's the full trip from raw socket data to a command result, and it's next.
