Why Redis Is Fast (and How to Watch It)

Ask a live server to tell on its own slowest commands: 

Terminal — catching a slow command red-handed$ redis-cli CONFIG SET slowlog-log-slower-than 1000  # microseconds
OK
$ redis-cli KEYS '*'
$ redis-cli SLOWLOG GET 1
1) 1) (integer) 3
   2) (integer) 1752400000
   3) (integer) 1420
   4) 1) "KEYS"
      2) "*"

Nothing in this book so far has been an accident — every chapter has been one specific answer to "how do we stay fast without sacrificing predictability." This chapter pulls those answers together and adds the tools for watching them work in a real, running server.

Memory Efficiency Is Performance

It's tempting to think of "fast" and "memory-efficient" as separate goals, but in an in-memory database they're nearly the same thing: every byte you don't allocate is a byte the CPU cache doesn't have to fetch from main memory, and every allocation you avoid is one less trip through the allocator and one less object for the memory allocator to manage. That's why shared integer objects (Chapter 4), listpacks and intsets (Chapter 5), and SDS's variable-size headers (Chapter 4) all matter for speed, not just for fitting more data in a box. MEMORY USAGE key and MEMORY DOCTOR let you inspect this directly on a live server.

Key Idea

In an in-memory system, a memory optimization is a performance optimization — smaller data structures mean fewer cache misses and less allocator overhead, on every single access, not just a smaller number on an INFO memory report.

Avoiding Latency Spikes: The Repeated Trick

By this chapter, a specific shape of solution should feel familiar: break a potentially large piece of work into small, bounded increments, and spread them across many turns of the event loop instead of doing them all in one command. Incremental rehashing (Chapter 6), the sampling-based active expire cycle (Chapter 10), and SCAN's cursor-based iteration (Chapter 3) are three independent instances of exactly the same idea. The flip side is operational: any command whose cost still scales with dataset size in one shot — KEYS, an unbounded SORT, a large LRANGE, an EVAL with a loop over every key — is a latency spike waiting for the right amount of data to trigger it.

Warning

Redis's own internals doing incremental work doesn't protect you from writing a non-incremental command. A Lua script that loops over every key in the database is just as capable of stalling the server as KEYS * — the engine's cleverness applies to its own bookkeeping, not automatically to application logic you write on top of it.

Observability: Slowlog, Latency Monitor, INFO

Three built-in tools cover most of what you need to actually watch this system, rather than reason about it in the abstract:

  • Slowlog — records the last slowlog-max-len commands (default 128) whose execution time exceeded slowlog-log-slower-than microseconds. SLOWLOG GET/RESET/LEN are your entry points, exactly as shown at the top of this chapter.
  • Latency monitor — samples specific latency-sensitive internal events (fork time, command execution, expire-cycle time, and more) into a time series you can inspect with LATENCY HISTORY <event>, and LATENCY DOCTOR turns the raw samples into a plain-English diagnosis.
  • INFO — sectioned server statistics. INFO stats for throughput and hit-rate counters, INFO commandstats for a per-command call count and total time, and INFO latencystats for per-command latency percentiles (p50/p99/p999 by default, configurable via latency-tracking-info-percentiles).
Try It Yourself

Run redis-cli INFO commandstats after a few minutes of real traffic and find your single most-called command. Then check INFO latencystats for that same command's p99. If the gap between its average and its p99 is large, you've found a candidate worth investigating with the slowlog and latency monitor — the exact workflow an on-call engineer reaches for first.

Pro Tip

MONITOR streams every command the server executes, in real time — extremely useful for a five-second debugging session, and extremely expensive to leave running, since it makes every single command do extra work to format and stream itself to your monitoring connection. Never leave it attached in production longer than you have to.

You now have both halves: the internal tricks that make Redis fast, and the tools to confirm they're working on your own server. What's left is the most useful part — turning all of this from trivia about one codebase into a repeatable way of reading any system, and a set of ideas worth stealing for whatever you build next.
