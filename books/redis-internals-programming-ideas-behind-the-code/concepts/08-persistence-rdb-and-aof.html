Persistence: RDB Snapshots and the Append-Only File

Trigger a background save on a server holding a few gigabytes of data, and keep issuing commands the entire time: 

Terminal — a snapshot that doesn't pause the server$ redis-cli BGSAVE
Background saving started
$ redis-cli SET still:working "yes"   # answers immediately, mid-save
OK
$ redis-cli INFO persistence | grep rdb_bgsave_in_progress
rdb_bgsave_in_progress:1

Writing gigabytes of data to disk is a slow operation by the standards of an in-memory database — exactly the kind of thing Chapter 1 said must never block the main thread. Redis solves this with one well-chosen operating-system trick, applied to both of its persistence formats.

RDB: fork(), Then Get Out of the Way

BGSAVE calls fork(). The child process receives, in effect, a private snapshot of the parent's entire memory — but thanks to the OS's copy-on-write page mapping, that snapshot costs almost nothing up front: parent and child share the same physical memory pages until one of them writes to a page, at which point the OS copies just that page for the writer. The child walks the (now-frozen, from its point of view) dataset and writes it out as one compact binary RDB file; the parent goes right back to serving commands, and any key the parent modifies while the child is still saving simply triggers a copy-on-write for that one page — the child's view stays exactly as it was at the instant of the fork.

Key Idea

fork() plus copy-on-write turns "take a consistent snapshot of a live, changing dataset" from an application-level problem into an operating-system-level one Redis gets almost for free. It's a great example of reaching for the right layer to solve a problem instead of building elaborate machinery yourself.

Warning

fork() itself is not entirely free — copying page tables (not the pages themselves) still takes time proportional to how much memory the process maps, so on a very large dataset the fork call itself can pause the parent for tens to hundreds of milliseconds. That's exactly the kind of latency spike Chapter 1 wants to avoid, which is why BGSAVE frequency and replica full-syncs (Chapter 9, which also forks for RDB) are operational levers worth tuning on memory-heavy deployments.

AOF: Log Every Write, Replay on Restart

The append-only file takes the opposite approach: instead of a point-in-time snapshot, it logs every write command as it happens, so recovery means replaying the log from the start. Three appendfsync policies trade durability against write latency directly:

PolicyBehaviorWorst-case data loss
alwaysfsync after every single writeeffectively none
everysecfsync once per second (the default)up to ~1 second
nolet the OS decide when to flushup to the OS flush interval

everysec is the default for a reason: it's close to always's durability in practice, but at a small fraction of its latency cost, because it batches many writes' worth of fsyncing into one syscall per second instead of one per command.

Rewrite: Compacting the Log

A pure append-only log of every write ever issued would grow forever — ten thousand increments to the same counter is ten thousand log lines for a value GET could reconstruct from just one. BGREWRITEAOF fixes this the same way BGSAVE does: it forks a child that writes a fresh, minimal AOF representing only the dataset's current state, while the parent keeps logging new writes to the old file (and, since Redis 7's multi-part AOF, to a small incremental file alongside a base file, tracked by a manifest) until the rewrite finishes and the files are swapped over.

Try It Yourself

Enable both: redis-cli CONFIG SET appendonly yes, then watch redis-cli INFO persistence for aof_rewrite_in_progress while you run BGREWRITEAOF. Compare the file sizes of dump.rdb and the AOF directory for the same dataset — RDB is almost always smaller, since it stores final values, not the history of commands that produced them.

Crash Recovery and Choosing a Durability Profile

On startup, Redis prefers the AOF over the RDB file if both are present and AOF is enabled — it's the more complete, more recent record. A corrupted AOF tail (from a crash mid-write) is handled with redis-check-aof, which can truncate the log back to the last complete, valid command rather than refuse to start. None of this is automatic magic: choosing RDB-only, AOF-only, or both (the common production choice) is a real decision about how much data loss on crash is acceptable, traded directly against write-path latency and operational complexity — Chapter 1's core theme, once more, now applied to durability instead of raw command latency.

Persistence answers "what happens if this one server crashes." Replication and clustering answer the next question up: what happens when one server isn't enough, either for availability or for scale? That's next.
