The Object Model: One Struct, Many Encodings

Store three strings that look similar to you and ask Redis how it actually stored them: 

Terminal — the same command, three different encodings$ redis-cli SET n 42
OK
$ redis-cli OBJECT ENCODING n
"int"
$ redis-cli SET short "hello there"
OK
$ redis-cli OBJECT ENCODING short
"embstr"
$ redis-cli SET long "a string that runs well past forty-four bytes long"
OK
$ redis-cli OBJECT ENCODING long
"raw"

Same command, same declared type (a Redis string), three completely different memory layouts. That's the theme of this whole chapter and the next: Redis draws a hard line between a value's logical type (what your client sees — a string, a list, a hash) and its physical encoding (how it's actually laid out in RAM). Every value picks the cheapest encoding that fits, silently, with no change to the API you use.

The Generic Object: robj

Every value in the keyspace is wrapped in the same small struct, robj ("Redis object"). It carries a type (string, list, hash, set, zset…), an encoding (which physical representation is in use right now), a reference count for sharing, an LRU/LFU field used by eviction (Chapter 10), and a raw pointer to the actual data. The command handlers in t_string.c, t_list.c, and friends mostly work against this generic wrapper and dispatch internally based on encoding — which is exactly what lets the physical layout change without the command surface changing at all.

Key Idea

Separating "what type is this" from "how is it physically stored" is what lets Redis quietly swap in a more compact representation when a value is small, and a more capable one when it grows — without ever changing the command you call. You'll see this same move made independently for every collection type in Chapter 5.

Strings: int, embstr, raw

A string picks one of three encodings, cheapest first:

  • int — the value parses cleanly as a 64-bit signed integer, so Redis stores the integer itself, no character data at all, directly in the robj's pointer field.
  • embstr ("embedded string") — short strings, up to 44 bytes, get the robj header and the string data allocated together in a single memory chunk.
  • raw — anything longer than 44 bytes gets its own separate allocation for the string data, referenced by pointer from the robj.

That 44-byte line isn't arbitrary. It's chosen so that an embstr's combined allocation — object header plus string header plus content — still fits inside a single 64-byte allocator bucket in jemalloc, the memory allocator Redis uses. One allocation instead of two also means one cache line to touch instead of two, and one free() instead of two when the key is deleted — a small win multiplied across millions of keys.

Shared Objects and the Small-Int Cache

Integers between 0 and 9999 are special-cased even further: Redis pre-allocates one shared robj for each of these small integers at startup and hands out a reference to the shared object (bumping its reference count) instead of allocating a fresh one every time a key happens to hold, say, the number 7. Counters, small IDs, and flags are extremely common values in real workloads, so this one cache quietly avoids an enormous number of tiny allocations.

Pro Tip

OBJECT REFCOUNT key on a small shared integer will report an unusually large number — that's not a bug, it's you observing the shared-integer pool directly. Reference counting only looks unusual here because sharing is normally rare in Redis; once a value is mutated in place, sharing stops (see the warning below).

Under raw and embstr: Simple Dynamic Strings

The actual bytes behind embstr and raw are stored as an SDS (simple dynamic string) rather than a plain C string. An SDS keeps a small header in front of the character data recording the current length and the allocated capacity, which buys two things a C string can't: strlen()-free O(1) length lookups, and safe storage of binary data containing embedded NUL bytes (a plain C string would silently truncate at the first \0). Redis even varies the header's own size (sdshdr5/8/16/32/64) based on how big the string is, so a short string doesn't pay for a header sized for gigabyte-scale content.

Warning

Encoding is a memory optimization Redis picks for you, not a promise about how a value behaves. APPENDing to a short string converts it away from embstr to raw (an embedded string's single allocation isn't resizable in place); once a value moves off a shared small-integer object because you ran INCR or APPEND on it, it gets its own private allocation and stops being shared. None of this changes correctness — only the memory footprint — but it's worth knowing if you're chasing memory numbers with MEMORY USAGE.

Strings are the simplest case: one value, three possible shapes. The collection types — lists, hashes, sets, sorted sets — play the exact same game, just with more encodings and more interesting trade-offs. That's next.
