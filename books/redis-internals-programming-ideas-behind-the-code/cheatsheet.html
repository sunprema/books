The 30-Second Summary

Redis runs one command at a time on a single thread (Ch. 2, 3, 7) — which is why it needs no locks and every single command is naturally atomic. Every value is a generic robj wrapper (Ch. 4) that picks the cheapest physical encoding that fits — int/embstr/raw strings, listpack/intset for small collections, converting one-way to quicklist/hashtable/skiplist once they grow (Ch. 5). The keyspace itself is a dict that grows by incremental rehashing — a few buckets at a time, never all at once (Ch. 6). Durability comes from two cooperating mechanisms: RDB snapshots via fork() + copy-on-write, and the AOF write-ahead log with a tunable fsync policy (Ch. 8). Scale comes from asynchronous replication (full/partial resync via a backlog ring buffer) and Cluster sharding across 16384 hash slots (Ch. 9). Keys clean themselves up via passive + active expiration, and memory pressure triggers approximated LRU/LFU eviction — both sampling-based, never an exhaustive scan (Ch. 10). The one idea behind all of it: never do a big, unpredictable chunk of work when a small, bounded, repeatable one will do.

⚠ Gotchas — Know These Before Production

  • ⚠KEYS *O(N) over the whole keyspace, done in one blocking command (Ch. 2–3). Use SCAN instead — same traversal, incremental cursor.
  • ⚠noeviction is the defaultWith maxmemory set and no explicit eviction policy, Redis starts rejecting writes at capacity instead of evicting anything (Ch. 10).
  • ⚠Encoding conversion is one-wayA collection that grows past its compact-encoding threshold never converts back, even after you delete most of its elements (Ch. 5).
  • ⚠A stuck Lua script or long command stalls everyoneSingle-threaded execution means there's no other thread to pick up the slack (Ch. 2, 3, 7). Keep any one command's work bounded.
  • ⚠Cluster has no cross-slot transactionsA command touching keys in two different hash slots is rejected unless you've forced them together with a {hash tag} (Ch. 9).
  • ⚠Async replication can lose the newest writes on failoverRedis trades the strongest consistency guarantee for simplicity and speed — a deliberate choice, not an oversight (Ch. 9).
