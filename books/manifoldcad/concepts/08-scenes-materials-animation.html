Chapter Eight

Scenes, Materials & Animation

Every script so far exported one solid. But real deliverables are rarely one solid — they're several parts, in different colours, arranged in a hierarchy, sometimes even animated. So let's ask: how do we hand back a whole assembly instead of a single blob? The answer is GLTFNode, a scene node we export in place of a bare Manifold.

Why not just union everything?

Tempting, but think it through. If two parts don't actually touch — a lid and a jar, two puzzle pieces — union fuses them into one nameless blob in the exported file. We'd lose the very thing that makes them separate parts. Wrapping each in its own GLTFNode keeps them distinct, individually coloured, and cleanly organised in the output .glb. Union is for material that's really one solid; nodes are for parts that merely share a scene.

import {GLTFNode, Manifold} from 'manifold-3d/manifoldCAD';

const node = new GLTFNode();
node.manifold = Manifold.cube([20, 20, 20], true);
node.material = { baseColorFactor: [0.2, 0.6, 1.0], metallic: 0 };

export default node;
GLTFNode fieldMeaning
manifoldThe solid this node draws.
materialA glTF PBR material — see below.
translation / rotation / scaleThe node's own transform, relative to its parent.
new GLTFNode(parent)Pass a parent node to build a hierarchy.

Building a hierarchy

Pass a parent to the constructor and the nodes form a tree — and a parent's transform carries its children with it, exactly like nested coordinate frames. The Rounded Frame example does precisely this: a root node holding an "inside" and an "outside" node, with the inside tinted cyan.

const root = new GLTFNode();
root.translation = [0, 0, 60];

const outsideNode = new GLTFNode(root);   // child of root
outsideNode.manifold = outside;

const insideNode = new GLTFNode(root);
insideNode.manifold = inside;
insideNode.material = { baseColorFactor: [0, 1, 1] };

export default getGLTFNodes();            // export the whole tree
Two ways to hand back a scene

We can export default a single node (its descendants come along for the ride), an array of nodes, or call getGLTFNodes() to collect every node we've created. Any of the three works — pick whichever makes the intent clearest at the bottom of the script.

Materials: the glTF metallic-roughness model

Materials follow the standard glTF PBR model, which means they render consistently anywhere a .glb opens — the same intent as writing portable code once and trusting it to run everywhere.

FieldMeaning
baseColorFactor: [r, g, b]Base colour, each channel 0–1.
metallic: 0–10 = plastic/matte, 1 = metal.
roughness: 0–10 = mirror-smooth, 1 = fully diffuse.
attributes: ['COLOR_0']Tell the exporter to read per-vertex colours set via setProperties.

That last field is the bridge back to Chapter Seven: the Menger Sponge and Scallop examples set a vertex colour per point, then declare attributes: ['COLOR_0'] so the gradient survives into the exported model. Set the colour, then remember to ask for it — a two-step handshake worth knowing about before it puzzles you.

Animation with morphs

Remember warp from the last chapter? Here's a lovely reuse of it. If we register that same function as a morph target, the viewer interpolates from the un-warped shape to the warped one over time — an animation, right in the browser, with no new machinery.

import {GLTFNode, Manifold, setMorphStart} from 'manifold-3d/manifoldCAD';

const ball = Manifold.sphere(1, 200);
setMorphStart(ball, func);        // animate FROM this warp toward the final shape

const node = new GLTFNode();
node.manifold = ball;

setAnimationDuration(5);          // seconds
setAnimationMode('ping-pong');    // 'loop' or 'ping-pong'
export default node;

This is the beating — literally — heart of the Heart example. Use setMorphStart/setMorphEnd to pin the warp functions for the first and last frames, and the global setAnimationDuration/setAnimationMode to control playback. The allowed modes are exactly two: 'loop' and 'ping-pong'.

Key idea & export

When a model is more than one solid, think in nodes: each part gets its own colour, transform, and place in the tree. Then export to .glb for a full-colour animated scene, or .3mf when we're sending it to a printer. The build stays code; the deliverable is whatever the moment needs.

So what did we gain — across the whole book? A complete pipeline: primitives, booleans, transforms, 2D-to-3D, organic warps, and now scenes, materials, and animation. Everything is code, everything stays watertight, and nothing needs the network. The final chapter steps back to ask the question that started us — when is describing a shape with code the right call?

Challenges

1. Build a two-piece box (base + lid) as two nodes, colour them differently, and translate the lid up so both are visible. Export a .glb and open it elsewhere — do the colours survive?
2. Give a sphere a per-vertex colour gradient, then export it with and without attributes: ['COLOR_0']. Explain the difference.
3. Animate a cube "breathing" by morphing a scale-warp, once in 'loop' and once in 'ping-pong'. Which reads as more natural, and why?
