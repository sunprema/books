Chapter Four

Moving Things Around

Notice something about every primitive we've made: it's born at the origin. That's a deliberate simplicity, but it means to build anything real we have to move shapes into place. Let's meet the movers — translate, rotate, scale, mirror — and one property of them that makes reuse almost free.

const arm = Manifold.cube([40, 6, 6])
  .translate([0, -3, 0])        // recentre on Y
  .rotate([0, 0, 30])           // spin 30 degrees about Z
  .translate([0, 0, 10]);       // lift 10mm up
Rotations are in DEGREES

Here's the surprise, so let's get ahead of it: unlike almost every other graphics API you've touched, ManifoldCAD's rotate takes degrees, not radians — applied as Euler angles in X, then Y, then Z. Write rotate([0, 0, 90]) for a quarter turn about the vertical axis. No Math.PI, no mental conversion. There's a quiet correctness win too: a 90° turn is exact, so no floating-point dust creeps in.

The four movers

MethodEffect
translate([x, y, z])Slide along each axis (mm). Also accepts separate args: translate(x, y, z).
rotate([x, y, z])Rotate by Euler angles in degrees, applied X→Y→Z.
scale(s)Resize. A single number scales uniformly; [x, y, z] scales per axis.
mirror([nx, ny, nz])Reflect across the plane through the origin with that normal — perfect for left/right symmetric parts.

There's also transform(matrix) if you already hold a 3×4 affine Mat4, and warp(fn) for bending every vertex by a function — we'll give warp its own chapter under "Smooth & Organic," because it earns one.

Immutable & chainable

Here's the property that makes all of this pleasant to work with: transforms never mutate the shape we call them on — they hand back a new one. If you've enjoyed immutable data structures in code, this is the same joy in geometry. It means reuse is trivial: build a part once, then place copies without any of them stepping on each other.

const post = Manifold.cylinder(50, 3);

// Four corner posts from one definition:
const posts = Manifold.union([
  post.translate([  0,   0, 0]),
  post.translate([100,   0, 0]),
  post.translate([  0, 100, 0]),
  post.translate([100, 100, 0]),
]);
Order matters — this is the bug you'll hit

.rotate(...).translate(...) is not the same as .translate(...).rotate(...). Each transform is relative to the shape's current position, and rotation always pivots about the origin. So the reliable recipe is: rotate first, while the shape is still at the origin, then translate it into place. Swap those and your part orbits the origin instead of spinning where it stands.

This is exactly how the built-in Rounded Frame example works, and it's worth studying: it builds a single rounded edge (a cylinder capped with a sphere), then rotates and unions copies of it into a full 12-edge wireframe cube. One part, transformed many ways — the geometric echo of a well-factored function called from a dozen places.

Key idea

Model a piece once at the origin, then use transforms to stamp it across the design. Symmetric and repetitive parts almost write themselves this way — and if a dimension changes, we change it in one place.

So what did we gain? A pipeline metaphor for placement: each transform is a stage, the input is immutable, and copies are free. That's four verbs for where, on top of three for which. Next we pick up a shortcut for a whole class of shapes — the ones that are really 2D outlines in disguise.
