Ownership & Moves

Every other clever thing Rust does grows out of one stubborn little rule. Get this, and the rest of the language stops feeling like a fight.

Let's start with a real problem

Forget Rust for a second. Imagine you've got a library book. Exactly one person checked it out — their name is on the card. When they're done, the book goes back on the shelf. Simple. Now imagine the library lets anybody photocopy the card and claim the book is theirs too. Two people show up to return it. One of them throws it in the trash while the other is still reading it. Chaos. People get hurt.

That, believe it or not, is exactly the bug that has haunted systems programming for fifty years. In C, a chunk of memory is the book. "Returning it" is calling free. If two parts of your program both think they own that memory, one frees it while the other is still reading — and you get a use-after-free, the single most exploited class of security bug on the planet.

So here's the question Rust asks: what if the compiler just enforced the library card rule? One owner. When the owner leaves, the book goes back. No photocopying. Let's build that up from nothing.

Rule one: every value has exactly one owner

In Rust, when you bind a value to a variable, that variable owns the value. Watch:

rustfn main() {
    let book = String::from("The Feynman Lectures"); // `book` owns this text
    println!("{book}");
} // `book` goes out of scope here → Rust automatically frees the memory

Notice what you didn't write: no free, no delete, no garbage collector humming in the background. When book reaches the closing brace, it goes out of scope, and Rust runs its cleanup (a function called drop) for you. The owner left; the book went back on the shelf. This is the whole deal: cleanup is tied to ownership, and ownership is tied to scope.

🔑
Key idea

The three rules of ownership, in full:

1. Each value has a variable that's its owner. 2. There can be only one owner at a time. 3. When the owner goes out of scope, the value is dropped.

Now the surprising part: a move

Here's where newcomers trip, so let's go slow and physical. What do you think happens here?

rust · won't compilelet a = String::from("hello");
let b = a;            // did we copy the string? ...no.
println!("{a}");     // ❌ error: borrow of moved value: `a`

"But I just assigned it!" Right. And in most languages b = a would either copy the whole string or make both names point at the same data. Rust refuses both, and here's why — this is the beautiful part.

A String is really two things: a little bookkeeping struct on the stack (a pointer, a length, a capacity) and the actual character bytes out on the heap. When you write let b = a, Rust copies the cheap little struct... but now two structs point at the same heap bytes. Remember the library card? Two owners. When both go out of scope, both try to free the same memory — the classic double-free.

let b = a — the heap bytes aren't copied; ownership moves to b, and a is retired

Rust's fix is delightfully blunt: when you assign a to b, it moves ownership. b is now the one owner, and a is marked dead — the compiler simply won't let you touch it again. No copy, no double-free, no runtime check. The bug is impossible because the code that would cause it doesn't compile.

💡
But why a move and not a copy?

Because copying the heap bytes might be huge and slow, and Rust never hides a cost like that from you. A move is always cheap — just a few machine words on the stack. If you genuinely want the deep copy, you ask for it by name: let b = a.clone();. Now both a and b are valid, each owning its own heap allocation. Explicit. Honest.

The exception that proves the rule: Copy

Now you might say: "Wait, surely simple things like integers don't move?" Correct, and your intuition is exactly right. Try this:

rust · compiles finelet x = 5;
let y = x;        // x is copied, not moved
println!("{x} and {y}"); // ✅ both work

Why the difference? An i32 lives entirely on the stack — there's no heap allocation behind it, so there's nothing to double-free. Copying four bytes is as cheap as moving them. So for these types Rust just copies, and both names stay valid. Types that opt into this behave-like-a-value-not-a-resource deal implement a trait called Copy.

Is Copy (cheap, duplicated)Is not Copy (moved)
All integers: i32, u64, …String
Floats: f32, f64Vec<T>
bool, charBox<T> and most owning types
Tuples/arrays of Copy typesAnything that owns a heap allocation or other resource
🧭
Rule of thumb

If a type manages a resource (memory, a file, a socket), it moves — because two owners would mean two cleanups. If a type is just plain bits with no cleanup, it's Copy. The compiler decides this for you; you just need to know which world you're in.

Functions move too

This is the same rule, just walking through a door. Passing a value to a function hands ownership to that function:

rustfn consume(s: String) {
    println!("I now own {s}");
} // s dropped here

fn main() {
    let name = String::from("Ferris");
    consume(name);          // ownership moves INTO consume
    // println!("{name}"); ❌ name is gone — consume ate it
}

Returning a value moves ownership back out. So one common early pattern is "take it, use it, hand it back" — though as you'll see in the very next chapter, there's a far nicer way: borrowing, where you lend a value without giving it up. That's where ownership stops feeling restrictive and starts feeling like freedom.

⚠️
Common first-week confusion

"Use of moved value" is the error you'll hit most. It is never the compiler being mean — it's the compiler catching a real bug at compile time that, in another language, would be a crash or a vulnerability at 3 a.m. Read it as: "you tried to use something you already gave away."

The thirty-second version

  • One value, one owner. Owner leaves scope → value is dropped (cleaned up) automatically.
  • Assigning or passing a non-Copy value moves ownership; the old name is dead.
  • Want a real duplicate? Say so: .clone().
  • Small stack-only types (i32, bool, char…) are Copy — duplicated, both names live.
  • No garbage collector, no manual free, no double-frees. The rules are checked at compile time and cost nothing at runtime.
