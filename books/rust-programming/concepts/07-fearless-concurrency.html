Fearless Concurrency

This is the chapter the whole book has been building toward. The very same ownership and borrowing rules that felt strict in chapter 1 turn out to make multithreaded bugs impossible to compile. Rust calls it fearless concurrency, and the name is earned.

Why concurrency is so terrifying everywhere else

Threads share memory. Two threads, one piece of data, at least one of them writing, no coordination — that's a data race, and it's a nightmare: the bug appears once in ten thousand runs, never in the debugger, never on your machine, only in production at peak traffic. Decades of tooling — sanitizers, race detectors, code review rituals — exist just to chase these ghosts after the fact.

Now look back at chapter 2's golden rule: many readers XOR one writer, never both at once. Read it again with threads in mind. That rule is the definition of "no data race." Rust already enforces it at compile time for single-threaded aliasing — it simply extends the very same check across threads. The bug class doesn't get caught earlier; it stops being expressible.

🔑
Two traits do the heavy lifting

The compiler reasons about thread safety with two marker traits it applies automatically:

Send — a type is safe to transfer ownership of to another thread.
Sync — a type is safe to share by reference (&T) across threads.

Almost everything is Send + Sync automatically. The types that aren't — like Rc<T>, whose reference count isn't atomic — are rejected by the compiler the moment you try to send them across a thread boundary. You don't annotate this; you just can't compile the unsafe thing.

Spawning threads, and the move that makes it safe

rustuse std::thread;

fn main() {
    let data = vec![1, 2, 3];

    let handle = thread::spawn(move || {     // `move` gives the closure ownership of `data`
        println!("from thread: {data:?}");
    });

    handle.join().unwrap();              // wait for the thread to finish
}

Why move? Because the new thread might outlive main's stack frame. If the closure merely borrowed data, that borrow could dangle the instant main returned — and chapter 2 already taught the compiler to forbid dangling references. So it requires you to move ownership in. Same lifetime rule, now keeping threads honest.

Sharing state: Arc + Mutex

Sometimes threads genuinely must share one piece of mutable state — a counter, a cache. Rust makes you build it out of two honest pieces, and the type you end up with is provably correct:

  • Mutex<T> — a lock. To touch the T you must call .lock(), which hands you a guard. While you hold the guard, you have exclusive access; when it drops, the lock releases (RAII again — you can't forget to unlock).
  • Arc<T> — an atomically reference-counted handle, so multiple threads can each own a share of the same allocation. (It's Rc's thread-safe twin from chapter 3.)
rust · ten threads, one counteruse std::sync::{Arc, Mutex};
use std::thread;

fn main() {
    let counter = Arc::new(Mutex::new(0));
    let mut handles = vec![];

    for _ in 0..10 {
        let counter = Arc::clone(&counter); // bump the refcount, hand a share to the thread
        handles.push(thread::spawn(move || {
            let mut n = counter.lock().unwrap(); // acquire the lock → exclusive access
            *n += 1;
        })); // guard drops here → lock auto-released
    }
    for h in handles { h.join().unwrap(); }

    println!("total = {}", *counter.lock().unwrap()); // always 10 — never a race
}
💡
The type system did the proof

Try to skip the Mutex and share a raw counter — won't compile (no synchronized access). Try to use Rc instead of Arc — won't compile (Rc isn't Send). Try to read the value without locking — won't compile (the data is inside the Mutex; the only door in is .lock()). Every wrong version is a build error. The one that compiles is the one that's correct. That is what "fearless" means: you refactor threaded code aggressively because if it builds, it doesn't have a data race.

The other style: message passing

Shared state isn't the only model. Often the cleaner design is "don't communicate by sharing memory; share memory by communicating." Rust ships channels for exactly this — threads send values to each other, and ownership moves through the channel:

rustuse std::sync::mpsc;     // "multiple producer, single consumer"
use std::thread;

fn main() {
    let (tx, rx) = mpsc::channel();
    thread::spawn(move || {
        tx.send(String::from("hi from the worker")).unwrap();
        // after send, the String is GONE from this thread — ownership moved
    });
    let msg = rx.recv().unwrap(); // blocks until a value arrives
    println!("main got: {msg}");
}

Because send moves the value, there's no shared mutable state to race over at all — the data is only ever owned by one thread at a time. Ownership turns a concurrency discipline into a compiler-checked guarantee.

🧭
And then there's async

For workloads that are I/O-bound — thousands of network connections mostly waiting — OS threads are too heavy. Rust's async/.await lets you write thousands of concurrent tasks on a handful of threads, scheduled by a runtime like Tokio. The 2024 edition matured this further with async closures (async || { … }) and the AsyncFn trait family. The mental model is the same — futures are values, ownership and Send still govern what may cross task boundaries — just cooperatively scheduled instead of pre-emptively.

⚠️
One thing Rust does NOT prevent

Be honest about the boundary: Rust eliminates data races, not deadlocks. If thread A locks mutex 1 then waits for mutex 2, while thread B locks mutex 2 then waits for mutex 1, you'll hang — and the compiler won't catch it, because each individual lock is perfectly valid. Lock ordering and avoiding held-locks-across-.await are still on you. "Fearless" means race-free, not logic-bug-free.

The thirty-second version

  • The borrow rule (readers XOR writer) is the no-data-race rule — Rust extends it across threads at compile time.
  • Send = safe to move to another thread; Sync = safe to share by reference. Auto-derived; unsafe types are rejected.
  • thread::spawn(move || …) moves owned data into a thread so nothing can dangle.
  • Share mutable state with Arc<Mutex<T>>: atomic shared ownership + a lock whose guard auto-releases.
  • Or pass ownership through channels (mpsc); for massive I/O concurrency, use async/.await on a runtime like Tokio.
  • Rust prevents data races — not deadlocks. Mind your lock ordering.
