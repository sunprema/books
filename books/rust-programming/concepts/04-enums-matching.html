Concept 04 · Making Illegal States Unrepresentable

Enums & Pattern Matching

Rust's enums aren't the feeble lists of constants you've met before. They can hold data, and combined with match they let you carve a problem into cases the compiler forces you to handle — all of them.

An enum is a "one of these" type

Here's a thought. Suppose you're modeling a message in a chat app. A message is exactly one of a few kinds: a quit signal, a bit of text, a color change. In a lot of languages you'd fake this with an integer tag plus a pile of maybe-null fields, and pray everyone reads the tag before the fields. Rust gives you a type that says it directly:

rustenum Message {
    Quit,                          // no data
    Write(String),                 // holds a String
    Move { x: i32, y: i32 },       // holds named fields, like a struct
    SetColor(u8, u8, u8),         // holds a tuple
}

Each variant can carry its own payload, of its own shape. A Message value is always exactly one variant. There is no way to be a Write and a Move at once, and no way to have a Write with a missing string. The illegal states simply can't be written down. That's the whole game: make illegal states unrepresentable.

The two enums you'll use every single day

Rust's standard library leans so hard on this idea that two enums replace entire language features from elsewhere.

Option<T> — instead of null

enum Option<T> {
    Some(T),
    None,
}

"Either there's a value, or there isn't." Rust has no null. A thing that might be absent has type Option<T>, and the compiler won't let you use the inside without first checking for None. Tony Hoare called null his "billion-dollar mistake" — Rust just deleted it.

Result<T, E> — instead of exceptions

enum Result<T, E> {
    Ok(T),
    Err(E),
}

"Either it worked and here's the value, or it failed and here's why." Errors are ordinary values you must deal with, not invisible exceptions that unwind your stack. We give this its own chapter next.

match: a fork in the road that checks every path

An enum is only half the tool. The other half is match — and it's not your grandfather's switch statement. match compares a value against patterns, runs the first that fits, and pulls the data right out of the variant:

rustfn process(msg: Message) {
    match msg {
        Message::Quit => println!("goodbye"),
        Message::Write(text) => println!("text: {text}"),   // binds the String to `text`
        Message::Move { x, y } => println!("move to ({x}, {y})"), // destructures the fields
        Message::SetColor(r, g, b) => println!("#{r:02x}{g:02x}{b:02x}"),
    }
}
🔑
Exhaustiveness — the superpower

match must cover every possibility. Delete one arm above and the program won't compile: "pattern Quit not covered." Now fast-forward six months: you add a Message::Ping variant. The compiler instantly lights up every match in your codebase that forgot to handle it. The type system turns "did I update all the places?" from a prayer into a guarantee.

Patterns are a little language of their own

Once you see patterns, you see them everywhere. They can match literals, ranges, bind variables, ignore parts, and add conditions called guards:

rustlet n = 7;
match n {
    0 => println!("zero"),
    1 | 2 | 3 => println!("small"),         // multiple patterns with |
    4..=9 => println!("medium"),             // an inclusive range
    x if x % 2 == 0 => println!("big even"),  // a guard
    _ => println!("big odd"),                 // _ = catch-all
}

When you only care about one case: if let

A full match is overkill when you want just one variant and don't care about the rest. if let is the shorthand:

rustlet maybe = Some(42);

// instead of a two-arm match:
if let Some(value) = maybe {
    println!("got {value}");
} else {
    println!("nothing");
}

// or loop until a pattern stops matching:
let mut stack = vec![1, 2, 3];
while let Some(top) = stack.pop() {
    println!("{top}");  // 3, 2, 1
}
🧭
Newer sugar: let-else

The 2024-edition idiom for "get the value or bail out" is let ... else. It binds on the happy path and must diverge (return/break/panic) on the unhappy one, keeping your main logic un-indented:

let Some(config) = load() else {
    return;  // no config → leave now
};
// `config` is available, unwrapped, for the rest of the function
💡
Why this design pays off

Enums + exhaustive match push errors from runtime to compile time. The classic crash — "I forgot the null/empty/error case" — becomes a build failure with a precise message. You spend your debugging budget on logic, not on whack-a-mole with states you forgot existed. This is the same philosophy as ownership: encode the invariant in the type, let the compiler enforce it, pay nothing at runtime.

The thirty-second version

  • An enum is a "one of these" type, and each variant can carry its own data.
  • Option<T> replaces null; Result<T, E> replaces exceptions. Both are just enums.
  • match destructures a value and must handle every case — exhaustiveness is checked by the compiler.
  • Patterns support literals, ranges (4..=9), alternatives (|), guards (if), and _ catch-all.
  • if let / while let / let-else are ergonomic shorthands for single-case matching.
