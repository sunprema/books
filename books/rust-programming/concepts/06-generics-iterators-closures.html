Concept 06 · Abstraction Without a Bill

Generics, Iterators & Closures

Write code once that works for every type, express loops as pipelines of transformations, and pass behavior around like data — all compiling down to the same tight machine code you'd write by hand. This is Rust's "zero-cost abstraction" promise made concrete.

Generics — write it once, for any type

You've already used generics without flinching: Vec<T>, Option<T>, Result<T, E>. The T is a placeholder — "some type, I don't care which." You can write your own functions and structs the same way:

rust// works for any T that can be compared with >
fn largest<T: PartialOrd>(items: &[T]) -> &T {
    let mut biggest = &items[0];
    for item in items {
        if item > biggest { biggest = item; }
    }
    biggest
}

let nums = vec![3, 7, 2];
let words = vec!["pear", "fig", "plum"];
largest(&nums);   // 7
largest(&words);  // "plum"

The T: PartialOrd is a trait bound (remember chapter 3): it says "T must be a type that knows how to be ordered." That's the contract that lets you write item > biggest. Generics and traits are two halves of one tool — generics give you the placeholder, traits constrain what it's allowed to do.

💡
Why generics cost nothing

At compile time Rust performs monomorphization: it generates a separate, specialized version of largest for each concrete type you actually call it with — one for i32, one for &str. So at runtime there's no "generic dispatch," no boxing, no type tags. The generated code is identical to what you'd have written by hand for each type. Abstraction at the source level, zero abstraction in the binary.

Closures — functions you can carry around

A closure is an anonymous function that can capture variables from where it's defined. Think of it as a little packet of behavior-plus-context you can hand to other code:

rustlet factor = 3;
let scale = |x| x * factor;  // captures `factor` from the surrounding scope
println!("{}", scale(10));    // 30

How a closure captures depends on what it does with the variable, and it mirrors the borrowing rules you already know — Rust picks the least-demanding option automatically:

Closure traitHow it capturesCall it
Fnby shared reference &Tmany times, read-only
FnMutby mutable reference &mut Tmany times, can mutate
FnOnceby value (takes ownership, e.g. move)once

The move keyword forces a closure to take ownership of what it captures — essential when you send a closure to another thread (chapter 7), so the data goes with it.

Iterators — loops as pipelines

Here's the idea that changes how you write Rust. Instead of an index-fiddling for loop, you describe a pipeline: a source of items, a series of transformations, and a final step that consumes them. Each stage is a method; closures do the per-item work:

rustlet nums = vec![1, 2, 3, 4, 5, 6];

let sum_of_even_squares: i32 = nums
    .iter()                       // 1,2,3,4,5,6
    .filter(|n| *n % 2 == 0)     // 2,4,6
    .map(|n| n * n)               // 4,16,36
    .sum();                       // 56

Read it top to bottom like a sentence: take the numbers, keep the evens, square them, add them up. No off-by-one bugs, no mutable accumulator to forget, no index out of bounds. This is far harder to get wrong than the hand-rolled equivalent.

🔑
Lazy & zero-cost

Iterator adapters like map and filter are lazy — they do nothing until a consumer like sum(), collect(), or a for loop drives them. Nothing is allocated for the intermediate "2,4,6" or "4,16,36" lists; the values flow through one at a time. And because it all monomorphizes and inlines, this pipeline compiles to essentially the same loop you'd write by hand — often faster, since the compiler can elide bounds checks it can prove are safe. Elegance with no performance tax.

Know the two halves of the iterator world:

Adapters (lazy, return another iterator)Consumers (eager, end the chain)
map, filter, take, skip, enumerate, zip, rev, flat_map, chaincollect, sum, count, fold, for_each, find, any, all, max

The three ways to start a pipeline echo ownership exactly:

rustv.iter()        // yields &T      — borrow each item
v.iter_mut()    // yields &mut T  — borrow mutably
v.into_iter()   // yields T       — consume the collection, take ownership

And collect() reverses the whole thing — it pours an iterator back into a collection. The target type tells it what to build:

rustlet doubled: Vec<i32> = (1..=5).map(|x| x * 2).collect();
// even into a HashMap, or — neatly — collecting Results:
let parsed: Result<Vec<i32>, _> =
    vec!["1", "2", "3"].iter().map(|s| s.parse::<i32>()).collect();
// → Ok(vec![1,2,3]); if any parse failed, the whole thing is Err. Magic, but principled.
🧭
Returning iterators & closures

You'll often want a function to return a pipeline or a closure without naming its gnarly concrete type. Use impl Trait: fn evens() -> impl Iterator<Item = i32> or fn adder(n: i32) -> impl Fn(i32) -> i32. It says "some type that implements this trait — you don't need to know which," and keeps static dispatch's zero cost.

The thirty-second version

  • Generics (<T>) write code once for many types; trait bounds (T: Trait) say what T must be able to do.
  • Monomorphization makes generics zero-cost — specialized code per type, no runtime dispatch.
  • Closures (|x| …) are anonymous functions that capture their environment as Fn/FnMut/FnOnce; move takes ownership.
  • Iterators turn loops into readable, lazy pipelines of adapters (map, filter) ending in a consumer (collect, sum, fold).
  • iter/iter_mut/into_iter mirror borrow/mut-borrow/own; impl Trait returns iterators and closures cleanly.
