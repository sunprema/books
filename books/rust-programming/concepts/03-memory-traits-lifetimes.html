Memory, Traits & Lifetimes

Three big ideas in one chapter, because they're really one idea seen from three angles: how Rust lays out data, how it shares behavior, and how it proves references stay valid. This is where Rust earns its keep.

Part 1 — Memory: the stack, the heap, and smart pointers

Let's get physical. Your program has two places to put things.

The stack is a stack of plates. Push a plate on top, pop it off — lightning fast, perfectly ordered, and the size of every plate must be known in advance. Local variables live here. The heap is a big messy warehouse: you ask for space, the allocator finds a shelf, hands you back the address (a pointer), and you can store things whose size you didn't know until runtime — like a string the user types, or a list that grows.

The stack is free; the heap costs a trip to the warehouse. So Rust keeps things on the stack whenever it can, and only reaches for the heap when it must — and when it does, it wraps the heap address in a smart pointer: a stack-sized handle that owns the heap data and knows how to clean it up.

rust · a recursive type needs Box// A cons-list can't be sized — it contains itself! Box gives it a known size.
enum List {
    Cons(i32, Box<List>),  // Box<List> is just a pointer → fixed size
    Nil,
}
use List::{Cons, Nil};

let list = Cons(1, Box::new(Cons(2, Box::new(Nil))));
🔑
RAII — the quiet hero

"Resource Acquisition Is Initialization": a resource's lifetime is tied to a value's scope. When the owner drops, the Drop trait runs and the resource is released — heap memory freed, file closed, lock unlocked. You saw this in chapter 1 with String. It's the same machinery for every resource. No finally blocks, no leaks, no garbage collector pauses. Deterministic cleanup, exactly when the value dies.

rust · Drop runs automaticallystruct Guard(&'static str);
impl Drop for Guard {
    fn drop(&mut self) { println!("releasing {}", self.0); }
}
fn main() {
    let _a = Guard("first");
    let _b = Guard("second");
} // prints "releasing second" then "releasing first" — reverse order, automatic

Part 2 — Traits: shared behavior without inheritance

Most languages share behavior through inheritance: a Dog is-a Animal, inherits its guts, and you climb a family tree. Rust throws that tree away and asks a sharper question: not "what is this thing?" but "what can this thing do?" A trait is a named set of capabilities — an interface, a contract. Anything that fulfills the contract can be used wherever the contract is required, related or not.

rusttrait Summary {
    fn summarize(&self) -> String;          // required method
    fn preview(&self) -> String {            // default method — free if you don't override
        format!("{}…", &self.summarize()[..10.min(self.summarize().len())])
    }
}

struct Article { title: String, body: String }

impl Summary for Article {
    fn summarize(&self) -> String {
        format!("{}: {}", self.title, self.body)
    }
}

Now, how do you write a function that accepts "anything summarizable"? Two ways — and the choice between them is one of the most important things to understand in Rust.

Static dispatch — generics

fn notify<T: Summary>(item: &T) {
    println!("{}", item.summarize());
}
// or the sugar:
fn notify(item: &impl Summary) { … }

The compiler stamps out a specialized copy for each concrete type (monomorphization). Zero runtime cost — the call is as direct as if you'd hand-written it. The price: bigger binaries, and the type must be known at compile time.

Dynamic dispatch — trait objects

fn notify(item: &dyn Summary) {
    println!("{}", item.summarize());
}
// store a mixed bag:
let feed: Vec<Box<dyn Summary>> = vec![…];

One function, one copy. The exact method is looked up at runtime through a vtable (a small table of function pointers). Tiny indirection cost, but you can mix different concrete types in the same Vec. That's the trade.

💡
Generics vs. dyn — how to choose

Default to generics (static dispatch): faster, and the compiler keeps full type information. Reach for dyn Trait when you genuinely need a heterogeneous collection — a list of different shapes that all implement Draw, a plugin registry — or when monomorphizing every type would bloat the binary. Same trait, two strategies, you pick the cost.

Traits also power Rust's most-loved convenience, derive. Common traits can be auto-implemented:

rust#[derive(Debug, Clone, PartialEq)]
struct Point { x: i32, y: i32 }
// now Point gets {:?} printing, .clone(), and == for free

And traits are how operators, iteration, formatting, and conversion all work: + is the Add trait, for loops consume the Iterator trait, {} printing is Display, ? conversions use From. The whole standard library is stitched together with traits.

Part 3 — Lifetimes: proving references stay valid

Back in chapter 2 we said a reference can never outlive what it points to. Lifetimes are simply the names the compiler uses to keep that promise. A lifetime is not how long a value lives at runtime — it's a compile-time label on a reference, a region of code for which that reference is guaranteed valid.

Here's the catch the compiler runs into. Consider a function returning the longer of two strings:

rust · won't compile without helpfn longest(a: &str, b: &str) -> &str {
    if a.len() > b.len() { a } else { b }
}

The compiler asks a fair question: "The thing you return is borrowed — but borrowed from a or from b? I need to know, so I can stop the caller from dropping it too early." You answer with a lifetime parameter, 'a — a label that says "all of these references share the same validity window":

rust · annotatedfn longest<'a>(a: &'a str, b: &'a str) -> &'a str {
    if a.len() > b.len() { a } else { b }
}

Read it as: "the returned reference lives as long as the shorter-lived of a and b." You haven't changed any runtime behavior — you've just given the compiler the proof it needed. If the caller tries to use the result after one input has dropped, the build fails. Dangling reference: impossible.

🧭
Good news: you rarely write them

The compiler applies lifetime elision rules that infer the obvious cases for you — which is why the length(&String) function back in chapter 2 needed no annotations. You only write explicit lifetimes when a function returns a reference and there's genuine ambiguity about where it came from, or when a struct stores a reference. The 2024 edition relaxed several elision rules further, so you'll write them even less often.

When a struct holds a reference, you annotate it so the struct can never outlive the data it points into:

ruststruct Excerpt<'a> {
    part: &'a str,   // Excerpt may not outlive the str it borrows
}

One special lifetime is worth knowing: 'static, meaning "valid for the entire program." String literals are &'static str because they're baked into the binary and never go away.

⚠️
The biggest lifetime misconception

'a does not make a reference live longer. It only describes and relates existing lifetimes so the compiler can check them. You're an accountant labeling who-borrows-from-whom, not a wizard extending anyone's life. If a lifetime annotation makes the borrow checker happy but the data really does die early, that's a bug it's catching, not a hoop it's making you jump through.

Putting the three together

See how they interlock? Memory says where data lives and who frees it. Traits say what behavior data shares, with you choosing the dispatch cost. Lifetimes let the compiler prove every reference into that memory stays valid. None of them costs anything at runtime — they're all the compiler doing bookkeeping at build time so your program can run flat-out, safely.

The thirty-second version

  • Stack = fast, fixed-size; heap = flexible, behind a smart pointer (Box, Rc, Arc) that owns and frees it.
  • Drop + RAII = deterministic, automatic cleanup when an owner goes out of scope. No GC.
  • Traits define shared behavior. Use generics (static dispatch, zero-cost) by default; use dyn Trait (vtable) for heterogeneous collections.
  • derive auto-implements common traits; operators and core features are all traits.
  • Lifetimes ('a) are compile-time labels that prove references never outlive their data. Usually elided; 'static = whole program.
