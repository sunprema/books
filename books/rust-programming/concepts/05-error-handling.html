Concept 05 · Errors as Values

Errors: Option, Result & the ? Operator

Rust has no exceptions. Failure is just a value you carry in your hands — and a single character, ?, makes carrying it almost invisible.

Two kinds of "something went wrong"

Let's be careful, because people lump these together and shouldn't. There are two genuinely different situations:

  • Recoverable — the file isn't there, the input won't parse, the network blinked. Normal life. The caller might reasonably want to retry, use a default, or report it. → use Result<T, E>.
  • Unrecoverable — a bug. You indexed past the end of an array; an invariant you swore was true isn't. There's nothing sensible to do but stop. → panic!.

Most languages blur these into one exception mechanism. Rust keeps them apart on purpose, because they call for completely different responses.

panic! — the emergency stop

rustlet v = vec![1, 2, 3];
v[99];  // panics: index out of bounds — this is a bug, so stop the thread

A panic unwinds the current thread, running every Drop on the way (so resources still get cleaned up), and by default aborts the program. You reach for panic! — or its cousins unwrap() and expect() — only when continuing would be worse than crashing: a violated invariant, or a quick prototype where you haven't written the real handling yet.

⚠️
unwrap() is a loaded gun

opt.unwrap() means "give me the value, and if it's None/Err, panic." It's fine in tests, examples, and throwaway scripts. In real code, every unwrap() is a potential crash you've signed off on. Prefer expect("a reason") when you must — the message becomes your crash report — and prefer the ? operator (below) almost everywhere else.

Result — failure you must acknowledge

For recoverable errors, a function hands back a Result. The caller can't pretend it didn't: the value is the success-or-failure, and you can't read the success without confronting the failure.

rustuse std::fs::File;

fn main() {
    let result = File::open("data.txt");
    match result {
        Ok(file) => println!("opened {file:?}"),
        Err(e) => println!("couldn't open: {e}"),
    }
}

That's honest but verbose, and most of the time what you actually want is: "if this failed, stop what I'm doing and pass the error up to my caller." Writing that match by hand at every step would bury your logic. Enter the star of the chapter.

The ? operator — the whole point

Put a ? after any Result (or Option) expression. It means: "unwrap the Ok and keep going; or, if it's Err, return that error from this function right now." Watch a multi-step operation collapse into something you can actually read:

Without ? — the staircase

fn read() -> Result<String, io::Error> {
    let f = File::open("x");
    let mut f = match f {
        Ok(f) => f,
        Err(e) => return Err(e),
    };
    let mut s = String::new();
    match f.read_to_string(&mut s) {
        Ok(_) => Ok(s),
        Err(e) => Err(e),
    }
}

With ? — the intent

fn read() -> Result<String, io::Error> {
    let mut s = String::new();
    File::open("x")?.read_to_string(&mut s)?;
    Ok(s)
}
// same behavior, same safety —
// the errors still propagate, you
// just don't have to see the plumbing.
🔑
What ? really does

expr? expands to roughly: match expr { Ok(v) => v, Err(e) => return Err(e.into()) }. Two things to notice. First, it can only be used in a function that itself returns Result (or Option) — the error has to have somewhere to go. Second, that .into() automatically converts the error into your function's error type via the From trait. That conversion is what makes mixing error types painless.

One error type to rule the function

Real functions fail in several ways — an I/O error here, a parse error there. ? only works if all of them can become one error type. Two common solutions:

rust · the easy way for apps// Box<dyn Error> accepts any error type — great for app/main code.
use std::error::Error;

fn run() -> Result<(), Box<dyn Error>> {
    let text = std::fs::read_to_string("port.txt")?; // io::Error → Box<dyn Error>
    let port: u16 = text.trim().parse()?;            // ParseIntError → Box<dyn Error>
    println!("port = {port}");
    Ok(())
}
🧭
Ecosystem shortcuts

In practice the Rust community reaches for two tiny crates. anyhow gives you an easy catch-all error type for applications (think "Box<dyn Error> with superpowers and a backtrace"). thiserror helps you derive clean, specific error enums for libraries, where callers want to match on exactly what went wrong. Apps → anyhow; libraries → thiserror. Both are built entirely on the Result/?/From machinery you just learned.

main can return a Result too

rustfn main() -> Result<(), Box<dyn std::error::Error>> {
    let text = std::fs::read_to_string("data.txt")?;
    println!("{} bytes", text.len());
    Ok(())  // returning Err makes the process exit non-zero and prints the error
}

So even your entry point can use ? all the way down. The error bubbles to main, the runtime prints it, and the process exits with a failure code — no exceptions, no surprises.

💡
Why errors-as-values beats exceptions

With exceptions, any line might secretly throw, and control can leap anywhere — you can't tell from a function's signature what might go wrong. In Rust, fallibility is written into the return type: -> Result<T, E> says "this can fail" right on the label. You can't accidentally ignore it (the compiler warns on unused Results), and ? makes handling it as terse as the exception version while staying completely explicit. Honesty and brevity — you usually only get to pick one.

The thirty-second version

  • Bugs → panic! (stop). Expected failures → Result<T, E> (handle).
  • unwrap()/expect() turn a missing value into a panic — fine for tests, risky in production.
  • ? unwraps Ok or early-returns the Err, auto-converting it via From. Only usable in functions returning Result/Option.
  • Unify mixed errors with Box<dyn Error> (or anyhow for apps, thiserror for libraries).
  • fn main() -> Result<…> lets ? reach all the way to the top.
