Concept 02 · Lending Without Losing

Borrowing & References

Moving everything around would be exhausting. So Rust lets you lend a value — and the rules of lending turn out to prevent a whole category of bugs for free.

The problem with giving everything away

Last chapter we saw that handing a String to a function moves it — the function eats it. But that's absurd for something simple. If I ask you "how long is this word?", I don't have to give you the word forever. I let you look at it, you tell me the length, and I keep it. That act of looking-without-taking is a reference, and the looking is called borrowing.

rustfn length(s: &String) -> usize {  // & means "I just want to borrow"
    s.len()
} // s is a reference, so nothing is dropped here — we never owned it

fn main() {
    let word = String::from("oxide");
    let n = length(&word);  // lend it with &
    println!("{word} has {n} letters"); // ✅ we still own word!
}

The & creates a reference — a little signpost that points at word without owning it. When the reference goes out of scope, nothing gets dropped, because a borrower was never responsible for cleanup. The owner stays the owner.

Two flavors of reference

Here is where Rust gets clever. There are exactly two kinds of borrow, and the difference is everything:

ReferenceWrittenMeansHow many?
Shared / immutable&T"I'll read, I promise not to change it"As many as you like
Exclusive / mutable&mut T"I need to change it, hands off"Exactly one

To change something through a reference, you need a mutable reference, and the value itself must be declared mut:

rustfn shout(s: &mut String) {
    s.push_str("!!!");  // mutate through the borrow
}

fn main() {
    let mut msg = String::from("fearless");
    shout(&mut msg);
    println!("{msg}"); // fearless!!!
}

The one rule that rules them all

Now the heart of the matter. Rust enforces this, at compile time, with no exceptions:

🔑
The borrowing rule

At any given moment, for any piece of data, you may have either:

• any number of immutable references &T (many readers), or
• exactly one mutable reference &mut T (one writer),

but never both at the same time.

Feynman would tell you to find the everyday version of this, so here it is: it's a whiteboard in a meeting room. Any number of people can read the whiteboard at once — reading doesn't interfere with reading. But if someone is going to erase and rewrite it, everyone else has to stop reading, because otherwise you'd be reading half-erased nonsense. One writer, or many readers. Never a writer while others read.

aliasing XOR mutation — the invariant the borrow checker guards
💡
Why this rule is a stroke of genius

A data race — two threads touching the same memory, at least one writing, with no coordination — is undefined behavior and the source of countless heisenbugs. Look at the rule again: "no writer while there are readers" is exactly what prevents a data race. Rust enforces it for single-threaded aliasing bugs too (like mutating a vector while you're iterating it). One rule, an entire bug class gone. And because it's checked at compile time, it costs zero at runtime.

Watch the borrow checker catch a real bug

rust · won't compilelet mut v = vec![1, 2, 3];
let first = &v[0];   // immutable borrow starts here
v.push(4);         // ❌ needs &mut v — but `first` is still reading!
println!("{first}");

Why is this a bug and not just Rust nagging? Because push might need to grow the vector, which means allocating new memory and moving all the elements there — leaving first pointing at freed memory. A dangling pointer! In C++ this compiles and occasionally explodes. In Rust it simply won't build. The fix is to finish reading before you write — and thanks to non-lexical lifetimes, a borrow ends at its last use, not at the closing brace, so the compiler is smarter than you'd fear.

References can't outlive what they point to

One more guarantee falls out of all this: a reference is never allowed to dangle. You cannot return a reference to something that's about to be destroyed:

rust · won't compilefn dangle() -> &String {
    let s = String::from("poof");
    &s   // ❌ s is dropped at the end of the function — this reference would dangle
}

The compiler tracks how long each reference must stay valid — these are lifetimes, and we'll meet them properly in the next chapter. For now the takeaway is simply: a borrow can never live longer than the thing it borrows. Use-after-free, structurally impossible.

🧭
Mental model

Ownership answers "who cleans this up?" Borrowing answers "who may look, and who may change?" Together they let you pass data around freely while the compiler quietly proves there are no double-frees, no dangling pointers, and no data races — before your program ever runs.

The thirty-second version

  • &value borrows (lends a read-only view); &mut value borrows mutably (one exclusive writer).
  • Many &T readers or one &mut T writer — never both at once.
  • This single rule kills data races and aliasing bugs at compile time, for free.
  • References may never outlive their referent — no dangling pointers.
  • Borrowing is how you use data without the friction of moving ownership everywhere.
