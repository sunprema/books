Chapter 01
The setup

You Type ./a.out — What Happens Next?

Here is a little program. Nothing fancy — it just says hello.

// hello.c
#include <stdio.h>

int main(void) {
    printf("Hello, kernel!\n");
    return 0;
}

You compile it — gcc -o hello hello.c — and you type ./hello at the shell. A blink later, Hello, kernel! appears on your screen. That's it. That's the whole experience, from the outside.

But why does that work? What actually happens between you pressing Enter and those characters appearing? This whole book is one long answer to that question, and I promise it's more interesting than it sounds — it's a story about disks, memory, and a surprisingly literal-minded kernel that does exactly what a tiny file tells it to, nothing more.

Think of it like ordering food at a restaurant. You (the shell) tell the waiter (also the shell, wearing a different hat) "I'll have the hello.c special." The waiter doesn't cook anything — it walks to the kitchen (the kernel) and hands over a ticket. Everything interesting happens back there, out of your sight, and what comes back to your table is the finished dish.

The shell's two-step move: fork, then exec

Your shell — bash, zsh, whatever — is just another ordinary program, running as an ordinary process. When you hit Enter on ./hello, the shell doesn't magically turn itself into your program. It does something cleverer: it makes a copy of itself with the fork() system call, and then that copy throws away its own insides and rebuilds itself as your program, using the exec family of system calls (in practice, execve()).

fork() is the "make a clone of me" step. exec() is the "now stop being me and become something else entirely" step. Same process ID, same open files, same terminal — but a completely different program running inside. We'll spend the whole next chapter on that second step, because it is where the real magic — execve — happens.

StepWho does itWhat changes
fork()the shella near-identical child process is born
execve()the childthe child's memory is entirely replaced by hello
wait()the shellthe parent waits for the child to finish and reports its exit code
Key idea The process (the "container": a PID, a memory address space, a set of open files) and the program (the actual code and data) are two different things. execve() keeps the container but swaps out everything inside it.

Where hello actually lives

Before the kernel can run hello, it has to know what hello is. Right now, on disk, hello is just a file — a sequence of bytes sitting in a directory, with an executable permission bit set. It is not "running" in any sense; nothing is running until the kernel decides to build a process around it.

And this is where our story really starts: how does the kernel look at a blob of bytes on disk and turn it into instructions, a stack, a heap, and a jump straight into your code? The file has to carry its own instructions for how to load itself — a format, a blueprint. On Linux (and most other Unix-like systems) that blueprint is called ELF: the Executable and Linkable Format. We'll meet it properly in Chapter 3.

Try this

Compile the program above and ask the shell what kind of file it produced:

$ gcc -O0 -g -o hello hello.c
$ file ./hello
./hello: ELF 64-bit LSB pie executable, x86-64, version 1 (SYSV),
dynamically linked, interpreter /lib64/ld-linux-x86-64.so.2, ...

That one line from file is basically the table of contents for this whole book: "ELF", "64-bit", "pie executable", "dynamically linked", "interpreter". We'll unpack every single word of it.

One more thing worth sitting with before we go further: nothing about this is specific to hello. Every program on your Linux machine — bash, ls, your web browser, the kernel's own loadable modules — is built from the same handful of ideas we're about to walk through. Learn this once, and you can point readelf at anything and read its story.
