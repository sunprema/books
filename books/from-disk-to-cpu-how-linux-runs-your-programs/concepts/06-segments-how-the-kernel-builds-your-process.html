Chapter 06
The loader's list

Segments: How the Kernel Builds Your Process

The program header table is a short list — usually under a dozen entries — of segments. Each entry is the kernel's construction instruction for one contiguous chunk of memory: "take these bytes from the file, put them at this virtual address, and mark the memory with these permissions."

typedef struct {
  uint32_t p_type;    // PT_LOAD, PT_INTERP, PT_DYNAMIC, ...
  uint32_t p_flags;   // PF_R=4, PF_W=2, PF_X=1 (bitmask)
  Elf64_Off p_offset;  // where in the FILE this segment starts
  Elf64_Addr p_vaddr; // where in MEMORY it should land
  Elf64_Addr p_paddr; // physical address (irrelevant on Linux)
  uint64_t p_filesz;  // bytes to copy FROM the file
  uint64_t p_memsz;   // bytes to reserve IN memory
  uint64_t p_align;   // required alignment
} Elf64_Phdr;

The segment types worth knowing

p_typeMeaning
PT_LOADA chunk to actually map into memory. A typical binary has 2–4 of these: one for code, one or two for data, sometimes one read-only.
PT_INTERP"Before you run me, go run this other program first" — holds the path to the dynamic linker, e.g. /lib64/ld-linux-x86-64.so.2.
PT_DYNAMICPoints to the dynamic linking info: which shared libraries are needed, where the symbol tables and relocations live.
PT_NOTEFree-form vendor/build metadata (build IDs, ABI tags).
PT_GNU_STACKTells the kernel whether the stack should be executable — modern toolchains always say no.
PT_GNU_RELRO"After you've finished relocations, make this region read-only" — a hardening trick, see Chapter 12.
PT_PHDRThe program header table itself, describing its own location — useful for the dynamic linker to find it.
Here's the physical analogy that made this click for me: imagine unpacking a flat-pack wardrobe. The file is the flat box — everything compressed and stacked efficiently to save shipping space. The program headers are the assembly instructions: "panel A goes here, screw it at this angle, use these four screws." PT_LOAD segments are the panels you actually place. And .bss — the space for uninitialized globals — is like the little bag of "extra shelf space, no parts needed" that the instructions tell you to leave empty. You don't ship empty space; you just leave room for it.

Why memsz can be bigger than filesz

This is the detail that trips people up the first time, and it's worth sitting with: for a data segment, p_memsz is often larger than p_filesz. The kernel copies p_filesz bytes from the file starting at p_offset, and then zero-fills the remaining p_memsz − p_filesz bytes. That trailing zero-filled region is exactly your program's .bss — every uninitialized global variable you declared. The file never had to store those zeros; the kernel just materializes them at load time, for free.

Watch out p_vaddr, p_offset, and p_align aren't independent — they must agree modulo the alignment (usually the page size, 4 KiB). That's why segment boundaries in a real binary look like oddly specific numbers, not round ones: the linker is satisfying this congruence so the kernel can map the file directly with mmap() rather than copying bytes around.

Try this

readelf -l ("l" for "program headers", historically called "the Elf program headers") prints exactly this table:

$ readelf -l ./hello
Program Headers:
  Type           Offset   VirtAddr   FileSiz  MemSiz   Flg
  PHDR           0x000040 0x000040   0x0002d8 0x0002d8 R
  INTERP         0x000318 0x000318   0x00001c 0x00001c R
      [Requesting program interpreter: /lib64/ld-linux-x86-64.so.2]
  LOAD           0x000000 0x000000   0x000638 0x000638 R
  LOAD           0x001000 0x001000   0x000191 0x000191 R E
  LOAD           0x002000 0x002000   0x0000f8 0x0000f8 R
  LOAD           0x002df0 0x003df0   0x000220 0x000228 RW
  DYNAMIC        0x002e00 0x003e00   0x0001e0 0x0001e0 RW
  ...

Find the last LOAD line and compare FileSiz to MemSiz — that small gap is your program's .bss.
