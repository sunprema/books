Hardening the handoff

Security and Modern Executables

Every mechanism in this book — fixed segments, a predictable entry point, a known dynamic linker — is also, if you squint, an attacker's map. If code always lands at the same address and the stack always sits in the same place, an attacker who can corrupt memory knows exactly where to aim. Modern Linux adds a handful of countermeasures directly into the ELF-and-exec pipeline we've been tracing.

Picture the old world: every house on the street has its front door in exactly the same spot, and the key is always under the same flowerpot. A burglar who's cased one house has cased them all. ASLR is the neighborhood deciding, on move-in day, to place the front door and the flowerpot in a different (but still functional) spot for every single house — the floor plan is identical, but you can't case the block from one visit.

ASLR + PIE: randomizing the floor plan

We've mentioned this twice already; here's the full picture. Address Space Layout Randomization (ASLR) is a kernel feature that randomizes the base addresses of the stack, heap, and memory-mapped libraries on every run. But randomizing the libraries alone leaves your own code sitting at a fixed, predictable address — unless your executable itself is built as position-independent.

That's exactly what e_type: ET_DYN on an executable means (Chapter 3, Chapter 5): a PIE (Position-Independent Executable) is structurally identical to a shared library — every internal reference is relative, so the kernel is free to load it at a random base address too. Modern gcc and clang build PIE by default; that's why readelf -h ./hello showed Type: DYN back in Chapter 5, not EXEC.

Try this

Run the same PIE binary twice and watch its own code move:

$ cat /proc/sys/kernel/randomize_va_space
2   # 2 = full ASLR (stack, heap, mmap base, AND PIE load base)
$ for i in 1 2; do ./hello & wait; grep -m1 r-xp /proc/$!/maps 2>/dev/null; done

(In practice, easier to see with cat /proc/self/maps printed from inside a small helper, or by comparing ldd $(which bash) output across two runs — the library base addresses shift each time.)

Three more locks on the same door

ProtectionWhere it lives in ELFWhat it stops
NX / DEPPT_GNU_STACK flags (Chapter 6)Marks the stack (and other data) non-executable, so injected shellcode on the stack can't just be jumped to and run.
RELROPT_GNU_RELRO (Chapter 6)After the dynamic linker finishes relocations, the GOT (Global Offset Table — the function-pointer table it just filled in) is remapped read-only, so a memory-corruption bug can't overwrite it to hijack a later call.
Stack canariesNot ELF structure — compiler-inserted, seeded from AT_RANDOM (Chapter 8)A random guard value before the return address; corrupted on overflow, checked before the function returns.
Key idea None of these protections change what a program does — they only change how hard it is to redirect what it does. That's why you can build the exact same hello.c with these on or off and get identical Hello, kernel! output either way; the difference only shows up when something goes wrong.

Try this

readelf and file will tell you which of these a given binary has enabled:

$ readelf -l ./hello | grep -A1 GNU_STACK
  GNU_STACK      0x000000 0x000000 0x000000 0x000000 RW    0x10   # no E flag = non-executable stack, good
$ readelf -d ./hello | grep -i flags
  0x000000006ffffffb (FLAGS_1)  Flags: NOW PIE  # NOW = full RELRO (BIND_NOW)

Stripped binaries: hiding the map, not the building

Recall from Chapter 4 that section headers — including the symbol table, .symtab — are for tools, not for execve(). strip deletes them, which is why a "stripped" binary still runs perfectly but gives a reverse engineer (or a crash-reporting tool) far less to work with: no human-readable function names, no debug info, just raw code and the program headers the kernel actually needs.

With that, we've covered the whole path a program takes from a file on disk to instructions in the CPU, plus the guardrails Linux puts around that path today. One last, lighter chapter: ELF shows up in a few places you might not expect.
