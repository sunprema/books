Chapter 02
The syscall

The Magic of execve

Here's the syscall that does the heavy lifting:

int execve(const char *pathname,
            char *const argv[],
            char *const envp[]);

Three arguments: which file to run, the argument list it should see as argv, and the environment variables it should see as envp. That's the whole interface. And yet the effect is total: on success, execve() does not return. Not "returns 0". Not "returns void". It does not return — because the process that called it no longer exists in any recognizable form. Its code, its heap, its stack, its memory mappings are gone, replaced wholesale by the new program.

Here's the physical picture I like: imagine you're driving a car down the highway, and — without stopping — you swap out the engine, the seats, the dashboard, and the entire body panel, keeping only the license plate and the fact that you're still "the car in lane two." The PID is the license plate. Everything else about the vehicle just got replaced, mid-motion, without you ever pulling over.

What survives, what doesn't

Because the process identity (PID) is preserved, a handful of things ride through execve() unchanged: the process ID and parent PID, open file descriptors (unless marked close-on-exec), the current working directory, and pending signals' disposition in some cases. But the address space — every byte of code, every global variable, every pointer you had — is thrown away and rebuilt from scratch out of the new file.

Survives execReplaced by exec
PID / PPIDEntire address space (code, data, heap)
Open file descriptors*The stack (rebuilt with new argv/envp)
Current working directoryRegisters, including the instruction pointer
Signal dispositions (mostly)Any memory-mapped libraries

*Except descriptors opened with O_CLOEXEC — those close automatically on exec, a deliberate safety valve so a program doesn't accidentally hand its file handles to whatever it execs next.

Two kinds of files execve understands

The kernel doesn't just blindly trust that pathname is runnable. It peeks at the first few bytes to decide what kind of file it's holding. There are, broadly, two answers it can find:

  • An ELF binary. The bytes start with the magic sequence 0x7f 'E' 'L' 'F'. The kernel's ELF loader takes over — this is the path our whole book follows.
  • A script with a shebang line. The bytes start with #!, like #!/bin/bash. The kernel reads the interpreter path off that first line and effectively re-execs that interpreter, handing it your script's path as an argument.

If the kernel doesn't recognize either shape — if the file isn't marked executable, or its first bytes are gibberish to every registered loader — execve() fails with ENOEXEC. That's the kernel's honest way of saying "I have no idea how to turn this into a running program."

Key idea A shebang script is really just a note to the kernel saying "don't run me directly — go run this other program instead, and pass it my path." The real execution always bottoms out in an ELF binary somewhere.

Where ELF's own instructions come in

Once the kernel recognizes the ELF magic bytes, it doesn't run the file the way you'd run a script — line by line. Instead, it reads a small structured header at the front of the file that tells it, precisely: how big is this file's header, where is the table describing which chunks of the file become which chunks of memory, and — critically — if this program needs help from another program (a dynamic linker) to finish loading, where is that program's path recorded?

That one detail — a field called PT_INTERP in the file, which we'll meet in Chapter 6 — is what lets a single system call, execve(), end up loading not one file but potentially two: your program, and the dynamic linker that finishes wiring it together. We'll trace that whole relay race in Chapter 9.

Try this

Watch execve fire, live, with strace:

$ strace -e trace=execve ./hello
execve("./hello", ["./hello"], 0x7ffc... /* 40 vars */) = 0
Hello, kernel!
+++ exited with 0 +++

One line, one syscall, and everything we're about to spend the rest of this book unpacking happens inside that single = 0.
