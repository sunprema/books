One shape, everywhere

Beyond Linux Executables

A short closing note, not a new deep chapter — just a nudge that everything you now understand about ELF keeps paying off in places you might not expect.

Core dumps are ELF too

When a process segfaults and the kernel writes out a core file, that file is — you guessed it — ELF, with e_type: ET_CORE (Chapter 3). Instead of program headers describing "load this from the file," a core dump's PT_LOAD segments describe "here is a literal snapshot of what was in memory the instant the process died." gdb ./hello core reads those segments to reconstruct the crashed process's exact stack and heap state — same format, opposite direction: instead of building a process from a file, you're rebuilding a file from a process.

Kernel modules: ELF that never gets execve'd

A .ko kernel module — the thing insmod/modprobe loads — is an ELF relocatable object, ET_REL, the same type as a plain .o file straight out of the compiler (Chapter 3). It's never handed to execve() at all; instead the kernel itself acts as the linker, resolving the module's symbol references against the running kernel's own symbol table and relocating it directly into kernel memory. Same container format, a completely different consumer.

Off Linux: Android, BSD, and beyond

ELF isn't a Linux-only invention — it's the standard executable format across most modern Unix-like systems, including the BSDs and Solaris. Android is a nice illustration of the ideas in this book showing up with different names: its C library, Bionic, ships its own dynamic linker (/system/bin/linker or linker64), found via the exact same PT_INTERP/DT_NEEDED mechanism we covered in Chapters 6–7 — and Android requires every executable to be a PIE (ET_DYN), enforcing the ASLR story from Chapter 12 as a hard rule rather than a default.

Key idea Whether it's a crash snapshot, a kernel module, or a phone app, the questions are always the same ones this book asked: what type of ELF object is this, which segments matter to whoever's loading it, and who resolves the symbols? Learn the shape once, and it keeps showing up.

Try this

Confirm a couple of these with tools you already know:

$ file /lib/modules/$(uname -r)/kernel/fs/fuse/fuse.ko
fuse.ko: ELF 64-bit LSB relocatable, x86-64, ...
$ file core
core: ELF 64-bit LSB core file, x86-64, ...

That's the tour. You started this book able to run ./hello and get an answer; now you can point readelf, strace, or a debugger at almost anything on a Linux box and read the same underlying story — a file describing itself precisely enough that a kernel, a linker, and a debugger can all agree on what it means. Keep the cheatsheet nearby; you'll reach for these commands more than you'd expect.
