Chapter 09
The relay race

First Instructions: From Kernel to Your main()

We now have everything staged: segments mapped, stack furnished with argv/envp/auxv. Time to actually press "go." This chapter is the relay race — four runners, one baton, each handing off at a precise point.

Wait — the kernel jumps into the linker's code first?

Yes, and this is the detail that surprises people: for a dynamically linked binary, the value the kernel actually treats as "the entry point" isn't your binary's e_entry at all — it's the interpreter's e_entry (ld.so's own ELF header has one, since it's an ELF file too). The kernel passes your binary's real entry point along inside the auxiliary vector as AT_ENTRY, so ld.so knows where to jump once it's done. Your code doesn't run first; it runs last, after everything it depends on has been wired up.

It's exactly like a relay race where the second runner doesn't just receive a baton — they're handed a folded note that says "after your leg, hand the baton to runner #4, who's waiting at the far corner of the track." The kernel is the race official who starts everyone off at the right spot; it doesn't run any leg itself.

What glibc does before you ever see it

By the time your main() gets control, glibc's __libc_start_main has already: parsed argc/argv/envp off the stack, set up the stack-protector canary (using that AT_RANDOM entropy from Chapter 8), initialized the heap allocator's bookkeeping, wired up stdio buffering for stdin/stdout/stderr, and run every function marked __attribute__((constructor)) (and every C++ global object's constructor) — all before a single line of your own main() executes. That's why a C++ global with a constructor "just works" without you calling anything: it ran during step 4 of the diagram above.

Key idea main() is not where your program "starts." It's the fourth or fifth stop on a relay that begins in the kernel. Everything before it exists to make main()'s simple contract — "here's argc, argv, envp; I'll call exit() with whatever you return" — actually true.

Try this

Set a breakpoint at _start instead of main and watch the handoff yourself:

$ gdb -q ./hello
(gdb) break _start
(gdb) run
Breakpoint 1, 0x0000000000001060 in _start ()
(gdb) bt
#0  0x...1060 in _start ()

Notice there's no caller on the backtrace above _start — as far as your process's own call stack is concerned, this really is the first frame. The kernel isn't a "caller" in the C sense; it just set the instruction pointer and walked away.
