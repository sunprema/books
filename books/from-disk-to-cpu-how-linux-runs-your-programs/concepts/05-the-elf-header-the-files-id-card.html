The ELF Header: The File's ID Card

Every ELF file opens with a fixed-size header — 64 bytes on a 64-bit system. Think of it as a passport: a strict, small, unchanging layout that tells any border agent (the kernel, a linker, a debugger) everything it needs to know before reading a single word further.

typedef struct {
  unsigned char e_ident[16];  // magic + class + encoding + ABI
  uint16_t e_type;       // ET_EXEC, ET_DYN, ET_REL, ET_CORE
  uint16_t e_machine;    // required CPU architecture
  uint32_t e_version;    // always 1, in practice
  Elf64_Addr e_entry;   // first instruction to run
  Elf64_Off e_phoff;    // program header table's file offset
  Elf64_Off e_shoff;    // section header table's file offset
  uint32_t e_flags;
  uint16_t e_ehsize;     // size of this header
  uint16_t e_phentsize, e_phnum;  // program header: entry size, count
  uint16_t e_shentsize, e_shnum;  // section header: entry size, count
  uint16_t e_shstrndx;  // which section holds section *names*
} Elf64_Ehdr;

e_ident: the first 16 bytes

The very first field is itself a small bundle of sub-fields, each answering one yes/no-shaped question before the kernel commits to anything else:

BytesFieldWhat it says
0–3magic0x7f 'E' 'L' 'F' — "yes, I am an ELF file"
4EI_CLASS1 = 32-bit, 2 = 64-bit
5EI_DATA1 = little-endian, 2 = big-endian
6EI_VERSIONELF version, always 1 today
7EI_OSABItarget OS ABI — usually 0 (System V) on Linux
8–15paddingreserved, currently zero-filled
This is exactly why the kernel can refuse, instantly, to run an ARM binary you scp'd onto your x86 laptop, or a 32-bit binary in a way that's silently wrong. It doesn't have to disassemble a single instruction — it just checks the ID card at the door. Wrong class, wrong endianness, wrong machine: rejected before anything else even loads.

The three fields that matter most, day to day

Of the whole header, three fields do almost all the work you'll personally care about:

  • e_entry — the virtual address of the very first instruction the CPU will execute. For a non-PIE executable this is a fixed address like 0x401020; for a PIE (position-independent) executable it's an offset, since the kernel will pick the real base address at load time (more on that in Chapter 12).
  • e_phoff / e_phnum — where the program header table starts, and how many entries it has. This is the table the kernel actually reads to build your process — the one we open properly in the next chapter.
  • e_type — which of the four flavors from Chapter 3 this file is: ET_REL, ET_EXEC, ET_DYN, or ET_CORE.
Key idea Everything downstream — where the code is, where the data is, whether a dynamic linker is needed — is reachable only because the header tells you where to look next. It's not the payload; it's the table of contents to the table of contents.

Try this

readelf -h prints the ELF header in human-readable form — this is usually the very first command a reverse engineer runs on an unfamiliar binary:

$ readelf -h ./hello
ELF Header:
  Magic:   7f 45 4c 46 02 01 01 00 00 00 00 00 00 00 00 00
  Class:                             ELF64
  Data:                              2's complement, little endian
  Type:                              DYN (Position-Independent Executable file)
  Machine:                           Advanced Micro Devices X86-64
  Entry point address:               0x1060
  Start of program headers:          64 (bytes into file)
  Start of section headers:          14264 (bytes into file)
  ...

Notice Type: DYN, not EXEC — on a modern distro, gcc builds position-independent executables by default. We'll come back to exactly what that means for e_entry in Chapter 12.

Now that we know how to find the program header table, let's go read what it actually says — because that table is the kernel's real construction manual.
