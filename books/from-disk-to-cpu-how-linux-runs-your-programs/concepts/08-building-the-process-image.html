Chapter 08
The address space

Building the Process Image

By now the kernel has mapped your PT_LOAD segments and, if needed, mapped the dynamic linker alongside them. But a process is more than mapped code and data — it needs somewhere to keep its call stack, its argv, its environment, and a small dossier of facts the C library needs before main() can even start.

Picture the whole 64-bit address space as an enormous, mostly-empty apartment building — trillions of possible addresses, almost all of them never touched. The kernel doesn't furnish the whole building; it just moves a few pieces of furniture into specific rooms: your code and data near the bottom floors, shared libraries and the dynamic linker somewhere in the middle floors, and the stack way up near the top, growing downward like a chandelier being lowered from the ceiling.

The auxiliary vector: a cheat sheet the kernel leaves behind

Right after the envp array (and its terminating NULL), the kernel writes one more structure onto the initial stack: the auxiliary vector, or auxv. It's a list of key/value pairs answering questions the C runtime would otherwise have to rediscover the hard way — things like "where is the program's own ELF header in memory" (AT_PHDR), "what's the entry point" (AT_ENTRY), "here are 16 bytes of real randomness for stack-protector cookies" (AT_RANDOM), and "what page size and CPU features does this machine have" (AT_PAGESZ, AT_HWCAP).

Key idea The kernel doesn't hand your program a "process object" or a struct — it hands it exactly one thing: an initial stack, pre-loaded with argc/argv/envp/auxv, and a starting instruction pointer. Everything C's runtime does before main() is just parsing that stack.

Try this

The tiny helper program LD_SHOW_AUXV=1 (read by ld.so itself) dumps the auxiliary vector before your program even starts:

$ LD_SHOW_AUXV=1 ./hello
AT_PHDR:       0x5590...1040
AT_PHENT:      56
AT_PHNUM:      13
AT_PAGESZ:     4096
AT_ENTRY:      0x5590...1060
AT_RANDOM:     0x7ffd...
AT_HWCAP:      0x...
Hello, kernel!

Note AT_ENTRY matches the e_entry value from your ELF header (Chapter 5) — the kernel is literally forwarding facts it already computed while parsing the file.

With the address space furnished and the stack loaded, everything is now in place for the CPU to actually start executing instructions. Where does it jump first — your code, or the dynamic linker's? That's Chapter 9.
