Chapter 04
The blueprint

The ELF "Anatomy" at a Glance

Before we crack open individual fields, let's look at the whole file the way an engineer looks at a cross-section drawing: one glance, all the major parts labeled, nothing explained in depth yet.

Two tables of contents, two audiences

The single most important thing to understand about ELF's structure is that it carries two independent indexes into the same bytes, aimed at two different readers:

  • The program header table — a short list of segments. This is read by the kernel at exec time. It answers one question: "which byte ranges of this file get mapped into memory, at what addresses, with what permissions?" A running process only cares about this table — it's the "execution view."
  • The section header table — a longer, finer-grained list of sections like .text, .rodata, .symtab, .debug_info. This is read by the linker when it's stitching object files together, and by tools like objdump, gdb, and nm when they want to find, say, "the symbol table" by name. This is the "linking view."
I like to think of it as one warehouse with two different maps taped to the wall. The forklift driver (the kernel) only needs the coarse map: "pallets A through D go on truck 1, truck 1 drives to this loading dock." The inventory clerk (the linker, your debugger) needs the fine-grained map: "item #4471 is in pallet B, third box from the left." Same warehouse, same boxes — different level of detail for different jobs.
Key idea A running process never looks at section headers. You can strip .symtab, .debug_info, and the entire section header table out of a binary with strip, and it will run exactly the same — because the kernel only ever consulted the program headers. Section headers are for humans and tools, not for execve().

Notice one more thing in the diagram: .bss — the space reserved for uninitialized global variables — has a size recorded but occupies zero bytes in the file. Why ship zeros on disk when the kernel can just hand you a block of already-zeroed memory at load time? That single optimization is a preview of the more general trick we'll see in Chapter 6: a segment's size in memory and its size in the file are allowed to differ.

Over the next two chapters we'll open each of these tables properly: first the ELF header itself (the file's "ID card," which tells you where to find both tables), then the program headers (the kernel's actual loading instructions).
