The other program

The Dynamic Linker and Shared Libraries

Back in Chapter 6 we saw a PT_INTERP segment carrying a path: /lib64/ld-linux-x86-64.so.2. That's not a footnote — it's an instruction to the kernel: "don't just run me — run that program first, and let it finish preparing me."

Here's the analogy: your hello binary is like a piece of flat-pack furniture that arrives missing a few screws on purpose — because the manufacturer expects your house (this particular machine, this particular Linux distribution) to already stock those screws in a shared drawer, and re-shipping a fresh set with every single box would be wasteful. The dynamic linker is the assistant who walks over to that shared drawer (/lib/x86_64-linux-gnu/ and friends), finds the exact screws your furniture calls for, and bolts them on before handing you the finished piece.

What ld-linux.so actually does

ld.so / ld-linux.so — the dynamic linker/loader — runs as a full program in its own right, mapped into your process's address space right alongside your code. Its job, roughly in order:

  1. Read your binary's PT_DYNAMIC segment to find the list of shared libraries it depends on (things like libc.so.6).
  2. mmap() each of those libraries into memory — they're ELF files too, with their own PT_LOAD segments.
  3. Resolve symbols — patch up every call to printf, every reference to a library global, so it points at the real address where that function or variable actually landed this time (which changes from run to run, especially under ASLR).
  4. Apply any final protections (like making relocated tables read-only — PT_GNU_RELRO, Chapter 12).
  5. Jump to your binary's real entry point, e_entry — only now does your code get to run.
Key idea Symbol resolution is why linking is "dynamic": the exact memory address of printf isn't baked into your binary at compile time. It's discovered fresh, on this machine, on this run, by the dynamic linker — which is precisely what lets you upgrade libc with a security patch without recompiling every program on your system.

Static vs. dynamic: the same program, two ways

A statically linked binary skips all of this — every library function it calls is copied straight into the executable at link time. No PT_INTERP, no relay race, no dynamic linker. The tradeoff is size and update story: a static binary is bigger and self-sufficient (great for containers and rescue disks); a dynamic binary is smaller and shares one copy of libc across every process on the machine (great for memory usage and patching).

Try this

Build hello.c both ways and compare:

$ gcc -O0 -o hello-dyn hello.c
$ gcc -O0 -static -o hello-static hello.c
$ ls -la hello-dyn hello-static
-rwxr-xr-x 1 you you   16656 hello-dyn
-rwxr-xr-x 1 you you  873728 hello-static
$ ldd hello-dyn
	linux-vdso.so.1
	libc.so.6 => /lib/x86_64-linux-gnu/libc.so.6
	/lib64/ld-linux-x86-64.so.2
$ ldd hello-static
	not a dynamic executable

The static binary is roughly 50× larger for the same 4 lines of C — that's the entire relevant slice of libc baked in. linux-vdso.so.1 in the dynamic listing is a special case: the kernel maps it in for free, with no file on disk at all, to make certain syscalls (like getting the time) faster.

Two knobs worth knowing

LD_LIBRARY_PATH adds extra directories the dynamic linker should search for shared libraries — handy for testing a library build without installing it system-wide. LD_PRELOAD is more powerful (and more dangerous): it forces specific shared objects to be loaded before anything else, letting their symbols override the normal ones — the classic trick behind tools that intercept malloc() for memory debugging.

$ LD_LIBRARY_PATH=./mylibs ./hello
$ LD_PRELOAD=./my_malloc_logger.so ./hello

Both are read directly by ld.so itself, before your program's main() has run a single line — which is exactly why they're such a common lever for both legitimate debugging tools and, historically, a few security tricks. (Notably: ld.so ignores both in "secure-execution mode" — e.g. for setuid binaries — precisely to close that door.)
