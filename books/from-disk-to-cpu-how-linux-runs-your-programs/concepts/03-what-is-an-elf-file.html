Chapter 03
The format

What Is an ELF File?

ELF stands for Executable and Linkable Format. It's not a Linux invention exactly — it came out of Unix System V and got adopted across nearly every modern Unix-like system — but on Linux it is the format for anything that touches a compiler or a running process.

Think of ELF less like "an executable" and more like a standardized shipping container. A shipping container doesn't care whether it holds bananas or bicycles — it just has a fixed shape, standard latch points, and a manifest stapled to the door telling the crane exactly where to grab it and where each pallet inside goes. ELF is that container. The "cargo" can be machine code, string constants, or debugging notes; ELF's job is just to describe, precisely, how the pieces are arranged so any tool — the kernel, a linker, a debugger — can find what it needs without guessing.

One format, four jobs

The same container shape gets reused for four different kinds of cargo, distinguished by a single field near the top of the file (e_type, which we'll meet properly next chapter):

e_typeWhat it isExample
ET_RELRelocatable object — not yet linked, positions still "to be determined"hello.o, a kernel module .ko
ET_EXECA finished executable with fixed load addressesan old-style static binary
ET_DYNA shared object or a position-independent executablelibc.so, most modern hello binaries (PIE)
ET_COREA snapshot of a crashed process's memorya core dump after a segfault

That's a genuinely elegant piece of design: the compiler's raw output (.o), the linker's finished product (an executable or a .so), and even the wreckage the kernel writes out after your program crashes — all three speak the same file format. Learn to read one ELF file, and readelf works identically on all of them.

Why not just dump raw machine code?

You might reasonably ask: the CPU just executes bytes — why not make an executable just be raw instruction bytes, no header, no wrapper? Early systems actually did something close to that (the ancient a.out format, whose name is why we still call compiler output a.out by default today). But a bare instruction stream can't answer questions the loader desperately needs answered before it dares run anything:

  • Which byte is the very first instruction to execute (the entry point)?
  • Which parts of the file are code (must not be writable — and ideally shouldn't be, for safety) versus initialized data (readable and writable) versus space that should just be zero-filled at runtime (uninitialized globals)?
  • Does this program need another program's help — a dynamic linker — to finish assembling itself before it can run?
  • What CPU architecture and byte order was this compiled for, so the kernel can refuse to run an ARM binary on an x86 machine instead of jumping into garbage?

ELF's entire reason for existing is to answer those questions in a fixed, predictable place near the front of the file, so the kernel's loader (and tools like readelf, objdump, and your debugger) can find the answers without parsing the whole file or guessing.

Key idea ELF is a metadata format wrapped around machine code and data. The magic bytes 0x7f 'E' 'L' 'F' at byte 0 are the kernel's first, cheapest check: "is this even a container I know how to open?"

Try this

Look at the raw magic bytes yourself with a hex dump — the first four bytes spell out the format, literally:

$ xxd -l 20 ./hello
00000000: 7f45 4c46 0201 0100 0000 0000 0000 0000  .ELF............
00000010: 0300 3e00                                ..>.

7f 45 4c 46 is 0x7f followed by the ASCII characters E, L, F. The kernel reads exactly these four bytes before deciding to hand the file to its ELF loader.
