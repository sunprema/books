Watching it happen

Tracing Execution with strace

Static tools show you the blueprint. strace shows you the building actually going up, syscall by syscall, in real time. It works by attaching to your process with ptrace() and pausing it at every syscall boundary, printing the call and its result before letting execution continue.

If readelf is reading the architect's drawings, strace is standing on the construction site with a clipboard, writing down every single instruction the foreman shouts to the crew: "open this door," "map this wall," "now go tell the neighbors' handyman (the kernel) to load the plumbing library." Nothing is hidden — you see the entire conversation between your program and the kernel.

Try this — just the exec

$ strace -e trace=execve ./hello
execve("./hello", ["./hello"], 0x7ffc... /* 41 vars */) = 0
Hello, kernel!
+++ exited with 0 +++

One line — the exact moment we studied in Chapter 2. Filtering to just execve is the fastest way to see when one program launches another, e.g. inside a shell script or a build system.

Try this — the whole startup sequence

Drop the filter and you'll see the dynamic linker's own work, live — every mmap, every library openat:

$ strace ./hello
execve("./hello", ["./hello"], 0x7ffc... ) = 0
brk(NULL)                              = 0x55a1...
openat(AT_FDCWD, "/lib/x86_64-linux-gnu/libc.so.6", O_RDONLY|O_CLOEXEC) = 3
read(3, "\177ELF\2\1\1\3...", 832)     = 832
mmap(NULL, 8365624, PROT_NONE, ...)    = 0x7f2c...
mmap(0x7f2c..., 1735672, PROT_READ|PROT_EXEC, ...) = 0x7f2c...
mmap(0x7f2c..., 360448, PROT_READ|PROT_WRITE, ...) = 0x7f2c...
close(3)                               = 0
...
write(1, "Hello, kernel!\n", 16)       = 16
exit_group(0)                          = ?

That block of openat + read + several mmap calls on libc.so.6 is ld.so doing exactly the work described in Chapter 7 and 9 — mapping the library's own PT_LOAD segments in, one PROT_READ|PROT_EXEC region and one PROT_READ|PROT_WRITE region, just like your own binary got mapped by the kernel a moment earlier.

Try this — statically linked, for contrast

Trace the static build from Chapter 7 and count how much of that startup noise disappears:

$ strace -c ./hello-static
% time     seconds  usecs/call     calls    syscall
------ ----------- ----------- --------- ----------------
 45.20    0.000123         123         1 execve
 30.10    0.000082          41         2 mmap
 12.40    0.000034          34         1 write
 ...
------ ----------- ----------- --------- ----------------
100.00    0.000273                    12 total

No openat for libc.so.6, far fewer mmap calls — the dynamic-linking relay race from Chapter 9 simply never runs, because everything it would have fetched is already baked into the file.

A few flags worth keeping in your fingers

FlagWhat it does
-e trace=SETOnly show syscalls in SET, e.g. execve, open,openat, or a category like %file
-fFollow forked child processes too — essential for shell scripts and multi-process programs
-cSkip the play-by-play; print a summary table of time and call counts per syscall
-o fileWrite the trace to a file instead of stderr (so it doesn't interleave with the program's own output)
-TShow time spent inside each syscall — useful for spotting a slow read() or connect()
Watch out strace works by pausing your process at every syscall via ptrace(), which is real overhead — a syscall-heavy program can run noticeably slower under trace. It's a diagnostic tool, not something you leave attached to production traffic.

We've now walked the entire path — from the shell's fork()+exec(), through the ELF file's own blueprint, to the kernel-to-linker-to-main() relay, and finally watched it all happen live with strace. One topic left: what does the system do to make sure none of this can be hijacked by an attacker?
