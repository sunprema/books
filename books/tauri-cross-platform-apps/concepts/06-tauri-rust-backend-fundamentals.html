Chapter 06

The Other Side of the Seam

Rust Backend Fundamentals

This is not a Rust tutorial. It's a tour of the small, specific shape Rust code takes when it lives inside src-tauri/: a command is a plain Rust function with a JSON-shaped doorway, and everything else — state, async, errors — exists to keep that doorway honest. Know this shape and you can read, review, and direct an agent through Rust-side code without knowing Rust deeply yourself.

A command is a function with a JSON-shaped doorway

Mark any Rust function #[tauri::command] and it becomes callable from the frontend by name (Chapter 5's invoke()). That's the whole contract: every argument and the return value must implement serde::Serialize / Deserialize, because they cross the Rust↔JS boundary as JSON. Inside the function body it's ordinary Rust — read a file, query a database, call another crate — none of that is Tauri-specific. 

#[tauri::command]
fn greet(name: String) -> String {
    format!("Hello, {}!", name)
}

A command doesn't run itself — it has to be listed in tauri::generate_handler![…], which Core wires into its dispatch table when the app builds. Forget to list a new command there and invoke() fails on the frontend with "command not found," even though the function compiles fine — a common first surprise, and a common thing an agent forgets too. 

tauri::Builder::default()
    .invoke_handler(tauri::generate_handler![greet, get_file_metadata])
    .run(tauri::generate_context!())
    .expect("error while running tauri application");

Ask your agent

Add a Rust command list_recent_files(dir: String, limit: usize) -> Result<Vec<FileEntry>, AppError> in src-tauri/src that reads the directory and returns up to `limit` entries sorted by modified time, newest first. Register it in generate_handler! and tell me the exact invoke() call the frontend should use, including the argument object shape.

What actually belongs on the Rust side

The dividing line isn't "logic vs. UI" — plenty of ordinary app logic is fine in JS. The dividing line is what the OS or the outside world will let you touch. The webview sandbox can't open an arbitrary file, spawn a process, or hold a raw database connection; Rust, running as a full native process, can. So Rust-side responsibility clusters around four things: filesystem and OS APIs beyond what a plugin already wraps, talking to a database or an external service, anything that must stay secret from the webview (API keys, credentials, license checks), and CPU-heavy work that would jank a UI thread if it ran in JS.

Belongs in RustWhy
Reading/writing files outside what a plugin covers, walking directory trees The webview has no raw filesystem access at all — it only ever gets what a command hands it
SQLite/Postgres connections, ORMs, migrations A DB driver and its connection pool have no business shipping to the webview's JS bundle
API keys, secrets, license/entitlement checks Anything in the webview's JS is readable via devtools; Rust memory isn't inspectable the same way
Heavy computation: parsing, image/video processing, crypto Runs off the UI thread and off the JS single thread, without the browser's usual limits
Where this shows up A local-first notes app keeps its SQLite file and every query behind Rust commands like save_note and search_notes — the frontend never sees a connection string or a SQL string, only typed function calls that happen to be async and cross a process boundary.

Sync by default; async when you'd otherwise block

A plain fn command runs on Tauri's main thread pool and should return fast — anything slow there stalls other commands queued behind it. Mark a command async fn and Tauri instead runs it on its async runtime (Tokio under the hood), so it can .await I/O — a file read, an HTTP request, a DB query — without blocking anything else. The rule of thumb: reach for async the moment a command does I/O or anything that isn't near-instant; for genuinely CPU-bound work (image processing, heavy parsing) inside an async command, hand it to a blocking thread pool (e.g. tokio::task::spawn_blocking) instead of awaiting it inline, since blocking the async runtime's own threads starves every other async command too. 

#[tauri::command]
async fn get_file_metadata(path: String) -> Result<FileMetadata, AppError> {
    let meta = tokio::fs::metadata(&path).await?;
    Ok(FileMetadata {
        size: meta.len(),
        is_dir: meta.is_dir(),
    })
}
Gotcha A sync command that does slow, blocking work (a large synchronous file read, a tight loop) still ties up Tauri's shared thread pool, delaying unrelated commands the frontend fires around the same time. If it's not instant, either make it async and .await async I/O, or explicitly spawn it onto a background thread and report back via an event (Chapter 7) instead of making the frontend wait on the command's return.

Error handling across the boundary

A command's return type is almost always Result<T, E>: Ok(T) resolves the frontend's promise, Err(E) rejects it. The catch is that everything crossing the boundary must implement serde::Serialize — including E — and most library error types (std::io::Error, a database driver's error enum) don't. The fix is a small custom error type for your app that does derive Serialize, with From impls that convert the library errors you actually hit; ? then just works inside a command body. 

#[derive(Debug, thiserror::Error, serde::Serialize)]
enum AppError {
    #[error("io error: {0}")]
    Io(String),
    #[error("not found: {0}")]
    NotFound(String),
}

impl From<std::io::Error> for AppError {
    fn from(err: std::io::Error) -> Self {
        AppError::Io(err.to_string())
    }
}

On the frontend, a rejected invoke() promise carries whatever shape E serialized to — here, a tagged object like { "Io": "no such file or directory" } — so design the error enum with the frontend's catch block in mind, not just as an internal Rust detail.

Ask your agent

Create a shared AppError enum in src-tauri/src/error.rs that derives serde::Serialize and thiserror::Error, with variants for Io, NotFound, and Database. Add From impls so `?` works for std::io::Error and our sqlx error type. Update every existing #[tauri::command] that currently returns Result<T, String> to return Result<T, AppError> instead.

Shared state: one value, many commands

Most non-trivial apps need something that outlives any single command call — a database connection pool, an in-memory cache, config loaded once at startup. Tauri calls this managed state: you hand a value to .manage(…) once while building the app, and any command can ask for it back by adding a tauri::State<T> parameter — Tauri injects it, matched by type. Because commands can run concurrently, state that needs mutation is almost always wrapped in a Mutex (or RwLock) so exactly one command locks and mutates it at a time. 

struct AppState {
    db: Mutex<Connection>,
}

#[tauri::command]
fn save_note(note: String, state: tauri::State<AppState>) -> Result<(), AppError> {
    let db = state.db.lock().unwrap();
    db.execute("INSERT INTO notes (body) VALUES (?1)", [note])?;
    Ok(())
}

// wiring it up once, at startup:
tauri::Builder::default()
    .manage(AppState { db: Mutex::new(conn) })
    .invoke_handler(tauri::generate_handler![save_note])
    .run(tauri::generate_context!())
    .expect("error while running tauri application");
Gotcha State<T> is matched by exact type, resolved at runtime, not compile time — ask for State<AppState> when you actually managed Mutex<AppState> (or vice versa) and the command doesn't fail to compile, it panics when called because nothing matches that exact type. If a command panics with "state not managed for type X," check the type match first.

For work that needs to reach the app itself — create a window, emit an event, fetch managed state from outside a command (a background task spawned earlier) — a command can also take an tauri::AppHandle parameter (Chapter 3), which is cheap to clone and safe to move into a spawned task.

Ask your agent

Add a managed AppState struct holding a Mutex<HashMap<String, String>> as an in-memory settings cache. Wire it up with .manage() in main.rs, and add two commands, get_setting(key) and set_setting(key, value), that read and write through the mutex. Keep the lock held for the shortest scope possible in each command.

Quick glossary

#[tauri::command]

Marks a Rust function callable from the frontend by name; args/return must be Serialize/Deserialize.

generate_handler![…]

The registration list — a command not listed here can't be invoked, even if it compiles.

async fn command

Runs on the async runtime so it can .await I/O without blocking other commands.

Result<T, E>

Ok resolves the invoke() promise; Err rejects it — E must implement Serialize too.

tauri::State<T>

Injects a value previously handed to .manage(<T>); matched by exact type, at runtime.

AppHandle

A cheap, cloneable handle reaching the running app from anywhere — windows, state, events.

Key idea Rust-side code in Tauri is ordinary Rust wearing one contract at its edges: a command's inputs and outputs must serialize to JSON, its errors must serialize too, and anything shared across calls goes through managed state behind a lock. Learn that contract and you can direct an agent through filesystem access, database work, or CPU-heavy tasks with the same precision you already have for the frontend side.

The takeaway

You now have both halves of the seam: invoke()/listen() on the frontend (Chapter 5), and #[tauri::command], Result<T, E>, and managed state on the Rust side. Chapter 7 puts them together as the two communication patterns — commands and events — and shows how they chain into the "trigger, work, report progress" shape almost every real feature ends up taking.
