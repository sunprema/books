Chapter 09

Two Ways Core Grows Without You Rewriting the World in Rust

Plugins and Sidecars

Chapter 3 gave you the two words and a one-line test: reimplementing an existing tool as a plugin, or reaching for a sidecar when a plugin already does the job, both waste real effort. This chapter is the deep version — how to reach for an official plugin first, when writing your own actually pays off, how a sidecar binary actually gets bundled and spawned, and why both still answer to the capability system from Chapter 8.

The test, restated

A plugin runs inside Core — a Rust crate (plus an optional JS package) that registers its own commands, events, and state into your app's own process at startup. A sidecar runs beside Core — a separately compiled binary that Core spawns as its own OS process and talks to over stdio, the way a shell script shells out to another program. Same question every time a feature needs something outside the webview: does Tauri (or the plugin ecosystem) already have Rust code for this, or does a working, already-written program for this already exist on disk?

Reach for an official plugin first

Before writing anything, check whether the Tauri team already shipped it. The official plugins-workspace repo covers most of what an app needs to touch the OS: dialogs, filesystem access, HTTP, notifications, shelling out, autostart, a persistent key-value store, in-app updates, deep links, clipboard read/write, global shortcuts, SQL databases, and window-state persistence, among others. Each is a normal tauri-plugin-* crate plus (usually) a small @tauri-apps/plugin-* npm package, wired in with one line in Rust and one import in JS — and governed by the exact same permission/capability system as your own commands, so adopting one doesn't open a side door around Chapter 8's model.

// src-tauri/src/lib.rs
pub fn run() {
    tauri::Builder::default()
        .plugin(tauri_plugin_store::Builder::default().build())
        .plugin(tauri_plugin_window_state::Builder::default().build())
        .invoke_handler(tauri::generate_handler![your_own_commands])
        .run(tauri::generate_context!())
        .expect("error while running tauri application");
}

Each registered plugin still needs its permissions granted in a capability file before any window can call it — registering tauri_plugin_store on the Rust side makes the commands exist, but a window with no capability naming store:default still gets nothing, exactly like Chapter 8's default-deny rule for your own commands.

Ask your agent

Add the official window-state plugin so the main window remembers its size and position between launches. Register it in src-tauri/src/lib.rs, add the npm package if one exists for this plugin, and grant only the "main" window's capability the permission it needs — don't add it to the "settings" window, which doesn't need persisted geometry.

In practice A note-taking app persists the sidebar width and last-opened note via store, restores window geometry via window-state, and registers a system-wide "quick capture" hotkey via global-shortcut — three cross-platform native behaviors, zero custom Rust, zero bundled binaries.

Writing your own plugin, when nothing official fits

Sometimes the feature is genuinely native and genuinely yours — a hardware integration, a proprietary SDK with a Rust or C binding, a piece of logic you want packaged once and reused across several apps. cargo tauri plugin new <name> scaffolds a real plugin project: a tauri-plugin-<name> crate, an optional tauri-plugin-<name>-api npm package (skip it with --no-api if the plugin is Rust-only), and Android/iOS project stubs if you pass --android / --ios. Inside, it looks exactly like your app's own backend — commands decorated with #[tauri::command], an init() that builds a tauri::plugin::Builder, its own permissions/ directory defining what a capability can grant. It's the same commands-and-permissions shape as Chapters 6 through 8, just packaged to be registered into a different app later instead of living in src-tauri/src/ directly.

Reach for this when the capability needs to live in Rust itself.

  • No existing compiled tool does the job — you're writing the logic from scratch anyway
  • The feature needs tight, low-latency access to Tauri's own state (windows, app handle, other commands)
  • You want it reusable across several apps, or shared with the mobile targets
  • Cost: real Rust code to write and maintain, same as any command in Chapter 6

Reach for this when a working program already solves it.

  • An existing CLI tool, model runtime, or script already does exactly this, in another language
  • Rewriting it in Rust would take weeks and add risk for no real benefit
  • It's fine to run as a separate, supervised OS process talking over stdio
  • Cost: one binary per target platform, plus explicit spawn permission — see below

Ask your agent

Scaffold a new Tauri plugin called barcode-reader with cargo tauri plugin new, including the npm API package. Give it one command, scan_from_bytes, that takes image bytes and returns a decoded string or an error — implement the decoding with an existing Rust barcode crate, not a bundled external binary. Define its permission as scan-from-bytes:allow so a consuming app's capability has to grant it explicitly.

Bundling and spawning a sidecar

A sidecar starts as a binary you already have — built any way you like, in any language. List it under bundle.externalBin in tauri.conf.json, and the bundler packages it alongside your app for every platform target you build for. The one wrinkle: a single binary can't run on every OS/architecture, so Tauri expects one file per target, each named with its Rust target triple appended — find your own host triple by running rustc -vV and reading the host: line. 

// src-tauri/tauri.conf.json
{
  "bundle": {
    "externalBin": ["binaries/whisper-cli"]
  }
}
// src-tauri/binaries/ must then contain, per platform you ship:
//   whisper-cli-x86_64-pc-windows-msvc.exe
//   whisper-cli-x86_64-apple-darwin
//   whisper-cli-aarch64-apple-darwin
//   whisper-cli-x86_64-unknown-linux-gnu

Spawning goes through the shell plugin — a sidecar is just a process, and the shell plugin already owns "run a process" as a permissioned capability. From Rust, call .sidecar() with the bare name you configured (not the platform-suffixed filename — Tauri resolves that for you at runtime); from JS, the equivalent Command.sidecar() call takes the same string. 

// Rust: spawn it, stream stdout lines back as events
use tauri_plugin_shell::ShellExt;

#[tauri::command]
async fn transcribe(app: tauri::AppHandle, audio_path: String) -> Result<(), String> {
    let (mut rx, _child) = app
        .shell()
        .sidecar("whisper-cli").map_err(|e| e.to_string())?
        .args(["--file", &audio_path])
        .spawn().map_err(|e| e.to_string())?;

    while let Some(event) = rx.recv().await {
        if let tauri_plugin_shell::process::CommandEvent::Stdout(line) = event {
            app.emit("transcribe-progress", String::from_utf8_lossy(&line).to_string())
                .ok();
        }
    }
    Ok(())
}
Gotcha A build that only produces the host machine's own target triple looks correct in dev and passes CI on one runner, then ships a Windows or Linux build with no matching binary at all — .sidecar() fails at runtime on exactly the platforms nobody tested locally. Building (or cross-compiling, or fetching a prebuilt release) one binary per shipped target is part of the release checklist, not an afterthought; Chapter 10 covers building per-platform in the CI pipeline.

Sidecars still answer to the capability system

Spawning a process is exactly the kind of power Chapter 8 default-denies. A window can call .sidecar() at all only if its capability grants shell:allow-execute (or a narrower permission naming just that command), and the permission's argument validator decides how much freedom the caller actually has: true allows any arguments through, false allows none, and an explicit array pins the exact arguments (or an exact shape of them) the frontend is allowed to pass — the same allow/scope pairing from Chapter 8, applied to "which process can this window start, with what." 

// src-tauri/capabilities/main.json
{
  "identifier": "main-capability",
  "windows": ["main"],
  "permissions": [
    {
      "identifier": "shell:allow-execute",
      "allow": [{ "name": "whisper-cli", "sidecar": true, "args": ["--file", true] }]
    }
  ]
}
In practice A local-first transcription app bundles a compiled whisper.cpp binary instead of binding it over FFI or reimplementing model inference in Rust; a video tool bundles ffmpeg for transcoding rather than writing a codec pipeline. Both grant shell:allow-execute scoped to that one binary name with a fixed argument shape — never a general "run this string as a command" permission, which is exactly the shell-injection shape Chapter 8 warned about.

Ask your agent

We're bundling the existing ffmpeg-cli binary as a sidecar named "ffmpeg" to transcode a user-picked video to mp4. Add it to bundle.externalBin, write a Rust command transcode_video(input_path, output_path) that spawns it with a fixed, hardcoded argument template (only the two paths are variable), and add a capability granting shell:allow-execute scoped to the "ffmpeg" sidecar only, with the args validator matching that exact template — not "true" for arbitrary arguments.

Deciding what a new capability needs

You're about to…Reach for
Add a common OS integration (dialogs, notifications, storage, shortcuts) An official plugin from plugins-workspace, permission scoped as narrowly as it allows
Add native logic that's genuinely yours, or reusable across apps cargo tauri plugin new, same commands/permissions shape as your own backend
Reuse an existing compiled tool or heavy runtime you won't rewrite in Rust A sidecar via externalBin, spawned through the shell plugin
Ship that sidecar to more than one OS/architecture One binary per Rust target triple, named and placed exactly as Tauri expects
Let a window spawn a sidecar at all shell:allow-execute scoped to that sidecar's name, args pinned — not true

Quick glossary

Plugin

A Rust crate (+ optional JS package) registered into Core's own process; runs in-process, no OS boundary crossed.

Sidecar

A separately compiled binary Core spawns as its own OS process and talks to over stdio.

externalBin

The tauri.conf.json bundle field listing sidecar binaries to package with the app.

Target triple

The platform-architecture suffix (e.g. x86_64-pc-windows-msvc) a sidecar binary's filename must carry per platform.

cargo tauri plugin new

Scaffolds a new plugin crate (+ optional npm package, optional mobile stubs) with the standard shape.

shell:allow-execute

The permission gating which processes (sidecars included) a window's capability may spawn, and with what arguments.

Asked to "add PDF export" or "add barcode scanning," an agent will often reach straight for a sidecar wrapping some CLI tool, because it's the fastest path to a demo that works. That's sometimes right — but check first whether an existing Rust crate already does it in-process (no binary to bundle, no target triples to chase), and if a sidecar genuinely is the right call, push back on any capability that grants shell:allow-execute with "args": true. Fixed, pinned arguments turn "this window can run any command" into "this window can run exactly this one conversion."

The takeaway

Official plugin, then custom plugin, then sidecar — in that order of preference, because each step trades less bundling and less process-boundary complexity for more code you'd have to write yourself. A sidecar earns its keep when a working program already exists; once it's in the app, it's still just another thing a capability has to explicitly allow, scoped as tightly as the feature allows. Chapter 10 picks up the thread on the bundling side: building those per-target binaries and packaging everything — plugins compiled in, sidecars alongside — into something installable.
