Chapter 01

The Big Picture

What Is Tauri, and Why Does It Exist?

Tauri is a framework for building small, fast, secure desktop and mobile apps by pairing any web frontend you like with a Rust backend, and rendering the UI in the operating system's own webview instead of shipping a browser inside your app. That's the whole idea. Everything else in this book is detail on top of it.

The problem: one codebase, every desktop

Say you want to ship a real app — not a website — on Windows, macOS, and Linux, and maybe iOS and Android too. You want a proper window, a system tray icon, native file dialogs, notifications, auto-updates, a real installer. Writing and maintaining that natively three or four times over is slow and expensive. Most teams don't have a Swift engineer, a C++/Win32 engineer, and a GTK engineer sitting around — they have web developers.

So the industry's answer for a decade has been: build the UI in HTML/CSS/JS, like a website, and find a way to wrap it in a native-feeling shell. Electron made that mainstream around 2013 — it bundles a full copy of Chromium and Node.js inside every app, so your JavaScript UI runs the same way everywhere. It works, and it's why apps like VS Code and Slack exist as they do. But bundling an entire browser engine per app is heavy: tens to hundreds of megabytes on disk, and a meaningful RAM tax at idle, multiplied by every Electron app a user has installed.

Tauri's answer: reuse what's already there

Tauri starts from a different premise: every desktop OS already ships a capable, evergreen web renderer — WebView2 (Edge/Chromium) on Windows, WKWebView (Safari's engine) on macOS and iOS, WebKitGTK on Linux, and the system webview on Android. Tauri's Rust libraries (WRY for webview rendering, TAO for window/event handling) talk to that, instead of shipping a browser of its own. Your app ships only its own code and assets — a minimal Tauri app can be under 600 KB. The application core — window and lifecycle management, OS integration, anything privileged — is written in Rust, a systems language chosen for memory safety and raw performance rather than JavaScript's runtime overhead.

The project started in 2019, reached a 1.0 stable release in June 2022, and shipped a 2.0 stable release on October 2, 2024 — the release that added first-class iOS and Android targets alongside desktop, plus the capability-based permission system this book covers in Chapter 8. It's open source and maintained by the Tauri Programme within the Commons Conservancy, with Rust and web-standards people, not one vendor, driving the roadmap.

  • Typical idle RAM: 120–300 MB
  • Backend language: Node.js / JavaScript
  • Rendering is identical across OSes (it's the same Chromium everywhere)
  • Desktop-only (Windows, macOS, Linux)

Reuses the OS webview. Ships only your app's code/assets; the Rust core is compiled native.

  • Typical installed size: 2–10 MB (a minimal app can be <600 KB)
  • Typical idle RAM: 30–50 MB
  • Backend language: Rust (plus Swift/Kotlin for native mobile plugin code)
  • Rendering can vary slightly across OS webviews — test on each target
  • Desktop and mobile (iOS, Android) from Tauri 2.0 on

Figures are representative, not a benchmark from a specific pair of apps — the direction and rough magnitude are what matter here.

The mental model, at a glance

Key idea The webview is your UI and nothing else. The Rust core is the only side allowed to touch the filesystem, the network, the shell, or any other OS capability. Everything the frontend needs from the outside world crosses that boundary explicitly, as a command or an event — never silently. Chapter 2 draws this same picture in full detail.

Where this actually gets used

Tauri fits best where a team already has web skills, wants a genuinely small and fast native shell, and doesn't want to (or can't) bundle a browser. In practice that's turned into a real ecosystem:

Developer tools Git clients (GitButler), database GUIs (Seaquel, RedisME) — fast-launching, low-overhead tools developers keep open all day, where Electron's RAM tax is felt most.
Media & local-first apps Music players and taggers (Musicat), media server clients (Jellyfin Vue) — small installers, quick startup, no need for a bundled browser to play local files.
Privacy- and security-sensitive tools On-device AI and dictation tools (Screenpipe, SilentKeys), wealth-tracking apps (Wealthfolio) — cases where "the Rust core is the only thing that can touch your data" is a feature, not a constraint.
One codebase, phone included Since Tauri 2.0, teams have shipped the same app to desktop and mobile — e.g. a blood-pressure tracker (Tension) reusing its desktop UI and Rust logic on iOS/Android with native plugin code where needed.

You + the agent: how the workflow fits

Here's the part that matters for how you'll use this book: Tauri gives you a very clean seam to hand to a coding agent. You, the human, decide the architecture and constraints — which capabilities the app actually needs, what's allowed to touch the filesystem or network, what the window layout looks like. The agent writes the frontend code, the Rust commands, and the wiring between them against those constraints. Because the frontend/Rust boundary is explicit (every crossing is a named command or event), it's easy to describe precisely in a prompt and easy to review afterward — you're not asking the agent to "figure out the architecture," you're asking it to fill in a shape you already specified.

Ask your agent

Scaffold a minimal Tauri v2 app: a <your framework> frontend and a Rust backend. Add one command, greet(name: String) -> String, that returns "Hello, {name}!". Wire a text input and button in the UI that calls it via invoke() and displays the result. Don't add any plugins or extra permissions yet — keep this to the smallest working example.

A gotcha worth knowing before you start: because Tauri renders with whatever webview the OS provides, you don't get Electron's "it's the same Chromium everywhere" guarantee. A CSS or JS feature that works fine in WebView2 on Windows can render slightly differently — or not at all — in WKWebView on macOS. It's rarely a dealbreaker, but it means you actually test on each platform you ship to, rather than trusting one dev machine.

The takeaway

Tauri exists because bundling a browser per app is expensive, and most of what a desktop or mobile app needs from the OS — a window, a webview, files, notifications — is already sitting there, free. Tauri's job is to give your web frontend a small, fast, memory-safe Rust core to stand on, with an explicit, auditable boundary between "UI" and "everything privileged." That boundary is the single idea to carry into every later chapter — and it's exactly the seam you'll hand to a coding agent, chapter by chapter, for the rest of this book.
