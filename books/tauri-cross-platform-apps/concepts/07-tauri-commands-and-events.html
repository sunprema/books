Chapter 07

The Two Directions, Working Together

Communication Patterns: Commands and Events

Chapters 5 and 6 gave you each half of the seam separately — invoke() calling into #[tauri::command], and emit()/listen() moving payloads the other way. This chapter is about the one habit that matters more than either fact on its own: knowing when a command alone is enough, and when you need to bolt events onto it — because almost every real feature in a Tauri app ends up taking the same four-beat shape.

Two verbs, two directions — a fast recap

A command is request/response: the frontend calls invoke("name", args), Rust runs exactly once, and a single value comes back — resolved or rejected. It's RPC, full stop. An event is publish/subscribe: Rust (or another window) calls emit(name, payload) with nobody waiting, and every listen(name, cb) registered for that name fires, zero, one, or many times, on its own schedule. Neither replaces the other — a command can't tell you about three things that happen over the next ten seconds, and an event can't hand the caller a typed return value it's waiting on.

The tell If the frontend's question is "what's the answer?", reach for a command. If the question is "tell me when something happens" — possibly more than once, possibly triggered by something that isn't a button click at all — reach for an event.

The shape almost every feature takes: trigger, work, report, arrive

The pattern that actually matters in practice combines both: the frontend triggers a command, Rust does the work — often slow enough that the user shouldn't just stare at a spinner — and while it runs, Rust reports progress by emitting events the frontend is already listening for, until a final event (or the command's own return) says the work has arrived. A file import, a large download, a batch of database migrations, an export to PDF — nearly all of them are this same four-beat shape wearing different clothes.

On the Rust side, the command itself stays thin: it spawns the real work (a background task, or an async body that .awaits in a loop) and returns almost immediately, so invoke() resolves fast even though the actual import keeps running. The AppHandle is what makes this possible — it's cheap to clone and safe to move into the spawned task, which is exactly what lets that task keep calling .emit(…) long after the command that started it has already returned. 

#[tauri::command]
fn start_import(app: tauri::AppHandle, path: String) {
    tauri::async_runtime::spawn(async move {
        let total = count_rows(&path).await;
        for (done, row) in read_rows(&path).enumerate() {
            import_row(row).await;
            let _ = app.emit("import-progress", serde_json::json!({
                "done": done + 1, "total": total
            }));
        }
        let _ = app.emit("import-done", total);
    });
    // returns immediately — the spawned task keeps running and emitting
}

On the frontend, register both listeners before calling invoke() — events are fire-and-forget with no replay buffer, so a listener that registers after the first emit() already fired simply misses it. Call the returned unlisten functions once the terminal event arrives, or you'll leak a listener every time the user re-runs the import. 

import { invoke } from '@tauri-apps/api/core';
import { listen } from '@tauri-apps/api/event';

const offProgress = await listen('import-progress', (e) => {
  updateProgressBar(e.payload.done, e.payload.total);
});
const offDone = await listen('import-done', (e) => {
  showComplete(e.payload);
  offProgress(); offDone();  // clean up once the run is over
});

await invoke('start_import', { path });  // fires after listeners are armed

Ask your agent

Add a Rust command export_report(app: AppHandle, format: String) that spawns an async task generating the report page by page. Emit an "export-progress" event with { page, totalPages } after each page and an "export-done" event with the final file path when finished, or an "export-error" event with a message if it fails. On the frontend, wire up a progress bar component that listens for all three, unlistens once export-done or export-error fires, and calls invoke to start the export only after the listeners are registered.

Targeting who hears an event

Plain app.emit(name, payload) broadcasts to every listener in every window — fine for a single-window app, noisy for a multi-window one. app.emit_to(target, name, payload) aims at one specific window/webview label instead, so a progress event meant for a background export window doesn't also wake up listeners in the main window that have nothing to do with it. 

let _ = app.emit_to("exporter", "export-progress", payload);
Gotcha emit_to targets a window/webview label, matched by exact string — misspell it, or target a window that already closed, and nothing errors. The event just goes nowhere and the frontend silently never updates, which is a much harder bug to spot than a rejected promise. Double-check the label against Chapter 3 and Chapter 4's window configuration when a listener that should be firing isn't.

When events get too chatty: Channels

The global event bus is convenient but not built for volume or ordering guarantees — fine for a handful of progress ticks, less fine for something emitting hundreds of times a second (piping a child process's stdout, streaming WebSocket frames, byte-level download progress). For that, Tauri v2 has a dedicated Channel<T> type: a command takes one as an argument, and every channel.send(…) call delivers ordered, low-overhead messages straight to the one frontend callback that opened it — no broadcast, no listener bookkeeping. Tauri uses Channels internally for exactly this: download progress, spawned-process output, and similar high-throughput streams.

Where this shows up A log-tail viewer streaming a growing file to the UI line by line, or a download manager reporting byte-level progress at high frequency, both fit the Channel shape better than the plain event bus — the volume and the ordering guarantee are the point.

Reach for a plain event first — it's simpler, and Chapter 12's file-inspector example only ever needs one. Reach for a Channel once you notice you're emitting fast enough, or often enough, that ordering or event-bus overhead becomes a real concern; it's an optimization on the same "report progress" idea, not a different pattern.

Deciding whether you need an event at all

SituationReach for
A quick lookup or write — get a setting, save a note, fetch a file's metadata A command alone; the return value is the whole answer
Work that takes long enough the UI should show progress, not just a spinner A command to start it, events (or a Channel) to report progress
Something Rust-side triggers on its own — a filesystem watcher, a timer, a tray-icon click, a deep link An event; there's no frontend call to hang a response on
One Rust-side happening that more than one window or component needs to react to An event; every listener gets it, a command reply only reaches the caller
Very high-frequency or strictly ordered streaming (child process output, byte-level progress) A Channel<T> passed into the command
Gotcha Emitting on every single unit of work — every row, every byte — floods the event bus and can make the UI feel less responsive, not more, as the frontend churns through a backlog of progress updates it can't render fast enough to matter. Throttle: emit on a percentage step, a time interval, or every N items, not every one.
Key idea A command is the frontend asking a question and waiting for one answer. An event is Rust announcing something with nobody required to be listening. Most real features need both, chained: invoke to trigger, events to report progress as it happens, a final event (or the original command's own resolution) to say it's done.

Quick glossary

Command

Request/response, RPC-style — invoke() waits for exactly one resolved or rejected value.

Event

Publish/subscribe — emit() with nobody waiting; every listen() for that name fires, any number of times.

emit() / emit_to()

Broadcast to every window, or aim at one window/webview by label.

listen() / unlisten

Registers a callback for an event name; call the returned function when you're done receiving it.

Trigger → work → report → arrive

Invoke to start, spawn the work, emit progress as it runs, a terminal event (or the command's own return) signals done.

Channel<T>

Ordered, low-overhead streaming to one caller — the fix when plain events get too chatty.

A mix-up worth catching in review: an agent asked to "show progress" will sometimes reach for polling — the frontend calling a get_progress() command on an interval — instead of events. It works, but it's strictly worse here: it wastes calls when nothing changed, and it can miss a fast-moving update between polls. If Rust already knows the moment something changes, have it emit that moment rather than making the frontend guess how often to ask.

The takeaway

You now have the full communication picture: commands for asking, events for announcing, and the trigger–work–report–arrive pattern that chains them into almost every real feature. What you haven't seen yet is what's allowed to cross that seam in the first place — Chapter 8 covers the security model and capabilities that constrain which commands a given window can even call.
