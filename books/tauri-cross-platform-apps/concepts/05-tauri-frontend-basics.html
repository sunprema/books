Chapter 05

What Your JS Actually Talks To

Frontend Basics from Tauri's Perspective

Your frontend is just a web app — React, Vue, Svelte, plain JS, whatever you already know — plus one small, deliberate seam where it talks to Rust. That seam is @tauri-apps/api and a handful of plugin packages. This chapter is not a framework tutorial; it's a tour of that seam, so you know exactly what to name when you ask an agent to wire a button up to the Rust side.

Tauri doesn't care about your UI framework

Everything Tauri gives your frontend arrives as a normal JavaScript/TypeScript package you npm install and import — there's no Tauri-specific component model, no required state library, no template syntax of its own. Whatever builds to static HTML/CSS/JS and can run an npm script qualifies: point devUrl and frontendDist (Chapter 4) at it and Tauri serves it in the webview like any other page. That's deliberate — it's the flip side of the "any web-stack frontend" claim from Chapter 2, and it means this chapter's content transfers unchanged whether your team ships React, Vue, Svelte, SolidJS, or a build-free pile of HTML files.

Where this shows up Teams that already have a React or Vue admin dashboard and want a desktop build usually change nothing about the frontend's structure or state management — they add src-tauri/, install @tauri-apps/api, and start replacing a handful of "fetch from our own backend" calls with invoke() calls to Rust commands.

@tauri-apps/api: a client, not a framework

@tauri-apps/api is a thin, typed client for the same two verbs from Chapter 3 — commands and events — plus a few facts about the runtime itself. As of Tauri 2, it only ships those core modules; everything that touches an actual OS feature (dialogs, filesystem, notifications, HTTP, the OS shell…) was deliberately split out into separate plugin packages you install one at a time, each pairing with a Rust plugin crate registered on the backend (Chapter 9). That split keeps your dependency tree — and your capability grants — scoped to exactly the OS features you actually use.

@tauri-apps/api/window, /webviewWindow getCurrentWindow, WebviewWindow, LogicalSize Resize, minimize, focus, or create windows from the frontend
@tauri-apps/api/path, /app appDataDir, join, getVersion… Resolve platform-correct paths; read app metadata
@tauri-apps/plugin-* dialog, fs, notification, http, shell, autostart… Touch an actual OS feature — each is a separate install, paired with a Rust plugin
If you're reading v1-era code Tauri 1 bundled dialog, fs, notification, and friends straight into @tauri-apps/api. If an agent (or an old tutorial) writes import { open } from '@tauri-apps/api/dialog', that's the v1 shape — on v2 it's import { open } from '@tauri-apps/plugin-dialog', and the matching Rust crate has to be registered separately too.

The three things your frontend code does

Strip away framework boilerplate and almost every line of Tauri-aware frontend code is doing one of three things: asking Rust to do something and waiting for the answer, listening for Rust to announce something unprompted, or calling a plugin's JS wrapper — which, under the hood, is just doing one of the first two on your behalf.

Calling a command with invoke() is the frontend's only door into Rust (Chapter 3): it returns a promise that resolves with the command's return value or rejects with its error, and arguments travel as a plain object matching the command's parameters. 

import { invoke } from '@tauri-apps/api/core';

const greeting = await invoke('greet', { name: 'Ada' });
// greeting is whatever the Rust #[tauri::command] fn greet(name: String) -> String returned

Listening for an event with listen() is the other direction — Rust announces something and every listener on that event name gets the payload, unprompted. listen() itself returns a promise that resolves to an unlisten function; call that when the listener should stop (a component unmounting, a task finishing) or it keeps firing for the life of the page. 

import { listen } from '@tauri-apps/api/event';

const unlisten = await listen('import-progress', (event) => {
  updateProgressBar(event.payload.done, event.payload.total);
});

// later, e.g. when the view unmounts:
unlisten();

Calling a plugin's JS wrapper looks like calling any other async JS function, but it's a thin convenience layer over the same invoke() mechanism — the plugin's Rust half registered its own hidden commands, and the JS package just calls them with a friendlier name and typed signature. 

import { open } from '@tauri-apps/plugin-dialog';
import { sendNotification } from '@tauri-apps/plugin-notification';

const path = await open({ multiple: false, directory: false });
if (path) sendNotification({ title: 'File selected', body: path });

Ask your agent

Add a "Refresh" button to the dashboard view that calls invoke("get_recent_projects") from @tauri-apps/api/core, replaces the project list in state with the result, and shows a toast on rejection. Don't touch any Rust code — the command already exists; this is frontend wiring only.

The pattern: isolate Tauri calls in a thin native-bridge module

The single habit worth building early: don't scatter invoke(), listen(), and plugin imports across your components. Put them behind one small module — often something like src/native/ or src/lib/tauri.ts — that exports plain, typed functions with names that describe intent, not mechanism: pickFile(), not "a component that calls dialog.open()." The rest of your UI code imports from that module and never imports @tauri-apps/* directly. 

// src/native/tauri.ts — the ONLY file that imports @tauri-apps/*
import { invoke } from '@tauri-apps/api/core';
import { listen } from '@tauri-apps/api/event';
import { open } from '@tauri-apps/plugin-dialog';

export async function pickFile(): Promise<string | null> {
  return await open({ multiple: false });
}

export async function getFileMetadata(path: string) {
  return await invoke('get_file_metadata', { path });
}

export function onImportProgress(cb: (done: number, total: number) => void) {
  return listen<{ done: number; total: number }>('import-progress', (e) => cb(e.payload.done, e.payload.total));
}

This earns its keep three separate ways: when the Rust side of a command changes shape, you edit one function instead of hunting through components; you can swap the bridge module for a mock in tests or a browser-only build without touching a single UI component; and it gives you a natural place to point an agent — "add a function to the native bridge" is unambiguous in a way "call Tauri somewhere in the file picker" is not.

Every component imports @tauri-apps/* directly. Fast to start, expensive later.

  • A Rust command signature change means grepping the whole frontend for its call sites
  • No single place to mock Tauri for a browser preview or a test suite
  • An agent prompt has to describe mechanism ("call invoke") instead of intent ("pick a file")

One module wraps every invoke/listen/plugin call. Slightly more setup, pays off fast.

  • Components import typed, intent-named functions — pickFile(), not dialog internals
  • Swap the whole module for a mock to run the UI outside Tauri
  • Agent prompts stay short: "add X to the native bridge," reviewed in one file
Where this shows up Apps that ship both a Tauri desktop build and a plain web/PWA build from the same frontend code lean on this hardest — the bridge module is exactly where you'd branch on isTauri() or swap in a fetch-based implementation for the web target, without the rest of the app knowing which build it's running in.

Ask your agent

Create a native bridge module at src/native/tauri.ts. Move every existing invoke(), listen(), and @tauri-apps/plugin-* import out of the components under src/views/ into typed, intent-named functions in that module, and update the components to import from it instead. Don't change any Rust code or component behavior — this is a pure refactor of where the Tauri calls live.

Knowing whether you're even in Tauri

isTauri() from @tauri-apps/api/core (and the window.isTauri global it mirrors) answers one question: is this code currently running inside a Tauri webview at all? It matters for exactly the shared-codebase case above — a component that should call the native file picker in the desktop build but fall back to an <input type="file"> in a plain-browser build reads this flag once, in the bridge module, and picks a path. It's not a permissions check and it's not a substitute for capabilities (Chapter 8) — it only tells you which build you're in.

Gotcha A promise from invoke() or a plugin call rejecting doesn't always mean your frontend code is wrong — it's just as often a missing capability grant (Chapter 8): the command exists and the arguments are fine, but the calling window's capability file never granted it. Before debugging the JS, check whether the window even has permission to call that command.

Quick glossary

@tauri-apps/api

The core JS client: invoke, event, window, path, app — no OS-feature plugins bundled in.

@tauri-apps/plugin-*

A separate JS package per OS feature (dialog, fs, notification…), paired with a Rust plugin crate.

invoke()

Calls a Rust command by name; returns a promise resolving to its return value.

listen() / unlisten

Registers a callback for an event name; call the returned function to stop receiving it.

isTauri()

True only when the current code is running inside a Tauri webview, not a plain browser.

Native-bridge module

One file that owns every @tauri-apps/* import; the rest of the UI calls typed, intent-named functions.

Key idea Your frontend stays an ordinary web app in whatever stack you already use. The only Tauri-specific surface is @tauri-apps/api's core/event/window modules plus whichever @tauri-apps/plugin-* packages you install, and every call through that surface is either an invoke(), a listen(), or a plugin wrapper doing one of those two things for you. Put that surface behind one bridge module and the rest of your app never has to know Tauri exists.

The takeaway

When you're directing an agent on frontend work, you now have the exact vocabulary to be precise: name the module (@tauri-apps/api/core vs a specific plugin package), name the verb (invoke vs listen), and say where it belongs (the native bridge module, not scattered through components). Chapter 6 flips to the other side of that same seam — what a command looks like from the Rust code an agent will actually write.
