Chapter 03

The Shared Vocabulary

Core Concepts and Terminology

You now have the picture: one Rust Core, one or more WebViews, commands and events crossing between them. This chapter puts precise names on every piece of that picture. That matters more than it sounds — when you ask a coding agent for "a command that reads a file," it needs to land on exactly the right primitive, not improvise one. The words in this chapter are the words you'll use in every prompt for the rest of this book.

App, window, webview — the three nouns

App is the single running instance of your Tauri backend — the Rust process that starts when tauri::Builder::default()….run(…) is called and keeps running until the last window closes. From inside a command, an async task, or another thread, you reach it through an AppHandle: a small, cheaply cloneable handle that gets you the same capabilities as the app itself — create a window, emit an event, reach shared state — from anywhere, not just the setup closure. It matters because "pass the AppHandle in" is how you make anything in Rust reachable from a background task without fighting the borrow checker.

A window is the native OS frame — the title bar, the border, the thing your window manager moves and resizes — created and owned by Core. A webview is the content surface living inside a window that actually renders your HTML/CSS/JS, using whichever engine the OS ships (Chapter 1). As of Tauri 2 these are deliberately two different types in the Rust API: the common case bundles them as one WebviewWindow (one window, one webview, which is the stable default you should design around), but the split exists because an unstable API lets one window host more than one webview for advanced multi-pane layouts. Every window and webview carries a unique string label (e.g. "main", "settings") — that label is how a command or event targets one specific window instead of all of them.

Ask your agent

Create a second window with the label "settings", 480x360, not resizable, hidden until the user opens it from the main window's menu. Store the AppHandle from setup() so a Rust background task can show or focus this window later without needing a reference passed through every function that spawned it.

Commands, events, and listeners — the two verbs

A command is a Rust function marked #[tauri::command] and registered with generate_handler![…], which the frontend calls by name with invoke("name", { args }) — request in, response out, exactly like calling an RPC endpoint. Arguments and return values cross as JSON, a command can be sync or async, and a Result<T, E> return maps directly to a resolved or rejected JavaScript promise. It matters because it's the only sanctioned way the frontend triggers Rust work — there's no other door.

An event runs the other direction and doesn't expect a reply: Core or a window calls emit(name, payload) to broadcast to every listener, or emit_to(label, name, payload) to reach just one window/webview by label. A listener is simply the callback registered with listen(name, callback) on the receiving side (frontend or Rust); it keeps firing until you call the unlisten function it returns. Use a command when you need an answer; use an event for a background task announcing "I made progress" or "I finished" with nobody blocking on it. Chapter 7 is the deep dive on choosing between them and chaining the two together.

Ask your agent

Add an async command import_large_file(path: String) that reads the file in chunks on a background task and, for every chunk processed, emits an "import-progress" event with { done: usize, total: usize } to the calling window only (use emit_to, not a broadcast emit). The command itself should return immediately after starting the task, not block until it's finished.

Plugins vs. sidecars — two ways to extend Core

A plugin is Rust code (often with a thin JS binding) that registers itself into the Core process at startup via .plugin(tauri_plugin_x::init()), adding its own commands, events, and state in the same address space as your app. Official plugins cover dialogs, filesystem access, HTTP, notifications, autostart, the OS shell, and more. A sidecar is the opposite: a separate compiled binary — listed under bundle.externalBin in tauri.conf.json — that Core spawns as its own OS process (through the shell plugin's sidecar() API) and talks to over stdio, the way a shell script shells out to another program. It matters because picking wrong wastes real effort: reimplementing an existing tool as a plugin, or reaching for a heavyweight sidecar when a plugin already does the job.

Runs inside Core. Rust crate (+ optional JS), same process, same address space.

  • Adds new #[tauri::command]s and events like they were your own
  • Reach for it: OS integration, simple native features, "someone already wrote this in Rust"
  • Governed by the same permissions/capabilities system as your own commands

Runs as its own OS process. A separately bundled, separately supervised binary.

  • Talks to Core over stdio — whatever protocol the binary itself speaks
  • Reach for it: an existing CLI tool, a heavy runtime (Python, Node) you won't reimplement in Rust
  • Needs explicit permission just to spawn it, on top of whatever the binary does internally

Ask your agent

We need to shell out to an existing Python script, transcode.py, that already does our video transcoding. Bundle it as a Tauri sidecar rather than rewriting it in Rust: add it to externalBin, grant the shell plugin's sidecar-execute permission scoped to only that binary name, and add a Rust command that spawns it, streams its stdout lines back as "transcode-log" events, and resolves once the process exits.

Permissions and capabilities — the gate, not the door v2+

Tauri 1 exposed every registered command to every window by default. Tauri 2 flipped that: nothing is reachable until you say so. A permission is an on/off toggle for one command or a named group of them (usually one ships per plugin, e.g. fs:allow-read-file). A scope narrows a permission with parameters — an fs permission scoped to $APPDATA/* still lets a command run, but only against paths under that directory. A capability is the file that ties permissions (and their scopes) to specific windows or webviews by label. A window's actual, effective access is the union of every capability that names it — and if no capability names it, that window can call nothing at all.

Why it matters This is what makes "a powerful, general-purpose webview" survivable: the frontend can only ever do what its window's capabilities explicitly grant, at the granularity you chose — not "can this window use the filesystem plugin" but "can this window read files, and only inside this one directory." Chapter 8 goes deep on designing a capability set; here, just recognize the three words when you see them.

Ask your agent

Restrict the fs plugin so only the main window can use it, and only for reading files under the user's chosen project directory — not the whole filesystem. Add a capability that grants fs:allow-read-file scoped to that directory to the "main" window label only; don't grant it to the "settings" window, which has no need to touch files.

Dev server vs. production build

tauri dev runs your frontend's own dev server (Vite, webpack, whatever it normally uses) alongside a debug build of the Rust backend, opens a window pointed at that dev server's URL, and rebuilds the Rust side automatically when its source changes — you get the frontend tooling's own hot reload for free, plus a slower but automatic recompile-and-relaunch on the Rust side. tauri build is the other end of the pipeline: it runs your frontend's production build, compiles Rust in release mode, and packages both into the installers/app bundles configured under tauri.conf.json's bundle section — an MSI/NSIS installer on Windows, a .app/DMG on macOS, an AppImage/deb/rpm on Linux, plus whatever mobile targets you've set up. It matters mostly so you don't accidentally profile performance, test the CSP, or judge install size against a dev build — only the release build reflects what ships.

Ask your agent

Set up the npm scripts so `tauri dev` points at our Vite dev server with hot reload working for the frontend, and confirm devUrl / beforeDevCommand / beforeBuildCommand in tauri.conf.json are wired so `tauri build` runs our production Vite build before packaging. Don't change any window or capability config — this is dev-workflow wiring only.

tauri.conf.json — the file that ties it together

One JSON file at the root of src-tauri/ is the app's central manifest: identity (app name, version, bundle identifier), default window settings, the frontend build commands Tauri should run and watch, the Content-Security-Policy and other security defaults, and everything the bundle step needs to package each platform target. It answers "what are we building and how do we package it" — a different question from the capability files under src-tauri/capabilities/, which answer "what is the running app allowed to do." Chapter 4 walks the file's shape area by area; for now, just know it's the one place almost every "configure X" request from you to an agent will touch.

Key idea Every term above names one piece of the same two-box picture from Chapters 1–2: App owns windows, each hosting a webview; the frontend crosses into Rust only through a command (asks, waits for an answer) or receives a event (told, doesn't ask); plugins and sidecars are the two ways Core grows; permissions, scopes, and capabilities decide who gets to use any of it; and tauri.conf.json is where you configure the shape of the whole thing.

Quick glossary

One line each, for when you just need the word, not the paragraph:

App / AppHandle

The running Rust process, and a cloneable handle to it usable from anywhere.

Window

The native OS frame Core owns; identified by a unique label.

WebView

The content surface inside a window that renders your frontend; one per window, by default.

Command

A Rust function the frontend calls via invoke() — request/response.

Event

A message emit/emit_to broadcasts to listeners — publish/subscribe.

Listener

The callback registered with listen() that receives an event.

Plugin

Rust (+ optional JS) that registers new commands/events into Core's own process.

Sidecar

A separate bundled binary Core spawns as its own OS process and talks to over stdio.

Permission

An on/off toggle allowing or denying one command (or a named group of them).

Scope

Parameters narrowing a permission — e.g. which paths, which URLs.

Capability

The file binding permissions (+ scopes) to specific window/webview labels.

tauri dev / tauri build

Dev-server-backed debug run with hot reload, vs. the release build that gets packaged and shipped.

A mix-up worth avoiding out loud: people say "window" when they mean "webview" constantly, and in casual conversation it rarely matters. It starts to matter the moment you're writing a precise agent prompt or reading Tauri's Rust docs — Window and Webview are genuinely different types with different methods. If an instruction is about native chrome (size, position, decorations, always-on-top), you mean the window. If it's about what's rendered or IPC, you mean the webview.

The takeaway

None of these terms are new ideas — they're names for the boxes and arrows Chapters 1 and 2 already drew. App owns windows, windows host webviews, commands and events cross the boundary between them, plugins and sidecars are the two ways to extend what Core can do, and permissions/scopes/capabilities decide who's allowed to use any of it. Carry this vocabulary forward: from here on, every chapter assumes you can name the piece it's talking about, and every "Ask your agent" prompt in this book leans on these exact words on purpose — precise nouns make for precise prompts.
