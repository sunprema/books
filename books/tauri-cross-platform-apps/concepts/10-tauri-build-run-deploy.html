Chapter 10

Three Modes: Watch It Change, Compile It Once, Hand It to a Stranger

Build, Run, and Deploy Workflow

Every chapter so far has been about what you write — commands, events, capabilities, plugins. This one is about what happens to all of that once you stop writing and start shipping: how the same project turns into a fast-iterating dev session, then a single compiled binary, then an installer someone double-clicks on a machine you'll never see. Three modes, three different sets of guarantees.

Dev, build, and distribute are three different questions

It's easy to think of "running the app" as one thing with different speeds. It isn't. Each mode answers a different question, and conflating them is where "works on my machine" release surprises come from.

ModeQuestion it answersWhat you get
tauri devDoes this change work, right now?An unpacked, unoptimized window that reloads as you edit
tauri buildDoes this compile clean, in release mode, for one platform?An optimized binary plus one or more installer artifacts
DistributeCan someone who isn't you install and trust this?A signed, versioned artifact on a channel they can reach

Dev mode: fast feedback, not a preview of the shipped app

tauri dev does two things at once and keeps them in sync: it launches (or attaches to) your frontend's own dev server — the same Vite/webpack/whatever process you'd run for a plain web app, driven by beforeDevCommand from Chapter 4's build config — and it compiles Core in debug mode and opens a real OS window pointed at that dev server's devUrl. Edit the frontend and the webview hot-reloads exactly like a browser tab; edit Rust and Tauri notices the source change, recompiles Core, and relaunches the window with your new command or event wired in. Debug builds skip most optimization to compile fast, which is the whole point in a loop you run dozens of times an hour — and exactly why a debug build's performance and binary size tell you nothing about what ships.

Key idea Dev mode is a development tool, not a smaller version of the release build. It runs unoptimized, it may skip some CSP enforcement for convenience, and it talks to a live dev server instead of bundled static files. "It works in tauri dev" is necessary but never sufficient — run at least one real tauri build before you believe the feature is done.

Ask your agent

Our frontend dev server isn't starting automatically when I run tauri dev — check that build.beforeDevCommand and build.devUrl in tauri.conf.json actually match how our frontend's own dev script starts and which port it binds, and fix the mismatch. Don't touch anything under the app or bundle sections.

What a production build actually does

tauri build runs three jobs in sequence, and every one of them can fail for a different reason — worth knowing which stage broke when a release goes red. First, build.beforeBuildCommand runs your frontend's own production build (minified, bundled, no dev server) into build.frontendDist. Second, Core compiles in --release mode — full optimizations, which is why a release compile takes noticeably longer than a dev one. Third, the bundler takes that compiled binary plus the built frontend plus everything bundle.resources and bundle.externalBin (Chapter 9's sidecars) name, and packages all of it into one or more installer formats for the host platform.

PlatformTypical bundle.targetsWhat a user double-clicks
Windowsmsi, nsisAn MSI installer, or a smaller NSIS .exe installer
macOSapp, dmgA .app bundle, usually wrapped in a .dmg disk image
Linuxdeb, rpm, appimageA .deb/.rpm package for the distro's manager, or a self-contained AppImage
 
// src-tauri/tauri.conf.json
{
  "bundle": {
    "active": true,
    "targets": ["msi", "nsis", "dmg", "appimage", "deb"]
  }
}
// or "targets": "all" to produce every format the host platform supports
Gotcha There is no such thing as "build the Windows installer from your Mac." Tauri does not cross-compile a native bundle for a different OS — you build a Windows target on a Windows machine (or Windows CI runner), a macOS target on macOS, a Linux target on Linux. That's exactly why release pipelines run a matrix: one job per target OS, in parallel, each producing that platform's own artifacts.

Platform targets: a CI matrix, not a single command

For desktop, "release the app" almost always means "run tauri build once per target OS, on a runner of that OS." The official tauri-action GitHub Action exists for exactly this: point it at a matrix of windows-latest, macos-latest, and ubuntu-22.04 runners, and it runs the whole build-and-bundle pipeline on each, uploading the resulting installers to a GitHub release. If Chapter 9's sidecar needs a binary per Rust target triple, this is where those triples actually get produced or fetched — one per matrix leg, named and placed before the bundler runs, or that platform's build ships silently without the sidecar it needs.

In practice A three-person team ships weekly: pushing a version tag triggers a GitHub Actions matrix that builds Windows, macOS (Intel and Apple Silicon), and Linux artifacts in parallel, signs each one, and attaches them to a draft release — nobody's laptop is a required part of shipping, and the same pipeline produces every release the same way.

Ask your agent

Set up a GitHub Actions workflow using tauri-apps/tauri-action that builds this app on a matrix of windows-latest, macos-latest, and ubuntu-22.04 whenever I push a tag matching v*, and attaches the resulting installers to a draft GitHub release. Don't add code signing yet — I'll handle certificates separately.

Mobile targets: the same app, a different init step

Tauri's mobile support reuses the same frontend, the same commands, the same capability system — tauri android init and tauri ios init generate a native Android Studio / Xcode project alongside your existing src-tauri/, wiring your Rust Core into a WKWebView on iOS and the system WebView on Android instead of a native OS window. From there, tauri android dev / tauri ios dev run on an emulator or simulator with the same hot-reload loop as desktop dev mode, and tauri android build / tauri ios build produce an APK/AAB or an IPA instead of a desktop installer. iOS builds require macOS and a full Xcode install — there's no way around that platform requirement, it comes from Apple's own toolchain, not from Tauri.

Note Not every plugin and not every OS API behaves identically across all five targets. Treat "does this feature exist on mobile" as a question to check per plugin, not an assumption — a command that shells out to spawn a sidecar process, for instance, has no equivalent on iOS's sandboxed process model.

Distribution: installers, portable builds, and stores

A built installer isn't yet "distributed" — it still needs a place a user can trust and reach. The common paths, roughly in order of how much platform ceremony they require:

ChannelWhat it buys youWhat it costs
Direct download (your own site, a GitHub release)Full control, fastest to shipYou handle code signing yourself, or OSes warn on install
Portable buildNo install step, no admin rights neededNo OS-level update hooks, no Start-menu integration
Microsoft Store / Mac App StoreUser trust, discoverability, store-managed updatesReview process, sandboxing rules, revenue cut
Package managers (winget, Homebrew, apt/dnf, Flathub)Familiar install path for that platform's usersA separate packaging definition to maintain per manager

Whichever channel, an unsigned installer earns a scary OS warning on first launch — "Windows protected your PC," an unidentified-developer block on macOS — so a real release pipeline signs the binary with a code-signing certificate for that platform before anyone downloads it. That's a separate credential from the updater's signing key below; both matter, neither substitutes for the other.

Auto-updates, at a high level

Once an app is installed somewhere, "ship a bug fix" means getting a new version onto machines you don't control. Tauri's official updater plugin handles this as: the running app periodically fetches a small JSON manifest (commonly named latest.json) from a URL you host, compares its version to its own, and if newer, downloads and installs that platform's artifact. The identifier from Chapter 4's tauri.conf.json is how the installed app and the manifest agree they're talking about the same product across versions. 

// latest.json — the update manifest the running app polls
{
  "version": "1.4.0",
  "pub_date": "2026-07-10T09:00:00Z",
  "platforms": {
    "windows-x86_64": { "signature": "...", "url": "https://.../app-1.4.0-x64.msi.zip" },
    "darwin-aarch64": { "signature": "...", "url": "https://.../app-1.4.0-aarch64.app.tar.gz" }
  }
}

Every artifact the updater installs must carry a signature it can verify — there's no opt-out. A keypair is generated once with the Tauri CLI; the public half goes into tauri.conf.json so every installed copy of the app can check signatures, and the private half signs each release's artifacts during tauri build (with createUpdaterArtifacts enabled) via the TAURI_SIGNING_PRIVATE_KEY environment variable, typically injected as a CI secret — never committed to the repo.

Gotcha Losing the private signing key doesn't just block your next release — it strands every copy of the app already installed. They keep polling for updates signed by a key you no longer have, so a lost key means every existing user needs a fresh, out-of-band install to get back on an updatable version. Treat it like a production database credential: generated once, stored in a secrets manager, never in the repo.

Ask your agent

Wire up the Tauri updater plugin: add it to src-tauri, set createUpdaterArtifacts to true in tauri.conf.json, and read the signing private key from the TAURI_SIGNING_PRIVATE_KEY environment variable in the release workflow — don't hardcode or commit it anywhere. Have the frontend check for updates once on startup and show a non-blocking "update available" banner rather than forcing an immediate restart.

In practice A internal ops tool used by twenty people auto-updates silently on launch with no user-facing prompt at all — acceptable because the team controls both ends. A consumer note-taking app instead shows a dismissible "restart to update" banner, because forcing a restart mid-edit on someone else's machine is a trust-eroding move a public app can't afford.

A release checklist mental model

Before you ship…Reach for
"Does this feature work at all?"tauri dev
"Does this compile and bundle cleanly for one platform?"tauri build
"Does every target OS get a matching artifact, including sidecars?"A CI matrix, one job per platform
"Will the OS trust this installer on first run?"Platform code signing, separate from updater signing
"Will installed copies find out about this release?"Updater plugin + signed manifest, keyed by identifier

Quick glossary

tauri dev

Runs the frontend dev server and a debug Core build together, hot-reloading both as you edit.

tauri build

Builds the frontend for production, compiles Core in release mode, and bundles both into installers.

bundle.targets

The installer formats to produce for the host platform — msi/nsis, dmg, deb/rpm/appimage.

CI matrix

One build job per target OS, run in parallel on that OS's own runner — Tauri doesn't cross-compile native bundles.

updater plugin

Polls a hosted JSON manifest for a newer signed version and installs it into the running app.

signing keypair

Public key ships in tauri.conf.json to verify updates; private key signs artifacts and must never leak.

Ask an agent to "set up releases" and it will often reach for the happy path — one platform, no signing, no matrix — because that's the fastest thing that visibly works. Push back explicitly: name every platform you actually ship, ask for the CI matrix by name, and ask for updater signing wired through CI secrets from day one. Retrofitting a signing key onto an app with users already installed is strictly harder than starting with it.

The takeaway

Dev mode optimizes for iteration speed and tells you nothing about what ships. A build is a real, release-mode compile plus a bundle step, native to one platform at a time, which is why shipping to several platforms is a CI matrix problem, not a flag problem. And distribution doesn't end at "installer exists" — it ends at "installed copies can trust and receive the next one," which is what platform code signing and the updater's own signing keypair are each, separately, for. Chapter 11 turns to the workflow side of all this: how to brief a coding agent so it builds toward this pipeline instead of away from it.
