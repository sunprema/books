Chapter 11

Brief Like an Architect, Review Like a Skeptic

Working with Coding Agents on Tauri Projects

Everything in Chapters 1 through 10 was vocabulary — commands, events, capabilities, plugins, the build pipeline. This chapter is about the job that vocabulary was for: you hold the architecture and the constraints in your head, and you hand a coding agent precisely-scoped pieces of it to type out. Tauri rewards this split more than most frameworks, because it has two languages and an explicit, opt-in security model — both are places where an agent left to guess will guess wrong in a way that compiles, runs, and quietly does the wrong thing.

Why Tauri specifically needs this discipline

Ask an agent to "add dark mode" to a plain web app and the blast radius is CSS. Ask it to "let the user export their data" in a Tauri app and the request touches a Rust command, a filesystem capability, maybe a dialog plugin, and the boundary between a frontend that can only ask and a backend that can actually act. Two things make Tauri unusually easy to get subtly wrong: a real language boundary (TypeScript/JavaScript talking to Rust across invoke, from Chapter 7) and a default-deny security model (from Chapter 8) where every OS capability is off until a capability file says otherwise. An agent optimizing for "make the feature work" will happily punch through either one — and the code will compile and demo fine, which is exactly why you can't rely on the demo to catch it.

Frame the task like an architecture brief, not a feature request

"Add an export feature" leaves an agent to invent the command name, the error type, which plugin to reach for, and how much filesystem access to grant — four architectural decisions it has no business making alone. A scoped brief answers those up front: which layer the change belongs in (frontend only, Rust only, or a named command/event contract crossing both), what the capability implications are, and what's explicitly out of scope for this pass. The difference isn't length — it's specificity about the boundary.

"Let the user export their notes to a file."

  • The agent picks the command name, the file format, and the save-location UI on its own
  • It typically reaches for the broadest fs capability that makes the demo work — often fs:allow-write-file scoped to nothing, or worse, fs:default
  • Error handling is whatever the agent's training data usually does, not what your app's conventions are
  • You find out what it actually built during review, not before

"Add a Rust command export_notes(format: ExportFormat) -> Result<String, String> that serializes the current notes to JSON or Markdown. Use the dialog plugin's save-file picker on the frontend to choose the destination, then pass that path to the command. Scope any fs capability to the exact path returned by the dialog, not a broad directory grant."

  • Command name, signature, and return type are already decided — the agent fills in the body
  • The capability boundary is stated as a constraint, not left to the agent's judgment
  • Frontend/Rust split is explicit: dialog picks the path, Rust does the writing
  • Review has one job: does the draft match this contract, not "what did it decide to build"

Prompt patterns that keep the boundary explicit

Three verbs cover most Tauri asks, and each one names the boundary it's supposed to cross — use them as sentence stems rather than reinventing phrasing each time:

StemWhat it scopesExample
Add a command that…One Rust function, one contract, one place it gets called from"Add a command get_file_metadata(path: String) that returns size, modified time, and MIME guess as a struct, erroring if the path doesn't exist."
Emit an event when…A pub/sub notification, decoupled from any one caller"Emit a copy://progress event with bytes-copied and total every 250ms during a long file copy, so any open window can show a progress bar."
Restrict this plugin to…A capability grant, narrowed to exactly what's needed"Restrict the fs plugin's capability to read-only access under $APPDATA/exports, not the whole filesystem, and only for the main window."

Ask your agent

Add a Rust command named get_file_metadata that takes a file path as a String and returns a struct with size in bytes, last-modified as an ISO timestamp, and a best-guess MIME type — Result, erroring with a readable message if the path doesn't exist or can't be read. Register it in the invoke_handler. Don't add any new capability grants — this command only needs to read metadata on a path the frontend already has permission to reach.

Ask your agent

When the copy_large_file command runs, emit a copy://progress event roughly every 250ms with { copied: u64, total: u64 } as the payload, and a final copy://done event with the destination path when it finishes or copy://failed with an error message if it doesn't. Don't block the main thread — run the copy on a background task and use AppHandle::emit for the events. Leave the frontend listener wiring for the next task.

Ask your agent

Restrict the fs plugin's capability for the main window to fs:allow-read-file and fs:allow-write-file scoped only to paths under $APPDATA/exports — remove any broader fs:default or unscoped grant currently in capabilities/. The app should not be able to read or write anywhere else on disk through this plugin.

The review checklist

Reading agent-generated Tauri code well means checking a short, specific list — not re-deriving the architecture from scratch. These are the failures that compile clean, run fine in tauri dev, and only bite in production or in a security review:

Capabilities: least privilege?

Does the diff grant exactly the permission identifiers the feature needs — not fs:default, not an unscoped shell:allow-execute — and only to the window that needs it?

Registration: is it wired in?

A new command that's never added to tauri::generate_handler![] compiles fine and fails silently at invoke time — no compiler error tells you.

Naming across the boundary?

Rust is snake_case, JS is camelCase — check argument structs carry #[serde(rename_all = "camelCase")] or the two sides silently disagree on field names.

Error handling: readable, not unwrapped?

Does a fallible command return Result<T, E> with a message a UI can show, rather than .unwrap()/.expect() that panics the whole backend on bad input?

State & async: not blocking the loop?

Shared state goes through tauri::State with a Mutex/RwLock, and slow work is async or spawned — not a long synchronous call sitting on the event loop.

Version drift: v2 idioms, not v1?

Watch for imports from @tauri-apps/api/tauri (v1) instead of @tauri-apps/api/core (v2) — a common tell that the agent's training data skewed to an older Tauri.

Gotcha An unregistered command is the single most common silent failure in agent-written Tauri code. The Rust function compiles, #[tauri::command] looks correct, and the frontend's invoke("do_thing") call simply rejects at runtime with "command not found" — because nobody added do_thing to tauri::generate_handler![...] inside .invoke_handler(...). Nothing about this shows up at compile time or in a type check; it only shows up when you actually click the button.
Key idea Treat every new line in a capabilities/*.json file as its own code-review event, the way you'd review a new database grant or an IAM policy change — not as boilerplate the agent "just needs" to make the feature work. Chapter 8's default-deny model means a capability grant is the one artifact in a Tauri diff that expands what the running app can do to the host machine.

Iterate outward from a minimal, verifiable core

The fastest path with an agent is rarely "describe the whole feature and let it run." Ask for a minimal slice, confirm the architecture holds, then widen the ask — each step small enough that a bad capability grant or a missed registration is obvious in the diff instead of buried in it.

  1. One window, one command. Get a single round trip working end to end — frontend calls, Rust responds — before adding a second command or a plugin.
  2. Add capabilities one at a time, by name. Each new OS capability the feature needs is its own small ask and its own review, not a bundle granted up front "in case."
  3. Reach for a plugin or sidecar only once the need is real (Chapter 9's decision framework) — not because the agent suggested one as a shortcut.
  4. Wire up build, CI, and distribution last (Chapter 10), once the feature's shape has stopped changing — churn there is expensive to keep re-verifying.
In practice A two-person team building a local-first note-taking app has their agent draft each new command against a one-page spec comment above the function — inputs, error cases, which capability it needs — before writing the implementation. The spec comment becomes the review checklist for that diff: if the implementation doesn't match what the comment promised, it goes back for a redraft against the same spec, not a fresh attempt from a new prompt.

The most useful thing you can do for an agent working on a Tauri codebase is say what layer you mean every time. "Fix the export bug" is ambiguous between a Rust panic and a frontend state bug; "the export_notes command is returning the wrong error message when the path doesn't exist" is not. The specificity that feels like over-explaining is exactly what keeps an agent from quietly widening a capability grant to route around a problem it doesn't understand.

The takeaway

An agent is very good at typing out a command, an event, or a capability file once you've decided its shape — and prone to guessing across exactly the boundaries this book has spent ten chapters naming when you haven't. Brief in terms of layer, contract, and capability; let three sentence stems — add a command that, emit an event when, restrict this plugin to — carry most of that precision; and run every draft through the same short checklist before it merges. Chapter 12 puts this loop to work on one small, complete feature, end to end.
