Chapter 08

A Powerful Webview You Must Constrain On Purpose

Security Model and Capabilities

Chapter 3 introduced the three words — capability, permission, scope — and promised this chapter would go deep on them. Here's the deep version: why the model exists at all, how the three pieces fit together into one enforceable rule per window, and the handful of patterns that keep an agent (or a tired version of you at 11pm) from quietly granting more than a feature actually needs.

Why this exists: the webview is not on your side

Your Rust core has real power — it can read files, spawn processes, talk to the network unfiltered, touch OS APIs. Your webview renders whatever HTML, CSS, and JavaScript your frontend build produces — which includes every npm dependency you didn't audit, every string of user content you render without escaping, every deep link or clipboard paste that ends up in the DOM. invoke() is the one bridge between those two worlds, so the entire security model boils down to one question asked at every crossing: should whatever is currently running in this webview be allowed to call this command, with these arguments, right now? Tauri answers it structurally, not by trusting the frontend to behave.

Key idea Tauri 1 exposed every registered command to every window by default — you opted out of access. Tauri 2 flipped it to default-deny: a window can call nothing until a capability says otherwise. An unclaimed window — one no capability names — can reach zero commands, full stop. That inversion is the whole model in one sentence.

Three layers, one job each

Capability, permission, and scope aren't three alternative ways to say the same thing — each answers a different question, and together they answer "who, what, and how narrowly."

LayerQuestion it answersLives where
CapabilityWhich windows/webviews get this bundle of access?A JSON/TOML file under src-tauri/capabilities/
PermissionWhich command(s) does this toggle on?Named identifier inside a capability's permissions array
ScopeAccess to what data, exactly — which paths, URLs, programs?Parameters attached to a scoped permission
 
{
  "$schema": "../gen/schemas/desktop-schema.json",
  "identifier": "main-capability",
  "windows": ["main"],
  "permissions": [
    "core:default",
    "dialog:default",
    { "identifier": "fs:allow-read-file", "allow": [{ "path": "$APPDATA/**" }] }
  ]
}

Save that as src-tauri/capabilities/main.json and it's picked up automatically (or reference it by identifier from tauri.conf.json if you want to be explicit). A window's effective access is the union of every capability that names it — you can split concerns across several small capability files instead of one growing monolith, which is usually the more maintainable shape once an app has more than two or three windows.

Ask your agent

Add a capability file scoping fs:allow-read-file and fs:allow-write-file to only the "$APPDATA/notes/**" path, and grant it to the "main" window only. Do not add core:default's broader permissions beyond what dialog and notification already need, and do not touch the "settings" window's capabilities — it has no filesystem needs.

Deny always wins

Permissions and scopes can carry explicit deny lists alongside their allow lists, and when a request matches both, deny wins — no exceptions, no ordering tricks. That precedence rule is what makes it safe to compose several plugin "default" permission sets (which tend to be generous) with a narrow, app-specific deny you bolt on top: you can grant a plugin's whole default bundle and still carve out the one path or command you never want reachable, without worrying about which permission was declared first.

In practice A password manager's main window gets fs access scoped to its own encrypted vault directory only — never the user's home folder. A local-first notes app grants shell:allow-execute for exactly one program name ("open in default editor") with fixed, non-user-controlled arguments, never a general "run this string as a command" capability. Both are the same move: grant the narrowest thing that makes the feature work, not the plugin's whole surface.

Scope to the window that actually needs it

A multi-window app rarely needs every window to have the same access. A settings window that only flips booleans and strings in a config store has no business holding filesystem or shell permissions just because the main window needs them — and because capabilities are addressed by window label, keeping that boundary is a one-line difference in the windows array, not a redesign.

One capability, every window, the plugin's whole default set. Fast to write, easy to reach for under deadline pressure.

  • "windows": ["main", "settings", "about"]
  • "permissions": ["fs:default", "shell:default"]
  • Every window can read/write anywhere the OS lets the process go, and can shell out arbitrarily
  • A bug or an injected script in any window can act on the whole filesystem

One capability per window, scoped to what that window's features actually touch. A few more minutes to write, once, per feature.

  • "windows": ["main"], a second file with "windows": ["settings"]
  • fs:allow-read-file / fs:allow-write-file scoped to $APPDATA/notes/**
  • No shell permission at all — this app never shells out
  • A compromised "settings" window can't touch a single file on disk

Content Security Policy: constraining the page itself

Capabilities govern the IPC bridge — what invoke() can reach. The Content-Security-Policy, set under app.security.csp in tauri.conf.json, governs the page itself: which script and style sources the webview is even willing to execute or load. It's the same CSP header the web platform has always had, but Tauri makes it load-bearing by default and does the tedious part for you — at build time it hashes your bundled local scripts and mints a cryptographic nonce for the rest, then appends those to the policy automatically, so you only have to declare what's unique to your app (a font CDN you actually use, a WebSocket endpoint you actually connect to). 

{
  "app": {
    "security": {
      "csp": "default-src 'self'; connect-src ipc: http://ipc.localhost https://api.example.com"
    }
  }
}
Gotcha A CSP restricts what the page can do even after something goes wrong — a dependency with an injected script, a rendered piece of unsanitized user content. Setting "csp": null "to stop fighting it in dev" and forgetting to restore it before shipping means an XSS that would otherwise be boxed in can now load arbitrary remote scripts. Keep a real policy in every build profile you actually ship.

Ask your agent

Set app.security.csp in tauri.conf.json so the app can only load scripts and styles bundled with it plus fonts from our existing self-hosted assets directory, and can only make network connections to api.example.com and the Tauri IPC endpoint. Don't add wildcard script-src or unsafe-inline — if a library genuinely needs it, tell me which one and why before adding it.

A capability opens a door; the command still has to check what's behind it

Scopes constrain the shape of an allowed request, not every value a caller might try inside it. An fs scope of $APPDATA/notes/** stops a call aimed outside that directory outright — but a hand-rolled command that builds its own paths from arguments (rather than going through a scoped plugin command) has to do that same check itself, because nothing upstream is doing it for a command Tauri doesn't know is filesystem-shaped. Path traversal ("../../../etc/passwd" smuggled into what looked like a filename), unbounded string lengths, and un-typed serde_json::Value blobs are the usual way a capability-respecting app still gets hurt: the door was the right door, nobody checked what walked through it. 

#[tauri::command]
fn read_note(app: tauri::AppHandle, name: String) -> Result<String, String> {
    // Never trust `name` to already be safe just because a scope exists —
    // reject path separators before it ever touches the filesystem.
    if name.contains('/') || name.contains("..") {
        return Err("invalid note name".into());
    }
    let dir = app.path().app_data_dir().map_err(|e| e.to_string())?;
    std::fs::read_to_string(dir.join("notes").join(name)).map_err(|e| e.to_string())
}

Deciding what a new feature needs

You're about to…Reach for
Let one window use a plugin's whole safe surface (dialog, notification) That plugin's "<plugin>:default" permission set
Let a window touch files, network hosts, or shell out Individual allow-* permissions with an explicit scope, not *:default
Give two windows very different access levels Two capability files, each with its own narrow windows array
Ship the same capability set differently per OS (e.g. no shell on mobile) The capability's platforms array
Accept structured data from the frontend inside an allowed scope Still validate it in the Rust command — the scope isn't a substitute for that

Quick glossary

Capability

A JSON/TOML file that names windows/webviews and lists the permissions granted to them.

Permission

A named toggle for one command or a bundle of them — plugin:name, or an unprefixed name for app-defined commands.

Scope

Parameters narrowing a permission — which paths, URLs, or program names it actually covers.

Default-deny

A window unnamed by any capability can call zero commands. Access must be granted, never merely un-denied.

CSP

app.security.csp in tauri.conf.json — constrains what the page can load/execute, nonced and hashed automatically for bundled assets.

Least privilege

Grant the narrowest permission and scope that makes a feature work, to only the window that needs it — not the plugin's whole default set everywhere.

A mix-up worth catching in review: asked to "let the frontend read a file," an agent will often reach for the plugin's whole fs:default permission set on every window, because it's the first thing that makes the error go away. That's a broad grant standing in for a narrow one. Push back for the specific allow-read-file permission, a scope limited to the directory the feature actually needs, and a windows array limited to the window that asked for it — three small edits that turn "this window can touch the filesystem" into "this window can read its own notes, and nothing else."

The takeaway

Default-deny, three layers (capability names the window, permission names the command, scope names the data), deny-always-wins precedence, and a CSP that boxes in the page itself — that's the model. Chapter 9 is where most new permissions actually get added in practice, since installing a plugin and granting it access are usually the same afternoon's work; Chapter 11's review checklist comes back to this chapter directly when it asks "does the agent's diff respect the security model."
