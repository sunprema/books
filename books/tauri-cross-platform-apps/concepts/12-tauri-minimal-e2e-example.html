Chapter 12

One Small Feature, Every Concept, End to End

A Minimal End-to-End Example

Eleven chapters of vocabulary condense into one small feature here: a file inspector. The user clicks a button, picks a file from the OS's native picker, and sees its size, modified time, and kind — nothing more. It's deliberately tiny, because the point isn't the feature, it's watching every seam from Chapters 3 through 11 — a plugin, a command, a capability, an invoke() round trip — show up in one walkthrough small enough to hold in your head at once.

The shape of the feature

Four beats, two round trips, zero events — this feature never needs to announce anything over time, so Chapter 7's decision table already tells you it's commands all the way down: (1) a button asks the dialog plugin to show a native file picker, (2) the frontend gets back a path (or nothing, if the user cancelled), (3) the frontend hands that path to a Rust command, and (4) Rust reads the file's metadata and returns it as structured data the frontend renders. Each arrow below is a full request/response round trip — the frontend asks a question and waits for exactly one answer, twice.

Why this example A file inspector touches a plugin (dialog), a hand-written command (get_file_metadata), a capability grant scoped to exactly one thing, and the plain invoke/response pattern — without needing events, a sidecar, or shared state. It's the smallest slice that still exercises the whole architecture, which is exactly why Chapter 11's "start minimal, verify, widen" advice uses it as the reference shape.

Step 1 — wire the "Choose File" button to the dialog plugin

This step never touches Rust code you write yourself — the dialog plugin's open() is a command the plugin already registered for you. The only jobs here are the button, the frontend state, and the capability that allows this window to use the dialog plugin at all.

Ask your agent

Add the dialog plugin (@tauri-apps/plugin-dialog on the frontend, tauri-plugin-dialog in Cargo.toml, registered with .plugin(tauri_plugin_dialog::init()) in lib.rs). Add a "Choose File" button that calls open({ multiple: false, directory: false }) from '@tauri-apps/plugin-dialog' and stores the returned path in component state. The user can cancel the picker — open() then resolves to null; leave the existing state untouched in that case rather than treating it as an error. Grant this window the dialog:default capability.

import { open } from '@tauri-apps/plugin-dialog';

// pseudo-snippet — the shape, not the whole component
async function chooseFile() {
  const picked = await open({ multiple: false, directory: false });
  if (picked) setPath(picked);  // null on cancel — state stays as it was
}

Expect

Clicking the button opens the OS's native file picker; cancelling does nothing observable; picking a file lands its absolute path in frontend state with no Rust call yet.

Check

A capabilities/*.json file grants this window dialog:default (or a narrower dialog:allow-open); a null return from a cancelled picker isn't routed through an error path.

Step 2 — write the Rust command that reads the file

This half never touches the frontend at all — it's a self-contained Rust function that takes a path and returns structured data or a readable error. Writing it before wiring the invoke call means you can reason about (and even test) the Rust side in isolation, exactly like Chapter 6 recommended.

Ask your agent

Add a Rust command get_file_metadata(path: String) -> Result using std::fs::metadata. FileMetadata should carry size in bytes, last-modified as an ISO-8601 string, and a best-guess kind (file extension or "unknown") — derive Serialize and use #[serde(rename_all = "camelCase")] so the frontend sees camelCase fields. Return a readable error string if the path doesn't exist or can't be read, not a panic. Register it in tauri::generate_handler![]. Don't add any new capability — this command only reads metadata on a path the frontend already has because the dialog plugin returned it.

#[derive(serde::Serialize)]
#[serde(rename_all = "camelCase")]
struct FileMetadata { size: u64, last_modified: String, kind: String }

#[tauri::command]
fn get_file_metadata(path: String) -> Result<FileMetadata, String> {
    let meta = std::fs::metadata(&path).map_err(|e| e.to_string())?;
    // ...size, last_modified, kind assembled from `meta` and the path's extension
    Ok(FileMetadata { /* ... */ })
}
// tauri::generate_handler![get_file_metadata, /* ...other commands */]

Expect

A plain #[tauri::command] function plus a Serialize struct — it compiles and could be unit-tested without a running webview.

Check

It's actually inside generate_handler![] (an unregistered command fails silently at invoke time, not at compile time — Chapter 11's most common gotcha); no .unwrap() on a fallible read; camelCase rename is present.

Step 3 — call the command and render what comes back

The last wire connects the two halves: when the stored path changes, invoke the command and show whatever it resolves — or rejects — with.

Ask your agent

When the picked file path changes, call invoke('get_file_metadata', { path }) from '@tauri-apps/api/core' and store the result in state. Render size, last-modified, and kind in a small results panel below the button. If the call rejects — e.g. the file was deleted between picking and invoking — show the error message inline in that panel instead of leaving the UI stuck or throwing unhandled.

import { invoke } from '@tauri-apps/api/core';

// pseudo-snippet — runs whenever `path` changes
try {
  const meta = await invoke('get_file_metadata', { path });
  setResult(meta);           // { size, lastModified, kind } — camelCase, matches the Rust rename
} catch (err) {
  setResult({ error: String(err) });   // rejection, not a crash
}

Expect

Picking a file makes a results panel appear in place, no reload; a file that vanishes between pick and invoke shows the Rust-side error text, not a blank screen.

Check

The argument key (path) matches the Rust parameter name; the invoke() call is wrapped so a rejection updates state instead of throwing; no broader filesystem capability was added — the path still only ever came from the dialog plugin.

Gotcha If the results panel renders every field as undefined, it's almost always the camelCase/snake_case seam from Chapter 7's checklist: the Rust struct returned last_modified but the frontend destructures lastModified because #[serde(rename_all = "camelCase")] is missing. It compiles, it runs, and it quietly shows blanks — nothing errors.

Verifying the whole loop, not just each step

Each step above checks out in isolation before you're done — but the feature isn't proven until you run it end to end in tauri dev and walk the failure paths on purpose, not just the happy one:

  1. Happy path. Click, pick a real file, see correct size/modified/kind rendered.
  2. Cancel the picker. Nothing changes — no error, no empty results panel flashing in.
  3. Pick, then delete the file before the panel updates (or pick something already gone) — the Rust error string should surface in the UI, not a silent freeze.
  4. Restart tauri dev after a Rust-side change. Frontend edits hot-reload; a new or changed #[tauri::command] needs the Rust side rebuilt before invoke() can see it (Chapter 10).
Key idea Every real Tauri feature is this same shape at some scale: a capability-scoped entry point on the frontend, a Rust command that does the trusted work, and a render step that handles both the answer and the error. Once this four-beat loop is boring to you, "add export," "add a settings panel," "add a background sync" are all the same loop with different verbs.

Notice what this walkthrough didn't need: no shared tauri::State, no events, no sidecar, no second window. Resist the urge to add any of those preemptively — Chapter 11's "iterate outward from a minimal, verifiable core" means this exact four-step loop is the minimal core, and the next feature is a reason to widen it, not a reason to have built it in on day one.

The takeaway

A file inspector this small still exercises a plugin, a hand-written command, a scoped capability, and the plain request/response round trip twice over — which is the point: almost every Tauri feature you'll ever brief an agent to build is this same loop, wearing a different verb. Chapter 13 closes the book with a glossary, decision checklists, and a resources list to keep nearby while you build the next one.
