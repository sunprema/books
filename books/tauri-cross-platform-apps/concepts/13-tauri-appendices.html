Glossary

Every term below got a full chapter earlier in the book; this is the one-sentence version to jog your memory, not to replace the chapter.

TermOne-sentence definitionCh.
App handleThe Rust-side handle to the running app — windows, state, and the ability to emit events, reachable from inside a command.3
Window / WebviewA window is the OS-level frame Tauri owns; the webview inside it is the OS's own browser engine rendering your frontend — two different layers, one visible box.2, 3
CommandA Rust function marked #[tauri::command] and registered in generate_handler!, callable from the frontend via invoke() — request/response, exactly one answer.3, 6, 7
EventA named message Rust emit()s and the frontend listen()s for (or vice versa) — publish/subscribe, zero-to-many deliveries, no reply expected.3, 7
PluginA packaged Rust crate (plus optional JS) that extends Tauri with its own commands and permissions, registered with .plugin(...) — runs inside Core.3, 9
SidecarA separate, pre-built binary bundled alongside your app and spawned as its own OS process — runs beside Core, not inside it.3, 9
CapabilityA named, versioned grant that lists exactly which permissions apply to exactly which windows — the v2 unit of "what this webview is allowed to touch."3, 8
PermissionA single named privilege (e.g. fs:allow-read-text-file) that a capability can include — defined by core Tauri or a plugin, granted by you.8
CSPThe Content-Security-Policy set in tauri.conf.json that restricts what the webview's own HTML/JS is allowed to load or execute — a second, independent lock from capabilities.4, 8
tauri.conf.jsonThe single source of truth for app identity, window defaults, security/CSP, and bundling — the file an agent should treat as intent, not boilerplate.4
src-tauri/The Rust backend crate: commands, plugin registration, capabilities, build config — everything with OS-level trust.2, 4
Dev server vs. production buildtauri dev runs your frontend's dev server with hot reload inside a live-reloading webview; a production build compiles Rust and bundles a static frontend into a platform installer.4, 10
tauri::StateShared, thread-safe application state (a DB pool, a config struct) registered once and injected into any command that asks for it by type.6
Sidecar handshakeThe stdio/IPC contract between Rust and a spawned sidecar process — JSON lines, length-prefixed frames, or a local socket, agreed on explicitly since Tauri doesn't supply one.9
Auto-updateA built-in mechanism that checks a signed manifest, downloads a new bundle, and installs it on next launch — opt-in, not automatic just from building the app.10

Decision checklist: Command or Event?

Chapter 7's full argument, compressed to a question you can ask in five seconds: does the frontend need exactly one answer to something it just asked, or does Rust need to say something on its own schedule?

The frontend asked a question and is waiting for one answer.

  • "Give me this file's metadata" — one request, one response, done
  • The work is fast enough (or you're fine awaiting it) that the caller can block on the result
  • Failure has a natural shape: the Result is Err, the frontend catches it
  • You want the type-checked, discoverable call shape — invoke('name', args) reads like a function call because it is one

Rust needs to speak without being asked a fresh question each time.

  • Progress updates during a long-running command ("62% converted")
  • Something happened that isn't a reply to anything — a file changed on disk, a background job finished, a hardware notification arrived
  • Zero, one, or many listeners might care, and Rust shouldn't need to know which
  • The natural pattern is frontend triggers → Rust works → Rust emits progress events → frontend updates UI (Ch. 7's default shape for anything longer than an instant)
Key idea These aren't mutually exclusive within one feature. The file inspector in Chapter 12 used two commands and zero events because both steps were quick request/response hops. A long-running import would start with one command (to kick it off) and then lean entirely on events for everything after — the command's job was just to say "started."

Decision checklist: Plugin or Sidecar?

Chapter 9's framework, compressed: is the capability something you'd write in Rust anyway, or does a working program already do it in another language?

OS integration, or logic you're comfortable writing (or already exists) in Rust.

  • Native dialogs, filesystem access, notifications, autostart, clipboard, geolocation, biometrics — the official plugins in tauri-apps/plugins-workspace already cover most of these
  • You want tight, low-latency access to Tauri's own state (windows, app handle, other commands)
  • It should be reusable across several apps, or need to reach the mobile targets too
  • Cost: real Rust (and maybe JS) to write and maintain, same as any command

An existing compiled tool, or a runtime you have no reason to reimplement.

  • A Python ML pipeline, a Go CLI, an ffmpeg binary — something battle-tested you'd otherwise spend weeks rewriting in Rust for no real benefit
  • It's fine to run as a separate, supervised OS process talking over stdio or a local socket
  • Cost: one binary per target platform bundled via bundle.externalBin, plus an explicit shell:allow-execute-style capability grant — and you own the IPC contract, since Tauri doesn't supply one

Ask your agent

Before starting a new Tauri feature, check the current @tauri-apps/api and plugin versions in package.json and Cargo.toml against the latest on v2.tauri.app — flag anything more than one minor version behind before writing code against APIs that may have moved.

That's the whole map: two layers, two kinds of traces between them, and a handful of decisions — command or event, plugin or sidecar, how wide a capability — that repeat on every feature you'll ever brief an agent to build. You don't need to memorize Tauri's API surface to direct it well; you need this shape in your head and the discipline to ask for exactly the access a feature needs, nothing wider.
