Chapter 04

Where Things Live, and What Decides Them

Project Structure and Configuration

Every Tauri repo has the same two-folder shape, and one file that ties it together. You don't need to memorize every key in that file — you need to know which area owns which decision, so that when you tell an agent "add a window" or "install a plugin" you already know which file it should be touching, and can tell at a glance whether the diff it hands back is in the right place.

The two-folder shape

A Tauri repo puts your frontend wherever a normal web project would — often src/ or whatever your framework's starter scaffolds — and adds exactly one sibling folder Tauri owns: src-tauri/. That folder is a normal Cargo crate with a few extra files layered on, and its presence at the project root is literally how the tauri CLI finds your app: it looks for src-tauri/tauri.conf.json and everything else follows from there. Nothing about your frontend folder is special to Tauri — React, Vue, Svelte, plain HTML, a static-site generator, it doesn't care, as long as it produces a folder of built assets at the end.

Where this shows up Teams adding Tauri to an existing web app usually don't move a single frontend file — they run tauri init at the repo root, point frontendDist at their existing build output, and get src-tauri/ as the only new folder in the diff.

Inside src-tauri/

Cargo.toml and Cargo.lock make it a normal Rust crate — your Rust dependencies (plugins, database drivers, whatever your commands need) go here, and the tauri and tauri-build crates must stay on the same minor version as your Tauri CLI. build.rs calls tauri_build::build(), Tauri's own build-script hook that wires code generation into cargo build — you'll rarely touch it by hand. icons/ holds the app icon at every size and format each platform's bundler expects (.icns, .ico, PNGs), usually generated once with tauri icon from a single source image.

Inside src/, the split between main.rs and lib.rs is deliberate, not boilerplate: main.rs is a thin desktop-only entry point that just calls into your app's run() function, while lib.rs holds the actual setup — the tauri::Builder chain, your registered commands, plugin initialization — behind a function marked #[cfg_attr(mobile, tauri::mobile_entry_point)]. That mark is what lets the exact same setup code run as the entry point on iOS/Android too, where there's no main.rs to call it. Practically: almost everything you'll ask an agent to add — a new command, a new plugin, a new piece of managed state — lands in lib.rs, not main.rs.

capabilities/ holds the permission files from Chapter 3 — it's a separate concern from everything else in this folder, and Tauri reads every capability file here by default. gen/ only appears once you initialize a mobile target (tauri android init / tauri ios init); it's the generated native Android Studio / Xcode project, treated as a build artifact rather than something you hand-edit.

Ask your agent

Register a new command called list_recent_projects that returns a Vec of recent project paths from app state. Add the function to lib.rs (not main.rs), mark it #[tauri::command], and add it to the generate_handler![...] list in the Builder chain so the frontend can call it with invoke().

tauri.conf.json, area by area

This is the file from the end of Chapter 3 — the one place almost every "configure X" request touches. Four areas matter most day to day:

AreaAnswersKey fields
identity What is this app, and who published it? productName, version, identifier
app What windows exist, and what can the webview load/run? app.windows[], app.security.csp
build Where's the frontend, dev and prod, and how do we build it? devUrl, frontendDist, beforeDevCommand, beforeBuildCommand
bundle What installers do we ship, and what rides along inside them? bundle.targets, bundle.icon, bundle.resources, bundle.externalBin

identity — identifier is a reverse-DNS string (com.yourcompany.appname) that platforms use as a stable key: macOS ties Keychain entries and code-signing to it, Windows uses it for install/uninstall registration, and your auto-updater (Chapter 10) uses it to recognize "this is the same app, apply the update." Treat it as effectively permanent once you've shipped a build.

app — app.windows is an array, one entry per window Tauri creates at startup (label, title, width/height, resizable, initial URL); adding a window your app creates later at runtime instead of on launch is a Rust-side WebviewWindowBuilder call, not a config entry. app.security.csp sets the webview's Content-Security-Policy — which scripts, styles, and connections the frontend is allowed to load at all — and Tauri v2 ships a restrictive default on purpose, so loosening it (e.g. to allow an external API's script) is a decision you make deliberately, not a default you fight.

build — devUrl is the URL tauri dev points the webview at (your frontend dev server, e.g. Vite on localhost:5173); frontendDist is the folder tauri build expects your frontend's production build to land in and packages from. beforeDevCommand and beforeBuildCommand are the shell commands Tauri runs for you first — typically your frontend's own dev and build scripts — so one tauri dev or tauri build drives both halves of the stack together.

bundle — bundle.targets picks which installer formats to produce per platform ("deb", "appimage", "msi", "nsis", "app", "dmg", or "all"); bundle.resources copies extra files (a bundled dataset, a config template) into the shipped app; bundle.externalBin is the sidecar list from Chapter 3 — separate binaries Tauri packages alongside your app.

Ask your agent

Add a new default window to tauri.conf.json's app.windows array: label "log-viewer", title "Logs", 640x480, resizable true, not visible on launch (we'll show it from a Rust command). Don't touch devUrl, frontendDist, or any capability file — this is a window-config change only.

Platform-specific overrides

Some settings only make sense on one platform — a Linux-only bundle target, a macOS-only entitlement, a Windows-only installer tweak. Rather than cramming platform conditionals into one file, Tauri looks for a sibling config file per platform — tauri.linux.conf.json, tauri.windows.conf.json, tauri.macos.conf.json, tauri.android.conf.json, tauri.ios.conf.json — and merges whichever one matches the current build target on top of the base tauri.conf.json, using a JSON Merge Patch (any key you set there overwrites the base value; keys you don't mention are left alone). The same mechanism, via the CLI's --config flag, is how teams build multiple flavors of one app (a "Pro" build with extra bundle resources, say) from one codebase without duplicating the whole config file.

Ask your agent

We only want the AppImage bundle target on Linux, not deb or rpm, without changing bundle targets for Windows or macOS. Create src-tauri/tauri.linux.conf.json with just bundle.targets set to ["appimage"], and leave the base tauri.conf.json's bundle config as the default for the other platforms.

Two config surfaces, one mental model

It's worth keeping tauri.conf.json and capabilities/*.json mentally separate, because agents (and people) mix them up: one describes the app you're building, the other describes what the running app is allowed to do. A prompt that says "let the app read files" is ambiguous between the two; a precise prompt names the file.

What are we building, and how do we package it. One file, read at build time.

  • App identity, window defaults, dev/build commands, bundle targets
  • Change it to add a window, point at a new dev server, or ship a new installer format
  • Has almost nothing to say about what the frontend is allowed to call

What is the running app allowed to do, and from which window. Multiple files, read at runtime.

  • Permissions (+ scopes) granted to specific window/webview labels
  • Change it to grant a plugin's command, narrow a filesystem scope, add a new window's access
  • Has nothing to say about installers, dev servers, or app identity

Ask your agent

Install the notification plugin and expose it to the frontend: add the tauri-plugin-notification crate to src-tauri/Cargo.toml, register it with .plugin(tauri_plugin_notification::init()) in lib.rs, install the matching @tauri-apps/plugin-notification JS package, and add a capability granting notification:default to the "main" window only. Don't add it to any other window's capabilities.

Key idea One frontend folder, one src-tauri/ crate. Inside it, lib.rs is where behavior gets added, tauri.conf.json is where the app's shape and build pipeline are configured, platform override files layer platform-only tweaks on top of it, and capabilities/ is the separate file that decides what any of it is allowed to touch at runtime.

Quick glossary

src-tauri/

The Rust crate Tauri owns; its presence at the project root is what the CLI looks for.

main.rs vs lib.rs

main.rs is a thin desktop entry point; lib.rs holds the real setup, shared with mobile.

tauri.conf.json

The central manifest: identity, windows, security defaults, build hooks, bundle targets.

identifier

The app's stable reverse-DNS key — code signing, updater, OS registration all key off it.

devUrl / frontendDist

Where the dev server lives, and where the production frontend build lands for packaging.

beforeDevCommand / beforeBuildCommand

Shell hooks Tauri runs first — usually your frontend's own dev/build scripts.

bundle.targets

Which installer formats to produce per platform — deb, appimage, msi, dmg, and more.

Platform override file

tauri.<platform>.conf.json, merged over the base config for that build target only.

A mix-up worth catching in review: an agent asked for "more permissions" will sometimes edit app.security.csp when it meant to edit a capability file, or vice versa. CSP controls what the webview can load and execute as content; capabilities control what Tauri commands it can call. Loosening the wrong one either does nothing for the feature you wanted, or opens a door you didn't mean to.

The takeaway

The shape never changes: your frontend folder plus one src-tauri/ crate, lib.rs for behavior, tauri.conf.json for the app's shape and build pipeline, platform files for the exceptions, capabilities/ for what's allowed. Once you can point at which of those five owns a given change, "ask your agent" prompts get short and specific instead of vague — and a diff that touches the wrong one is easy to spot in review.
