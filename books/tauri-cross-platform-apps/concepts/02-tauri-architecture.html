Chapter 02

The Big Picture, Zoomed In

High-Level Architecture of a Tauri App

Chapter 1 drew two boxes and two arrows. This chapter opens the right-hand box. A Tauri app is really one Rust process, called Core, that owns the window(s) and everything privileged, plus one or more WebView processes that render your frontend and nothing else. Every crossing between them — a command or an event — is routed through Core, which is exactly what makes the whole thing inspectable and securable.

Two layers, one boundary

Layer one is your frontend: whatever HTML/CSS/JS you'd ship to a browser — React, Svelte, Vue, plain JS, even a static site generator's output. Tauri doesn't care; it just needs a folder of built assets to hand to the webview. Layer two is the Rust backend, built on the tauri crate, which is where windows get created, files get read, and anything that talks to the operating system happens. The boundary between them is not a suggestion — it's enforced by process isolation, the same way your browser tab can't reach into your filesystem without going through an explicit browser API.

The process model: one Core, many WebViews

Under the hood, Tauri is a multi-process architecture, the same shape Electron and modern browsers use: a single Core process supervises one or more WebView processes, rather than one process trying to be both the UI renderer and the privileged backend. Core is written in Rust against the tauri-runtime trait, which is implemented for desktop by tauri-runtime-wry — itself built on two Tauri-maintained crates: WRY, which drives whichever webview engine the OS already ships (WebView2 on Windows, WKWebView on macOS/iOS, WebKitGTK on Linux, the system webview on Android), and TAO, which creates and manages the native windows and their OS event loop. Core never renders a pixel of your UI itself — it asks the OS's webview to do that, then owns everything around it: window lifecycle, the system tray, global app state, and every OS capability your app is allowed to use.

Key idea Anything that can touch the OS either runs inside the Core process (Core itself, or a plugin registered into it) or is an explicitly spawned, supervised sidecar. A WebView process can only ask — via a command or an event — it can never reach out on its own.

How a message actually crosses the boundary

When your frontend calls invoke("some_command", { args }), that call doesn't hit Rust directly — it becomes a request against a Tauri-internal ipc://localhost custom protocol (on Android, which can't read request bodies on custom protocols, it falls back to postMessage). Core intercepts that request, looks up the matching #[tauri::command] function, runs it, and serializes the return value back to the caller's JavaScript promise. Because every IPC message passes through this one chokepoint in Core, Tauri can validate, filter, or reject a call before it ever reaches your Rust code — that's the mechanism the capability system in Chapter 8 builds on. Events work the other direction: Core (or another window) calls emit, and every listener registered with listen on the target webview(s) gets called — no request required, which is why events fit progress updates and background notifications better than commands do. Chapter 7 goes deep on choosing between the two; here, just note that both are IPC, and both are funneled through the same Core-owned pipe.

One per app. Rust, compiled native, async runtime (Tokio) underneath.

  • Creates and owns every window and (optionally) system tray/menu
  • Holds global app state — settings, DB handles, anything shared across windows
  • The only side with a syscall boundary to files, network, shell, hardware
  • Routes and can inspect every IPC message before it's handled
  • Hosts plugins directly; spawns and supervises sidecars

One per window (and, on an unstable API, possibly more than one per window).

  • Renders your HTML/CSS/JS using the OS's own webview engine
  • Has no direct filesystem/network/shell access of its own
  • Talks to Core only via invoke() (commands) and listen() (events)
  • Can crash or hang without taking Core, or other windows, down with it
  • Framework-agnostic — Core doesn't know or care what's rendering inside

Where plugins and sidecars attach

Two different escape hatches extend what Core can do, and they attach at different points in this diagram. A plugin is Rust code (plus, often, a thin JS binding) that registers itself into the Core process at startup — it adds its own commands and events and runs in the same address space as your app's own Rust code. Official examples cover dialogs, filesystem access, notifications, HTTP, autostart, and more. A sidecar is the opposite: a separate executable — a Python script, an existing CLI tool, a heavy runtime you don't want to reimplement in Rust — that Core spawns as its own OS process and talks to over stdio or a local socket, the same way a shell script would shell out to another program. Chapter 9 covers the decision in depth; the architectural takeaway for now is just where each one lives:

LayerRuns asDirect OS access?
Frontend (webview)WebView process, per windowNo — only via commands/events

Multiple windows, and (soon) multiple webviews

Because Core, not any one window, is the source of truth, a single app can own several windows — a main window plus a settings window, say — and they're all just additional WebView processes Core supervises, able to emit events to each other through Core. Tauri also maintains a clean split between a window (the native OS frame) and a webview (the content surface inside it); as of Tauri 2, attaching more than one webview to a single window is possible via an unstable API, useful for things like a browser-style app with several panes, but it's still evolving — the one-webview-per-window shape is the stable, default mental model to design around today.

Ask your agent

Add a second Tauri window called "settings", built from its own frontend route, separate from the main window. When the user saves settings there, emit a settings-saved event with the new values; have the main window's Rust setup register a listener that updates in-memory app state when that event arrives. Keep window creation and the event wiring in Core — the frontend should not need to know a second window exists unless the event fires.

A gotcha worth internalizing early: Core runs your async runtime, and it's the same process handling every window's IPC traffic. A command handler that blocks synchronously — a long CPU loop, a blocking file read on a huge file — can stall IPC for windows that have nothing to do with that command. Prefer async fn commands, or hand genuinely heavy work to tauri::async_runtime::spawn, so one slow operation doesn't freeze the rest of the app. Chapter 6 covers this in more depth on the Rust side.

The takeaway

"Frontend talks to Rust via commands; Rust owns OS capabilities" from Chapter 1 is the mental model. This chapter is the mechanism underneath it: one Core process, written in Rust on top of tauri-runtime/WRY/TAO, supervising one or more WebView processes that render your UI and nothing else, with every crossing between them funneled through Core as a command or an event. Plugins extend Core from the inside; sidecars extend it from the outside, as separately supervised processes. Once this shape is solid in your head, most later chapters — terminology, project structure, security — are just naming and configuring the pieces already on this diagram.
