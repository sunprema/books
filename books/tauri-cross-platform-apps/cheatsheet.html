The dense version — for when you already have the mental model from the 13 chapters and just need the exact syntax, the decision table, or the CLI flag.

The 30-second summary

  1. Two layers: an untrusted webview (your frontend, any stack) and a trusted Rust Core (windows, OS access, lifecycle).
  2. Two traces cross the seam: command = request/response (invoke), event = publish/subscribe (emit/listen).
  3. Security is default-deny: a window gets zero commands until a capability names it, grants a permission, optionally narrowed by a scope.
  4. Extend Core with a plugin (Rust+JS, runs inside Core) or a sidecar (a separate bundled binary, runs beside it).
  5. Three modes, three guarantees: tauri dev (fast, unoptimized) → tauri build (release compile + bundle, one platform) → distribute (signed, reachable, updater-aware).
  6. Direct an agent by naming the exact layer, command/event, and permission/scope — never "just make it work."

Frontend syntax (@tauri-apps/api)

import { invoke } from '@tauri-apps/api/core';
import { listen, emit } from '@tauri-apps/api/event';

const result = await invoke('cmd_name', { arg1, arg2 });

const unlisten = await listen('evt-name', (e) => {
  // e.payload
});
unlisten();  // always unlisten when done

Rust syntax

#[tauri::command]
fn cmd_name(app: tauri::AppHandle, arg1: String) -> Result<T, String> { .. }

// register:
tauri::Builder::default()
  .invoke_handler(tauri::generate_handler![cmd_name])

// emit, from anywhere holding an AppHandle:
app.emit("evt-name", payload);
app.emit_to("window-label", "evt-name", payload);

Capability file skeleton

{
  "identifier": "main-capability",
  "windows": ["main"],
  "permissions": [
    "core:default", "dialog:default",
    { "identifier": "fs:allow-read-file",
      "allow": [{ "path": "$APPDATA/**" }] }
  ]
}

Save under src-tauri/capabilities/*.json — picked up automatically.

Permission & scope quick syntax

plugin:defaultthat plugin's whole safe default bundle
plugin:allow-xone specific action, on
plugin:deny-xone specific action, off — deny always wins
core:defaultcore (non-plugin) commands' default bundle
"allow"/"deny": [{..}]scope objects narrowing a permission to paths/URLs/programs
Unnamed windowreaches zero commands — default-deny

Security checklist

  • Default-deny: every window needs an explicit capability, nothing is implicit
  • Least privilege: narrow allow-* + scope, not *:default everywhere
  • A scope isn't validation — still check inputs (path traversal, length) inside the Rust command
  • Keep a real CSP in every shipped profile; don't leave dev's relaxed policy in place
  • Deny always wins over allow, at both the permission and scope level
  • Never commit TAURI_SIGNING_PRIVATE_KEY — CI secret only
