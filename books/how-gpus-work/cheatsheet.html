The Cheatsheet

The whole book, compressed for exam week. 30-second version GPUs = throughput machines: thousands of simple cores run one small program over huge independent data, in lockstep warps of 32, hiding memory latency with spare threads — and the real bottleneck is almost always memory, not math.

The vocabulary

TermMeaning
kernelfunction launched across many GPU threads (the loop body, minus the loop)
threadone execution of the kernel; handles one element
warp32 threads executing in lockstep (SIMT); the hardware's real unit of work
block≤1,024 threads on one SM; can share memory & __syncthreads()
gridall blocks of a launch; blocks must be independent → scales to any GPU
SMStreaming Multiprocessor — the "factory" (cores + Tensor Cores + shared mem + warp schedulers)
occupancyfraction of an SM's warp slots filled — fuel for latency hiding
divergencewarp threads taking different branches → paths serialized → idle lanes
coalescingneighboring threads reading neighboring addresses → few memory transactions
HBMstacked high-bandwidth DRAM next to the die (data-center); GDDR = gaming equivalent
Tensor Coreunit computing D = A×B + C on whole matrix tiles per clock
arithmetic intensitymath ops per byte moved — decides memory- vs compute-bound (roofline)
KV cacheper-token attention state kept in GPU memory during LLM generation

The memory pyramid (H100-era ballparks)

LevelSizeLatencyScope
Registers~256 KB/SM~1 cyper thread
Shared / L1~228 KB/SM~20–30 cyper block
L250–96 MB~200 cywhole chip
HBM/GDDR24–192 GB~400–800 cywhole chip
  • Fast-kernel recipe: coalesce global reads → stage tiles in shared memory → reuse many times.
  • Break-even on H100 ≈ ~300 math ops per byte. Below it, FLOPS don't matter.

CUDA in six lines

__global__ void add(float* a, float* b, float* c, int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) c[i] = a[i] + b[i];
}
// launch: grid covers N, 256 threads/block
add<<<(N+255)/256, 256>>>(a, b, c, N);
  • Flow: cudaMalloc → cudaMemcpy(H→D) → kernel → cudaMemcpy(D→H). Move data once, work a lot.
  • Loops inside a thread are fine; the loop over the data becomes the grid.

Precision menu (Tensor Core era)

FormatBitsUse
FP3232science, accumulators, master weights
TF3219free training speedup on Ampere+ matmuls
FP16 / BF1616standard training compute (BF16 = FP32's range, less detail)
FP88frontier training + fast inference (Hopper+)
FP4 / INT44quantized inference (Blackwell hardware)
  • Rule: multiply small, accumulate big. Each halving ≈ 2× speed, ½ memory.
  • Spec-sheet trap: "with sparsity" doubles the marketing number.

LLM speed napkin math

  • Decode tokens/sec ≤ bandwidth ÷ model bytes. 70B @ 8-bit on H100: 3350/70 ≈ 48 tok/s.
  • Prefill = batch of prompt tokens = compute-bound (Tensor Cores). Decode = one token at a time = memory-bound.
  • Fixes: batching (reuse each weight-load), quantization (fewer bytes), speculative decoding (verify 5 guesses per load), FlashAttention (keep attention on-chip).
  • Buying for local LLMs? Rank specs: 1. memory capacity, 2. memory bandwidth, 3. FLOPS.

"Will it be fast on a GPU?" checklist

  • ☑ Same operation over many elements?
  • ☑ Elements independent (no step-N-needs-N−1 chains)?
  • ☑ Enough work — tens of thousands of threads minimum?
  • ☑ Data can stay on the GPU (copies amortized)?
  • ☑ Neighboring threads touch neighboring memory?
  • ☑ Branches mostly agree within a warp?
  • Any ☒ → profile before celebrating; two ☒ → maybe keep it on the CPU.

Timeline in one breath

  • 1999 GeForce 256, "the first GPU" · 2006–07 unified shaders + CUDA — GPUs go general
  • 2012 AlexNet on 2 gaming cards — deep learning ignites · 2017 Tensor Cores (Volta) + Transformers
  • 2018 RT cores — ray tracing · 2020 A100 · 2022 H100 + FP8 — the LLM workhorse
  • 2024–26 Blackwell: chiplets, FP4, NVL72 racks, AI factories, petaflop desktops
