Shipping: Modules, Tooling & One Binary

Code that runs on your laptop isn't done — done is "running on a server, reachable by users." Here's the part Go makes almost suspiciously easy, and the last bit of vocabulary you need.

A project is a module

Every Go project is a module: a directory with a go.mod file at its root that names the project and records its dependencies. You make one with a single command:

shellgo mod init github.com/you/bookservice

That writes a go.mod like this — the entire dependency manifest for the whole project:

go.modmodule github.com/you/bookservice

go 1.26

require github.com/some/dep v1.4.2

When the agent adds an import, you run go mod tidy and Go figures out exactly which dependencies (and which versions) are needed, writing them here plus a go.sum file of cryptographic checksums. That go.sum is your supply-chain seatbelt: it guarantees you get byte-for-byte the same dependency code every build, so nobody can swap a package out from under you.

🧭
Two files, total

That's the whole dependency story: go.mod (what you depend on) and go.sum (the checksums). No lockfile zoo, no separate package manager to install — go is the package manager, the build tool, the test runner, and the formatter, all one binary. Tell your agent to run go mod tidy after changing imports and to commit both files.

The payoff: one self-contained binary

Now the moment that makes Go people grin. To build your program for production:

shellgo build -o bookservice ./...

Out pops a single executable file. Not a folder of scripts plus an interpreter; not a node_modules the size of a small moon. One file, with the Go runtime and every dependency statically linked inside it. To deploy, you copy that one file to the server and run it. There is nothing to install on the target — no Go, no libraries, no runtime. This is why Go conquered CLI tools and cloud infrastructure (Chapter 1): "ship one binary" is the whole deployment story.

💡
Why does a vibe coder care about this so much?

Because deployment is where a lot of AI-built projects quietly die — "it worked in the chat, but I can't get it running anywhere." Go collapses that whole class of problem. No dependency-version mismatch on the server, no "works on my machine," no runtime to provision. The thing the agent built is the thing you run. Fewer moving parts between code and production means fewer places for the project to fall apart after the vibes wear off.

Cross-compiling: build for anywhere, from anywhere

Here's a genuinely magic trick. Go can build a binary for a different operating system and CPU than the one you're on, with two environment variables and no special toolchain:

shell · build a Linux server binary from your MacGOOS=linux GOARCH=amd64 go build -o bookservice ./...

# a Windows .exe, an ARM Raspberry Pi build, etc. — same idea
GOOS=windows GOARCH=amd64 go build -o app.exe ./...

You develop on a Mac and ship a Linux binary to your cloud box without ever leaving your laptop. This is also why Go binaries drop so cleanly into tiny Docker images — you can put a single static binary in a scratch or distroless base and get a container measured in megabytes.

CommandWhat it does

⚠️
One asterisk on "fully static"

Go binaries are static by default — until you use a package that calls into C (via cgo), which a few do (some database drivers, for instance). If you hit a deployment that complains about a missing system library, that's the usual culprit; building with CGO_ENABLED=0 forces a pure-Go static build. Worth knowing the word cgo so the error message isn't a mystery — and so you can tell the agent to avoid cgo dependencies when a clean static binary matters.

🔑
Key idea

A Go project is a module (go.mod + go.sum), and one go build turns it into a single static binary with everything inside. Cross-compile for any OS/arch with GOOS/GOARCH. Deployment is "copy one file and run it" — which is exactly why Go is the language of CLIs, containers, and the cloud.

You've got the whole loop now

Step back and look at the arc. You can read Go (Ch. 2), reason about its concurrency (Ch. 3), its errors (Ch. 4), and its types (Ch. 5); you can build a real service with it (Ch. 6); you can drive an agent to write it well (Ch. 7); and now you can ship it. That's the complete cycle of a vibe coder who chose the right tool: a small, honest language whose code is easy to verify, whose feedback loop is fast, and whose output is one file you can run anywhere. Go forth and build something — and let the graders, not your eyeballs, do the boring part.

The thirty-second version

  • A project is a module: go mod init creates go.mod; go mod tidy syncs deps; commit go.mod + go.sum.
  • go build emits one static binary with the runtime and all deps inside — copy it to the server and run.
  • Cross-compile for any platform with GOOS/GOARCH; perfect for tiny Docker images.
  • go is the package manager, builder, tester, and formatter — one tool, no extras to install.
  • Watch for cgo if you need a guaranteed-static build; CGO_ENABLED=0 forces pure Go.
