A Real Web API — Standard Library Only

Time to make something. We'll build a small JSON HTTP service — the bread and butter of backend work — using zero third-party packages. This is where Go stops being theory.

Why no framework?

Coming from other ecosystems you'd reflexively reach for a web framework — Express, Flask, Rails. In Go you usually don't need one, and that's not bravado: HTTP serving, routing, and JSON are all in the standard library, and since Go 1.22 the built-in router learned to match methods and path parameters. Fewer dependencies means less to install, less that can break, less for an agent to get wrong, and a single binary at the end with nothing to npm install on the server. Let's lean into it.

The whole server, one file

Here's a complete API for a tiny "books" service — list all, and fetch one by id. Read it once top to bottom; we'll dissect it after.

go · main.go — a complete JSON API, no dependenciespackage main

import (
    "encoding/json"
    "log"
    "net/http"
)

type Book struct {
    ID    string `json:"id"`
    Title string `json:"title"`
}

var books = []Book{
    {ID: "1", Title: "The Go Programming Language"},
    {ID: "2", Title: "Learning Go"},
}

func main() {
    mux := http.NewServeMux()

    // GET /books  → list them all as JSON
    mux.HandleFunc("GET /books", func(w http.ResponseWriter, r *http.Request) {
        writeJSON(w, http.StatusOK, books)
    })

    // GET /books/{id}  → one book, by path parameter
    mux.HandleFunc("GET /books/{id}", func(w http.ResponseWriter, r *http.Request) {
        id := r.PathValue("id")        // new in Go 1.22
        for _, b := range books {
            if b.ID == id {
                writeJSON(w, http.StatusOK, b)
                return
            }
        }
        http.Error(w, "book not found", http.StatusNotFound)
    })

    log.Println("listening on :8080")
    log.Fatal(http.ListenAndServe(":8080", mux))
}

// one small helper, reused everywhere
func writeJSON(w http.ResponseWriter, status int, v any) {
    w.Header().Set("Content-Type", "application/json")
    w.WriteHeader(status)
    json.NewEncoder(w).Encode(v)
}

Run it with go run main.go, then in another terminal: curl localhost:8080/books. You get JSON back. You just wrote a production-shaped web service, and your go.mod has not a single dependency in it.

Let's dissect the four key moves

1 · The handler signature is the whole API

Every HTTP handler in Go has the same shape: func(w http.ResponseWriter, r *http.Request). The request r is what came in; you write your response into w. Recall Chapter 5 — http.ResponseWriter is an interface (an io.Writer, in fact), which is why writing JSON to it is the same act as writing to a file. One pattern, learned once.

2 · Routing with methods and path parameters

The string "GET /books/{id}" packs three things into one pattern: the HTTP method, the path, and a named wildcard {id}. Inside the handler, r.PathValue("id") pulls the matched value out. A request with the wrong method (say POST /books) automatically gets a 405 Method Not Allowed — you didn't write that logic; the router did.

🧭
Version note — this is genuinely new

Method matching and {id} path wildcards arrived in Go 1.22 (early 2024). Before that, everyone reached for a third-party router like chi or gorilla/mux for exactly this. An LLM trained partly on older code may still suggest those out of habit — when you don't need their extras, tell it "use the standard library net/http router with method patterns." Cleaner, dependency-free, and current.

3 · JSON via struct tags

Look at Title string `json:"title"`. That backtick string is a struct tag — metadata the encoding/json package reads to decide that the Go field Title becomes the JSON key title. json.NewEncoder(w).Encode(v) turns any Go value into JSON and streams it to the writer; json.Decoder does the reverse for incoming request bodies. Remember the capital-letter rule once more: only exported (capitalized) fields get encoded — a lowercase field is invisible to JSON. This trips up newcomers exactly once.

4 · Starting the server

http.ListenAndServe(":8080", mux) binds the port and serves forever, handing each incoming request to your router. And here's the quiet superpower from Chapter 3: the standard server runs each request in its own goroutine, automatically. Ten thousand simultaneous clients? Ten thousand cheap goroutines. You wrote straight-line, blocking-looking code, and got massive concurrency for free.

⚠️
What this toy skips (and a real one wouldn't)

To stay readable, this example omits things production needs: request timeouts (use a configured &http.Server{} with ReadTimeout/WriteTimeout rather than the bare helper), graceful shutdown via context, input validation, structured logging (log/slog), and a real datastore instead of a slice. When you vibe-code a service for real, name these explicitly — "add read/write timeouts, graceful shutdown, and slog" — so the agent builds them in from the start.

🔑
Key idea

A complete JSON web service in Go needs no framework: net/http serves and routes (with method + {id} patterns since 1.22), encoding/json marshals structs via tags, every request runs in its own goroutine, and the result is one dependency-free binary. This standard-library-first style is the most reliable thing to ask an agent for.

The thirty-second version

  • Handlers are all func(w http.ResponseWriter, r *http.Request) — read r, write into w.
  • Route with patterns like "GET /books/{id}"; read params with r.PathValue("id") (Go 1.22+).
  • JSON in/out via encoding/json + struct tags; only exported fields encode.
  • http.ListenAndServe serves every request in its own goroutine — free concurrency.
  • For real services add timeouts, graceful shutdown, validation, and log/slog — ask for them by name.
