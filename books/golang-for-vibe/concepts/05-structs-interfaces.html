Structs, Methods & Interfaces

Go has no classes, no inheritance — and people who arrive from Java keep waiting for the other shoe to drop. It never does. Here's the smaller, sturdier idea Go uses instead.

Structs: just a bag of fields

A struct is the most honest data type there is: a named bundle of fields, sitting next to each other in memory. No hidden machinery.

gotype Book struct {
    Title  string
    Author string
    Pages  int
}

b := Book{Title: "The Go Programming Language", Pages: 380}
fmt.Println(b.Title)   // field access with a dot

That's a struct. Most Go programs are, at bottom, structs being filled in, passed around, and read out. (Remember the capital-letter rule from Chapter 2: capitalized fields like Title are visible outside the package — which is exactly why JSON encoding can see them, as you'll find in Chapter 6.)

Methods: a function with a receiver

You attach behavior to a type by writing a function with a little extra: a receiver in parentheses before the name. That's the whole "method" concept.

go// (b Book) is the receiver — this method belongs to Book
func (b Book) Summary() string {
    return fmt.Sprintf("%s (%d pages)", b.Title, b.Pages)
}

fmt.Println(b.Summary())   // call it like a method

If a method needs to modify the struct, it takes a pointer receiver, (b *Book), so it edits the original instead of a copy. That's nearly the only place a beginner meets pointers in Go — and the agent will get it right far more reliably than you'd get it right by hand.

Interfaces: the quietly brilliant part

Now the idea that replaces inheritance. An interface is a list of method signatures — a description of behavior, with no data attached:

gotype Stringer interface {
    String() string
}

Here's the twist that surprises everyone: you never declare that a type "implements" an interface. There's no implements keyword. If a type happens to have all the methods an interface lists, it satisfies that interface — automatically, silently. This is called structural typing, or affectionately, duck typing that the compiler checks: if it walks like a duck and quacks like a duck, the compiler agrees it's a duck.

gotype Shape interface {
    Area() float64
}

type Circle struct { R float64 }
func (c Circle) Area() float64 { return 3.14159 * c.R * c.R }

type Square struct { Side float64 }
func (s Square) Area() float64 { return s.Side * s.Side }

// Both satisfy Shape automatically — no declaration needed.
shapes := []Shape{Circle{R: 2}, Square{Side: 3}}
for _, s := range shapes {
    fmt.Println(s.Area())   // calls the right one for each
}
💡
Why is "no implements keyword" a big deal?

Because it decouples the user of a behavior from the provider of it. You can define an interface that describes exactly what your code needs, and any existing type — even one from a library you don't control — automatically fits if it has the methods. Interfaces are defined where they're used, not where types are declared. This is why Go code composes so cleanly, and why it's so testable: to fake a dependency in a test, you just write a small struct with the right methods. No mocking framework required.

Two interfaces you'll meet immediately

  • error — from the last chapter. Any type with an Error() string method is an error. Now you see the whole picture: it's just structural typing.
  • io.Reader / io.Writer — the most famous interfaces in Go. A Reader is "anything with a Read method"; a Writer is "anything with a Write method." A file, a network socket, an in-memory buffer, an HTTP response — all are Writers. So a function that writes to an io.Writer works with all of them, unchanged. That's the payoff of small interfaces.
🧭
Composition over inheritance, literally

Instead of "a Manager is-a Employee" inheritance trees, Go embeds one struct inside another to reuse its fields and methods — composition you can see. The community proverb is "the bigger the interface, the weaker the abstraction": keep interfaces tiny (one or two methods) and they stay reusable. When you ask an agent for a design, nudge it toward small interfaces and composition — that's idiomatic Go, and it's what the training corpus is full of.

🔑
Key idea

Data lives in structs; behavior attaches via methods; abstraction comes from small interfaces that types satisfy implicitly, just by having the right methods. No classes, no inheritance, no implements — and as a bonus, trivially fakeable dependencies for testing.

The thirty-second version

  • A struct is a named bundle of fields. Most Go data is structs.
  • A method is a function with a receiver: func (b Book) Summary() string. Use a pointer receiver (b *Book) to mutate.
  • An interface lists methods. A type satisfies it automatically by having those methods — no implements.
  • Define small interfaces where they're used; error, io.Reader, io.Writer are the canonical examples.
  • Reuse via composition/embedding, not inheritance. Small interfaces make code testable without mocking frameworks.
