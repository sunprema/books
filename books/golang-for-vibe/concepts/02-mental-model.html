The Five-Minute Mental Model

You don't need to memorize syntax to vibe-code well — but you do need to be able to read what the agent hands you. Here's the whole shape of a Go program in one sitting.

The smallest complete program

Let's not theorize. Here is a real, runnable Go program — the entire thing:

go · main.gopackage main

import "fmt"

func main() {
    fmt.Println("Hello, vibe coder")
}

Save that as main.go and type go run main.go. It prints the line. Let's read it the way you'd read a sentence, left to right, because every piece is a real idea:

  • package main — every Go file belongs to a package. The special package named main says "this is a program you can run," not a library.
  • import "fmt" — pull in the standard library's fmt ("format") package for printing.
  • func main() — the function named main is where a runnable program starts. One per program.
  • fmt.Println(...) — call the Println function from the fmt package. The capital P is not decoration — read on.
🔑
The capital-letter rule (this one's load-bearing)

In Go, a name that starts with a Capital letter is exported — visible outside its package, public. A name starting lowercase is private to its package. That's the entire access-control system. No public/private keywords. Println is callable because it's capitalized; fmt's internal helpers are lowercase and you can't touch them.

Variables: three spellings, one idea

Go is statically typed — every value has a type the compiler knows. But it's not chatty about it; it infers types when it can.

govar name string = "Ada"   // explicit type
var age = 37             // type inferred as int
city := "London"          // short form — only inside functions

That last one, :=, is the workhorse you'll see everywhere: "make a new variable and figure out its type from the value." The types are still there and still checked — Go just spares you writing them when they're obvious.

⚠️
Go is strict in two endearing ways

An unused variable or an unused import is a compile error, not a warning. It feels nitpicky at first, but it's a gift to a vibe coder: dead code can't quietly accumulate, and an agent that imports something it didn't use gets told off immediately. The mess never compiles.

Functions can hand back more than one thing

This is the feature that quietly shapes all idiomatic Go. A function can return multiple values, and by deep convention the last one is an error:

gofunc divide(a, b float64) (float64, error) {
    if b == 0 {
        return 0, fmt.Errorf("cannot divide %v by zero", a)
    }
    return a / b, nil   // nil error means "all good"
}

result, err := divide(10, 2)

Notice a, b float64 — when consecutive parameters share a type, you write it once. We'll spend a whole chapter on that err (Chapter 4), because it's the heart of how Go thinks. For now just absorb the shape: here's the answer, and here's whether anything went wrong.

Control flow: deliberately few shapes

There is exactly one loop keyword — for — and it does every job. There's no while, no do. Fewer shapes, less to misremember.

gofor i := 0; i < 3; i++ {        // classic counted loop
    fmt.Println(i)
}

for n < 100 {                    // "while" — just omit the other parts
    n *= 2
}

for i, item := range items {      // range over a slice: index + value
    fmt.Println(i, item)
}

if score >= 60 {
    pass()
} else {
    retry()
}

The two collections you'll use constantly

Almost all data wrangling in Go runs through two built-in types: the slice (a growable list) and the map (a key→value dictionary).

go// slice: an ordered, growable list of strings
langs := []string{"Go", "Rust"}
langs = append(langs, "Zig")     // append returns a new slice header
fmt.Println(len(langs))            // 3

// map: keys to values
age := map[string]int{"Ada": 37}
age["Alan"] = 41
v, ok := age["Ada"]              // ok is false if the key is absent

That v, ok pattern — "the value, and a bool saying whether it was really there" — is the same multiple-return idea again. You'll see Go reuse a handful of patterns like this everywhere instead of inventing new ones. That repetition is exactly what makes it easy to read.

💡
Why so few features? Isn't that limiting?

It's a trade, made on purpose. Every feature a language doesn't have is a feature you never have to learn, debug, or argue about in code review — and that an agent can never misuse. Go's designers came from C++ and decided the discipline of "small and obvious" was worth more than the thrill of "powerful and subtle." For code you have to verify rather than savor, they were right.

The thirty-second version

  • Every file starts with package; the runnable one is package main with a func main().
  • Capitalized = public, lowercase = private. That's the whole visibility system.
  • := declares-and-infers inside functions; types are real and checked, just often unwritten.
  • Functions return multiple values; the last is usually an error. Internalize this shape.
  • for is the only loop. Slices ([]T) and maps (map[K]V) carry most of your data.
  • Unused variables/imports won't compile — the mess is caught at the door.
