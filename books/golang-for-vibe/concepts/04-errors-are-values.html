Errors Are Just Values

Newcomers groan at Go's error handling — "so verbose!" — and miss that the verbosity is the feature. Let's see why handling failure in the open is exactly what a vibe coder wants.

First, the thing Go refuses to do

In a lot of languages, when something fails you throw an exception. It flies up the call stack, invisible, until some catch somewhere — maybe — grabs it. The seductive part: your happy-path code looks clean. The treacherous part: you cannot tell, by reading a function call, whether it might blow up and unwind your program. The control flow is hidden. For code you're trying to verify at a glance, hidden control flow is the enemy.

Go made a stark choice: no exceptions for ordinary errors. Instead, a function that can fail simply returns an error as an ordinary value, right alongside its result. Failure isn't a trapdoor in the floor — it's a thing handed to you, in your hand, that you must look at.

The pattern you'll see ten thousand times

gof, err := os.Open("config.json")
if err != nil {
    return err          // can't open it — say so, and stop
}
defer f.Close()         // (more on defer below)

// ...only here, past the check, do we know f is good...

That little block — call something, get val, err, then immediately if err != nil — is the rhythm of all Go code. Yes, you'll write it a lot. But look at what it buys: every place a thing can go wrong is visible, named, and right next to the call. There are no surprises hiding up the stack. When you skim AI-generated Go, the error checks are sign-posts saying "here is exactly what this code worried about." That's a reviewer's dream.

💡
But isn't all that boilerplate noise?

It feels like noise until you've debugged the alternative. The Go designers' bet — and Rob Pike's famous line, "errors are values" — is that making failure explicit and ordinary means you actually think about each failure instead of vaguely hoping a catch upstairs handles everything. For a vibe coder it's even better: the pattern is so uniform that an agent reproduces it perfectly, and you can audit error handling just by scanning for if err != nil.

What an error actually is

No magic here — and that's the point. error is just a built-in interface with a single method:

go · the entire definitiontype error interface {
    Error() string
}

Anything with an Error() string method is an error. The value nil means "no error." Most of the time you create one with fmt.Errorf, and crucially you wrap lower-level errors with %w to add context while preserving the original:

go · wrapping adds context without losing the causefunc loadConfig(path string) (Config, error) {
    data, err := os.ReadFile(path)
    if err != nil {
        // %w wraps the original error so callers can still inspect it
        return Config{}, fmt.Errorf("loading config %s: %w", path, err)
    }
    // ...parse data...
}

Now an error reads like a breadcrumb trail: loading config app.json: open app.json: no such file or directory. You see what failed and where, top to bottom. Callers who care can dig into the wrapped cause with errors.Is (does this match a known error?) or errors.As (is this a specific error type?).

defer: cleanup that can't be forgotten

You saw defer f.Close() above. defer schedules a call to run when the surrounding function returns — no matter how it returns. It's how Go guarantees cleanup without exceptions.

gofunc process() error {
    f, err := os.Open("data.txt")
    if err != nil {
        return err
    }
    defer f.Close()    // runs on EVERY return path below — you can't forget it

    // ...lots of code with many possible returns...
    return nil
}

Put the defer right after you acquire a resource and you've made the cleanup impossible to leak, even if a dozen error-returns sit between acquire and close. Files, network connections, mutex unlocks, database rows — same pattern every time.

⚠️
"But Go has panic — isn't that an exception?"

It has panic/recover, yes — but they're for truly unexpected, programmer-error situations (an impossible state, an index out of range), not for ordinary failures like "file not found" or "bad user input." The idiom is firm: return errors for things you expect can happen; panic only for bugs. If you catch an agent reaching for panic to handle a normal failure, that's a code smell — send it back to return an error.

🔑
Key idea

In Go, failure is data, not a trapdoor. Functions return (result, error); you check if err != nil immediately; you wrap with %w to add context; you defer cleanup so it can't leak. Control flow stays visible and local — which is exactly what makes machine-written Go easy to trust.

The thirty-second version

  • No exceptions for ordinary errors — a fallible function returns (value, error).
  • The universal rhythm: v, err := do(); if err != nil { return err }.
  • error is just an interface with Error() string; nil means success.
  • Wrap with fmt.Errorf("context: %w", err); inspect with errors.Is / errors.As.
  • defer guarantees cleanup on every return path. panic is for bugs, not for expected failures.
