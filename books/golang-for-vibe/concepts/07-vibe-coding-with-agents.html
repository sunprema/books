Vibe Coding Go with Agents

You now know enough Go to read what an agent writes. This chapter is about the part that actually decides whether you ship something good: the loop you run the agent in, and the guardrails that keep it honest.

The mental shift: you're the test harness, not the typist

Let me be Feynman-blunt about this. An LLM does not "understand" your program the way you imagine. It's an extraordinarily well-read pattern-completer. Left unsupervised it will write code that looks right — confident, plausible, idiomatic — and is sometimes quietly wrong. That's not a flaw you can prompt away; it's the nature of the tool. So the entire game of vibe coding is building a loop where wrong code gets caught fast and cheaply, by a machine, not by you reading every line.

And this is precisely where Go earns its place (Chapter 1). Go ships, in the box, an unusually strong set of automatic graders. Your job is to wire them into the loop and make the agent answer to them.

Go's built-in graders — the agent's report card

Every one of these is a single command. Together they form a fast, brutal, honest feedback signal — exactly what an agent needs to self-correct:

CommandWhat it grades
go build ./...Does it even compile? Strict static types catch hallucinated methods and type mismatches instantly.
go vet ./...Suspicious-but-compiling mistakes: bad format strings, unreachable code, misused locks.
gofmt / go fmtCanonical formatting. One true style — no diff noise, no style debates.
go test ./...Runs the tests. The real arbiter of "does it do the right thing."
go test -race ./...Runs tests with the data-race detector on — catches concurrency bugs (Chapter 3).

Notice they're fast (Chapter 1's point about the compiler). A full build-vet-test cycle on a normal project is seconds. That speed is what lets the agent take many corrective passes per minute instead of a few per hour.

🔑
Key idea

Vibe coding works when the feedback loop is automated and tight. Go hands you a free, fast, strict grading rig — build, vet, fmt, test, -race. Your leverage as a vibe coder is to put the agent inside that loop and refuse to accept code until every grader is green.

The loop, concretely

Here's the cycle that actually produces trustworthy Go with an agent. It's not fancy — it's discipline:

Spec → agent writes → graders run → you review. Failures loop straight back as the next prompt.
  1. Write a spec, and write the tests first (or have the agent write them, then you check). In Go, a test is just a function func TestThing(t *testing.T) in a _test.go file. Tests are the executable definition of "correct" — and Go's testing is built in, no framework to choose. A spec the agent can check itself against is worth more than any amount of careful prompting.
  2. Let the agent write the implementation. Small, named tasks beat "build the whole app." Go's small surface area means the agent rarely wanders into exotic territory.
  3. Run the graders — and feed every failure straight back. A compile error or failing test isn't a problem; it's the highest-quality prompt you have. "go vet says X, the test TestParse fails with Y — fix it." The agent's strength is reacting to a concrete error message.
  4. Review the green code yourself. Green graders mean "compiles, formatted, tests pass" — not "correct and well-designed." This is the step you cannot delegate. You're checking: does it solve the actual problem? Are the error paths real (Chapter 4)? Are the interfaces small (Chapter 5)? Is it doing something dumb the tests didn't cover?

Habits that make Go agents reliable

  • Keep functions small and the package boundaries clear. Smaller units are easier for both you and the agent to reason about, test, and replace. Go's "capitalized = public" rule makes the intended surface obvious.
  • Demand tests with the code, every time. "Add the handler and a table-driven test for it." Table-driven tests (a slice of input/expected cases in a loop) are the beloved Go idiom and agents write them well.
  • Pin the standard library. Tell the agent to prefer the stdlib and justify any new dependency. Fewer deps = less supply-chain risk and less for the model to get subtly wrong (Chapter 6).
  • Manage the agent's context, not just its prompt. The real work in agentic coding is controlling what the model sees: the relevant files, the failing test output, the spec — and trimming the rest. A focused context produces focused code.
  • Always run -race on anything concurrent, and let the agent read the detector's output. Concurrency bugs are exactly the class humans miss on review.
⚠️
The trap: green is necessary, not sufficient

The single biggest failure mode is trusting a green checkmark as proof of correctness. An agent can write code that compiles, is formatted, and passes the tests it also wrote — while completely missing the point, or while its tests assert the wrong thing. Karpathy's own update on vibe coding was exactly this: the professional version is "increasingly the default workflow… but with more oversight and scrutiny." The graders catch the mechanical mistakes so that your human attention is freed for the judgment ones. Spend it there.

The thirty-second version

  • You're not the typist — you're the harness. Build a loop where wrong code is caught fast and cheaply.
  • Go's free graders are your loop: go build, go vet, gofmt, go test, go test -race. Make the agent answer to all of them.
  • Spec + tests first → agent implements → graders run → you review. Feed every failure back as the next prompt.
  • Demand tests with code, keep functions/interfaces small, prefer the stdlib, manage context, always -race concurrency.
  • Green ≠ correct. The graders buy you time; spend it on the judgment the machine can't do.
