Concept 03 · Concurrency

Goroutines & Channels

This is Go's party trick: doing thousands of things at once without your brain melting. Let's build the intuition from a single physical question — what is a thread, really?

The problem, made physical

Picture a restaurant kitchen. A naïve cook does one ticket start-to-finish before touching the next: chop, simmer twenty minutes standing there watching the pot, plate, repeat. Absurd — most of the time is spent waiting, doing nothing. A real cook starts the soup, and while it simmers, chops the salad for another order. The work that blocks (waiting on the pot, the oven, the network) overlaps with other work. That overlap is concurrency, and almost all server work — waiting on databases, disks, other servers — is mostly waiting.

The old way to get this overlap was operating-system threads. But an OS thread is expensive: it grabs a megabyte or two of stack and the kernel has to babysit it. Spin up ten thousand and you've drowned the machine. So everyone built awkward workarounds — callbacks, async/await coloring, event loops — to fake concurrency without paying for threads.

Goroutines: threads, but almost free

Go's answer is the goroutine: a function running concurrently, managed by Go's own runtime rather than the OS. The whole trick is the price tag. A goroutine starts with about a 2 KB stack — a thousand times smaller than an OS thread — and grows only if it needs to. The Go runtime quietly multiplexes thousands of goroutines onto a handful of real OS threads. So launching a hundred thousand of them is genuinely fine.

And the syntax is the punchline. To run something concurrently, you put the word go in front of it:

gofunc fetch(url string) {
    // ... do the slow network call ...
}

func main() {
    go fetch("https://a.example")   // runs concurrently, doesn't block
    go fetch("https://b.example")   // so does this one
    // main keeps going immediately...
}

That's it. go fetch(...) means "start this running and don't wait for it." No thread pool to configure, no async keyword to thread through your whole call stack. The word go is the feature.

⚠️
The catch you just created

In that example main launches two goroutines and then… reaches its end and exits, killing the whole program before either fetch finishes. A goroutine is not a promise the program will wait for it. So we need a way to coordinate — to say "wait until the soup is done." That's the other half of the story.

Channels: how goroutines talk safely

Here's the deepest idea in Go concurrency, and it's worth slowing down for. In most languages, threads share memory and you bolt on locks to stop them from stepping on each other — and getting those locks right is famously where the 3 a.m. bugs live. Go's motto turns it around:

🔑
The Go proverb

"Do not communicate by sharing memory; instead, share memory by communicating." Rather than several goroutines poking the same variable behind locks, one goroutine hands a value to another down a pipe. At any moment the value has one clear owner. The pipe is called a channel.

A channel is a typed conduit. You send with ch <- v and receive with v := <-ch. The magic property: a receive blocks until something is sent, and a send blocks until someone receives. So a channel doesn't just pass data — it synchronizes. It's the pot timer and the hand-off, in one.

go · fetch N URLs concurrently, collect resultsfunc main() {
    urls := []string{"a", "b", "c"}
    results := make(chan string)        // a channel of strings

    for _, u := range urls {
        go func(url string) {
            // ...slow work...
            results <- "done: " + url     // send result into the channel
        }(u)
    }

    for range urls {                    // receive exactly len(urls) results
        fmt.Println(<-results)            // blocks until a value arrives
    }
}

Read what happened: three goroutines run their slow work at the same time, each drops its answer into results, and main calmly collects three values. The total wall-clock time is roughly the time of the slowest single fetch, not the sum of all three. That's the whole win, and there's not a lock in sight.

Many goroutines send into one channel; main receives. The channel both moves data and synchronizes.

Two everyday helpers

Channels are the foundation, but for two common chores Go's standard sync package gives you blunt tools — and an agent will reach for these constantly:

  • sync.WaitGroup — "wait for N goroutines to all finish." wg.Add(n), each goroutine calls wg.Done(), and wg.Wait() blocks until the count hits zero. This is the clean fix for the "main exits too early" bug above.
  • context.Context — the standard way to say "stop now" or "give up after 5 seconds." You pass a ctx down through your calls; cancel it (or let its timeout fire) and every goroutine watching it can bail out. Essential for network code, and the backbone of cancellation in real Go services.
🧭
For the vibe coder: name the pattern, not the plumbing

You rarely need to hand-write this machinery. The leverage is in asking precisely: "fetch these URLs concurrently with a WaitGroup, collect results on a channel, and cancel everything if a context deadline of 5s is exceeded." Those four nouns — goroutine, channel, WaitGroup, context — are the vocabulary that turns a vague "do it in parallel" into code the agent can write correctly. And Go ships a built-in race detector: run your tests with go test -race and it will flag concurrent-access bugs at runtime. Always tell the agent to run it.

The thirty-second version

  • A goroutine is a near-free concurrent function — start one with go f(). You can have hundreds of thousands.
  • The runtime multiplexes them onto a few OS threads; each starts at ~2 KB.
  • A channel (make(chan T)) passes typed values between goroutines and synchronizes them — send/receive block until matched.
  • "Share memory by communicating" — hand values down channels instead of sharing variables behind locks.
  • Use sync.WaitGroup to wait for goroutines, context.Context to cancel/timeout, and go test -race to catch data races.
