Concept 01 · The Why

Why Go, and Why Now

If you're going to let an agent write most of the code, you should pick the language that's easiest to get right. Let's reason out, from scratch, why that's so often Go.

Start with what "vibe coding" actually is

In February 2025 Andrej Karpathy gave a name to something everyone had started doing: you describe what you want in plain English, a large language model writes the code, and you steer by looking at the results and asking for changes. He called it vibe coding — "fully give in to the vibes… and forget that the code even exists." A year later even he admitted the pure forget-the-code version was already passé; the grown-up version is programming through agents, with oversight. You still read, test, and judge — you just don't type most of the characters.

Now here's the part people skip. If a machine is writing the code and you're judging it, the bottleneck moved. It's no longer "how fast can I type clever things?" It's "how quickly and confidently can I tell whether this code is correct?" That single shift changes which language is the smart bet. And it points, almost embarrassingly directly, at Go.

Why plainness beats cleverness here

Let me give you a physical picture. Imagine two workshops. One is a jazz musician's studio — a thousand ways to express any idea, gorgeous when it works, but you have to know the player to read the sheet. The other is IKEA: a small set of identical parts, one Allen key, and instructions a child can check. When you personally are the master craftsman, the jazz studio wins. But when a tireless apprentice is doing the assembly and your job is quality control — you want IKEA. Fewer ways to do a thing means fewer ways to do it wrong, and far less to verify.

Go was designed, on purpose, to be the IKEA. Listen to its actual design rules:

  • One way to format code. gofmt is not a preference — it's the format. Every Go file in the world looks the same, so an agent never argues about style and you never review a diff full of cosmetic noise.
  • One way to handle an error. No exceptions hiding in the bushes. If a function can fail, it hands you back an error value, right there in the open (Chapter 4).
  • A tiny language. The whole spec fits in an afternoon. There are no classes, no inheritance, no operator overloading, no macros, no decorators. Less surface area means an agent has less rope to hang itself with.
  • Static types, checked instantly. The compiler catches a whole genus of "the AI hallucinated a method that doesn't exist" before the program ever runs.
  • It basically hasn't changed. Go promises backward compatibility and hasn't made a breaking change to the language in over a decade. So the millions of correct Go examples an LLM trained on are still correct today — the training data hasn't rotted.
💡
But why does that last point matter so much?

An LLM is, at heart, a very good average of everything it has read. If a language has six fashionable ways to do one thing and they churn every two years, the model's "average" is a muddle of contradictory, half-deprecated patterns. Go gives the model a stable, consistent, decade-deep corpus where there's usually one idiomatic answer. The model's average lands on the right answer more often. That's not magic — it's just clean training data.

Where Go genuinely shines

Plainness would be worthless if Go weren't also good at real work. It is — and it's sharply good at a specific shape of work. Be honest about the shape and you'll know exactly when to reach for it.

Go is a great fit for…Because…
Command-line toolsIt compiles to one self-contained binary. No runtime, no node_modules, no "works on my machine." You scp the file and it runs.
Web APIs & backendsHTTP, JSON, and routing are in the standard library. A production-shaped JSON service needs zero third-party packages (Chapter 6).
Microservices & networked glueGoroutines make "handle 10,000 connections at once" almost boring (Chapter 3), and the small memory footprint loves containers.
Cloud-native infrastructureDocker, Kubernetes, Prometheus, Terraform, and Istio are written in Go. It is the native tongue of the cloud-native world.
The plumbing around AICalling LLM APIs, fanning out requests, streaming tokens, building agent harnesses — Go's concurrency and reliability make it a favorite for the systems around the model.
🧭
And where it's the wrong tool — say so honestly

Go is not where you do data science, train models, or do heavy numeric/ML work — that's Python's turf, and it's not close. Go has no real GUI story, and for tiny throwaway scripts a shell one-liner is often faster. Picking the right tool is half the craft. Go's sweet spot is servers, tools, and infrastructure that must be fast, small, and reliable. Aim it there.

The compiler is fast enough to be part of the conversation

One more thing, and it matters more than it sounds. Go compiles fast — seconds, not minutes. Why should a vibe coder care? Because the agent's whole loop is write → compile → test → read the error → fix. If each turn of that crank takes ten seconds instead of three minutes, the agent can take ten times as many corrective steps in the same wall-clock time. A fast compiler with strict types is, for an agent, a fast and brutally honest tutor that grades every attempt instantly. Cleverness you can't verify is worthless; correctness you can verify in two seconds compounds.

🔑
Key idea

When an agent writes the code and you judge it, optimize for verifiability, not expressiveness. Go's small, stable, one-true-way design and its fast strict compiler make AI-written code easy to trust — and its sweet spot (CLIs, APIs, cloud infrastructure) is exactly the kind of software vibe coders ship most.

The thirty-second version

  • Vibe coding moves the bottleneck from typing to verifying. Pick the language that's easiest to verify.
  • Go is deliberately plain: one formatter, one error idiom, a tiny stable spec, static types. Fewer ways to go wrong.
  • A decade of backward compatibility means the LLM's training corpus is still correct — so it guesses right more often.
  • Go's home turf is CLIs, web APIs, microservices, cloud-native infra, and the plumbing around AI. It is not for ML or GUIs.
  • A fast, strict compiler turns the agent's edit loop into a tight, honest feedback cycle.
