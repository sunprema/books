The Future of Protobuf

We've reached the last chapter, so let's do the thing every good closing chapter does: step back from the bytes and ask where all of this is going. For a book, that's a satisfying question. For a serialization format, it's almost a strange one — protobuf's entire promise is that a message you write today still decodes cleanly in five years, which is another way of saying the format is engineered not to change out from under you. So the interesting story isn't a dramatic reinvention. It's the quiet, deliberate way the language is shedding an old wart (the proto2/proto3 split), the way its tooling grew from “a compiler called protoc” into a whole ecosystem, the way protobuf-the-schema and gRPC-the-transport have started to lead separate lives, and the honest question of whether, in a world of zero-copy formats and JSON everywhere, protobuf still earns its keep. Let's take those one at a time, and by the end we'll be able to say what actually holds this whole thing together.

The end of the proto2/proto3 split: Editions

Start with the change you're most likely to see in a real .proto file. For its entire public life, protobuf has forced you to pick a dialect on the very first line — syntax = "proto2"; or syntax = "proto3"; — and that one word silently decided a pile of behaviors: whether fields track presence, whether required exists, whether repeated numbers pack, whether unknown enum values are preserved. Two dialects, two mental models, and (as we saw painfully in Chapter 15) a seam between them where bugs love to hide. Protobuf Editions is the plan to end that split for good.

The idea is simple and, in hindsight, obvious. Replace the syntax line with an edition — a year, like edition = "2023"; — and then break out all those implicit behaviors into named, individually-selectable features. The syntax version used to be a bundle you couldn't unbundle; a feature is a dial you can turn one at a time, at the file level (and, from edition 2024 on, right down to a single field). Editions were announced on June 29, 2023, edition 2023 is the first stable one, and edition 2024 became generally available in Protobuf v32. 

// No more `syntax = "proto3";`. An edition, plus explicit features where you want them.
edition = "2023";

package tutorial;

message User {
  string id   = 1;
  string name = 2;

  // field_presence is EXPLICIT by default in edition 2023 (so has_login_count() exists).
  // Opt THIS field back into proto3's old implicit behavior, and nothing else:
  int32 login_count = 3 [features.field_presence = IMPLICIT];
}

Read that snippet against everything we learned in Chapters 2, 7, and 15 and it should click. The thing that used to be a global, invisible consequence of your syntax choice — is a set-to-zero field the same as an unset one? — is now a feature you declare. That's the whole spirit of Editions: make the implicit explicit. Here are the six features edition 2023 defines, and notice how each one is just a behavior we've already met in this book, now given a name and a switch.

FeatureEdition 2023 defaultWhat it controls (and where we met it)
field_presence EXPLICIT Whether a scalar tracks set-vs-unset. IMPLICIT is proto3's old no-presence behavior; LEGACY_REQUIRED is proto2 required. (Ch. 2, 15)
enum_type OPEN Whether an unknown enum number is preserved as-is (OPEN) or coerced (CLOSED, proto2-style). (Ch. 7, 15)
repeated_field_encoding PACKED Packed vs EXPANDED (unpacked) wire layout for repeated numerics. (Ch. 4, 7)
utf8_validation VERIFY Whether string bytes are validated as UTF-8, or left unchecked (NONE).
message_encoding LENGTH_PREFIXED Normal length-delimited submessages, or DELIMITED — the return of proto2 “groups.” (Ch. 4)
json_format ALLOW Whether the message is guaranteed JSON-mappable (Ch. 10), or only LEGACY_BEST_EFFORT.

Most of those defaults line up with proto3, which is intentional — migrating shouldn't move your cheese. The one that jumps out is field_presence = EXPLICIT: an editions file tracks presence by default, quietly fixing the single most common proto3 footgun from Chapter 15, where “set to zero” and “never set” were indistinguishable. And look at the two exiles — LEGACY_REQUIRED and DELIMITED. proto2's required and groups aren't gone; they've been demoted from first-class syntax to opt-in feature values you can reach for on legacy schemas without spreading their defaults to anyone else. That's the elegant part: editions doesn't delete history, it contains it.

Editions is a source-language change, not a new wire format This is the part to hold onto: adopting an edition does not change the bytes on the wire. Field numbers, varints, tags, packed repeated fields, the compatibility rules from Chapter 7 — all unchanged. Editions changes how a .proto declares its semantics, not how a message is encoded. So “proto3 is dead” is the wrong reading: proto2 and proto3 files keep compiling and interoperating, migration tools (prototiller, protoc's editions support) convert them mechanically, and an old file is simply treated as edition-equivalent. Editions unifies the future; it doesn't invalidate the past.

A language with a spec — and split governance

Here's a fact that surprises people: for most of its life, the .proto language had no written specification. The “spec” was “whatever the C++ protoc does.” That's a shaky foundation for something running inside half the world's backends, and the future is quietly fixing it. protobuf.dev now publishes a formal, per-edition language specification (there's an edition-2023 spec and an edition-2024 spec), and independently, Buf maintains a community grammar for the language. When a format gets a real spec, it's a sign it has stopped being one company's implementation detail and started being infrastructure.

That maturation shows up in the compilers too. For years, protoc was the only program that could authoritatively parse a .proto file. Now Buf ships protocompile, a pure-Go parser and linker — the guts of a protoc replacement — which is exactly the kind of second, independent implementation a spec makes possible and a healthy ecosystem needs.

But the most important structural shift is about who owns what, and it's easy to miss because we so often say “protobuf” and “gRPC” in the same breath. They are governed differently now. Protocol Buffers — the schema language, the wire format, the protoc reference — remains a Google-led open-source project. gRPC — the RPC framework we built in Chapter 9 — is a Cloud Native Computing Foundation project, run under vendor-neutral open-collaboration governance with maintainers across many companies. Two layers of the same stack, two different homes.

Why the governance split matters to you It's not org-chart trivia. Different governance means the two layers can — and do — evolve on their own clocks and answer to different communities. It's the structural reason the next section is even possible: because protobuf-the-schema isn't chained to gRPC-the-transport, other people are free to build different transports on top of the very same .proto files.

The ecosystem grew up around the schema

When you started this book, “protobuf tooling” meant protoc and a pile of language plugins. Today there's a whole layer above that, and much of it has coalesced around Buf. We met the pieces across Chapters 5, 7, and 11 — the buf CLI (build, lint, format, breaking-change detection, codegen), the Buf Schema Registry as a versioned home for your modules with hosted docs and published SDKs, managed mode and remote plugins for reproducible generation, buf-action to fail a pull request that breaks the wire. The throughline is that the schema has been promoted from “a file in your repo” to a governed, versioned, discoverable artifact — exactly the long-term discipline this book has argued for since Chapter 1.

Two younger projects show where that discipline is heading next. The first is protovalidate, the successor to protoc-gen-validate. The old approach generated a Validate() method per language, which meant your validation logic was recompiled — and could subtly diverge — in every target. protovalidate changes the shape entirely: it's a runtime library driven by CEL (the Common Expression Language), so you write a rule once, as an option on the field, and it evaluates identically in Go, Java, Python, C++, and TypeScript. It reached v1.0, and it's a beautiful expression of the book's core idea — push more of your contract into the schema, where every language agrees on it. 

// Validation as a property of the schema, checked at runtime by protovalidate.
// One rule, evaluated the same way in every language. No per-language codegen.
import "buf/validate/validate.proto";

message SignUp {
  string email = 1 [(buf.validate.field).string.email = true];
  uint32 age   = 2 [(buf.validate.field).uint32.gte = 13,
                          (buf.validate.field).uint32.lte = 150];
}

The second is Connect (ConnectRPC), and it's the clearest signpost of all — so much so that it earns its own section.

gRPC and protobuf: diverging, and coexisting

Back in Chapter 9 we treated gRPC as protobuf's natural partner — you define a service, gRPC carries the calls. That's still true, but the future complicates the tidy picture in a healthy way. gRPC is a specific, powerful, sometimes-heavy stack: it wants HTTP/2, its browser story needs a proxy, and its client libraries are large. None of that is protobuf's fault — protobuf is just the schema and the bytes. And once you internalize that separation, an obvious question follows: if the schema is independent of the transport, why should there be only one transport?

Connect is the answer several teams reached. A single Connect handler simultaneously speaks three protocols — gRPC, gRPC-Web, and Connect's own simple POST-based protocol that runs happily over plain HTTP/1.1 and is pleasant to call from a browser with fetch. Crucially, it's wire-compatible with existing gRPC clients and servers and reuses your .proto completely unchanged. You don't migrate your schema; you don't rewrite your gRPC callers; you just gain the option of a lighter protocol where you want one. And in April 2024, Connect joined the CNCF as a Sandbox project — a Protobuf-based RPC framework, distinct from gRPC, with its own vendor-neutral home.

The mental model to carry forward: two layers, not one Stop thinking “protobuf = gRPC.” Think of a stack. The bottom layer is protobuf: the schema, the types, the wire format, the compatibility rules — the durable contract. The top layer is a transport that carries messages defined by that contract — and that layer is now plural: gRPC, gRPC-Web, Connect, and REST/JSON via a gateway. Connect's whole existence is proof the layers have separated: same .proto, same generated types, different wire on the outside. Protobuf is the noun; the RPC framework is a verb you can swap.

This is divergence and coexistence at the same time. The transports diverge — different protocols, different governance, different trade-offs. But they coexist effortlessly precisely because they all agree on the layer underneath. That shared layer, it turns out, is the thing worth betting on.

Staying power: an honest reckoning

Let's end where an honest book should: with the competition, and without flattery. Protobuf is not the fastest way to move structured bytes, and it's important to say so plainly. Zero-copy formats — FlatBuffers (from Google's own games group) and Cap'n Proto (from a former protobuf maintainer, no less) — lay data out so you can read a field without parsing the message at all. On raw serialize/deserialize benchmarks, they win, sometimes dramatically. If your bottleneck is genuinely per-message CPU in a hot loop — a game engine, a market-data feed — protobuf may be the wrong tool, and you should know that.

FormatWhere it winsWhat it gives up
Protobuf Schema-as-contract, backward/forward compat, breaking-change tooling, codegen across a dozen languages, huge ecosystem Not the fastest; must parse to read; not human-readable
FlatBuffers Zero-copy reads, tiny memory overhead — great for games/mobile Thinner ecosystem & docs; more awkward to write and evolve
Cap'n Proto Fastest (de)serialization; zero-copy; RPC built in Less flexible evolution story than protobuf's optional/default/deprecation model; smaller community
Avro Dynamic, data-embedded schemas — the default in big-data/Kafka pipelines Less natural for typed RPC service contracts
JSON Human-readable, universal, zero setup Bulkier, slower, no enforced schema or types by itself

So why does protobuf keep winning the middle of the map anyway? Because for the job most teams actually have — a typed, versioned contract between services and languages that has to keep working for years — speed was never the deciding factor. The widely-argued case, and the one this whole book has quietly been making, is that the real reason to reach for protobuf is not performance. It's the discipline: one source-of-truth schema; compatibility rules the wire enforces; a compiler that generates honest, idiomatic code everywhere; and now a spec, a registry, CI gates, runtime validation, and a choice of transports around it. Cap'n Proto is faster; JSON is friendlier to a human; Avro fits the data lake better. Protobuf's moat is that it turned serialization into a governable, long-lived contract — and that moat is exactly what got wider this chapter, not narrower.

So what did we actually gain — from the whole book? Sixteen chapters ago, protobuf looked like “a faster JSON with a compiler.” I hope it looks different now. Every chapter was secretly about one idea: a schema is a promise you make to your future self and to every other program that will ever read your bytes. The wire format (Chapter 4) is engineered so that promise survives change. Field numbers, presence, and reserved (Chapters 2, 7, 15) are the rules that keep the promise honest. Codegen and cross-language work (Chapters 5, 10) are how the promise reaches every language at once. And this final chapter is just that same idea, projected forward: Editions makes the promise explicit instead of implied; a formal spec makes it durable; the Buf ecosystem and protovalidate make it enforceable; and the separation of protobuf from gRPC makes it portable across transports. The bytes were never the point. The contract was. That's what you learned to build — and it's why, whatever comes next, the discipline you now have will outlast any one tool.
Challenges — before you close the book
  1. Take a proto3 message with an int32 status_code where the team has always needed to tell “unset” from “explicitly 0.” Rewrite its header as edition = "2023" and explain, referencing the default value of features.field_presence, why the ambiguity from Chapter 15 simply disappears — and what single feature override would bring the old proto3 behavior back for one field.
  2. A colleague says “we can't adopt Editions, it'll change our wire format and break every deployed client.” They're wrong. Explain precisely why, and name the two things Editions does change instead.
  3. Your team defines services in .proto and currently serves them over gRPC, but the web frontend struggles to call them from the browser. Describe how adopting Connect lets the browser talk to the same service — and state exactly what you would not have to change about your schema or your existing gRPC backend clients.
  4. You're asked to migrate a field-validation rule (“email must be valid, age between 13 and 150”) off a hand-written check in three languages. Sketch the protovalidate annotation, and explain the one architectural reason it can't drift between Go and TypeScript the way the old protoc-gen-validate-generated methods could.
  5. A performance-obsessed teammate benchmarks protobuf against Cap'n Proto, sees Cap'n Proto win, and proposes switching your inter-service API. Give the strongest honest argument for staying on protobuf that does not dispute the benchmark — and the one scenario where you'd agree with them and switch.
