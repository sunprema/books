Chapter 13 · Production & Beyond

Storage and Event Systems

We've spent twelve chapters treating protobuf as a way to get a message from one process to another and back — a wire format, born in the request/response world of gRPC. But here's a quiet truth the RPC framing hides: protobuf doesn't know or care whether its bytes are flying across a network or sitting on a disk. A serialized message is just a byte string. You can send it, sure — or you can write it to a file, drop it into a database column, or append it to a Kafka log where it might be read back years from now. And that last part changes everything. A wire message lives for milliseconds; a stored message lives until someone deletes it. This chapter is about what happens when protobuf stops being a courier and becomes an archive — how you frame many messages into one stream, how bytes can carry their own schema, how event systems like Kafka wire protobuf up to a schema registry, and why data at rest demands a stricter schema discipline than anything on the wire ever did.

The same bytes, at rest

Start with the thing that's easy to miss because it's so simple: SerializeToString returns a byte string, and a byte string doesn't care where it goes. The exact bytes you'd put on a gRPC connection are the exact bytes you can write to events.pb on disk, stuff into a bytea column in Postgres, cache in Redis, or store as the value of a Kafka record. Google built protobuf for exactly this dual life — it has always been a format for storing structured data as much as for exchanging it. Nothing special is required; the write side is SerializeToString, the read side is ParseFromString, and in between the bytes can sit still for as long as you like.

The reframing A protobuf message is not “an RPC payload.” It's a compact, schema-defined byte string that happens to work beautifully on the wire — and just as well in a file, a column, or a log. Everything in this chapter falls out of taking that seriously: if the bytes can persist, they will, and persistence has rules the wire never enforced.

But two problems appear the instant the bytes stop moving. First: a single serialized message has no idea where it ends — fine when a connection or a length-header framed it for you, a real question when you want to store thousands of messages in one file. Second: the bytes carry field numbers, not field names, so a stored message is meaningless without the .proto that describes it — and the .proto that described it might be long gone by the time someone reads it back. Let's take those in order, because they're the two hinges this whole chapter turns on: framing and self-description.

Framing: where does one message end?

Here's a fact that surprises people the first time it bites them: the protobuf wire format is not self-delimiting. A parser reading a stream of bytes cannot, on its own, tell where one message stops and the next begins. On a single request that's a non-issue — the whole byte string is the message. But the moment you want to append a second message after the first in the same file, you've created an ambiguity protobuf can't resolve for you.

And it's worse than “it can't tell” — it will confidently do the wrong thing. Recall from Chapter 4 that concatenating two serialized messages of the same type and parsing the result is defined to behave like merging them: singular fields take the last value seen, repeated fields concatenate. So if you naively write msgA + msgB to a file and read it back, you don't get two messages — you get one Frankenstein message that's the merge of both, with no error to warn you. Raw concatenation doesn't stream messages; it silently fuses them.

The trap in one line Two serialized messages stuck end-to-end don't parse as two messages — a same-type concatenation parses as their merge, and a mixed-type one parses as garbage. There is no exception thrown to save you. If a stream needs boundaries, you must put them there.

The fix is the oldest trick in binary I/O: length-prefixing. Before each message, write its size — as a varint, naturally, to keep it compact — then the message bytes. To read, you decode the varint, read exactly that many bytes into a buffer, and parse the buffer. The length tells the reader precisely where the message ends, so the next varint it reads is the length of the next message. This is the canonical “stream of messages” layout, and it's what turns a set of structures into a tidy set of byte strings you can append to forever.

Some languages hand you this for free. In Java, every message has writeDelimitedTo(OutputStream) and the static parseDelimitedFrom(InputStream) — they do exactly the varint-then-bytes dance, and parseDelimitedFrom returns null at end-of-stream so you can loop cleanly. Most other languages have no built-in delimited API: in C++ you reach for CodedOutputStream::WriteVarint32 plus the message's serialize call (and CodedInputStream with a byte limit to read it back); in Python and Go you write the length yourself. The convention is universal — a varint length before each message — but the convenience method is Java's alone. 

// A length-delimited log of events, in one file (Java).
// Each write emits: varint length, then the message bytes.
try (OutputStream out = Files.newOutputStream(path)) {
  for (Event e : events) {
    e.writeDelimitedTo(out);       // framing handled for you
  }
}

// Reading mirrors it — parseDelimitedFrom returns null at EOF.
try (InputStream in = Files.newInputStream(path)) {
  Event e;
  while ((e = Event.parseDelimitedFrom(in)) != null) {
    handle(e);
  }
}
One message, or a stream of many? This is also the answer to a question Chapter 12 raised: protobuf wants each message to stay small — the project's own guidance is to keep individual messages well under a megabyte and treat anything larger as a smell. A million-row export shouldn't be one giant repeated field; it should be a delimited stream of a million small messages. Same data, but now it parses incrementally, streams without buffering the whole thing, and never triggers the multi-copy memory spikes a monster message causes.

When the schema travels with the data

Framing solves “where does a message end?” Now the second problem: a stored message is opaque. Unlike JSON, protobuf bytes carry no field names — just numbers and wire types. Hand someone a .pb file and no .proto, and the best they can do is guess at a soup of tags and varints. On the wire that's fine, because both ends were compiled from the same schema. But bytes on a disk might outlive the code that wrote them, and “which .proto was this?” becomes a genuine archaeology problem.

Protobuf's answer is elegant and a little mind-bending: describe the schema using protobuf itself. The structure of a .proto file — its messages, fields, types, and dependencies — is itself modeled as a message, defined in google/protobuf/descriptor.proto. The compiled, resolved form of one or more .proto files is a FileDescriptorSet: literally the parsed schema, as data. It's not some side artifact, either — it's the same structure protoc feeds to every code-generator plugin (we'll meet it again from the plugin side in Chapter 14). You can emit one straight from the compiler: 

# Compile the schema itself INTO data: a portable FileDescriptorSet.
# --include_imports folds in every dependency so the set is self-contained.
protoc --include_imports \
       --descriptor_set_out=schema.desc \
       user.proto order.proto

# schema.desc IS a serialized FileDescriptorSet message. Ship it beside
# your data and you can decode old blobs later with no .proto on hand.

Put those two ideas together — the schema-as-data and the payload — and you get a self-describing message, the pattern the protobuf Techniques guide spells out. Wrap the actual payload in a google.protobuf.Any (which carries a type URL plus the bytes), and pair it with the FileDescriptorSet that defines that type. Now the message contains everything a reader needs: the data, and the schema to interpret it. 

// The classic self-describing message (protobuf Techniques guide).
syntax = "proto2";   // descriptor.proto is proto2
import "google/protobuf/any.proto";
import "google/protobuf/descriptor.proto";

message SelfDescribingMessage {
  // The schema: FileDescriptorProtos for the type and its deps.
  google.protobuf.FileDescriptorSet descriptor_set = 1;
  // The payload: the message and its type, as an Any.
  google.protobuf.Any message = 2;
}

A tool that receives this needs no compiled stubs at all. It reads the FileDescriptorSet, builds the type at runtime, and uses DynamicMessage (available in C++, Java, and others) to walk the payload field by field — field names and all. This is exactly how generic protobuf tooling works: a debugger that can pretty-print any message, a log viewer, an archival format that stays readable decades later. The cost is real — you're paying to carry the schema, and dynamic decoding is slower than generated code — so you reserve it for genuinely generic or long-lived storage, not your hot path. But when you need bytes that can explain themselves, this is the mechanism.

Protobuf in event systems: Kafka and the schema registry

A Kafka topic is the purest example of protobuf-as-storage. Producers append serialized messages to a log; consumers read them back — maybe seconds later, maybe after replaying from the beginning of a topic that's months old. Every message on that log is a stored protobuf message with all the durability concerns we've been building toward. So how do you keep a fleet of independently-deployed producers and consumers agreeing on the schema, without baking a FileDescriptorSet into every single record?

The industry answer is a schema registry — a central service that stores each schema once, under a named subject, and hands back a small integer ID. The most common is Confluent's Schema Registry. Instead of embedding the schema, each message carries a tiny reference to it. Confluent's serializer wraps the ordinary protobuf bytes in a small envelope: a magic byte (0x00), a 4-byte big-endian schema ID, and — unique to protobuf — a message-index that names which message type in the schema was used (a .proto can define several), which is usually a single 0x00 for the first top-level message. 

# Confluent Schema Registry framing around a protobuf value:

  byte  0      0x00              # magic byte — "this is registry wire format"
  bytes 1..4   00 00 00 2A       # schema ID = 42, big-endian int32
  byte  5      00                # message-index (varint array; 0 = first msg)
  rest         08 96 01 12 03 .. # ordinary protobuf bytes

# The consumer reads the schema ID, fetches THAT schema from the
# registry (and caches it), then decodes the trailing protobuf bytes.

The registry does more than deduplicate the schema — it governs it. Every time a producer tries to register a new version of a subject's schema, the registry checks it against a configured compatibility mode. Confluent's default is BACKWARD: a new schema must be able to read data written with the previous one. That specific direction is what makes the common deploy safe — upgrade your consumers first, and they can still read every old message already sitting on the topic, then producers roll forward behind them. Set it wrong (or to NONE) and the registry will happily accept a change that orphans every message written before it. The registry only enforces the rule you told it to enforce.

Why Kafka reaches for a registry at all — the Avro contrast Chapter 12 noted protobuf and Avro make opposite bets about where the schema lives. It matters most here. Avro writes no field tags in the payload, so a reader needs the exact writer's schema to decode anything — a registry is effectively mandatory. Protobuf carries field numbers and wire types in the bytes, so it stays structurally parseable with no registry at all. On Kafka you still use a registry with protobuf — for names, types, and that governance — but understand what it's adding: discipline, not basic decodability. That difference is a big part of why Avro grew up on Kafka and protobuf grew up on gRPC.

Data at rest is a stricter discipline

Everything so far leads to one shift in mindset, and it's the real point of the chapter. On the wire, a schema mismatch is a momentary bug: fix the deploy, retry the call, it's gone. In storage, a schema mistake is permanent, because the bad bytes persist. The evolution rules from Chapter 7 weren't optional guidelines there — and for stored data they're closer to sacred law. A field number is permanent identity, so you must never reuse one; deleted fields and their numbers must be reserved so no future edit can recycle them. Reuse a number on the wire and you break a deploy; reuse a number in storage and you silently misread every record ever written with the old meaning — records that will outlive you on that disk.

There's a subtler durability property that stored data leans on hard: unknown-field preservation. When a message arrives with fields your schema doesn't know about (a newer producer, an older reader), proto3 — since v3.5, matching proto2 — keeps those unknown fields and writes them back out on re-serialization. So a service can read a record, modify one field, and save it, faithfully round-tripping data it doesn't even understand. That's forward compatibility you get almost for free… until you break it. And it's startlingly easy to break: projecting the message into typed database columns, converting it to JSON, or copying it field-by-field into a fresh message all silently drop the unknowns. A relational table has no column for a field it was never told about.

If round-tripping matters, store the bytes The only way to guarantee end-to-end preservation of unknown fields is to keep the original serialized bytes — a bytea/BLOB column, the raw Kafka value — not an exploded set of typed columns. Shred the message into columns and you've thrown away exactly the forward-compatibility protobuf handed you. Keep the blob (index a few columns beside it if you must query), and old readers preserve tomorrow's fields untouched.

And circle back to self-description one last time, because it's the quiet failure mode that ruins archives. A raw protobuf blob is not readable without its schema. If the only copy of the .proto that wrote a five-year-old record has been deleted, refactored beyond recognition, or lost to a repo migration, that record is now undecodable garbage — the bytes are intact and utterly meaningless. So durable storage carries an obligation the wire never did: keep the schema recoverable. Pin the .proto version in source control, store a FileDescriptorSet alongside the data, or lean on a registry that never forgets an old version. Whichever you choose, decide it on purpose — because the bytes will certainly outlast your memory of what they meant.

So what did we actually gain? A second life for a format we thought we understood. The same protobuf we've used to talk turns out to be a fine way to remember — provided we respect what remembering demands. We learned to frame many messages into one stream with varint length prefixes (because the wire format won't mark its own boundaries); to make bytes self-describing with a FileDescriptorSet and Any when they must outlive their code; to lean on a schema registry so an event log like Kafka can carry a 4-byte reference instead of a whole schema; and to treat data at rest with a stricter discipline than any wire message — permanent field numbers, preserved unknowns, a schema kept forever recoverable. The wire was forgiving because it forgot. Storage remembers everything, including your mistakes — so here, more than anywhere, the schema discipline this whole book has been teaching stops being good practice and becomes the thing standing between you and an unreadable disk.
Before you turn the page

  2. Your team stores a protobuf message by mapping each field to a Postgres column. A newer producer starts sending a field the reader's schema doesn't have yet. After a read-modify-write cycle through your service, where did that field go, and what storage layout would have preserved it?
  3. A Kafka consumer reads a message whose first byte is 0x00. What are the next four bytes, what does the consumer do with them, and why can't it simply parseFrom the record value directly?
  4. Someone proposes reusing field number 7 — freed when an old field was deleted — for a brand-new field, arguing “no current code uses 7 anymore.” Explain why this is far more dangerous for a database table than for a gRPC endpoint.
