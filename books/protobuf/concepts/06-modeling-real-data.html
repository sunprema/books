Chapter 06 · Encoding & Tooling

Modeling Real Data

Back in Chapter 2 we left a few threads dangling on purpose: Timestamp, Any, Struct, FieldMask, the wrapper types — a “small standard vocabulary,” we called it, and moved on. That was fair while Product only had a name and a quantity. It stops being fair the moment your schema needs a price, a creation time, an update that should touch exactly three fields and no others, or a payload whose exact shape won't be known until runtime. This chapter cashes in that promise. It's less about new syntax — you already know message, enum, repeated, map — and more about judgment: which type actually fits the data you have, and which of protobuf's own escape hatches are worth reaching for versus worth avoiding.

Picking scalar types like you mean it

Start with the field type every tutorial glosses over: money. It's tempting to reach for double — it's a number, prices are numbers, done. Don't. Binary floating point can't represent common decimal fractions like 0.10 exactly, and that inexactness compounds every time you add, multiply, or sum a column of prices. The standard fix is fixed-point: store an integer count of the currency's smallest unit. A single-currency field can just be int64 total_amount_minor_units (cents, for USD); a currency-aware field follows the shape Google's own google.type.Money uses — an int64 units, an int32 nanos for the sub-unit remainder, and a string currency_code. Either way, the arithmetic happens in integers, where it's exact.

The same instinct — prefer the type that can't quietly lie to you — applies elsewhere. Reach for int64 over int32 for IDs and counters; the extra headroom against overflow costs you nothing you'd notice and saves you a very bad day later. Reach for bytes instead of string for an identifier that isn't guaranteed to be text — raw UUID bytes, a hash, an opaque token — because a proto3 string field is validated as UTF-8, and binary data either fails that check or pays for a pointless text re-encoding to satisfy it. string stays exactly right for identifiers that really are human-readable text. None of this is new mechanism; it's just the field-type table from Chapter 2, read with an eye for which row actually matches the data.

The bug that ships quietly A double price field will work fine in every demo, every unit test with round numbers, and most of production — right up until someone sums ten thousand line items and the total is off by a cent. By then it's shipped, and it's in an invoice.

Time: Timestamp and Duration

The same “don't roll your own” instinct applies to time even more strongly, because a bare integer field is ambiguous in ways a comment can't fully fix: is created_at seconds or milliseconds? UTC or local? What happens at the year 2038 boundary if it's a 32-bit field? google.protobuf.Timestamp answers all of that once, in one place, so every consumer inherits the same answer. It's a point in time — seconds since the Unix epoch, plus non-negative nanosecond fractions — valid from 0001-01-01T00:00:00Z to 9999-12-31T23:59:59.999999999Z, with leap seconds “smeared” linearly over 24 hours so no reader needs a leap-second table to make sense of it. In JSON it's an RFC 3339 string: "2017-01-15T01:30:15.01Z" is exactly 15.01 seconds past 01:30 UTC on that date.

google.protobuf.Duration is Timestamp's companion, not a competitor: a signed, calendar-independent span of time (again seconds plus nanos, range roughly ±10,000 years) rather than a point on the calendar. The two compose the way you'd hope — subtract two Timestamps and you get a Duration; add a Duration to a Timestamp and you get another Timestamp. In JSON, a Duration is a string ending in s: "3s", or "3.000000001s" for three seconds and one nanosecond.

One sign rule is easy to get backwards the first time you hand-build a Duration: whenever seconds is non-zero, nanos must be zero or share its sign. A duration of −1.5 seconds is seconds=-1, nanos=-500000000 — not nanos=+500000000, which would mix signs. Only for sub-second durations does seconds sit at zero while nanos alone carries the sign.

The presence problem, revisited: wrapper types are legacy now

Chapter 2 mentioned the wrapper messages — Int32Value, StringValue, BoolValue, and their siblings — as the “old-fashioned way” to get presence on a scalar: wrap it in a message, since message fields always have explicit presence. That was true, and for years it was the only way. It isn't anymore. Protobuf's own doc comments on the wrapper types are blunt about it: they were “needed for legacy reasons” and are “not recommended for use in new APIs,” now that proto3's optional keyword gives a plain scalar field real, native presence — no wrapping required, no .value field to unwrap on every read. The same doc comments go further: a wrapper has no meaningful use inside a repeated field, a map, or a oneof, because each of those already has its own way to detect presence per element, per entry, or per member.

So when would you still reach for one? Exactly one place: packing a bare scalar into an Any, which is coming up next. Any can only hold a message, so a lone int32 or string has nowhere to live unless it's wrapped in something message-shaped first — and protobuf's guidance, even there, is to prefer a small, purpose-documented wrapper message of your own over the generic Int32Value for that job going forward. Outside of that one corner, if you're writing a new proto3 field and you want presence, write optional int32, not Int32Value.

A promise, kept a little differently Chapter 2's dangling thread said wrappers solve presence “the old-fashioned way.” That phrasing was earning its keep: it's now genuinely the old way. The new way is a keyword you already know.

Any: honest extensibility, or an escape hatch?

google.protobuf.Any holds two fields: a type_url string naming the message type it contains (something like type.googleapis.com/google.protobuf.StringValue — the part after the last / is the type's fully-qualified name, which is the only part that actually matters), and a bytes value holding that message's serialized bytes. Three operations cover everything you do with one: pack wraps a message into an Any; unpack reads it back into a typed message, checking type_url first; is tests the type without unpacking. The URL prefix is bookkeeping only — protobuf never built a type-resolution service that actually lives at those URLs, and every implementation is explicitly told not to try contacting one.

Its JSON form has a matching two-tier shape: for a type without a special-cased JSON encoding, an Any looks like that message's own JSON with an extra @type key holding the type URL; for a well-known type that does have a special JSON form (Timestamp, the wrappers, Struct, and so on), the Any's JSON instead carries @type plus a value key holding that special form.

Here's the part worth actually weighing before you reach for Any: it discards the static type safety protobuf otherwise gives you for free. Every consumer still has to know the closed set of types it might receive to do anything useful with the unpacked result — so the “openness” is frequently an illusion. If that set is known and controlled by one schema, a oneof of concrete message types beats Any almost every time: you get compiler-checked exhaustiveness and a direct typed accessor instead of string-matching a type_url. Any earns its place when producer and consumer genuinely don't share one schema — an event envelope on a bus where publishers evolve independently of subscribers, a multi-tenant system injecting different payloads per tenant, a plugin point the schema author can't enumerate in advance.

The question to ask first Before reaching for Any: “do I actually know every type that could show up here, and do I own that list?” If yes, that's a oneof. Any is for when the honest answer is no.

Struct and Value: the JSON-shaped escape hatch

Struct is an unordered map<string, Value>; Value is a oneof over null_value, number_value, string_value, bool_value, struct_value (recursive), and list_value. Put those together and you can round-trip an arbitrary JSON object through a protobuf field — which is exactly, and only, the job it exists for. Protobuf's own doc comment doesn't hedge about this: “If you do not intend to parse arbitrary JSON into your message, a custom typed message should be preferred instead of using this type.” It also inherits JSON's own limits, since it's modeled on RFC 8259 — no NaN, no Infinity, and no large int64 values that JSON's number type can't carry exactly.

The cost shows up the moment the shape is known. Every field access through a Struct is a string-keyed map lookup followed by a oneof kind-check, instead of a direct typed field read — you've traded away the exact thing a schema is supposed to buy you. The practitioner consensus matches the doc comment: reserve Struct/Value for data that's genuinely dynamic or arriving as passthrough from somewhere you don't control — a config blob, a feature-flag payload, a third-party webhook body — and define a real message the moment you can describe the shape in a sentence.

FieldMask: naming which fields you actually touched

FieldMask is almost aggressively simple — one field, repeated string paths — and it earns its whole chapter entry from how that one field gets used. In a get or list response, a mask filters the result down to just those paths (everything else reads back at its default and is dropped from proto text output). In an update, it's the mechanism behind partial updates: the request carries a full message plus a mask naming exactly which fields to write, and the server is expected to leave every field not named in the mask untouched, even if the request message happens to carry other populated values. This is the standard shape behind a PATCH-style RPC — formalized in Google's own API design guidance as the update-mask pattern — and it's the cleanest answer to a question every CRUD-shaped service eventually asks: how do you update three fields on a hundred-field resource without re-sending, and risking overwriting, the other ninety-seven?

Two structural rules matter once the target has nested messages or repeated fields. A repeated field named by a path gets new values appended to what's already there, not replaced wholesale — and a repeated field is only legal at the last position of a path, never in the middle. A sub-message named at the last position is merged into the existing sub-message rather than replacing it outright. Both defaults can be overridden by a particular implementation, but that's the behavior to assume unless you're told otherwise. 

// target before: f { b { d: 1  x: 2 }  c: [1] }
// update message:   f { b { d: 10 }       c: [2] }
// mask: paths: ["f.b", "f.c"]

// result: f { b { d: 10  x: 2 }  c: [1, 2] }
//         (b merged, keeping x=2; c appended, not replaced)
The casing gotcha that ships anyway A path uses the field's real lower_snake_case name in a proto message — but the JSON encoding of a FieldMask is one comma-separated string with each path converted to lowerCamelCase: mask: "user.displayName,photo", not "user.display_name,photo". A path built in the wrong casing for its transport doesn't error — it just quietly fails to match anything, and that's a real, recurring class of bug at REST/JSON gateway boundaries.

Two more edges worth knowing before you ship an update RPC. Google's own API guidance expects update methods to support the special path "*" to mean “replace every field,” and to treat a wholly omitted mask as “replace only the fields present in the request” — but different FieldMask library implementations handle an empty mask or "*" inconsistently in practice, so confirm your library's actual behavior rather than assume it matches the spec's intent. And a well-behaved server is expected to validate every incoming path against the target message's real fields, returning an error for one that doesn't map to anything — skip that check, and a typo'd path silently does nothing instead of failing loudly.

Maps: convenient, with real limits

You met map<K, V> in Chapter 2 as sugar over a repeated key/value message. The restrictions on K and V are worth having memorized before you design around one. A key must be an integral or string/bool scalar — no floating point, no bytes, no message, no enum. A value can be almost anything except another map: map<string, map<string, int32>> simply doesn't compile. Need nested maps anyway? Wrap the inner one in a message and map to that.

The other limit is about behavior, not syntax: a map's iteration and wire order are unspecified, and language runtimes don't agree on how they fill that gap. Go's protobuf library only sorts map entries when a caller explicitly asks for deterministic serialization; that guarantee has been inconsistent, or missing outright, in other language implementations at various points. If a field genuinely needs stable, repeatable iteration order or ordered output, don't fight the map for it — model it as repeated SomeEntry with explicit key and value fields, the same shape a map compiles down to anyway, minus the ordering ambiguity.

Repeated fields don't have opinions about presence

Chapter 2 established that a plain scalar field can't tell “never set” apart from “set to the zero value” — that's implicit presence. A repeated field has the exact same blind spot, just one level up: an empty repeated field and a repeated field that was never touched are wire-indistinguishable, for repeated int32 just as much as for repeated SomeMessage. There's no such thing as repeated optional int32, and there never will be — presence, in protobuf, is a property of a field slot, and a repeated field is a list of values with no individual slots to be absent from.

So what do you do when a list needs presence — a list of measurements where some readings are missing, a list of settings that will eventually need more per-entry metadata than a bare value can carry? You give each element its own slot by making the element a message: repeated Reading, where Reading itself has an optional double value, instead of repeated double. A repeated message field gets per-element presence and per-element room to grow for free, because each element is now a whole message with its own field-presence rules — something a repeated scalar field structurally cannot offer, no matter how you squint at it.

Zero means “didn't say” — enums, done right

Chapter 2's Status enum started its list at STATUS_UNSPECIFIED = 0 rather than, say, IN_STOCK = 0, and we noted the compiler requires some value at zero. Here's the design reasoning behind reserving it for an explicit sentinel instead of spending it on a real business value: an unset enum field silently reads back as its zero value, exactly like an unset scalar. If zero also means something — “in stock,” say — then “the caller never set a status” and “the caller explicitly said in stock” become the same bit pattern, and there is no way, ever, for downstream code to tell them apart. Reserving zero for _UNSPECIFIED/ _UNKNOWN is the enum version of the same discipline optional gives you for scalars — it just can't be opted out of, because the compiler makes you pick a zero value one way or the other.

There's a second reason this convention matters, and it's about the future, not just today: proto3 enums are open. A field declared as an enum can legally hold any int32, not only the values named in your .proto — which is what lets a newer peer add DISCONTINUED = 5 without breaking an older reader that's never heard of it. What that older reader actually sees depends on its language, and this is worth knowing before it surprises you in production: C++ and Go simply hand back the raw, out-of-range integer through the normal accessor — not coerced to UNSPECIFIED, just an int your switch statement doesn't have a case for. Java's generated enums are effectively closed by comparison and instead surface a synthetic UNRECOGNIZED constant for any value they don't recognize, while still exposing the real integer through a separate accessor if you go looking for it. Either way, the original number survives a round trip through re-serialization untouched — it's only a rebuild-from-scratch, field-by-field copy that would lose it, the same caveat unknown fields carried back in Chapter 4.

Design for the reader who hasn't updated yet An UNSPECIFIED zero value and an open enum together mean: old code degrades to “I don't recognize this value” instead of crashing, misreading, or quietly treating a new value as the default. That's the whole point — and it only works if you actually reserved zero for it.

So what did we actually gain?

Put together, everything in this chapter is one habit wearing different outfits: before reaching for a type, ask what it can't tell you. double can't tell you it rounded your money. Plain int32 can't tell you whether a status was set on purpose. A bare repeated double can't tell you an entry is missing rather than zero. Any can't tell your compiler what it actually contains. Struct can't tell a reader what shape to expect. Protobuf's well-known types and language features — Timestamp, Duration, optional, oneof, the UNSPECIFIED convention — exist specifically to close those gaps, one at a time, and each one is worth its small ceremony exactly when the data genuinely needs what it buys.

None of this stays true forever without discipline, though. A schema that models today's data well can still break tomorrow's readers if a field number gets reused or a type quietly changes shape. That's exactly the discipline the next chapter is about.
