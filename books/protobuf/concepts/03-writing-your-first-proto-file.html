Writing Your First .proto File

Last chapter closed with a promise: you now have the vocabulary — field, number, presence, shape, tag — to write one of these schemas yourself. A vocabulary isn't a file, though. message Product { ... } on its own doesn't compile into anything, because a real .proto file needs a bit of ceremony around the message before protoc will touch it. This chapter is that ceremony, end to end — from the first line of the file to the command that turns it into code you can actually import.

Every file starts with the same two lines

Before a single message appears, a proto3 file opens with two declarations: 

syntax = "proto3";
package catalog.v1;

syntax = "proto3"; has to be the first non-comment statement in the file, and skipping it isn't a safe shortcut — it silently tells protoc to compile the file as proto2 instead, an older syntax with different field-presence defaults and a required keyword that proto3 deliberately removed. You want this line every time, not just when you remember it.

package catalog.v1; is the namespace line from last chapter, now doing its actual job: every type this file defines — Product, its nested Status and Dimensions — is scoped under catalog.v1 once compiled, so a different team's billing.v1.Product can coexist without a collision.

The mistake this prevents Forget syntax = "proto3"; and nothing crashes. The file just quietly becomes proto2. You'll notice something's off only once a teammate asks why required compiles, or why a scalar field's presence behaves differently than this book describes — a confusing bug to track back to one missing line.

Telling each language where the code should land

A .proto file is compiled once but read by many languages, and each of those languages has its own idea of "where generated code lives" — a Java package, a Go import path, a C# namespace. File options are how you tell protoc that, one language at a time, without touching the schema itself. Here's the real thing: the options block from Google's own tutorial .proto file.

From protocolbuffers/protobuf's tutorial file, addressbook.proto

option java_multiple_files = true;
option java_package = "com.example.tutorial.protos";
option java_outer_classname = "AddressBookProtos";
option csharp_namespace = "Google.Protobuf.Examples.AddressBook";
option go_package = "github.com/protocolbuffers/protobuf/examples/go/tutorialpb";
OptionSteers…
java_package / java_multiple_files / java_outer_classnamewhich Java package the generated classes land in, and whether they're one file per message or bundled into a single outer class
go_packagethe Go import path for the generated package
csharp_namespacethe C# namespace for the generated types
Key idea None of these options change one byte on the wire. Two peers — one compiled with this go_package, one with a different one — exchange identical bytes. What changes is purely where each language's generated code lives on disk and how you import it in your own program.

That "no effect on the wire" framing has a sharp edge, though: changing go_package on a file that's already shipped isn't free just because the wire format didn't move. It changes the generated package's import path, so every existing import "github.com/…" line in every consumer's code now points at the wrong place. Treat a file option change with the same care you'd give a package rename in any other language — because that's exactly what it is.

In practice, go_package is the option you'll miss and regret: without it, protoc's Go plugin has nothing reliable to base an import path on, and generation either fails or falls back to guessing. If you're generating Go code at all, write this option in from the start.

Borrowing a type instead of reinventing it

The tutorial file also has this line, sitting right after package: 

import "google/protobuf/timestamp.proto";

import pulls another .proto file's types into scope, exactly the way you'd expect. google/protobuf/timestamp.proto is one of a small set of well-known types that ships bundled with your protoc installation — you don't vendor it, write it, or download it; it's just there, which is why an import of it resolves with zero extra setup.

Import your own files the same way, but know how protoc finds them: an import path is resolved against the include directories you hand it on the command line with -I (equivalently --proto_path), not against wherever the importing file happens to sit on disk. Skip -I entirely and protoc falls back to the current working directory — which is exactly the kind of thing that works on your machine and fails the moment CI runs the same command from a different folder.

Comments the compiler throws away, on purpose

.proto supports the comment styles you already know from C-family languages: // to end of line, /* … */ across several lines. protoc discards every one of them — they never reach the compiled descriptor, let alone the wire. Their entire job is to help the next human. The tutorial file's own style is worth copying: a short, specific comment trailing the field it documents. 

int32 id = 2;  // Unique ID number for this person.

That one line answers a question the type alone can't: not just "what's the type of id," but "what does this number actually mean." Multiply that by every non-obvious field in a schema three teams share, and comments stop being decoration.

Putting it together: catalog/v1/product.proto

Here's last chapter's Product message, now dressed in everything above — a real, compilable file, not a fragment.

catalog/v1/product.proto

// Product catalog schema. Owned by the Catalog team.
syntax = "proto3";

package catalog.v1;

option go_package = "example.com/catalog/gen/catalogpb";
option java_package = "com.example.catalog.v1";

message Product {
  reserved 4;
  reserved "sku_legacy";

  string name = 1;
  int32 quantity = 2;  // units on hand, not units sold
  optional string description = 3;

  enum Status {
    STATUS_UNSPECIFIED = 0;
    IN_STOCK = 1;
    BACKORDERED = 2;
  }
  Status status = 5;

  repeated string tags = 6;

  oneof pricing {
    int64 price_cents = 7;
    string price_quote_id = 8;
  }

  map<string, int32> warehouse_counts = 9;

  message Dimensions {
    float width_cm = 1;
    float height_cm = 2;
  }
  Dimensions dimensions = 10;
}

Notice the filename in that caption: catalog/v1/product.proto, lowercase with underscores, one clear match for the package catalog.v1; line inside it. That pairing — lower_snake_case.proto file names, TitleCase message and enum type names, lower_snake_case field names, UPPER_SNAKE_CASE enum values — is the style every .proto file in this book follows, and it's worth adopting even before a linter makes you.

From text file to real code: running protoc

A .proto file is inert until protoc reads it. The shape of every invocation is the same: 

protoc -I=<import-path> --<lang>_out=<output-dir> <path/to/file.proto>

-I sets the include path import statements resolve against (and where your input file's path is relative to); --<lang>_out is where that language's generated source lands. At least one --<lang>_out flag is required — protoc refuses to run with none — and you can stack several in one call to generate every language you need from a single pass over the file: 

protoc -I=. --go_out=. --go_opt=paths=source_relative catalog/v1/product.proto
protoc -I=. --java_out=./gen/java catalog/v1/product.proto
protoc -I=. --python_out=./gen/py catalog/v1/product.proto

--go_opt=paths=source_relative is worth calling out: it's a language-specific modifier (not every plugin has one) that changes how the output path is derived, in this case mirroring the input file's own relative path instead of building one from the Go import path. Flags like it exist because "generate the code" hides a surprising number of small, language-specific decisions.

You may need a plugin, not just protoc C++, Java, Python, C#, Objective-C, and PHP are built into protoc itself. Go, and several other languages, are not — generating them means installing a separate plugin binary (Go's is protoc-gen-go) on your PATH first. protoc shells out to it the moment it sees a matching --go_out flag; without the binary installed, that flag simply fails.

A glance at what comes out the other side

We'll spend a whole chapter on generated code and tooling shortly, so just the shape of it for now. Every target language gets, roughly: a type per message (a struct in Go, a class in Java and Python), a type per enum, and a getter-style accessor per field — plus, for any field marked optional, a way to ask whether it was actually set. In Go, illustratively, that Product message becomes something in this shape:

Illustrative, simplified — actual generated code carries more scaffolding

type Product struct {
    Name        string
    Quantity    int32
    Description *string  // pointer: nil means "not set"
    Status      Product_Status
    Tags        []string
    // oneof, map, and nested-message fields follow the same pattern
}

func (p *Product) GetQuantity() int32 { return p.Quantity }

Notice Description is a pointer, not a plain string — that's exactly the explicit-presence machinery from last chapter, made concrete: nil means "never set," a pointer to "" means "set, to empty." Everything you learned about the model in Chapter 2 is about to keep showing up like this, translated into whatever language you generate for.

So what did we actually gain?

We took the abstract vocabulary from last chapter and gave it a real file, a real filename, a real compile step, and real generated code in more than one language. None of the ceremony — syntax, package, file options, import, comments — is arbitrary busywork. Each piece answers one concrete question a schema shared across teams and languages actually needs answered: which language rules apply, where does this fit among every other schema, where does each language's generated code live, what other types am I borrowing, and what does this field mean to the human reading it.

What we haven't done yet is watch Product actually become bytes. We've talked around the wire format for two chapters now — tags, field numbers, wire types — and it's time to stop gesturing at it and look at the bytes themselves.

  2. You change option go_package on a file three services already import. Concretely, what breaks, for whom, and when do they find out?
