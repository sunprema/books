Chapter 10 · Evolution & Interop

Working Across Languages

Here is the promise that has been humming under every chapter so far, finally said out loud: a Go service writes a message, a Python worker reads it, a TypeScript front-end renders it, and a C++ batch job archives it — and not one of them shares a line of code with the others. All they share is a .proto file. In this chapter we'll take that promise apart to see why it holds. We'll watch one schema become idiomatic types in four different languages, meet the second wire format (canonical JSON) that every implementation must agree on, and then spend real time in the places where the abstraction leaks — presence, 64-bit integers in JavaScript, and enums — because knowing exactly where interop almost works is the difference between a system that ships and one that corrupts data at 3 a.m. We'll close with the machinery that keeps every language honest: the conformance suite.

The contract is the schema, not the library

Let's fix the mental model first, because everything else falls out of it. When two programs in different languages talk protobuf, they are not sharing a library. There is no common runtime they both link against, no shared object passed between them. What they share is the .proto file — a language-neutral description of the data — and two things that every conformant implementation independently agrees to produce: the same binary wire format and the same canonical JSON mapping. That's the whole contract. The Go library and the Python library were written by different people, in different decades, and they interoperate because they both obey the same spec, not because they descend from the same code.

The one-sentence version A .proto file plus a per-language plugin generates native types in each language; interoperability comes from every implementation independently honoring the same two serialization formats — binary and canonical JSON — not from any shared code. The schema is the API; the generated code is disposable.

This is why, back in the tooling chapter, running protoc for a new language felt so anticlimactic: you point the compiler at the same file with a different --<lang>_out flag and get idiomatic code for free. Each generator's job is to make the schema feel native — like a library a human wrote for that language — while keeping the bytes on the wire bit-for-bit identical to every other generator's.

One schema, many idioms

Let's make it concrete with a message we'll carry through the whole chapter. It's deliberately loaded: a 64-bit id, an enum, a bytes field, and a timestamp — the four fields that expose almost every cross-language wrinkle. 

syntax = "proto3";
package catalog.v1;

import "google/protobuf/timestamp.proto";

enum Availability {
  AVAILABILITY_UNSPECIFIED = 0;   // the required zero value
  AVAILABILITY_IN_STOCK    = 1;
  AVAILABILITY_BACKORDER   = 2;
}

message Product {
  int64  sku          = 1;   // a 64-bit id
  string display_name = 2;
  Availability status  = 3;
  bytes  thumbnail    = 4;
  google.protobuf.Timestamp updated_at = 5;
}

Notice one style rule doing quiet work here: in the .proto the field is display_name, lower_snake_case, always. We never write the field the way any one language spells it, because the file belongs to all of them. It's each generator's job to translate that single canonical name into whatever reads as natural locally. Watch the same sku field land in four languages: 

// proto:   int64 sku = 1;        // one field, four renderings

// Go     — exported CamelCase field, nil-safe getter
p.GetSku()                 // int64

// Java   — getter/Builder setter off the snake_case name
p.getSku();                // long;  builder.setSku(long)

// Python — a plain snake_case attribute
p.sku                       # an int, arbitrary precision

// C#     — a PascalCase property
p.Sku;                     // long { get; set; }

Each generator also matches its language's grain in bigger ways. Go emits a struct with exported CamelCase fields and GetX() accessors that are safe to call on a nil pointer. Java gives you an immutable message plus a mutable Builder to construct it, with getSongName()-style accessors. Python's generated module is really just a static descriptor — the actual class is conjured at import time by a metaclass, with the fast upb C library doing the real work underneath since protobuf 4.21. C# produces PascalCase properties and even trims the enum's own name off its values so AVAILABILITY_IN_STOCK reads as InStock. Five faces, one schema.

Two live options dominate — ts-proto, which emits lean idiomatic interfaces and functions, and Protobuf-ES (@bufbuild/protobuf), which chases full spec conformance. It's the one ecosystem where you pick the runtime, so pick deliberately.

Where it first leaks: field presence

Now the leaks. The first one is subtle and it's caused by a decision we made chapters ago: proto3's implicit presence. For a plain scalar field — no optional label — a value equal to the default (0, "", an empty list) is indistinguishable from never having been set. On the wire, a zero-valued field simply isn't written. And here's the interop bite: this shows up differently in each language's API even though the bytes are identical.

In Go, an implicit int64 sku is just an int64 — there's no HasSku(), so you literally cannot ask whether the sender meant “zero” or “didn't set it.” Java is the same for implicit fields: no hazzer. The fix is the same everywhere and we've seen it before — add the optional label to opt into explicit presence, and now every language generates a hasSku()/HasSku() that tells the truth. 

// implicit presence (proto3 default): 0 and "unset" are the same fact
int64 sku = 1;

// explicit presence: adds has-methods in every language
optional int64 sku = 1;   // Go: HasSku()  Java: hasSku()  ...
The presence gotcha that crosses a language boundary If your Go producer leaves sku at 0 and your Python consumer branches on “is sku present?”, the two disagree about reality — not because of a bug, but because implicit presence defines 0 as absent. Reach for optional the moment “set to the default” and “not set” must mean different things. (One oddity worth knowing: Dart is the outlier that generates has_ methods even for implicit proto3 fields — don't let it lull you into assuming the others do too.)

The second wire format: canonical JSON

So far “the wire” has meant the compact binary encoding from Chapter 04. But protobuf actually mandates two serializations, and every conformant library must implement both: the binary format, and a canonical JSON mapping — often called ProtoJSON. It exists for the moments binary can't reach: a browser's dev-tools, a REST gateway, a log line a human has to read, a config file. Crucially it is a specified mapping, not “whatever your library felt like emitting,” which is exactly what lets a C# service and a TypeScript client exchange JSON and agree on what it means.

Here is our Product as ProtoJSON. Read it slowly — almost every token is a rule you have to know: 

{
  "sku": "90071992547409910",          // int64 -> a QUOTED string
  "displayName": "Field Notebook",       // display_name -> lowerCamelCase
  "status": "AVAILABILITY_IN_STOCK",    // enum -> its value NAME
  "thumbnail": "iVBORw0KGgoAAA==",       // bytes -> base64
  "updatedAt": "2026-07-20T17:04:00Z"    // Timestamp -> RFC-3339 UTC string
}
// A field left at its default (sku==0, empty string) is OMITTED entirely.

Let's name the rules, because “it's just JSON” is the fastest way to ship a bug. Field names become lowerCamelCase (or the json_name option's value if you set one); a good parser accepts both that and the original snake_case. Enums serialize as their value name by default (a number is also legal on input). bytes is base64. Floating point is a normal JSON number — except the three values JSON can't spell, which become the strings "NaN", "Infinity", and "-Infinity". And the one that surprises everyone: the 64-bit integer types (int64, uint64, fixed64, sint64…) are emitted as quoted strings. Hold that thought — it's about to earn a whole section.

The well-known types have their own JSON forms A handful of google.protobuf types get bespoke ProtoJSON shapes you can't guess from the message-as-object rule: Timestamp is an RFC-3339 UTC string ending in Z; Duration is decimal seconds with an s suffix ("3.5s"); Struct/Value/ListValue round-trip to plain JSON; the wrapper types collapse to just their wrapped value (or null); and Any becomes an object carrying a special "@type" key. A generic JSON consumer that doesn't know these will be baffled by them — and that's a very common integration surprise.

The 64-bit integer trap in JavaScript

Why on earth does ProtoJSON wrap 64-bit integers in quotes? Because of one of the most consequential facts in cross-language data: JavaScript has no integer type. Every JS Number is an IEEE-754 double, which can represent integers exactly only up to 2^53 - 1. Past that, arithmetic silently lies. A uint64 is a perfectly ordinary id in Go or Java; drop it into a JS number and it rounds out from under you: 

// Why ProtoJSON strings 64-bit ints — run this in any JS console:
JSON.parse('{"sku": 90071992547409910}').sku    // 90071992547409920  <-- WRONG
JSON.parse('{"sku": "90071992547409910"}').sku  // "90071992547409910" exact
Number.MAX_SAFE_INTEGER                          // 9007199254740991  (2^53 - 1)

Look at the first line: the last two digits changed, with no error, no warning. An order id, a ledger balance, a Snowflake timestamp — quietly corrupted the instant it crossed into a JS number. The quoted-string mapping in ProtoJSON is the spec's defense against exactly this. And it's why serious JS protobuf runtimes never hand you a bare number for a 64-bit field: they give you a string, a Long object, or, in Protobuf-ES, a native BigInt. The value is preserved; you just have to remember it isn't a plain number.

The rule that saves you In any JavaScript/TypeScript that touches protobuf, treat every int64/uint64/fixed64 as opaque — keep it a string or BigInt, never coerce it to Number, and never let it flow through JSON.parse as an unquoted numeric literal. The bug is invisible in tests with small ids and catastrophic in production with real ones.

Wire-compatibility — and its one sharp edge

Here's the reassuring half. The binary wire format is the same algorithm in every language — a message is nothing but a flat stream of field # · wire type · value records, as we dissected in Chapter 04. There is no per-language dialect of the bytes. So anything a Go service serializes, a Python service parses, and vice versa, forever, without either knowing the other exists. This is the bedrock the whole promise rests on, and it genuinely just works.

Now the sharp edge, and it cuts people who assume “same message ⇒ same bytes.” It does not. Protobuf gives you no guarantee that two implementations — or even two runs of the same implementation — produce byte-for-byte identical output for equal messages. Fields may be emitted in any order. Most notoriously, map iteration order is unspecified and routinely differs. Every reader still parses every ordering correctly, because the field numbers carry the meaning. But the bytes are not canonical.

Never treat serialized bytes as a canonical fingerprint Do not hash serialized protobuf and compare the hashes for equality across languages, across library versions, or even across two runs. Do not sign the raw bytes and expect another language to reproduce them. Do not deduplicate records by their serialized form. The moment a second implementation (or an upgraded map library) touches the data, “equal message, different bytes” breaks all three. If you truly need a stable digest, normalize into a canonical form yourself — the wire format won't do it for you.

Enums across runtimes: open vs. closed

Enums are the last big leak, and it comes from forward compatibility. Picture a new server that adds AVAILABILITY_PREORDER = 3 and sends it to an old client whose schema stops at 2. The value 3 is unrecognized — so what does the old client do with it? The answer depends on the language, and it's the difference between open and closed enums.

In open enums — C++ and Go — the runtime keeps the raw integer 3 as-is; the field just holds a value your generated enum doesn't have a name for. In Java's generated closed-style enums, the value maps to a synthetic UNRECOGNIZED sentinel, and you fish the original number out through a separate accessor. Either way — and this is the saving grace — the original 3 round-trips: if the old client reserializes the message untouched, 3 goes back out on the wire intact. But your switch statement won't agree across languages.

The enum switch that misbehaves in exactly one language Always have a default / unknown arm when you branch on an enum. A value from the future arrives as a raw int in Go and C++ but as UNRECOGNIZED in Java, so a naive switch with no fallthrough handles it in one language and silently mis-handles it in another. And never renumber or reuse enum values across a schema change — reserved the retired names and numbers, exactly as the evolution chapter warned. (The zero value stays special: it must be the first entry and it's the default for implicit-presence fields.)

Keeping everyone honest: the conformance suite

One last question hangs over all of this: if there's no shared code, only a spec, how do we know a brand-new Rust or Swift or TypeScript library actually obeys it? You test it — against the same battery of tests every other language faces. That battery is the protobuf conformance suite: over 2,000 cases that probe binary and JSON, parsing and serializing, well-known types, the awkward edges of presence and defaults.

The mechanism is beautifully low-tech. A tester process speaks to your implementation over stdin/stdout. It writes a ConformanceRequest — a payload plus “now give it back to me in this format” — and reads back a ConformanceResponse with either the re-serialized bytes or an error. Your job is a tiny harness that parses the payload and re-emits it; the C++ reference is about 150 lines. Pass all 2,000+ and your library is, by definition, interoperable. 

// conformance.proto — the contract for testing a contract (abridged):
message ConformanceRequest {
  oneof payload {
    bytes  protobuf_payload = 1;
    string json_payload     = 2;
  }
  WireFormat requested_output_format = 3;
  string     message_type            = 4;
}
message ConformanceResponse {
  oneof result {
    string parse_error      = 1;
    bytes  protobuf_payload = 3;
    string json_payload     = 4;
    string skipped          = 5;
  }
}
enum WireFormat { UNSPECIFIED=0; PROTOBUF=1; JSON=2; JSPB=3; TEXT_FORMAT=4; }
So what did we actually gain? A genuinely language-neutral contract — and a clear-eyed map of where it holds and where it leaks. It holds because every implementation obeys the same binary and JSON specs, so one schema becomes idiomatic Go, Java, Python, C#, and TypeScript that all interoperate without sharing a line of code, and the conformance suite proves each one honest. It leaks in a handful of named, learnable places: implicit presence makes “zero” and “unset” the same fact, JavaScript can't hold a 64-bit integer, serialized bytes aren't canonical, and a future enum value lands differently in open vs. closed runtimes. Know those four and cross-language protobuf stops being magic and becomes engineering.
