gRPC and Service Definitions

For eight chapters protobuf has been a way to turn a message into bytes and back. But look again at the .proto grammar and you'll notice two keywords we've been stepping around — service and rpc — that don't describe data at all. They describe verbs: things a server can do. That's the seam where protobuf hands off to gRPC. In this chapter we'll define a service, meet the four shapes an RPC can take, watch protoc spin your schema into a client stub and a server skeleton, and follow a single call all the way down to the HTTP/2 bytes and back up through a status code. Most of all we'll draw one line clearly: exactly where protobuf ends and gRPC begins — because confusing the two is the single most common misunderstanding in this whole corner of the field.

First, the line between them

Let's settle the big confusion up front, because everything else is easier once it's fixed in your head. Protobuf is a serialization format: a schema language, a binary wire encoding, and the compiler that turns one into the other. You can use it with a file on disk, a Kafka topic, a Redis value — no network required. gRPC is a Remote Procedure Call framework: a runtime for calling a method on another machine as if it were local. Google built it from an internal system called Stubby and open-sourced it in 2015; it's now a CNCF project. gRPC uses protobuf as its default interface definition language (IDL) and its default message format — but the two are genuinely separable layers.

The one-sentence version Protobuf owns the nouns and the bytes — message schemas, the wire encoding, and the syntax of service/rpc/stream. gRPC owns the verbs and the runtime — how a call travels (HTTP/2), what streaming actually does, deadlines, metadata, the status/error model, and the generated stubs. The keywords parse under plain proto3, but they only mean something when a gRPC-aware plugin generates code against them.

That last sentence has teeth, so let's make it concrete now and pay it off later: if you run protoc with only a message plugin over a .proto full of service blocks, it compiles happily and generates nothing at all for the services. No stub, no skeleton, no error. The service definitions are inert until something that understands them — gRPC's code generator, or another RPC system's — is asked to look. Protobuf will describe your API; it won't wire it up.

Defining a service

A service is a named bundle of methods. You declare it with the service keyword, and inside it each method is an rpc line: a name, one request message in parentheses, and one response message after returns. Notice the shape of that contract — exactly one message in, exactly one message out. There's no such thing as an RPC that takes two arguments or returns a bare integer; the parameter and the return are always message types. 

syntax = "proto3";
package routeguide.v1;

service RouteGuide {
  // A method is a verb the server offers. One message in, one out.
  rpc GetFeature(Point) returns (Feature);
}

message Point   { int32 lat = 1; int32 lng = 2; }
message Feature { string name = 1; Point location = 2; }

This is where the schema discipline from the last two chapters comes home to roost. Because a request and response must be messages, you get, for free, the one thing that lets an API age well: room to grow. That GetFeature(Point) returns (Feature) can never also return a freshness timestamp, because Feature is about a feature. The fix — and the reason the best-practices chapter told you to wrap every RPC in a dedicated GetFeatureRequest/GetFeatureResponse even when it feels like ceremony — is that the wrapper is the only place you can add a field later without changing the method signature, and changing a signature breaks every client. gRPC is why that rule exists.

The four kinds of RPC

Here's the part that feels like magic the first time and obvious ever after. An RPC doesn't have to be a single question and a single answer. gRPC gives you four shapes, and — this is the lovely bit — you select between them with a single keyword, stream, placed on the request side, the response side, both, or neither. 

service RouteGuide {
  // 1. Unary — one request, one response. A normal function call.
  rpc GetFeature(Point) returns (Feature);

  // 2. Server streaming — one request, a sequence of responses.
  rpc ListFeatures(Rectangle) returns (stream Feature);

  // 3. Client streaming — a sequence of requests, one response.
  rpc RecordRoute(stream Point) returns (RouteSummary);

  // 4. Bidirectional streaming — two independent streams.
  rpc RouteChat(stream RouteNote) returns (stream RouteNote);
}

Read those four lines top to bottom and the pattern is unmistakable: stream in front of the request means “the client sends many,” stream in front of the response means “the server sends many.” Everything else about the four is a consequence of that. Here they are as pictures, because the message flow is the whole point.

KindSignatureReach for it when…
Unaryrpc M(Req) returns (Resp)A plain request/response — the default, and most RPCs.
Server streamingrpc M(Req) returns (stream Resp)One query, many results: search hits, a subscription feed, a large export.
Client streamingrpc M(stream Req) returns (Resp)Many inputs, one summary: uploading chunks, batching metrics.
Bidirectionalrpc M(stream Req) returns (stream Resp)A live conversation: chat, telemetry with acks, interactive control.

Two clarifications that save confusion. A stream carries a sequence of the same message type — it is not a way to send different shapes; it's the same Feature, many times. And “streaming” here has nothing to do with how big the data is; it's about the conversation pattern. A tiny chat message and a huge file upload can both be streams; a hundred-megabyte response can still be unary. Pick the kind by the shape of the interaction, not the size of the payload.

Stubs and skeletons: what code generation gives you

Now the payoff for writing that service down. Feed the .proto to protoc with a gRPC plugin and, for each service, you get two artifacts. On the calling side, a stub (sometimes called the client): a local object whose methods are your RPCs. Call stub.GetFeature(point) and it marshals the argument to protobuf bytes, opens a call over the network, waits for the reply, decodes it, and hands you back a Feature. The remote call reads exactly like a local method call — hiding that gap is the entire reason the letters “RPC” are in the name.

On the serving side you get a skeleton: an abstract base class or service interface with one method per RPC, each one empty. Your job is to subclass it and fill in the business logic. gRPC's runtime does the rest — it listens, decodes the incoming bytes into a Point, calls your method, and serializes whatever you return. You write the what; the generated skeleton handles the how. 

// CLIENT — the generated stub makes a remote call look local (Go-ish):
resp, err := client.GetFeature(ctx, &pb.Point{Lat: 409146138, Lng: -746188906})
if err != nil { /* err carries the gRPC status code */ }
fmt.Println(resp.GetName())

// SERVER — you implement the generated interface; gRPC calls you:
func (s *server) GetFeature(ctx context.Context, p *pb.Point) (*pb.Feature, error) {
  return &pb.Feature{Name: s.lookup(p)}, nil
}
Two plugins, not one — the gotcha that bites everyone Generating messages and generating stubs are separate jobs done by separate plugins. In Go, protoc-gen-go emits the Point/Feature structs, and a different plugin, protoc-gen-go-grpc, emits the RouteGuideClient stub and RouteGuideServer interface. Run only the first and your services vanish with no warning — the exact “protoc ignores services” behavior we flagged at the top. Other languages bundle the two steps, but the mental model is the same: a message plugin plus a service plugin.

The transport: one call, over HTTP/2

So what actually happens when that stub method runs? gRPC's transport is HTTP/2, and the choice is not incidental — it's what makes the four RPC shapes possible. HTTP/2 multiplexes many independent streams over one TCP connection, frames everything in binary, compresses headers, and is full-duplex: both ends can send at once. A gRPC call is one HTTP/2 stream. That's why a bidirectional RPC isn't a special trick — it's just a stream where both sides keep sending DATA frames.

Mechanically, a call is an HTTP POST. The method name becomes the URL path — /<package>.<Service>/<Method>, e.g. /routeguide.v1.RouteGuide/GetFeature — and the content-type is application/grpc (often +proto). Your request messages travel in the stream's DATA frames, each one wrapped in a tiny five-byte header so the receiver knows where one message stops and the next begins:

Alongside the payload, HTTP/2 headers carry the metadata: on the way out, key/value pairs plus a grpc-timeout header that encodes the call's deadline — a first-class idea in gRPC. Set a deadline and a call that overruns it fails fast with DEADLINE_EXCEEDED instead of hanging forever; forget to set one and a dead server can wedge your client indefinitely. (Binary metadata is allowed too: keys ending in -bin are base64-encoded.) Stubs issue their calls over a channel — a managed, reusable connection to an endpoint — so you're not paying for a new TCP handshake per call.

How a call ends: status codes and errors

Here's a detail that surprises people who come from REST: the HTTP status of a gRPC response is 200, even when the call failed. The transport succeeded — a well-formed response came back — so at the HTTP layer everything is fine. The real outcome rides in a trailer: after the response messages, gRPC sends a final HEADERS frame carrying grpc-status (an integer) and an optional, percent-encoded grpc-message. If you check the HTTP status to decide whether your RPC worked, you will be wrong every time it matters. 

# Response trailers — the actual result of the call:
grpc-status: 5
grpc-message: Feature%20not%20found     # NOT_FOUND, percent-encoded

# (the HTTP :status was 200 the whole time)

Every call therefore ends with a status: a code plus a message. There are 17 canonical codes, 0–16, shared identically across every gRPC library and independent of whether the payload was protobuf or something else. You don't need to memorize all of them, but a working handful covers most days:

Code#What it meansSet by

INVALID_ARGUMENT3Caller sent a bad field.app

Who's allowed to say what Some codes only the runtime ever produces — DEADLINE_EXCEEDED, UNAVAILABLE, UNIMPLEMENTED, CANCELLED, INTERNAL, RESOURCE_EXHAUSTED, UNAUTHENTICATED, UNKNOWN. Others should only come from your application code: INVALID_ARGUMENT, NOT_FOUND, ALREADY_EXISTS, FAILED_PRECONDITION, ABORTED, OUT_OF_RANGE, DATA_LOSS. The practical lesson: don't reach for INTERNAL or UNKNOWN when a caller simply sent a bad argument — that's INVALID_ARGUMENT, and picking the right code is how a client knows whether retrying could ever help. (There's no universal “always retry these” set; it depends on your semantics and whether the call is idempotent.)
So what did we actually gain? A clean division of labor. Protobuf gave us a typed, evolvable schema and a compact wire format — the bytes. gRPC borrowed that schema as its IDL and built a runtime on top: it turns service and rpc into real client stubs and server skeletons, carries the messages over HTTP/2 with length-prefixed framing, offers four conversation shapes from a single stream keyword, and reports every outcome through one shared status model. Protobuf never had to know about any of it, and your business logic never had to know about sockets. That seam — schema on one side, transport on the other — is the whole idea.
