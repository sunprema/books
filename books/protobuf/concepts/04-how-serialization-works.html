Chapter 04 · Encoding & Tooling

How Serialization Works

We closed the last chapter with a promise to stop gesturing at the wire format and actually look at the bytes. So here's the question this chapter answers, as concretely as possible: you call product.SerializeToString() on a Product message with name = "ink" and quantity = 300. What comes back is eight bytes: 0A 03 69 6E 6B 10 AC 02. Every one of those bytes has a reason to be exactly what it is, and by the end of this chapter you'll be able to derive all eight by hand.

The tag: one small integer doing two jobs

Start with the smallest unit of the format: the tag. Every field, when it's about to be written to the wire, is preceded by one tag that packs two pieces of information into a single varint: 

tag = (field_number << 3) | wire_type

The field number is the same integer from Chapter 2 — the one thing that actually identifies a field on the wire, never the name. Wire type is new: a 3-bit code, 0 through 5, that says what shape of data follows. Six wire types cover every protobuf value:

CodeNameCarries
0VARINTint32/int64/uint32/uint64/sint32/sint64, bool, enum
1I64 (64-bit)fixed64, sfixed64, double
2LEN (length-delimited)string, bytes, embedded messages, packed repeated fields
3SGROUPdeprecated proto2 group start — not used in proto3
4EGROUPdeprecated proto2 group end — not used in proto3
5I32 (32-bit)fixed32, sfixed32, float

Notice what the wire type is for. It's fully determined by a field's declared type — every int32 field always gets wire type 0, every string field always gets wire type 2 — so it's redundant information as far as a schema-aware reader is concerned. But strip the schema away, and the wire type is the only thing telling a decoder how many bytes to skip. Hold onto that; it's the hinge the last section of this chapter swings on.

Varints: small numbers, small bytes

The tag itself is a varint, and so is most of the rest of the format, so let's actually build one. A varint spends one or more bytes on a number, and the rule per byte is simple: the low 7 bits carry data, and the high (8th) bit is a continuation flag — 1 means “another byte follows,” 0 means “you're done.” The 7-bit groups go out least-significant first. In pseudocode: 

encode_varint(n):
  bytes = []
  loop:
    b = n & 0x7F          // low 7 bits
    n = n >> 7
    if n != 0: b |= 0x80  // more bytes follow
    bytes.append(b)
    if n == 0: break
  return bytes

Trace it on 300, which is 1 0010 1100 in binary — nine significant bits, more than fits in one byte's 7 data bits. Split it into 7-bit groups, low group first: 0101100 (44) and 0000010 (2). The first byte still has a group after it, so its continuation bit is set: 10101100 = 0xAC. The second byte is the last one, continuation bit clear: 00000010 = 0x02. So 300 encodes as the two bytes AC 02 — which is exactly the quantity value in this chapter's opening example, sitting right after its tag byte 0x10 (field 2, wire type 0: (2 << 3) | 0 = 16).

Why varints stop at 10 bytes Protobuf caps a varint's length in theory it could encode any integer, arbitrarily large, but the format is built for 64-bit values, so 10 bytes (70 usable bits) is the ceiling. A conformant decoder rejects anything longer as malformed rather than reading forever.

The catch: a negative number is never small

Here's a trap the varint algorithm doesn't warn you about. Plain int32/int64 fields use ordinary two's complement, and a negative value gets sign-extended to a full 64-bit pattern before it's varint-encoded. That sets the top bit, which means n = -2 in a plain int64 field costs the full 10-byte varint — the same 10 bytes as the largest value the format can represent, for one of the smallest-magnitude numbers there is.

The mistake this causes Reach for int32 on a field you expect to sometimes hold small negative numbers — a temperature delta, a signed offset — and every negative value on the wire quietly costs 10 bytes instead of 1 or 2. Nothing errors. The schema still compiles, the values still round-trip correctly. The only symptom is a payload that's larger than it has any reason to be.

sint32/sint64 exist specifically to close this gap, using ZigZag encoding: instead of writing a signed integer's two's-complement bits directly, it remaps signed values onto unsigned ones so that small magnitudes — positive or negative — land near zero:

Signed value0-11-22-33
ZigZag-encoded0123456

The exact formula, straight from protobuf's own C++ source: 

// google/protobuf/wire_format_lite.h
uint32_t ZigZagEncode32(int32_t n) {
  return (uint32_t(n) << 1) ^ uint32_t(n >> 31);
}

n >> 31 is an arithmetic shift, so it produces all-1-bits when n is negative and all-0-bits when it isn't. XOR-ing against that either flips every bit of n << 1 (negative case) or leaves it alone (non-negative case) — which is exactly the zig, and the zag. Once ZigZag has remapped the number, the result gets varint-encoded exactly like any other unsigned value, small numbers, small bytes, no exception this time.

Fixed-width types: when predictability beats compactness

Not every type wants varint's variable length. fixed32, sfixed32, and float use wire type 5 (I32); fixed64, sfixed64, and double use wire type 1 (I64). Both skip the continuation-bit machinery entirely: they're always exactly 4 or 8 bytes, little-endian, no more, no less.

varint types (int32/int64/…)fixed types (fixed32/fixed64/…)
Size1–10 bytes, value-dependentalways 4 or 8 bytes
Best forsmall or usually-small numberslarge, high-entropy, or already-binary values
Worst forlarge or negative (non-ZigZag) numbersvalues that are usually small — still 4/8 bytes

That's the honest reason float and double land here rather than getting some clever variable encoding: an IEEE-754 float's bits already are its value, bit for bit. There's nothing to compress without lying about the number, so protobuf just writes the bits and moves on. And for hashes, UUIDs split into two fixed64s, or random IDs — values with no small-magnitude case to optimize for — a predictable 8 bytes beats a varint that's almost always going to cost 10 anyway.

Length-delimited: one shape, four different payloads

Wire type 2 is the workhorse of the format, and its shape is exactly three parts: a tag, a length (itself a varint), then that many raw bytes. 

[tag varint][length varint][ …length bytes of payload… ]

That one shape covers four genuinely different kinds of field, and the wire can't tell them apart — only the schema can:

Field kindWhat the payload bytes are
stringUTF-8 bytes of the text, unescaped
bytesopaque bytes, no encoding assumed at all
an embedded messagethat message's own complete serialization, inlined as-is
a packed repeated scalar fieldeach element's own varint/fixed encoding, concatenated with no per-element tag

Now finish the opening example. name = "ink" is field 1 — tag (1 << 3) | 2 = 0x0A — followed by length 0x03, followed by the three ASCII bytes of "ink": 69 6E 6B. Put that ahead of quantity's three bytes from earlier (10 AC 02) and you have the whole message: 

0A 03 69 6E 6B   10 AC 02
└─┬─┘ └┬┘ └──────┬──────┘   └┬┘ └──┬──┘
  tag  len   "ink"       tag   300 (varint)
(field 1, LEN)      (field 2, VARINT)

Eight bytes, every one of them accounted for — and notice an embedded Dimensions submessage from Chapter 2 would slot in exactly the same way: its own tag, its own length, then its own fully-formed bytes, produced by running this entire process again, one level deeper.

A quiet compatibility rule A proto3 writer emits repeated scalar fields packed — one length-delimited entry — by default. But every proto3 parser is still required to accept the older unpacked form too (one ordinary tag+value pair per element). That's what lets a peer compiled against an old protobuf release, writing unpacked, interoperate with one that packs.

Reading a tag you don't recognize

Now the payoff for tracking wire types so carefully. Say a peer running an older copy of this schema receives a message with a field 11 it's never heard of. It can't interpret field 11 — it has no idea what that data means — but it doesn't need to interpret it to get past it. It reads the tag, reads off the wire type, and skips exactly the right number of bytes for that type:

Wire typeHow to skip it without the schema
0 VARINTread one varint, discard it
1 I64discard the next 8 bytes
2 LENread the length varint, discard that many bytes
5 I32discard the next 4 bytes

That's the entire trick: the wire type is self-describing enough to skip, even though it isn't self-describing enough to interpret. Protobuf's own C++ implementation goes one step further than just skipping, though — it keeps what it skipped. Unrecognized (field number, wire type, raw bytes) triples get stored in an UnknownFieldSet attached to the message, specifically so that re-serializing the message later writes those bytes right back out, unchanged.

Where this guarantee quietly breaks Preservation is a promise about binary reserialization of the original parsed object — nothing more. Convert that message to JSON, or build a brand-new message by copying over only the fields you recognize, and field 11 is gone. It was never read, and now it's never written either. (It's also worth knowing this wasn't always true: early proto3 releases dropped unknown fields outright; the language restored preservation in v3.5, matching proto2's original behavior.)

One more piece falls out of everything in this chapter almost for free: the wire format never promises field order, and the same field number can legally appear more than once (think of two serialized messages simply concatenated). A conformant decoder has to accept fields in any order, and for a non-repeated field, the rule is last-one-wins. None of that requires special-case code — a decoder that already reads tag, dispatches on wire type, and stores the result by field number handles it automatically.

So what did we actually gain?

We opened this chapter with eight mystery bytes and closed it having derived every one of them by hand: a tag that fuses field number and wire type into one varint, a varint algorithm that spends bytes only on the bits a number actually needs, ZigZag as the fix for the one case plain varints handle badly, fixed-width types for when predictability matters more than compactness, and a length-delimited shape flexible enough to carry a string, a blob, a nested message, or a whole packed array. And running underneath every one of those: a wire type that lets any decoder, schema or no schema, skip past what it doesn't understand instead of choking on it.

That last part is the one worth carrying forward. It's not a footnote about error handling — it's the whole reason Chapter 7's schema evolution rules work at all. You'll never hand-write a single byte of this by hand in real code, though; that's protoc's job, and the shape of exactly what it generates is next.

Before you turn the page
  1. Encode the number 16384 as a varint by hand, byte by byte. How many bytes did you need, and why does that number sit right at a boundary?
  2. A teammate stores a signed delta (±a few hundred) in a plain int32 field instead of sint32. Say exactly what that costs on the wire for a typical negative value, in bytes.
  3. A relay service parses a message, logs message.quantity, and forwards the same in-memory object onward. A second relay instead reads every known field into a freshly built message before forwarding that. Which one keeps a field 11 it's never heard of, and which one silently drops it?
