Custom Extensions and Plugins

For thirteen chapters we've treated protoc as a closed appliance: you feed it a .proto, it hands you generated code, and the only knobs are the ones Google gave you — java_package, deprecated, optimize_for. But here's a question that should have been nagging at you: where do those knobs come from? What makes option java_package = "..." legal, and could you add a knob of your own? The answer turns out to be one of the most elegant ideas in the whole system — protobuf is built out of protobuf, its compiler is designed to be extended by strangers, and once you see how, you can teach your schema to carry any metadata you like and teach the compiler to do anything you want with it. This chapter is about opening the appliance. We'll add our own options, read them back through reflection, and then write a protoc plugin from scratch — and discover that a plugin is a far smaller, humbler thing than its name suggests.

Options were extensions all along

Let's start by demystifying the option keyword, because the trick hides in plain sight. When you write option java_package = "com.example"; at the top of a file, it looks like special compiler syntax. It isn't. Every .proto element — a file, a message, a field, an enum, a service, a method — has an options message hanging off it, and those messages are ordinary protobuf messages defined in a file that ships with the compiler: google/protobuf/descriptor.proto. There's a FileOptions, a MessageOptions, a FieldOptions, an EnumOptions, and so on. java_package is just field number 1 of FileOptions. Setting an option is setting a field on a message. It was protobuf all the way down the entire time.

That reframing is the key that unlocks everything else. Recall from Chapter 8 that protobuf has a feature called extensions: a way to declare "some other file may add fields to this message at these numbers." The options messages in descriptor.proto are deliberately declared with wide extensions ranges — Google left the door open on purpose. So a custom option is nothing more exotic than an extension of one of those options messages. You extend FieldOptions, and suddenly there's a new option you can write on any field, as native-feeling as the built-ins. 

syntax = "proto3";
package myco;
import "google/protobuf/descriptor.proto";   // the option messages live here

// Declare custom options by EXTENDING descriptor.proto's option messages.
extend google.protobuf.FieldOptions {
  string sensitivity = 50001;   // a new knob for any field
}
extend google.protobuf.MessageOptions {
  bool audited = 50002;         // a new knob for any message
}
The whole idea in one sentence An option is a field on a descriptor message; a custom option is an extension field you add to that message. Nothing about the mechanism is new — it's the extensions you already learned, pointed at the compiler's own data structures.

Applying them — and the number that matters

Once declared, a custom option is applied with the exact same syntax as a built-in one, with a single cosmetic difference: because it's an extension, its name goes in parentheses. At the message or file level you write option (myco.audited) = true;; inside a field's inline square brackets you write [(myco.sensitivity) = "pii"]. The parentheses are how the parser knows "look this name up as an extension, not as a built-in field." 

message User {
  option (myco.audited) = true;                     // message-level custom option

  string id    = 1;
  string email = 2 [(myco.sensitivity) = "pii"];    // field-level
  string ssn   = 3 [(myco.sensitivity) = "secret"];
}

Now, that number — 50001 — is not arbitrary, and getting it wrong is the one way to really hurt yourself here. Extension field numbers share the same flat space as regular field numbers, which means two different custom options claiming the same number on the same options message is a genuine collision. The protobuf project carved out a safe range for exactly this: the numbers 50000–99999 are reserved for internal, in-house use. Inside a single organization, where one team can eyeball who's using what, you may grab any number in that band freely.

In-house numbers are a private agreement, not a global guarantee The 50000–99999 range is safe only because everyone in your org agrees to stay in it and not step on each other. The moment a custom option travels in a .proto that other organizations will compile — a public schema, an open-source annotation library — that informal agreement evaporates. Public options need a globally unique number, and the protobuf project maintains a global extension registry you should reserve one in. Two libraries that both grabbed 50001 can never be imported into the same build.

Reading options back: annotations are inert until someone looks

Here's the part that trips up everyone the first time. You've declared sensitivity, you've tagged email as "pii", you recompile… and nothing happens. No redaction, no warning, no behavior at all. That's not a bug — it's the whole design. A custom option is pure metadata. All protoc does is record the value into the compiled descriptor and move on. The option does nothing until some other piece of code — a plugin at generation time, or your own program at runtime — goes and reads it back off the descriptor. Options are notes you leave on the schema; someone still has to read the notes.

Reading them back is a reflection operation, and it follows the same shape in every language: reach the descriptor of the element you care about (the message descriptor, the field descriptor), then pull the extension off its .Options(). In Go that's proto.GetExtension; in Python it's the Extensions[] map on the options; in Java it's getExtension; in C++ it's GetExtension. Same move, different spelling. 

// Go: read our custom field option back off a field descriptor.
fd := msgDesc.Fields().ByName("email")
opts := fd.Options().(*descriptorpb.FieldOptions)

if proto.HasExtension(opts, mycopb.E_Sensitivity) {
    level := proto.GetExtension(opts, mycopb.E_Sensitivity).(string)
    // level == "pii" — now YOU decide what that means:
    // redact it in logs, encrypt it at rest, flag it in an audit report…
}
The same descriptors you met in Chapter 13 This is the reflection API from the storage chapter wearing a different hat. There we walked a DynamicMessage to decode unknown bytes; here we walk a field descriptor to read the annotations a schema author left behind. Custom options and runtime reflection are the same toolkit: the schema, examined as data, at runtime.

The annotate-then-process loop

Step back and you'll see the shape of the whole extension story. It's a two-beat rhythm: someone annotates the schema with custom options, and someone else processes those annotations — either at build time, generating code, or at runtime, changing behavior. Every extension use case in this chapter, from validation to documentation, is just a different answer to "who processes the annotations, and when?"

Writing a protoc plugin: a program that reads stdin

So who is this build-time "plugin" that reads the descriptors? The word sounds heavyweight — you picture a shared library, an SDK, a registration API. The reality is almost anticlimactic, and it's my favorite thing about protobuf's design. A protoc plugin is just an executable that reads a protobuf from stdin and writes a protobuf to stdout. That's the entire contract. No linking against protoc, no callbacks, no framework. If your language can read standard input, parse a message, and print to standard output, it can host a plugin.

The two messages it speaks are defined in google/protobuf/compiler/plugin.proto. When you run protoc, it parses your .proto files and resolves all their descriptors — the same descriptors we've been discussing — packs them into a CodeGeneratorRequest, and pipes the serialized bytes into the plugin's stdin. The plugin does its work and writes back a serialized CodeGeneratorResponse listing the files it wants created. protoc reads that response and writes the files to disk. The compiler handles all the hard parts — parsing, importing, type resolution — and hands your plugin a fully-chewed schema as data.

The wiring is a naming convention, and it's beautifully low-tech. If you pass --NAME_out=./gen to protoc, it looks on your PATH for a binary literally called protoc-gen-NAME and runs it. That's why the Go plugin is protoc-gen-go and you invoke it with --go_out; the docs plugin is protoc-gen-doc, invoked with --doc_out. Convention is the configuration. Any options you want to pass your plugin ride along in --NAME_opt=key=value, which shows up as the parameter string inside the request. 

# protoc runs protoc-gen-myext (found on PATH) because of --myext_out.
protoc -I. --myext_out=./gen --myext_opt=paths=source_relative user.proto
#          |            |               |
#    import path   runs protoc-gen-myext   arrives as request.parameter

# Or wire an arbitrary binary to a name explicitly:
protoc --plugin=protoc-gen-myext=./bin/myext --myext_out=./gen user.proto

Inside the request and response

Let's look at what's actually in those two messages, because two of their fields hide gotchas worth knowing before you write a line of plugin code. The CodeGeneratorRequest carries three things you care about:

FieldWhat it holdsThe subtlety
file_to_generate The .proto files the user actually named on the command line. Just the roots — not their imports. Generate output for these.
proto_file Full FileDescriptorProtos for the roots and every transitive import, topologically sorted. Read descriptors from here — it's the complete schema, deps included.
parameter The string from --NAME_opt (or the key=value:dir form of --NAME_out). Your plugin parses it however it likes — there's no schema for it.

The CodeGeneratorResponse is what you send back. Its file list is the meat — each entry is a File with a name and its content as a string, and protoc writes exactly those. There's an error field for reporting trouble, and a supported_features bitmask where you advertise capabilities like FEATURE_PROTO3_OPTIONAL or FEATURE_SUPPORTS_EDITIONS so protoc knows your plugin can handle those schemas. 

// The ENTIRE contract of a plugin, by hand (real plugins use a helper).
func main() {
    in, _ := io.ReadAll(os.Stdin)               // 1. read request bytes
    var req pluginpb.CodeGeneratorRequest
    proto.Unmarshal(in, &req)                     // 2. parse it

    var resp pluginpb.CodeGeneratorResponse
    for _, f := range req.GetProtoFile() {          // 3. walk descriptors,
        //    read custom options off each element, build output…
        resp.File = append(resp.File, &pluginpb.CodeGeneratorResponse_File{
            Name:    proto.String(f.GetName() + ".myext.go"),
            Content: proto.String(generated),
        })
    }
    resp.SupportedFeatures = proto.Uint64(
        uint64(pluginpb.CodeGeneratorResponse_FEATURE_PROTO3_OPTIONAL))
    out, _ := proto.Marshal(&resp)                // 4. write response bytes
    os.Stdout.Write(out)
}
Report errors in the response, then exit zero This one surprises everyone. If your plugin hits a generation error — a schema it can't handle, a bad option — you set CodeGeneratorResponse.error to a message and still exit with status 0. protoc reads the error out of the response and fails the build gracefully. A non-zero exit code is reserved for your plugin actually crashing (write those to stderr). Exit non-zero on a normal generation error and you'll get a confusing "plugin failed" instead of your nice message.

In practice you almost never write that skeleton by hand. Every mature ecosystem ships a helper that does the stdio dance and hands you resolved, walkable descriptors: Go has google.golang.org/protobuf/compiler/protogen, Scala has ScalaPB's protocgen, and there are equivalents for Python, Java, and more. You write the interesting part — walk the types, read the custom options, emit strings — and let the helper own the pipe. There's one more trick worth naming: the insertion_point. A plugin can emit content containing a marker comment like // @@protoc_insertion_point(NAME), and a later plugin can target that marker to splice its own code into the first plugin's file instead of overwriting it. That's how a gRPC generator can weave service methods into another generator's output.

What people actually build with this

This machinery is not academic — a large slice of the protobuf ecosystem is exactly "custom options plus a plugin (or a runtime reader)." Once you can annotate a schema and process the annotations, your .proto becomes a single source of truth that spins off validators, documentation, gateways, and bespoke code. Here are the load-bearing examples.

ToolAnnotationWhat processes it, and when
protovalidate (Buf) (buf.validate.field) rules + CEL A runtime library reads the options and checks the message. No generated code.
protoc-gen-validate (legacy) (validate.rules) field options A build-time plugin emits Validate() methods. Superseded by protovalidate.
protoc-gen-doc Leading // comments on messages/fields A plugin turns the schema + comments into HTML / Markdown / JSON API docs.
grpc-gateway google.api.http method options A plugin generates a REST/JSON reverse proxy in front of a gRPC service.
your codegen whatever options you invent SQL schemas, TS clients, config loaders, mock servers — anything.

The validation story is worth dwelling on, because it captures a real shift in taste. The original protoc-gen-validate was a classic plugin: you annotated fields with (validate.rules), and it generated a Validate() method in each language you targeted. It worked, but it meant a code-generation step per language and a pile of generated validation code to compile in. Buf's successor, protovalidate, keeps the annotations but throws away the codegen: the constraints live as buf.validate custom options in the descriptor, and a tiny runtime library reads them and validates on the fly — using Google's Common Expression Language (CEL) for anything beyond the built-in rules, including cross-field checks. Same annotate-then-process loop, but the "process" moved from build time to runtime. 

// protovalidate: the rules live in the schema; a runtime reads them.
message Signup {
  string email = 1 [(buf.validate.field).string.email = true];
  uint32 age   = 2 [(buf.validate.field).uint32.gte = 13,
                    (buf.validate.field).uint32.lte = 150];
  string id    = 3 [(buf.validate.field).string.uuid = true];
}
// No Validate() is generated. At runtime: protovalidate.Validate(msg)
// reads these options off the descriptor and returns the violations.
Running plugins without the flag soup In Chapter 5 you assembled protoc --NAME_out --NAME_opt flags by hand. Once you're running several plugins, that gets unwieldy, which is why buf generate exists: a buf.gen.yaml lists your plugins — local binaries, or remote plugins hosted on the Buf Schema Registry and pinned by version — with their out/opt, and runs them reproducibly on every laptop and CI machine. Underneath, it's still exactly the CodeGeneratorRequest/Response protocol we just dissected. The protocol is the bedrock; buf generate is a nicer way to drive it.
So what did we actually gain? We opened the appliance and found it was never sealed. The option keyword turned out to be fields on descriptor messages, and because those messages allow extensions, we can add options of our own — safe in the 50000–99999 range, registered globally if they ever leave home. But an option is only a note; the value came alive only when something read it back through reflection. That "something" is a plugin — and a plugin, stripped of its intimidating name, is just a program that reads a CodeGeneratorRequest from stdin and writes a CodeGeneratorResponse to stdout, wired up by nothing more than a protoc-gen-NAME filename. Put the two halves together — annotate the schema, process the annotations — and you have the engine behind validation, documentation, REST gateways, and every bespoke generator a team ever wished for. Your .proto stops being merely a message definition and becomes the single source of truth an entire toolchain reads from. That's the real prize: not a new feature, but a door you can walk through to build features nobody at Google ever anticipated.
Before you turn the page
  1. You add option (myco.audited) = true; to a message, recompile, and nothing changes in your program's behavior. A teammate says the option is broken. Explain precisely why nothing happened, and what would have to exist for the annotation to actually do something.
  2. Two open-source annotation libraries both chose extension number 50123 on FieldOptions. What concretely goes wrong when a project tries to import both, and what rule were the libraries supposed to follow to prevent it?
  3. Your plugin receives a CodeGeneratorRequest for order.proto, which imports user.proto. Which of the two shows up in file_to_generate, which in proto_file, and why would generating output for everything in proto_file be a bug?
  4. Your plugin encounters a field it can't handle and you want to fail the build with a clear message. Describe exactly how to report it — which field you set and what exit code you use — and what happens if you exit non-zero instead.
  5. A colleague wants field-level validation and is about to adopt protoc-gen-validate. Explain how protovalidate differs in where the validation logic runs, and why that difference removes a per-language build step.
