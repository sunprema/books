Advanced Schema Design

By now you can write a schema that's correct and one that survives change. This chapter is about the schema that's a pleasure to use — the one where a reader can tell “unset” from “zero,” where a field means exactly one thing, where a service can grow for five years without a single breaking release. None of this is exotic. It's a handful of design moves — oneof, explicit presence, how you carve messages into files and packages, how you shape an RPC's inputs and outputs — each of which trades a little verbosity now for a lot of room to move later. We'll take them one at a time, and every one of them comes back to the same instinct you built in the last chapter: leave yourself a way out.

Oneof: exactly one of these, and the compiler knows it

Start with the case where a value is genuinely a choice. A payment is funded by a card or a bank transfer or a stored wallet — never two at once, and the “which one” is as much a part of the data as the values themselves. You could model that as three optional fields and a comment begging everyone to only set one. Don't. That's what oneof is for: it groups a set of fields so that at most one is set at any moment, and it makes that constraint the schema's job instead of a convention nobody enforces. 

message Payment {
  // exactly one funding source is ever set
  oneof source {
    CardDetails  card      = 1;
    BankTransfer bank      = 2;
    string       wallet_id = 3;
  }
  int64 amount_cents = 10;   // NOT in the oneof — always present
}

Two things happen once you write that. First, the members share memory: setting card automatically clears whatever was set before — there is no state in which both card and bank hold a value. Second, the generated code carries a single discriminator — a case enum, or a WhichOneof call, depending on the language — that tells you which member is set, or that none of them is. That discriminator is the payoff. Instead of checking three fields and hoping the invariant held, you switch on one value the schema guarantees is consistent.

Clear-on-set has teeth Because members share storage, setting one member frees the previously-set one. In C++ especially, holding a pointer to the old member across a set_other(...) call is a use-after-free — the object it pointed at is already gone. And an accessor for a member that isn't the current case returns that member's default, not some value it once held. Read the discriminator first; trust a member's value only when it's the set one.

A oneof also has hard structural limits, and they trip everyone once. A oneof field cannot be repeated; its members cannot be maps; and no member may carry the optional, required, or repeated label. The workaround is always the same and always clean: if you need a repeated or map value as one of the choices, wrap it in a little message and make that a member. 

message Filter {
  oneof match {
    string exact = 1;
    TagList any_of = 2;   // repeated lives INSIDE a wrapper message
  }
}
message TagList { repeated string tags = 1; }

One evolution note before we move on, because it connects straight to the last chapter. Adding a brand-new field — fresh, never-used number — into an existing oneof is wire-safe. But moving a field that already shipped as a standalone field into a oneof, or pulling one out, or shuffling it between two oneofs, is a breaking change, even though the field's number, name, and type never move. Old and new readers disagree about whether setting it should clear its neighbors, and that disagreement is exactly the kind of silent divergence buf breaking is built to catch. Decide whether a field belongs in a oneof when you first add it.

Presence: telling “unset” apart from “zero”

Here's a bug that has cost more engineer-hours than it has any right to. You have a proto3 field int32 retry_limit = 4;. A client sends a message where retry_limit is 0. Did they ask for zero retries, or did they just not set the field and you're seeing the default? In plain proto3, you cannot tell. A singular scalar with no optional keyword has no presence: a value equal to the default — 0, "", false — is indistinguishable from unset, gets no has_ accessor, and isn't even written to the wire. The two states are the same state.

Field presence is precisely the ability to distinguish “this was explicitly set” from “this holds its default.” You opt into it, per field, with the optional keyword. That one word buys you a has_retry_limit() accessor, and it changes the wire behavior too: an optional field that's explicitly set — even to the default value — is serialized, so the other side can see you meant it. 

message FeatureFlag {
  int32    count   = 1;  // no presence: 0 == unset, indistinguishable
  optional bool enabled = 2;  // presence: has_enabled() separates unset from false
}
The three-state test A plain proto3 bool can only ever say true or false — false is the default, so it's invisible. The moment “unset” is a meaningful third answer (“user hasn't chosen yet” vs. “user chose off”), you need presence: reach for optional bool, or the boxed google.protobuf.BoolValue wrapper, whose message-ness gives it presence for free.

Two field kinds already have presence in proto3 without you asking, and it's worth knowing why. A singular message field distinguishes “unset” from “set to an empty message” inherently — a null sub-message is a different thing from a present-but-empty one — so it never needs optional. Repeated fields go the other way: they have no per-element presence at all. An absent list and an explicitly-empty list are the same bytes on the wire. If “the client sent an empty list on purpose” has to be distinguishable from “the client said nothing,” a bare repeated field can't carry that — you'll need a wrapper message around it.

The trick behind optional: synthetic oneofs

This is a lovely little piece of compiler engineering, and knowing it demystifies a name you'll eventually see in generated code or a descriptor dump. proto3 didn't grow a whole new presence-tracking machine for optional. It already had one — the machinery behind oneof, which by definition tracks “is a member set, and which.” So the compiler reuses it. When you write an optional scalar, protoc quietly rewrites it into a one-field oneof whose name is an underscore followed by the field's name: 

// You wrote:
optional int32 retry_limit = 4;

// protoc rewrites it internally as a one-field synthetic oneof:
oneof _retry_limit {
  int32 retry_limit = 4 [proto3_optional = true];
}

That's the whole trick: every proto3 optional field becomes its own one-member oneof, and the oneof's “is a member set” bit becomes the field's presence bit. But you never want that plumbing leaking into the public API — nobody should call WhichOneof("_retry_limit") to ask whether one field is set. So these are marked as synthetic oneofs. Presence is tracked with an ordinary hasbit and surfaced as the plain has_retry_limit() you expected. Reflection exposes OneofDescriptor::is_synthetic(), and code generators call real_containing_oneof() (null for synthetic ones) and real_oneof_decl_count() to filter the synthetic oneofs out of any user-facing case/WhichOneof API. To you, it's just optional. Under the hood, it's a oneof of one.

Editions says the same thing, differently In Protobuf Editions — the successor to the proto2/proto3 split we'll meet later — you don't write optional; you request the same behavior with the feature features.field_presence = EXPLICIT. Different spelling, same hasbit underneath.

Nested or top-level? A namespacing choice, not a wire choice

You can declare a message or an enum inside another message, and it reads like ownership: Order.LineItem looks like a line item that belongs to an order. Here's the fact that keeps this decision from mattering more than it should: nesting has zero effect on the wire format. A field's number and encoding are identical whether its type is nested or top-level. Nesting only scopes the name — it's an organizational choice, pure and simple, so make it on organizational grounds.

And the organizational advice leans toward not nesting. The moment you nest LineItem inside Order, referencing it from anywhere else means dragging Order.LineItem around, and if a second message ever needs line items you're stuck between an awkward name and a painful move. The guidance: don't nest a type you might later want to reference elsewhere — and strongly avoid nested enums. Some target languages can't declare an enum inside a struct-like type at all, and enum value names leak into the enclosing scope, so a nested enum is both a portability risk and a naming hazard. Prefer a top-level enum with prefixed values (ORDER_STATUS_PENDING) — it's the option that comes out consistent across every language you generate.

Organizing a schema that outgrew one file

A three-message .proto is fine as one file. A schema with two hundred messages across a dozen teams is a different animal, and the way you carve it up is itself a design decision. The protobuf project's rule of thumb is called 1-1-1: one top-level entity — one message, one enum, or one service — per file, named after that entity. It feels almost too granular at first. The payoff shows up later: a file that holds exactly one thing can be moved, split off, or depended upon on its own, and a change to one message touches the smallest possible surface, which is exactly what you want when a breaking-change tool is diffing your pull request.

That's the file axis. The other axis is the package, and it's the one people skip and regret. Every .proto should declare a package: a dot-delimited, lower_snake_case namespace that keeps type names from colliding when files from unrelated projects get compiled into the same binary — and they will. That package maps straight onto each language's namespace: a C++ namespace, a Java package (unless java_package overrides it), a Go import path via go_package. 

syntax = "proto3";
package mycorp.billing.v1;   // versioned namespace so v2 can live beside it

option go_package   = "mycorp/billing/v1;billingv1";
option java_package = "com.mycorp.billing.v1";
Put the version in the package Name packages short and project-based, and end them with a version component — mycorp.billing.v1. When the day comes that you need an incompatible redesign, ...v2 is a brand-new namespace that coexists peacefully with ...v1 while callers migrate. The alternative — mutating v1 in place — is the breaking-change trap the whole last chapter warned you off.

Designing the API surface: wrap everything

When your schema starts describing RPCs — requests and responses for a service — a few patterns separate an API that ages gracefully from one that calcifies. We'll do gRPC properly in the next chapter; this is the schema-shaped half of it, and it's worth internalizing before you write a single rpc line.

The first rule is almost mechanical: give every RPC its own dedicated request and response message, even when the request is empty or the response wraps a single value, and never use a bare scalar (or a raw well-known type) as an RPC's parameter or return. It feels like pointless ceremony to wrap one string in a whole message. It isn't. The wrapper is the only place you can add a field later without changing the method's signature — and changing a method signature breaks every client. A GetUser that returns User can never also return, say, a cache-freshness timestamp; a GetUser that returns GetUserResponse can grow forever.

The second rule catches almost everyone eventually: build pagination in from the very first version of any RPC that returns a collection. Not when the list gets long — on day one. Adding pagination later silently changes the method's behavior for every existing caller (they used to get everything; now they get a page), which is a breaking change dressed up as a feature. The shape is a small, standard one, codified in Google's AIP-158: the request carries a page_size and an opaque page_token; the response returns the page plus a next_page_token that's empty when there's nothing left. 

// Every RPC gets its own wrapper — even the "simple" ones.
rpc ListInvoices(ListInvoicesRequest) returns (ListInvoicesResponse);

message ListInvoicesRequest {
  string customer_id = 1;
  int32  page_size   = 2;   // designed in up front
  string page_token  = 3;   // opaque cursor from a previous response
}
message ListInvoicesResponse {
  repeated Invoice invoices = 1;
  string next_page_token = 2;   // empty when iteration is done
}

Escape hatches: Any and extensions, and their price

Sometimes a field genuinely has to hold “a message, but we don't know which type yet” — an audit log that records events of every kind, a plugin payload defined by code you've never seen. google.protobuf.Any is the tool for that. It embeds an arbitrary serialized message alongside a type_url string (by default type.googleapis.com/<fully.qualified.Name>); pack/unpack helpers write and read it, unpacking by matching on the type name after the last /. In JSON it renders as the embedded message's normal JSON with an extra @type member.

But be clear-eyed about what Any costs, because it's easy to reach for and hard to walk back. It trades static typing for dynamic dispatch. The compiler can't check that the embedded message is one the reader expects; both peers must already share the concrete .proto to unpack it; and every schema tool — including the breaking-change checker you now rely on — is blind to what's inside. So the rule is simple: when the set of variants is known and bounded, use a closed oneof and keep your static typing. Reach for Any only when the set is genuinely open, unbounded, or plugin-defined. 

import "google/protobuf/any.proto";

// Open-ended — the variant set is unknown, so Any earns its keep:
message AuditEvent {
  string actor = 1;
  google.protobuf.Any payload = 2;   // type_url + serialized bytes
}

// Bounded — the variants are KNOWN, so a oneof keeps static typing:
message AuditEventClosed {
  string actor = 1;
  oneof payload {
    LoginEvent    login    = 2;
    LogoutEvent   logout   = 3;
    PurchaseEvent purchase = 4;
  }
}

The older cousin of Any is the proto2 extension. A message reserves a range of field numbers — extensions 100 to 199; — and other files fill that range with typed fields later, attaching data to a shared type from the outside without editing it. Extensions keep static typing (each extension is a real, typed field), which is a genuine advantage over Any, but they add real complexity, and proto3 dropped them for everything except declaring custom options. Protobuf Editions brings back extension declarations, mostly as a way to reserve and announce extension numbers. For the vast majority of schemas the honest advice is: prefer a plain field or a bounded oneof. Any, extensions, and the free-form google.protobuf.Struct are all escape hatches — useful, occasionally necessary, and each one a small surrender of the static guarantees that made you choose protobuf in the first place.

Don't use an escape hatch to dodge schema work google.protobuf.Struct / Value — a JSON-shaped tree — is the documented home for genuinely arbitrary, schema-less data. It's also the fastest way to throw away every reason you adopted protobuf: no static types, nothing for tooling to check, no compatibility guarantees. Fine as a deliberate last resort. Never as a shortcut around modeling the data you actually have.
So what did we actually gain? Every move in this chapter buys the same currency: room to change your mind later. A oneof makes an invariant the compiler's job. optional lets a reader tell zero from unset. A wrapper message lets an RPC grow a field without a breaking release. A versioned package lets a redesign coexist with the thing it replaces. You're not decorating the schema — you're paying a little now so that Future You, six months and three teams from here, has somewhere to stand.
Before you turn the page
  1. A field is int32 timeout_ms = 3; and a client sends 0. Your service needs to distinguish “use zero timeout” from “client didn't specify.” What one change makes that possible, and what does it do to the wire bytes when the value is zero?
  2. You wrote optional string nickname = 5;. Describe, in the compiler's terms, what protoc actually generates — and why WhichOneof never shows _nickname as a choice.
  3. You need a oneof where one of the choices is a list of tags. Why won't repeated string tags work as a direct member, and what's the fix?
  4. A teammate proposes rpc GetBalance() returns (google.protobuf.Int64Value) because “it's just a number.” Give the one-sentence reason this will hurt in a year, and the one-line change that fixes it.
  5. When is google.protobuf.Any the right call over a oneof, and what exactly does your breaking-change tool lose the ability to check the moment you use it?
