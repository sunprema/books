Chapter 05 · Encoding & Tooling

Code Generation and Tooling

We closed the last chapter with a promise: you'll never hand-write a single byte of the wire format yourself — that's protoc's job. Fair enough, but that just moves the question one level up. What does protoc actually do when you point it at a .proto file? How does it know how to talk to a program it's never heard of, written in a language it doesn't even ship a compiler for? And once a team has fifteen services and four target languages, is anyone really typing --python_out flags by hand every day? This chapter answers all three, in that order.

protoc: one compiler, many exits

Strip away everything else and protoc does one thing: it parses .proto files into an in-memory schema (the FileDescriptorProto graph you've already met in passing), resolves every import against the paths you gave it with -I/--proto_path, and then hands that resolved schema to a generator — once per --<lang>_out flag on the command line. Everything downstream of parsing is a generator's problem, not the compiler's. 

protoc -I=<import-path> --<lang>_out=<output-dir> <path/to/file.proto>

A handful of generators live inside protoc itself — C++, Java, Python, C#, Objective-C, PHP, Ruby, and now Kotlin and Rust — so --java_out or --python_out just works. Ask for anything else and protoc doesn't know how to refuse gracefully: it goes looking on your PATH for an executable named protoc-gen-<name> and shells out to it. --go_out works because you installed protoc-gen-go; --go-grpc_out works because you also installed protoc-gen-go-grpc. Forget one, and protoc fails with something like “program not found” — not a friendly message, but at least a diagnosable one once you know the naming convention.

One invocation, several outputs Nothing stops you from combining flags: protoc --go_out=. --go-grpc_out=. --java_out=./java file.proto runs three generators in a single pass over the same parsed schema. Each one only cares about its own --<name>_out (and optional --<name>_opt) flags; they don't know or care that siblings are running too.

The same schema, five different shapes

Here's something worth sitting with: every one of these generators starts from exactly the same input — the identical resolved FileDescriptorProto for your Product message — and yet what comes out the other end looks nothing alike from language to language. The wire format is universal; the generated code is not, and was never meant to be. Each generator's job is to make the schema feel native to its own language.

LanguageGeneratorWhat you get
Goprotoc-gen-goa struct per message, exported fields, nil-safe Get<Field>() accessors
Javabuilt into protocan immutable class built through Message.newBuilder()....build()
Pythonbuilt into protoca thin descriptor literal, handed to a metaclass that builds the real class at import time
C++built into protoca class with direct getters/setters, backing the reference runtime itself
TypeScriptts-proto or protoc-gen-esplain interfaces + functions, or a full-conformance class — your choice, no default

Go is the most literal translation: one struct field per protobuf field, each carrying a struct tag that encodes the wire-format details the runtime needs (protobuf:"varint,2,opt,name=quantity,proto3"), plus a plain json tag for good measure. What you don't get automatically is nil-safety on direct field access — that's what the generated getter is for. 

type Product struct {
  Name     string `protobuf:"bytes,1,opt,name=name,proto3" json:"name,omitempty"`
  Quantity int32  `protobuf:"varint,2,opt,name=quantity,proto3" json:"quantity,omitempty"`
}

func (x *Product) GetQuantity() int32 {
  if x != nil {
    return x.Quantity
  }
  return 0
}

Python goes the opposite direction: about as little generated code as it can get away with. The _pb2.py file mostly serializes a FileDescriptorProto literal, then hands it to a metaclass (GeneratedProtocolMessageType, if you go looking) that builds the actual message class when the module is imported — the per-field behavior lives in shared runtime code, not in code generated fresh for every message. That runtime, in turn, picks between a few backends at load time: a pure-Python implementation, an older C++ extension, or upb, a small C library that's been the default since protobuf 4.21 for its speed. Same generated Python file, three possible engines underneath, chosen automatically.

Not every language has “the” generator TypeScript is the odd one out on that table for a reason: it isn't bundled in protoc at all, and unlike Go or Java there's no single official plugin to reach for. ts-proto produces idiomatic plain interfaces and functions with zero native dependencies; protoc-gen-es (the Protobuf-ES project) generates a fully spec-conformant runtime and pairs naturally with Connect-ES for RPC. Picking between them is a real decision your team makes, not a default protoc hands you.

Plugins: how protoc talks to a program it's never heard of

This is the part worth slowing down for, because the mechanism is smaller and stranger than it sounds. protoc doesn't know anything about Go, Rust, or whatever language your plugin happens to be written in. It doesn't need to. A protoc plugin's entire contract is two steps:

  1. read one serialized CodeGeneratorRequest protobuf message from stdin;
  2. write one serialized CodeGeneratorResponse protobuf message to stdout.

That's it. The plugin is itself just another protobuf consumer — it decodes a message using whatever protobuf library its own language has, does some work, encodes a response message, and exits. There's a pleasing bit of eating-your-own-dog-food here: the tool that generates code from schemas is, at the transport level, exercising the exact same encode/decode machinery Chapter 4 walked through byte by byte.

What's actually inside that request is worth naming, because it explains a lot of what plugins can do that might otherwise feel like magic. CodeGeneratorRequest carries:

FieldWhat it holds
file_to_generateonly the files named explicitly on the command line — generate code for these, not their dependencies
parameterthe raw string passed via --<name>_opt, e.g. paths=source_relative
proto_filethe full resolved schema: every file needed to generate the targets, including every transitive import, topologically ordered
compiler_versionthe protoc version that invoked the plugin

Notice the gap between file_to_generate and proto_file. A plugin isn't handed a lonely single file to generate code for in isolation — it's handed the entire compiled graph the target file depends on, descriptors and all. That's exactly why a plugin can do things that sound like they'd require re-parsing your whole schema from scratch: it never has to, because protoc already did that part and handed over the result.

The cost hiding in that convenience A plugin scales with your whole import graph, not just the file you're generating. A schema with a deep or wide import tree makes every plugin invocation proportionally slower, even for a change that only touched one small leaf file — because proto_file still carries the transitive closure every time.

The response side has its own small trick worth knowing. CodeGeneratorResponse.File has a name (the output path) and content (the generated text) — the obvious pair — plus an optional insertion_point. Set that, and instead of writing a new file, protoc splices your content into another generator's already-written output, at a matching @@protoc_insertion_point(NAME) marker comment left behind in that file. It's how two independent plugins can cooperate on one generated file without either knowing the other exists — one leaves markers, the other fills them in.

The plugin ecosystem: gRPC is just the start

Once you see plugins as ordinary programs speaking a fixed request/ response contract, the rest of the ecosystem stops looking like a pile of unrelated tools and starts looking like the same trick, reused. gRPC service code, for instance, isn't folded into the message generator at all — it's a second, separate plugin, layered on top. The current Go workflow runs protoc-gen-go (messages) and protoc-gen-go-grpc (service interfaces and clients) as two distinct invocations, producing two distinct files: 

protoc -I=. \
  --go_out=. --go_opt=paths=source_relative \
  --go-grpc_out=. --go-grpc_opt=paths=source_relative \
  catalog/v1/product.proto
# -> catalog/v1/product.pb.go       (messages)
# -> catalog/v1/product_grpc.pb.go  (service)
A form you'll still see in older code An older, combined form — --go_out=plugins=grpc:. — wrote both messages and service code into one file with a single plugin. It still works in many setups, but it isn't the path current tooling and docs are written against, and it produces a different file layout than the split invocation above. If you're starting fresh, use the split form.

The same request/response contract also powers plugins that have nothing to do with generating a target-language runtime at all: protoc-gen-doc walks the descriptor graph and renders HTML, Markdown, or JSON documentation straight from your schema's own comments; grpc-gateway's protoc-gen-grpc-gateway reads google.api.http annotations on a service definition and generates a reverse-proxy server that exposes the same RPCs as a RESTful JSON API. Neither one emits code you'd run your business logic through — and neither needs to be special-cased by protoc to work. They're plugins in exactly the sense protoc-gen-go is.

Ecosystem churn, illustrated protoc-gen-validate was, for years, the standard way to generate per-language Validate() methods from constraint annotations on your fields. Its own repository now points newcomers to protovalidate as the maintained successor for the same job. The core protoc/plugin contract from the previous section is stable; the plugins built on top of it — especially anything validation- or gateway-adjacent — move faster and get superseded more often. Check what a given plugin's own README currently recommends before you commit to it.

Stop hand-assembling flags: buf

Picture a repo with a dozen services, four target languages, and a grpc-gateway pass on top. That's a lot of -I paths and --<lang>_out flags to keep in sync by hand across Makefiles, and it's exactly the itch buf scratches. Buf isn't a replacement for the wire format or for protoc's underlying job — it's a CLI that wraps day-to-day protoc use in two declarative config files instead of a shell script:

FileReplacesDeclares
buf.yamlad hoc -I juggling across a workspacea module's lint rules, breaking-change rules, and dependencies
buf.gen.yamla hand-assembled string of --<lang>_out/_opt flagswhich plugins to run, and where each one's output goes
 
# buf.gen.yaml -- replaces the hand-assembled protoc flags above
version: v2
plugins:
  - local: protoc-gen-go
    out: gen/go
    opt: paths=source_relative
  - local: protoc-gen-go-grpc
    out: gen/go
    opt:
      - paths=source_relative
      - require_unimplemented_servers=false

Run buf generate and it reads both files and runs every configured plugin in one command — the same plugins, the same request/response contract from the last section, just with the bookkeeping centralized instead of copy-pasted into every service's build script.

Codegen is the part that touches this chapter most directly, but it's not the point of buf on its own. buf lint checks a schema against style/API-shape rules; buf format canonicalizes .proto formatting the way gofmt does for Go; and buf breaking diffs your current schema against a prior version — a git ref, a tag, a snapshot in the Buf Schema Registry — and fails if it finds an incompatible change, like a reused or retyped field number from Chapter 2's rules. That last one is the piece worth remembering going into Chapter 7: it's the tool that turns “don't break the wire format” from a rule you hope everyone remembers into a check that fails a pull request automatically.

Wiring it into CI Buf's official GitHub Action (bufbuild/buf-action) runs build, lint, format, and breaking checks in one workflow step and posts the result as a pull-request comment. That's the concrete shape of the “CI breaking-change detector” this book keeps promising: a schema change that would silently corrupt an old reader's understanding of the wire never merges in the first place.

Build systems: Bazel's language-neutral proto_library

Bazel takes a slightly different cut at the same problem, and it's worth seeing because the shape generalizes: separate declaring a schema from compiling it into any particular language. A proto_library target names a .proto file's sources and its dependencies on other proto_library targets, mirroring that file's own import statements — and nothing else. It doesn't pick Go or Java or generate a single line of code by itself. 

# BUILD.bazel
proto_library(
    name = "product_proto",
    srcs = ["product.proto"],
)

go_proto_library(
    name = "product_go_proto",
    importpath = "example.com/catalog/v1",
    proto = ":product_proto",
)

py_proto_library(
    name = "product_py_proto",
    deps = [":product_proto"],
)

Each language then layers its own rule on top of that bare proto_library — cc_proto_library, java_proto_library, py_proto_library, or a community rule like go_proto_library from rules_proto_grpc. Those wrapper rules are what actually invoke protoc: each one implicitly depends on the protoc binary and on a proto_lang_toolchain target describing exactly how to call it for that language. The payoff is the same one-schema-many-languages story as before, but with Bazel's caching: generate Go and Python from the same proto_library, and Bazel treats each as its own cacheable output, rebuilt only when the schema or the toolchain actually changes.

If you're setting this up today Don't reach for the standalone bazelbuild/rules_proto repository — as of a 2024 community decision, proto_library and the core per-language rules moved directly into the protocolbuffers/protobuf repository itself, and rules_proto is kept around only as a migration pointer, not an actively developed rule set.

Editors caught up too

One last, smaller piece, but worth a mention because it changes how writing a .proto file actually feels day to day. .proto editor support has converged on the same Language Server Protocol that powers modern editor tooling for most other languages, rather than a pile of one-off syntax-highlighting plugins. Buf ships its own production-grade Protobuf language server, bundled automatically into its VS Code extension, giving you go-to-definition, completion, and inline lint diagnostics right as you type — the same buf lint rules from two sections ago, surfaced live instead of only at CI time. Independent community language servers (protobuf-language-server, protols) cover other editors, and formatting can run through that LSP, through clang-format, or by shelling out to buf format directly.

So what did we actually gain?

Start to finish, this chapter has really been one idea wearing several outfits: protoc parses a schema once and hands the result, not the source text, to whatever generates code next. When that generator is built in, you don't notice the handoff at all. When it's an external plugin, the handoff becomes an honest-to-goodness protobuf message crossing a pipe — which is also why a plugin can be written in any language, why it can see your entire import graph instead of just one file, and why two plugins can cooperate on one file through insertion points without ever importing each other's code. buf and Bazel are both, underneath, organizing that same handoff at a larger scale: one config file or one build graph standing in for a hundred hand-typed --<lang>_out invocations.

None of this has said anything yet about what to actually put in a schema once you're modeling something more real-world than a product with a name and a quantity — nested types, cross-file references, well-known types for time and dynamic data, and the judgment calls that separate a schema that ages well from one that doesn't. That's next.
