Debugging and Common Mistakes

Here's a scene you will eventually live through: a service on the far end of a socket is getting your messages, and the values are wrong. A name comes through as gibberish, a timestamp lands in 1970, an enum you never chose shows up as 0. Nobody threw an exception — that's the cruel part. Protobuf is a schema-driven format, and when the schema on one side quietly disagrees with the bytes on the wire, you don't get a crash. You get a plausible, silent lie. So this chapter is two things at once. First, the one tool that cuts through all of it: how to open up the raw bytes and look, with and without the .proto in hand. And second, a field guide to the mistakes that produce those silent lies in the first place — nearly all of which we've foreshadowed in earlier chapters, now gathered in one place where you can recognize them on sight. We'll end with a checklist you can actually run down at 2 a.m.

Rule one: stop guessing, look at the bytes

When a message decodes wrong, the temptation is to stare at your code — the serializer, the field you just added, the network layer. Resist it. The bytes on the wire are the ground truth, and protobuf ships a tool that reads them straight back to you. It lives inside protoc, the same compiler you've used since Chapter 5, and it has two modes that differ by one crucial question: do you have the schema or not?

If you have the .proto, use --decode. You name the fully qualified message type and pass the file; protoc reads the binary from standard input and prints it as human-readable text with real field names and typed values. This is your first move whenever you suspect the sender and you disagree about the schema — decode the exact bytes you received and see what your schema makes of them. 

# You HAVE the schema: named, typed fields come back out.
protoc --decode=tutorial.Person addressbook.proto < person.bin

# name: "Ada Lovelace"
# id: 4
# phones { number: "555-1234" type: HOME }

# The same tool round-trips the other way (text -> bytes),
# which is gold for building a known-good fixture by hand:
protoc --encode=tutorial.Person addressbook.proto < person.txt > person.bin

But the really magical mode is the one for when you don't have the schema — a capture off the wire, a blob in a database column, a payload from a service whose .proto you can't find. --decode_raw takes no proto file at all. It reads arbitrary bytes from stdin and prints the raw field-number/value pairs it can recover, nesting sub-messages in braces. How is that possible without the schema? Because — and this is the payoff of everything we learned about the wire format in Chapter 4 — every field on the wire is a self-describing tag: a field number and a wire type, glued to its value. You don't need names to walk that structure; the structure is in the bytes. 

# You have NO schema. See the shape anyway:
protoc --decode_raw < message.bin

1: 4
2: "Ada Lovelace"
5 {                 # a nested message — protoc guessed wire-type 2 is a submessage
  1: "555-1234"
  2: 1
}
--decode_raw shows you structure, not truth Trust --decode_raw for the shape of a message — which field numbers are present, how they nest — but not for the exact types. Recall from Chapter 4 that the wire carries a wire type, not the declared type. Wire type 0 (varint) could be an int32, int64, uint32, bool, an enum, or a ZigZag sint32 — and protoc can't tell which, so a negative sint32 will print as a big meaningless unsigned number. Likewise a length-delimited (wire type 2) field might be a string, raw bytes, a packed repeated field, or an embedded message — protoc guesses, and sometimes guesses wrong. When the value matters, get the .proto and use --decode.

The original sin: touching a live field number

Now to the mistakes. If you remember one from this whole chapter, make it this one, because it's the mistake that produces the silent-lie scenario from the opening. A field number, once it has shipped, is permanent. Reusing a number for a different field, or renumbering an existing one, corrupts deserialization for every peer still running the old schema — and it does so exactly the way changing a field's type would, which is to say quietly and catastrophically. The old reader sees the old number, applies the old type, and confidently decodes your new field's bytes as something it is not.

The trap is that nothing stops you at compile time. You delete string email = 2;, everything builds, and six months later a teammate who's never heard of the old email field innocently writes bool is_verified = 2;. The compiler is delighted. The wire is not. The defense is the reserved keyword you met in Chapter 7 — when you retire a field, reserve its number (and its name, so nobody re-adds the name either) and the compiler will now reject the collision for you. 

message User {
  reserved 2, 15, 9 to 11;      // numbers of removed fields — never reuse
  reserved "email", "legacy_id";  // and their names, so they can't come back either

  string id   = 1;
  string name = 3;                // note: NOT 2 — that number is retired forever
}
The rule in one line Field numbers are addresses, not labels. You can rename a field freely (the name is a local convenience), but the number is the identity every serialized byte agrees on. Change a name, and nothing on the wire notices. Change or reuse a number, and everything on the wire lies.

Ghosts of proto2: required and the tyranny of zero

A cluster of mistakes comes from the seam between proto2 and proto3, and from proto3's central design choice about presence. These are worth understanding together, because they all trace back to the same question: "how does the reader know whether a field was actually set?"

Don't write required — and be wary of code that did

In proto2 you could label a field required, and it felt responsible — "this field must always be present, so I'll enforce it." proto3 removed the label entirely, and not by accident. required turned out to be a one-way door. Once a required field is deployed, you can never relax it: making it optional later means every reader and writer everywhere has to be upgraded in a single synchronized rollout, or an old reader will reject a new message that omits the field. That's precisely the lockstep-deploy nightmare the entire schema-evolution discipline from Chapter 7 exists to prevent. Google's own retrospective is blunt: adding required was a mistake. If you're on proto2 or Editions, treat the option as if it doesn't exist.

Give the enum a zero value, and make it mean nothing

proto3 requires that the first value of every enum be 0. This isn't bureaucracy — the zero value is the enum's implicit default, the thing you get when the field is absent. And here's the subtler reason it must be a harmless sentinel: because of the forward-compatibility we discussed in Chapter 7, when a newer peer sends an enum value your older schema has never heard of, many runtimes surface it to you as the zero value (or preserve it as an unknown number you have to check for). So if you make 0 mean something real — say ACTIVE — then both "field was never set" and "value I don't recognize yet" collapse into "ACTIVE," and you will ship a bug that looks like data corruption. Make zero an inert *_UNSPECIFIED, always. 

// GOOD: zero is a sentinel that means "I don't know," not a real state.
enum Status {
  STATUS_UNSPECIFIED = 0;  // absent OR unknown-to-me decodes to this
  STATUS_ACTIVE      = 1;
  STATUS_CLOSED      = 2;
  reserved 3;              // a removed value's number — reserve it, same as a field
}
// Note the STATUS_ prefix on every value: enum value names live in the
// ENCLOSING scope, not inside the enum, so unprefixed names collide across enums.

Two follow-on traps hide here. First, enum values share the same reuse hazard as field numbers: delete CLOSED = 2 and a future edit can reclaim 2 for something else, silently reinterpreting old data — so reserved the number just as you would a field's. Second, enum value names are scoped to the enclosing namespace, not to the enum, which is why the convention is to prefix every value with the enum's name. Two enums in one file that both define a bare ACTIVE = 1 simply won't compile.

Default value is not the same as "present"

This is the single most common conceptual mistake in proto3, and we flagged it back in Chapter 2. In proto3, a plain scalar field that equals its zero value — 0, "", false — is indistinguishable from a field that was never set. There's no has_retries() for a bare int32 retries. So the classic bug: an API takes an int32 page_size, a client that wants the server default simply doesn't set it — and the server, unable to tell "unset" from "explicitly zero," reads 0 and returns an empty page.

The fix is to opt back into explicit presence when you need it. Adding the optional label in proto3 (yes, the keyword came back for exactly this) restores a real has_ accessor. A oneof or a wrapper message (google.protobuf.Int32Value) does the same job. The rule of thumb: if the difference between "set to zero" and "not set" is meaningful in your domain, you must model presence explicitly — the zero value will not carry that distinction for you. 

message ListRequest {
  int32 page_size = 1;            // 0 could mean "unset" OR "the client asked for 0" — ambiguous
  optional int32 max_results = 2;  // proto3 presence: has_max_results() distinguishes the two
}
Editions makes this a choice, not a guess Mixing proto2 and proto3 in your head is a mistake all its own — the two dialects genuinely disagree about "set vs. zero," so a field that behaves one way in a proto2 file behaves differently in a proto3 one. Protobuf Editions, which the next chapter covers, exists partly to end this: instead of inheriting presence and default behavior implicitly from the syntax line, you select it per-field as an explicit feature. Same wire format, but the semantics stop being a thing you have to remember and become a thing you declare.

Encoding traps: signed ints, maps, and giant messages

The last cluster of mistakes is about the encoding fighting your assumptions — cases where the code is correct but the bytes misbehave in ways that only surface under load or under a signature check.

Negative numbers in an int32 are a size bomb

From Chapter 4: varints are efficient for small non-negative numbers. But a plain int32 or int64 encodes a negative value by sign-extending it to the full width, which means -1 costs a full ten bytes on the wire, every single time. Store a field of temperatures or deltas or geo-offsets as int32 and you've quietly made every negative reading maximally expensive. The fix is sint32/sint64, which ZigZag-encode so that small-magnitude negatives stay small. The catch — and it's a real one — is that sint32 is a different, wire-incompatible type from int32: they share wire type 0 but decode the same bytes to different numbers, so you cannot swap one for the other on a field that's already in use. Choose it up front for fields you know will hold negatives.

Maps don't have an order, and your bytes aren't a fingerprint

Here's one that burns people building caches, signatures, and content-addressed storage. It is tempting to serialize a message and use the resulting bytes as a hash key or sign them — "same message, same bytes, right?" Wrong. Protobuf's default serialization is not deterministic. The biggest culprit is maps: map iteration order and the wire order of map entries are explicitly undefined, so two serializations of the same message can differ byte-for-byte. Even setting maps aside, the library makes no promise that repeated serializations are identical.

Most mature libraries offer a deterministic serialization option that, among other things, sorts map keys — and you should turn it on for anything that hashes or signs bytes. But even then, know the limits the docs are careful to spell out: deterministic serialization is not canonical. The byte output can still differ across languages, across versions of the same library, and in the presence of unknown fields carried through from a peer. So: never treat a serialized proto as a stable fingerprint across time or across services. If you need a canonical identity, define one explicitly.

Protobuf is not a file format for large blobs One more assumption to shed: protobuf is built to hold the whole message in memory, and during parse or serialize it may transiently hold several copies. That's fine for the kilobyte-to-low-megabyte messages it's designed for, and miserable for multi-megabyte payloads or unbounded streams — you'll spike memory and, in some runtimes, hit hard size limits. The idiom is to chunk a large blob into a sequence of smaller messages, or stream them (Chapter 13), rather than stuffing a 100 MB video into a single bytes field.

The 2 a.m. troubleshooting checklist

When a message decodes wrong and you need a path, not a lecture, run down this list roughly in order. It's built from every mistake above, arranged as symptoms.

SymptomFirst thing to check
Values are garbage / shifted / nonsensical protoc --decode_raw the exact received bytes. Do the field numbers match what you expect? A mismatch means the two sides disagree about the schema — almost always a reused or renumbered field.
One field is wrong; the rest are fine That field's number was probably reused or its type changed. Diff the two .protos at that number. Reserve retired numbers going forward.
An enum shows up as 0 / UNSPECIFIED unexpectedly Either the field was never set, or the sender used a value newer than your schema knows. Both funnel to zero — confirm your zero value is a harmless sentinel, and check for unknown-enum handling.
Can't tell "unset" from "zero / empty / false" That's proto3 default-vs-presence. Add optional, a wrapper, or a oneof to the field so a has_ accessor exists.
Old readers reject new messages / a rollout is stuck Look for a required field (proto2) or a field whose semantics you can't relax without a synchronized deploy. This is the required-is-forever trap.
A signed field is huge on the wire It's an int32/int64 holding negatives (10 bytes each). Use sint32/sint64 — but only on a new field, since it's wire-incompatible.
Signatures / hashes / cache keys intermittently break You're hashing non-deterministic bytes — almost certainly a map. Enable deterministic serialization, and stop treating serialized protos as canonical fingerprints.
A relay/proxy drops fields it doesn't know It's rebuilding a new message or round-tripping through JSON instead of re-serializing the parsed object. Unknown fields survive a straight re-serialize; they don't survive a manual copy.
Memory spikes / hard failures on big payloads The message is too large for protobuf's whole-in-memory model. Chunk or stream it.
The fastest diagnosis is a diff of two decodes When two services disagree, capture the same bytes and run them through both schemas: --decode with the sender's .proto, then --decode with the receiver's. The place the two text dumps diverge is, almost every time, the exact field whose number or type drifted between them. It turns a philosophical argument about "who's right" into a two-line shell command.
So what did we actually gain? A way to stop guessing. Protobuf's great strength — that it's a compact, schema-driven binary format — is also the source of its signature failure mode: when the schemas drift, you get silent wrong answers instead of loud errors. The cure is to remember that the bytes are self-describing, and that protoc --decode and --decode_raw will read them back to you with or without the .proto in hand. Around that one tool we gathered the recurring mistakes, and notice they nearly all share a shape: they're places where something feels safe to change but isn't — a field number, an enum value, a required label, an int32 that "obviously" holds a small number, a map you assumed had an order. Protobuf's whole discipline, the thread running through this book, is knowing which changes the wire forgives and which it silently punishes. Learn to see the bytes, and you can always find out which kind you're looking at.
Before you turn the page
  1. You capture a message off the wire, run protoc --decode_raw, and one field prints as 18446744073709551615. Nothing in your schema has a number that large. Explain what type that field probably is and why --decode_raw can't show it correctly — then say what you'd run instead to see the real value.
  2. A teammate deletes string nickname = 4; from a message and ships it. Six months later someone adds int64 created_at = 4;. Walk through exactly what an old client that still knows nickname does when it receives a message from the new server, and name the one keyword that would have made the compiler stop the second developer.
  3. An enum Color { COLOR_RED = 0; COLOR_GREEN = 1; } is deployed. A newer service sends COLOR_BLUE = 2 to an older reader. What value does the older reader most likely observe, and why does that make picking a meaningful 0 value dangerous?
  4. Your service signs serialized protobuf messages and verifies the signature downstream. It works in tests and fails ~30% of the time in production. The message contains a map<string, string>. Diagnose it, and describe the two changes you'd make.
  5. Give a one-sentence rule for when to reach for sint32 instead of int32, and explain why you can't fix a mis-chosen int32 field by simply editing it to sint32 in place.
