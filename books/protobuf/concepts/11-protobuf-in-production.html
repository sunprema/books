Protobuf in Production

Everything so far has been about getting a single schema right: the model, the bytes, the codegen, the compatibility rules. This chapter is about what happens when there are two hundred schemas, forty services, six languages, and a hundred engineers who have never met — all of them depending on the same .proto files, all deploying on their own schedule. The wire format was never the hard part. The hard part is that your schema quietly became a distributed contract, and nobody owns it the way they own their own code. So we'll build the machinery that keeps a shared schema honest: a registry to version and distribute it, a CI gate that refuses breaking changes, a build pipeline that generates code the same way on every machine, rollout discipline for producers and consumers that upgrade out of step, and finally a way to actually see schema change as it happens. This is protobuf as an operational practice, not a syntax.

The schema is a distributed system now

Let's start by naming the shift, because it reframes everything else. On one team, a .proto file is just a file — you edit it, regenerate, run the tests, ship. The moment a second team compiles that same file, it stops being your file. It's an interface between programs that upgrade independently, and — here's the sharp bit — those programs are running right now, in production, at versions you don't control. A field you delete is a field some binary out there still reads. A number you reuse is a landmine for a peer three deploys behind you.

This is why the recurring argument — the one we met back in Chapter 1 — is that protobuf's durable value isn't the compact bytes; it's the enforceable contract. But a contract nobody enforces is just a suggestion. Everything in this chapter exists to turn “please don't break the schema” into something a machine checks, blocks, and records. Treat the schema like what it has become: a shipped artifact with versions, a distribution channel, a test suite, and an audit log.

The one-sentence version Once more than one independently-deployed program depends on your .proto files, the schema is a distributed contract — and it needs the same operational discipline as any other production artifact: it must be versioned, distributed, compatibility-checked in CI, rolled out in a safe order, and observable.

buf: one workspace, one source of truth

Before the machinery, meet the tool most of it runs on. Running protoc by hand — a wall of -I paths and --<lang>_out flags, retyped slightly differently on every laptop — doesn't survive contact with a real organization. buf is the modern toolchain that replaces it, and its central idea is simple: declare the workspace once, in a buf.yaml at the repo root, and let every operation read that same definition. 

# buf.yaml — the workspace contract, checked into the repo root
version: v2
modules:
  - path: proto
deps:
  - buf.build/googleapis/googleapis   # resolved into buf.lock by `buf dep update`
lint:
  use:
    - STANDARD
breaking:
  use:
    - WIRE_JSON            # the recommended minimum; FILE is stricter

With that one file in place, each subcommand is a verb over the same schema: buf build compiles the workspace, buf lint catches API-shape problems while you're still editing, buf format tidies, buf breaking checks compatibility, buf generate produces code, buf dep resolves dependencies, and buf push publishes. They can't disagree about what “the schema” is, because they all read buf.yaml. (A nice bonus: buf's compiler uses every core and tends to be about twice as fast as protoc on a non-trivial workspace — which you feel on every CI run.)

Lint is the cheap half of the discipline buf lint won't stop a wire break — that's buf breaking's job — but it enforces the style rules from earlier chapters (enum-zero values, UPPER_SNAKE_CASE enum names, package-mirrors-directory) so a schema stays consistent across a hundred authors. Consistency now is what makes the automated compatibility rules legible later.

A schema registry: versioned, not vendored

Here's the first real production question: how does the payments team get the catalog team's .proto files? The naive answer — copy them in, or git-submodule the other repo — is the origin of most schema pain. Copies drift. Nobody can say which version of the contract a given service actually compiled against. A fix in the source repo doesn't reach the ten repos that snapshotted it months ago. You've turned a shared interface back into a pile of divergent files.

A schema registry solves this the way a package manager solves it for libraries. The reference implementation is the Buf Schema Registry (BSR). It turns a set of .proto files into a versioned, addressable module — buf.build/acme/catalog — with real history. buf push publishes the module, and every push is a commit; labels mark specific commits for consumers, with a default label tracking releases. A downstream repo doesn't copy anything — it declares a dependency in buf.yaml, and buf dep update resolves it into a buf.lock, exactly the lockfile idea you already trust from npm or Go modules.

Distribution is the other half of the story, and it's where a registry earns its keep every day. The BSR doesn't just store the schema — it serves generated documentation, runs codegen plugins on demand, and publishes ready-made SDKs into native package ecosystems: Go modules, PyPI, Maven, npm, NuGet, Cargo, Swift Package Manager. A Python consumer runs pip install on a normal package; it never sees a .proto file or runs protoc at all. The schema's authors publish; the schema's users install. That's the whole game.

The CI gate: breaking-change detection

Now the single most valuable piece of production tooling in the whole chapter. Back in the evolution chapter we learned which changes are safe and which are poison — adding a field is fine, reusing a number is a disaster. But knowing the rules and never once violating them across thousands of PRs by tired humans are very different things. So we don't rely on humans. We let buf breaking compile the proposed schema and a past version, diff them, and fail the build on any change that would break clients, servers, or generated code.

What counts as “breaking” isn't one thing — it's a policy you choose, expressed as one of four nested categories. They run from strictest to most lenient, and passing a stricter one implies passing every looser one:

CategoryCatches a break in…Use it when
FILEgenerated source, per file (an import path, a symbol location)consumers compile the generated code directly (esp. C++/Python)
PACKAGEgenerated source, per packageyou want source stability but tolerate file moves
WIRE_JSONbinary or JSON encodingthe recommended minimum — services that also speak JSON
WIREbinary encoding onlybinary-only systems that never serialize JSON

The check always diffs against a baseline, and the --against flag is refreshingly flexible about where that baseline lives: a Git branch, tag, or commit; a module in the registry; a tarball or zip; a plain directory; or a prebuilt Buf image. The everyday form compares your branch against main: 

# does this branch break the schema on main?
buf breaking --against '.git#branch=main'

# CI does shallow clones? point at the remote; buf fetches only the one
# commit it needs for the diff.
buf breaking --against 'https://github.com/acme/api.git#branch=main'
“buf breaking passed” is not the same claim as “wire-compatible” A breaking-change tool can be deliberately stricter than the raw wire format. Moving an enum's definition to a different file, for instance, can be a FILE-category break even though the bytes would round-trip perfectly. That's a feature, not a bug — it protects source compatibility too — but it means the category you enforce is a real decision. Pick WIRE/WIRE_JSON if you only care about the bytes; pick FILE if your consumers depend on the generated code's shape.

Wiring the gate into CI

A check nobody runs protects nothing, so the gate belongs in the pull-request pipeline where it can't be skipped. The unified bufbuild/buf-action GitHub Action runs the whole suite — build, lint, format, and breaking — in one step, and can auto-publish to the registry on merge. Critically for reviewers, it surfaces failures where humans look: lint and breaking violations become GitHub annotations and inline comments right next to the offending lines, plus a summary comment on the PR. An unsafe schema edit gets a red check and a comment explaining exactly which field broke — at review time, before it can ever reach a running peer. 

# .github/workflows/buf.yaml — the gate
on: pull_request
jobs:
  buf:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with: { fetch-depth: 0 }        # breaking needs the baseline commit
      - uses: bufbuild/buf-action@v1
        with:
          # build + lint + format + breaking; comments on the PR,
          # diffs against the target branch, fails on a break
          breaking_against: 'https://github.com/acme/api.git#branch=main'
The one config that always trips people up Breaking detection needs to see the baseline. A default shallow checkout (fetch-depth: 1) doesn't have main's history, so the diff silently has nothing to compare against. Either set fetch-depth: 0 or point --against at the remote URL and let buf fetch just the one commit. This is the single most common reason a “working” breaking check quietly checks nothing.

Code generation as a build product

Generated code should be a reproducible build output, not an artisanal thing each engineer conjures with slightly different flags. In production you want two guarantees: the same schema produces the same code on every machine, and nobody hand-edits the result. buf gets you there with a versioned buf.gen.yaml and two features worth knowing by name.

The first is remote plugins. Instead of asking every laptop and CI runner to install protoc-gen-go at exactly the right version, the registry hosts plugins pinned by version and executes them server-side. buf generate sends the schema up, the plugin runs, the code comes back. Pin the version once (...:v1.36.11) and the output is identical everywhere, forever. The second is managed mode, which moves the language-specific file options — go_package, java_package, csharp_namespace — out of the shared .proto and into buf.gen.yaml. The schema stays language-neutral (no consumer's convention leaks into the contract), yet every consumer still gets correctly-namespaced code. 

# buf.gen.yaml — codegen, made reproducible
version: v2
managed:
  enabled: true              # go_package etc. live HERE, not in the .proto
  override:
    - file_option: go_package_prefix
      value: github.com/acme/gen/go
plugins:
  # remote + version-pinned -> identical output on every machine
  - remote: buf.build/protocolbuffers/go:v1.36.11
    out: gen/go
    opt: paths=source_relative
  - remote: buf.build/grpc/go:v1.5.1
    out: gen/go
    opt: paths=source_relative
Layout is a build-performance decision too How you split .proto files feeds straight into this pipeline. The 1-1-1 convention — one top-level type per file, one package, one build rule, in a directory tree that mirrors the package — minimizes transitive dependencies, so a small change recompiles and regenerates less, builds faster, and produces leaner binaries. Moving a type becomes a file move instead of a surgical extraction. It's an ideal to bend when messages are genuinely coupled or a circular dependency forces your hand — but bend it knowingly.

Rollout discipline: producers and consumers out of step

Here's the fact that makes schema changes genuinely operational rather than theoretical: a rolling deploy runs old and new binaries at the same time. For minutes or hours, a v2 producer is sending data to a v1 consumer that hasn't restarted yet — and a v1 producer is still emitting data a v2 consumer must accept. Two compatibility directions are in play at once, so let's be precise about them:

DirectionMeansWho needs it
Backwardnew code reads old datathe upgraded reader, meeting not-yet-upgraded writers
Forwardold code reads new datathe not-yet-upgraded reader, meeting upgraded writers
Fullboth at onceany rolling deploy where upgrade order is unknown

Full compatibility is the prize, because it's what lets producers and consumers upgrade in any order — you stop having to choreograph deploys at all. And it's exactly what protobuf's additive-change rules buy you: give new fields new numbers, never reuse a retired number, reserved everything you delete. A well-behaved old consumer even preserves fields it doesn't understand — unknown fields survive a binary round-trip — so new data flows through it intact.

When a change is one-directional, order becomes correctness Not every change can be fully compatible. When one isn't, deployment order stops being a preference and becomes a requirement: teach every reader about the change before any writer produces it — i.e. deploy consumers ahead of producers. Ship a producer first and an old reader meets data it was never built for. The safe default is to prefer additive, fully-compatible changes so order doesn't matter; reach for choreographed ordering only when you truly can't.

And a trap that hides inside a change the tools call safe: widening int32 to int64 is wire-compatible — buf breaking waves it through — but an un-upgraded 32-bit reader will truncate any value that overflows the old range. The parser doesn't error; the number is just silently wrong. Wire-compatibility means “the decoder won't choke,” not “the value survives.” During such a migration, keep emitted values inside the old type's range until every reader has upgraded. 

message Order {
  string id = 1;
  // v1 shipped this as int32. v2 widens to int64 — WIRE-compatible,
  // but a v1 reader TRUNCATES values > 2^31-1. Keep totals in range
  // until every reader is on v2.
  int64 total_cents = 2;

  string coupon_code = 3 [deprecated = true];  // gradual retirement
  reserved 4;                                  // a number we deleted, never to reuse
  reserved "legacy_status";
}

Compatibility testing beyond the gate

buf breaking is a static check — it diffs schemas. That catches the mechanical breaks, but it can't tell you whether a specific producer/consumer pair actually round-trips real payloads, or whether your new Rust client encodes a oneof the way your Go server expects. Mature teams add two more layers. First, golden-payload tests: serialize a fixture with the old schema, parse it with the new one (and vice-versa), and assert the values you care about. This is where you'd catch that int32→int64 truncation with an actual over-range value, not just a rule. Second, the conformance suite from the previous chapter, run against every language runtime you deploy, so it's the implementations under test, not just the schema.

Three layers, three different questions buf breaking asks “is this schema change compatible in principle?” Golden-payload tests ask “do my old and new versions actually agree on my data?” The conformance suite asks “does each language runtime implement the spec correctly at all?” You want all three; skipping the middle one is how a technically-compatible change still breaks a real integration.

Observability: seeing the schema change

The last piece is knowing what's actually deployed. In a fleet of forty services you can't answer “which version of the Order contract is this service running?” by reading source. You answer it through descriptors. A FileDescriptorSet is the compiled, self-describing form of a schema — the schema as data — and a registry (or a gRPC server's reflection endpoint) can serve it over HTTP. With it you can introspect a running service's exact fields, options, and deprecation flags, and hydrate dashboards or conformance checks from CI images that don't even have the buf CLI installed.

Two everyday practices fall out of this. The first is deprecation as a visible signal: marking a field [deprecated = true] (or a message/enum/service with option deprecated = true;) makes generators emit the symbol as deprecated, so consumers get a compiler warning — a loud, gradual “this is going away” long before the field is ever deleted and its number reserved. That warning is observability at the source level. The second is the registry as an audit log: because every push is an immutable commit with an author and timestamp, the history of who changed the contract, when, and how becomes queryable — a git blame across every API in the organization. For most teams, that is what “observability of schema changes” concretely means.

A practical runtime signal to watch Unknown fields are protobuf's forward-compatibility mechanism — but a rising count of unknown-field bytes at a consumer is also a signal: producers are already emitting a newer schema this consumer hasn't caught up to. Surfacing that as a metric turns “we forgot to redeploy the reader” into a graph you can alert on, instead of a mystery you debug at 3 a.m.
So what did we actually gain? A schema that behaves like a real production artifact instead of a shared file everyone hopes stays stable. The registry versions and distributes it, so consumers install packages instead of vendoring drift. The CI gate (buf breaking, surfaced as PR comments) makes an incompatible change un-mergeable. The build pipeline (pinned remote plugins + managed mode) makes generated code identical on every machine. Rollout discipline — prefer fully-compatible changes; when you can't, deploy readers first — keeps out-of-step producers and consumers honest. And descriptors, deprecation, and the registry's commit history let you see schema change as it happens. None of this is about bytes. All of it is about running a contract that outlives any single deploy.
Before you turn the page
  1. Your teammate copies another team's .proto files into your repo “to keep it simple.” Name two concrete failures this causes over the next six months, and what a registry does instead.
  2. A PR widens a field from int32 to int64. buf breaking passes. Is the change safe? Under what exact runtime condition does it silently corrupt data, and what's the mitigation during rollout?
  3. You must ship a one-directional schema change (not fully compatible). In which order do you deploy producers and consumers, and why does the wrong order break?
  4. Your breaking-change CI check has been green for months, but a wire break reached production anyway. Give the single most likely misconfiguration — and the one line that fixes it.
  5. Someone asks “which version of the Order schema is the checkout service actually running right now?” What artifact answers that, and how do you fetch it without the service's source?
