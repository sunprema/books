Understanding the Protobuf Model

Last chapter we landed on a claim: a .proto file is a contract, checked by a compiler instead of hoped for by convention. Fair enough — but a contract needs terms. Before we write our first schema, let's slow down and look at the small, deliberately limited set of shapes a protobuf message is actually built from. Nothing here is arbitrary. Every rule in this chapter exists to make the evolution story from the last chapter — renaming fields safely, adding fields without breaking old readers — actually hold up.

A message is a box of numbered fields, not named ones

A protobuf message is a named, ordered collection of typed fields. So far that sounds like a struct in any language. Here's the twist: every field actually has two identities. It has a name, which is what you'll type in your generated Go struct or Python class — and it has a field number, a small positive integer that is the field's only identity once the message hits the wire. The name never leaves the .proto file and the code generated from it.

Here's a slightly larger message we'll use as a running example for the rest of this chapter. Don't worry about absorbing every line yet — it's here so we can point at concrete pieces of it as we go.

A miniature schema exercising every field shape in this chapter

syntax = "proto3";

package catalog.v1;

message Product {
  reserved 4;
  reserved "sku_legacy";

  string name = 1;
  int32 quantity = 2;
  optional string description = 3;

  enum Status {
    STATUS_UNSPECIFIED = 0;
    IN_STOCK = 1;
    BACKORDERED = 2;
  }
  Status status = 5;

  repeated string tags = 6;

  oneof pricing {
    int64 price_cents = 7;
    string price_quote_id = 8;
  }

  map<string, int32> warehouse_counts = 9;

  message Dimensions {
    float width_cm = 1;
    float height_cm = 2;
  }
  Dimensions dimensions = 10;
}

Field numbers are identity — and they're forever

A field number must be unique within its message, and it can be anywhere from 1 up to 536,870,911 (that's 229 − 1). One narrow slice, 19,000–19,999, is off-limits everywhere: it's reserved for the protobuf implementation itself, and protoc will simply refuse to compile a schema that uses one of those numbers.

Why low numbers are worth hoarding A field's tag on the wire bundles its number together with a small type code. Numbers 1–15 fit that whole tag into a single byte; 16–2047 need two. It costs nothing to know this up front — just give your most frequently set fields (in Product, that's probably name and quantity) the lowest numbers, and save the middle of the range for fields you're not sure you'll keep.

Notice the top of Product: reserved 4; and reserved "sku_legacy";. Those two statements exist because a field called sku_legacy used to sit at number 4, and somebody removed it. Reserving both the number and the name tells the compiler — and every future engineer who touches this file — that neither one is available for reuse. You can't combine them into one statement; numbers and names each get their own reserved line. The exact same mechanism works on enum values, which matters in a page or two.

The mistake this prevents Skip the reserved statement and a teammate, six months later, adds a brand-new field and innocently reuses number 4 for it. Every message still in flight from an old binary, or sitting in a log or a database, that was written when 4 meant sku_legacy now gets silently misread as the new field's type. No crash, no error — just quietly wrong data. This is the sharpest edge of "the wire only sees numbers": it cuts both ways.

Every field has a type — and a well-defined empty state

Proto3 doesn't let you invent scalar types; you pick from a fixed catalog, grouped by how they're meant to be used:

FamilyTypesEmpty-state default
Plain integers int32, int64, uint32, uint64 0
Sign-optimized integers sint32, sint64 0 — cheaper on the wire for values that are often negative
Fixed-width integers fixed32, fixed64, sfixed32, sfixed64 0 — always 4 or 8 bytes, never smaller
Floating point float, double 0.0
Text & bytes string, bytes empty string / empty bytes
Logic bool false

Read that last column again, because it's the setup for the next section: an unset scalar field doesn't read back as null. It reads back as its type's default — and by default, protobuf has no way to tell you whether that default is really there, or just standing in for nothing.

The gotcha hiding in "empty": field presence

Look again at quantity in our schema — a plain int32, no modifier. If a client sends a Product with quantity left untouched, your server reads 0. If a client explicitly sets quantity = 0 because that product really is out of stock, your server also reads 0. Proto3 calls this implicit presence: for a plain scalar or enum field, "unset" and "set to the default" are the same bit pattern, and the generated API simply cannot tell them apart.

That's exactly why description in our schema is marked optional. The keyword was dropped from early proto3 and then reintroduced (in protobuf 3.15) specifically to close this gap: an optional scalar field gets explicit presence, meaning generated code can ask "was this actually set?" the way proto2 always could. Under the hood, the compiler does something almost sneaky to pull this off — it wraps the field in a one-member synthetic oneof that never appears in your .proto source but does exist in the compiled descriptor, quietly reusing the presence tracking that oneofs already needed.

Key idea Message-typed fields never have this problem — an unset Dimensions dimensions is always distinguishable from one explicitly set to an empty Dimensions{}, in both proto2 and proto3. Presence ambiguity is specifically a scalar and enum quirk, and it's exactly what optional exists to fix.

Keep this distinction in your back pocket. When we get to schema evolution, "can a reader tell an old default from an intentional zero?" turns out to be one of the first questions you need to answer honestly about any field you're about to add.

Four shapes a field can take

Beyond singular (one value, the case we've been discussing), a field can take three other shapes — and our Product message uses all of them.

ShapeIn our schemaBehavior
repeated repeated string tags An ordered list of the element type — protobuf's array. Repeated scalar fields are packed into one compact entry on the wire by default.
oneof oneof pricing { price_cents; price_quote_id; } At most one member is ever set. Assigning any member silently clears whichever member was set before — oneof members can't be repeated or map.
map<K,V> map<string, int32> warehouse_counts Shorthand for a repeated key/value entry message. Wire-compatible with that longhand form, but entry order isn't preserved and duplicate keys aren't well-defined.
A oneof isn't a group of optionals It's tempting to reach for a oneof just to get explicit presence on a couple of related fields. Remember what actually happens: setting price_cents and then setting price_quote_id doesn't give you both — the second assignment clears the first. A oneof is for fields that are genuinely mutually exclusive alternatives, not a batch way to get optional's presence tracking.

Enums: closed sets that must start at zero

Our Status enum starts at STATUS_UNSPECIFIED = 0 and that's not a style choice — it's a rule. An unset Status field reads back as whichever enum value is numbered 0, so every enum you write needs one, and it needs to actually mean "no opinion yet" rather than a real state you'd be surprised to see show up by default. Two enum names are allowed to share the same number, but only if you opt in explicitly with option allow_alias = true — without it, the compiler treats a duplicate number as an error, not a convenience.

And because an enum is really just a small closed set of number/name pairs, the exact same reserved mechanism from earlier applies here too: remove BACKORDERED someday, and you'd write reserved 2; reserved "BACKORDERED"; inside the enum, for the same reason you'd reserve a removed message field.

Nesting and packages keep a large schema from colliding

Dimensions is declared inside Product, which is nothing more than scoping — generated code refers to it as Product.Dimensions. You'll do this whenever a type only makes sense attached to its parent and you don't want it cluttering the top-level namespace of the file.

The package catalog.v1; line at the top does the same job one level up, for the whole file: it namespaces every type this file defines, so a Product message in catalog.v1 and an unrelated Product message in, say, billing.v1, compile into two distinct types in your target language instead of colliding. Once you have more than a couple of .proto files importing each other, this stops being optional hygiene and starts being the thing that keeps the whole schema buildable.

You also don't have to invent every type yourself. Protobuf ships a small standard vocabulary under google.protobuf for problems every schema eventually hits: Timestamp and Duration for time, Any for a payload that carries its own type tag, Struct and Value for dynamic JSON-shaped data, FieldMask for naming which fields a partial update actually touched, and a family of scalar wrapper messages (Int32Value, StringValue, and friends) that solve the presence problem from a few pages ago the old-fashioned way — by making the scalar a message field, which always has explicit presence.

A peek ahead: what a tag actually carries

One last thread to leave dangling on purpose. Every field's tag on the wire isn't just its number — it also packs in a 3-bit wire type, computed as (field_number << 3) | wire_type. There are four of them, and which one a field gets is fully determined by its declared type:

Wire typeCarriesUsed by
Varinta variable-length integerint32/int64/uint32/uint64/bool/enums
64-bita fixed 8 bytesfixed64, sfixed64, double
Length-delimiteda length, then that many bytesstring, bytes, embedded messages, packed repeated fields
32-bita fixed 4 bytesfixed32, sfixed32, float

That's genuinely all a decoder needs to skip a field it doesn't recognize — it reads the wire type, knows how many bytes to consume (or where to find the length), and moves on without ever consulting your schema. We'll write our first real .proto file next, and then dedicate a whole chapter to watching this byte-by-byte. For now, just notice that everything in this chapter — numbers instead of names, mandatory enum zeros, explicit-presence opt-ins — exists in service of that one decoder trick.

So what did we actually gain?

Compare this to a JSON object, which will accept any shape you throw at it. Protobuf's model looks restrictive by comparison — a fixed catalog of scalar types, only four field shapes, an enum that's forced to define its own "nothing to see here" value. That restriction is the entire point. Because a message can only be built from these few, well-understood pieces, the compiler can reason about every one of them: it knows exactly what changing a field's number means, exactly what an unset field looks like, exactly how a decoder should behave when it meets a byte pattern it's never seen before.

None of that is incidental. It's the raw material the schema-evolution discipline from Chapter 1 is built out of. You now have the vocabulary — field, number, presence, shape, tag — to actually write one of these schemas yourself, which is exactly where we're headed next.
