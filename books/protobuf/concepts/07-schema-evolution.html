Chapter 07 · Evolution & Interop

Schema Evolution

Every chapter so far has been about writing a schema. This one is about the day, six months or six years later, when you have to change it — while an old mobile client, a half-updated backend fleet, and a three-year-old message sitting in a queue all keep working without anyone coordinating a synchronized rollout. That's not a hope. It's a small, learnable set of rules, and protobuf's whole design — field numbers instead of names, no schema on the wire, unknown fields kept instead of dropped — exists to make those rules possible in the first place. Break one of them and nothing crashes. Something just goes quietly, plausibly wrong, which is the worse failure to ship.

Two directions, and why both matter

Get the vocabulary straight first, because the two terms get swapped constantly and they mean different things. Backward compatible means your new code can correctly read data an old version wrote. Forward compatible means your old code can correctly read data a new version wrote. A change that's fully compatible holds in both directions at once — and that's the property that actually matters in production, because it's what lets one service deploy a schema change on Tuesday while three other teams' services, holding the old schema, keep running untouched until they get around to it. Nobody has to deploy in lockstep. That's the whole payoff of this chapter.

Everything below is really one question, asked twice: does an old reader survive this change, and does a new reader survive it too? Most of the rules in this chapter turn out to answer “yes” to both at once — that's not a coincidence, it's the reason protobuf's format was shaped the way it was.

Names are free. Numbers are forever.

Here's the single fact almost everything else in this chapter derives from, and you already have half of it from Chapter 2: the wire format carries a field's number, never its name. That asymmetry cuts both ways, and both halves matter. Renaming quantity to stock_count touches nothing on the wire — it's a generated-code change, full stop, and every peer exchanging bytes with you never notices. Changing a field's number, on the other hand, is not a small edit. It is, on the wire, indistinguishable from deleting that field and inventing an entirely new one that happens to share a type. Every message already serialized under the old number, and every peer still compiled against it, is now reading (or about to write) the wrong identity. There is no safe way to change a field number once it has shipped. Not “risky” — not possible. This is the one rule in the whole chapter that never bends.

The rule underneath all the other rules If you remember nothing else from this chapter: a field's number is a permanent identity, assigned once. Its name, type, and even its existence can change around it — the number can't.

Adding a field

Adding a field is the easy, load-bearing case, and it's worth seeing exactly why it's safe rather than just trusting that it is. Pick a number that was never used before in this message and add the field. An old reader, compiled before this field existed, encounters bytes tagged with a number it has no schema entry for. It doesn't error. It reads the wire type off the tag alone — no schema needed for that part — skips exactly that many bytes, and, since protobuf 3.5, holds onto them as an unknown field rather than throwing them away. If that old reader turns around and re-serializes the same parsed message (forwarding it, storing it, relaying it), those bytes come back out unchanged, even though the code that did the relaying still has no idea what they mean. That's forward compatibility, mechanically: new data survives old code completely intact.

Run it the other direction and it's backward compatibility, just as mechanically: a new reader parsing an old message that predates the field simply finds the field absent, and reads back whatever “absent” means for that field's presence rules — the zero value for a plain scalar, false for has_x() if it's optional, an empty list for repeated. Nothing to reconcile, nothing to migrate. Adding a field is compatible in both directions, for free, as long as the number is genuinely unused.

Removing a field — and why reserved exists

Deleting a field is also wire-safe in both directions — there's nothing left in the schema for a decoder to choke on. The part that isn't automatically safe is what happens next, to a number nobody is watching anymore. Nothing stops a teammate, six months from now, from adding a brand-new field and letting the editor autocomplete hand them the next integer — which might be the exact number you just freed up. If any peer anywhere is still running the old schema, it now decodes your teammate's new field as if it were the field you deleted. No error. No warning. Just a plausible, wrong value, silently, forever, until someone notices the data doesn't add up.

reserved exists to turn that landmine into a compile-time error instead. Say it once for the number, once more for the name — the two need separate statements, you can't mix them in one — and protoc refuses to let anyone reintroduce either: 

message Product {
  // v1 shipped: name=1, quantity=2, sku_legacy=3

  string name = 1;
  int32 quantity = 2;

  reserved 3;
  reserved "sku_legacy";   // number AND name, both off-limits forever
}

reserved also takes a comma-separated list or a range (reserved 2, 15, 9 to 11;, or 9 to max; to retire everything from 9 up), and the identical mechanism protects enum values — delete a value, reserve its number and its name the same way, for the same reason: JSON and text-format encodings address fields and enum values by name, so a reused name can corrupt those formats even when the binary wire format wouldn't care. A documented alternative, if you'd rather the retirement stay visible in the .proto to a human skimming it, is to keep the field but rename it with an OBSOLETE_ prefix instead of deleting it outright — readable, but it doesn't get the compiler's enforcement the way reserved does.

Never, ever reuse a field number Not “avoid it when you can.” Never. A reused field number is the one schema mistake in this entire book that produces no error anywhere in the pipeline — just quietly wrong data, possibly for a long time before anyone notices. reserved is cheap insurance against a very expensive bug.

Deprecating, as a step before deleting

Between “in active use” and “deleted and reserved,” there's a middle step worth using deliberately rather than skipping: the [deprecated = true] field option. 

string internal_note = 5 [deprecated = true];

This changes nothing about the wire format and nothing about runtime behavior — the field still works exactly as before. What it changes is what a generated-code caller sees: most language codegens surface it as a compiler or IDE warning on the accessor, a nudge to new callers before you ever get to the actual deletion-plus-reserved step. It's a signal, not an enforcement mechanism — nothing stops someone from ignoring the warning and setting the field anyway — but it's the honest, visible way to announce “this is going away” before it actually does.

Safe vs. unsafe: changing a field's type

This is where the wire-type groundwork from Chapter 4 finally pays off directly. A field's wire type — varint, 64-bit, length-delimited, or 32-bit — is fixed by its declared protobuf type and travels on every tag. Two types sharing a wire type is necessary for a type change to be safe, but it is not sufficient — the bytes can parse cleanly and still mean the wrong thing. Both failure modes exist in this table: a hard parse-level break, and the quieter, worse one, a clean parse that silently produces the wrong value.

The gotcha that shares a wire type on purpose int32 and sint32 both use wire type 0 — so a decoder never rejects the bytes. It just silently decodes the wrong number. Watch what happens to the same logical value, −2, depending only on which of the two the field was declared as: 
// field declared int32, value -2:
//   sign-extended to 64 bits -> costs the full 10-byte varint
//   wire bytes: FE FF FF FF FF FF FF FF FF 01

// SAME field redeclared sint32, SAME logical value -2:
//   ZigZag(-2) = 3 -> fits in one byte
//   wire bytes: 03

// an old peer still expecting int32, decoding byte 0x03 as a plain varint:
//   reads back the value 3 -- not -2, and no parse error anywhere

That's the general shape of every unsafe-but-same-wire-type change in the table above: nothing throws, nothing logs, a value just quietly stops meaning what it used to mean. Treat “does it parse” and “does it mean the same thing” as two separate questions — this chapter's whole safe/unsafe table exists because those two questions don't always have the same answer.

Singular and repeated: the packed asterisk

The singular-vs-repeated row above deserves its own beat, because the answer genuinely depends on the field's underlying type in a way none of the other rows do. A repeated message field is simple: an embedded message is always length-delimited, singular or repeated, so an old singular-field reader parsing what's now a repeated stream of that message just keeps merging every occurrence it sees — which lands it on the last one, the same “last write wins” behavior a singular field always had. Safe, no asterisk.

A repeated scalar, bool, or enum field is where the asterisk lives. Proto3 defaults a repeated scalar field to packed encoding — every element's varint or fixed bytes concatenated back-to-back inside one length-delimited entry, with no per-element tag. That's a genuinely different byte shape than a singular field of the same type expects to find. A parser must accept a repeated field written either packed or unpacked no matter which one the current .proto declares — that bidirectional acceptance is exactly what makes toggling [packed=true]/[packed=false] itself a safe change. But it doesn't rescue the singular-vs-repeated case: for that change to be safe, the repeated field has to be explicitly unpacked.

One asterisk, easy to remember Repeated message field ↔ singular: always fine. Repeated scalar/bool/enum field ↔ singular: fine only with [packed=false]. Same “singular vs. repeated” question, two different answers, depending on what's inside the list.

Why forwarding services don't need a rebuild

Chapter 4 introduced unknown-field preservation as a wire-format fact; here's why it's the specific mechanism that makes forward compatibility actually usable in a real system, not just theoretically true. A proxy, a message bus, an audit-log writer — any service that receives a message, does something with the fields it recognizes, and then passes the same parsed object along — carries every field it doesn't understand along for the ride, byte-for-byte, without ever being rebuilt to know that field exists. Since protobuf 3.5 restored this behavior (proto3 briefly dropped it, then brought it back to match proto2), that's not a special case you have to opt into. It's the default.

The guarantee is precisely scoped, though, and the scope is worth being exact about: it covers re-serializing the original parsed object. The moment you convert that message to JSON, or build a brand-new message and copy fields into it one at a time, you've stepped outside the guarantee — both operations only carry over the fields the code explicitly names, and anything else is gone. A service that forwards the object it received stays forward-compatible for free. A service that reconstructs one field-by-field has quietly opted back out.

The cautionary tale: proto2's required

If you want the whole chapter's argument distilled into one historical example, it's this. proto2 let a field be marked required, and it seemed reasonable at the time — some fields really are always present in a well-formed message, so why not have the parser enforce it? The trap wasn't in adding the constraint. It was in ever trying to remove it later. Once one deployed reader anywhere rejects a message that omits a required field, and one deployed writer anywhere might legitimately want to omit it, there is no safe order to roll out that change — you need every reader and every writer updated in the same instant, which is precisely the coordination problem this whole chapter's rules exist to make unnecessary.

Proto3 didn't patch required. It removed the concept entirely, on the theory that a field that looks required today may turn out not to be next year, and a schema language shouldn't offer a constraint that can be added painlessly but never safely taken away. Every rule in this chapter is a smaller version of the same lesson: ask not just “is this change safe now,” but “does this change leave me a safe way out later.”

The lesson underneath the lesson required wasn't removed because it was unsafe to add. It was removed because there was no safe way to ever take it back off. That asymmetry — easy to add, impossible to undo — is the exact shape to watch for in your own schema decisions.

Let a tool hold the line

None of these rules are exotic once you've seen them, but a schema shared across a dozen services and three languages doesn't stay safe because every contributor memorized this chapter — it stays safe because something checks. Tools like buf breaking diff a proposed schema against a previous version (a git ref, a tag, a stored snapshot) and fail the build the moment a change violates one of these rules: a reused or renumbered field number, an unsafe type swap, a field or enum value gone without a matching reserved. Wired into CI as a required check on every pull request, that turns “remember not to reuse field 3” from institutional knowledge into something the build enforces whether or not anyone remembers it. We'll wire exactly this up in the production chapters ahead.

One honest caveat: a breaking-change tool's rules aren't always identical to raw wire compatibility. Some are deliberately stricter — flagging, say, a schema reorganization that wouldn't actually corrupt a single byte on the wire, because it's still risky enough in practice to want a human's sign-off. “The tool didn't flag it” and “this is wire-compatible” are related claims, not identical ones — the tool errs toward caution on your behalf.

Before you turn the page
  1. A teammate deletes a field, forgets to write reserved, and a month later a different teammate adds a new field that happens to land on the freed-up number. Walk through exactly what an old, undeployed peer does with the bytes — does it error, and if not, what does it actually read?
  2. Why is sint32 ↔ int32 unsafe even though both use wire type 0, while int32 ↔ uint32 is safe on that same wire type? Name the one difference between them that matters.
  3. You want to change repeated int32 scores to singular, since it turns out only one score is ever set. What field option, if any, do you need before that's actually safe — and would the answer change if the field were repeated ScoreRecord instead?
  4. Explain, to someone who's never heard of proto2, why required was removed in proto3 — not what it did, but specifically why it was dangerous.
