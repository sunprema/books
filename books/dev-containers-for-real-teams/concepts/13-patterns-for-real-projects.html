Patterns for Real Projects

Every project has constraints that a generic Dev Container tutorial never covers. This chapter presents seven battle-tested patterns—each with a concrete devcontainer.json snippet you can lift and adapt. Pick the pattern that fits your repo’s shape.

Pattern 1: Monorepo with Multiple Configurations

A monorepo with a Go API, a React frontend, and shared infrastructure code needs different toolchains per sub-project. The Dev Container spec supports named configurations—multiple devcontainer.json files under .devcontainer/, each in its own subdirectory.

# Directory layout
.devcontainer/
  api/
    devcontainer.json       # Go + Delve + protobuf
    Dockerfile
  web/
    devcontainer.json       # Node 20 + Playwright
  infra/
    devcontainer.json       # Terraform + AWS CLI
  devcontainer.json         # Default: full-stack (all tools)

When a user opens the repo, VS Code and Codespaces prompt them to pick a configuration. Each config targets the right workspace folder:

// .devcontainer/api/devcontainer.json
{
  "name": "API (Go)",
  "build": {
    "dockerfile": "Dockerfile",
    "context": "../.."
  },
  "workspaceFolder": "/workspaces/${localWorkspaceFolderBasename}",
  "features": {
    "ghcr.io/devcontainers/features/go:1": { "version": "1.22" },
    "ghcr.io/devcontainers-extra/features/protoc:1": {}
  },
  "customizations": {
    "vscode": {
      "extensions": ["golang.go", "zxh404.vscode-proto3"],
      "settings": {
        "go.lintTool": "golangci-lint",
        "go.testFlags": ["-race", "-count=1"]
      }
    }
  },
  "forwardPorts": [8080],
  "postCreateCommand": "cd api && go mod download"
}
Key insight The default .devcontainer/devcontainer.json at the root is important—it’s what Codespaces uses if the user doesn’t pick a named config. Make it the “kitchen sink” config that installs everything, so nobody gets a broken environment by accident.

One gotcha: named configurations share the same Git repo mount. If your API config installs Go and your web config installs Node, you’re fine—they’re separate containers. But watch out for postCreateCommand scripts that assume they’re the only tenant. Use relative paths to the sub-project, not absolute repo-root assumptions.

Pattern 2: Polyglot Stack

You have a Python backend and a React frontend. The question: one container or two?

ApproachProsConsUse when
Single container Simple config, shared filesystem, easier debugging Heavier image, version conflicts possible Small team, tightly coupled stack
Docker Compose Isolated runtimes, mirrors production topology More YAML, cross-container debugging harder Teams >5, independent deploy cycles

Single container approach—install both runtimes via Features:

{
  "name": "Fullstack (Python + React)",
  "image": "mcr.microsoft.com/devcontainers/base:ubuntu",
  "features": {
    "ghcr.io/devcontainers/features/python:1": { "version": "3.12" },
    "ghcr.io/devcontainers/features/node:1": { "version": "20" }
  },
  "forwardPorts": [8000, 3000],
  "postCreateCommand": "pip install -r api/requirements.txt && cd web && npm ci",
  "customizations": {
    "vscode": {
      "extensions": [
        "ms-python.python",
        "dbaeumer.vscode-eslint"
      ]
    }
  }
}

Compose approach—the backend and frontend are separate services:

// .devcontainer/devcontainer.json
{
  "name": "Fullstack (Compose)",
  "dockerComposeFile": "docker-compose.yml",
  "service": "api",
  "workspaceFolder": "/workspaces/${localWorkspaceFolderBasename}",
  "forwardPorts": [8000, 3000],
  "customizations": {
    "vscode": {
      "extensions": ["ms-python.python", "dbaeumer.vscode-eslint"]
    }
  }
}
Foot-gun With Compose, the "service" field determines which container VS Code attaches to. Extensions run inside that container. If you attach to the Python API container, the ESLint extension won’t find Node—install it in that container or use a multi-root workspace.

Pattern 3: Data Science & ML

ML workloads need GPU passthrough, large dataset mounts, and Jupyter. The spec supports all three, but the setup has sharp edges.

{
  "name": "ML Workbench",
  "build": {
    "dockerfile": "Dockerfile"
  },
  "hostRequirements": {
    "gpu": "optional"
  },
  "features": {
    "ghcr.io/devcontainers/features/python:1": { "version": "3.11" },
    "ghcr.io/devcontainers/features/nvidia-cuda:1": { "cudaVersion": "12.2" }
  },
  "mounts": [
    "source=/data/datasets,target=/datasets,type=bind,readonly"
  ],
  "containerEnv": {
    "JUPYTER_TOKEN": "devtoken",
    "CUDA_VISIBLE_DEVICES": "0"
  },
  "forwardPorts": [8888],
  "postCreateCommand": "pip install -r requirements.txt && jupyter lab --no-browser --port=8888 &",
  "customizations": {
    "vscode": {
      "extensions": [
        "ms-python.python",
        "ms-toolsai.jupyter"
      ],
      "settings": {
        "jupyter.notebookFileRoot": "${workspaceFolder}"
      }
    }
  }
}
Note "gpu": "optional" means the container works without a GPU but uses one if available. Use "gpu": true to require it—Codespaces will provision a GPU-enabled machine. Locally, you need the NVIDIA Container Toolkit installed on the host and Docker configured with the nvidia runtime.

Datasets are the big gotcha here. Never COPY a 50 GB dataset into the image. Bind-mount it read-only from the host, or use a named Docker volume that persists across rebuilds. For cloud environments, mount from object storage at container start via initializeCommand.

Pattern 4: API-First Development

When your service depends on five other APIs you don’t own, you need mock services and contract testing baked into the container.

{
  "name": "API Service",
  "dockerComposeFile": "docker-compose.yml",
  "service": "app",
  "workspaceFolder": "/workspaces/${localWorkspaceFolderBasename}",
  "containerEnv": {
    "PAYMENT_API_URL": "http://wiremock:8080",
    "AUTH_API_URL": "http://wiremock:8080",
    "NODE_ENV": "development"
  },
  "forwardPorts": [3000, 8080],
  "postCreateCommand": "npm ci && npm run generate:client",
  "customizations": {
    "vscode": {
      "extensions": [
        "42Crunch.vscode-openapi",
        "humao.rest-client"
      ]
    }
  }
}

# docker-compose.yml
services:
  app:
    build:
      context: ..
      dockerfile: .devcontainer/Dockerfile
    volumes:
      - ..:/workspaces/${LOCAL_WORKSPACE_FOLDER_BASENAME}
    depends_on:
      - wiremock

  wiremock:
    image: wiremock/wiremock:3.5.4
    volumes:
      - ./mocks:/home/wiremock/mappings
    ports:
      - "8080:8080"

Place your WireMock stubs in .devcontainer/mocks/. Every developer gets the same mock responses, and your contract tests validate against the same stubs that back your dev environment.

Why this matters: When mocks live alongside the dev container config, they’re versioned with the code. A new API field? Update the stub and the client code in the same PR.

Pattern 5: Legacy Application

You maintain a Rails 5 app on Ruby 2.7, or a .NET Framework 4.8 service. The host OS has moved on, but your codebase hasn’t. Dev Containers are an escape hatch.

{
  "name": "Legacy Rails App",
  "build": {
    "dockerfile": "Dockerfile"
  },
  "containerEnv": {
    "RAILS_ENV": "development",
    "BUNDLE_PATH": "/bundle"
  },
  "mounts": [
    "source=legacy-bundle-cache,target=/bundle,type=volume"
  ],
  "forwardPorts": [3000],
  "postCreateCommand": "bundle install && rails db:setup",
  "customizations": {
    "vscode": {
      "extensions": ["shopify.ruby-lsp"]
    }
  }
}

# Dockerfile
FROM ruby:2.7-bullseye

# Pin OS to Debian Bullseye — Bookworm dropped libs this app needs
RUN apt-get update && apt-get install -y \
    libmysqlclient-dev \
    imagemagick \
    && rm -rf /var/lib/apt/lists/*

RUN gem install bundler:2.3.27

WORKDIR /workspaces
Gotcha Old runtimes often need old OS packages. Pin the base image to a specific Debian/Ubuntu release, not :latest. Document why in a comment so the next person doesn’t “helpfully” upgrade it and break everything.

Legacy apps are exactly where Dev Containers pay for themselves. New developers don’t need to figure out how to install Ruby 2.7 on an M3 Mac. They open the repo, pick the container, and they’re running the app in five minutes.

Pattern 6: Open Source Project

The goal: a contributor clones your repo, opens it in VS Code (or Codespaces), and can run tests within minutes. Zero setup docs needed.

{
  "name": "my-oss-project",
  "image": "mcr.microsoft.com/devcontainers/base:ubuntu",
  "features": {
    "ghcr.io/devcontainers/features/node:1": { "version": "20" },
    "ghcr.io/devcontainers/features/git:1": {},
    "ghcr.io/devcontainers/features/github-cli:1": {}
  },
  "containerEnv": {
    "CI": "true"
  },
  "postCreateCommand": "npm ci && npm run build",
  "postStartCommand": "echo '\\n\\033[1;33mRun: npm test\\033[0m\\n'",
  "customizations": {
    "vscode": {
      "extensions": [
        "dbaeumer.vscode-eslint",
        "esbenp.prettier-vscode"
      ],
      "settings": {
        "editor.formatOnSave": true,
        "editor.defaultFormatter": "esbenp.prettier-vscode"
      }
    }
  }
}
Key insight Set "CI": "true" in containerEnv so formatters and linters run with the same strictness as CI. Contributors catch issues before pushing. Add a postStartCommand that prints the next step—new contributors shouldn’t have to hunt for it.

For Codespaces, add a .devcontainer/devcontainer.json plus a badge in your README:

<!-- In README.md -->
[![Open in GitHub Codespaces](
  https://github.com/codespaces/badge.svg
)](https://codespaces.new/your-org/your-repo?quickstart=1)

Pattern 7: Microservices with Docker Compose

You own one service but need the surrounding mesh (database, message queue, neighbor services) running locally. Docker Compose describes the topology; the Dev Container attaches to your service.

// .devcontainer/devcontainer.json
{
  "name": "Order Service",
  "dockerComposeFile": ["../docker-compose.yml", "docker-compose.dev.yml"],
  "service": "order-service",
  "workspaceFolder": "/workspaces/${localWorkspaceFolderBasename}",
  "shutdownAction": "stopCompose",
  "forwardPorts": [3000, 5432, 15672],
  "containerEnv": {
    "DATABASE_URL": "postgres://dev:dev@db:5432/orders",
    "RABBITMQ_URL": "amqp://guest:guest@rabbitmq:5672"
  },
  "postCreateCommand": "npm ci && npx prisma migrate dev"
}

# .devcontainer/docker-compose.dev.yml
# Overrides for dev — adds volume mounts and dev tools
services:
  order-service:
    build:
      context: ..
      dockerfile: .devcontainer/Dockerfile
    volumes:
      - ..:/workspaces/${LOCAL_WORKSPACE_FOLDER_BASENAME}
      - node_modules:/workspaces/${LOCAL_WORKSPACE_FOLDER_BASENAME}/node_modules
    command: sleep infinity

volumes:
  node_modules:

The trick is the two-file Compose array: the first file is your production docker-compose.yml that your ops team maintains. The second is a dev-only overlay that adds source mounts and swaps the entrypoint to sleep infinity so the Dev Container runtime takes over. You never fork the production Compose file.

Foot-gun node_modules on a bind mount between macOS/Windows hosts and Linux containers is catastrophically slow. Use a named volume for node_modules (as shown above) to keep native binaries inside the container filesystem.

Choosing a Pattern

If your project is…Start withThen layer on
A monorepo with distinct sub-projectsPattern 1 (named configs)Pattern 7 for shared services
Python + JS in the same codebasePattern 2 (single container)Split to Compose when the image gets heavy
ML/Data Science with notebooksPattern 3 (GPU + Jupyter)Pattern 5 if pinning CUDA versions
An API that depends on external servicesPattern 4 (mocks)Pattern 7 for the full mesh
Stuck on old runtimesPattern 5 (legacy pinning)Pattern 6 to ease onboarding
An open source libraryPattern 6 (contributor-friendly)Codespaces badge for zero-install
One service in a larger systemPattern 7 (Compose mesh)Pattern 1 if the service has sub-components
Why this matters: Patterns are combinable. A real project often blends two or three. Start with the dominant shape of your project, get that config working, then layer in additional patterns as complexity demands it.
