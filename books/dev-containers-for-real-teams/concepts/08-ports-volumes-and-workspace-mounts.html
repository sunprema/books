Ports, Volumes, and Workspace Mounts

Your container runs services. Your host runs the browser. Ports bridge that gap. Your code lives on the host. The container needs to read and write it. Mounts bridge that gap. Getting both wrong means slow builds, unreachable servers, and mysterious file permission errors. This chapter covers the mechanics and the production trade-offs.

Port Forwarding

VS Code auto-detects ports that processes listen on inside the container and offers to forward them. But relying on auto-detection in a team setting is fragile. Declare your ports explicitly:

{
  "forwardPorts": [3000, 5432, 6379],

  "portsAttributes": {
    "3000": {
      "label": "App",
      "onAutoForward": "openBrowser"
    },
    "5432": {
      "label": "Postgres",
      "onAutoForward": "silent"
    },
    "6379": {
      "label": "Redis",
      "onAutoForward": "silent"
    }
  }
}

portsAttributes Options

Property Values Effect
label string Human-readable name shown in the Ports panel
onAutoForward notify | openBrowser | openBrowserOnce | silent | ignore What happens when the port is detected. silent forwards without a notification. ignore does not forward at all.
protocol http | https Protocol VS Code uses when opening the browser
requireLocalPort boolean If true, fails instead of picking an alternate port when the host port is taken
elevateIfNeeded boolean Request elevated permissions for privileged ports (<1024)

appPort vs. forwardPorts

These look similar but behave differently:

Property Mechanism When to use
forwardPorts VS Code tunnels the port via its server connection Default choice. Works with any image. No Docker flag needed.
appPort Maps to Docker's -p flag (publishes the port on the host network) When non-VS Code tools on the host need the port, or when you need the port available before VS Code connects.
Note

appPort exposes ports on your host network directly via Docker. On shared machines or CI, this can conflict with other containers. Prefer forwardPorts unless you have a specific reason not to.

Workspace Mounts

By default, VS Code bind-mounts your project folder into the container at /workspaces/<project-name>. This is convenient but comes with trade-offs you need to understand.

# The default behavior is equivalent to:
{
  "workspaceMount": "source=${localWorkspaceFolder},target=/workspaces/${localWorkspaceFolderBasename},type=bind,consistency=cached",
  "workspaceFolder": "/workspaces/${localWorkspaceFolderBasename}"
}

You can override both. workspaceMount controls how the source is mounted. workspaceFolder controls where VS Code opens inside the container.

Foot-gun

If you set workspaceMount without setting workspaceFolder, VS Code may open the wrong directory. Always set both together.

Named Volumes and the macOS Performance Problem

On macOS, bind mounts go through a virtualization layer (VirtioFS or gRPC-FUSE) that adds latency to every filesystem call. For a Node project with 100,000+ files in node_modules, this means:

  • npm install takes 3-5x longer than native
  • TypeScript compilation and webpack/vite hot-reload become sluggish
  • Test suites with heavy file I/O crawl

The fix: move high-churn directories to named volumes that live inside Docker's Linux VM:

{
  "mounts": [
    {
      "source": "devcontainer-node_modules",
      "target": "/workspaces/myapp/node_modules",
      "type": "volume"
    },
    {
      "source": "devcontainer-dotcache",
      "target": "/home/node/.cache",
      "type": "volume"
    }
  ],
  "postCreateCommand": "npm ci"
}

Named volumes are fast because they bypass the host filesystem entirely—the data lives in Docker's Linux VM. The trade-off: you cannot browse node_modules from Finder or Explorer. For dependency directories, that is almost always fine.

Common Volume Targets by Language

Language Directory Why volume-mount it
Node.js node_modules 100k+ small files, constant reads during bundling
Rust target Incremental compilation artifacts, heavy I/O
Go ~/go/pkg/mod Module cache, reused across builds
Python .venv Virtual environment with many small files
PHP vendor Composer dependencies
Java ~/.m2, ~/.gradle Large artifact caches

tmpfs Mounts for Test Artifacts

For throwaway data like test outputs, coverage reports, or build caches that do not need to survive container restarts, use a tmpfs mount. It lives in memory—maximum speed, zero disk I/O:

{
  "mounts": [
    {
      "target": "/workspaces/myapp/coverage",
      "type": "tmpfs"
    },
    {
      "target": "/tmp/test-results",
      "type": "tmpfs"
    }
  ]
}

Mount Consistency Flags

Bind mounts accept a consistency flag that controls how aggressively Docker syncs between host and container. This is primarily relevant on macOS with the older osxfs/gRPC-FUSE driver:

Flag Behavior When to use
consistent Full consistency. Writes visible immediately on both sides. Default. Safe but slowest on macOS.
cached Host view is authoritative. Container may see stale reads briefly. Best for source code. This is what Dev Containers uses by default.
delegated Container view is authoritative. Host may see stale reads. Good for build output directories the host only reads after builds finish.
Note

Docker Desktop 4.15+ on macOS defaults to VirtioFS, which largely eliminates the performance gap between consistency modes. The flags still work but have less impact. Named volumes remain faster for extremely I/O-heavy directories.

Data Flow: Host, Bind Mount, and Named Volume

Why this matters

Port forwarding and mount configuration are where Dev Containers go from "neat demo" to "daily driver." Get forwardPorts right and your team stops fighting port conflicts. Move node_modules to a named volume on macOS and npm install drops from 90 seconds to 20. Use tmpfs for test artifacts and your CI containers stop filling up disk. These are small config changes with outsized impact on daily velocity.
