Extensions, Settings, and Environment

A Dev Container without extensions and settings is just a blank Docker image with VS Code bolted on. The customizations.vscode object is where you turn that blank slate into a fully configured workspace—linters running, formatters on save, environment variables injected, Git configured. Get this right and your team opens the container ready to ship code in under a minute.

Installing Extensions Automatically

The extensions array inside customizations.vscode installs extensions the moment the container builds. No marketplace clicks, no "did you install the linter?" in Slack.

{
  "customizations": {
    "vscode": {
      "extensions": [
        "dbaeumer.vscode-eslint",
        "esbenp.prettier-vscode",
        "ms-python.python",
        "eamodio.gitlens",
        "bradlc.vscode-tailwindcss"
      ]
    }
  }
}

Use the full publisher.extension-name identifier. You can find it in the VS Code marketplace URL or by running code --list-extensions on a working machine.

UI Extensions vs. Workspace Extensions

VS Code runs two extension hosts. Understanding which one runs where saves you from "extension not found" mysteries:

Extension type Runs where Examples
UI extensions Local VS Code (your laptop) Themes, icon packs, keybinding managers
Workspace extensions Inside the container Language servers, linters, debuggers, formatters

Extensions listed in customizations.vscode.extensions install inside the container by default. If you need a UI extension (like a theme), the user installs it locally—it is not part of the devcontainer config. The exception: VS Code will detect some extensions that need both sides and install accordingly.

Foot-gun

Extensions install the latest version unless you pin. If eslint ships a breaking major version, every new container build picks it up. Pin versions with "dbaeumer.vscode-eslint@2.4.4" when stability matters.

Settings Inside the Container

The settings object applies VS Code settings at the machine scope inside the container. These override user defaults but can still be overridden by workspace .vscode/settings.json.

{
  "customizations": {
    "vscode": {
      "settings": {
        "editor.formatOnSave": true,
        "editor.defaultFormatter": "esbenp.prettier-vscode",
        "editor.tabSize": 2,
        "eslint.validate": ["javascript", "typescript"],
        "python.linting.enabled": true,
        "terminal.integrated.defaultProfile.linux": "zsh"
      }
    }
  }
}

Settings Precedence

Settings resolve in this order (last wins):

  1. VS Code defaults — built-in baseline
  2. User settings — your personal settings.json (synced via Settings Sync)
  3. Dev Container settings — the customizations.vscode.settings object
  4. Workspace settings — .vscode/settings.json in the repo

The Dev Container settings sit between your personal preferences and the repo's workspace settings. This means the team can enforce formatter-on-save without overriding individual keybinding or theme preferences. A good split: put formatter/linter config in devcontainer, put project-specific paths in .vscode/settings.json.

Environment Variables and Secrets

Dev Containers give you three knobs for environment variables, each with different scope and timing:

Property Scope Available to Set when
containerEnv Entire container All processes (build scripts, daemons, terminals, VS Code server) Container start
remoteEnv VS Code server only Integrated terminals, tasks, debug sessions, extensions VS Code connects
.env file Depends on config Docker Compose services or runArgs passthrough Container start 
{
  // Available to everything inside the container
  "containerEnv": {
    "NODE_ENV": "development",
    "TZ": "UTC"
  },
  // Available only to VS Code's remote server
  "remoteEnv": {
    "EDITOR": "code --wait",
    "GIT_EDITOR": "code --wait",
    // Pass through from host (note the ${localEnv:...} syntax)
    "AWS_PROFILE": "${localEnv:AWS_PROFILE}"
  }
}
Key distinction

Use containerEnv when background services need the variable (database URLs, timezones). Use remoteEnv when only your interactive sessions need it (editor prefs, host credentials passthrough). The ${localEnv:VAR} syntax in remoteEnv pulls values from your host machine—perfect for credentials you never want in version control.

Secrets Management

The number-one rule: never bake secrets into images. A committed Dockerfile with ENV API_KEY=sk-... is a secret sitting in every layer cache and every registry pull.

Safer patterns, in order of preference:

  1. ${localEnv:VAR} passthrough — secrets stay on the host, never in the image or config files
  2. .env file + .gitignore — local-only file, never committed. Reference via runArgs: ["--env-file", ".devcontainer/.env"]
  3. Docker secrets / vault integration — for production-grade setups, mount secrets at runtime from a vault
Gotcha

containerEnv values end up in the container's environment metadata and can be inspected with docker inspect. For sensitive values, prefer remoteEnv with ${localEnv:...} or mount a .env file.

Lifecycle Commands and Practical Patterns

initializeCommand: Host-Side Setup

initializeCommand runs on your host machine before the container builds. Use it for host-side prerequisites:

{
  "initializeCommand": "echo 'Checking Docker...' && docker info > /dev/null 2>&1"
}

Common uses: checking Docker is running, pulling base images, generating .env files from templates, running host-side auth (like aws sso login).

Git Config Inside Containers

VS Code automatically forwards your local Git credentials and GPG setup. But if you need custom Git config inside the container:

{
  "postCreateCommand": "git config --global core.editor 'code --wait' && git config --global pull.rebase true",
  "remoteEnv": {
    "GIT_EDITOR": "code --wait"
  }
}

SSH Agent Forwarding

VS Code forwards your SSH agent automatically when the container is started from VS Code. For private Git repos accessed via SSH inside the container, just ensure your host has an SSH agent running with keys loaded:

# On your host machine
eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519

The Dev Container spec forwards the SSH_AUTH_SOCK into the container. If it is not working, check that your devcontainer feature includes the sshd support or add the mount manually:

{
  "mounts": [
    "source=${localEnv:SSH_AUTH_SOCK},target=/ssh-agent,type=bind"
  ],
  "containerEnv": {
    "SSH_AUTH_SOCK": "/ssh-agent"
  }
}

Complete Linter/Formatter Setup

Here is a full-stack example putting everything together for a TypeScript project:

{
  "name": "ts-fullstack",
  "image": "mcr.microsoft.com/devcontainers/typescript-node:20",

  "customizations": {
    "vscode": {
      "extensions": [
        "dbaeumer.vscode-eslint",
        "esbenp.prettier-vscode",
        "ms-azuretools.vscode-docker",
        "eamodio.gitlens"
      ],
      "settings": {
        "editor.formatOnSave": true,
        "editor.defaultFormatter": "esbenp.prettier-vscode",
        "editor.codeActionsOnSave": {
          "source.fixAll.eslint": "explicit"
        },
        "eslint.validate": ["javascript", "typescript"]
      }
    }
  },

  "containerEnv": {
    "NODE_ENV": "development"
  },
  "remoteEnv": {
    "NPM_TOKEN": "${localEnv:NPM_TOKEN}"
  },

  "postCreateCommand": "npm ci",
  "forwardPorts": [3000, 5432]
}

Why this matters

When a new teammate opens the project, they should see ESLint errors inline, Prettier formatting on save, and the right terminal shell—with zero setup steps. The customizations.vscode object plus environment variable passthrough is the difference between "clone and code" and a two-page onboarding doc nobody reads.
