Custom Features and Reusable Setup

Dev Container Features are self-contained units of installation logic — a shell script, a metadata file, and nothing else. They let you compose environment capabilities like LEGO bricks: add Node, add the AWS CLI, add your team’s internal tooling, all without touching the Dockerfile.

Using Published Features

The simplest way to use Features is to reference them from the public registry. Each Feature is an OCI artifact hosted on ghcr.io, versioned with semver.

// .devcontainer/devcontainer.json
{
  "image": "mcr.microsoft.com/devcontainers/base:ubuntu",
  "features": {
    "ghcr.io/devcontainers/features/node:1": {
      "version": "20"
    },
    "ghcr.io/devcontainers/features/aws-cli:1": {},
    "ghcr.io/devcontainers/features/docker-in-docker:2": {
      "dockerDashComposeVersion": "v2"
    }
  }
}

Each key in features is an OCI reference. The value is an options object — every Feature defines its own set of configurable options in its devcontainer-feature.json. Pass an empty object {} for defaults.

Foot-gun Omitting the version tag (e.g., writing ghcr.io/devcontainers/features/node with no :1) pulls latest. That means your environment can silently change when a Feature publishes a breaking major version. Always pin to at least a major version.

Features are installed after the base image is built but before lifecycle commands like postCreateCommand run. This means you can rely on Feature-installed tools in your lifecycle scripts.

Option patternExampleEffect
"version""20"Installs a specific version of the tool
"installTools"trueOpt into companion utilities
"moby"falseUse Docker CE instead of Moby (docker-in-docker)

Think of Features as apt install with guardrails. Each one runs in isolation, handles its own dependencies, and exposes a typed options contract.

How Features Layer onto the Base Image

When the Dev Container CLI builds your environment, it processes Features in a specific order:

  1. The base image (from image, Dockerfile, or Compose) is built or pulled.
  2. Each Feature is applied as a separate build stage. The CLI generates a temporary Dockerfile that COPYs each Feature’s files and runs its install.sh.
  3. Features run in dependency order — if Feature A declares installsAfter referencing Feature B, B runs first.
  4. Without explicit ordering, Features run in the order they appear in your JSON (but don’t rely on this — use installsAfter).
Key detail Feature install.sh scripts run as root, even if your container’s remoteUser is non-root. This is by design — Features install system-level tooling. If your Feature needs to set up user-level config, use the $_REMOTE_USER and $_REMOTE_USER_HOME variables the runtime provides.

Writing Your Own Feature

A Feature is a directory with two required files: devcontainer-feature.json (metadata + options) and install.sh (the script that runs during build).

Directory structure

src/
  internal-cli/
    devcontainer-feature.json
    install.sh
  another-feature/
    devcontainer-feature.json
    install.sh

The metadata file

// src/internal-cli/devcontainer-feature.json
{
  "id": "internal-cli",
  "version": "1.2.0",
  "name": "Acme Internal CLI",
  "description": "Installs acme-cli and authenticates to internal registries",
  "options": {
    "version": {
      "type": "string",
      "default": "latest",
      "description": "Version of acme-cli to install"
    },
    "includePlugins": {
      "type": "boolean",
      "default": true,
      "description": "Install recommended acme-cli plugins"
    }
  },
  "installsAfter": [
    "ghcr.io/devcontainers/features/node"
  ],
  "containerEnv": {
    "ACME_HOME": "/usr/local/acme"
  }
}

The install script

#!/bin/bash
# src/internal-cli/install.sh
set -e

# Options arrive as environment variables: VERSION, INCLUDEPLUGINS
# (uppercased, no underscores)
VERSION="${VERSION:-latest}"
INCLUDE_PLUGINS="${INCLUDEPLUGINS:-true}"

echo "Installing acme-cli ${VERSION}..."

# Download from internal registry
curl -fsSL "https://releases.acme.internal/cli/${VERSION}/install.sh" | bash

# Set up for the container user, not root
if [ "${INCLUDE_PLUGINS}" = "true" ]; then
    su - "${_REMOTE_USER}" -c "acme plugin install lint deploy test"
fi

# Add to PATH for all users
echo 'export PATH="/usr/local/acme/bin:$PATH"' >> /etc/profile.d/acme.sh
Conventions
  • Options become uppercased env vars: includePlugins becomes $INCLUDEPLUGINS.
  • $_REMOTE_USER, $_REMOTE_USER_HOME, and $_CONTAINER_ID are injected at install time.
  • Scripts must be POSIX-compatible or explicitly set their shell with a shebang.
  • Do not assume apt — check the distro or use Feature dependencies to guarantee a base.

Publishing and Distribution

Features are published as OCI artifacts, typically to ghcr.io. The Dev Containers CLI provides a publish command, or you can use the devcontainers/action GitHub Action.

# Publish all features in src/ to ghcr.io/your-org/devcontainer-features
devcontainer features publish ./src \
  --registry ghcr.io \
  --namespace your-org/devcontainer-features

# .github/workflows/publish-features.yml
name: Publish Dev Container Features
on:
  push:
    branches: [main]
    paths: ['src/**']

jobs:
  publish:
    runs-on: ubuntu-latest
    permissions:
      packages: write
      contents: read
    steps:
      - uses: actions/checkout@v4
      - uses: devcontainers/action@v1
        with:
          publish-features: "true"
          base-path-to-features: "src"

Dependencies and conflicts

Use installsAfter to declare ordering dependencies. If your Feature requires Node to be present, list the Node Feature. The CLI will reorder installation so Node goes first.

Watch out There is no formal “conflicts” declaration in the spec. Two Features that both install Python (say, different versions) will silently overwrite each other. The last one wins. Document conflicts in your Feature’s README and use options to let users choose versions explicitly.

Features vs. Templates

AspectFeatureTemplate
PurposeAdd a capability to an existing containerScaffold an entire .devcontainer/ directory
When it runsEvery build (baked into the image)Once, when creating a new project
OutputInstalled tools, env vars, configFiles: devcontainer.json, Dockerfile, scripts
ComposableYes — stack as many as you wantYou pick one template, then customize
VersionedOCI artifact with semverOCI artifact, but applied once

Templates are starters. Features are ingredients. Use a Template to create the project skeleton, then add Features for the tools. Your team’s internal CLI? That’s a Feature. Your project’s boilerplate .devcontainer folder? That’s a Template.

Why this matters

Without Features, teams end up with enormous Dockerfiles that copy-paste the same installation blocks across repositories. When the internal CLI gets a security update, someone has to find and patch every repo. Features flip this: publish once, reference everywhere, update by bumping a version number. One CI pipeline, one source of truth, every repo gets the fix on next rebuild.
