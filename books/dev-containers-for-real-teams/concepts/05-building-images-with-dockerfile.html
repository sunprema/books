Building Images with Dockerfile

An image reference in devcontainer.json gets you started fast. A Dockerfile gets you exactly what your team needs. Here is when to cross that line and how to do it without turning every container rebuild into a coffee break.

Image Reference vs. Dockerfile

The simplest devcontainer.json points at a pre-built image:

// .devcontainer/devcontainer.json — image-only approach
{
  "image": "mcr.microsoft.com/devcontainers/typescript-node:1-20-bookworm"
}

That works until you need something the image does not ship: a specific CLI version, a system library for native modules, or a second runtime. At that point, switch to a Dockerfile.

ApproachRebuild speedCustomizationBest for
imageInstant (pull only)NoneStandard stacks, quick starts
build.dockerfileCached layersFullCustom tooling, polyglot, pinned versions
Rule of thumb If you are adding more than two postCreateCommand install steps, move them into a Dockerfile. Those commands run on every rebuild; Dockerfile layers are cached.

Point devcontainer.json at the Dockerfile with the build object:

{
  "build": {
    "dockerfile": "Dockerfile",
    "context": "..",
    "args": { "NODE_VERSION": "20" }
  }
}

The context is relative to the .devcontainer/ folder. Setting it to ".." means the project root, which lets you COPY files like package.json from the repo.

Base Image Selection

Microsoft publishes Dev Container base images at mcr.microsoft.com/devcontainers/. They come pre-wired with a non-root vscode user, common utilities, and the plumbing VS Code needs. Starting from one saves work.

BaseWhen to use
devcontainers/base:bookwormDebian-only, no language runtime pre-installed
devcontainers/python:3.12Python projects needing a specific minor version
devcontainers/typescript-node:20Node/TS projects, includes nvm
ubuntu:24.04Full control, no dev container scaffolding (you wire it yourself)
Foot-gun Alpine-based images save ~200 MB but break many native Node modules and Python wheels that expect glibc. Unless your production image is also Alpine, start with Debian bookworm.

Layer Ordering and Cache Efficiency

Docker caches each layer. When a layer changes, every layer after it is rebuilt. The golden rule: put things that change rarely at the top, things that change often at the bottom.

Copy dependency manifests (package.json, requirements.txt, go.sum) before copying the full source tree. This lets Docker cache the expensive install step as long as your dependencies have not changed.

Practical Example: Python + Node Polyglot

Here is a production-shaped Dockerfile for a project that needs Python 3.12 and Node 20. It demonstrates every pattern discussed above.

# .devcontainer/Dockerfile
ARG PYTHON_VERSION=3.12
ARG NODE_VERSION=20

FROM mcr.microsoft.com/devcontainers/python:${PYTHON_VERSION}-bookworm

# ---- System packages (changes rarely) ----
RUN apt-get update \
 && apt-get install -y --no-install-recommends \
      build-essential \
      libpq-dev \
      curl \
 && rm -rf /var/lib/apt/lists/*

# ---- Node via nvm (pinned version) ----
ARG NODE_VERSION
RUN su vscode -c \
  "source /usr/local/share/nvm/nvm.sh && nvm install ${NODE_VERSION}"

# ---- Global CLI tools ----
RUN pip install --no-cache-dir poetry==1.8.3 \
 && su vscode -c \
  "source /usr/local/share/nvm/nvm.sh && npm install -g pnpm@9"

# ---- Dependency manifests (copy before source) ----
COPY pyproject.toml poetry.lock ./
RUN poetry config virtualenvs.in-project true \
 && poetry install --no-root --no-interaction

COPY package.json pnpm-lock.yaml ./
RUN su vscode -c \
  "source /usr/local/share/nvm/nvm.sh && pnpm install --frozen-lockfile"

# Source code is mounted by VS Code — no COPY . needed
apt-get best practices Always combine apt-get update and apt-get install in one RUN layer. Use --no-install-recommends to skip optional packages. End with rm -rf /var/lib/apt/lists/* to discard the package index and save ~30 MB per layer.

Using Build Args for Team Flexibility

The ARG instructions above let teammates override versions without forking the Dockerfile:

// devcontainer.json
{
  "build": {
    "dockerfile": "Dockerfile",
    "context": "..",
    "args": {
      "PYTHON_VERSION": "3.12",
      "NODE_VERSION": "20"
    }
  }
}

The Non-Root User Pattern

Microsoft base images ship a vscode user (UID 1000). If you use a plain ubuntu base, create one yourself:

RUN groupadd --gid 1000 dev \
 && useradd --uid 1000 --gid dev --shell /bin/bash \
      --create-home dev

USER dev

Then set "remoteUser": "dev" in devcontainer.json. VS Code will run as this user inside the container while Docker operations stay root-capable.

Avoiding bloat Dev container images do not ship to production, so trimming them to the last megabyte is not worth the effort. But avoid obvious waste: do not install full desktop environments, do not leave build artifacts in intermediate layers, and do not install packages you will never use. A 2 GB image is fine. A 6 GB image means something went wrong.

Why this matters

A well-layered Dockerfile means rebuilds take seconds instead of minutes. That is the difference between a team that iterates on their dev environment and a team that treats it as frozen lava. Get the layer order right once and your container becomes a living tool, not a build-time tax.
