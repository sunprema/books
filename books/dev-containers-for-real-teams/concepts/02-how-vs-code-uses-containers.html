Chapter 02 · Architecture

How VS Code Uses Containers

VS Code doesn’t just “attach to a container.” It installs a full server process inside the container, splits extensions between client and server, and proxies every file operation, terminal session, and debug connection through a structured protocol. Understanding this architecture explains most of the behavior you’ll observe—and most of the bugs you’ll hit.

The Client-Server Split

VS Code Remote (the umbrella that includes Dev Containers, Remote-SSH, and WSL) uses a client-server architecture. The desktop application you interact with is the client. It handles UI rendering, theming, and keybindings—everything that touches pixels. The heavy lifting—language servers, file I/O, terminal processes, debugging—runs on the VS Code Server, which lives inside the container.

# Where VS Code installs its server inside the container
~/.vscode-server/
  bin/
    <commit-hash>/
      node              # bundled Node.js runtime
      bin/
        code-server     # the server binary
      extensions/       # workspace extensions installed here

The client and server communicate over a JSON-RPC protocol, multiplexed over a single connection (a Unix socket when local, or a WebSocket/SSH tunnel when remote). This is not HTTP—it’s a persistent, bidirectional channel.

How this differs from Remote-SSH Remote-SSH installs the same VS Code Server on a remote host via SSH. Dev Containers does the same thing but targets a Docker container instead. The protocol, extension split, and server binary are identical. The only difference is the transport layer and how the environment is provisioned.

The Extension Host Split

This is where most confusion lives. VS Code classifies every extension into one of two categories:

Category Runs where Examples
UI Extension Client (your desktop) Themes, icon packs, keymaps, some status-bar widgets
Workspace Extension Server (inside container) Language servers, linters, formatters, debuggers, Git

When you specify extensions in devcontainer.json, those extensions are installed inside the container’s VS Code Server. They run against the container’s file system and runtimes, which is exactly what you want—ESLint uses the container’s Node, the Python extension uses the container’s Python, the Go extension uses the container’s Go toolchain.

{
  "customizations": {
    "vscode": {
      "extensions": [
        "dbaeumer.vscode-eslint",      // workspace ext → runs in container
        "esbenp.prettier-vscode",      // workspace ext → runs in container
        "ms-python.python"             // workspace ext → runs in container
      ]
    }
  }
}

A common gotcha: you install a theme extension via devcontainer.json, rebuild the container, and it doesn’t show up. That’s because themes are UI extensions—they run on the client side. They should be installed locally, not in the container. The extension marketplace page tells you which kind it is, but the labeling is easy to miss.

Foot-gun Extensions that need native binaries (e.g., C++ compiler tools) must match the container’s architecture. If your container runs linux/arm64 but an extension only ships linux/x64 binaries, it will silently fail to activate. Check the Output panel → “Extension Host” for clues.

The Dev Container Lifecycle

When you open a folder with a .devcontainer/ config, the Dev Containers extension executes a specific sequence. Knowing these steps helps you debug “why is my container stuck?” issues.

The Six-Step Startup

  1. Detect — the extension finds .devcontainer/devcontainer.json (or offers to create one)
  2. Build — runs docker build (or docker compose build) using your Dockerfile. Caches layers when possible
  3. Create — runs docker run (or docker compose up) with the arguments derived from devcontainer.json: mounts, ports, env vars, runArgs
  4. Install Server — copies the VS Code Server binary + bundled Node into the running container
  5. Install Extensions — installs workspace extensions inside the container’s server
  6. Connect — establishes the JSON-RPC channel and opens the remote window

Lifecycle scripts fire at specific points within this sequence:

// Execution order of lifecycle hooks

initializeCommand     // runs on HOST before anything else
  ↓
onCreateCommand       // runs IN CONTAINER on first create only
  ↓
updateContentCommand  // runs when content changes (e.g., git pull in Codespaces)
  ↓
postCreateCommand    // runs after create + content update (good for npm install)
  ↓
postStartCommand     // runs every time container starts (including restart)
  ↓
postAttachCommand    // runs every time VS Code attaches (including window reload)
Production trade-off Put slow, one-time setup in onCreateCommand (dependency installs, database migrations). Put fast recurring setup in postStartCommand (starting background services). Put user-specific setup in postAttachCommand (shell customizations). Getting this wrong means either rebuilds take forever or you lose state on restart.

Port Forwarding and Container Rebuilds

How Port Forwarding Works

When a process inside the container listens on a port, VS Code can automatically detect it and forward it to localhost on the host. This happens through the VS Code Server, not through Docker’s -p flag.

{
  // Explicitly forward ports
  "forwardPorts": [3000, 5432],

  // Control auto-detection
  "otherPortsAttributes": {
    "onAutoForward": "silent"    // "notify" | "openBrowser" | "silent" | "ignore"
  },

  // Per-port labels and visibility
  "portsAttributes": {
    "3000": { "label": "Frontend", "onAutoForward": "openBrowser" },
    "5432": { "label": "Postgres", "onAutoForward": "silent" }
  }
}

The port forwarding is more intelligent than Docker’s -p. It detects new listeners dynamically—you start a dev server, and VS Code offers to open the browser. This works because the server inside the container monitors /proc/net/tcp and reports new listeners back to the client.

How Rebuilds Work

When you change devcontainer.json or the Dockerfile, you need to rebuild. VS Code offers two commands:

Command What it does When to use
Rebuild Container Rebuilds the image and recreates the container. Runs onCreateCommand + postCreateCommand again Dockerfile or Feature changes
Rebuild Without Cache Same as above but passes --no-cache to Docker. Full clean build Debugging broken layers or stale caches
Gotcha A rebuild destroys the container. Anything stored outside of mounted volumes is lost—including apt install packages you ran manually in the terminal, pip packages installed outside a requirements file, and any files written outside the workspace mount. If you need something to survive a rebuild, it must be in the Dockerfile, a Feature, or a lifecycle script.
Container persistence Between rebuilds, the container persists. Closing VS Code does not destroy it—it stops the container, and reopening reconnects to the same instance. Only an explicit rebuild, or docker rm, destroys it. Named volumes and bind mounts survive rebuilds because they live outside the container filesystem.

Why this matters

Every debugging session with Dev Containers eventually comes down to understanding this architecture. “Why can’t my extension find Python?”—because it’s a workspace extension running inside the container, and Python isn’t installed there. “Why did my terminal history vanish?”—because you rebuilt the container without a persistent volume for shell history. “Why is port 3000 not reachable?”—because the process is binding to 127.0.0.1 inside the container, and you need 0.0.0.0. The architecture is not complex, but it is unfamiliar, and most issues trace back to forgetting which side of the client-server boundary you’re on.
