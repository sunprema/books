Chapter 12 · Troubleshooting

Debugging, Performance, and Common Problems

Something will break. The container won’t start, the build takes forever, file watching stops working, or permissions go sideways. This chapter is a field reference for diagnosing and fixing the problems you will actually hit—organized so you can find the answer fast when your team is blocked.

“Container Failed to Start”

This is the single most common support request. VS Code shows a vague error, and most developers don’t know where to look. Here is the systematic debugging workflow:

# Step 1: Get the build log (VS Code: "Dev Containers: Show Log")
# Or from CLI:
devcontainer up --workspace-folder . 2>&1 | tee /tmp/devcontainer.log

# Step 2: If build succeeded but container exits, check the container
docker ps -a --filter "label=devcontainer.local_folder" \
  --format "{{.ID}} {{.Status}} {{.Names}}"

# Step 3: Inspect the dead container
docker inspect <container-id> | jq '.[0].State'

# Step 4: Read its logs
docker logs <container-id>

# Step 5: If it's a lifecycle script failure, run the image manually
docker run --rm -it <image-id> /bin/bash
# Then manually run your postCreateCommand to see where it fails

Follow the arrows. Most problems resolve at the first or second decision point.

Slow Builds and I/O Performance

A container that takes 8 minutes to build is a container nobody uses. Here are the fixes, in order of impact:

Layer Caching

Docker caches layers, but one wrong line order invalidates everything downstream. Put things that change rarely at the top:

# BAD: copying everything first busts cache on any file change
FROM node:20
COPY . /app          # any code change = reinstall everything
RUN npm ci

# GOOD: copy only dependency files first
FROM node:20
COPY package.json package-lock.json /app/
WORKDIR /app
RUN npm ci           # cached unless lock file changes
COPY . /app          # only this layer rebuilds on code change

Pre-built Images

For teams, build the Dev Container image in CI and push it to a registry. Developers pull instead of build:

# CI builds and pushes the image nightly
devcontainer build --workspace-folder . \
  --image-name ghcr.io/myorg/devcontainer-myproject:latest \
  --cache-from ghcr.io/myorg/devcontainer-myproject:latest
docker push ghcr.io/myorg/devcontainer-myproject:latest

# devcontainer.json uses the pre-built image
{
  "image": "ghcr.io/myorg/devcontainer-myproject:latest"
}

File I/O on macOS

Bind mounts on macOS are slow—often 10x slower than native I/O. This kills webpack, Jest, and any tool that scans the file tree.

# Option 1: Use a named volume for node_modules (biggest win)
{
  "mounts": [
    "source=myproject-node-modules,target=/workspace/node_modules,type=volume"
  ]
}

# Option 2: Use Docker's VirtioFS backend (Docker Desktop 4.15+)
# Settings > General > "Use VirtioFS" (replaces gRPC FUSE)

# Measure the difference:
time find /workspace -name "*.ts" | wc -l    # bind mount
time find /workspace -name "*.ts" | wc -l    # named volume
File watching (inotify) Bind mounts on Docker Desktop do not propagate native inotify events. Tools like webpack-dev-server, nodemon, or tsc --watch fall back to polling, eating CPU. Fixes: (1) Use CHOKIDAR_USEPOLLING=true with a high interval to reduce CPU; (2) Use legacyWatch in VS Code settings; (3) Switch to a named volume for the watched directory.

Common Problems Quick Reference

ProblemCauseFix
Permission denied writing files Container runs as root; workspace owned by vscode user (UID 1000) Set "remoteUser": "vscode" or match UID in Dockerfile: ARG UID=1000
Out of disk space Docker Desktop default is 64GB virtual disk; old images fill it docker system prune -a or increase disk in Docker Desktop settings
Extensions won’t install Extension needs a specific platform binary not available in container arch Check extension supports linux-x64. Use "extensions" array only for verified ones.
Git credentials not forwarded SSH agent or credential manager not mounted VS Code auto-forwards SSH agent. For HTTPS, ensure credential.helper is set on host.
DNS failures inside container Corporate VPN overrides DNS; container inherits host’s broken resolv.conf Set "runArgs": ["--dns=8.8.8.8"] or configure Docker DNS in daemon.json
Proxy / VPN blocks image pulls Docker daemon not configured for corporate proxy Set HTTP_PROXY in ~/.docker/config.json or daemon proxy config
postCreateCommand hangs Interactive prompt (e.g., npm init) waiting for stdin Use --yes flags. All lifecycle commands must be non-interactive.

Performance Profiling and Scripted Debugging

When you need numbers, not guesses:

# Measure build time with cache vs. no cache
time devcontainer build --workspace-folder .
time devcontainer build --workspace-folder . --no-cache

# Profile I/O throughput inside the container
devcontainer exec --workspace-folder . \
  dd if=/dev/zero of=/workspace/testfile bs=1M count=256 oflag=direct
# Compare: >200 MB/s (volume) vs ~50 MB/s (bind mount on macOS)

# Check Docker Desktop resource limits
devcontainer exec --workspace-folder . \
  sh -c "echo CPUs: $(nproc) && free -h | head -2"

# List what's eating disk
docker system df -v
The devcontainer CLI is your best friend for scripted debugging. You can build, start, exec, and tear down containers from a shell script or CI job—no VS Code needed. When someone reports a container problem, your first move is to reproduce it with the CLI, not the GUI. CLI output is greppable; GUI error dialogs are not.

Docker Desktop ships with conservative defaults. If builds are slow or tests OOM, check these:

# Inside the container, check what you've actually got:
$ nproc
2                    # default: 2 CPUs. Bump to 4+ for real work.
$ free -h
              total   used   free
Mem:          3.8G    1.2G   2.6G  # default: 4GB. Bump to 8+ for Node/Java.

# Fix: Docker Desktop > Settings > Resources
# CPUs: 4+, Memory: 8GB+, Disk: 128GB+
Production trade-off Giving Docker more resources means less for the rest of your laptop. On a 16GB machine, giving Docker 8GB leaves your host with 8GB—which may not be enough if you run Slack, Chrome, and VS Code outside the container. Find the balance; document it in your CONTRIBUTING.md so the team uses the same settings.
Why this matters: Container debugging is a skill, not a talent. Every problem in this chapter has a systematic fix. When your team knows the flowchart—check the build, check the logs, check the resources—they stop treating container issues as mysterious black-box failures and start fixing them in minutes instead of hours.
