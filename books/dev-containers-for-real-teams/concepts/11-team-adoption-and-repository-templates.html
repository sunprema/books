Chapter 11 · Teams

Team Adoption and Repository Templates

A Dev Container config sitting in a repo is inert until people actually use it. This chapter is the playbook for getting your team from “interesting demo” to “this is how we work”—without mandates, without breaking anyone’s flow, and without a six-month migration plan that never finishes.

Start Opt-In: The .devcontainer Folder Is Harmless

The single best property of Dev Containers for adoption: the .devcontainer/ folder does nothing to developers who don’t use it. It sits in the repo like any other config directory. No build step changes, no CI impact, no workflow disruption. This is your wedge.

Ship the config in a PR with a one-paragraph README change. Don’t announce it at standup. Don’t send a Slack message. Just merge it. The people who are curious will find it. The people who aren’t won’t even notice. That’s the point.

Your first PR should be minimal and battle-tested. Here is exactly what to land:

# Directory structure for the initial PR
.devcontainer/
  devcontainer.json
  Dockerfile          # only if base image needs customization
CONTRIBUTING.md       # add a "Dev Container" section

A minimal but complete devcontainer.json that covers the 80% case:

{
  "name": "myproject-dev",
  "image": "mcr.microsoft.com/devcontainers/typescript-node:20",
  "features": {
    "ghcr.io/devcontainers/features/docker-in-docker:2": {}
  },
  "postCreateCommand": "npm ci",
  "customizations": {
    "vscode": {
      "extensions": [
        "dbaeumer.vscode-eslint",
        "esbenp.prettier-vscode"
      ]
    }
  },
  "forwardPorts": [3000]
}
CONTRIBUTING.md pattern Add a section like this—short, no hard sell: 
## Quick Start (Dev Container)
If you have VS Code + Docker, open this repo and accept
the "Reopen in Container" prompt. Everything installs
automatically. See .devcontainer/ for details.

## Quick Start (Local)
Install Node 20, run `npm ci`, then `npm run dev`.
Keep both paths documented. Removing the local instructions signals a mandate you haven’t earned yet.

Making It the Default Onboarding Path

The inflection point comes when new hires and new contributors use the Dev Container first—not because you told them to, but because it’s the path of least resistance. Here is a phased rollout plan that has worked on teams from 5 to 200 engineers:

PhaseDurationActionsSuccess Signal
1. Seed Week 1–2 Land .devcontainer/ in 1–3 key repos. Update CONTRIBUTING.md. No announcements. PR merges without pushback
2. Adopt Week 3–6 Onboard one new hire exclusively via Dev Container. Fix every friction point they hit. Add postCreateCommand for all setup steps. New hire productive in <1 hour
3. Expand Week 7–12 Add configs to remaining active repos. Publish a GitHub template repo. Set up pre-built images for slow builds. Share a short demo at team meeting. >50% of team has used it at least once
4. Default Week 13+ Make Dev Container the first option in onboarding docs. Use same image in CI. Establish review process for config changes. New contributors default to container path
Foot-gun Never frame Dev Containers as mandatory before Phase 4. The moment someone feels forced, you have created an adversary who will catalog every rough edge. Let adoption be pulled by convenience, not pushed by policy.

Repository Templates and Org-Wide Config

Once your Dev Container config is stable, bake it into repository templates so every new project starts correctly. GitHub template repositories are the simplest mechanism.

# Create a template repo with your standard Dev Container
gh repo create myorg/template-node-service \
  --template --public \
  --description "Node.js service template with Dev Container"

# New repos inherit the full .devcontainer/ directory
gh repo create myorg/billing-service \
  --template myorg/template-node-service \
  --private

A good template repo includes:

# Template repository structure
.devcontainer/
  devcontainer.json       # points to org base image
  docker-compose.yml      # if services needed (Postgres, Redis)
.github/
  CODEOWNERS              # platform team owns .devcontainer/
  pull_request_template.md
src/
  index.ts                # minimal hello-world
package.json
tsconfig.json
CONTRIBUTING.md           # Dev Container is first option

For org-wide consistency, publish a base Dev Container image to your container registry. Individual repos extend it rather than duplicating setup:

{
  "name": "billing-service",
  "image": "ghcr.io/myorg/devcontainer-base:node20-2024.1",
  "features": {
    "ghcr.io/myorg/devcontainer-features/billing-tools:1": {}
  },
  "postCreateCommand": "npm ci && npm run db:migrate"
}
Versioning tip Tag your base images with a date-based version (node20-2024.1) rather than latest. This lets teams upgrade on their own schedule while still getting a reproducible environment. Pin in devcontainer.json; update via Dependabot or Renovate.

One base image feeds templates, project repos, and CI—same tools everywhere.

CI Parity and Governance

The most powerful argument for Dev Containers on a team is CI parity: developers build and test in the exact same image that CI uses. No more “it passes on my machine.” The devcontainer CLI makes this straightforward.

# .github/workflows/ci.yml - Use the Dev Container for CI
name: CI
on: [push, pull_request]
jobs:
  build-and-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: devcontainers/ci@v0.3
        with:
          runCmd: |
            npm ci
            npm run lint
            npm test
          push: never    # don't push image from PR builds

For other CI systems, the devcontainer CLI works anywhere Docker runs:

# Install the CLI
npm install -g @devcontainers/cli

# Build the Dev Container image (uses cache)
devcontainer build --workspace-folder .

# Run tests inside the container
devcontainer up --workspace-folder .
devcontainer exec --workspace-folder . npm test
Governance model Assign ownership of .devcontainer/ via CODEOWNERS. A platform or DevEx team reviews changes—this prevents well-meaning tweaks from breaking everyone else. Keep the review bar low: approve fast, iterate in follow-ups. 
# .github/CODEOWNERS
/.devcontainer/    @myorg/platform-team

Some team members will always prefer local development. That is fine. The contract is simple: the Dev Container is the reference environment. If something works in the container but not locally, the container is right. Document this clearly and enforce it in CI. Never break the local path on purpose—just stop maintaining two sets of setup instructions.

Why this matters: The goal is not 100% Dev Container adoption. The goal is zero time spent debugging environment differences. Even if half your team runs local, the Dev Container as reference environment plus CI parity eliminates the entire class of “works on my machine” problems. That is the win.
