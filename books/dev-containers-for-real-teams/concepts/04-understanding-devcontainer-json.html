Understanding devcontainer.json

The devcontainer.json file is the single source of truth for your development environment. This chapter breaks down every major section: how the container gets built, what runs and when, how environment variables flow, and where your editor customizations live.

Three Ways to Define the Container

Every devcontainer.json must tell the tooling how to get a container. There are exactly three modes, and they are mutually exclusive:

Mode 1: image

Pull a pre-built image directly. Fastest to start, least flexible. Good for simple projects or when your team publishes a shared base image.

{
  "image": "mcr.microsoft.com/devcontainers/python:3.12"
}

Mode 2: build.dockerfile

Build from a Dockerfile in your repo. Use this when you need to install system packages, add tools, or customize the base image.

{
  "build": {
    "dockerfile": "Dockerfile",
    "context": "..",
    "args": {
      "NODE_VERSION": "22"
    }
  }
}
Note The context path is relative to the .devcontainer/ directory, not the project root. Setting it to ".." gives your Dockerfile access to the full project tree during build. This is the most common value.

Mode 3: dockerComposeFile

Use an existing Docker Compose file. Required for multi-container setups (app + database + cache). You tell it which service is the dev container.

{
  "dockerComposeFile": "docker-compose.yml",
  "service": "app",
  "workspaceFolder": "/workspaces/my-project"
}
Foot-gun You cannot mix modes. Setting both "image" and "build" is an error. Setting "dockerComposeFile" alongside either is also an error. Pick one. If you start with image and later need customization, switch to build.dockerfile—it is a one-line change.

Lifecycle Hooks

Dev Containers define five lifecycle hooks. Getting them wrong means your dependencies install at the wrong time, or run on every reconnect when they should run once. Here is the full picture:

Hook When it fires Runs again on... Use for
onCreateCommand After container is created (first time only) Rebuild only One-time OS-level setup, global tool installs
updateContentCommand After onCreateCommand, and when source changes (Codespaces prebuild) Rebuild, content update npm install, pip install, dependency restoration
postCreateCommand After updateContentCommand, first time container runs Rebuild only Project bootstrap, seed data, generate configs
postStartCommand Every time the container starts Every start (including restart) Start background services, refresh tokens
postAttachCommand Every time a VS Code window attaches Every attach Terminal greeting, status check, open a browser

The most common mistake: putting npm install in postStartCommand. That re-runs every time you restart the container, wasting 30 seconds each time. Use updateContentCommand or postCreateCommand instead.

// A practical lifecycle setup
{
  "image": "mcr.microsoft.com/devcontainers/javascript-node:22",

  // Install global tools once
  "onCreateCommand": "npm install -g turbo",

  // Restore deps when source changes
  "updateContentCommand": "npm ci",

  // Seed the local DB on first create
  "postCreateCommand": "npm run db:seed",

  // Start the dev DB on every container start
  "postStartCommand": "npm run db:start &",

  // Print status on every attach
  "postAttachCommand": "node --version && git status"
}

Environment Variables

Dev Containers distinguish between two scopes of environment variables. Getting the scope wrong means your variable is invisible where you need it.

{
  // Available inside the container at the OS level
  // (visible to all processes, including lifecycle hooks)
  "containerEnv": {
    "NODE_ENV": "development",
    "CI": "false"
  },

  // Available only inside the VS Code remote server
  // (visible to terminals and tasks, NOT to lifecycle hooks)
  "remoteEnv": {
    "DATABASE_URL": "postgres://localhost:5432/devdb",
    "EDITOR": "code --wait"
  }
}
Rule of thumb Use containerEnv for things your build and lifecycle hooks need. Use remoteEnv for things only your editor and terminal sessions need. If a variable contains a secret, use neither—use remoteEnv with ${localEnv:VAR_NAME} to forward it from your host without committing it to the repo. 
// Forward a secret from your host machine
"remoteEnv": {
  "AWS_ACCESS_KEY_ID": "${localEnv:AWS_ACCESS_KEY_ID}",
  "AWS_SECRET_ACCESS_KEY": "${localEnv:AWS_SECRET_ACCESS_KEY}"
}

Features, Ports, Mounts, and runArgs

Beyond the container source and lifecycle hooks, four fields come up in nearly every real config:

features

Features are reusable, composable units of dev tooling. Instead of writing Dockerfile lines to install the AWS CLI, Docker-in-Docker, or Terraform, you declare them:

"features": {
  "ghcr.io/devcontainers/features/aws-cli:1": {},
  "ghcr.io/devcontainers/features/docker-in-docker:2": {
    "version": "latest"
  },
  "ghcr.io/devcontainers/features/github-cli:1": {}
}

Each Feature is an OCI artifact with an install.sh that runs during image build. They compose cleanly because they layer on top of your base image. The full catalog is at containers.dev/features.

forwardPorts

// Map container ports to your host
"forwardPorts": [3000, 5432, 6379],

// Control visibility and labels
"portsAttributes": {
  "3000": { "label": "App", "onAutoForward": "notify" },
  "5432": { "label": "Postgres", "onAutoForward": "silent" }
}

mounts

Persist data across container rebuilds or share host directories:

"mounts": [
  // Named volume for node_modules (survives rebuilds)
  "source=node_modules,target=/workspaces/app/node_modules,type=volume",

  // Bind-mount your SSH keys (read-only)
  "source=${localEnv:HOME}/.ssh,target=/home/node/.ssh,type=bind,readonly"
]

runArgs

Pass arbitrary flags to docker run. Use sparingly—these are escape hatches:

"runArgs": [
  "--cpus=2",
  "--memory=4g",
  "--network=host"
]
Foot-gun --network=host skips Docker networking and exposes all container ports directly on the host. It simplifies port forwarding but breaks container isolation. Use it only when you genuinely need it (e.g., mDNS, multicast). On macOS and Windows, --network=host does not work as expected because Docker runs inside a VM.

customizations

Editor-specific settings live under a namespaced key. For VS Code:

"customizations": {
  "vscode": {
    "extensions": [
      "dbaeumer.vscode-eslint",
      "ms-python.python"
    ],
    "settings": {
      "editor.formatOnSave": true,
      "editor.defaultFormatter": "esbenp.prettier-vscode",
      "terminal.integrated.defaultProfile.linux": "zsh"
    }
  }
}

This is how you eliminate the “make sure you have these extensions installed” section from your README. The container is the README.

Why this matters

A well-structured devcontainer.json replaces your onboarding wiki, your “local setup” Slack channel, and your “works on my machine” debugging sessions. Every field you set is one fewer thing a new team member has to figure out. Every lifecycle hook in the right slot is one fewer surprise at 2 AM.
