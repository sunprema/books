Remote and Multi-Container Workflows

Your Dev Container doesn’t have to run on your laptop. GitHub Codespaces, remote Docker hosts, DevPod, and Kubernetes-based environments all speak the same devcontainer.json spec. This chapter covers when to go remote, how to set it up, and how to keep costs from surprising your finance team.

GitHub Codespaces

Codespaces is the most direct path from devcontainer.json to a cloud-hosted environment. GitHub provisions a VM, builds your container, attaches VS Code (browser or desktop), and you’re coding. Your existing Dev Container config works without changes.

Machine types and prebuilds

// .devcontainer/devcontainer.json
{
  "image": "mcr.microsoft.com/devcontainers/typescript-node:20",
  "features": {
    "ghcr.io/devcontainers/features/docker-in-docker:2": {}
  },
  "hostRequirements": {
    "cpus": 4,
    "memory": "8gb",
    "storage": "32gb"
  },
  "postCreateCommand": "npm ci"
}

The hostRequirements field tells Codespaces which machine tier to use. Without it, the org default applies (usually 2-core).

Prebuilds save minutes Configure prebuilds in your repo settings: Settings → Codespaces → Prebuild configuration. Prebuilds run docker build and postCreateCommand ahead of time, so opening a Codespace takes seconds instead of minutes. Trigger them on push to main and on PR branches for best coverage.

Cost foot-gun Codespaces charge for compute while running and storage while stopped. An idle Codespace sitting stopped for a month at 64 GB still costs ~$5. Set org-level policies: auto-stop after 30 minutes of inactivity, retention period of 14 days, and max machine type per repo.

Remote Docker Hosts and Alternatives

You don’t need Codespaces to go remote. Any machine running Docker can host your Dev Container — a beefy EC2 instance, a shared build server, or a colleague’s workstation.

Docker context over SSH

# Point Docker at a remote host
docker context create dev-server \
  --docker "host=ssh://ubuntu@dev.acme.internal"

# Switch to it
docker context use dev-server

# Now devcontainer CLI builds on the remote machine
devcontainer up --workspace-folder .

VS Code’s Docker extension picks up your active context. Set docker.environment in settings to {"DOCKER_HOST": "ssh://ubuntu@dev.acme.internal"} for a per-workspace override.

DevPod

DevPod is an open-source, client-only tool that provisions Dev Containers on any backend: local Docker, SSH hosts, AWS, GCP, Azure, or Kubernetes. It reads your devcontainer.json and handles the rest.

# Create a workspace on an SSH provider
devpod up github.com/your-org/your-repo \
  --provider ssh \
  --option HOST=dev.acme.internal

# Or on AWS
devpod up github.com/your-org/your-repo \
  --provider aws \
  --option REGION=us-east-1 \
  --option INSTANCE_TYPE=m5.xlarge
Why DevPod DevPod is vendor-neutral and client-only — no server component to manage. It works with VS Code, JetBrains, and terminal-only SSH. Good for teams that want Codespaces-like convenience without GitHub lock-in or for orgs that must run environments on their own infrastructure.

Attaching to existing containers

Sometimes the container already exists — a long-running database, a CI runner, or a container started by another tool. Use attachContainer mode:

// .devcontainer/devcontainer.json
{
  "attachable": true,
  "overrideCommand": false,
  "workspaceFolder": "/app",
  "customizations": {
    "vscode": {
      "extensions": ["ms-python.python"]
    }
  }
}

In VS Code: Dev Containers: Attach to Running Container, or use the CLI: devcontainer up --attach targeting an existing container ID.

Choosing Your Architecture

Patterns for Hybrid Teams

In practice, most teams end up with a mix: some developers run local containers (fast SSDs, beefy machines), others use Codespaces (company laptops with limited RAM, Windows without WSL2, or remote contractors). The good news: the same devcontainer.json works everywhere. The challenge is in the edges.

Kubernetes-based dev environments

Tools like Gitpod and internal platforms built on Eclipse Che run Dev Containers in Kubernetes pods. This gives you cluster-level resource sharing, RBAC, and the ability to connect to in-cluster services without port-forwarding. The trade-off: more infrastructure to manage, and you need Kubernetes expertise on the platform team.

When to consider K8s dev envs Your microservices need to talk to other in-cluster services during development. You have a platform team that already operates Kubernetes. You need fine-grained resource quotas across many developers. Otherwise, Codespaces or remote Docker is simpler.

Multi-root workspaces

VS Code multi-root workspaces let you open multiple folders in one window. Each folder can have its own .devcontainer/, but VS Code will use the configuration from the first folder (or let you choose). For true multi-container development, use Docker Compose instead — it’s more predictable.

When local is better

Remote environments add latency. If your workflow involves rapid save-and-test cycles, GPU-heavy work, or massive local file trees, local containers will feel faster. Go remote when onboarding speed, hardware constraints, or security boundaries demand it — not because it sounds cool.

Latency reality check Typing in a remote VS Code session adds 50–150ms of round-trip latency depending on geography. File watchers and hot-reload can feel sluggish. Prebuilds help with startup but not with ongoing I/O. Test locally if your feedback loop is tight.

Why this matters

The Dev Container spec is deliberately runtime-agnostic. Your devcontainer.json doesn’t know or care whether it runs on a MacBook, a Codespace, or an EC2 instance. That portability is the real value: one config file, multiple deployment targets, zero “works on my machine.” Invest in making that config robust, and let each developer choose the runtime that fits their constraints.
