Chapter 03 · Hands-On

First Dev Container Project

Enough theory. Open an empty folder, add two files, and watch VS Code spin up a fully configured Node.js environment inside a container. This chapter walks through every step—and every thing that happens behind the curtain when you click Reopen in Container.

The Minimal Config

Start with any folder that has a package.json. You only need one new directory and one new file to make it a Dev Container project.

// .devcontainer/devcontainer.json
{
  "name": "node-app",
  "image": "mcr.microsoft.com/devcontainers/javascript-node:22",
  "forwardPorts": [3000]
}

That is it. Three fields. The image pulls a pre-built Microsoft container image with Node 22, git, and common CLI tools already installed. forwardPorts maps port 3000 from inside the container to your host so you can hit localhost:3000 in your browser.

Why mcr.microsoft.com? Microsoft publishes official Dev Container images optimized for this workflow. They include the VS Code Server prerequisites, a non-root user (node or vscode), and sensible defaults. You can use any Docker image, but these save you twenty minutes of plumbing on day one.

The “Reopen in Container” Flow

Open the folder in VS Code. The Dev Containers extension detects .devcontainer/devcontainer.json and shows a notification: “Reopen in Container.” Click it. Here is what happens next, in order:

  1. Image pull — Docker pulls mcr.microsoft.com/devcontainers/javascript-node:22 (first time only; cached after that).
  2. Container create — Docker creates a container from the image, bind-mounting your project folder into /workspaces/node-app.
  3. VS Code Server install — The extension copies a headless VS Code Server binary into the container and starts it.
  4. Extensions install — Any extensions listed in customizations.vscode.extensions get installed inside the container.
  5. Lifecycle hooks — onCreateCommand, updateContentCommand, postCreateCommand, postStartCommand, and postAttachCommand run in sequence (more on these in Chapter 04).
  6. Port forwarding — Ports in forwardPorts get mapped to your host.
  7. Terminal connects — VS Code opens a terminal session inside the running container. You are in.
Gotcha The first docker pull can take 2–5 minutes on a typical connection. The base Node image is ~1 GB. After it is cached, reopening takes seconds. If your team is on a corporate VPN with a slow Docker registry mirror, this is the number one complaint. Pre-pull in CI or use a local registry cache.

Verifying It Works

Once the container is running, open the integrated terminal. You are now inside the container:

# Check you're inside the container
$ hostname
a3f8c2e91b47

# Node is installed and matches the image version
$ node --version
v22.x.x

# Your project files are mounted
$ ls /workspaces/node-app
package.json  node_modules/  src/  .devcontainer/

# Run the app
$ npm start
Server listening on http://localhost:3000

Open http://localhost:3000 on your host machine. The port forward makes the containerized server reachable as if it were running locally.

If node --version shows the version you expect and your app starts, you are done. Everything from here is refinement.

Adding postCreateCommand

You do not want every developer to manually run npm install after opening the container. Add a postCreateCommand:

// .devcontainer/devcontainer.json
{
  "name": "node-app",
  "image": "mcr.microsoft.com/devcontainers/javascript-node:22",
  "forwardPorts": [3000],
  "postCreateCommand": "npm install",
  "customizations": {
    "vscode": {
      "extensions": [
        "dbaeumer.vscode-eslint",
        "esbenp.prettier-vscode"
      ]
    }
  }
}

postCreateCommand runs once after the container is created (and again if the container is rebuilt). It is the right place for npm install, pip install -r requirements.txt, or any project bootstrapping. Extensions listed in customizations.vscode.extensions get installed automatically—no more “please install the ESLint extension” messages in your README.

Key concept Changing devcontainer.json does not auto-update a running container. You need to run Dev Containers: Rebuild Container from the Command Palette (Ctrl+Shift+P / Cmd+Shift+P) for changes to take effect.

Here is what a minimal Dev Container project looks like on disk. Two files in a .devcontainer/ directory are all it takes. The Dockerfile is optional when using "image"—but once you need custom tooling (Chapter 05), you will add one.

Why this matters

Every developer on your team now opens this repo and gets the same Node version, the same extensions, and the same bootstrapping—without reading a setup doc. The container is disposable: break something, rebuild, and you are back to a known-good state in under a minute.
