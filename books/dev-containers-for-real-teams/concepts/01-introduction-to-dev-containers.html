Introduction to Dev Containers

A Dev Container is a full development environment defined as code—a JSON file, an optional Dockerfile, and a set of conventions that let any tool spin up an identical workspace in seconds. This chapter covers what the spec actually says, the problem it solves, and when you should not use it.

The Problem: “Works on My Machine”

Every team eventually writes a setup document. It starts as a quick README section, then grows into a wiki page with platform-specific tabs, and eventually becomes a tribal ritual where a senior engineer sits with each new hire for half a day.

I’ve seen teams where onboarding a new developer takes two full days. Two days of “oh, you also need to install X” and “that version doesn’t work on ARM Macs.” Dev Containers compress that to the time it takes to pull an image.

The root cause is always the same: the development environment is implicit. It lives in each developer’s head and their laptop’s global state—brew installs, nvm aliases, Python virtualenvs, background Postgres instances. Dev Containers make that environment explicit and portable.

Key insight Dev Containers don’t replace Docker in your workflow—they standardize the development side of Docker that every team was already improvising.

The Spec and the Ecosystem

The Development Containers Specification started inside Microsoft as part of VS Code Remote. In 2022, Microsoft open-sourced it and contributed it to the community. It is now maintained as an open standard under the Dev Containers GitHub organization, with contributions from multiple vendors.

The spec defines three things:

  1. devcontainer.json — the metadata file that describes the environment (base image, extensions, ports, lifecycle scripts, environment variables)
  2. Dev Container Features — shareable, composable units of installation logic (e.g., “install Node 20” or “add the AWS CLI”)
  3. Dev Container Templates — starter configurations you can scaffold from

The key architectural decision: the spec separates container definition from tool integration. Any editor or CI system can read the same devcontainer.json and build the same environment.

Tool Type Notes
VS Code + Dev Containers ext. Editor Reference implementation; richest integration
GitHub Codespaces Cloud IDE Reads devcontainer.json natively
JetBrains Gateway Editor IntelliJ, GoLand, etc. via remote dev
DevPod CLI / Desktop Open-source, provider-agnostic launcher
devcontainer CLI CLI Official reference CLI; used in CI pipelines

The .devcontainer Folder Convention

By convention, configuration lives in a .devcontainer/ folder at the repo root. The minimal structure:

# Minimal
.devcontainer/
  devcontainer.json

# Typical
.devcontainer/
  devcontainer.json
  Dockerfile          # custom image
  docker-compose.yml  # multi-service stack
  post-create.sh      # lifecycle script

Here is the simplest possible devcontainer.json:

{
  "name": "My Project",
  "image": "mcr.microsoft.com/devcontainers/base:ubuntu"
}

That’s it. Two fields. Any tool that understands the spec can pull that image and drop you into a working shell. Everything else—Features, lifecycle hooks, port forwarding—layers on top of this.

Key Terminology

Term Meaning
Dev Container A Docker container configured for development via devcontainer.json
Feature A self-contained, shareable install script + metadata. Composable—you can stack multiple Features on any base image
Template A starter .devcontainer folder you scaffold into a new project
Lifecycle script A hook that runs at a specific point: onCreateCommand, postCreateCommand, postStartCommand, postAttachCommand
Workspace folder The directory mounted into the container (default: the repo root at /workspaces/<repo>)

When NOT to Use Dev Containers

Dev Containers are not universally the right answer. Skip them when:

  • GPU-heavy local workflows — GPU passthrough into Docker is brittle, especially on macOS (where it doesn’t exist). If your daily loop requires CUDA, you may be better off with a remote VM.
  • Ultra-low-latency file systems — Bind-mounted volumes on macOS have measurable I/O overhead. For very large repos (100k+ files), this can slow file watchers and builds. Named volumes help, but add complexity.
  • The team won’t run Docker — If half the team is on locked-down corporate laptops that prohibit Docker Desktop, you need a remote-first strategy (Codespaces, remote Docker hosts) rather than a local Dev Container.
  • Trivial projects — A single-file script that needs only a system Python does not benefit from container isolation. Don’t add ceremony to a task that has none.
Foot-gun On Apple Silicon Macs, the container runs under linux/arm64 by default. If your production target is linux/amd64, you need "runArgs": ["--platform=linux/amd64"]—but that triggers Rosetta emulation, which is significantly slower. Test early.

Why this matters

Dev Containers shift environment configuration from oral tradition to version-controlled code. A new hire clones the repo, opens it in a supporting tool, and gets the exact environment the team designed—same runtimes, same extensions, same shell config. That’s not a nice-to-have; it’s the difference between onboarding in minutes and onboarding in days. The rest of this book shows you how to build that environment correctly.
