Chapter 06 · Multi-Service

Using Docker Compose for Full Stacks

Your app does not run alone. It needs a database, a cache, maybe a queue. Docker Compose lets you define the whole stack in one file and have VS Code attach to exactly the right container. Here is how to wire it up without losing your mind.

When You Need Compose

A single-container dev environment works until your first integration test hits a real database. At that point you have two choices: mock everything (and discover the bugs in production) or run the actual services locally. Compose is option two.

Use Compose when your app depends on:

  • A database (PostgreSQL, MySQL, MongoDB)
  • A cache or message broker (Redis, RabbitMQ)
  • A sidecar service (auth proxy, localstack, mail catcher)
  • Another microservice your team owns
Single container is fine too If your app only talks to external APIs and has no local service dependencies, skip Compose. A Dockerfile-based dev container is simpler and faster to rebuild. Do not add Compose just because you can.

Wiring devcontainer.json to Compose

Instead of image or build, you set two fields: dockerComposeFile and service.

// .devcontainer/devcontainer.json
{
  "dockerComposeFile": "docker-compose.yml",
  "service": "app",
  "workspaceFolder": "/workspace",
  "shutdownAction": "stopCompose",
  "forwardPorts": [3000, 5432],
  "postCreateCommand": "npm install && npm run db:migrate"
}

The service field is critical: it tells VS Code which container to attach to. You will have your shell, extensions, and editor inside that container. The other services (database, cache) run alongside it but VS Code does not attach to them.

The service field Think of service as "which container is my workstation." Everything else is infrastructure. VS Code opens a terminal in the service container; you connect to Postgres from there just like you would in production.

The docker-compose.yml

Here is a complete Compose file for a web app with PostgreSQL and Redis. Place it at .devcontainer/docker-compose.yml.

# .devcontainer/docker-compose.yml
services:
  app:
    build:
      context: ..
      dockerfile: .devcontainer/Dockerfile
    volumes:
      - ..:/workspace:cached
    command: sleep infinity
    depends_on:
      db:
        condition: service_healthy
      redis:
        condition: service_started
    environment:
      DATABASE_URL: postgres://dev:dev@db:5432/app_dev
      REDIS_URL: redis://redis:6379/0

  db:
    image: postgres:16-bookworm
    restart: unless-stopped
    volumes:
      - pgdata:/var/lib/postgresql/data
    environment:
      POSTGRES_USER: dev
      POSTGRES_PASSWORD: dev
      POSTGRES_DB: app_dev
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U dev"]
      interval: 3s
      timeout: 3s
      retries: 10

  redis:
    image: redis:7-alpine
    restart: unless-stopped

volumes:
  pgdata:

The command: sleep infinity keeps the app container alive so VS Code can attach. Without it, the container exits immediately because there is no long-running process yet — VS Code is the process.

Key Patterns

Health Checks for Dependent Services

The depends_on with condition: service_healthy pattern prevents your app container from starting before the database is ready to accept connections. Without it, your migration script races the database startup and fails intermittently.

Foot-gun Using depends_on without a condition only waits for the container to start, not for the service inside it to be ready. PostgreSQL takes 1–3 seconds to initialize. Your migration will fail. Always add a health check to database services.

Named Volumes for Data Persistence

The pgdata named volume keeps your database contents across container rebuilds. Without it, every Rebuild Container command wipes your development data and re-runs all migrations from scratch.

# To wipe and start fresh when you actually want to:
$ docker volume rm devcontainer_pgdata
# Or from inside VS Code terminal:
$ docker compose -f .devcontainer/docker-compose.yml down -v

Running Migrations in Lifecycle Hooks

Put database migrations in postCreateCommand or postStartCommand depending on when you need them:

HookRuns whenUse for
postCreateCommandAfter first build or rebuildnpm install, db:migrate, db:seed
postStartCommandEvery time the container startsLightweight checks, starting watchers
postAttachCommandEvery time VS Code attachesPrinting a welcome message, activating venvs

For the stack above, a solid lifecycle setup is:

{
  "postCreateCommand": "npm install && npx prisma migrate deploy && npx prisma db seed",
  "postStartCommand": "npx prisma migrate deploy"
}

Running migrations in postStartCommand too catches any schema changes pulled from git since the last rebuild.

Networking Between Services

Docker Compose creates a shared network automatically. Inside the app container, you reach PostgreSQL at db:5432 and Redis at redis:6379 — using the service name as the hostname. No IP addresses, no localhost.

Common mistake Do not use localhost in your connection strings inside the container. localhost means "this container," not the database container. Use the Compose service name (db, redis) instead. Set these as environment variables in the Compose file so your app config stays container-aware.

Shutdown Behavior

The shutdownAction field controls what happens when you close VS Code:

ValueEffect
stopComposeStops all services. Database data persists in the named volume. Use this.
noneLeaves everything running. Useful if you run services for other projects too.

Why this matters

A Compose-backed dev container means every developer on the team gets the same stack — same database version, same Redis version, same network topology — without installing anything on their host. The new hire clones the repo, opens it in VS Code, and ten minutes later they are running the full application against real services. That is a team multiplier that compounds every single week.
