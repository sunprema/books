Chapter 06 · For Engineers

What This Means for Software Engineers

Everything so far has been hardware people solving a hardware problem. But you don't have to design a chip to be affected by the memory wall — you hit it every time your code is slower than it "should" be for the amount of math it's doing. Here's how the last five chapters change how you should actually write software in 2026.

If there's one habit I want you to walk away with, it's this: stop asking "how many operations does this do?" first. Start asking "how many bytes does this touch, and how many times?" That single reframing explains most of the performance work happening in serious systems right now.

Think in bytes moved, not just FLOPs executed

The mental model professionals use for this is the roofline model: plot your code's arithmetic intensity (FLOPs performed per byte moved) against two hard ceilings — the machine's peak compute rate, and its peak memory bandwidth. Low arithmetic intensity means you're memory-bound no matter how fast the math units are (that's decode, from Chapter 3, living permanently under the memory roof). High intensity means you're compute-bound (that's prefill). Tools like Intel Advisor's Cache-Aware Roofline Model extend this idea to the whole memory hierarchy at once — not just DRAM bandwidth, but every cache level a core actually sees — so you can tell exactly which rung of Chapter 2's pyramid is choking your code. The practical takeaway: profile bytes moved with a tool that reports memory traffic (perf counters, CARM, vendor profilers), not just cycles or instruction counts. A kernel that "should" be fast on paper and isn't is almost always paying a memory tax you haven't measured yet.

Know when to offload: CPU vs. GPU vs. NPU

Chapters 1 through 4 gave you the hardware menu; here's the ordering heuristic that follows from it:

  • Keep hot, frequently-reused data on the GPU's own HBM — active model weights, the working slice of the KV cache, anything read thousands of times a second.
  • Offload cold, infrequently-touched data to CPU DRAM (or further, to NVMe) only when your latency budget can absorb the trip — aging KV cache entries, inactive MoE experts, batch-time-tolerant work.
  • Use the CPU for what CPUs are still best at: branchy orchestration, request scheduling, and control flow that doesn't vectorize.
  • Reach for fixed-function accelerators (NPUs, Tensor Cores) for dense, low-precision, repetitive math — but only once you've confirmed the bottleneck really is compute, not the memory feeding it.

Design for KV-cache management as a first-class concern

If you build or operate LLM-serving systems, the KV cache (Chapter 3) deserves the same deliberate engineering as any other scarce resource: quantize it aggressively once accuracy allows, evict or page out cold entries the way an OS pages out cold memory, shard it by expert if you're running MoE, and consider GPU–CPU cooperative schemes that compute on cached state in place rather than hauling it back and forth. This isn't a nice-to-have anymore — for long-context, high-concurrency serving, KV-cache strategy is frequently the single biggest lever on both latency and cost.

Zooming out: the bigger picture

Step back from LLMs for a second, because this pattern is bigger than chatbots. Modern chips already can't power all their own transistors at once at full clock — a phenomenon architects call dark silicon, the direct legacy of Dennard scaling breaking down in the mid-2000s. You simply cannot turn on every square millimeter of a modern die without melting it. That's the real "end of the free lunch" this book opened with: it's not only that memory bandwidth lags compute — it's that even the compute you can build, you often can't afford to power. The industry's answer on both fronts rhymes: specialize. Instead of one power-hungry, general-purpose core doing everything, build focused, efficient blocks — domain-specific accelerators for AI, video, crypto — and turn on only the ones a given workload needs. Open instruction sets like RISC-V are accelerating this shift because they let a team bolt a small, purpose-built, royalty-free core onto a chiplet (Chapter 1) without licensing a whole general-purpose CPU architecture just to get a tiny accelerator control core. Put the whole book's argument in one sentence: the engineers who get the most out of 2026-and-beyond hardware won't be the ones who write the cleverest arithmetic — they'll be the ones who understand, and design around, where the data lives and how far it has to travel.

Key idea Moore's Law gave you free compute for forty years. It's not giving you free bandwidth, and it's not giving you free power. Both of those now have to be designed for on purpose — in silicon, and in your code.

Why this matters

Every fix in Chapter 5 — HBF, PNM, 3D stacking, new decoding algorithms — exists because hardware alone can't out-run the memory wall forever. The software you write is the other half of that fight. Profile bandwidth. Place data deliberately. Question whether your algorithm needs to touch memory as often as it currently does. That's the whole book, distilled.
