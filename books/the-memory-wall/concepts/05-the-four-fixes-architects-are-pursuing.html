Chapter 05 · The Fixes

The Four Fixes Architects Are Pursuing

You can't repeal the memory wall, but you can out-engineer it from four different directions at once. Two of these fixes are about hardware getting closer to the data. One is about hardware getting closer to the data literally. And one doesn't touch hardware at all — it changes the algorithm so it needs less data, less often. Real architecture teams are pursuing all four simultaneously, because none of them alone is enough.

This is my favorite part of the story, honestly. When an industry hits a wall this hard, you get four genuinely different, genuinely clever answers all at once — not because anyone agreed on a plan, but because smart people converge on the same problem from wherever they're standing.

1. High Bandwidth Flash (HBF): cheap capacity for the data you rarely change

Chapter 2 introduced HBF as HBM's packaging trick applied to ordinary NAND flash instead of DRAM. The reasoning is purely economic: a huge fraction of an inference server's memory footprint is model weights, which barely change request to request, so they don't need DRAM's fast-write, low-latency guarantees — they need to be read, fast and often, and stored cheaply. HBF trades some read latency and all but the occasional write for a dramatically lower cost per gigabyte, letting architects afford far more capacity at HBM-adjacent bandwidth. The honest caveat: it's not shipping in real systems yet — expect first products around 2027 — but the memory hierarchies being designed today already have a labeled slot waiting for it.

2. Processing-Near-Memory (PNM): move the computation, not the data

If moving bytes is the expensive part, the boldest fix is to stop moving them: put a thin logic layer directly next to the memory and do part of the computation right there, shipping back only a small result instead of the whole dataset. Processing-Near-Memory (PNM) designs add a logic die to a 3D-stacked memory package while keeping the standard memory interface intact, so the rest of the system barely notices anything changed — it's just a memory chip that happens to also compute. Recent research designs aim high: China's Unisplendour has described a "Zixuan" architecture centered on 3D DRAM with a PNM mode targeting roughly 30 TB/s of memory bandwidth for exactly this kind of near-memory compute. PNM is a great fit for the simple, repetitive, bandwidth-hungry parts of LLM inference — attention score reduction, embedding lookups — where the operation is simple but the data volume is enormous.

3. 3D memory-logic stacking: density's price is heat

Stack a compute logic die directly beneath (or between) memory dies, connected by dense through-silicon vias, and you get PNM's benefits with even shorter wires — the natural extension of the 3D V-Cache trick from Chapter 1, now applied to main memory itself. The catch is physics, not economics: heat generated deep inside a 3D stack has nowhere quick to go, since it has to conduct through every layer above it before reaching a heatsink. Research on thermal-aware 3D-stacked designs for LLM serving finds that stack height is fundamentally limited by cumulative thermal resistance — most estimates put the practical ceiling around 6–8 device tiers without exotic cooling, and any stack dissipating north of 350 W likely needs liquid microchannel or vapor-chamber cooling built right into the package. Every extra layer of density you stack up is also, quite literally, an extra layer of insulation you have to engineer your way through.

4. Rethinking the autoregressive limit itself

The fourth fix doesn't touch silicon at all — it questions the algorithm. Standard decoding generates exactly one token per expensive memory-bound pass through the model. Speculative decoding breaks that assumption: a small, cheap "draft" model guesses several tokens ahead, and the big model verifies (or rejects) that whole guess in a single pass — turning one expensive memory fetch into a check for several tokens at once instead of one. It went from research curiosity to production default astonishingly fast: by 2026 it ships in vLLM, SGLang, and TensorRT-LLM, and MLPerf Inference v6.0 added a DeepSeek-R1 scenario that effectively mandates it to hit latency targets. A more radical version of the same idea drops autoregression altogether: diffusion-inspired and non-autoregressive approaches (recent examples include DART and block-diffusion serving systems) generate whole blocks, or entire sequences, in parallel passes instead of one token at a time — trading a different kind of compute for a lot fewer trips to memory.

Key idea Notice the pattern: three of the four fixes shorten the distance between data and computation; the fourth shortens how often you have to make that trip at all. Both strategies attack the same number — bytes moved per useful result.

Why this matters

None of these fixes are exotic research-lab curiosities anymore — HBF has vendor roadmaps, PNM has shipping-adjacent prototypes, 3D stacking is already in production (just not yet with logic underneath), and speculative decoding is a one-line config flag in the serving stack you're probably already running. The final chapter is about what all of this means for the code you write.
