Chapter Six

Embedding Lua in Elixir

Here's where the two languages meet. The lua library (by Tristan Pratt) is an ergonomic wrapper over Luerl — a pure-Erlang Lua VM. No NIFs, no C: a Lua state is just an Elixir term you evaluate, copy, and pass between processes.

# mix.exs
def deps do
  [{:lua, "~> 0.4"}]
end

Evaluate a script

Lua.new/0 builds a fresh, sandboxed VM. eval!/2 runs a chunk and returns {results, new_state} — results is a list, because Lua functions return multiple values.

iex> {[4], _lua} = Lua.eval!(Lua.new(), "return 2 + 2")
iex> {[6, 7], _} = Lua.eval!(Lua.new(), "return 6, 7")
Compile once with the ~LUA sigil

Parsing Lua text on every call is wasteful. The ~LUA sigil with the c modifier pre-compiles the chunk at Elixir compile time, so there's no runtime parse:

import Lua, only: [sigil_LUA: 2]
{[4], _} = Lua.eval!(~LUA[return 2 + 2]c)   # validated at compile time

Passing values in and out

set!/3 puts an Elixir value at a (possibly nested) Lua path; get!/3 reads one back. This is your data bridge.

lua =
  Lua.new()
  |> Lua.set!([:user], %{"name" => "Ada", "tier" => "pro"})
  |> Lua.set!([:limits, :max], 100)     # nested table path

{[msg], _} =
  Lua.eval!(lua, ~LUA[return user.name .. " can use " .. limits.max]c)
# msg = "Ada can use 100"

# read a Lua global back into Elixir
lua = Lua.set!(Lua.new(), [:greeting], "hi")
"hi" = Lua.get!(lua, [:greeting])
Mind the table/list rule from Chapter Three

Elixir maps with string keys become Lua tables; Elixir lists become 1-based Lua sequences. Coming back, a dense sequence decodes to a list and a mixed table to a list of {key, value} tuples. Keep arrays dense and objects string-keyed to avoid surprises.

Exposing Elixir functions to Lua

The real power: let scripts call your code. The quick way is an anonymous function via set!. It receives the Lua args as a list and must return a list.

lua = Lua.set!(Lua.new(), [:sum], fn args -> [Enum.sum(args)] end)
{[10], _} = Lua.eval!(lua, "return sum(1, 2, 3, 4)")

# need to mutate Lua state? take a second arg and return {results, state}
lua = Lua.set!(Lua.new(), [:remember], fn args, state ->
  {[], Lua.set!(state, [:count], Enum.count(args))}
end)

Organized APIs with deflua

For a real surface area, define a module with use Lua.API and the deflua macro, then load_api/2 it. A scope namespaces the functions into a Lua table.

defmodule MyApp.LuaKit do
  use Lua.API, scope: "kit"

  deflua double(v), do: 2 * v

  # `state` gives you the VM — here we call a built-in Lua function
  deflua shout(text), state do
    Lua.call_function!(state, [:string, :upper], [text])
  end
end

lua = Lua.load_api(Lua.new(), MyApp.LuaKit)
{[10], _}     = Lua.eval!(lua, ~LUA[return kit.double(5)]c)
{["HEY"], _}  = Lua.eval!(lua, ~LUA[return kit.shout("hey")]c)

Secrets the script must never see: private state

You often need an API key or the current user inside your Elixir functions, but those values must stay invisible to the Lua code. Use private storage — it travels with the VM but isn't reachable from Lua.

lua = Lua.put_private(Lua.new(), :api_key, "sk-secret")

defmodule MyApp.Weather do
  use Lua.API, scope: "weather"
  deflua fetch(city), state do
    key = Lua.get_private!(state, :api_key)   # available to Elixir only
    [MyApp.WeatherClient.get(city, key)]
  end
end
Key idea

The VM is the whole boundary. The script sees exactly the globals you set! and the functions you load_api — nothing else. Keys, the database, the network: all reachable from your Elixir functions, none reachable from Lua. That asymmetry is the entire security model, and it's why this pairing is so comfortable.

Source: lua Elixir library docs (lua.hexdocs.pm). Underlying VM: Luerl (github.com/rvirding/luerl).
