Chapter Five

Coroutines

A coroutine is a function you can pause and resume. It's not a thread — nothing runs in parallel — but it lets a computation yield control and pick up exactly where it left off. This is how Lua does generators, cooperative schedulers, and state machines without callbacks.

local function counter()
  for i = 1, 3 do
    coroutine.yield(i)        -- pause here, hand `i` back to the caller
  end
  return "done"
end

local co = coroutine.create(counter)
print(coroutine.resume(co))   -- true   1
print(coroutine.resume(co))   -- true   2
print(coroutine.resume(co))   -- true   3
print(coroutine.resume(co))   -- true   done
print(coroutine.status(co))   -- dead

resume starts (or continues) the coroutine; yield suspends it and sends values back to whoever resumed. Each resume returns true plus whatever yield (or return) produced.

Values flow both ways

The arguments you pass to resume become the return value of the yield that woke up. That two-way channel is what makes coroutines more than generators.

local function echo()
  while true do
    local got = coroutine.yield()   -- receive on resume
    print("got: " .. tostring(got))
  end
end

local c = coroutine.create(echo)
coroutine.resume(c)                 -- prime it (runs to first yield)
coroutine.resume(c, "ping")         -- got: ping
coroutine.resume(c, "pong")         -- got: pong
resume never crashes your host

Like pcall, coroutine.resume returns false, err instead of raising if the coroutine errors. So a coroutine is also a natural fault boundary — handy when the code inside it is untrusted.

The classic: a lazy generator

coroutine.wrap turns a coroutine into a plain function you can call (or drive a for loop with). Each call resumes it and returns the yielded value.

local function fib()
  local a, b = 0, 1
  while true do
    coroutine.yield(a)
    a, b = b, a + b
  end
end

local nextfib = coroutine.wrap(fib)
for _ = 1, 8 do io.write(nextfib(), " ") end
-- 0 1 1 2 3 5 8 13
Why this matters for embedding

Coroutines are how a host drives long-running scripts incrementally. A game engine resumes each NPC's script for one "tick" then yields; a workflow engine yields whenever a script awaits I/O the host must perform. When you embed Lua in Elixir, you can resume a script, let it yield a request ("fetch this URL"), do the work on the BEAM, and resume it with the answer — cooperative async, with the host firmly in control.

Not parallelism

Coroutines share one line of execution — they don't use multiple cores. For real concurrency on the BEAM you run each Lua VM in its own Elixir process. Coroutines are about structure (pausing/resuming), not throughput.

Source: Lua 5.5 Reference Manual §6.2 (coroutine manipulation).
