Chapter Nine

Sandboxing in Production

Running other people's code is a privilege you grant carefully. The good news: between Luerl and the BEAM you get three independent layers of containment. The job is to turn all three on.

Layer 1 — a sandboxed VM by default

Lua.new/0 already removes the dangerous standard libraries. io, os, package, and require are sandboxed out of the box, so a script can't read files, shell out, or load native code.

# default: dangerous libs already removed
lua = Lua.new()

# tighten or loosen explicitly
Lua.new(sandboxed: [[:os, :exit]])   # sandbox just os.exit
Lua.new(sandboxed: [])               # DANGER: nothing sandboxed
Lua.sandbox(lua, [:os, :exit])       # sandbox after the fact
Never pass sandboxed: [] for untrusted code

That re-arms os.execute, file I/O, and dynamic loading. Use it only for code you wrote and trust. For anything user-supplied, keep the defaults and add to the deny-list, never empty it.

Layer 2 — the BEAM process boundary

This is the advantage native Lua can't match. Because a Luerl VM is just an Erlang term, you run each evaluation inside its own short-lived, supervised process — then cap its wall-clock time and heap size. A runaway while true do end dies without touching your app.

def run_guarded(src, timeout_ms \\ 1_000) do
  task =
    Task.async(fn ->
      # cap memory: kill the process if its heap exceeds the limit
      Process.flag(:max_heap_size, %{size: 50_000, kill: true})
      Lua.eval!(Lua.new(), src)
    end)

  case Task.yield(task, timeout_ms) || Task.shutdown(task) do
    {:ok, {result, _lua}} -> {:ok, result}
    nil -> {:error, :timeout}        # script ran too long; killed
  end
end
Key idea

On the BEAM, "stop a misbehaving script" is just "kill a process." Timeouts, memory ceilings, and crash isolation come from the runtime you already trust to run your whole app. That's the single biggest reason to embed Lua here rather than in a C host.

Layer 3 — a curated API surface

Sandboxing removes capabilities; your API adds back exactly the safe ones. Every door is one you chose: a deflua function or an ash_lua action, each enforcing its own checks. Keep secrets in put_private (Chapter Six), keep authority in the actor/tenant (Chapter Seven), and the script's whole world is the surface you designed.

Know the trade-offs

  • Luerl is not C-Lua. It implements Lua 5.x semantics in Erlang, so it's slower than the reference interpreter and trails the very newest 5.5 features. For glue logic and rules that's irrelevant; for tight numeric loops, do the heavy lifting in Elixir and call out from Lua.
  • Marshalling has a cost. Converting big tables across the boundary isn't free. Pass identifiers, not megabytes; let Ash fetch what's needed inside the call.
  • Sandboxes need review. A capability you expose is a capability scripts have. Treat deflua functions and exposed actions like a public API — because to your users, they are.
A sensible default recipe

Default Lua.new() + one BEAM process per run + a 1-second timeout + a heap cap + an explicitly curated API + (for data) ash_lua with a fixed actor. That stack safely runs untrusted, user-authored scripts in a multi-tenant SaaS — which is exactly where we came in.
