Chapter Seven

Lua & Ash: Scripting Your Domain

Ash models your application as resources and actions, with authorization, validation, and multitenancy baked in. ash_lua exposes that domain to Lua scripts — so a script can create, read, update, and delete records through the exact same policies your app already enforces.

# mix.exs, then: mix igniter.install ash_lua
{:ash_lua, "~> 0.1.0"}

Opt a domain and its resources in

Add the AshLua.Domain extension to the domain and AshLua.Resource to each resource you want reachable. By default everything nests as domain.resource.action; you can rename the Lua tables.

defmodule MyApp.Accounts do
  use Ash.Domain, otp_app: :my_app, extensions: [AshLua.Domain]

  lua do
    name "accounts"      # the Lua table name (optional)
  end

  resources do
    resource MyApp.Accounts.User
  end
end

defmodule MyApp.Accounts.User do
  use Ash.Resource, domain: MyApp.Accounts, extensions: [AshLua.Resource]
  lua do
    name "user"
  end
end

Run a script — with the actor attached

AshLua.eval!/2 evaluates Lua and threads an actor, tenant, and context through every Ash call the script makes. Crucially, those are read-only from Lua: the script cannot promote itself to another user.

{[user_id], _lua} =
  AshLua.eval!(
    """
    local user, err = accounts.user.create({
      name = "Zach",
      email = "z@example.com",
      fields = { "id" }
    })
    assert(err == nil)
    return user.id
    """,
    otp_app: :my_app,
    actor: current_user      # enforced on every call below
  )

The result/err convention

Every call returns (result, err). On success err is nil; on failure result is nil and err is a table with a message. Wrap with assert() to raise, or branch explicitly.

local user, err = accounts.user.create({ name = "Zach" })
if err then
  print("failed: " .. err.message)
else
  print("created " .. user.id)
end

-- assert() raises on the (nil, err) shape — concise when you want fail-fast
local post = assert(posts.post.create({ title = "Hello", fields = {"id"} }))

Reads: filter, sort, paginate, aggregate

-- a filtered, sorted, limited list with selected fields
local results = assert(posts.post.read({
  filter = { published = true },
  sort   = "-created_at",
  limit  = 10,
  offset = 20,
  fields = { "title", { author = { "name" } } }   -- nested relationship
}))

-- aggregates use the same call
local total = assert(posts.post.read({ operation = "count" }))
local avg   = assert(posts.comment.read({ operation = { "avg", "rating" } }))
Fields are opt-in

By default a call returns only primary keys. Ask for what you need with fields — stored attributes, calculations (even with args), and nested relationships. That keeps payloads tight and avoids leaking columns a script shouldn't see.

Update & delete

local updated = assert(posts.post.update({
  id = post.id, title = "Hello again", fields = { "title" }
}))
assert(posts.post.destroy({ id = post.id }))
Why this is the safe way to script a database

The script never sees a connection string or a raw query. It calls named actions, and Ash runs them as the supplied actor and tenant — so your policies, validations, and tenant scoping all still apply. The host (your Elixir code) is the sole source of authority; the actor/tenant/context cannot be read or changed from Lua. This is the database-grade version of Chapter Six's boundary.

Discoverability for your users

AshLua.Docs generates a callable list and per-call markdown straight from your domains — perfect for an in-app "what can I script?" reference.

AshLua.Docs.list_callables(otp_app: :my_app)
# => ["accounts.user.create", "accounts.user.read", ...]
{:ok, md} = AshLua.Docs.callable_doc([otp_app: :my_app], "accounts.user.create")
Still apply limits

ash_lua governs authorization, not resource use. A script can still write an infinite loop or a 100k-row read. Run it under the time/step/memory guards from the next chapter, and design actions with sensible default page sizes.
