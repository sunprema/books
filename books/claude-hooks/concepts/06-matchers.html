Chapter 06 · Targeting

Matchers

A hook without a matcher fires on everything — every tool, every time. Usually you want to be precise: format only after file edits, guard only Bash. The matcher string is how you aim, and its rules are refreshingly small.

Three ways a matcher is read

Claude Code looks at the characters in your matcher and decides how to interpret it:

You write…Read asMatches
"*", "", or omittedmatch-allevery occurrence of the event
Bashexact stringonly the Bash tool
Edit|Writelist (pipe or comma)Edit or Write
^Notebookregex (unanchored)any tool name matching the pattern

The switch is automatic: plain letters, digits, spaces, _ - , | are treated as an exact string or a simple list. The moment you use any other character (^ . * ( )…), the whole thing becomes a JavaScript regular expression, matched unanchored. So Edit|Write is a list, but ^(Edit|Write)$ is a regex — both work, one is just fussier.

The matchers you’ll write most Edit|Write (any file change) · Bash (shell commands) · Write (new files only) · * (all tools). Ninety percent of hooks use one of these four.

Matching MCP tools

Tools that come from MCP servers have compound names in a fixed shape: mcp__<server>__<tool>. That regularity makes them easy to target with a regex matcher:

matching MCP tools"mcp__memory__create_entities"   // one exact MCP tool
"mcp__memory__.*"                // every tool from the memory server
"mcp__.*__write.*"               // any "write*" tool from any server

This is how you’d guard, say, every write tool exposed by any MCP server your project uses, without naming them one by one. (Plugin-scoped servers add a prefix: mcp__plugin_<plugin>_<server>__<tool>.)

Not every event matches on a tool

Here’s the subtlety that trips people up: matcher means different things on different events. On tool events it’s a tool name. On others it selects a variant of the event — and some events don’t take a matcher at all.

So SessionStart with matcher "resume" fires only when you resume a session — perfect for “re-load the context I care about, but only when picking up where I left off.” And a Stop hook simply has no matcher: it fires on every turn’s end, so your script does the filtering.

Finer aim: the `if` field On tool events, a handler can add an "if" permission-rule filter like "Bash(git *)" or "Edit(*.ex)" to narrow within a matched tool — e.g. only Bash commands that start with git, or only edits to .ex files. Handy when a bare tool name is too broad.

So what did we actually gain?

Precision. We can fire on exactly the right tool, the right MCP call, or the right session variant — and we understand why the same word means “tool name” in one place and “trigger” in another. That’s the last primitive. Time to build something worth keeping.

Challenges

  1. Write a matcher that fires for edits to Elixir source and tests but nothing else. (Two approaches: a tool matcher plus an if, or a regex — try both.)
  2. You only want a SessionStart hook to run on a fresh start, never on resume. What matcher do you use?
