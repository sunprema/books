Chapter 04 · Input

The Input Contract

When a hook fires, Claude Code hands your script a single JSON object on standard input. Part of it is the same on every event — the “envelope” — and part is specific to what just happened. Learn the envelope once; look up the payload per event.

Your script is a little clerk. A form slides under the door on stdin. Some fields are always there — who, where, which window. The rest depends on which counter the form came from.

The envelope — on every event

common fields (all events){
  "session_id":      "abc123",
  "transcript_path": "/…/transcript.jsonl",
  "cwd":             "/home/you/my_app",
  "permission_mode": "default",
  "hook_event_name": "PreToolUse"
}
FieldWhat it tells you
session_ididentifier for the current session
transcript_pathpath to the conversation JSON, if you want to inspect it
cwdthe current working directory
permission_modedefault, plan, acceptEdits, bypassPermissions…
hook_event_namewhich event fired — so one script can serve several
One script, many events Because hook_event_name is always present, you can point several events at a single dispatcher script and case on it. That’s exactly how we’ll structure the Elixir hooks in Chapter 07 — one file, a few branches.

The payload — what each event adds

On top of the envelope, each event slips in the fields that make it useful. The ones you’ll read constantly:

EventAdds
PreToolUsetool_name, tool_input (the args about to run)
PostToolUsetool_name, tool_input, tool_response (stdout/stderr/exit_code)
UserPromptSubmitprompt (the text you just typed)
SessionStartsource (startup/resume/clear/compact), model
Stop · SubagentStoplast_assistant_message
Notificationmessage, type
PreCompacttrigger (manual / auto)

So a PreToolUse hook on Bash can read the exact command before it runs; a PostToolUse hook on Edit can read which file was just written. That’s the whole basis of the useful recipes.

Reading it — jq is your friend

In a Bash hook, pull fields out of stdin with jq. Here’s a PreToolUse guard reading the Bash command that’s about to run:

.claude/hooks/read-input.sh#!/usr/bin/env bash
# Slurp all of stdin once — you can only read it a single time.
input="$(cat)"

event=$(jq -r '.hook_event_name' <<<"$input")
tool=$(jq -r '.tool_name'     <<<"$input")
cmd=$(jq -r '.tool_input.command // ""' <<<"$input")

printf '%s ran %s: %s\n' "$event" "$tool" "$cmd" >&2
exit 0
Read stdin exactly once Standard input is a stream — once you’ve consumed it, it’s gone. Capture it into a variable first (input="$(cat)"), then query that variable repeatedly. A hook that pipes stdin into two commands will find the second one empty.

Prefer Elixir? A hook can be any executable. Read stdin, decode with Jason (or :json on recent OTP), and branch:

.claude/hooks/read_input.exs — run with `elixir`# stdin arrives as JSON; decode it and inspect.
input = IO.read(:stdio, :eof) |> Jason.decode!()

case input["hook_event_name"] do
  "PreToolUse" ->
    cmd = get_in(input, ["tool_input", "command"])
    IO.puts(:stderr, "about to run: #{cmd}")

  _other ->
    :ok
end

System.halt(0)

So what did we actually gain?

Eyes. Our script can now see exactly what the agent is about to do — the tool, the arguments, the prompt, the working directory. Seeing is half the job. The other half is answering: allow, block, or nudge. That’s next.
