What a Hook Is

Here is the whole idea in one breath: a hook is a shell command that Claude Code runs for you, automatically, at a fixed moment in its own lifecycle. You register it once in a settings file; from then on, Claude Code shells out to your script every time that moment arrives — before a tool runs, after a file is edited, when it’s about to stop — and listens to what your script says back.

A prompt is a suggestion. You can ask the model, ever so politely, to “always run the formatter.” A hook is a law. It runs whether the model remembered to or not.

Why we want this

Let’s start with the pain. You’re pairing with the agent on an Elixir project. You’ve told it, three times, to run mix format after touching a file. It does — mostly. Then on the fourth edit it forgets, and your diff picks up a spurious whitespace change. The instruction lived in prose, and prose is probabilistic. Sometimes the model follows it; sometimes the context is full and it slips.

The fix is to stop asking. Move the rule out of the conversation and into the machinery. If a formatter must run after every edit, then edits should be wired to the formatter — deterministically, in code you control, not in a paragraph you hope the model reads. That wire is a hook.

The one word that matters Deterministic. Hooks are application-level configuration that executes every time, the same way, regardless of what the model “decided.” They turn a soft request into a hard guarantee.

The anatomy of a firing

Every hook works the same way, and it’s worth drawing once so the rest of the book has a picture to hang on. When the moment arrives, Claude Code:

  1. gathers a small JSON object describing what’s happening, and
  2. runs your command, piping that JSON to its standard input, then
  3. reads your command’s exit code and stdout, and acts on them.

That’s the entire protocol. Your script is an ordinary program — Bash, Elixir, a compiled binary, anything you can run — that reads JSON and returns a number. Everything else in this book is details of that conversation: which moments you can hook, what’s in the JSON, and the vocabulary your script uses to answer.

Your first hook

Let’s make something tiny and real. We’ll log a line to a file every time Claude finishes a turn. It doesn’t block anything; it just proves the wiring. Two pieces: a script, and a registration.

.claude/hooks/log-stop.sh#!/usr/bin/env bash
# The event's JSON is on stdin. We don't even need to read it here.
printf '%s  turn finished\n' "$(date -Iseconds)" \
  >> "$CLAUDE_PROJECT_DIR/.claude/turns.log"
exit 0   # 0 = all good, don't interfere
.claude/settings.json{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          { "type": "command",
            "command": "$CLAUDE_PROJECT_DIR/.claude/hooks/log-stop.sh" }
        ]
      }
    ]
  }
}

Make the script executable (chmod +x), and from then on every completed turn appends a timestamp to turns.log. You didn’t ask the model to do it. You didn’t hope. It simply happens. Notice $CLAUDE_PROJECT_DIR — Claude Code sets that variable to your project root, so your paths don’t depend on where the agent’s working directory happens to be.

Hooks run as you A hook is arbitrary code executing with your credentials and file access, with no confirmation dialog. That is exactly what makes it powerful — and why Chapter 08 is entirely about not shooting your own foot. Only register commands you’ve read.

So what did we actually gain?

We moved one rule out of the conversation and into the machinery. The model can forget, run low on context, or be swapped for a different model entirely — the hook fires anyway. That’s the trade at the heart of hooks: you give up a little flexibility and buy back a lot of certainty. Now let’s learn every moment you’re allowed to hook.
