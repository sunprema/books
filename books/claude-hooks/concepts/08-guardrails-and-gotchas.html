Chapter 08 · Operating Safely

Guardrails & Gotchas

Hooks are power tools. They run real code with your real permissions, at machine speed, on every event. This last chapter is the safety briefing — the handful of habits that separate a hook that helps from one that quietly corrupts your afternoon.

Security: you are the trust boundary

Read this twice A hook executes arbitrary shell commands with your credentials and file access, without a confirmation prompt. Claude Code shows a warning to that effect for a reason. You are responsible for what a hook does — the tool won’t stop it.
  • Never register a command you haven’t read. Treat a hook from the internet like curl | bash — because that’s what it is.
  • Quote your variables. Unquoted $file in a hook is a shell-injection hole; a filename with a space or a ; becomes a command. Always "$file".
  • Use absolute paths. Anchor scripts with $CLAUDE_PROJECT_DIR so a changed working directory can’t point you at the wrong file.
  • Validate before you act. Check that a path is inside the repo before writing to it.

Debugging: make it visible

When a hook “doesn’t fire,” it usually did — you just couldn’t see it. Turn the lights on:

Do thisTo see
claude --debugeach hook match, the command run, its exit code and output
/hooksthe exact configuration Claude Code loaded this session
test the script by handecho '{"tool_input":{"command":"rm -rf /"}}' | ./guard-bash.sh; echo $?
write to stderryour own trace — stderr is surfaced, stdout may be parsed as JSON

That third row is the best habit in the book: a hook is just a program that reads JSON and returns a code, so run it outside Claude Code. Pipe a sample event in, check the exit status. If it behaves at the terminal, it behaves in the agent.

infinite Stop loopa Stop hook that always exit 2s — add a cap (see Ch. 07)

Timeouts, async, and idempotency

  • Keep hooks fast. They run inline and the agent waits. A full mix test belongs on Stop (once per turn), not PostToolUse (every edit).
  • Reach for async:true when a hook’s result doesn’t need to block — a notification, a background log. The agent keeps moving.
  • Set a real timeout. The default is generous (600s); a formatter should cap at 15–30 so a hang can’t freeze the session.
  • Make hooks idempotent. They may fire more than you expect; running twice should be harmless. mix format is; a hook that appends a line to a file every time is not.

Order & the off switch

Within one matcher group, handlers run in array order — so put the formatter before the linter if the linter should see formatted code. Across files, configs merge: your user hooks and the project’s both fire, so a personal guard and a team formatter coexist. And when you need everything off at once — debugging, a demo — "disableAllHooks": true in settings silences the lot without deleting a thing.

A sane default posture Commit a small, readable set of hooks in .claude/settings.json; keep experiments in settings.local.json; make each script runnable and testable on its own; and log to stderr so --debug tells the truth.

So what did we actually gain — across the whole book?

We took the agent’s good intentions and made them structural. A hook is nothing but a program that reads JSON and returns a number, fired at a known moment. From that one idea we built context injection, tool guards, a compile check, and a test gate that won’t quit on red — each one deterministic, auditable, and yours. Prompts ask nicely. Hooks make it so. Go wire your repo.

Challenges

  3. Design a “panic switch”: a personal settings.local.json that disables all hooks, and a note on when you’d flip it.
