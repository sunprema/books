Wiring It Up

A hook is registered in a settings.json file under a single top-level "hooks" key. The shape has three nested layers, and once you see them named, every hook config in the wild reads the same way.

The three layers

From the outside in: an event maps to a list of matcher groups; each group has a matcher (which tool/variant to target — Chapter 06) and a hooks array of handlers that actually run. Read it as a sentence: “on this event, for calls matching this pattern, run these handlers.”

.claude/settings.json — the skeleton{
  "hooks": {
    "PostToolUse": [           // ← the EVENT
      {
        "matcher": "Edit|Write",  // ← the MATCHER (Ch. 06)
        "hooks": [               // ← the HANDLERS (run in order)
          { "type": "command",
            "command": "mix format",
            "timeout": 30 }
        ]
      }
    ]
  }
}
They’re arrays for a reason Both levels are lists: an event can have several matcher groups, and a group can have several handlers. This is how hooks compose — a project can add its rules without erasing yours, and one edit can trigger a formatter and a linter in sequence.

Where the file lives — and who wins

The same "hooks" block can appear in several files. They don’t overwrite each other — Claude Code merges them, so a hook defined in your user settings and one defined in the project both fire. Knowing the locations tells you where to put a rule so the right people get it:

FileScopeShared in git?
~/.claude/settings.jsonevery project you openno — personal machine
.claude/settings.jsonthis project, for everyoneyes — commit it

Skill / agent frontmatterwhile that component is activeships with the component

The practical rule: a formatter your whole team should get goes in .claude/settings.json and gets committed. A noisy personal logger goes in settings.local.json. A machine-wide guard you want on every repo goes in ~/.claude/settings.json.

Config is snapshotted at startup For safety, Claude Code captures your hook configuration when the session begins. Editing settings.json mid-session doesn’t silently change what runs — you’ll review changes through the /hooks menu (or restart). This stops a malicious edit from hot-patching a hook under you.

Five kinds of handler

Every handler has a type. The one you’ll use ninety percent of the time is "command" — a shell command. But the reference gives you five, and the others are genuinely useful once you know they exist:

typeWhat it doesAnswers via
commandrun a shell command / scriptexit code + stdout
httpPOST the event JSON to a URLthe JSON response body
mcp_toolcall a tool on an MCP serverthe tool’s output
promptask a fast model a yes/no questionits JSON decision
agentspawn a subagent (Read/Grep/Glob) to verifyits decision

The last two are quietly remarkable: your hook can defer a decision to another model. “Before running this Bash command, ask a cheap model whether it looks destructive.” We stay with command for the recipes — it’s deterministic and free — but keep prompt/agent in your back pocket for fuzzy judgment calls.

Fields worth knowing on a command handler

FieldMeaning
commandthe shell command (or executable path when args is set)
argsargument list; when present, command runs directly, no shell
timeoutseconds before it’s killed (default 600; 30 for prompt/UserPromptSubmit)
asynctrue = run in the background, don’t block the agent
statusMessagecustom text for the spinner while it runs
shell"bash" (default) or "powershell"
You don’t have to hand-edit JSON Claude Code ships a /hooks slash command that walks you through adding a hook interactively and writes the settings for you. Hand-editing is fine — and clearer for version control — but the menu is a good way to learn the shape.

So what did we actually gain?

A registration grammar. Event → matcher → handlers, layered across files that merge by scope. From here the only unknowns are the data flowing in and the vocabulary flowing out — the next two chapters.
