Talking Back

A hook that only watches is a logger. A hook that decides is a guardrail. There are two ways your script talks back to Claude Code — a simple one and a rich one — and you should reach for the simple one first.

The simple channel: exit codes

Your script’s exit code carries the whole decision for most jobs. There are exactly three cases to remember:

Exit codeMeaningEffect
0successproceed; if stdout is valid JSON, apply its decisions
2blocking errorblock the action; stderr is fed back as the reason
othernon-blocking errorshow stderr; the action still happens

That is the crux of the whole system: exit 2 blocks, and whatever you wrote to stderr becomes the message Claude (or you) sees. Want to stop a dangerous Bash command? Print why, then exit 2. Done.

.claude/hooks/block-rm.sh — a PreToolUse guard#!/usr/bin/env bash
cmd=$(jq -r '.tool_input.command' < /dev/stdin)

if grep -qE 'rm[[:space:]]+-rf' <<<"$cmd"; then
  echo "Blocked: refuse to run 'rm -rf' from the agent." >&2
  exit 2    # ← blocks the tool call; stderr is the reason
fi
exit 0
exit 2 doesn’t mean the same thing everywhere For most events, exit 2 blocks. But its meaning is per event: on PostToolUse the tool already ran, so 2 can only surface stderr to Claude — it can’t un-edit the file. On Stop, 2 means the opposite of “stop”: it keeps the agent working. Use the simulator below to feel the difference.

The rich channel: JSON on stdout

Exit codes are blunt — block or don’t. When you need nuance (allow and add a note; block and tell the user why; rewrite the command before it runs), print a JSON object to stdout and exit 0. Claude Code parses it and obeys.

A few fields are common to every event:

FieldEffect
continuefalse stops Claude entirely (wins over any decision below)
stopReasonmessage shown to you when continue:false
systemMessagea warning surfaced to the user
suppressOutputtrue hides stdout from the transcript

The interesting control is hookSpecificOutput, whose shape depends on the event. For PreToolUse, it carries a permissionDecision — the cleanest way to allow, deny, or escalate:

PreToolUse — allow / deny / ask via JSON{
  "hookSpecificOutput": {
    "hookEventName": "PreToolUse",
    "permissionDecision": "deny",   // allow · deny · ask · defer
    "permissionDecisionReason": "writes outside the repo"
  }
}
permissionDecisionEffect
allowrun the tool, skipping the usual permission prompt
denyblock the tool call
askescalate to the user with a permission dialog
deferfall through to the normal permission flow

Injecting context (the sleeper feature)

Several events let you add text straight into Claude’s context via additionalContext. On UserPromptSubmit and SessionStart this is gold: silently prepend the current git branch, open tickets, or house rules to every prompt — no manual pasting.

UserPromptSubmit — add context, then let it proceed{
  "hookSpecificOutput": {
    "hookEventName": "UserPromptSubmit",
    "additionalContext": "Branch: feat/auth · run mix test before finishing."
  }
}
Blocking with a reason, the JSON way Events like Stop, UserPromptSubmit, and PostToolUse accept a top-level {"decision":"block", "reason":"…"}. The reason is handed to Claude as its next instruction — this is how a Stop hook says “tests failed, keep going and fix them.”

So what did we actually gain?

A voice. Our scripts can now allow, block, escalate, rewrite input, and quietly inject context — the full vocabulary. Reach for exit codes when the answer is yes/no, and JSON when you need to say why or say more. With input (Ch. 04) and output (Ch. 05) in hand, we can build real guardrails.
