Chapter 01Cache-Control Directives

One response header decides whether a browser, a phone app, or a CDN edge node is allowed to keep a copy of your response and hand it out again without asking you. That header is Cache-Control, and its directives are the vocabulary for saying exactly how long, to whom, and under what conditions.

Start with a running example

Say your app serves a fingerprinted JS bundle and a personalized cart total. They need opposite caching behavior, and you say so with two very different headers:

# A fingerprinted static asset — the URL itself changes on every
# deploy, so this exact response body is genuinely permanent.
GET /static/app.a3f9c2.js HTTP/1.1

HTTP/1.1 200 OK
Cache-Control: public, max-age=31536000, immutable

# A personalized API response — must never be cached at all.
GET /api/cart HTTP/1.1

HTTP/1.1 200 OK
Cache-Control: private, no-store

Neither line is decoration. The first tells every cache in the path — browser, corporate proxy, CDN — "keep this for a year, don't even ask me again." The second tells them "don't keep this anywhere, for anyone, ever." Everything else in Cache-Control is a variation between those two poles.

The directive vocabulary

Cache-Control is a comma-separated list of directives. Some only make sense on a response (from your server), a few only on a request (from the browser), and they combine — you'll usually see two or three together, not one.

DirectiveWhereMeans
max-age=NresponseFresh for N seconds after generation. The single most important directive.
s-maxage=NresponseSame idea, but only shared caches (CDNs, proxies) obey it — overrides max-age for them, ignored by browsers.
no-cacheresponseMay be stored, but must be revalidated with the origin before every reuse. Not "don't cache."
no-storeresponseMust not be stored anywhere, by anyone. The real "don't cache."
privateresponseOnly a single-user cache (the browser) may store it — shared caches (CDNs) must not.
publicresponseMay be stored by shared caches even if the response would normally look non-cacheable (e.g. it has an Authorization header).
must-revalidateresponseOnce stale, it must be revalidated before reuse — a cache may not serve it stale even to survive an outage.
proxy-revalidateresponsemust-revalidate, but binding only on shared caches.
immutableresponseThe body will never change while it's fresh — skip revalidation entirely, even on a user-triggered refresh.
stale-while-revalidate=NresponseServe the stale copy immediately for up to N more seconds while refetching in the background.
stale-if-error=NresponseIf the origin errors (5xx) on revalidation, keep serving the stale copy for up to N seconds instead of failing.
no-transformbothIntermediaries (e.g. a carrier's image-compressing proxy) must not modify the body.
⚠️
The trap everyone falls into

no-cache does not mean "don't cache." It means "cache it, but check with me first every time." If you actually want a response to never be stored — a bank balance, a one-time download link — you want no-store. Reach for no-cache when you want the round trip but not the full payload (see Chapter 2's 304 flow); reach for no-store when the answer must never sit on a disk anywhere.

The freshness lifetime, visually

A cached response has an age — how long ago it was generated — and a freshness lifetime, set by max-age. While age < max-age the cache may serve it with zero network calls. Past that boundary it's stale, and (unless told otherwise) the cache must check back in before reusing it.

private / public and who is asking

A request rarely passes through just one cache. A typical path is browser cache → corporate/ISP proxy → CDN edge → your origin. private and public control which of those are allowed to keep a copy:

private

Only the end-user's own browser may store it. Shared caches (CDN, corporate proxy) must not. Use this for anything tied to one logged-in user — a profile page, a cart, a dashboard.

forgetting it

Omit private on a personalized response and a shared cache in front of your app can legally store it — and serve user A's page to user B. This is a real, recurring class of data leak, not a theoretical one.

public is the opposite signal: it tells a shared cache "you may store this even though it looks non-cacheable" — for example, a response that carries an Authorization header, which caches otherwise treat as personal by default. Static assets served under a CDN nearly always combine public with a long max-age.

immutable: skip revalidation on purpose

Even with a year-long max-age, pressing a browser's reload button traditionally triggers a revalidation request anyway — the user asked for a refresh, so the browser plays it safe. immutable overrides exactly that case: it tells the browser "this body will not change while it's fresh, full stop, don't bother checking even on reload." It only makes sense paired with a fingerprinted URL (a content hash or build id in the path), because the only way that file's content legitimately changes is for the URL to change with it.

🛠️
Production trade-off

Cache-Control: public, max-age=31536000, immutable is the standard header for a fingerprinted static asset — but it only works if your build pipeline actually changes the filename on every change. Ship the same filename with different bytes under this header and clients keep the stale version for up to a year, with no way to force a refresh short of a URL change.

What we've got so far

Cache-Control answers who may cache a response and for how long. What it doesn't answer is what happens the moment a stored copy goes stale and a client checks back in — that handshake is ETag and conditional GET, next.
