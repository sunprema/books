Chapter 03Caching in Practice

You now have two independent tools — freshness (Cache-Control) and revalidation (ETag). The practical question for any given endpoint isn't "which header do I add," it's "which family of caching does this response belong to." There are really only two families, and almost every header combo you'll write is one of them.

Two families: expiration and validation

Expiration caching

max-age (+ immutable for the extreme case). The cache trusts the timer — zero requests reach your server until it expires. Fastest possible option, but the server has no way to push an early update.

Validation caching

no-cache + ETag/Last-Modified. Every use makes a round trip, but the response is usually a cheap 304 instead of the full body. Freshness is always exactly correct.

Everything else — no-store, private/public, s-maxage — is a modifier on top of one of these two, answering who may cache and where, not how.

A decision guide

Worked example: one small app, three endpoints

# 1. Fingerprinted bundle — content-addressed URL, never changes underneath itself
GET /static/app.a3f9c2.js
Cache-Control: public, max-age=31536000, immutable

# 2. The HTML shell — same URL forever, but its content changes on every deploy
GET /index.html
Cache-Control: no-cache
ETag: "build-4821"

# 3. A logged-in user's cart — must never leave their own browser
GET /api/cart
Cache-Control: private, no-store

Notice endpoint 2: the HTML shell can't be immutable — its bytes change on every deploy while its URL stays the same — so it falls back to validation caching. This is the standard pattern for the classic "index.html + hashed assets" deploy: the entry document is always revalidated (cheaply, via ETag), and it points at asset URLs that change name every time their content does, so those can be cached forever.

Layering the CDN in

s-maxage lets you give the edge a different freshness window than the browser gets — commonly shorter, so you can push an update by purging one CDN layer without waiting out a long browser-side max-age:

Cache-Control: public, max-age=60, s-maxage=600
# browser: revalidate after 60s   CDN edge: revalidate after 600s
🛠️
Production trade-off

stale-while-revalidate is the best of both worlds — instant response, background refresh — but browser support for it on the client is inconsistent, and it's most reliable when a CDN or a service worker is the one honoring it, not the end browser directly. Treat it as an edge/CDN optimization first, a browser guarantee second.

✅
The whole book, compressed

Cacheable-forever content gets a fingerprinted URL and immutable. Everything else that's safe to share gets no-cache + a validator, so a stale check costs headers, not bytes. Anything tied to one user gets private, and anything that must never be written to disk anywhere gets no-store.
