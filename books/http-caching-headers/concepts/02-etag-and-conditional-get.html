Chapter 02ETag and Conditional GET

Chapter 1 ended on a cliffhanger: what happens when a cached response goes stale and the client checks back in? If the answer were "re-download the whole thing," no-cache would be nearly useless. The answer is a conditional GET — a request that says "send me the body only if it's actually different" — and ETag is how the server names "different."

The exchange, end to end

First visit: the server returns the body and tags it with an ETag — an opaque identifier for this exact representation of the resource.

GET /api/products/42 HTTP/1.1

HTTP/1.1 200 OK
Cache-Control: no-cache
ETag: "33a64df551425fcc55e4d42a148795d9f25f89d"

{"id": 42, "name": "Mechanical Keyboard", "price": 89.00}

The cache stores the body and the ETag. Next time it needs this resource — because no-cache said "always check" — it echoes the tag back in If-None-Match instead of asking for the resource cold:

GET /api/products/42 HTTP/1.1
If-None-Match: "33a64df551425fcc55e4d42a148795d9f25f89d"

HTTP/1.1 304 Not Modified
Cache-Control: no-cache
ETag: "33a64df551425fcc55e4d42a148795d9f25f89d"
# no body at all — the cache reuses what it already has

If the price had changed, the server would compute a different ETag, the comparison would fail, and it would answer with a normal 200 and the new body. The entire point of the dance: the response headers are cheap, and a 304 carries no body — for a large resource that's checked often, this is the difference between shipping kilobytes and shipping a couple hundred bytes of headers.

Strong vs. weak validators

An ETag value is an opaque double-quoted string — the spec doesn't mandate how you compute it (a content hash, a version/revision number, a hash of the last-modified timestamp are all common). What it does define is two flavors:

FormMeansUse it for
ETag: "abc123"Strong — byte-for-byte identicalRange requests, exact-match caches
ETag: W/"abc123"Weak — semantically equivalent, bytes may differContent that varies trivially (gzip vs. brotli of the same page, whitespace-only template changes)

Weak ETags exist because "identical" is often too strict a bar: a gzip-compressed and an uncompressed response to the same Accept-Encoding-varied request are the same resource but not the same bytes. If-None-Match on a plain GET always uses the weak comparison — the W/ prefix is stripped before comparing — so ordinary revalidation doesn't care which kind you sent. Byte-range requests (If-Range) are the one place strong validators matter, because splicing two different bodies together silently corrupts the response.

Last-Modified: the older, coarser cousin

Before ETag, HTTP had — and still supports — timestamp-based validation:

Last-Modified: Wed, 21 Oct 2015 07:28:00 GMT

# next request:
If-Modified-Since: Wed, 21 Oct 2015 07:28:00 GMT
# → 304 if unchanged since that instant, 200 + fresh body otherwise

It works the same way, but at one-second resolution — two edits inside the same second are indistinguishable to it. When a server sends both headers, If-None-Match takes precedence if the client sent it; If-Modified-Since is the fallback for resources cheap enough that hashing the body isn't worth it (static files served straight off disk, where the filesystem's mtime is already sitting right there).

⚠️
The gotcha that bites at scale

If your ETag is derived from something node-local — an inode number, a process-local counter, a timestamp with clock skew across instances — then the same logical content gets a different ETag from each backend behind your load balancer. Every request round-robins to a "different" resource as far as If-None-Match is concerned, every comparison fails, and you get a 200 full body on every request instead of the 304 you expected — a completely silent cache-effectiveness bug. Derive the tag from the content itself (a hash) or a value that's identical across every node serving it.

Production notes

Most frameworks generate ETags for you — Express's etag middleware, nginx's default weak ETag from mtime+size, ASP.NET Core's static file middleware. That's usually the right default for static files. For API responses you compute yourself, hash the serialized body (or a version/updated-at column already in your database) rather than re-deriving it from scratch on every request — the whole point is to make the 304 path cheaper than the 200 path, and an expensive ETag computation undermines that.
