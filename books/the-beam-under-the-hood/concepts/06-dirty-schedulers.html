Chapter 06 · When C Won't Behave

Dirty Schedulers & NIFs

A separate room for the guests who won't take turns.

Back in Chapter 02 we made a promise: no process can hog a core, because every process is counted and preempted at 4000 reductions. That promise holds beautifully — for Erlang code. But sometimes you need to call out to C: a cryptographic routine, an image codec, a number-cruncher that would be too slow in the VM. That native function is a NIF, and a NIF can quietly break the promise. This chapter is about how, and about the clever room the BEAM built to contain the damage.

Why a NIF is dangerous

Remember how preemption works: the scheduler runs a process until its reduction budget runs out, but the budget only ticks down at safe points between function calls. A NIF isn't Erlang — it's a lump of C running directly on the scheduler thread. While it runs, no reductions tick, no safe point arrives, and the scheduler simply cannot take the core back. If the NIF returns in a few microseconds, fine, nobody notices. If it runs for 50 milliseconds, that scheduler — that whole clerk — is frozen for 50 milliseconds, and every process in its run queue waits. The soft real-time guarantee, gone, for one rude guest.

The rule of thumb

A NIF that runs on a normal scheduler should return in under about a millisecond. Longer than that and you're stealing time from every process queued behind you. This isn't a style preference; it's the load-bearing assumption the whole scheduler rests on.

Two ways to keep the promise

So what do you do when the native work honestly takes longer than a millisecond? There are two good answers, and it's worth knowing both.

1. Yield cooperatively

The polite NIF does the scheduler's job for it: it does a slice of work, tells the VM how much of its timeslice it used with enif_consume_timeslice, and when it's used enough it stops early and asks to be rescheduled with enif_schedule_nif, picking up where it left off next turn. In effect the C code chops itself into millisecond-sized pieces and voluntarily yields the core between them. It's more work to write, but it keeps the native call a well-behaved citizen of the normal schedulers.

2. Send it to a dirty scheduler

The easier answer, and usually the right one: run the long native call on a dirty scheduler — a separate pool of OS threads that exists precisely so that long or blocking native work can run without tying up the normal schedulers. You flag the NIF as dirty, and the VM routes it to the dirty pool. Your sixteen normal clerks keep serving Erlang processes at full speed and low latency; the heavy C call grinds away on a dirty thread off to the side. When it finishes, its result flows back as if nothing unusual happened.

The two dirty pools, and their sizes

There are two kinds, because "slow" comes in two flavours. Dirty-CPU schedulers are for work that's busy — it wants a core and will use it (that crypto routine, that codec). By default there are as many dirty-CPU schedulers as normal ones, so heavy CPU work can't monopolise the box. Dirty-I/O schedulers are for work that mostly waits — a blocking file read, a slow syscall — where you can afford many more threads because they're idle most of the time; the default is 10. You can tune both at boot with +SDcpu and +SDio.

A little history

Dirty schedulers arrived experimental in OTP 17 and became a default, always-there feature in OTP 20 — so on any modern system they're simply present. Before them, a long NIF was a genuine footgun with no safety net; today the net is built in, and much of the ecosystem's native code (crypto, compression, JSON parsers) uses it for you without your ever thinking about it.

Not a magic "make it fast"

A dirty scheduler removes the latency hazard — it stops one slow call from freezing your normal clerks. It does not make the work cheap. Heavy native code still burns real cores, and dirty-CPU schedulers still compete with everything else for the machine. And do match the pool to the job: a CPU-bound task dumped on a dirty-I/O scheduler (or a blocking one on dirty-CPU) uses the wrong room.

So what did the dirty pool buy us? A way to invite badly-behaved guests — long, blocking, native — into the house without letting them ruin the party for everyone else. It's the same instinct as the whole BEAM: don't trust a component to behave, contain it so its misbehaviour stays local. Isolate the crash, isolate the heap, and now, isolate the slow C call. Same idea, three floors down.

Try it yourself

  1. Check your dirty scheduler counts: :erlang.system_info(:dirty_cpu_schedulers) and :dirty_io_schedulers.
  2. Find a NIF-backed library you use (crypto, jason's native paths, an image lib) and read whether it marks calls dirty. Notice how much native code you already lean on.
  3. Read about enif_schedule_nif in the erl_nif docs and sketch, in words, how you'd chop a 100 ms C loop into yielding slices.
