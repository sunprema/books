Chapter 07 · Across the Wire

Distribution: Nodes Talking to Nodes

The same letters, now posted between buildings — and the network is not reliable.

Everything so far has happened inside one machine — one BEAM, many processes. But the same idea stretches, rather magically, across machines. Start Erlang on two computers, connect them, and a process here can send a message to a process there with exactly the same send it would use locally. Distribution is the BEAM taking its message-passing model and running it over the network. It's a lovely trick — and, being honest, it's the place where the tidy in-memory picture meets the messy real world, so we'll spend equal time on what it gives you and where the edges are.

Nodes, names, and a shared secret

A running BEAM that can talk to others is a node, and it has a name like app@host. Nodes find each other's ports through a little helper called epmd, the Erlang Port Mapper Daemon, which runs on each host and answers the question "what TCP port is the node named app listening on?" To connect at all, two nodes must share the same secret — the magic cookie, an atom kept in ~/.erlang.cookie. Same cookie, they may talk; different cookie, they're refused. It's a coarse, all-or-nothing trust, so keep that cookie private.

The one idea

Distributed send is location-transparent. Given a pid, send(pid, msg) works the same whether that pid lives in your VM or on a node across the room — the BEAM copies the message and ships it over a TCP link for you. The syntax and the semantics are identical to a local send. What is not identical is the failure and latency profile — and forgetting that is how distributed systems break hearts.

The full mesh

By default, distributed Erlang builds a fully connected mesh. The moment your node connects to one visible node, it also connects to every node that one already knows about, until everybody has a direct link to everybody. For a small cluster this is delightful: any node can reach any other in one hop, no routing to think about. But look at the shape of it — with N nodes you have roughly N² connections, each with its heartbeats and buffers. That cost grows fast.

How a node knows another has died

Links across a network need a heartbeat, because a silent peer might be busy, or might be gone. So net_kernel sends periodic ticks between connected nodes. If the ticks stop arriving, the node is declared down. The interval is net_ticktime, defaulting to 60 seconds, and detection lands somewhere within roughly a quarter of that either way. Your code can subscribe to these life-and-death events with net_kernel:monitor_nodes/1 and receive nodeup / nodedown messages — the same "tell me when it dies" pattern you use for processes, lifted up to whole machines.

Sixty seconds is a long time

That default 60-second tick means a crashed node can look alive for up to a minute. For many systems that's fine; for a chat or trading system it's an eternity. You can shorten net_ticktime, but tighten it too far and a momentary network hiccup will falsely declare a healthy node dead, triggering needless failover. Choosing this number is the design of your failure detector — there's no free lunch, only a dial between "slow to notice" and "jumpy."

Where the model strains

There's one connection per pair of nodes, and historically a single big message could clog it — a 100 MB term in flight would make small, urgent messages wait behind it (head-of-line blocking). OTP 22 softened this with message fragmentation: large terms are chopped into chunks and interleaved, so a little message can slip between the pieces of a big one. But note what didn't change — it's still one connection per pair. There is no built-in "many parallel links per node pair," even in recent OTP. When people genuinely need to go past the full mesh — hundreds or thousands of nodes — they reach outside core distribution for it (libraries like Partisan, or purpose-built topologies), rather than expecting the mesh to stretch that far.

So what did distribution buy us? The whole programming model — spawn, send, monitor, let it crash — extended across machines with the syntax unchanged. That transparency is a gift and a trap in one. A gift, because you already know the API. A trap, because the network is not reliable, and a send that always succeeded in memory can now vanish, arrive late, or arrive after the node you sent it to is already gone. The BEAM hands you the same tools; it's on you to remember you're now using them over something that lies.

Try it yourself

  1. Start two named nodes (iex --sname a, iex --sname b), connect with Node.connect/1, and Node.spawn/2 a process on the other. Same send, different machine.
  2. Subscribe with :net_kernel.monitor_nodes(true), kill one node, and watch the :nodedown arrive. Time how long it takes with the default tick.
  3. Set net_ticktime low on both nodes, then pause one with Ctrl-Z for a few seconds. Watch a healthy node get wrongly declared down — and feel the tradeoff in the dial.
