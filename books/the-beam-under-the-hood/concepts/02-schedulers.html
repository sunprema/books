Chapter 02 · Fairness

Schedulers, Reductions & Preemption

A few hands, a great many jobs, and a rule that keeps it fair.

In the last chapter we cheerfully made hundreds of thousands of processes. Now comes the obvious question, the one a sensible person asks straight away: you have four cores, maybe sixteen — how on earth do you run three hundred thousand things on sixteen cores fairly, so that no single greedy process can freeze all the others? The answer is one of the loveliest bits of engineering in the whole machine, and it is almost embarrassingly simple once you see it.

One clerk per core

The BEAM starts, by default, one scheduler — think of it as a worker, a clerk — for each logical core on your machine. Four cores, four clerks. Each clerk keeps its own queue of processes that are ready to run, its run queue. (That queue is really several queues stacked by priority — max, high, normal, low — but hold that thought; normal is where nearly everything lives.) A clerk picks the process at the front of its queue, runs it for a little while, then puts it to the back and picks the next. Round and round. That is the whole loop.

The one idea

The trick is the phrase "for a little while." An operating system interrupts a thread with a hardware timer — it can yank the CPU away at any instant. The BEAM does something different and, for its purposes, better: it counts. Every process is given a small budget of work, and when the budget runs out the process is politely set aside so the next one gets a turn. This is reduction counting, and it is how the BEAM preempts without a timer interrupt.

What on earth is a reduction?

A reduction is a unit of work. To a first approximation, one function call costs one reduction. Call a function, spend a reduction. Do it again, spend another. When a process has spent its allowance, the scheduler stops it exactly where it is — mid-loop, between calls — remembers the spot, and moves on to the next process in the queue. Later it comes back and carries on as if nothing happened. The process never knows it was interrupted.

How big is the allowance? 4000 reductions per turn, on any modern Erlang/OTP (it was 2000 before OTP 20, so you'll still see the old number quoted in older writing). Roughly four thousand function calls, then your turn is over and somebody else runs. Sends cost reductions too, and built-in functions cost a variable amount depending on how much work they really do — a big lists:reverse on a long list honestly charges you for the length. The accounting isn't perfect, but it doesn't need to be. It needs to be fair enough that no process runs forever, and that it is.

Why count instead of using a timer?

Because counting is portable, cheap, and precise about the thing we care about. A hardware timer interrupt can fall in the middle of anything, including a moment when the machine is halfway through updating its own bookkeeping — messy to handle safely. Reductions, by contrast, are only ever counted at safe points, between function calls, where the process is in a tidy state. So the BEAM can stop a process, garbage-collect it, move it to another core, all without fear, because it only ever stops at a clean boundary. The soft real-time feel you get from Erlang — the way latency stays low and even under load — grows directly out of this: everyone gets a small turn, very often, and nobody waits long for a core.

Sharing the load

What if one clerk's queue is stacked to the ceiling while another is idle? The idle scheduler simply steals work — it takes processes from a busier queue and runs them itself. The BEAM also recomputes migration paths periodically to keep the queues roughly even. You get multicore load-balancing for free, and you never wrote a line of it.

Where it goes wrong: the process that won't yield

Here is the honest weakness, and it matters because you will meet it. The whole scheme rests on processes bumping reductions — calling functions, coming back to safe points. Ordinary Elixir does this constantly, so it's a non-issue. But drop down into native C code — a NIF, a natively implemented function (Chapter 06) — and that C code runs on the scheduler thread without bumping reductions. If it runs for a long time, it never hits a safe point, the scheduler can't take the core back, and that clerk is stuck. The rule of thumb is blunt: a NIF should return in under a millisecond, or it doesn't belong on a normal scheduler.

Two failures people muddle

A single scheduler blocked by a rude NIF is one problem. Scheduler collapse is a different, subtler one: under bursty, low load, schedulers that go to sleep can fail to wake up promptly, so work piles onto the few that are awake and your other cores sit idle while latency climbs. It's tuned with busy-wait flags — +sbwt, +sbwtdcpu, +sbwtdio — which tell a scheduler how long to spin looking for work before sleeping. Both problems come from work that doesn't yield, but they are not the same beast. Reach for the right one when you diagnose.

So what did the counting buy us? Fairness we didn't have to design, on hardware we didn't have to know about. A million processes, sixteen cores, and a guarantee that the chatty ones can't starve the quiet ones — all from the humble act of keeping a tally and calling "next" when it runs out. The price is that native code must play along, and when it won't, we have a whole separate room to put it in. That room is Chapter 06.

Try it yourself

  1. Run :erlang.system_info(:schedulers_online). Does it match your core count? Try starting the VM with +S 2 and watch it change.
  2. In :observer (the Load Charts tab), watch per-scheduler utilisation while you spin up a busy workload. See the work spread across clerks.
  3. Spawn one process running a tight Enum.reduce over a huge range, and another printing a heartbeat every 100 ms. Confirm the heartbeat stays steady — the busy one gets preempted at 4000 reductions and can't starve it.
