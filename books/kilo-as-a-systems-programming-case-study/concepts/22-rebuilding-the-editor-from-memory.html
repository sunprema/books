Chapter 22 · Part VII — Beyond the Tutorial

Rebuilding the Editor from Memory

Twenty-one chapters have taken kilo.c apart, one function group at a time. This one closes the book — literally, for a minute. The exercise is simple to state: without re-opening any earlier chapter, sketch the whole architecture from memory. Not every line. Just the shape: what struct runs the show, what loop drives it, what the eight modules are, and which invariants keep them from stepping on each other. Then check yourself against what's actually here.

Motivation: rebuilding beats rereading

Rereading Chapters 1–21 feels like progress, because the material is instantly familiar. That familiarity is mostly an illusion — recognizing a sentence isn't the same skill as producing the idea it describes when the page isn't in front of you. Over a century of research on what's called the testing effect backs this up directly: recalling material from memory, not just re-studying it, is what actually strengthens retention, inference, and the ability to transfer it to a new problem — and the effect holds for closed-book reconstruction, not just formal quizzes. There's even a second-order payoff: retrieving old material well changes how the next new thing you learn gets encoded in the first place. That's the whole rationale for this chapter existing as an exercise instead of a summary.

Kilo was built this way too

Chapter 1 established that antirez's own README says kilo.c was "written in just a few hours taking code from my other two projects, load81 and linenoise." That's not a lookup-and-assemble process — it's building from an internalized model of terminal handling he already carried in his head. Rebuilding this book's version of that model, from memory, is the same move in reverse.

Design: one architecture, eight parts, one page

Before checking anything, try to fill this table in yourself — module, job, one invariant it depends on — then compare. Chapter 19 counted 34 top-level functions in kilo.c's 1308 lines and found they sort into exactly these eight groups, in dependency order:

ModuleJobChapter(s)Invariant it owns
terminalenter/exit raw mode, read one keypress3–4, 17 orig_termios is the only copy that can restore the user's shell
rowown the erow array, chars/render split8 render is always rebuilt from chars, never the reverse
highlightfill erow.hl from render14–16 hl is parallel to render, one byte per rendered character
editinsert/delete chars, split/merge rows10 every mutation ends by calling editorUpdateRow
fileload and persist the buffer11–12 save writes chars only, never render
screenassemble one frame, one write()5–6 nothing reaches the terminal until the append buffer is complete
searchincremental find, cancel restores state13 saved_cx/cy/rowoff/coloff are captured before the query loop starts
inputdispatch a keypress to everything above4, 9–10 E.cx/E.cy are screen-relative; filerow/filecol are recomputed, not stored

Every one of those eight rows reads or writes the same single instance: static struct editorConfig E (Chapter 7). That's the one fact that makes the whole table cohere — there's no handle, no context pointer, just one global every module touches directly. If your own sketch didn't have a box for E sitting in the middle, that's the gap worth noticing first.

The rebuild: start from the loop, radiate outward

The fastest way back to the whole architecture is the same one Chapter 6 used to introduce it: main()'s steady state is two calls, forever.

kilo.c — main(), steady state 
while (1) {
    editorRefreshScreen();
    editorProcessKeypress(STDIN_FILENO);
}

Refresh runs first on every pass, so each iteration paints whatever the previous keypress produced before it blocks on the next one. From there the reconstruction is just following what each call touches:

editorProcessKeypress calls editorReadKey, which blocks on read() against a file descriptor that enableRawMode (Chapter 4) put into raw mode — canonical mode and echo off, so bytes arrive one at a time with no line buffering and nothing echoed back automatically. The keypress lands in a twelve-case switch that either moves the cursor (editorMoveCursor, reading and writing E.cx/E.cy/E.rowoff/E.coloff), mutates text (editorInsertChar/editorDelChar/editorInsertNewline, each ending in editorUpdateRow so render and hl stay in sync with chars), opens a mode (editorFind, which saves four cursor/scroll fields up front so Escape can restore them exactly), or touches the file (editorSave, which flattens every row to one string and writes it with ftruncate doing the work a missing O_TRUNC doesn't). editorRefreshScreen then reads that same state back — E.row[], E.cx/E.cy, E.rowoff/E.coloff — and assembles one append buffer before a single byte reaches the terminal.

Self-test before you read the next section

Close your eyes on the previous paragraph and answer these without looking back: (1) Is Kilo's text buffer a gap buffer, a rope, or something simpler — and what is it? (2) What does E.cx actually measure — a position in the file, or a position on the screen? (3) Why does editorSave need ftruncate at all? If any of those three come back fuzzy, that's exactly the piece worth re-reading, not the whole chapter it came from.

Edge cases: what a rebuild-from-memory tends to lose first

Four specific details are the ones a from-memory sketch of this architecture most often gets wrong, each one a case where the obvious-sounding guess is the incorrect one.

Easy guessWhat's actually trueChapter
Kilo uses a gap buffer or rope for text a plain, dynamically-grown array of erow structs — O(n) realloc per inserted line8
E.cx/E.cy are the file position they're screen-relative; filerow/filecol are recomputed from them plus the scroll offsets, every time9
ftruncate in editorSave is just defensive it's load-bearing — the open() flags omit O_TRUNC, so without it a shorter save leaves old trailing bytes behind 12
There's a dedicated editorScroll() function there isn't one — scrolling is folded directly into each arrow-key branch of editorMoveCursor9

So what did we actually gain?

Not new material — every fact in this chapter was established somewhere in the previous twenty-one. What changes is whether it's yours: one static struct editorConfig E that every one of 34 functions across eight modules reads or writes directly; one steady-state loop, refresh then read, that drives all of it; one chars/render/hl triple per row that keeps what's on disk, what's on screen, and how it's colored from ever tangling into each other. 1308 lines, no libraries, no header files — and by now, ideally, no notes either. That's the actual test this book has been building toward: not whether you can find the answer in a chapter, but whether you can reconstruct the design without one. The same move works on the next unfamiliar codebase you read that isn't this one.

Before you close the book
  1. On a blank page, draw the eight-module table from this chapter without looking at it — module, job, one invariant — then check it against the one above and circle whatever you missed.
  2. Pick one keystroke — say, pressing Backspace at the start of a line — and narrate, from memory, every function it passes through from editorReadKey to the next repainted frame.
  3. Without opening Chapter 19, list the three refactors that chapter proposed and, for each, say what it would cost and what it would buy. Then open Chapter 19 and see what you left out.
