Chapter 03 · Part I — Orientation

C and Terminal Prerequisites

Chapter 02 gave us the five jobs every editor does. Before we watch Kilo do them, we need a shared vocabulary for how it does them — a handful of C idioms that show up on nearly every page of kilo.c, and a handful of facts about what a terminal actually is, underneath the blinking cursor. This isn't a C course. It's the specific, reusable toolbox one very disciplined systems programmer reached for, over and over, to keep a real program under a thousand lines.

What this chapter is, and isn't

It is not a general introduction to C, and it won't teach you the language from zero. It is a fast, concrete tour of the specific patterns Kilo leans on repeatedly — one global struct, pointers with hand-tracked lengths, growing arrays with realloc(), NULL-terminated arrays of strings, bitflags, variadic functions, and C's plain, exception-free way of failing — plus the terminal facts (file descriptors, canonical vs. raw mode, what an escape sequence even is) that Chapter 04 assumes you already have. If a term below is new, it will not be new again.

Motivation: name the tools before you watch someone use them

Reading someone else's C is a different skill from writing your own. The syntax is familiar, but the idioms — the small, repeated decisions an experienced systems programmer makes without comment — are not always obvious on a first pass. Kilo is short, but it is not simplified for teaching; it is real code, written the way antirez writes real code. If you don't already recognize “grow this array by one and reassign the pointer” or “this struct is one giant global and everything touches it” as patterns, you'll spend Chapter 04 decoding syntax instead of learning raw mode. So we decode the syntax now, once, here.

The prerequisites split cleanly into two halves, and we'll take them in order: first the C idioms, which apply to the whole file; then the terminal facts, which apply specifically to the input/output layer we're about to build in Chapters 04 and 05.

One global struct, not a hundred parameters

Open kilo.c and search for struct editorConfig. You'll find it defined once, near the top, and then a single line right after it: static struct editorConfig E;. That's the entire state of the editor — cursor position, scroll offsets, screen size, the array of rows, the dirty flag, the open filename, the status message, the current syntax — as one global variable, declared static so it's private to this file.[1] Nearly every function in Kilo reads or writes a field of E directly, instead of receiving state through parameters and returning it back out.

That's a real design decision, not laziness, and it's worth naming the trade-off honestly now because we'll live with it for the rest of the book. A single global struct means no function signature ever has to grow just to thread one more piece of state through three layers of calls — editorMoveCursor() just reaches into E.cx and E.cy directly. The cost is that every function is implicitly coupled to all of the editor's state, which only stays manageable because the program is small enough to hold in your head — which was the entire premise of Chapter 01. We'll meet this struct properly, field by field, in Chapter 07 (Editor State); for now, just recognize the pattern when you see bare E.something appear with no argument list in sight.

Pointers, strings, and the length you track yourself

A C string is a convention, not a type: a char * pointing at the first byte, with a '\0' byte marking the end. C itself enforces none of this — strlen() will happily walk off the end of a buffer that forgot its terminator. Kilo's answer is to not lean on strlen() for anything that matters: every buffer that holds text also carries an explicit length field right next to the pointer — erow.size and erow.rsize sit beside erow.chars and erow.render; struct abuf (the screen-write append buffer we met briefly in Chapter 02) pairs char *b with an explicit int len.[1] Copies then use memcpy() or memmove(), which take a length argument and are binary-safe — they'll copy embedded zero bytes correctly — rather than strcpy(), which stops at the first '\0' it finds.

Keep that pairing in mind as a reading habit: whenever you see a char * field in a Kilo struct, look for the sibling length field right next to it. It's almost always there, and it's almost always what the code actually trusts.

Dynamic memory: grow by one, reassign the pointer

Kilo never pre-allocates a big array and hopes it's big enough. It grows arrays one element at a time with realloc(), and — this is the part worth memorizing — always reassigns the result back into the owning pointer, because realloc() is free to return a different address than the one it was given (it may have had to move the whole block to find room):

void abAppend(struct abuf *ab, const char *s, int len) {
    char *new = realloc(ab->b,ab->len+len);
    if (new == NULL) return;
    memcpy(new+ab->len,s,len);
    ab->b = new;
    ab->len += len;
}

The row array grows the identical way, one line at a time, in editorInsertRow(): E.row = realloc(E.row, sizeof(erow)*(E.numrows+1)).[1] Same shape, same discipline, every time Kilo's data needs to get bigger. You'll type this exact pattern yourself starting in Chapter 08.

One more pointer idiom shows up wherever Kilo needs a variable-length list of strings rather than one string: a plain C array carries no length of its own, so the standard trick is to terminate the array with a NULL pointer and walk it until you hit one. Kilo's syntax-highlighting table uses exactly this for both a language's file extensions and its keyword list:

struct editorSyntax {
    char **filematch;
    char **keywords;
    char singleline_comment_start[2];
    char multiline_comment_start[3];
    char multiline_comment_end[3];
    int flags;
};

char **filematch is a pointer to an array of C strings — a list of patterns like ".c", ".h", NULL — and char **keywords is the same shape for a language's reserved words. We'll fill this exact struct in for real in Chapter 16 (File Types and Highlight Rules); for now, just recognize char ** as “a NULL-terminated list of strings,” not as anything more exotic.

Macros, enums, and bitflags

The very first non-comment lines of kilo.c are preprocessor directives, and they matter more than they look:

#define KILO_VERSION "0.0.1"

#ifdef __linux__
#define _POSIX_C_SOURCE 200809L
#endif

That _POSIX_C_SOURCE line is a feature-test macro: it tells glibc “expose the POSIX.1-2008 functions,” unlocking library calls like getline() that a strict-ISO-C build would otherwise hide. Skip it, and functions Kilo depends on for file loading simply wouldn't be declared. It's easy to gloss over as boilerplate; it's actually load-bearing.

Key handling leans on a plain C enum, and it's worth reading closely because the values are not arbitrary:

enum KEY_ACTION{
    KEY_NULL = 0,
    CTRL_C = 3,         /* Ctrl-c */
    CTRL_D = 4,         /* Ctrl-d */
    CTRL_F = 6,         /* Ctrl-f */
    CTRL_H = 8,         /* Ctrl-h */
    CTRL_L = 12,        /* Ctrl+l */
    CTRL_Q = 17,        /* Ctrl-q */
    CTRL_S = 19,        /* Ctrl-s */
    CTRL_U = 21,        /* Ctrl-u */
    ESC = 27,           /* Escape */
    BACKSPACE = 127,    /* Backspace */
    ARROW_LEFT = 1000,  /* …and more soft codes above 1000 */
};

Every one of those Ctrl-key numbers comes from the same rule: on an ASCII terminal, holding Ctrl while typing a letter clears bits 5 and 6 of that letter's ASCII code — which is exactly letter & 0x1F. The popular community walkthrough of this same program even spells that rule out as a macro, #define CTRL_KEY(k) ((k) & 0x1F) — but the actual kilo.c you'll be reading has no such macro; it just writes the already-computed numbers straight into the enum.[3] Do the arithmetic once and you'll never be surprised by one of these constants again:

KeyLetter (ASCII, binary)& 0x1FResult
Ctrl-Q'Q' = 0x51 = 0101 00010001 11110001 0001 = 17
Ctrl-S'S' = 0x53 = 0101 00110001 11110001 0011 = 19
Ctrl-F'F' = 0x46 = 0100 01100001 11110000 0110 = 6

Bitflags follow the same “small integer, big meaning” spirit: Kilo's syntax-highlight flags are declared as #define HL_HIGHLIGHT_STRINGS (1<<0) and #define HL_HIGHLIGHT_NUMBERS (1<<1) — each one a single bit, combinable with |, testable with &. We'll use exactly this trick again in a moment, on the terminal side.

One more idiom: variadic functions

You'll see calls like editorSetStatusMessage("%d lines written to disk", n) scattered through Kilo — printf-style calls to a function Kilo wrote itself. It works because the function is declared variadic, using the same <stdarg.h> machinery printf() itself is built on:

void editorSetStatusMessage(const char *fmt, ...) {
    va_list ap;
    va_start(ap,fmt);
    vsnprintf(E.statusmsg,sizeof(E.statusmsg),fmt,ap);
    va_end(ap);
    E.statusmsg_time = time(NULL);
}

va_start/va_end bracket access to the ... arguments, and vsnprintf() — the “v” means “takes a va_list” — formats them into E.statusmsg[80] safely, truncating instead of overflowing if the message runs long. Recognize this shape and the status-bar code in Chapter 06 will read as plain arithmetic.

Finally: C has no exceptions, and Kilo doesn't fake any. Functions that can fail return -1, often by jumping to a goto fatal: label that sets errno first; failures the editor truly can't recover from — a file that won't open, a terminal that won't report its size — call perror() (print the C library's text for the current errno) and then exit(1). There's no try/catch anywhere in this file, and there never will be — every error path is a return value or a process exit, checked (or knowingly not checked) right where it happens.[1]

The terminal side: what's actually underneath the cursor

Now the other half. A terminal — even a modern GPU-accelerated one like iTerm2 or Windows Terminal — still behaves, by contract, like the physical VT100 terminals of the 1970s: it's a device your program writes bytes to and reads bytes from, mediated by a piece of the operating system called the line discipline. Understanding that mediation is the whole key to Chapter 04.

File descriptors and raw syscalls

Kilo doesn't use printf() or getchar() for its interactive I/O. It talks to the terminal through the same two syscalls everything in POSIX ultimately reduces to: read() and write(), on plain integer file descriptors — STDIN_FILENO (0) for keyboard input, STDOUT_FILENO (1) for screen output. editorReadKey() is, at its core, one line: read(fd,&c,1) — read exactly one byte, blocking until it arrives.[1] There's no buffering layer, no line-editing library, no curses. Just syscalls.

Canonical mode vs. raw mode: the termios API

By default, your shell's terminal is in canonical mode. In canonical mode the kernel's line discipline buffers everything you type, handles backspace and line-editing itself, and only hands your program a complete line once you press Enter — which is why, right now, a plain C program calling read() would receive nothing at all until you hit Enter, and would receive a whole line at once, backspaces already applied.[4] That's exactly wrong for an editor: Kilo needs to react the instant you press an arrow key, not after you press Enter.

The fix is the POSIX termios API: tcgetattr() reads the terminal's current settings into a struct termios; tcsetattr() writes modified settings back. Turning off the ICANON flag switches to noncanonical (“raw”) mode, where read() can return individual bytes as they arrive rather than waiting for a whole line; turning off ECHO stops the terminal from printing back what you type (Kilo will decide for itself what to draw); and in noncanonical mode, two more fields — c_cc[VMIN] and c_cc[VTIME] — govern exactly how many bytes, or how long, read() waits before it returns.[4] Here's Kilo setting all of this at once:

raw = orig_termios;  /* modify the original mode */
raw.c_iflag &= ~(BRKINT | ICRNL | INPCK | ISTRIP | IXON);
raw.c_oflag &= ~(OPOST);
raw.c_cflag |= (CS8);
raw.c_lflag &= ~(ECHO | ICANON | IEXTEN | ISIG);
raw.c_cc[VMIN] = 0;   /* Return each byte, or zero for timeout. */
raw.c_cc[VTIME] = 1;  /* 100 ms timeout (unit is tens of second). */
if (tcsetattr(fd,TCSAFLUSH,&raw) < 0) goto fatal;

Notice the bitflag idiom from the C section reappears immediately: &= ~(...) clears a set of flags, |= (...) sets them — the exact same two operators, applied to the terminal's control struct instead of Kilo's own HL_* constants. We'll walk this function line by line, flag by flag, in Chapter 04; for now, just recognize that “raw mode” is not a mysterious terminal superpower — it's a handful of bits in a struct, turned off on purpose.

Asking the terminal how big it is

A full-screen editor needs to know its canvas size. Kilo asks the kernel directly with ioctl(1, TIOCGWINSZ, &ws) — a general-purpose “device, please tell me something about yourself” syscall, here requesting the terminal's row and column count. If that fails or reports zero columns (some terminal multiplexers don't support TIOCGWINSZ), Kilo falls back to a cruder trick: move the cursor to the bottom-right corner with a raw escape sequence, then ask the terminal to report back where the cursor landed, and infer the screen size from that position.[1] Two different ways to ask the same question — one a syscall, one a conversation conducted entirely in escape sequences.

A first look at escape sequences

That fallback is worth pausing on, because it previews all of Chapter 05. An escape sequence is nothing more than ordinary bytes — starting with the ESC character (0x1B) followed by '[' — written to the terminal with the same write() call you'd use for any other text. Kilo's C program has no direct access to “the screen”; it never draws a pixel and never calls a graphics API. It writes text. The terminal emulator — a separate program, watching everything written to it — recognizes the ESC-[ prefix, parses what follows as a command, and reacts: move the cursor, clear a line, change a color. Kilo and the terminal are two separate programs, cooperating entirely through a shared, decades-old text protocol.

So what did we actually get out of this?

A shared vocabulary, on both halves of the problem. On the C side: one global struct instead of parameter-passing, pointers paired with hand-tracked lengths, realloc()-and- reassign for growing arrays, NULL-terminated arrays of strings, macros and bitflags for compact state, variadic functions for printf-style helpers, and return-value-based error handling. On the terminal side: file descriptors and raw read()/write() instead of stdio, canonical mode as the kernel's default line-buffered behavior, the termios struct as the lever that turns it off, and escape sequences as plain text a cooperating terminal chooses to interpret specially. Every one of these ideas gets used for real, not just mentioned, starting in the next chapter.
