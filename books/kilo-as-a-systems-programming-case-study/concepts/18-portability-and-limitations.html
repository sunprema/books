Chapter 18 · Part VI — Robustness

Portability and Limitations

Chapter 17 closed on a promise: that judging whether a design choice is "fine" always depends on where the program runs and what happens if it's wrong. This chapter is where that judgment gets made explicit, all at once. "Portable" turns out to be two different questions wearing one word — will this program even compile and run on a given operating system, and, separately, will it render correctly on a given terminal once it's running — and kilo.c answers those two questions very differently. Then there's the list of things it simply doesn't do at all. Neither the assumptions nor the omissions are accidents. Read the source looking for them, and they turn out to be about a dozen small, deliberate lines.

Motivation: portable to where, exactly?

kilo.c's own README states the tradeoff it's making in three sentences, and it's worth reading literally rather than skimming past:

From the README, verbatim

"Kilo does not depend on any library (not even curses). It uses fairly standard VT100 (and similar terminals) escape sequences. The project is in alpha stage and was written in just a few hours."

Every clause there is a portability claim. "Does not depend on any library" means no ncurses to install, link, or version-match — drop kilo.c on almost any Unix-like box with a C compiler and it builds. "Fairly standard VT100 escape sequences" is the other half of the same coin, and it cuts the opposite way: those sequences are written straight into the source, not looked up from a database of what this particular terminal supports. And "alpha stage" is antirez naming the scope honestly — this was never meant to be a program that handles every environment it might be run in.

Design: a POSIX program, not a plain C program

Everything kilo.c does to the terminal — entering raw mode (Chapter 4), reading the window size, reacting to a resize — goes through POSIX, not through the C standard library. The #include list at the top of the file is the tell:

#define KILO_VERSION "0.0.1"

#ifdef __linux__
#define _POSIX_C_SOURCE 200809L
#endif

#include <termios.h>
/* ...stdlib.h, stdio.h, string.h, ctype.h, time.h... */
#include <sys/ioctl.h>
#include <sys/time.h>
#include <unistd.h>
#include <fcntl.h>
#include <signal.h>

termios.h is what raw mode is built on. sys/ioctl.h is what getWindowSize queries the terminal with. signal.h is what lets Chapter 17's handleSigWinCh hear about a resize at all. None of these three exist on Windows, and there is no conditional compilation anywhere in kilo.c that reaches for a different API when they're missing.

Not a missing shim — a missing terminal layer

Porting kilo.c to Windows isn't a matter of adding an #ifdef _WIN32 branch next to the POSIX calls. There is no Windows equivalent of termios and tcsetattr to drop in — the closest analog is the Win32 Console API (or, on modern Windows, ConPTY), a different model entirely for the same underlying idea of "control this terminal directly." ioctl(fd, TIOCGWINSZ, ...) becomes GetConsoleScreenBufferInfo. signal(SIGWINCH, ...) has no direct counterpart at all — console resize notifications work completely differently. A real port means writing a second implementation of the whole terminal-handling layer this book has spent four chapters on, not patching the existing one.

Look again at that first conditional, though, because it's not actually about Windows — it's a portability wrinkle within the POSIX world kilo.c already targets. _POSIX_C_SOURCE 200809L is defined only #ifdef __linux__, never on macOS or the BSDs. The reason is editorOpen's use of getline() (Chapter 11) — a POSIX.1-2008 function that glibc, Linux's C library, only declares when a program explicitly opts into that POSIX revision via a feature-test macro like this one. macOS's libc doesn't gate the same function behind that macro, so the guard would be a no-op there, and kilo.c simply skips it. One three-line conditional, and it exists purely because "POSIX-compliant" doesn't mean two libc implementations expose identical symbols under identical default settings.

Implementation: two ways to ask how big the terminal is

getWindowSize, first introduced in Chapter 17 for its exit path, is also kilo.c's clearest piece of defensive portability — it doesn't trust a single API to always answer the question:

int getWindowSize(int ifd, int ofd, int *rows, int *cols) {
    struct winsize ws;

    if (ioctl(1, TIOCGWINSZ, &ws) == -1 || ws.ws_col == 0) {
        /* ioctl() failed. Try to query the terminal itself. */
        int orig_row, orig_col, retval;

        retval = getCursorPosition(ifd,ofd,&orig_row,&orig_col);
        if (retval == -1) goto failed;

        /* Go to right/bottom margin, then read the cursor position back. */
        if (write(ofd,"\x1b[999C\x1b[999B",12) != 12) goto failed;
        retval = getCursorPosition(ifd,ofd,rows,cols);
        if (retval == -1) goto failed;
        /* ...restore the cursor to orig_row/orig_col, return 0... */
    }
    /* ...set *rows/*cols from ws, return 0... */
}

The primary path, ioctl(1, TIOCGWINSZ, &ws), asks the kernel directly and is the fast, normal answer on a real terminal device. The fallback doesn't touch the kernel at all: it writes \x1b[999C\x1b[999B, an escape sequence that shoves the cursor to column 999 and row 999 — clamped to the actual bottom-right corner by any VT100-compatible terminal — then asks where the cursor ended up with the \x1b[6n device-status-report sequence from Chapter 5. Whatever position comes back is the terminal's size. It's a clever piece of portability precisely because it doesn't depend on ioctl or TIOCGWINSZ at all — only on the same minimal VT100 escape vocabulary the rest of the editor already assumes.

A parameter the function doesn't actually use

Look at the signature again: getWindowSize(int ifd, int ofd, ...) takes an output file descriptor — but the ioctl call hardcodes the literal 1 instead of ofd. It works only because of a POSIX convention, not because the code enforces it: file descriptor 1 is standard output for any process that hasn't had it explicitly reassigned. ofd happens to always be STDOUT_FILENO (which is 1) at kilo.c's one call site, so the bug never bites — but it's a real gap between what the function signature promises and what the function body actually does.

Design: no ncurses, no terminfo, no fallback

Chapter 5 already showed how kilo.c draws the screen — literal escape sequences written straight to STDOUT_FILENO. This chapter is about what that choice costs. A library like ncurses ships a terminfo database describing hundreds of terminal types and negotiates, at startup, which specific byte sequences this terminal actually understands. kilo.c has none of that. It picks one dialect — the VT100-family sequences that xterm and virtually every modern terminal emulator happen to support — and emits exactly those, unconditionally, to whatever's on the other end of stdout.

That's a genuinely good trade for a ~1000-line teaching program run on a modern development machine: it works everywhere that matters, with zero setup, in a program small enough to read start to finish. But it means kilo.c has no graceful-degradation path. A terminal that doesn't understand a sequence it emits — a bare serial VT52, TERM=dumb, some minimal embedded console — doesn't get a simplified rendering. It gets garbage on screen, because there's no terminfo lookup standing between "here's what I want to draw" and "here's the exact byte sequence for the terminal in front of me" to catch the mismatch.

Limitations: bytes, not characters

Chapter 15 showed isprint() flagging non-printable bytes for the HL_NONPRINT highlight. Zoom out from that one check and a wider pattern shows up: kilo.c treats a file as a stream of bytes, full stop. There is no concept of a multi-byte character anywhere in the source. isprint() itself is C-locale-dependent and answers one byte at a time. Every place that measures how much horizontal space a character takes — the tab expansion in editorUpdateRow, the cursor's on-screen column, the length checks against E.screencols — assumes one byte is one column.

That assumption is invisible on an ASCII source file and wrong the moment it isn't one. A single accented letter or emoji encoded as UTF-8 is two to four bytes, each of which kilo.c will count and render as its own column. The result isn't a crash — it's a cursor that silently drifts out of sync with what's visually under it, one multi-byte character at a time, with nothing in the source flagging that anything went wrong.

Limitations: what Kilo leaves out on purpose

Beyond the terminal and character-set assumptions, a short, honest list of features simply isn't there. None of these are bugs — they're the visible edge of a program that stayed small on purpose:

kilo.c — features that don't exist 
Line wrap          none — long lines scroll via E.coloff instead (Ch. 9, 13)
Undo / redo         none — every edit is immediate and final
Clipboard           none — no code path touches an OS clipboard at all
Multiple buffers    none — one file per process, set once from argv[1]
Split windows       none — a single full-screen view, always
Mouse                none — editorProcessKeypress is keyboard-only
Config file          none — tab width, buffer sizes, quit-confirm count are
                     all compile-time #define constants (edit + recompile to change)
Language support     one — HLDB ships a single C/C++ entry (Ch. 16)

That "compile-time constants" row is worth naming precisely, since it's a limitation in its own right: the tab stop is hardcoded to 8 columns inside editorUpdateRow, the incremental-search query buffer is KILO_QUERY_LEN (256 bytes, Chapter 13), the status-message buffer is a fixed 80 bytes (Chapter 7), and the unsaved-changes quit confirmation is KILO_QUIT_TIMES (3 presses, Chapter 17). Every one of them is a number baked into the binary at compile time, not something a user or a config file can adjust at runtime.

Even the compiler isn't fully portable

handleSigWinCh's unused signal-number parameter is marked with __attribute__((unused)) — a GCC/Clang extension, not standard C, with no fallback for a compiler that doesn't recognize it. It's a small thing next to "no Windows support," but it's the same story at a different scale: kilo.c targets a specific, common toolchain rather than the broadest possible one, and does it without hiding the choice behind a portability macro.

So what did we actually gain?

A precise answer to "what does Kilo run on?" that splits cleanly into two halves. On the operating system axis, it's narrow and absolute: POSIX or nothing, no Windows path, one Linux-only feature-test macro papering over the one place even POSIX systems disagree with each other. On the terminal axis, it's wide but brittle: any VT100-family terminal works with zero setup, because there's no library to install — and any terminal that isn't VT100-family gets no fallback at all, because there's no terminfo database standing between the code and the screen. Layer on top of that a file model that only understands single bytes, not characters, and a feature list that stops well short of undo, clipboard, multiple buffers, or a config file. None of this is Kilo failing at portability. It's Kilo being honest about which portability it paid for — "compiles and runs anywhere POSIX-y with a common terminal, does the twelve things it does and nothing else" — and which it simply didn't, the same kind of scope discipline that kept it under 1,000 lines in the first place. The next three chapters pick that discipline apart on purpose: Chapter 19 asks what changes if you refactor this code for maintainability, and Chapter 20 asks what it costs to add back some of what's missing here.

Before you turn the page
  1. Sketch, in plain language, what would have to change to get raw-mode terminal control working on Windows. Name the specific POSIX call in kilo.c — tcgetattr, tcsetattr, ioctl with TIOCGWINSZ, signal with SIGWINCH — that each Windows replacement would need to stand in for, and say whether you think it's a drop-in swap or a genuinely different design.
  2. Trace through what happens if a line in the open file contains a single UTF-8 character encoded as three bytes (a common case for, say, a curly quote or a currency symbol). Walk through editorUpdateRow's size math and isprint() check for those three bytes, and describe exactly what the user would see happen to the cursor as they move past that character.
  3. Fix getWindowSize's hardcoded ioctl(1, ...) to use its own ofd parameter instead. Then describe a hypothetical caller (not the one in kilo.c today) for which the hardcoded version would silently query the wrong descriptor.
