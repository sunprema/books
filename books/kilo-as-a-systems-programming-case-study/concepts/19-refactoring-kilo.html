Chapter 19 · Part VII — Beyond the Tutorial

Refactoring Kilo

Chapter 18 ended with a promise: that after cataloguing what kilo.c assumes and what it leaves out on purpose, the next question is what changes if you refactor this code for maintainability. So let's actually do it — or rather, let's do the part that matters more than typing the new code: deciding, with evidence from the source itself, what a refactor of kilo.c would touch, what it would buy, and what it would cost. Every chapter so far has read kilo.c as it is. This one reads it as a starting point.

Motivation: refactor is not a synonym for rewrite

It's worth being precise about the word before we use it forty more times. Martin Fowler's Catalog of Refactorings defines refactoring as a disciplined sequence of small, behavior-preserving transformations — each one verified before the next is attempted. Not "restructure until it feels better." Not "rewrite from scratch with better judgment this time." The whole point is that the program does exactly the same thing before and after, and you can prove it at every step along the way.

The catch, for this specific program

"Verified before the next step" needs something to verify against. A direct check of the antirez/kilo repository for the usual places a test suite or CI config would live — .travis.yml, .github/workflows, tests/, test.c, CMakeLists.txt — turns up nothing. Every one of those paths 404s. So every refactor idea in this chapter carries an asterisk: without an automated way to confirm a keystroke still does what it did before, "behavior-preserving" is a promise you keep by careful manual re-checking, not one the toolchain checks for you. Chapter 21 exists to close exactly this gap — read this chapter as "here's what we'd change," and that one as "here's how we'd know it still works."

With that caveat named honestly, we can go looking for real candidates. And kilo.c, read as a whole rather than one function at a time the way the last eighteen chapters did it, turns out to have exactly two structural properties worth talking about: everything lives in one file, and almost everything shares one global. Let's take them one at a time.

Design: the module map that's already there

kilo.c's build system is about as simple as build systems get:

all: kilo

kilo: kilo.c
	$(CC) -o kilo kilo.c -Wall -W -pedantic -std=c99

clean:
	rm kilo

One file compiled as one translation unit, no headers, no dependency graph to reason about. That simplicity is real and it's worth naming before we start proposing to break it: there's nothing here to accidentally get wrong at build time. But "one file" and "no internal structure" aren't the same claim. Counting the top-level function definitions in the shipped source turns up 34 functions, and they sort themselves into eight groups with almost no effort:

Implementation: the global that touches everything

Chapter 7 introduced static struct editorConfig E as the one struct that runs the show, and nearly every function in the file reads or writes a field of it directly. That's convenient — no parameter lists to thread state through — and it's also exactly the kind of thing general refactoring literature on C programs calls out by name: a function that reads a global can't be tested on its own, and it can't be run twice, independently, in the same process. A function that takes its state as an explicit argument can be both. Here's the transformation on the smallest function in the file that touches E at all:

int editorFileWasModified(void) {
    return E.dirty;
}
int editorFileWasModified(const struct editorConfig *ed) {
    return ed->dirty;
}

Three lines, one new parameter, and the function no longer cares that E exists. Now imagine doing that to all 34 functions. The payoff scales with the program: you could construct two independent editorConfigs in a test harness and exercise them without one leaking into the other — which is precisely the setup Chapter 21 will want. The cost scales too, in the least interesting way possible: every one of those 34 call sites grows an argument, and every one of the (currently effortless) direct reads of E.cy or E.numrows becomes ed->cy, ed->numrows. Mechanical, not clever — which is exactly what a refactoring step is supposed to be.

Implementation: the switch that decides everything

The other standout candidate is editorProcessKeypress itself — one switch, twelve case groups, deciding what every single keystroke means:

void editorProcessKeypress(int fd) {
    static int quit_times = KILO_QUIT_TIMES;
    int c = editorReadKey(fd);
    switch(c) {
    case ENTER: editorInsertNewline(); break;
    case CTRL_C: break; /* ignored on purpose */
    case CTRL_Q:
        if (E.dirty && quit_times) { /* ...warn, decrement, return... */ }
        exit(0); break;
    case CTRL_S: editorSave(); break;
    case CTRL_F: editorFind(fd); break;
    case BACKSPACE: case CTRL_H: case DEL_KEY:
        editorDelChar(); break;
    case PAGE_UP: case PAGE_DOWN: { /* ...loop calling editorMoveCursor... */ } break;
    case ARROW_UP: case ARROW_DOWN:
    case ARROW_LEFT: case ARROW_RIGHT:
        editorMoveCursor(c); break;
    case CTRL_L: break;
    case ESC: break;
    default: editorInsertChar(c); break;
    }
    quit_times = KILO_QUIT_TIMES;
}

Fowler's catalog has a name for the general shape of this move: Replace Conditional with Polymorphism — when a switch keeps growing on the same discriminant, replace the branches with dispatch instead of a chain of cases. C has no classes to be polymorphic with, so the idiomatic C version swaps the switch for data: an array mapping each key to a function pointer, walked in a loop instead of matched in a case list.

/* Illustrative sketch — not in antirez's source. */
typedef void (*KeyHandler)(struct editorConfig *ed, int key);

static const struct { int key; KeyHandler fn; } KEYMAP[] = {
    { ENTER,  handle_enter }, { CTRL_Q, handle_quit }, { CTRL_S, handle_save },
    { CTRL_F, handle_find }, /* ... */
    { 0, NULL } /* sentinel */
};

void editorProcessKeypress(struct editorConfig *ed, int fd) {
    int c = editorReadKey(fd);
    for (int i = 0; KEYMAP[i].fn; i++)
        if (KEYMAP[i].key == c) { KEYMAP[i].fn(ed, c); return; }
    handle_insert(ed, c); /* the old default: case */
}

Whether this is actually an improvement depends entirely on what you're optimizing for. Adding a new key binding to the table version means appending one line to KEYMAP and writing one small handler function — no editing a 70-line function to find the right spot. But the switch version has something the table quietly gives up: you can read every single thing a keystroke can mean in kilo.c in one scroll, top to bottom, in the order the cases are checked, with the PAGE_UP/PAGE_DOWN special case sitting right there in context. The table trades that at-a-glance completeness for per-entry extensibility. Neither is free.

Edge cases: what a refactor is not for

Two things in kilo.c look, at first glance, like refactoring targets and aren't — and it's worth being as precise about what to leave alone as about what to change.

The goto labels are not a bug

enableRawMode's goto fatal: and editorSave's goto writeerr: are a recognized, idiomatic C pattern — one shared cleanup block at the bottom of a function, reached from several failure points, instead of repeating the same three cleanup lines at each one. Chapter 17 already named the real gap here: it's that every function invents its own version instead of sharing one die() helper, not that goto appears at all. "Remove the gotos" would make this code worse, not more maintainable — it would mean duplicating the cleanup logic at every failure point instead of jumping to one copy of it.

The other trap is treating "one file" itself as the defect. Splitting kilo.c into eight files buys real things — parallel work by more than one person without merge conflicts on every line, and a compiler that only has to look at row.c to know whether editorInsertRow typechecks. It also spends something Chapter 1 named as the actual point of this program: a reader can hold the whole thing in their head and read it start to finish in an afternoon. Eight files means eight headers, decisions about whether struct editorConfig and erow get exposed across each new boundary or hidden behind an opaque pointer (an indirection this program has never needed before), and a program that's no longer one thing you open and read — it's eight things you navigate between. For a 1,000-line teaching program, that's not obviously a win. For a program that's still growing — which is exactly Chapter 20's premise — it starts to look different.

So what did we actually gain?

Not a rewrite — a short, honest list of two specific, mechanical changes and two specific things deliberately left untouched. Threading editorConfig through as a parameter instead of reaching for a global buys testability at the cost of one new argument on every one of 34 functions. Replacing the keypress switch with a dispatch table buys per-entry extensibility at the cost of the one-scroll readability the switch gave for free. The goto-based cleanup stays exactly as it is, because it was never the problem. And splitting the file itself is the one move this chapter deliberately doesn't recommend outright — not because it's wrong, but because whether it's worth it depends on a question this chapter can't answer alone: is this program going to keep growing? Chapter 20 answers that question by actually growing it, and Chapter 21 builds the safety net — the tests this chapter kept having to caveat around — that any of these changes would actually need before you'd trust them in the program you use every day.

Before you turn the page
  1. Pick one more function from the input module — editorMoveCursor — and write out its signature the way it would look after the global-to-parameter refactor from this chapter. List every field of E the function body would need to reach through the new pointer instead.
  2. The dispatch-table sketch in this chapter drops the PAGE_UP/PAGE_DOWN case's special handling (the loop calling editorMoveCursor E.screenrows times). Sketch the small handler function that case would become, and decide whether it needs anything beyond the (ed, key) signature every other entry in KEYMAP uses.
  3. Make the case for not refactoring kilo.c at all — using only claims this book has already established (Chapters 1, 17, and 18 are good places to start) rather than general opinions about small programs.
