Chapter 11 · Part IV — Files

Opening Files

Chapter 10 closed by pointing straight at this chapter: editorInsertRow() already knows how to build a file's rows one at a time, by hand, from whatever bytes you hand it. Everything left to do here is get those bytes from disk instead of from a keystroke. That turns out to be a short chapter — not because loading a file is trivial, but because kilo.c already did the hard part, and opening a file is mostly wiring that existing machine up to fopen() and a loop.

Motivation: a file is just rows you haven't built yet

Look at where editorOpen() sits in main(), and it tells you something before you've read a line of its body:

int main(int argc, char **argv) {
    if (argc != 2) {
        fprintf(stderr,"Usage: kilo <filename>\n");
        exit(1);
    }

    initEditor();
    editorSelectSyntaxHighlight(argv[1]);
    editorOpen(argv[1]);
    enableRawMode(STDIN_FILENO);
    editorSetStatusMessage(
        "HELP: Ctrl-S = save | Ctrl-Q = quit | Ctrl-F = find");
    while(1) {
        editorRefreshScreen();
        editorProcessKeypress(STDIN_FILENO);
    }
    return 0;
}

Two things worth noticing before editorOpen itself. First, argc != 2 is checked before anything else runs — not even initEditor(). Kilo has no “start with an empty, unnamed buffer” mode; a filename, real or not, is mandatory from the very first line of the program. Second, editorOpen(argv[1]) runs before enableRawMode(STDIN_FILENO). That ordering isn't an accident, and it matters more than it looks — hold onto it, because the next section explains exactly why it's there.

Design: load before you touch the terminal

enableRawMode() is the function that calls atexit(editorAtExit), registering the cleanup handler that will later restore the user's terminal settings. In main(), editorOpen() runs one line before that call. Which means: if loading the file fails badly enough that editorOpen() calls exit(), no cleanup handler has been registered yet, because the terminal was never put into raw mode in the first place. There is nothing to restore, so a hard failure can just exit(1) and walk away. Chapter 17 will spend a whole chapter on the general problem of cleaning up after yourself; this chapter gets to sidestep it entirely, just by going first.

Sequencing is a design decision

Nothing in editorOpen()'s own code depends on raw mode being off. The safety here comes entirely from where the call sits in main(). It's a small reminder that in a program this size, the order you call things in is doing real work — it's not just a list of setup steps that happen to be arranged top to bottom.

Implementation: editorOpen, in full

Here is the entire function — thirty-one lines, and every one of them earns its place:

int editorOpen(char *filename) {
    FILE *fp;

    E.dirty = 0;
    free(E.filename);
    size_t fnlen = strlen(filename)+1;
    E.filename = malloc(fnlen);
    memcpy(E.filename,filename,fnlen);

    fp = fopen(filename,"r");
    if (!fp) {
        if (errno != ENOENT) {
            perror("Opening file");
            exit(1);
        }
        return 1;
    }

    char *line = NULL;
    size_t linecap = 0;
    ssize_t linelen;
    while((linelen = getline(&line,&linecap,fp)) != -1) {
        if (linelen && (line[linelen-1] == '\n' || line[linelen-1] == '\r'))
            line[--linelen] = '\0';
        editorInsertRow(E.numrows,line,linelen);
    }
    free(line);
    fclose(fp);
    E.dirty = 0;
    return 0;
}

Four things happen in order, and each one deserves its own look.

1. Kilo makes its own copy of the filename

Before it opens anything, editorOpen frees whatever E.filename already pointed to, then allocates strlen(filename)+1 bytes and memcpys the argument in, null terminator included. E.filename is never a pointer aliased into argv — it's a heap buffer kilo owns outright, which is exactly what lets editorOpen be called again later (say, “Save As” in Chapter 12) without worrying about who owns the string it's replacing.

2. fopen, and the one error kilo forgives

fopen(filename, "r") either succeeds or sets errno and returns NULL. Kilo's response is deliberately asymmetric: if errno == ENOENT — the file simply doesn't exist yet — the function just returns 1 with zero rows loaded, and the editor comes up with an empty buffer under that name. That's the whole mechanism behind “run kilo newfile.txt to start a new file”: there's no special “new file” branch anywhere in the program, just a tolerated errno value. Any other reason fopen could fail — permission denied, too many open files, whatever errno the platform reports — is treated as fatal: perror("Opening file") prints it to stderr and exit(1) ends the program on the spot. There is no in-editor error message for this one; it never gets far enough to draw a screen.

3. Reading lines with getline, and stripping exactly one byte

The read loop leans on POSIX getline(&line, &linecap, fp): it grows line's heap buffer itself as needed (tracked by linecap), returns the byte length of whatever it just read, and returns -1 at end of file. This is also why the very top of kilo.c defines _POSIX_C_SOURCE 200809L under #ifdef __linux__ — getline is a POSIX.1-2008 addition, not part of standard C, and glibc won't expose it without being asked. Each line getline hands back still has its terminator attached, so the very next line strips it:

if (linelen && (line[linelen-1] == '\n' || line[linelen-1] == '\r')) line[--linelen] = '\0'; — read that carefully: it's one if, not a loop. It removes exactly one trailing byte, once.

A single if, not a while — verify it yourself

On a Unix file, each line ends in a lone '\n', and one strip removes it cleanly. But a Windows-style CRLF file ends every line in "\r\n" — two bytes. The if above fires once, sees '\n' as the last byte, and removes only that. The '\r' right before it is never touched, and becomes the new last character of line — which then gets copied straight into row->chars by the very next call. Kilo does not normalize CRLF line endings on load. Open a CRLF file in kilo and every single row quietly carries a trailing '\r' you can't see on screen. Trace the condition yourself against a fresh clone of kilo.c before taking this on faith — the same habit this book has asked of you at every callout like this one.

Whatever survives the strip is handed to editorInsertRow(E.numrows, line, linelen) — always at E.numrows, i.e. always appended to the end of the row array currently being built. That's the same primitive Chapter 08 built for splitting a line and Chapter 10 reused for typing Enter. editorOpen doesn't need a separate “load a line from disk” code path; it's a loop around the one function that already knows how to add a row, called once per line getline reads.

4. Cleanup, and a second E.dirty = 0 that isn't redundant

After the loop ends, editorOpen frees line (the buffer getline was managing), fcloses the file, and sets E.dirty = 0 a second time — the same assignment already made at the very top of the function. It's tempting to read that as a leftover duplicate. It isn't. editorInsertRow() increments E.dirty on every call — that's how it flags real edits elsewhere in the program. During the load loop above, it gets called once per line read, so a freshly opened hundred-line file would leave E.dirty at 100 if nothing reset it afterward. The second E.dirty = 0 is what makes “just opened” read as “nothing to save yet” instead of “a hundred unsaved changes,” the instant the file finishes loading and before the user has typed a single character.

What loading a file changes elsewhere on screen

editorOpen never draws anything itself — but two things it sets, E.filename and E.numrows, are read by code you've already met in earlier chapters, and both change what the very next editorRefreshScreen() call puts on screen.

Small reminder from a small program: even a codebase this deliberately minimal, this widely read, still has a typo nobody's fixed. Worth remembering the next time a piece of code looks too authoritative to double-check.

Edge cases: what editorOpen quietly guarantees (and doesn't)

  • A missing file is not an error the user ever sees. ENOENT is the one errno value editorOpen forgives, and it forgives it silently — no status message, no beep, just an empty buffer under the name you gave it.
  • Every other open failure is fatal, immediately. Permission denied and every other errno route through perror + exit(1) before the editor has drawn a single frame — there's no recovering into an empty buffer for those.
  • editorOpen's own return value goes nowhere. It's declared to return 0 on success or 1 on a tolerated not-found, but main() calls editorOpen(argv[1]); as a bare statement and never reads the result. The errno branch inside the function is what actually decides behavior; the return value is documentation that nothing downstream consults.
  • CRLF files keep a stray trailing '\r' per line. The newline strip is a single if, and only ever removes one byte.
  • Loading reuses insertion, it doesn't duplicate it. Every line read becomes one more call to the same editorInsertRow() Chapters 08 and 10 already built — there is no parallel, file-specific way to add a row to the buffer.

So what did we actually gain?

Almost nothing new, and that's the point. editorOpen adds one system call (fopen), one library loop (getline), and about a dozen lines gluing them to editorInsertRow() — a function this book had already fully built two chapters ago. The genuinely new ideas are small and specific: forgive exactly one errno, run before the terminal is touched so failure stays cheap to handle, and remember to zero the dirty counter a second time because the loading loop itself trips it. Everything else is reuse. Chapter 12 is the mirror image of this chapter — instead of turning disk bytes into rows, it turns rows back into disk bytes — and it will lean on that same E.dirty counter this chapter was careful to reset.

Before you turn the page
  1. Rewrite the trailing-newline strip so it correctly removes a full CRLF pair ("\r\n"), not just its last byte, while still leaving a lone Unix '\n' handled correctly too. What has to change about the condition, and does it still only need to look at the last one or two bytes of line?
  2. Trace what happens if you run kilo on a path that exists but is a directory, not a regular file. Does fopen fail on your system, and with which errno? Work out, before you test it, which branch of editorOpen that lands in.
  3. editorOpen's int return value is never checked by its only caller. Sketch the smallest change to main() that would let it tell the user, in the status bar, “New file” versus an existing file was loaded — using information editorOpen already computes but currently discards.
