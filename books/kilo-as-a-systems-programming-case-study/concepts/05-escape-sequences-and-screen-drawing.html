Chapter 05 · Part II — The Terminal & The Loop

Escape Sequences and Screen Drawing

Chapter 04 got you a raw stream of keystrokes. This chapter is about the other direction: how Kilo puts pixels — well, characters — on the screen with no curses, no terminfo database, and no library standing between it and the terminal. The trick is that a terminal emulator already understands a small vocabulary of special byte sequences. Kilo's whole visual output is just write(), called with the right bytes.

Motivation: the screen is not a canvas, it's a conversation

Here's the thing that has to click before any of this makes sense: a terminal program does not have a frame buffer it can poke pixels into. It has one file descriptor it can write() bytes to, and a terminal emulator on the other end that decides what those bytes mean. Ordinary bytes become ordinary characters. But a byte sequence starting with ESC (0x1B) followed by '[' — the Control Sequence Introducer, or CSI — is a signal the terminal treats specially: move the cursor, erase some text, change a color, hide the cursor entirely. Kilo never draws anything. It asks the terminal, politely and in writing, to draw things for it.

That reframes the whole chapter. There's no new I/O mechanism to learn — it's the same write(STDOUT_FILENO, ...) you'd use to print anything else. What's new is the vocabulary: a compact set of escape sequences that, strung together in the right order, redraw an entire terminal screen every time the user presses a key.

Design: one buffer, one write, zero flicker

The naive approach — call write() once per escape sequence, once per character, as you figure out what to draw — would work, but it would also mean dozens or hundreds of tiny system calls per frame, each one a chance for the terminal to repaint partially and for the user to see a flicker mid-redraw. Kilo's answer is the same idea Chapter 03 previewed: an append buffer, one heap-allocated string that every escape sequence and every character gets appended to, flushed to the terminal with exactly one write() call per frame.

/* We define a very simple "append buffer" structure, that is an heap
 * allocated string where we can append to. This is useful in order to
 * write all the escape sequences in a buffer and flush them to the standard
 * output in a single call, to avoid flickering effects. */
struct abuf {
    char *b;
    int len;
};

#define ABUF_INIT {NULL,0}

void abAppend(struct abuf *ab, const char *s, int len) {
    char *new = realloc(ab->b,ab->len+len);

    if (new == NULL) return;
    memcpy(new+ab->len,s,len);
    ab->b = new;
    ab->len += len;
}

void abFree(struct abuf *ab) {
    free(ab->b);
}

Read abAppend() and you'll recognize the pattern from erow's dynamic array in Chapter 02: grow with realloc(), reassign the pointer to whatever (possibly relocated) block comes back, then memcpy() the new bytes in at the end. There's no string formatting here, no null terminators to worry about — len is tracked explicitly the whole way, exactly the discipline Chapter 03 flagged as a running theme in this codebase. By the time editorRefreshScreen() is done, every escape sequence and every visible character for the entire frame lives in one contiguous block of memory, and it hits the wire in a single write().

The clear-screen sequence you won't find

If you've seen the popular Build Your Own Text Editor tutorial that walks this same program, you may be expecting ESC[2J — “erase entire display” — as the first thing Kilo does each frame. It isn't there. A search of the real kilo.c for [2J comes back empty. Kilo never clears the whole screen. Instead it goes home (ESC[H) and redraws every visible row, every frame, erasing only to the end of each line as it goes. Since nothing is left un-redrawn, there's nothing a full clear would need to clean up — and skipping it means one less full-screen repaint for the terminal to do.

Implementation: reading editorRefreshScreen top to bottom

The whole function is worth walking in the order it actually executes, because that order is the drawing order on screen: top row to bottom row, then the two status rows, then the cursor.

Step 1 — hide the cursor, go home

void editorRefreshScreen(void) {
    int y;
    erow *r;
    char buf[32];
    struct abuf ab = ABUF_INIT;

    abAppend(&ab,"\x1b[?25l",6); /* Hide cursor. */
    abAppend(&ab,"\x1b[H",3); /* Go home. */

Two escapes, and the order matters. ESC[?25l is a DEC private mode sequence (that leading ? marks it as one) that turns the text cursor invisible — the terminal keeps tracking where it “is” internally, it just stops drawing the blinking block there. ESC[H is CUP (Cursor Position) called with no parameters, which defaults to row 1, column 1 — home. Hiding the cursor first means the user never watches it hop around mid-redraw; it reappears exactly once, at the very end, already in its correct final spot.

Step 2 — redraw every row, tilde or text

Kilo then loops E.screenrows times (one iteration per visible row) and, for each row past the end of the file, writes a bare ~ — the vi-family convention for “this line doesn't exist.” If the file is completely empty, one row near the top instead centers a welcome banner:

for (y = 0; y < E.screenrows; y++) {
    int filerow = E.rowoff+y;

    if (filerow >= E.numrows) {
        if (E.numrows == 0 && y == E.screenrows/3) {
            char welcome[80];
            int welcomelen = snprintf(welcome,sizeof(welcome),
                "Kilo editor -- verison %s\x1b[0K\r\n", KILO_VERSION);
            int padding = (E.screencols-welcomelen)/2;
            if (padding) {
                abAppend(&ab,"~",1);
                padding--;
            }
            while(padding--) abAppend(&ab," ",1);
            abAppend(&ab,welcome,welcomelen);
        } else {
            abAppend(&ab,"~\x1b[0K\r\n",7);
        }
        continue;
    }
    /* ...render the row's characters... */
}

Notice ESC[0K at the end of every one of these lines — EL, Erase in Line, with parameter 0 meaning “from the cursor to the end of the line.” That's the per-line stand-in for the full-screen clear Kilo doesn't do: whatever character happened to be sitting at the end of this terminal row from some previous, longer frame gets wiped before the carriage return. Erase locally, every row, every frame — never globally.

Step 3 — text rows, colored lazily

For a row that does hold file content, Kilo walks its render buffer (the tab-expanded display version from Chapter 02, one byte at a time) alongside its hl array (one highlight-type byte per rendered character, from Chapter 14 onward). The color-changing logic is worth isolating, because it's a small, deliberate optimization:

current_color: only speak up when something changes

A local int current_color tracks the SGR (Select Graphic Rendition) color code most recently written. Kilo only appends a new ESC[<n>m sequence when the character's color actually differs from current_color — a run of ten same-colored characters costs one color escape, not ten. Falling back to HL_NORMAL appends ESC[39m (reset foreground to the terminal's default) exactly once, the moment color is no longer needed — not on every plain character.

The color numbers themselves come from a small, honest mapping — editorSyntaxToColor() — that has nothing clever in it: comments and multi-line comments are 36 (cyan), the first keyword class is 33 (yellow), the second is 32 (green), strings are 35 (magenta), numbers are 31 (red), a search match is 34 (blue), and anything else defaults to 37 (white). These are the classic 8-color SGR codes every VT100-descended terminal understands — no 256-color or truecolor codes, on purpose, for maximum compatibility.

One more category gets special treatment: HL_NONPRINT — raw control bytes that end up inside a line. Rather than write them out and risk the terminal misinterpreting a stray control character as its own escape sequence, Kilo wraps a printable stand-in in reverse video:

if (hl[j] == HL_NONPRINT) {
    char sym;
    abAppend(&ab,"\x1b[7m",4);
    if (c[j] <= 26)
        sym = '@'+c[j];
    else
        sym = '?';
    abAppend(&ab,&sym,1);
    abAppend(&ab,"\x1b[0m",4);
}

ESC[7m is SGR 7, reverse video — it swaps foreground and background so the glyph stands out as “something unusual is here” rather than blending in or corrupting the display. Bytes 1–26 (the classic Ctrl-A through Ctrl-Z range) get mapped back to their letter ('@'+1 = 'A', and so on) so a reader can tell which control character it was; anything else just shows a reversed ?. ESC[0m resets every SGR attribute back to plain before continuing. Every text row still ends with its own ESC[39m (color reset) + ESC[0K (erase to end of line) + \r\n, regardless of what happened inside it — belt-and-suspenders against a color escape leaking into the next line.

Step 4 — two status rows, reserved on purpose

Below the text, Kilo appends exactly two more rows, both built the same way as everything above — more escape sequences into the same buffer, no separate mechanism:

RowSequenceContent
Status barESC[7m ... ESC[0m\r\n Reverse video the whole row; left side shows filename - N lines (modified), right side shows currentline/totallines, space-padded to fill exactly E.screencols.
Message barESC[0K then plain text E.statusmsg, shown only if time(NULL) - E.statusmsg_time < 5 — a message older than five seconds simply stops being written, no explicit clear required.

These two rows aren't drawn into leftover space by accident — they're reserved. updateWindowSize(), which we'll get to in a moment, subtracts 2 from the queried terminal height before storing it in E.screenrows. The text-row loop's for (y = 0; y < E.screenrows; y++) physically cannot reach these last two lines, because they were never counted as part of it.

Step 5 — place the cursor, show it, flush once

The very last work before flushing is figuring out where the terminal cursor actually belongs:

int j;
int cx = 1;
int filerow = E.rowoff+E.cy;
erow *row = (filerow >= E.numrows) ? NULL : &E.row[filerow];
if (row) {
    for (j = E.coloff; j < (E.cx+E.coloff); j++) {
        if (j < row->size && row->chars[j] == TAB) cx += 7-((cx)%8);
        cx++;
    }
}
snprintf(buf,sizeof(buf),"\x1b[%d;%dH",E.cy+1,cx);
abAppend(&ab,buf,strlen(buf));
abAppend(&ab,"\x1b[?25h",6); /* Show cursor. */
write(STDOUT_FILENO,ab.b,ab.len);
abFree(&ab);

cx is deliberately not E.cx+1. The loop re-walks the row from the horizontal scroll offset up to E.cx, and every time it crosses a TAB byte it advances cx to the next multiple of 8 first — the exact same expansion rule editorUpdateRow() used to build the render buffer back in Chapter 02. If it didn't, a cursor sitting after a tab would land on the wrong visual column: correct in terms of bytes into chars, wrong in terms of columns on screen. The %d;%dH escape is CUP again, this time with real parameters, 1-indexed because terminals count from row/column 1 while E.cy/cx are 0/1-indexed respectively in the code.

Then: reveal the cursor (ESC[?25h), and only now — after every escape sequence and every character for the entire frame has been appended — one write(), one abFree(). Everything since ESC[?25l at the top of the function happened entirely in memory.

Escape sequences that ask questions, not just draw

So far every sequence has been one-directional: Kilo tells the terminal what to do. But one sequence works the other way — Kilo asks the terminal a question and reads back an answer, over the very same file descriptor Chapter 04's editorReadKey() normally reads keystrokes from.

int getCursorPosition(int ifd, int ofd, int *rows, int *cols) {
    char buf[32];
    unsigned int i = 0;

    /* Report cursor location */
    if (write(ofd, "\x1b[6n", 4) != 4) return -1;

    /* Read the response: ESC [ rows ; cols R */
    while (i < sizeof(buf)-1) {
        if (read(ifd,buf+i,1) != 1) break;
        if (buf[i] == 'R') break;
        i++;
    }
    buf[i] = '\0';

    /* Parse it. */
    if (buf[0] != ESC || buf[1] != '[') return -1;
    if (sscanf(buf+2,"%d;%d",rows,cols) != 2) return -1;
    return 0;
}

ESC[6n is DSR, Device Status Report, with parameter 6 specifically requesting the cursor position. The terminal replies — as ordinary bytes on the input side — with ESC[<row>;<col>R. Kilo reads that reply one byte at a time until it sees the terminating 'R', then parses it with a plain sscanf(). Nothing exotic: the “response to a query” is just more input, read with the same read() syscall as everything else, arriving on a channel Kilo otherwise treats as pure keystrokes.

getWindowSize: ioctl first, escape sequences as a fallback

getCursorPosition() exists mainly to serve a second function, one this chapter's drawing code depends on entirely: knowing how big the terminal actually is.

int getWindowSize(int ifd, int ofd, int *rows, int *cols) {
    struct winsize ws;

    if (ioctl(1, TIOCGWINSZ, &ws) == -1 || ws.ws_col == 0) {
        int orig_row, orig_col, retval;

        retval = getCursorPosition(ifd,ofd,&orig_row,&orig_col);
        if (retval == -1) goto failed;

        /* Go to right/bottom margin and get position. */
        if (write(ofd,"\x1b[999C\x1b[999B",12) != 12) goto failed;
        retval = getCursorPosition(ifd,ofd,rows,cols);
        if (retval == -1) goto failed;

        char seq[32];
        snprintf(seq,32,"\x1b[%d;%dH",orig_row,orig_col);
        if (write(ofd,seq,strlen(seq)) == -1) { /* Can't recover... */ }
        return 0;
    } else {
        *cols = ws.ws_col;
        *rows = ws.ws_row;
        return 0;
    }
failed:
    return -1;
}

The preferred path costs no escape sequences at all: ioctl(1, TIOCGWINSZ, &ws) asks the kernel directly, which knows the terminal's dimensions without needing to ask the terminal emulator anything. Only if that fails, or reports ws_col == 0 (which the code's own comment notes happens on some terminals or terminal multiplexers), does Kilo fall back to the escape-sequence route. That fallback is a neat trick: write ESC[999C ESC[999B — cursor forward 999, cursor down 999 — then read the cursor's position back. 999 isn't a real column or row count; it's deliberately oversized, relying on the fact that VT100-family terminals clamp cursor motion at the actual screen edge instead of scrolling or erroring. Wherever the cursor lands after “move by 999, twice” is the bottom-right corner, and reading its position back with getCursorPosition() tells you exactly how many rows and columns the terminal has. The cursor is then walked back to wherever it started (ESC[%d;%dH with the saved orig_row/orig_col) so a size query the user never asked for doesn't visibly move anything.

Where the “minus 2” comes from

updateWindowSize() calls getWindowSize() and then does E.screenrows -= 2 — the one line that reserves this chapter's two status rows. It's also registered as the response to a live terminal resize: signal(SIGWINCH, handleSigWinCh) means the kernel-delivered SIGWINCH signal re-runs updateWindowSize() (and clamps E.cy back inside the new bounds) whenever the user resizes their terminal window mid-session, so the next editorRefreshScreen() draws against dimensions that are actually still correct.

Edge cases: what this design deliberately doesn't do

  • No full-screen clear, ever. As the callout above says, ESC[2J doesn't appear anywhere in kilo.c. This is a deliberate trade for less flicker, not an oversight — but it only works because every visible row is genuinely redrawn every single frame, with its own per-line erase. Skip a row in that loop and stale content from a previous, longer frame could survive on screen with no clear to remove it.
  • Ctrl-L is a no-op, on purpose. The conventional “refresh the screen” keystroke in many terminal programs does essentially nothing special in Kilo's key dispatch: 
    case CTRL_L: /* ctrl+l, clear screen */
    /* Just refresht the line as side effect. */
    break;
    (Yes, that comment has a typo in the real source.) It isn't ignored — the keystroke is consumed so it doesn't fall through to text insertion — it's just that Chapter 06's read-react-repaint loop already calls editorRefreshScreen() unconditionally on every trip through main(), whether or not Ctrl-L was pressed. There's nothing left for the keystroke itself to trigger.
  • The size-query fallback assumes edge-clamping behavior. ESC[999C/ESC[999B only work as a size probe because real terminals clamp cursor motion at the screen boundary instead of, say, ignoring an out-of-range request or scrolling the viewport. It's a documented VT100-family behavior, not an assumption unique to Kilo, but it's still an assumption — a hypothetical terminal that didn't clamp would break this fallback silently.
  • Terminal-emulator quirks aren't Kilo's problem to solve. Every sequence in this chapter — ESC[H, ESC[0K, ESC[?25l/h, SGR colors, ESC[6n — is old, widely-supported VT100/ANSI vocabulary, chosen precisely because virtually every terminal emulator in real use honors it. Kilo doesn't detect terminal capabilities or fall back to a lesser drawing mode; it simply assumes a reasonably compliant VT100-compatible terminal on the other end, the same assumption Chapter 03 flagged when it first introduced escape sequences as “ordinary bytes the terminal has agreed to treat specially.”

So what did we actually gain?

A complete answer to “how does a program with no GUI toolkit draw a screen?” — and it turns out to be smaller than it sounds. One append buffer that turns dozens of small drawing decisions into one write(). A handful of escape sequences — hide/show cursor, home, erase-to-end-of- line, set/reset color, position the cursor — reused for every row, every frame, with no special-casing. And one sequence, ESC[6n, that runs the whole conversation in reverse: the program asks, the terminal answers, over the same channel keystrokes arrive on. Chapter 06 picks this up immediately — editorRefreshScreen() is one of exactly two calls in Kilo's entire main loop, and now you know everything that one call actually does.

Before you turn the page
  1. Trace what would visibly go wrong if ESC[?25h (show cursor) were appended before the row-drawing loop instead of after it, at the very end. What would the user see during a redraw that they don't see today?
  2. The current_color optimization only tracks syntax-highlight color, not the reverse-video escapes used for HL_NONPRINT or the status bar. Why might it be safe to skip that optimization for those two cases specifically? (Hint: how often does each one actually occur in a typical frame?)
  3. Suppose ioctl(1, TIOCGWINSZ, &ws) succeeded but returned ws_row as 1 more than the terminal's real height — an off-by-one in the kernel's report. Walk through what would happen on the next editorRefreshScreen() call, and explain why this class of bug would be easy to miss during quick manual testing.
