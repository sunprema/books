Chapter 08 · Part III — Data & Navigation

Text Storage

Chapter 07 told you where the document lives: E.row, a pointer, and E.numrows, a count. It deliberately didn't tell you what's on the other end of that pointer. That's this chapter's whole job — one struct per line, a second buffer inside it that exists only because tab characters are one byte but many columns, and the small, careful arithmetic that keeps those two buffers in sync every time you type.

Motivation: a file isn't the same shape as a screen

Suppose Kilo stored a line exactly as the file has it — one C string, nothing else — and drew that string straight to the terminal. Most bytes would draw fine: one byte, one printed cell, one column. A tab breaks that assumption immediately. A tab is a single byte, '\t', ASCII 9. But a terminal that receives a raw tab byte jumps its cursor forward to the next multiple of some tab-stop width — a variable amount of visual space from a fixed one-byte cause. If the cursor's column in the file (“you're at the 5th character of this line”) and the cursor's column on screen (“you're 12 cells from the left edge”) are the same number for every ordinary character but diverge the moment a tab is involved, then one number can't answer both questions. Kilo needs two.

Its answer is to keep two buffers per line instead of one: the bytes you actually typed, and a second, disposable buffer built from the first, with every tab expanded into the right number of spaces. Editing always happens on the first buffer. The second is regenerated, in full, every time the first one changes. That's the entire idea of this chapter — everything below is really just “how is that second buffer built, and what has to happen to keep a whole array of them consistent as lines are inserted, deleted, and edited.”

Design: one struct per line, two buffers each

Here's erow again, the struct Chapter 07 introduced but didn't dwell on. E.row points at an array of these — one per line of the file, in order:

typedef struct erow {
    int idx;            /* Row index in the file, zero-based. */
    int size;           /* Size of the row, excluding the null term. */
    int rsize;          /* Size of the rendered row. */
    char *chars;        /* Row content. */
    char *render;       /* Row content "rendered" for screen (for TABs). */
    unsigned char *hl;  /* Syntax highlight type for each character in render.*/
    int hl_oc;          /* Row had open comment at end in last syntax highlight
                       check. */
} erow;

Two pairs do the work. chars/size are the file's own bytes — exactly what you typed or loaded, exactly what gets saved back to disk. render/rsize are a second, independently-allocated buffer built from chars, with every tab expanded into spaces, that exists purely so the screen has something honest to draw. Nothing about editing ever touches render directly — you insert and delete characters in chars, and render gets rebuilt, wholesale, as a consequence. hl rides along as a third buffer, sized to match render/rsize rather than chars — one highlight-class byte per rendered character, which is Chapters 14–16's business, not this one's, but worth noting now: whatever rebuilds render is going to need to rebuild hl too, in the same breath.

A constant that doesn't exist

If you've read the popular “Build Your Own Text Editor” walkthrough of this same program, you may be expecting a KILO_TAB_STOP constant somewhere near the top of the file. It isn't there. antirez's shipped kilo.c hard-codes the tab width — 8 columns — as a bare literal, repeated at three separate call sites you'll meet in this chapter. Both approaches produce the same behavior; only one of them is what's actually in the repository this book studies.

Implementation: building render from chars

editorUpdateRow() is the one function that performs this rebuild, and it is called after every mutation to a row's characters — you'll see it at the tail of every insert and delete function later in this chapter. It frees the old render, sizes a fresh one, and refills it:

void editorUpdateRow(erow *row) {
    unsigned int tabs = 0, nonprint = 0;
    int j, idx;

    free(row->render);
    for (j = 0; j < row->size; j++)
        if (row->chars[j] == TAB) tabs++;

    unsigned long long allocsize =
        (unsigned long long) row->size + tabs*8 + nonprint*9 + 1;
    if (allocsize > UINT32_MAX) {
        printf("Some line of the edited file is too long for kilo\n");
        exit(1);
    }

    row->render = malloc(row->size + tabs*8 + nonprint*9 + 1);
    idx = 0;
    for (j = 0; j < row->size; j++) {
        if (row->chars[j] == TAB) {
            row->render[idx++] = ' ';
            while((idx+1) % 8 != 0) row->render[idx++] = ' ';
        } else {
            row->render[idx++] = row->chars[j];
        }
    }
    row->rsize = idx;
    row->render[idx] = '\0';

    /* Update the syntax highlighting attributes of the row. */
    editorUpdateSyntax(row);
}

Read it as two passes over the same line. The first pass just counts: how many TAB bytes (ASCII 9) are in chars? That count sizes the allocation — worst case, every tab costs 8 bytes of expansion, so tabs*8 is a safe upper bound, not an exact one. (There's also a hard ceiling here: if the computed size would overflow a 32-bit length, Kilo refuses to open the file rather than silently under-allocate.) The second pass does the actual expansion: an ordinary character copies straight across; a tab writes one space, then keeps writing spaces (while((idx+1) % 8 != 0)) until the write position lands on a column that's a multiple of 8. That's the whole tab-stop rule, stated as a loop instead of a formula.

A variable that does nothing

nonprint is declared, folded into both size calculations (tabs*8 + nonprint*9), and never incremented anywhere in this function or anywhere else in the file. It is always zero — dead code, left over from what was presumably an earlier version that reserved extra bytes in render for non-printable characters. Kilo does still give non-printable bytes special visual treatment, but it does that later, at draw time in editorRefreshScreen() (Chapter 06), not by budgeting for it here. Even a widely-read, celebrated ~1000-line program ships the occasional variable that quietly does nothing — worth noticing, not worth losing sleep over.

Notice the very last line of the function: editorUpdateSyntax(row). That call means chars, render, and hl are not three independently-maintained buffers you have to remember to keep in sync by hand — they're one pipeline. Change the characters, call editorUpdateRow(), and the rendered text and its highlight colors come along for free. Every function later in this chapter ends by calling it.

Implementation: the row array itself

One erow knows how to render itself. The array of them — E.row, E.numrows lines long — needs its own operations: insert a line, delete a line. Both are C's classic array-shift dance, done with realloc() and memmove():

/* Insert a row at the specified position, shifting the other rows on the bottom
 * if required. */
void editorInsertRow(int at, char *s, size_t len) {
    if (at > E.numrows) return;
    E.row = realloc(E.row,sizeof(erow)*(E.numrows+1));
    if (at != E.numrows) {
        memmove(E.row+at+1,E.row+at,sizeof(E.row[0])*(E.numrows-at));
        for (int j = at+1; j <= E.numrows; j++) E.row[j].idx++;
    }
    E.row[at].size = len;
    E.row[at].chars = malloc(len+1);
    memcpy(E.row[at].chars,s,len+1);
    E.row[at].hl = NULL;
    E.row[at].hl_oc = 0;
    E.row[at].render = NULL;
    E.row[at].rsize = 0;
    E.row[at].idx = at;
    editorUpdateRow(E.row+at);
    E.numrows++;
    E.dirty++;
}

Every insertion grows the array by exactly one slot — a fresh realloc() per line. That's O(n) work per insert, which means loading an n-line file by inserting one row at a time costs O(n²) overall in the worst case. For the file sizes Kilo targets, that's a perfectly reasonable price for the simplicity of “an array, plus realloc”; it's exactly the kind of trade-off a 1000-line editor is allowed to make and a text editor meant to handle gigabyte logs is not. If at isn't the very end of the array, memmove() shoves every row at or after it one slot toward higher memory to open a gap, and the loop right after renumbers each shifted row's .idx field to match its new position. Then the new row's own fields get filled in: its bytes are deep-copied (len+1, deliberately including one byte past the caller's stated length — the caller is trusted to have a valid null terminator sitting right there), render/hl are zeroed so editorUpdateRow() builds them fresh, and only then does E.numrows actually grow.

Deletion is insertion run in reverse — almost:

void editorFreeRow(erow *row) {
    free(row->render);
    free(row->chars);
    free(row->hl);
}

/* Remove the row at the specified position, shifting the remainign on the
 * top. */
void editorDelRow(int at) {
    erow *row;

    if (at >= E.numrows) return;
    row = E.row+at;
    editorFreeRow(row);
    memmove(E.row+at,E.row+at+1,sizeof(E.row[0])*(E.numrows-at-1));
    for (int j = at; j < E.numrows-1; j++) E.row[j].idx++;
    E.numrows--;
    E.dirty++;
}

editorFreeRow() releases the three buffers one erow owns — render, chars, hl — but not the erow struct itself, because erows live inline inside E.row's array, not as individual heap allocations. editorDelRow() then closes the gap with a memmove() in the opposite direction from insertion's, and shrinks E.numrows. Notice what doesn't shrink: the array's actual allocated capacity. Delete every line in a huge file and E.row's underlying block is still sized for all of them — only the count says otherwise.

A loop worth checking against your own clone

Look closely at that renumbering loop: for (int j = at; j < E.numrows-1; j++) E.row[j].idx++; — it increments the .idx of every row that just shifted down into a lower array position. That's the same direction editorInsertRow() uses to shift rows up out of an insertion's way. And .idx isn't decorative: the syntax highlighter (Chapters 14–16) uses row->idx to look up a line's immediate neighbor when deciding whether a multi-line comment is still open. Incrementing after a deletion, where every surviving row's true position just went down by one, reads as a genuine indexing bug in the shipped source — not a deliberate choice. It's a small, narrow-impact one (it can only misdirect a highlight lookup by one row after a deletion), but it's real, and it's exactly the kind of thing worth verifying against a fresh clone of the repository yourself rather than taking a book's word for it.

Implementation: editing bytes inside one row

One more layer down: inserting or deleting a single character within one line's chars buffer. Three small functions, all following the same shape — resize with realloc(), shift bytes with memmove()/memcpy(), call editorUpdateRow(), bump E.dirty:

void editorRowInsertChar(erow *row, int at, int c) {
    if (at > row->size) {
        /* Pad the string with spaces if the insert location is outside the
         * current length by more than a single character. */
        int padlen = at-row->size;
        /* In the next line +2 means: new char and null term. */
        row->chars = realloc(row->chars,row->size+padlen+2);
        memset(row->chars+row->size,' ',padlen);
        row->chars[row->size+padlen+1] = '\0';
        row->size += padlen+1;
    } else {
        /* If we are in the middle of the string just make space for 1 new
         * char plus the (already existing) null term. */
        row->chars = realloc(row->chars,row->size+2);
        memmove(row->chars+at+1,row->chars+at,row->size-at+1);
        row->size++;
    }
    row->chars[at] = c;
    editorUpdateRow(row);
    E.dirty++;
}

Two cases, and the split is the interesting part. Typing past the end of a line — at > row->size, the cursor is sitting somewhere in the blank space after the last real character — pads the gap with literal spaces up to the insertion point (the comment's own arithmetic note, +2 means: new char and null term, is the kind of detail that's easy to skim past and worth reading twice). Typing inside or at the exact end of existing text instead opens a single-byte gap with memmove(), carrying the existing null terminator forward along with everything after the insertion point (row->size-at+1 — that trailing +1 is the null byte). Either way, the one new byte gets written, render gets rebuilt, and E.dirty ticks up.

/* Append the string 's' at the end of a row */
void editorRowAppendString(erow *row, char *s, size_t len) {
    row->chars = realloc(row->chars,row->size+len+1);
    memcpy(row->chars+row->size,s,len);
    row->size += len;
    row->chars[row->size] = '\0';
    editorUpdateRow(row);
    E.dirty++;
}

/* Delete the character at offset 'at' from the specified row. */
void editorRowDelChar(erow *row, int at) {
    if (row->size <= at) return;
    memmove(row->chars+at,row->chars+at+1,row->size-at);
    editorUpdateRow(row);
    row->size--;
    E.dirty++;
}

editorRowAppendString() is a plain memcpy() onto the end — it's what Chapter 10 will use to implement “Backspace at column zero merges this line onto the end of the previous one.” editorRowDelChar() closes a one-byte gap by memmove()-ing everything after at (including the null terminator) one step left. Compare its last three lines to the two functions above it: both of those update size before calling editorUpdateRow(); this one calls editorUpdateRow() first, then decrements size afterward. The practical effect is harmless — a leftover duplicate byte at the old last position gets ignored on the next real access — but it's a genuine inconsistency in an otherwise uniform pattern, and a good instinct to develop: notice when three nearly-identical functions don't do their bookkeeping in quite the same order.

Implementation: flattening the array back into one file

Saving (Chapter 12's subject) needs the whole document as one contiguous buffer, not an array of independent allocations. editorRowsToString() is the function that builds it — worth seeing now, because it's really a text-storage operation that happens to be used by the save chapter, not the other way around:

char *editorRowsToString(int *buflen) {
    char *buf = NULL, *p;
    int totlen = 0;
    int j;

    /* Compute count of bytes */
    for (j = 0; j < E.numrows; j++)
        totlen += E.row[j].size+1; /* +1 is for "\n" at end of every row */
    *buflen = totlen;
    totlen++; /* Also make space for nulterm */

    p = buf = malloc(totlen);
    for (j = 0; j < E.numrows; j++) {
        memcpy(p,E.row[j].chars,E.row[j].size);
        p += E.row[j].size;
        *p = '\n';
        p++;
    }
    *p = '\0';
    return buf;
}

Two passes, same shape as editorUpdateRow()'s count-then-fill: the first sums every row's size+1 (the +1 is a '\n' this function appends after every row, including the last) to size one allocation; the second copies each row's bytes across and manually writes that newline. The detail worth circling: it copies chars, never render. Whatever tab expansion the screen shows you, the file on disk gets the original bytes back, untouched.

The rule that explains everything above

Every function in this chapter enforces the same boundary. Editing operates on chars. Saving reads chars. render exists purely for the screen, is rebuilt from chars on every change, and is never the source of truth for anything. If you remember one sentence from this chapter, make it that one — it's the answer to almost every “wait, which buffer does this touch?” question the rest of the book will raise.

Implementation: turning a character index back into a screen column

There's one more place tab expansion matters, and it's not inside any of the row-storage functions above — it's inside editorRefreshScreen() (Chapter 06), at the very moment it positions the terminal's actual cursor. E.cx is a character index into chars; the terminal wants a column number. On a line with no tabs those are the same number. On a line with tabs, they aren't — and kilo.c's own comment says so directly:

/* Put cursor at its current position. Note that the horizontal position
 * at which the cursor is displayed may be different compared to 'E.cx'
 * because of TABs. */
int j;
int cx = 1;
int filerow = E.rowoff+E.cy;
erow *row = (filerow >= E.numrows) ? NULL : &E.row[filerow];
if (row) {
    for (j = E.coloff; j < (E.cx+E.coloff); j++) {
        if (j < row->size && row->chars[j] == TAB) cx += 7-((cx)%8);
        cx++;
    }
}
snprintf(buf,sizeof(buf),"\x1b[%d;%dH",E.cy+1,cx);

This loop walks row->chars — the raw bytes, not render — from the left edge of the current viewport out to the cursor's logical position, and for every tab byte it passes, it jumps the running screen-column counter cx forward to the next multiple of 8 before the ordinary cx++. It's the same 8-column rule as editorUpdateRow(), arrived at through slightly different arithmetic (7-((cx)%8) here, versus a while loop there) — and there is no shared function or shared constant connecting the two. Search the file for editorRowCxToRx or anything like it and you'll find nothing; this inline loop, run fresh on every screen redraw, is the only cx-to-column conversion kilo.c has.

One rule, two independent implementations

If Kilo's tab width ever needed to change, a maintainer would have to find and update it in three places: twice inside editorUpdateRow() (the allocation-size formula and the expansion loop) and once more here, inside editorRefreshScreen(). None of the three refer to a shared constant. This is the direct, concrete cost of not naming that 8 — three independent places carrying the same assumption, with nothing enforcing that they agree.

Edge cases: where the two-buffer design shows its seams

  • Every row-array operation is O(n). editorInsertRow() and editorDelRow() both realloc()/memmove() the whole tail of the array on every single call. Loading a file of n lines one editorInsertRow() call at a time is O(n²) in the worst case. That's a real cost, accepted deliberately, because Kilo's whole premise is a small program for small-to-medium files — it's the same trade-off Chapter 02's buffer-design survey named: an array of lines is the simplest point on the spectrum, and simplicity is exactly what a ~1000-line editor is buying with it.
  • Deleting a row never shrinks the array's capacity. editorDelRow() frees a row's own three buffers and shrinks E.numrows, but never calls realloc() to give memory back to the allocator. Delete every line in a large file and E.row's block is still sized for the largest the document ever was.
  • The .idx renumbering after a delete looks backwards. Covered above, worth repeating in the edge-case list because it's the kind of bug that only bites in a narrow situation — a multi-line comment's open/closed state, computed via a neighbor lookup through row->idx, could consult the wrong row immediately after a deletion. Everyday single-line edits never trigger it.
  • Typing past the end of a line is a padding operation, not an error. editorRowInsertChar()'s at > row->size branch fills the gap with literal space characters. A cursor that's visually “past the last character” (which the padding-and-scrolling logic in Chapter 09 allows) becomes real, saved spaces the instant you type there — not an out-of-bounds write, and not silently ignored either.
  • chars and render can only ever drift apart between an edit and the next call to editorUpdateRow() — and every mutation function calls it before returning. There's no code path in kilo.c that changes a row's characters and leaves the old render in place; the two buffers are never observably out of sync from outside these functions.

So what did we actually gain?

A precise answer to a question that sounds simple and isn't: what does “the text” actually mean inside a running editor? Kilo's answer is that there are two texts, not one — chars, which is the truth, and render, which is a disposable, rebuilt-on-demand translation of that truth into what a terminal can honestly display. Every operation in this chapter, from inserting one character to flattening the whole document for a save, respects that boundary. You now know exactly which buffer any future feature should touch: if it's about what the file contains, it's chars; if it's about what the file looks like, it's render. Chapter 09 builds cursor movement and scrolling on top of this same split — and reuses the very cx-to-column arithmetic you just read.

Before you turn the page
  1. Trace what happens, function by function, when the user presses Tab in the middle of a line that already contains one tab earlier in it. Which fields of erow change, in what order, and which function is ultimately responsible for making render agree with chars again?
  2. The tab width (8) appears as a bare literal in three independent places in kilo.c. Sketch how you'd introduce a single named constant and thread it through all three — and consider whether a fourth, currently-unnoticed use might be lurking somewhere you haven't checked.
  3. Convince yourself, one way or the other, whether editorDelRow()'s E.row[j].idx++ loop is really a bug. What test case (a specific sequence of keystrokes on a specific small file) would make the consequence — a wrong neighbor row consulted for multi-line comment state — actually visible on screen?
