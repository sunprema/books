Chapter 13 · Part V — Search & Highlighting

Search

Every chapter so far has added one piece to a single, always-on picture: the editor draws the file, and you move around and change it. Search is the first thing in this book that isn't that. Press Ctrl-F and, for a little while, Kilo stops being a text editor and becomes a tiny, purpose-built search prompt — a different loop, a different set of keys that do anything, and a promise to hand you back exactly where you started if you back out. That promise, and the small pile of state it takes to keep it, is the real subject of this chapter. The searching itself is nine lines and one library call.

Motivation: a mode is a loop, not a flag

It would be easy to imagine "search mode" as a boolean somewhere in struct editorConfig — E.searching = 1 — checked here and there by the ordinary keypress handler. Kilo doesn't do that. editorFind(int fd) is called once, from one case in editorProcessKeypress, and it does not return until you're done searching. Inside it, there's a second while(1) loop with the exact same shape as main()'s own — show something, read a key, act on it, repeat — except this one shows a search prompt instead of the file, and reads keys for a much smaller vocabulary: type to extend the query, arrows to jump between matches, Backspace to shrink it, Escape or Enter to leave. While that inner loop is running, the outer one isn't. Ordinary movement, editing, save, and quit are simply not being checked. A mode, in Kilo, is not a flag threaded through the normal dispatch — it's its own event loop, borrowed and returned.

The only other while(1) in the file

main()'s loop is while(1){ editorRefreshScreen(); editorProcessKeypress(fd); }. editorFind's is the same two moves, reordered and specialized: set the status line to the query, refresh, read one key, decide. Across the roughly one thousand lines of kilo.c, these are the only two top-level loops that exist. Search isn't layered on top of the main loop with extra bookkeeping — it's a second, smaller loop that temporarily takes over.

Design: four numbers saved, so Escape can lie about time

Before that inner loop even starts, editorFind saves the entire on-screen situation:

int saved_cx = E.cx, saved_cy = E.cy;
int saved_coloff = E.coloff, saved_rowoff = E.rowoff;

Four integers — the exact same four fields Chapter 09 built the whole navigation and scrolling model around. That's the entire mechanism behind "cancel and go back to where I was." There's no undo stack, no snapshot of the buffer — search never edits anything, so the buffer needs no protecting. It only ever moves the cursor and the viewport, and those are four ints. Save them going in; if you press Escape, write them straight back on the way out, and the screen redraws next frame as if the search had never happened.

Alongside those four, three more locals track the search itself: query (a fixed 256-byte buffer, KILO_QUERY_LEN) and qlen hold what you've typed; last_match is the row index of the most recent hit, or -1 for "no search has run yet against this query." That -1 does double duty as both "haven't searched" and "the query just changed, search fresh" — both Backspace and typing a new character reset it, so the very next loop iteration always re-searches rather than reusing a stale match.

Implementation: editorFind, start to finish

Here is the whole function. It's worth reading top to bottom once before the chapter picks it apart, because nothing here is hidden behind another call — this is Kilo's search, all of it:

#define KILO_QUERY_LEN 256

void editorFind(int fd) {
    char query[KILO_QUERY_LEN+1] = {0};
    int qlen = 0;
    int last_match = -1; /* Last line where a match was found. -1 for none. */
    int find_next = 0; /* if 1 search next, if -1 search prev. */
    int saved_hl_line = -1;  /* No saved HL */
    char *saved_hl = NULL;

#define FIND_RESTORE_HL do { \
    if (saved_hl) { \
        memcpy(E.row[saved_hl_line].hl,saved_hl, E.row[saved_hl_line].rsize); \
        free(saved_hl); \
        saved_hl = NULL; \
    } \
} while (0)

    /* Save the cursor position in order to restore it later. */
    int saved_cx = E.cx, saved_cy = E.cy;
    int saved_coloff = E.coloff, saved_rowoff = E.rowoff;

    while(1) {
        editorSetStatusMessage(
            "Search: %s (Use ESC/Arrows/Enter)", query);
        editorRefreshScreen();

        int c = editorReadKey(fd);
        if (c == DEL_KEY || c == CTRL_H || c == BACKSPACE) {
            if (qlen != 0) query[--qlen] = '\0';
            last_match = -1;
        } else if (c == ESC || c == ENTER) {
            if (c == ESC) {
                E.cx = saved_cx; E.cy = saved_cy;
                E.coloff = saved_coloff; E.rowoff = saved_rowoff;
            }
            FIND_RESTORE_HL;
            editorSetStatusMessage("");
            return;
        } else if (c == ARROW_RIGHT || c == ARROW_DOWN) {
            find_next = 1;
        } else if (c == ARROW_LEFT || c == ARROW_UP) {
            find_next = -1;
        } else if (isprint(c)) {
            if (qlen < KILO_QUERY_LEN) {
                query[qlen++] = c;
                query[qlen] = '\0';
                last_match = -1;
            }
        }

        /* Search occurrence. */
        if (last_match == -1) find_next = 1;
        if (find_next) {
            char *match = NULL;
            int match_offset = 0;
            int i, current = last_match;

            for (i = 0; i < E.numrows; i++) {
                current += find_next;
                if (current == -1) current = E.numrows-1;
                else if (current == E.numrows) current = 0;
                match = strstr(E.row[current].render,query);
                if (match) {
                    match_offset = match-E.row[current].render;
                    break;
                }
            }
            find_next = 0;

            /* Highlight */
            FIND_RESTORE_HL;

            if (match) {
                erow *row = &E.row[current];
                last_match = current;
                if (row->hl) {
                    saved_hl_line = current;
                    saved_hl = malloc(row->rsize);
                    memcpy(saved_hl,row->hl,row->rsize);
                    memset(row->hl+match_offset,HL_MATCH,qlen);
                }
                E.cy = 0;
                E.cx = match_offset;
                E.rowoff = current;
                E.coloff = 0;
                /* Scroll horizontally as needed. */
                if (E.cx > E.screencols) {
                    int diff = E.cx - E.screencols;
                    E.cx -= diff;
                    E.coloff += diff;
                }
            }
        }
    }
}

Walk it in three parts: what a keystroke does to the query, how a query becomes a match, and what a match does to the screen.

1. One keystroke, six possible outcomes

Every trip through the loop reads exactly one key and falls into exactly one branch. Delete/Backspace shrinks the query and invalidates last_match. Escape and Enter both leave — the difference is entirely in the four lines guarded by if (c == ESC), which restore the saved position; Enter skips that and simply accepts wherever the last match left the cursor. The four arrow keys don't touch the query at all — they just set find_next to 1 or -1 and let the search block below re-run from the current match in that direction. Anything else printable extends the query. That's the whole vocabulary; every other key you might press — Ctrl-S, Ctrl-Q, a Page Up/Down — simply falls through every else if and does nothing.

Escape rewinds. Enter keeps.

It's tempting to think of Escape and Enter as two spellings of "close the search box," but they leave you in different places. Enter accepts the match you're looking at as your new cursor position — the saved cx/cy/coloff/rowoff are simply discarded. Escape is the only key that writes them back. Search for something, land on a match, and press Enter: you're there. Press Escape instead, after the exact same search: you're back where you started, as if you'd never searched at all.

2. The search itself: a bounded, wrapping walk

Everything above only decides whether to search this iteration (find_next nonzero) and in which direction. The actual search is nine lines: start at last_match, step by find_next, and wrap around the ends of the file — current == -1 becomes the last row, current == E.numrows becomes row zero. The loop runs for (i = 0; i < E.numrows; i++), so it visits at most every row once and always terminates, whether or not the query matches anywhere. Each row is tested with one call: strstr(E.row[current].render, query).

Two things about that call are easy to assume wrong. First, strstr is a plain C standard-library substring search — case-sensitive, literal, no regular expressions, no wildcards, no whole-word matching. Second, it searches row->render, not row->chars. Chapter 08 drew that line carefully: chars is the raw bytes you typed or loaded, render is the tab-expanded copy built purely so columns line up on screen. Search runs against the on-screen version.

Search wraps — in both directions

Because current is explicitly wrapped rather than clamped, searching forward past the last line continues at line 0, and searching backward past line 0 continues at the last line. There's no "not found, stop at the end" message distinct from an ordinary miss — if the query isn't anywhere in the file, the loop simply runs its full E.numrows iterations, finds nothing, and match stays NULL. The cursor and screen just don't move that iteration; the prompt stays up, waiting for you to keep typing or give up.

3. Marking the match: borrow one row's highlight, briefly

A found match gets a visual cue: the matched span turns blue. Kilo reaches for the machinery it already has — row->hl, the per-character highlight-type array that Chapter 14 will cover in full — and repurposes it for exactly one iteration. HL_MATCH (value 8, the last entry in the highlight-type list) is defined for this single purpose, and editorSyntaxToColor() maps it to ANSI color 34, blue. Before overwriting anything, editorFind copies the matched row's current hl bytes into saved_hl, then memset()s HL_MATCH across just the matched span: memset(row->hl+match_offset, HL_MATCH, qlen).

The FIND_RESTORE_HL macro undoes exactly that — copies saved_hl back over row->hl and frees it. It's called twice: once at the top of every new match (so moving to match two restores match one's row before touching anything else), and once more on the way out through Escape or Enter (so no stray blue survives after the prompt closes). Notice the shape: saved_hl_line and saved_hl are single variables, not a stack. Kilo only ever remembers one patched row at a time, because only one match is ever highlighted at once.

Not a new highlighting feature — a one-row memset

It's easy to read HL_MATCH and assume the syntax highlighter has learned about search results as a category it understands. It hasn't. editorUpdateSyntax() (Chapter 14) never emits HL_MATCH itself; nothing about parsing keywords, strings, or comments knows search exists. editorFind just borrows the array those functions already fill, overwrites a few bytes of it directly, and puts them back. It's a display trick riding on top of the highlighter's storage, not a change to the highlighter.

Landing on a match: search moves the viewport, not just the cursor

Once a match is found, four assignments reposition the screen: E.cy = 0; E.cx = match_offset; E.rowoff = current; E.coloff = 0;. Recall from Chapter 09 that the row actually on screen is always E.rowoff + E.cy — so setting E.cy to zero and E.rowoff to the matched row's index doesn't just bring that row into view, it puts it on the very first visible screen line, every time, no matter where it was before. That's a genuinely different scrolling policy from editorMoveCursor, which only scrolls the minimum amount needed to keep the cursor somewhere on screen. Search doesn't nudge the viewport — it recenters it, hard, on every single match.

Callback to Chapter 09: a one-column difference from the established pattern

Right after repositioning, editorFind does its own horizontal-scroll check for a match that lands past the right edge: if (E.cx > E.screencols) { int diff = E.cx - E.screencols; E.cx -= diff; E.coloff += diff; }. Work through the arithmetic and E.cx ends up at exactly E.screencols. Compare that to editorMoveCursor's equivalent clamp, established back in Chapter 09: if (E.cx > E.screencols-1) { E.coloff = E.cx-E.screencols+1; E.cx = E.screencols-1; } — which lands one column earlier. Chapter 09 fixed [0, E.screencols) as the valid on-screen range for E.cx; this branch in editorFind can leave it one past that. It's a narrow inconsistency, only reachable when a match sits right at the screen's right edge — not a documented feature, just two independent pieces of clamping code that don't quite agree.

There's a second, quieter subtlety in that same block, worth having once you already know the render/chars split from Chapter 08. match_offset is computed against row->render — the tab-expanded string, where one tab byte can occupy up to eight screen columns. But E.cx, everywhere else it's used (the cursor-drawing loop inside editorRefreshScreen, for one), is counted against row->chars — the raw, un-expanded bytes, from E.coloff. For a line with no tabs before the match, chars and render agree byte-for-byte and nothing goes wrong. For a match on a line that does have a tab earlier in it, the two indices diverge, and E.cx ends up counted in the wrong unit for that row. It's a real gap, and a narrow one — it only shows up on tabbed lines, and Kilo's own source files rarely mix tabs with the kind of text you'd search for.

Edge cases: what editorFind quietly guarantees (and doesn't)

  • Search is fully modal. While editorFind's loop runs, editorProcessKeypress's normal switch never executes. You cannot save, move with Page Up/Down, or start editing while the search prompt is open — only the keys editorFind explicitly checks do anything.
  • An empty query always "matches" the current position. Before you've typed anything, last_match == -1 forces a forward search with query empty; strstr(anything, "") returns the haystack itself at offset 0, so the very first row scanned counts as a hit at column 0. Opening search and immediately pressing Enter jumps you to the top of whichever row the scan starts from.
  • No match anywhere leaves you exactly where you are. The scan loop always terminates after at most E.numrows steps; if nothing matches, match stays NULL, the cursor/viewport block is skipped entirely, and the prompt just keeps showing your query, waiting.
  • Search is case-sensitive and literal. It's strstr(), nothing more — no case folding, no regex metacharacters treated specially. A query of "." matches a literal period, not "any character."
  • The highlight patch never outlives the search. FIND_RESTORE_HL runs on both exit paths, so no row keeps a phantom HL_MATCH span after you leave search mode — whether you left by finding something (Enter) or backing out (Escape).
  • The query has a hard ceiling. KILO_QUERY_LEN is 256; typing past it is silently ignored (the qlen < KILO_QUERY_LEN guard just stops appending), not truncated with a warning or rejected with a bell.

So what did we actually gain?

Two ideas, and they're both small. First: a temporary UI mode doesn't need a flag threaded through the main dispatch loop — it can just be its own tiny event loop, called once, returning when it's done. Everything about "how do I get back to normal editing" falls out of that shape for free: normal editing resumes automatically the instant editorFind returns, because it's the same editorProcessKeypress call stack that invoked it in the first place. Second: state that a mode needs to undo is just data you save before you start and restore if you bail — here, four integers already defined by Chapter 09's navigation model, no new struct required. The search algorithm itself, strip away all of that scaffolding, is a single strstr() call in a wrapping loop. Everything genuinely interesting in this chapter was about the mode, not the matching.
