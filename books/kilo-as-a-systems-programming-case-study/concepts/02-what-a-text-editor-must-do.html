Chapter 02 · Part I — Orientation

What a Text Editor Must Do

Before we read a single line of kilo.c, it's worth asking a question that sounds too simple to bother with: what does any text editor actually have to do? Not Kilo specifically — Vim, nano, VS Code, a Notepad clone, all of them. Strip away the toolkit, the plugins, the theme, and every editor that has ever existed is doing the same five jobs. Kilo just does them in the open, in about a thousand lines, where you can watch it happen.

The whole program, in eight lines

Skip to the very bottom of kilo.c and you'll find main(). After one-time setup, it does exactly one thing, forever:

int main(int argc, char **argv) {
    initEditor();
    editorSelectSyntaxHighlight(argv[1]);
    editorOpen(argv[1]);
    enableRawMode(STDIN_FILENO);
    editorSetStatusMessage("HELP: Ctrl-S = save...");
    while(1) {
        editorRefreshScreen();
        editorProcessKeypress(STDIN_FILENO);
    }
}

Two function calls, looping. Refresh the screen, then process one keypress. Everything Kilo does — every feature in every chapter still ahead of us — happens because of what those two calls, and the state they share, do underneath.[1]

Motivation: name the shape before you build it

It's tempting to start reading kilo.c top to bottom and let the structure reveal itself function by function. That works, eventually — but it's slow, and it's easy to mistake an implementation detail for a fundamental requirement. Better to fix the skeleton first: five responsibilities every text editor must satisfy, in some form, or it isn't a text editor. Once you can name them, every function in Kilo sorts itself into one of five buckets, and the rest of this book is just: which bucket, how, and why this way and not another.

Here they are, plainly:

ResponsibilityThe question it answersKilo's answer, roughly
InputHow do keystrokes reach the program at all?Raw-mode read() of the terminal, one byte at a time
BufferWhat does “the text” live in, in memory?An array of erow structs, one per line
RenderHow does memory become pixels/glyphs the user sees?VT100 escapes, batched into one write per frame
NavigateWhere is the cursor, and what part of the buffer is visible?Cursor coordinates + a scroll offset into the row array
PersistHow does memory become a file, and back again?getline() in, one flattened write() out

Notice what's not on that list: syntax highlighting, search, undo, multiple buffers, a mouse. Those are real features, and Kilo has some of them — but they're built on top of the five above, not alongside them. A program that has all five in some minimal form is already a text editor. Everything else is enrichment.

Design: the read-react-repaint cycle

The main() loop above names the pattern that ties all five responsibilities together: read a keystroke, react to it, repaint the screen, repeat. It is worth being precise about what kind of loop this is, because it's simpler — and stricter — than you might expect from something that feels this responsive. editorProcessKeypress() calls a helper that blocks on a single read() of the terminal until exactly one keypress arrives, then dispatches on it with one large switch: distinct cases for Enter, Ctrl-Q (quit), Ctrl-S (save), Ctrl-F (find), the delete keys, the four arrow keys, page up/down, Ctrl-L, and Esc.[1] There is no timer. No idle callback. No background thread quietly reflowing text while you think. Kilo only does work in direct response to a key you pressed — which is exactly why its whole architecture can be a two-line loop instead of an event system.

That single design choice cascades outward. Because nothing happens between keystrokes, the buffer can never be “mid-update” when you look at it — every repaint sees a fully-settled state. Because the loop is synchronous, there's no race between “the user typed another key” and “the screen finished drawing”. Simplicity here isn't a missing feature; it's what makes the rest of the program simple enough to hold in your head, which was the entire premise of Chapter 01.

The buffer: what “the text” actually is

Every editor has to answer one hard question honestly: when you're editing a 50,000-line file and you insert one character on line 30,000, what actually moves in memory? The naive answer — one giant string, shift everything after the cursor — is correct but slow at scale. Over the decades editors have converged on a handful of standard answers, each trading simplicity for performance differently:

  • Array of lines. One string per line, in an array. Simple to reason about; cheap for small files; gets expensive when a structural edit has to shift many lines.
  • Gap buffer. One contiguous allocation with a movable empty “gap” sitting right at the cursor; typing just writes into the gap. Because real editing is usually clustered near wherever you last typed, a gap buffer's locality tends to beat a rope in practice on small-to-medium files.[4]
  • Rope. A balanced tree of text chunks. Every operation — insert, delete, even concatenating two huge documents — is O(log n) worst case, anywhere in the document. The cost is real implementation complexity, which only pays for itself at large scale or under non-local editing patterns like collaborative editing.[4]
  • Piece table. Never move the original bytes at all — index small, mostly-immutable “pieces” instead. Popular where cheap, unlimited undo matters as much as edit speed.

Kilo picks the first option, on purpose, and doesn't apologize for it: an array of erow structs, one per line of the file, that grows with realloc() as you insert rows. For a program built to teach raw terminal I/O in under a thousand lines, a gap buffer or rope would be scope creep — the array is the right amount of engineering for the job in front of it, and we'll feel exactly where that choice starts to cost something once we get to Chapter 10, Editing Text.

Look closer at one erow

Here's the actual struct, unmodified. Notice it holds two copies of a line's text, not one:

typedef struct erow {
    int idx;            /* Row index in the file, zero-based. */
    int size;           /* Size excluding null term. */
    int rsize;          /* Rendered row size (after TAB expansion). */
    char *chars;        /* Raw row content. */
    char *render;       /* Content rendered for display. */
    unsigned char *hl;  /* Syntax highlight type per character. */
    int hl_oc;          /* Open comment state flag. */
} erow;

chars is exactly what you typed or loaded from disk — the file's ground truth. render is a second, derived buffer with tabs expanded into spaces, rebuilt from chars after every edit, that exists purely so the screen has something byte-aligned to draw. Editing operates on chars; the screen only ever reads render. Keep that split straight now — it's the difference between “the file” and “what you see”, and later, hl (one highlight-color byte per rendered character) hangs off render too, not chars.[1]

Rendering: memory becomes a screen, in one write

“Render” is easy to picture wrong. It's tempting to imagine editorRefreshScreen() writing to the terminal as it goes — draw this row, then that row, then the status bar. It doesn't. It hides the cursor, walks every visible row applying whatever highlight colors that row currently has, writes the status bar and a message line, and repositions the cursor — but every one of those steps only appends escape sequences and text to an in-memory buffer (Kilo calls it struct abuf, an append buffer). Only after the whole frame is assembled does the function make a single write() syscall to flush it to the terminal.[1]

That batching is not incidental polish — it's the difference between a screen that redraws cleanly and one that visibly flickers. Many small writes to a terminal each cost a round-trip through the kernel and the terminal emulator; if the screen were cleared with one write and repainted with a hundred more, you'd see the blank flash between them. One write, one frame, no flicker. We'll build this exact append-buffer pattern by hand in Chapter 05.

Persistence: the buffer is not the file

Opening and saving are mirror images of each other, and Kilo keeps them almost suspiciously simple. editorOpen() reads the target file line by line with getline(), strips the trailing newline, and calls editorInsertRow() once per line to build up the erow array — a file that doesn't exist yet (ENOENT) is tolerated and just starts an empty buffer, which is how kilo newfile.txt works at all. Saving reverses it: editorRowsToString() flattens every row back into one contiguous byte buffer, and editorSave() writes that buffer to disk.[1]

Read that save path closely, though, because it's a genuinely instructive piece of honesty from the source itself:

int fd = open(E.filename,O_RDWR|O_CREAT,0644);
if (fd == -1) goto writeerr;

/* Use truncate + a single write(2) call in order to make saving
 * a bit safer, under the limits of what we can do in a small editor. */
if (ftruncate(fd,len) == -1) goto writeerr;
if (write(fd,buf,len) != len) goto writeerr;

That comment is antirez talking directly to future readers of his own code. There's no write-to-a-temp-file-then-rename here — the industry-standard way to make a save crash-safe — just a truncate and a direct write to the real file, with an honest label on the trade-off: a bit safer, under the limits of what we can do in a small editor. If write() fails partway through, you can be left with a shorter file than you started with. That's not a bug we're going to pretend isn't there; it's a limitation we'll come back to by name in Chapter 17 (Error Handling and Cleanup) and Chapter 18 (Portability and Limitations), once we've built the machinery to talk about it precisely.

So what did we actually get out of this?

A map, before the territory. Input, buffer, render, navigate, persist — five jobs, one blocking read-react-repaint loop tying them together, and a buffer that deliberately keeps “what you typed” (chars) separate from “what's drawn” (render). Every chapter from here on is one of those five buckets, built out in real C. When you hit a function later and aren't sure why it exists, come back to this table — it almost always answers the question.
