Chapter 01 · Part I — Orientation

Why Build Kilo?

Most programs you'll ever read hide their interesting decisions behind years of accreted layers — frameworks on top of libraries on top of runtimes. Kilo doesn't have that luxury. It is a complete, usable text editor — open a file, move around it, edit it, search it, save it, get a little syntax color back — in about a thousand lines of plain C. Small enough to hold the whole thing in your head at once. That's exactly the point, and exactly why we're going to take it apart.

Straight from the source

Open kilo.c and the very first lines tell you its own goal, unprompted — before a single line of logic:

/* Kilo -- A very simple editor in less than 1-kilo lines of code (as counted
 *         by "cloc"). Does not depend on libcurses, directly emits VT100
 *         escapes on the terminal. */

Motivation: a whole program beats a pile of fragments

Kilo was written by Salvatore Sanfilippo — antirez, better known for Redis — as a for-fun weekend-scale exercise, around the time the wider community was talking about the lineage of GNU nano. He didn't set out to replace anything. He set out to see how small a real editor could be, and he built it fast: by his own account in the project's README, Kilo was written in just a few hours taking code from my other two projects, load81 and linenoise — reused building blocks, not a blank-page marathon.[1]

That's worth sitting with. Most of us learn systems programming from fragments: a raw-mode snippet here, a malloc example there, a gist that reads a file but never writes one back. Fragments don't have to survive contact with each other. Kilo does — every function in it exists because a real editor needed it to exist, and every design choice has to cooperate with every other one still running in the same process. Reading (and rebuilding) a complete, working program is a different kind of learning than reading isolated examples: you see how raw terminal input, screen redrawing, a text buffer, and file I/O actually fit together under one roof, because in Kilo they have no choice but to.

antirez said as much when he released it — Kilo's README explicitly invites this: People are encouraged to use it as a starting point to write other editors or command line interfaces that are more advanced than the usual REPL style CLI.[1] That's this book's whole premise. We're taking him up on it.

What “small” actually means here

“Less than 1K lines” is a real number, but it hides a detail worth being precise about, because it's the kind of thing a careful reader should notice and a sloppy one won't. The kilo.c file you'll clone is 1,308 lines long, license header, comments, and blank lines included — cloc's code-only count, which drops comments and blank lines before counting, is what lands under a thousand. The name is a pun on that stricter number, not the raw file size. It's a small nuance, but it's the first of many places in this book where being exact about what a claim actually measures will save you from a wrong mental model later.

The more consequential design decision is what Kilo refuses to depend on. No ncurses. No curses of any kind. No third-party anything. Every #include in the file is a standard POSIX/C header — termios.h, unistd.h, sys/ioctl.h, fcntl.h, signal.h, and a handful more.[1] That's not minimalism for its own sake — it's the whole pedagogical point. A curses-backed editor hides the terminal from you behind a library that already solved raw mode, cursor movement, and screen redraws. Kilo can't hide any of that, because it is the code that solves it. Every escape sequence you'll see later in this book, Kilo writes itself.

Set your expectations

This isn't a book about building a production editor. Kilo has three keybindings worth memorizing up front — Ctrl-S save, Ctrl-Q quit, Ctrl-F find — and it stays that spare on purpose.[1] No undo, no clipboard, no multiple buffers, no Windows console support — it assumes a POSIX terminal that understands VT100-style escapes. Parts VI and VII exist precisely to name those limits honestly and show you what growing past them would actually cost, rather than pretend the tiny version was always going to be the finished one.

Who this is for

You already know how to code. What you may not have done yet is write software that talks to the terminal directly instead of through printf and scanf, manage a growing buffer by hand instead of reaching for a container library, or read every line of a real, shipped program instead of a curated excerpt. If you're the kind of programmer who wants to build genuinely useful software — not just pass a course — that gap is worth closing, and a thousand lines of working C is a small enough investment to close it completely.

So what do we actually get out of this?

By the end of this book you won't just know what raw terminal mode, VT100 escapes, and a row-based text buffer are — you'll have read the exact lines of C that implement all three, in a program small enough that nothing is left unexplained. That's a rarer thing than it sounds. Most tutorials show you code; Kilo lets you own an entire codebase.

Before you turn the page
  1. Clone antirez/kilo and open kilo.c in whatever editor you're using right now. Scroll from top to bottom once, slowly. Don't try to understand it — just notice its shape: how many functions, how many sections, where the main() loop lives.
  2. Count the #include lines. Every one of them is a standard header. Ask yourself what each one probably gives the program, before you're told.
  3. Run cloc kilo.c if you have it installed, and compare its number to wc -l kilo.c. That gap is the lesson from this chapter, made concrete.
