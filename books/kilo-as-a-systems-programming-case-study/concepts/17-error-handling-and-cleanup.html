Chapter 17 · Part VI — Robustness

Error Handling and Cleanup

Every chapter so far has been about kilo.c doing something on purpose: reading a key, drawing a screen, coloring a token. This chapter is about what happens when something goes wrong, or when the user just wants out. Raw mode (Chapter 4) put the terminal into a state no shell expects to inherit — ECHO off, ICANON off, signals disabled. If Kilo exits without putting that back, the next thing you type into your shell prompt won't echo, won't line-buffer, and will look like your terminal is broken. So before we can call Kilo "done," we owe it one honest question: on every single way this program can stop running, does the terminal get put back the way it found it?

Motivation: a program has more exits than a while loop

Look at the shape of main() and it's tempting to think Kilo has exactly one way to end: the user hits Ctrl-Q, the while(1) loop breaks, the function returns. It doesn't work that way, and it was never going to. Real programs die from bad arguments, missing files, I/O errors, and the occasional line of source code so long it overflows an internal buffer — and every one of those is a legitimate reason for kilo.c to call exit() from somewhere deep inside a helper function, nowhere near main. Count them and there are seven separate call sites across kilo.c that can end the process: a bad argument count, two different file-I/O failures, a read error on the terminal itself, an over-long line, a window-size query failure, and the ordinary Ctrl-Q quit. Each one needs the same answer to the same question — is the terminal still in raw mode right now, and if so, who's putting it back?

Don't reach for a die() helper

If you've read the popular snaptoken walkthrough of Kilo, you may remember a single die() function that every error path calls. That's a teaching simplification from that tutorial, not what antirez's actual kilo.c does. The real source has no such function — instead each call site handles its own failure inline, with perror() and exit(), or a local goto label for its own cleanup. It's less uniform to read, but it's the real code, so that's what this chapter walks through.

Design: one flag, one atexit, one restore function

Kilo's whole cleanup story rests on three small pieces working together. First, a place to remember what the terminal looked like before Kilo touched it — a file-scope struct termios that Chapter 4's enableRawMode already fills in with tcgetattr. Second, a function that puts it back. Third, a way to guarantee that function actually runs, no matter which of those seven call sites is the one that ends up calling exit(). Here's the first two pieces, read whole:

static struct termios orig_termios; /* In order to restore at exit.*/

void disableRawMode(int fd) {
    /* Don't even check the return value as it's too late. */
    if (E.rawmode) {
        tcsetattr(fd,TCSAFLUSH,&orig_termios);
        E.rawmode = 0;
    }
}

/* Called at exit to avoid remaining in raw mode. */
void editorAtExit(void) {
    disableRawMode(STDIN_FILENO);
}

disableRawMode guards itself with E.rawmode, the same flag enableRawMode sets to 1 on success — call it when raw mode was never entered, or was already turned off, and it's a no-op. And notice what it doesn't do: it doesn't check whether tcsetattr succeeded. That comment isn't laziness, it's an honest admission. This call only ever runs while the process is already on its way out. If restoring the terminal fails here, there is no next step to take — no retry, no fallback, nowhere left in the program to report the failure to. Checking a return value only matters if you have something different to do with the two outcomes; here you don't, so the code doesn't pretend to.

editorAtExit is the thin wrapper that makes disableRawMode callable with the exact signature atexit() requires — void (*)(void), no arguments. It's one line, and it exists purely so the C runtime has something to call for you.

Implementation: atexit() is the guarantee, and it's registered just in time

Here's where the guarantee actually gets made — inside enableRawMode itself, the same function Chapter 4 walked through for what it does to the terminal flags:

int enableRawMode(int fd) {
    struct termios raw;

    if (E.rawmode) return 0; /* Already enabled. */
    if (!isatty(STDIN_FILENO)) goto fatal;
    atexit(editorAtExit);
    if (tcgetattr(fd,&orig_termios) == -1) goto fatal;

    raw = orig_termios;
    /* ...clear ECHO, ICANON, ISIG, IXON, and friends (Chapter 4)... */

    if (tcsetattr(fd,TCSAFLUSH,&raw) < 0) goto fatal;
    E.rawmode = 1;
    return 0;

fatal:
    errno = ENOTTY;
    return -1;
}

Read the order of operations carefully, because it's the whole design in three lines. atexit(editorAtExit) runs before tcgetattr and tcsetattr — the two calls in this function that can actually fail and jump to fatal. That's deliberate. Once enableRawMode has committed to trying, the cleanup handler is armed for the rest of the process's life, whether this particular attempt succeeds or not. From this point forward, calling exit() from anywhere in the program — not just from this function, not just from this file's obvious error paths, but from any function that happens to detect a fatal condition ten call frames later — will run editorAtExit first. That's the entire trick: register the safety net once, early, and then every unrelated piece of code that calls exit() gets the benefit without having to know the net exists.

exit() is not the only way a process ends

atexit()-registered functions run when the process terminates normally — a call to exit(), or falling off the end of main. They do not run if the process is killed by an unhandled signal: a real SIGSEGV from a memory bug, or a SIGKILL from outside the process, both skip editorAtExit entirely. Since kilo.c's malloc/realloc calls never check for a NULL return (more on that below), a genuine out-of-memory fault is exactly the kind of bug that could leave your terminal stuck in raw mode with no cleanup run at all — the reason stty sane or reset exists as an escape hatch independent of the program that broke it.

Edge case: the exit that happens before raw mode ever starts

Now look at main() again, specifically the order it calls things in:

int main(int argc, char **argv) {
    if (argc != 2) {
        fprintf(stderr,"Usage: kilo <filename>\n");
        exit(1);
    }

    initEditor();
    editorSelectSyntaxHighlight(argv[1]);
    editorOpen(argv[1]);
    enableRawMode(STDIN_FILENO);
    editorSetStatusMessage(
        "HELP: Ctrl-S = save | Ctrl-Q = quit | Ctrl-F = find");
    while(1) {
        editorRefreshScreen();
        editorProcessKeypress(STDIN_FILENO);
    }
    return 0;
}

Three of Kilo's seven exit points sit above enableRawMode in this call chain: the argument-count check right here, the window-size query inside initEditor, and a file-open failure inside editorOpen. None of those three register or rely on editorAtExit, and none of them need to — at the moment any of them could fire, enableRawMode hasn't run yet, so the terminal is still in whatever cooked, normal state your shell left it in. There is nothing to restore, because nothing has been changed. This is the payoff of putting atexit(editorAtExit) exactly where it sits, rather than at the very top of main: the guarantee only needs to cover the exit points that can happen after raw mode is actually on, and the source's own function ordering keeps every before-raw-mode exit provably safe by construction, not by an extra check anyone had to remember to add.

Here's editorOpen's fatal path, one of those three:

int editorOpen(char *filename) {
    FILE *fp;
    /* ...set E.filename... */

    fp = fopen(filename,"r");
    if (!fp) {
        if (errno != ENOENT) {
            perror("Opening file");
            exit(1);
        }
        return 1;
    }
    /* ...getline() loop building rows... */
    return 0;
}

Notice the two outcomes are treated completely differently, on purpose. A missing file (errno == ENOENT) is not an error at all — it's the ordinary case of running kilo notes.txt against a file you haven't created yet. editorOpen just returns 1 and lets the editor open on an empty buffer, ready to create that file on save. Anything else — permission denied, a directory instead of a file, whatever else fopen can fail on — is treated as genuinely fatal: report it with perror, which prints the message plus a human-readable strerror of errno, and exit. Since this call happens before enableRawMode, that exit(1) drops the user straight back to a perfectly normal shell prompt with a one-line explanation on stderr — no raw-mode cleanup required, none skipped either.

Implementation: the exits that happen after raw mode is on

The remaining four exit points all happen from inside the main loop, after enableRawMode has already succeeded and armed editorAtExit. First, a hard read error on the terminal itself, inside editorReadKey:

int editorReadKey(int fd) {
    int nread;
    char c, seq[3];
    while ((nread = read(fd,&c,1)) == 0);
    if (nread == -1) exit(1);
    /* ...escape-sequence parsing (Chapter 5)... */
}

The while loop spins on a read returning exactly 0 — the VTIME timeout with no byte available, from Chapter 4's VMIN=0, VTIME=1 setup, which is completely normal and just means "no key yet." A return of -1 is different: that's a real error on the file descriptor, and Kilo doesn't try to interpret which one — it exits immediately. Because this line can only run once the editor's main loop is going, raw mode is already on, and this exit(1) is exactly the case the earlier atexit registration was for: it runs editorAtExit before the process actually dies, and the terminal comes back.

Second, the ordinary way out — Ctrl-Q, gated by a small quit-confirmation counter:

#define KILO_QUIT_TIMES 3
/* inside editorProcessKeypress(int fd): */
static int quit_times = KILO_QUIT_TIMES;
/* ... */
case CTRL_Q:        /* Ctrl-q */
    /* Quit if the file was already saved. */
    if (E.dirty && quit_times) {
        editorSetStatusMessage("WARNING!!! File has unsaved changes. "
            "Press Ctrl-Q %d more times to quit.", quit_times);
        quit_times--;
        return;
    }
    exit(0);
    break;
/* ...at the very end of editorProcessKeypress: */
quit_times = KILO_QUIT_TIMES; /* Reset it to the original value. */

quit_times is a function-local static, so it persists between calls without being part of the global editor state. If the buffer is unmodified (E.dirty == 0), Ctrl-Q exits on the first press, same as always. If it's dirty, the first two presses just count down and warn; only the third Ctrl-Q in a row actually calls exit(0). The "in a row" part matters: that last line resets quit_times to 3 after every keypress that reaches it, so pressing Ctrl-Q twice, typing one character, then pressing Ctrl-Q again starts the countdown over from three, not from one. And Ctrl-C gets its own explicit case, right above this one, that does nothing at all:

case CTRL_C:        /* Ctrl-c */
    /* We ignore ctrl-c, it can't be so simple to lose the changes
     * to the edited file. */
    break;

Raw mode's ISIG flag being off (Chapter 4) already means the terminal driver won't turn Ctrl-C into a SIGINT before Kilo even sees it — it arrives as an ordinary byte, like any other key. This case is what happens to that byte: nothing, silently, so a stray Ctrl-C out of muscle memory can never be the thing that throws away unsaved work.

Third, an over-long line during rendering, inside editorUpdateRow:

unsigned long long allocsize =
    (unsigned long long) row->size + tabs*8 + nonprint*9 + 1;
if (allocsize > UINT32_MAX) {
    printf("Some line of the edited file is too long for kilo\n");
    exit(1);
}
row->render = malloc(row->size + tabs*8 + nonprint*9 + 1);

This one is defending against a specific, narrow failure mode: the render buffer's size is computed by widening every term to unsigned long long first, precisely so the overflow check itself can't overflow, and only then compared against UINT32_MAX. Without that check, a pathological line (megabytes of tabs, say) could compute a size that wraps around back to something small once it's actually passed to malloc, which would allocate far less memory than the code goes on to write into. Refusing to proceed and exiting is the honest choice here — better than silently corrupting memory on a line no reasonable source file should ever contain.

Fourth, a window-size query failure, which can be reached either once at startup or again on every resize, via SIGWINCH:

void updateWindowSize(void) {
    if (getWindowSize(STDIN_FILENO,STDOUT_FILENO,
                      &E.screenrows,&E.screencols) == -1) {
        perror("Unable to query the screen for size (columns / rows)");
        exit(1);
    }
    E.screenrows -= 2; /* Get room for status bar. */
}

void handleSigWinCh(int unused __attribute__((unused))) {
    updateWindowSize();
    if (E.cy > E.screenrows) E.cy = E.screenrows - 1;
    if (E.cx > E.screencols) E.cx = E.screencols - 1;
    editorRefreshScreen();
}

void initEditor(void) {
    /* ...zero E.cx, E.cy, E.rowoff, E.numrows, etc... */
    updateWindowSize();
    signal(SIGWINCH, handleSigWinCh);
}

initEditor calls updateWindowSize once, directly, and only then installs the SIGWINCH handler that will call it again on every terminal resize for the rest of the session. That first call happens before enableRawMode, in the same "nothing to restore yet" window as the argument check and editorOpen's fatal path. But SIGWINCH can fire again at any point after that — the user resizes their terminal emulator mid-session, well after raw mode is on — and if getWindowSize fails on that later call, the same exit(1) runs with editorAtExit already registered, so cleanup still happens correctly either way. Same function, same failure branch, safe under both timings — not because anyone special-cased the signal-handler call, but because the atexit registration doesn't care who calls exit() or when.

A return value nobody checks

Go back to main(): enableRawMode(STDIN_FILENO); is called as a bare statement. The function's int return value — 0 on success, -1 if isatty, tcgetattr, or tcsetattr failed — is silently discarded. If raw mode setup fails (stdin redirected from a file instead of a real terminal, say), the editor doesn't stop or warn; it proceeds straight into the main loop as though everything worked. editorReadKey then reads from a terminal still in canonical mode — line-buffered, still echoing, still delivering Ctrl-C as SIGINT — and every symptom of that looks like a bug in Kilo's input handling rather than what it actually is: a setup failure nobody was told about. It's the one real gap in an otherwise careful chain, worth remembering the next time a program's own error code goes unchecked at its only call site.

Design: saving fails softly, on purpose

Not every error in kilo.c ends the process. editorSave is the clearest counter-example — a failure here is reported to the user and handed back as a return code, never exit()ed:

int editorSave(void) {
    int len;
    char *buf = editorRowsToString(&len);
    int fd = open(E.filename,O_RDWR|O_CREAT,0644);
    if (fd == -1) goto writeerr;

    /* Use truncate + a single write(2) call in order to make saving
     * a bit safer, under the limits of what we can do in a small editor. */
    if (ftruncate(fd,len) == -1) goto writeerr;
    if (write(fd,buf,len) != len) goto writeerr;

    close(fd);
    free(buf);
    E.dirty = 0;
    editorSetStatusMessage("%d bytes written on disk", len);
    return 0;

writeerr:
    free(buf);
    if (fd != -1) close(fd);
    editorSetStatusMessage("Can't save! I/O error: %s",strerror(errno));
    return 1;
}

This is the same idea as enableRawMode's goto fatal, reused for a very different situation: one shared cleanup block that every failing step jumps to, instead of duplicating the same free/close pair at each if. But the ending is different on purpose. There's no unsaved-terminal-state problem to worry about here — raw mode isn't touched by any of this — so there's no reason to kill the whole editing session over a disk error. Instead writeerr frees the buffer, closes the descriptor if it was ever opened, and reports the failure the same way any other status message appears, through editorSetStatusMessage with strerror(errno) spelling out exactly what went wrong. The user keeps their in-memory edits (E.dirty stays set) and can just try Ctrl-S again.

"A bit safer," not fully safe — read the comment literally

The comment above the ftruncate/write pair says exactly what it is: safer than the alternative, not bulletproof. ftruncate shortens the file to the new length before write puts the new bytes in, so if the write fails partway — a full disk, an ENOSPC — the file on disk can end up shorter than either the version you started with or the version you meant to save. A more defensive editor writes to a fresh temporary file and rename()s it over the original only once the write fully succeeds, so a failed save can never corrupt the file that was already there. Kilo doesn't do that — it favors the simpler, more readable two-syscall version, and reports honestly when it isn't enough. That's a tradeoff a ~1000-line editor is allowed to make; it's not one a program guarding files you can't afford to lose should copy without noticing.

Edge case: the failure mode Kilo doesn't handle at all

Scan back through every snippet in this chapter and one thing is conspicuously absent: nowhere does kilo.c check whether malloc or realloc returned NULL. Not in editorInsertRow, not in the append buffer, not in editorOpen's E.filename allocation, not in editorRowsToString's save buffer. Every allocation in this program is written as though it cannot fail.

On a modern Linux system with memory overcommit, that assumption is usually harmless in practice — the kernel will hand out virtual address space it doesn't back with real memory until you touch it, and an editor working on ordinary source files is unlikely to ever ask for enough to matter. But "usually harmless" is not "handled." If an allocation genuinely did fail, the very next line dereferences a NULL pointer, the process receives SIGSEGV, and — per the callout earlier in this chapter — a signal-based death skips every atexit() handler, including editorAtExit. That's the one path in this whole chapter where the terminal really can get left in raw mode with nothing left to fix it automatically.

Naming this clearly beats pretending it isn't there. A ~1000-line editor built for reading, not for running unattended on a server with someone else's data, can reasonably decide that a NULL check on every one of a dozen or so allocation sites isn't worth the line count it would add. Whether that's the right call depends entirely on where the program runs and what happens if it's wrong — which is exactly the kind of judgment call Chapter 18's look at Kilo's portability and limitations will keep making, one assumption at a time.

So what did we actually gain?

A precise accounting of every way kilo.c's process can end, and proof — not a hand-wave, an actual trace through the source — that raw mode is restored on all of them. The trick was never complicated: one flag (E.rawmode) guards a restore function that doesn't bother checking its own return value, one atexit() call arms that function for the rest of the process's life, and that registration is placed exactly where it needs to be — inside enableRawMode, before the calls that could fail — so it's live for every exit that can happen once raw mode is actually on, while every exit that can happen before that is safe for the much simpler reason that there's nothing yet to clean up. What this chapter also surfaced, by reading the code instead of assuming it, is where that guarantee has real edges: enableRawMode's own failure return is silently ignored by its only caller, editorSave's two-syscall write can leave a file shorter than either version if it fails midway, and every unchecked allocation is one out-of-memory condition away from a signal death that atexit() can't reach at all. None of these are secret bugs — they're the visible, honest shape of the tradeoffs a program this size makes on purpose.
