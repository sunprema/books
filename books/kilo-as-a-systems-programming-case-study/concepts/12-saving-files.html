Chapter 12 · Part IV — Files

Saving Files

Chapter 11 ended with a promise: this chapter is the mirror image of that one. Opening turns disk bytes into an array of erow structs, one editorInsertRow() call per line. Saving turns that array back into disk bytes, one flatten-and-write. Same data, opposite direction — and, true to Kilo's style so far, almost no new machinery. The interesting part isn't how the bytes get written. It's the handful of small, deliberate decisions around when they get written, and what the editor remembers about whether they need to be.

Motivation: a save is a flatten, then a write

Everything Kilo knows about your file lives as an array of rows, each one holding its own raw bytes in erow.chars. A file on disk, though, is just one contiguous stream of bytes with newlines in it. Saving has to bridge that gap before a single byte reaches the filesystem, and Kilo does it as two separate, small functions: one that flattens the rows into a single heap buffer, and one that writes that buffer to disk. Keeping them separate is itself a small design choice worth noticing — the flattening logic doesn't know or care about file descriptors, and the writing logic doesn't know or care about rows.

Design: flatten first, ask questions never

editorRowsToString() does the flattening. It walks E.row[] twice: once to add up every row's size (plus one byte per row, for a newline) into a total length, and once to actually copy each row's raw bytes into a freshly malloc'd buffer, writing a literal '\n' after every row as it goes — including the last one.

/* Turn the editor rows into a single heap-allocated string.
 * Returns the pointer to the heap-allocated string and populate the
 * integer pointed by 'buflen' with the size of the string, escluding
 * the final nulterm. */
char *editorRowsToString(int *buflen) {
    char *buf = NULL, *p;
    int totlen = 0;
    int j;

    /* Compute count of bytes */
    for (j = 0; j < E.numrows; j++)
        totlen += E.row[j].size+1; /* +1 is for "\n" at end of every row */
    *buflen = totlen;
    totlen++; /* Also make space for nulterm */

    p = buf = malloc(totlen);
    for (j = 0; j < E.numrows; j++) {
        memcpy(p,E.row[j].chars,E.row[j].size);
        p += E.row[j].size;
        *p = '\n';
        p++;
    }
    *p = '\0';
    return buf;
}

Two things worth locking in before moving on to the write itself. First, it copies row.chars — the raw bytes, exactly as typed or loaded — never row.render, the tab-expanded copy Chapter 08 built purely so tabs line up on screen. A tab character you typed is a tab character on disk; the expansion to spaces you see in the terminal never touches what gets saved. Second, that '\n' goes after every row, unconditionally, including the last one in the file. A file that had no trailing newline when you opened it will have one the moment you save it. Kilo's save is faithful to your edits, not to the exact original bytes on disk.

Callback to Chapter 11: the stray \r round-trips by accident

Opening a CRLF file leaves a stray '\r' baked onto the end of row.chars, because the load-time strip only ever removes the final '\n'. Now look at what happens on save: editorRowsToString copies that '\r'-terminated chars buffer verbatim, then appends its own '\n' right after it — reconstructing "...\r\n". The CRLF line ending survives the round trip. But don't credit either function with CRLF support: neither one was written with CRLF in mind, and nothing else in the program — cursor movement, column math, syntax highlighting — treats that trailing '\r' as anything but an ordinary character. It's two narrow, unrelated pieces of code happening to compose correctly, not a feature.

Implementation: editorSave, in full

With a flat buffer in hand, editorSave() does the actual disk write — nineteen lines, one goto, and a comment that tells you exactly what trade-off it's making:

/* Save the current file on disk. Return 0 on success, 1 on error. */
int editorSave(void) {
    int len;
    char *buf = editorRowsToString(&len);
    int fd = open(E.filename,O_RDWR|O_CREAT,0644);
    if (fd == -1) goto writeerr;

    /* Use truncate + a single write(2) call in order to make saving
     * a bit safer, under the limits of what we can do in a small editor. */
    if (ftruncate(fd,len) == -1) goto writeerr;
    if (write(fd,buf,len) != len) goto writeerr;

    close(fd);
    free(buf);
    E.dirty = 0;
    editorSetStatusMessage("%d bytes written on disk", len);
    return 0;

writeerr:
    free(buf);
    if (fd != -1) close(fd);
    editorSetStatusMessage("Can't save! I/O error: %s",strerror(errno));
    return 1;
}

Walk it in the order it actually runs.

1. Open for read-write, create if missing — but don't truncate yet

open(E.filename, O_RDWR|O_CREAT, 0644) either finds the existing file or creates a new one at that path with permissions 0644 (rw-r--r--, filtered through the process's umask like any newly created file — a requested ceiling, not a guarantee). Look closely at the flags: there is no O_TRUNC. That's not an omission; it sets up the next line to matter.

2. ftruncate does the job O_TRUNC would have done — explicitly

Because the file wasn't opened with O_TRUNC, it still has whatever length it had before — possibly longer than the new content, if you deleted text since you last saved. ftruncate(fd, len) resizes it to exactly len bytes, the length editorRowsToString just computed. Skip this call and a shorter save over a longer old file would leave stale bytes from the previous version sitting past the end of your new content — ftruncate here isn't a safety margin, it's load-bearing.

3. One write() call, and the comment that explains why

The whole buffer goes out in a single write(fd, buf, len), checked against len to catch a short write. Kilo's own comment, sitting right above these two calls, says exactly what this buys you: “Use truncate + a single write(2) call in order to make saving a bit safer, under the limits of what we can do in a small editor.” Read that carefully — it says a bit safer, not safe.

Not crash-atomic — a named, deliberate limit

One write() call narrows the window in which a crash, power loss, or killed process mid-save can corrupt your file, compared to writing lots of small chunks. It does not close that window. If the process dies between ftruncate and the completion of write, the file on disk can be left truncated or half-written — worse off than before you saved. The genuinely crash-safe pattern is different: write the new content to a temporary file, then rename() it over the original. A rename that replaces an existing file is atomic at the filesystem level on POSIX — the old file is either fully there or fully replaced, never half of either. Kilo doesn't do that. It's a real limitation, named in the source itself, in keeping with the size budget this whole book keeps coming back to — and it's exactly the kind of trade-off Chapter 18 (Portability and Limitations) will revisit.

4. One failure path, reached three different ways

Three separate checks — fd == -1 (open failed), ftruncate(...) == -1, and write(...) != len — all goto the same writeerr label. That's one cleanup path instead of three copies of it: free the buffer, close the fd if it's actually open (if (fd != -1) — needed precisely because the open-failed branch lands here too, and closing -1 would itself be an error), report strerror(errno) in the status bar, and return 1. Notice what doesn't happen on this path: E.dirty is untouched. A failed save correctly leaves the buffer marked modified, because it still is.

How a keystroke becomes a save

All of the above is reached through exactly one line in editorProcessKeypress():

    case CTRL_S:        /* Ctrl-s */
        editorSave();
        break;

No confirmation dialog, no filename prompt, no branch for saving under a different name. Kilo has no “Save As” — editorSave() always writes to E.filename, the exact path editorOpen() recorded back in Chapter 11. And just like editorOpen's own return value in that chapter, editorSave()'s int result is computed carefully (0 success, 1 error) and then thrown away here, its only caller. Every bit of feedback you actually get about whether a save worked — "%d bytes written on disk" or "Can't save! I/O error: %s" — comes from the status message editorSave sets on its own way out, not from anything editorProcessKeypress does with the number it gets back.

Dirty-state tracking: one flag, read in three places

E.dirty is the whole mechanism. It starts at 0, climbs by one on every mutating row operation (typing, deleting, splitting a line — the editing chapter's territory), and this chapter adds its one and only reset: E.dirty = 0, reached exactly once, on editorSave()'s success path. Three places downstream read that single flag, and each does something different with it:

ReaderWhat it does with E.dirty
editorRefreshScreen()Status bar formats E.dirty ? "(modified)" : "" next to the filename, every single frame — the visible, always-current indicator.
editorProcessKeypress() — CTRL_QChecks truthiness only: if (E.dirty && quit_times) gates the “unsaved changes” warning below.
editorFileWasModified()return E.dirty; — a tiny wrapper defined near the bottom of kilo.c that is never actually called anywhere else in the file. Declared, never consulted — the same shape as editorOpen's unread return value.

Quitting dirty: a throttle, not a guard

Kilo does not stop you from quitting with unsaved changes. It makes you press Ctrl-Q a few extra times:

#define KILO_QUIT_TIMES 3
void editorProcessKeypress(int fd) {
    static int quit_times = KILO_QUIT_TIMES;
    ...
    case CTRL_Q:        /* Ctrl-q */
        /* Quit if the file was already saved. */
        if (E.dirty && quit_times) {
            editorSetStatusMessage("WARNING!!! File has unsaved changes. "
                "Press Ctrl-Q %d more times to quit.", quit_times);
            quit_times--;
            return;
        }
        exit(0);
        break;
    ...
    }
    quit_times = KILO_QUIT_TIMES; /* Reset it to the original value. */
}

Trace it slowly, because the mechanism is smaller than the warning message makes it feel. quit_times is a static local — it survives between calls to editorProcessKeypress, but there's exactly one of it in the whole program, seeded to KILO_QUIT_TIMES (3). While the buffer is dirty and quit_times is still positive, Ctrl-Q just warns and counts down, then returns — the editor keeps running. Once quit_times reaches 0 (three consecutive Ctrl-Q presses), or the buffer isn't dirty at all, the very next Ctrl-Q falls straight through to exit(0). No save happens on the way out — if you never pressed Ctrl-S, three Ctrl-Q presses later your edits are simply gone.

The countdown only tracks consecutive presses

The line at the very bottom of editorProcessKeypress — quit_times = KILO_QUIT_TIMES; — runs after every keypress that falls through the switch without returning early. Type one character between two Ctrl-Q presses and the counter resets to 3. The “three more times” warning is a countdown of consecutive Ctrl-Q's, not a running total across your whole editing session.

Edge cases: what editorSave quietly guarantees (and doesn't)

  • A failed save leaves E.dirty untouched. Every writeerr exit skips the reset, so the “(modified)” marker keeps showing after a failed save — correctly, since the disk copy still doesn't match memory.
  • No Save As, ever. There is no code path in kilo.c for writing to any path other than E.filename. Renaming a file means renaming it outside the editor.
  • Tabs save as tabs. The saved bytes come from row.chars, never the space-expanded row.render — what you see on screen after tab expansion is not what lands on disk.
  • A missing trailing newline gets added. editorRowsToString appends '\n' after every row, including the last, unconditionally — save is not a byte-for-byte echo of what was on disk before.
  • Quitting dirty is throttled, not blocked. Three consecutive Ctrl-Q presses (or one, if the buffer isn't dirty) always exits, unsaved changes and all. There is no save-before-quit prompt anywhere in the program.
  • editorSave's own return value goes nowhere. Exactly like editorOpen's in the previous chapter — computed, documented, and discarded at its only call site. The status-bar message is the only channel that actually reaches the user.

So what did we actually gain?

One new idea, really: chars-array-in, single-write()-out, with truncate doing the resizing job the missing O_TRUNC flag doesn't. Everything else is bookkeeping this book had already built — E.dirty was defined back in Chapter 07, incremented throughout Chapter 10, and this chapter only adds the one place it gets reset. The genuinely load-bearing design decisions are small and specific: truncate explicitly because O_TRUNC was left out, batch the write into one call because that's “a bit safer” under this program's size budget (explicitly not full crash-atomicity), and never touch E.dirty on any failure path. Persistence, start to finish across these two chapters, is symmetrical and small: read lines in with getline, flatten rows out with memcpy, and one flag that remembers whether the two sides still agree.

Before you turn the page

  2. Add a real “Save As”: a new keybinding that prompts for a filename and calls a version of editorSave that writes there instead of E.filename. Should a successful Save As update E.filename itself? What does the status bar show either way?
  3. editorFileWasModified() is dead code today. Find a real place in main() or editorAtExit() where calling it would let Kilo warn about unsaved changes on an abnormal exit path, not just on Ctrl-Q.
