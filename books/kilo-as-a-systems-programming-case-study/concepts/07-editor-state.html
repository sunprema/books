Chapter 07 · Part III — Data & Navigation

Editor State

Chapter 06 named the loop — paint, then react — and promised that everything else in this book is just more work slotted into one of its two calls. But a switch statement and a paint routine can't do anything on their own; they need something to read and something to write. This chapter is about that something: one C struct, one global instance of it, and the surprisingly small set of facts an entire text editor needs to remember between keystrokes.

Motivation: a loop needs somewhere to keep its memory

Every iteration of while(1) is, from C's point of view, a fresh function call. editorRefreshScreen() runs, returns, and every local variable it declared is gone. editorProcessKeypress() runs next, and it has no idea what the previous call to either function did — unless something outlives both calls. A text editor is inherently stateful: it has to remember where the cursor was, what's actually in the file, whether you've saved, and what the terminal looks like, across thousands of these short-lived function calls. That memory has to live somewhere that isn't a local variable.

Kilo's answer is about as direct as C allows: one struct, holding every field the editor needs to remember, declared once at file scope so it outlives every function call for the life of the process. No context object gets allocated and threaded through calls, no handle gets passed around — there's a single answer to “where does the cursor's row live?” and it's the same answer everywhere in the file: E.cy.

Design: one struct, one static instance

Here's the whole thing, field for field, exactly as it appears in kilo.c:

struct editorConfig {
    int cx,cy;  /* Cursor x and y position in characters */
    int rowoff;     /* Offset of row displayed. */
    int coloff;     /* Offset of column displayed. */
    int screenrows; /* Number of rows that we can show */
    int screencols; /* Number of cols that we can show */
    int numrows;    /* Number of rows */
    int rawmode;    /* Is terminal raw mode enabled? */
    erow *row;      /* Rows */
    int dirty;      /* File modified but not saved. */
    char *filename; /* Currently open filename */
    char statusmsg[80];
    time_t statusmsg_time;
    struct editorSyntax *syntax;    /* Current syntax highlight, or NULL. */
};

static struct editorConfig E;

Thirteen fields. That's the entire memory of a working text editor — cursor, scroll position, screen size, the document itself, a dirty flag, a filename, a status line, and a pointer to whichever syntax rules are active. Everything you'll build in the rest of this book — moving around the file, editing, opening and saving, searching, highlighting — is implemented as code that reads and updates some subset of these thirteen fields. If you can name where a feature's data lives in E, you already understand most of how that feature works.

Why static, not just global

static on a file-scope declaration in C isn't about storage duration here — E would live for the whole process either way. It's about linkage: a plain global would be visible to any other .c file that declared extern struct editorConfig E;, while static restricts it to this translation unit alone. Since Kilo is one file, the practical effect is the same either way — but the keyword documents the intent: E is this file's own business, not an exported API.

What each field actually tracks

Grouping the thirteen fields by the question they answer makes the struct easier to hold in your head than reading it top to bottom:

QuestionField(s)What it holds
Where's the cursor?cx, cyCursor column and row, in characters into the file — not screen pixels, not terminal columns after tab expansion.
What part of the file is visible?rowoff, coloffHow far the viewport has scrolled down and right past row/column zero. (Chapter 09 builds the scrolling logic that keeps these in sync with the cursor.)
How big is the terminal?screenrows, screencolsThe usable drawing area, queried once at startup and refreshed on SIGWINCH.
What's actually in the file?numrows, rowHow many lines exist, and a pointer to the heap-allocated array of erow that holds them. (Chapter 08 is entirely about this pair.)
Is the terminal in raw mode right now?rawmodeNot document state at all — a safety flag so cleanup code never restores terminal settings it never actually changed.
Have edits been saved?dirtyA counter incremented by every mutation, zeroed only by a successful save; drives the Ctrl-Q confirmation you saw in Chapter 06.
Which file, and what's the user being told?filename, statusmsg, statusmsg_timeThe open path, and the one-line status message shown under the file, plus when it was set (so it can expire).
How should this file's syntax be colored?syntaxA pointer into a compile-time table of highlight rules, or NULL for an unrecognized file type. (Chapters 14–16 build on this.)

Two of those groupings are worth pulling out on their own, because it's easy to mentally fold them into “the state struct” when they're really neighbors of it, not members.

erow is declared right above editorConfig, but it isn't part of it. E.row is a pointer to an array of this separate struct — one erow per line of the file, each owning its own chars/render buffers:

typedef struct erow {
    int idx;            /* Row index in the file, zero-based. */
    int size;           /* Size of the row, excluding the null term. */
    int rsize;          /* Size of the rendered row. */
    char *chars;        /* Row content. */
    char *render;       /* Row content "rendered" for screen (for TABs). */
    unsigned char *hl;  /* Syntax highlight type for each character in render.*/
    int hl_oc;          /* Row had open comment at end in last syntax highlight
                       check. */
} erow;

Chapter 08 is dedicated entirely to this struct and the array it lives in; for now, the point is just that editorConfig doesn't contain the document, it points at it — E.row is a handle, not storage.

E.syntax works the same way, but it never needs freeing. It's a pointer into HLDB, a static, compile-time table of struct editorSyntax entries (file-match patterns, keyword lists, comment delimiters) that editorSelectSyntaxHighlight() sets by taking the address of a matching table entry. editorConfig never owns that memory, so there's no corresponding free anywhere — unlike E.row and E.filename, which the editor does allocate and must eventually clean up.

Implementation: how E comes to life

Because E is a plain file-scope struct (not a pointer), the C runtime zero-initializes it before main() ever runs — there is no malloc() call for E itself. But Kilo doesn't rely on that implicit zeroing; initEditor(), called first thing in main(), sets every field explicitly:

void initEditor(void) {
    E.cx = 0;
    E.cy = 0;
    E.rowoff = 0;
    E.coloff = 0;
    E.numrows = 0;
    E.row = NULL;
    E.dirty = 0;
    E.filename = NULL;
    E.syntax = NULL;
    updateWindowSize();
    signal(SIGWINCH, handleSigWinCh);
}

Notice what's not in that list: E.rawmode. That's not an oversight — it's a sign that rawmode belongs to a different part of the program than the rest of the struct, and its own two functions manage it end to end:

void disableRawMode(int fd) {
    /* Don't even check the return value as it's too late. */
    if (E.rawmode) {
        tcsetattr(fd,TCSAFLUSH,&orig_termios);
        E.rawmode = 0;
    }
}

int enableRawMode(int fd) {
    if (E.rawmode) return 0; /* Already enabled. */
    /* … open the terminal, save orig_termios, set raw flags … */
    if (tcsetattr(fd,TCSAFLUSH,&raw) < 0) goto fatal;
    E.rawmode = 1;
    return 0;
    /* … */
}

enableRawMode() only sets E.rawmode = 1 after tcsetattr() has actually succeeded, and disableRawMode() checks the flag before touching the terminal at all. That check is what makes cleanup safe to call unconditionally: editorAtExit() (registered once via atexit(), and the subject of Chapter 17) calls disableRawMode() on every exit path, including ones that fire before raw mode was ever successfully entered. Without the flag, that call could try to restore terminal settings that were never saved in the first place.

One field, a different job than its twelve neighbors

rawmode sits in the same struct as cx and dirty, and it's easy to read the struct top to bottom and assume everything in it is document/UI state. It isn't. rawmode tracks the terminal's state, not the file's, has exactly one writer pair (enableRawMode / disableRawMode), and exists purely so exit-time cleanup can't misfire. Kilo puts it in E anyway, for the same reason almost everything else is there too: one place to look, no second global to remember.

updateWindowSize(), called at the end of initEditor(), fills in the screen-dimension fields and reserves room for the status bar in the same breath:

void updateWindowSize(void) {
    if (getWindowSize(STDIN_FILENO,STDOUT_FILENO,
                      &E.screenrows,&E.screencols) == -1) {
        perror("Unable to query the screen for size (columns / rows)");
        exit(1);
    }
    E.screenrows -= 2; /* Get room for status bar. */
}

void handleSigWinCh(int unused __attribute__((unused))) {
    updateWindowSize();
    if (E.cy > E.screenrows) E.cy = E.screenrows - 1;
    if (E.cx > E.screencols) E.cx = E.screencols - 1;
    editorRefreshScreen();
}

That -= 2 is a small but important detail: E.screenrows is deliberately not the terminal's true row count — it's the true count minus two, so the rest of the editor can treat screenrows as “rows available for file content” without re-subtracting the status bar's height at every call site. And because handleSigWinCh() is a SIGWINCH handler (registered back in initEditor()), it re-runs this same sizing logic — and clamps E.cx/E.cy back in bounds — the moment the terminal window is resized, asynchronously, outside the loop's own scheduled call. It's the one place in Kilo where E is mutated from signal-handler context rather than from inside while(1).

How the rest of the file leans on E

Two more fields are worth tracing end to end, because they show the pattern every later chapter repeats: one narrow write path, read from wherever the value is needed. E.dirty starts at zero and is incremented by every mutating call — editorInsertChar(), editorDelChar(), editorInsertNewline(), editorInsertRow(), editorDelRow() each do their own E.dirty++ — and is only ever reset to zero inside editorSave(), after a write actually succeeds. Chapter 06 already showed you the read side: the Ctrl-Q handler checks E.dirty before deciding whether to demand extra confirmation presses.

The status line follows the same shape, funneled through one function:

void editorSetStatusMessage(const char *fmt, ...) {
    va_list ap;
    va_start(ap,fmt);
    vsnprintf(E.statusmsg,sizeof(E.statusmsg),fmt,ap);
    va_end(ap);
    E.statusmsg_time = time(NULL);
}

Every status message in Kilo — the startup help line, the Ctrl-Q warning, a save confirmation, a search prompt — goes through this one printf-style function, which writes into a fixed 80-byte buffer and stamps the current time alongside it. statusmsg_time is state in its own right, not an afterthought: editorRefreshScreen() (Chapter 06) reads it to decide whether the message is still recent enough to keep showing, or old enough to let the status line go quiet. Trace either dirty or statusmsg through the file and you've seen the whole pattern this book will keep repeating: a field in E, a small number of writers, and readers scattered wherever the value matters.

Edge cases: where the “one struct” design shows its seams

  • Not everything stateful lives in E. Chapter 06 already flagged quit_times — the Ctrl-Q countdown — as a static local inside editorProcessKeypress(), not a field on E. The same is true of orig_termios, the saved terminal settings enableRawMode() needs to restore: it's its own static struct termios orig_termios;, declared separately, because it's needed by exactly the raw-mode functions and nothing else. E holds what's genuinely shared across many parts of the program — not literally every piece of state that outlives one function call.
  • One process, one document, one terminal — no exceptions. Because E is a single instance, not something you construct per document, Kilo can only ever edit one file in one terminal per running process. There's no equivalent of a multi-buffer or multi-window editor's “list of open documents” here; opening a second file means running a second kilo process with its own, entirely separate E.
  • No getters, no setters, no protection. Nothing stops any function in the file from writing any field of E directly — there's no accessor layer, no validation on assignment. That's a fair trade for a single-file, ~1000-line program where every writer is a few screens away from every other, but it's exactly the kind of shortcut a larger codebase can't take for long. Chapter 19 (Refactoring Kilo) comes back to this struct specifically when it asks what would have to change to make Kilo safely maintainable at ten times the size.

So what did we actually gain?

A map. Thirteen fields, one instance, zero indirection: E is the complete answer to “what does this editor remember?” From here on, every chapter is really about one or two of these fields — Chapter 08 is E.row and E.numrows in depth, Chapter 09 is E.cx/E.cy/E.rowoff/E.coloff working together, Chapter 12 is E.dirty and E.filename at save time. You don't need to hold the whole editor in your head anymore — you need to know which handful of fields a given feature touches, and this struct is the index.

Before you turn the page
  1. Suppose Kilo needed to support editing two files at once in the same process (a simple split view). List every field of editorConfig that would need to become per-buffer rather than global, and every field (like rawmode) that could safely stay a single process-wide value. What does that split tell you about which fields represent the document versus the terminal session?
  2. initEditor() explicitly zeroes nine of the struct's thirteen fields, even though a file-scope static struct editorConfig E; is already zero-initialized by the C runtime before main() runs. Is that explicit zeroing redundant, or is there a reason to write it anyway? Would removing initEditor()'s body change the program's behavior?
  3. E.rawmode is read and written only by enableRawMode() and disableRawMode(), yet it lives in the same struct as cx, dirty, and filename. Argue both sides: what's gained by keeping it in E alongside document state, and what would be gained by pulling it out into its own file-scope static int next to orig_termios instead?
