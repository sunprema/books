Chapter 04 · Part II — The Terminal & The Loop

Raw Mode and Key Input

Chapter 03 told you the terminal starts in canonical mode — the kernel buffers your keystrokes, applies backspace for you, and only hands a program a complete line once you hit Enter. That's exactly wrong for an editor. This chapter is where we actually turn it off, byte by byte, and then read what the keyboard sends one keystroke at a time — including the arrow keys, which don't arrive as one byte at all.

Motivation: an editor needs every keystroke, the instant it happens

Try this thought experiment: you're holding down the ↑ key to scroll up. In canonical mode, nothing happens — not one thing — until you press Enter, and by then the kernel has already handed your program a whole line of escape-sequence bytes to sort out after the fact, with backspace edits baked in if you fixed a typo along the way. An editor can't work like that. It needs to react to the arrow key the moment it's pressed, and it needs to draw its own cursor rather than let the terminal echo raw bytes wherever it pleases. Both of those requirements point at the same fix: leave canonical mode entirely, and take over reading input one byte at a time.

That fix is called raw mode, and it isn't a single switch — it's roughly a dozen individual bits in a struct termios, each one disabling a specific piece of kernel behavior you'd otherwise take for granted. Kilo turns every one of them off, on purpose, in one function: enableRawMode().

Design: a snapshot, a mutated copy, and a promise to put it back

Before Kilo changes anything, it saves what was there. Right at file scope, separate from the big struct editorConfig E we'll meet properly in Chapter 07, sits one more global:

static struct termios orig_termios; /* In order to restore at exit.*/

That's deliberate placement, not an oversight: orig_termios isn't editor state, it's terminal-restoration bookkeeping — the one snapshot needed to put the user's shell back exactly the way it found it, whether the editor exits cleanly, quits on an error, or the user hits Ctrl-Q. The whole design of this chapter is really one promise: whatever we do to the terminal, we will undo it before we hand control back. Everything below exists to keep that promise even when the program doesn't exit through its normal path.

Three guards, before a single flag is touched

enableRawMode(fd) doesn't rush to tcsetattr(). It checks, in order: (1) is raw mode already on (E.rawmode) — if so, return immediately, the function is idempotent; (2) is stdin even a terminal (isatty()) — if input is piped from a file instead of typed, there's no terminal to put in raw mode, and Kilo bails out cleanly instead of calling tcgetattr() on something that isn't a tty; only then does it (3) register a cleanup handler with atexit() and take the snapshot with tcgetattr(). Order matters here — you register the safety net before you touch anything.

Implementation: enableRawMode, flag by flag

Here is the whole function, unabridged:

int enableRawMode(int fd) {
    struct termios raw;

    if (E.rawmode) return 0; /* Already enabled. */
    if (!isatty(STDIN_FILENO)) goto fatal;
    atexit(editorAtExit);
    if (tcgetattr(fd,&orig_termios) == -1) goto fatal;

    raw = orig_termios;  /* modify the original mode */
    raw.c_iflag &= ~(BRKINT | ICRNL | INPCK | ISTRIP | IXON);
    raw.c_oflag &= ~(OPOST);
    raw.c_cflag |= (CS8);
    raw.c_lflag &= ~(ECHO | ICANON | IEXTEN | ISIG);
    raw.c_cc[VMIN] = 0;   /* Return each byte, or zero for timeout. */
    raw.c_cc[VTIME] = 1;  /* 100 ms timeout (unit is tens of second). */

    if (tcsetattr(fd,TCSAFLUSH,&raw) < 0) goto fatal;
    E.rawmode = 1;
    return 0;

fatal:
    errno = ENOTTY;
    return -1;
}

Notice the very first line inside the flag section: raw = orig_termios; copies the snapshot into a local variable, and every flag change below mutates raw, never orig_termios. The original stays untouched in memory for exactly as long as the program runs, ready to be handed straight back to tcsetattr() at exit.

Then come four groups of flags, and the direction of the operator is the entire meaning of each line — &= ~(...) clears bits (turns features off), |= (...) sets bits (turns features on):

FieldFlags cleared / setWhat it disables (or enables)
c_iflag&= ~(BRKINT|ICRNL|INPCK|ISTRIP|IXON) No break-condition signal, no CR→NL translation, no parity checking, no 8th-bit stripping, no software flow control (Ctrl-S/Ctrl-Q pausing/resuming output) — Kilo wants those two keys for itself.
c_oflag&= ~(OPOST) Disables output post-processing — no automatic \n→\r\n translation on write(). Kilo writes its own \r\n where it wants one.
c_cflag|= (CS8) Sets the character size to 8 bits per byte (a positive set, the one flag here that turns something on).
c_lflag&= ~(ECHO|ICANON|IEXTEN|ISIG) No echoing typed characters back automatically (Kilo draws its own screen), no canonical line-buffering (this is the big one — see below), no extended input processing, no Ctrl-C/Ctrl-Z turned into signals before Kilo ever sees the byte.

That last cleared flag, ISIG, is easy to skim past and important not to: without it, pressing Ctrl-C would send SIGINT and kill the process before a single line of Kilo's own code ran. With ISIG off, Ctrl-C arrives as an ordinary byte — CTRL_C = 3 in the enum KEY_ACTION from Chapter 03 — and it's Kilo's own editorProcessKeypress(), not the kernel, that decides what it means. Turning ISIG off is what lets an editor claim Ctrl-C as a keybinding at all.

c_cc[VMIN] and c_cc[VTIME]: how long read() is willing to wait

Two more fields, not flags this time but small integers, control read()'s patience once ICANON is off. Kilo sets VMIN = 0 and VTIME = 1, which together mean: return as soon as a byte is available, or after 100 milliseconds (VTIME's unit is tenths of a second) with zero bytes if nothing arrived. read() in this mode never blocks forever, and it never returns an error just because nothing was typed yet — a timeout comes back as nread == 0, which is not an error and not end-of-file. Kilo's own comment on the VMIN line says it plainly: “Return each byte, or zero for timeout.”

The whole modified raw struct is applied with tcsetattr(fd, TCSAFLUSH, &raw). TCSAFLUSH is the detail worth naming: it applies the new settings only after discarding any input that's already been typed but not yet read. Without it, keystrokes buffered in canonical mode right before the switch could leak through afterward and get misread as something else entirely — the flush makes the transition clean.

Putting it back: disableRawMode and atexit

A promise to restore the terminal is only as good as the mechanism that keeps it even when the program doesn't exit the way you expect. Kilo's answer is two small functions and one standard-library call:

void disableRawMode(int fd) {
    /* Don't even check the return value as it's too late. */
    if (E.rawmode) {
        tcsetattr(fd,TCSAFLUSH,&orig_termios);
        E.rawmode = 0;
    }
}

/* Called at exit to avoid remaining in raw mode. */
void editorAtExit(void) {
    disableRawMode(STDIN_FILENO);
}

editorAtExit() is registered inside enableRawMode() itself, with atexit(editorAtExit) — step 3 of the guard sequence above. The C standard library guarantees an atexit()-registered function runs whenever the process reaches exit(), from anywhere: a normal fall-through out of main()'s loop, or an exit(1) called from deep inside some unrelated error path. You don't have to remember to call disableRawMode() at every place the program might stop — you register it once, and the runtime calls it for you.[1]

Notice the comment above disableRawMode(): “Don't even check the return value as it's too late.” That's not carelessness, it's an honest read of the situation — by the time this function runs, the process is already on its way out. If tcsetattr() itself somehow fails here, there is no meaningful recovery action left to take. It's a small, deliberate acknowledgment that error handling has limits, and this is past them.

A limit worth naming honestly

atexit() only covers exit paths the C runtime controls. It does not run on an unhandled signal that terminates the process abnormally (a segfault, an unhandled SIGKILL) or on a hard kill from outside the process. That gap is a real, known limitation of this approach — Chapter 17 (Error Handling and Cleanup) comes back to it directly, once you've seen the whole shape of Kilo's error handling to compare it against.

Reading one keystroke — and un-mangling the ones that aren't one byte

Raw mode alone gets you individual bytes, on demand. It does not get you individual keys — because not every key a user presses corresponds to one byte. The arrow keys, Home, End, Delete, Page Up, and Page Down all arrive as multi-byte escape sequences: the terminal, watching what you type, sends a short burst of bytes starting with ESC (0x1B) that encodes which key you pressed. Turning that burst back into one logical keypress is editorReadKey()'s entire job.

int editorReadKey(int fd) {
    int nread;
    char c, seq[3];
    while ((nread = read(fd,&c,1)) == 0);
    if (nread == -1) exit(1);

    while(1) {
        switch(c) {
        case ESC:
            if (read(fd,seq,1) == 0) return ESC;
            if (read(fd,seq+1,1) == 0) return ESC;

            if (seq[0] == '[') {
                if (seq[1] >= '0' && seq[1] <= '9') {
                    if (read(fd,seq+2,1) == 0) return ESC;
                    if (seq[2] == '~') {
                        switch(seq[1]) {
                        case '3': return DEL_KEY;
                        case '5': return PAGE_UP;
                        case '6': return PAGE_DOWN;
                        }
                    }
                } else {
                    switch(seq[1]) {
                    case 'A': return ARROW_UP;
                    case 'B': return ARROW_DOWN;
                    case 'C': return ARROW_RIGHT;
                    case 'D': return ARROW_LEFT;
                    case 'H': return HOME_KEY;
                    case 'F': return END_KEY;
                    }
                }
            }
            else if (seq[0] == 'O') {
                switch(seq[1]) {
                case 'H': return HOME_KEY;
                case 'F': return END_KEY;
                }
            }
            break;
        default:
            return c;
        }
    }
}

Read it in three movements.

First, get one real byte. while ((nread = read(fd,&c,1)) == 0); is a loop with an empty body — it keeps calling read() for exactly one byte, and keeps discarding nread == 0 results (that's the VMIN/VTIME timeout from a moment ago, not an error) until either a byte actually arrives or something goes genuinely wrong (nread == -1, which exits the whole program). From the caller's point of view, this loop is what turns Kilo's technically-non-blocking, 100ms-polling read() back into an ordinary blocking “wait for the next keystroke” call. The polling is invisible from outside editorReadKey().

Second, the plain case. Once c holds a real byte, the outer switch(c) has exactly two paths: everything that isn't ESC falls to default and is returned immediately, unmodified. A letter, a digit, a Ctrl-key byte from Chapter 03's enum — all of them are already one complete key, so there's nothing to decode.

Third, the interesting case. If c == ESC, the function doesn't yet know whether the user pressed a lone Esc key or the terminal is mid-way through sending a multi-byte sequence for something like ↑. It reads up to two more bytes to find out — and if either of those reads times out (== 0), that is the answer: nothing else was coming, so it really was just Esc, and the function returns ESC plain. This is worth sitting with, because it's a genuinely clever piece of design: Kilo tells a bare Esc apart from the start of an arrow-key sequence purely by timing — by whether the rest of the sequence's bytes show up within the same 100 ms window VTIME already set up, not by looking for some special terminating byte. The two mechanisms — raw mode's read timeout, and escape-sequence disambiguation — are the same trick, reused.

Reading the sequence table like the terminal writes it

An arrow key sends three bytes: ESC, then '[' (the CSI, or Control Sequence Introducer), then a single letter — 'A' for up, 'B' down, 'C' right, 'D' left. Home and End use the same introducer with 'H'/'F' — or, on some terminal emulators, an older ESC O introducer (SS3) instead of ESC [, which is exactly why editorReadKey() checks seq[0] == 'O' as a separate case: without it, Home/End would silently break on whichever terminal family doesn't use CSI for those two keys. Delete, Page Up, and Page Down are one byte longer still — ESC [ 3 ~, ESC [ 5 ~, ESC [ 6 ~ — because CSI has room for a numeric parameter before its final byte, and these three keys are distinguished by that number rather than by a unique letter.

Every path through the escape-sequence decoding returns one of the “soft codes” from Chapter 03's enum KEY_ACTION — ARROW_UP, DEL_KEY, PAGE_UP, and the rest, all numbered 1000 and above precisely so they can never collide with a real byte value (which tops out at 255). Those codes were never sent by any terminal as a single byte — they're synthesized right here, inside editorReadKey() — but once synthesized, they're indistinguishable from an ordinary keypress to every function downstream. editorProcessKeypress()'s big dispatch switch, which we'll meet properly in Chapter 06, never has to know or care whether the int it received was one byte typed directly or three bytes decoded on its behalf. That uniformity is the entire payoff of doing this decoding here, in one place, instead of scattering “is this an escape sequence?” checks throughout the rest of the program.

Edge cases: what this design deliberately doesn't handle

  • Piped input. If stdin isn't a terminal at all — input redirected from a file — isatty() catches it in enableRawMode()'s second guard, and the function fails cleanly with errno = ENOTTY rather than calling tcgetattr() on something it can't query.
  • A slow or laggy connection. The Esc-vs-sequence disambiguation depends on the remaining bytes of a sequence arriving within the 100ms VTIME window. On an unusually slow terminal connection (a long-haul SSH session over a bad link, for instance) it's theoretically possible for an arrow-key sequence's bytes to arrive spread further apart than that and get misread as a lone Esc followed by stray characters. It's a known, named trade-off of this exact technique, not a bug specific to Kilo — every terminal program that disambiguates Esc this way inherits it.
  • An unhandled signal or a hard kill. As the callout above says plainly: atexit() only fires on paths the C runtime controls. A crash that bypasses normal exit, or a kill -9 from another terminal, leaves raw mode set and the user's shell looking broken until they run reset or stty sane. Chapter 17 returns to this gap once you've seen the rest of Kilo's error handling to weigh it against.
  • Calling enableRawMode twice. The very first guard, if (E.rawmode) return 0;, makes the function idempotent on purpose — safe to call again without re-snapshotting a terminal that's already in raw mode (which would otherwise overwrite orig_termios with the raw settings instead of the original ones, corrupting the very thing you're trying to restore).

So what did we actually gain?

One function, enableRawMode(), that trades a dozen small pieces of kernel convenience — line buffering, auto-echo, signal generation, newline translation — for total control over every byte the keyboard sends, the instant it's sent. One promise, kept by atexit(editorAtExit), that whatever raw mode changes gets undone no matter how the program ends (within the honest limits we named). And one function, editorReadKey(), that hides the ugliest part of that trade — multi-byte escape sequences for arrow keys and friends — behind a single, uniform return value: one int per logical keypress, real byte or synthesized soft code alike. Everything from Chapter 06 onward gets to assume that uniformity and never think about raw termios flags again.
