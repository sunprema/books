Chapter 06 · Part II — The Terminal & The Loop

The Refresh Cycle

Chapter 04 gave you a raw stream of keystrokes. Chapter 05 gave you a way to paint a screen. Neither one, on its own, is an editor. What actually makes Kilo run is the thing sitting between them — a loop so small it's easy to skim past, and so central that everything else in this book only ever executes because that loop calls it. Read main() once you know what to look for, and you've read the architecture.

Motivation: an editor is a loop, not an event system

Modern UI frameworks train you to think in callbacks and event queues — register a handler, let the framework decide when to call it, trust some scheduler to interleave input, timers, and redraws. Kilo has none of that. It has no event loop library, no dispatcher, no concurrency at all. It has one function, running forever, doing exactly two things in exactly the same order, every single time:

while(1) {
    editorRefreshScreen();
    editorProcessKeypress(STDIN_FILENO);
}

That's it. That's the whole runtime. Paint the screen, then wait for and react to one keystroke. Paint again, then wait for the next one. Forever, until something inside the loop calls exit(). There's no branch that skips the paint because “nothing changed,” and no branch that reads more than one key before painting. The entire mental model of how Kilo behaves — why it feels instant, why it never redraws stale content, why a resize just works — falls out of taking those two lines completely literally.

Design: paint first, then block

The order of the two calls is the whole design decision, and it's easy to read past. editorRefreshScreen() runs before editorProcessKeypress() on every trip through the loop. That means each iteration opens by showing the user the result of whatever the previous iteration's keystroke did, and only then blocks waiting for the next one. The very first iteration, before any key has ever been read, paints the editor's just-opened, unedited state — there's no keystroke to react to yet, and none is needed for the first frame to appear.

Why this order and not the reverse

Flip the two calls — process a keypress, then refresh — and the program would still basically work, but the very first thing the user would ever see is a blank blocking wait with nothing painted yet, and every subsequent frame would still be correct only by accident of the loop repeating forever. Painting first guarantees the screen is never more than one keystroke stale, starting from the very first frame.

The “block” half deserves a second look too, because it isn't a naive spin. editorProcessKeypress() calls editorReadKey(), which you first saw in Chapter 04 wrapped in this loop:

while ((nread = read(fd,&c,1)) == 0);
if (nread == -1) exit(1);

Recall from Chapter 04 that raw mode sets VMIN=0 and VTIME=1: read() returns zero bytes read if 100ms passes with nothing typed, rather than blocking indefinitely or erroring. That zero isn't treated as an event — the while condition just loops back around and calls read() again. So “blocking” here really means: sleep in short, OS-scheduled bursts, waking every tenth of a second to check for a byte, until one actually shows up. Only a genuine error (nread == -1) breaks that loop abnormally, with exit(1). The refresh cycle isn't burning CPU between keystrokes; the kernel is doing the waiting.

Implementation: what one iteration actually does

With the shape of the loop settled, the interesting work is entirely inside editorProcessKeypress(). One key comes in from editorReadKey(), and one switch decides what happens to it:

#define KILO_QUIT_TIMES 3
void editorProcessKeypress(int fd) {
    /* When the file is modified, requires Ctrl-q to be pressed N times
     * before actually quitting. */
    static int quit_times = KILO_QUIT_TIMES;

    int c = editorReadKey(fd);
    switch(c) {
    case ENTER:
        editorInsertNewline();
        break;
    case CTRL_C:
        /* We ignore ctrl-c, it can't be so simple to lose the changes
         * to the edited file. */
        break;
    case CTRL_Q:
        /* Quit if the file was already saved. */
        if (E.dirty && quit_times) {
            editorSetStatusMessage("WARNING!!! File has unsaved changes. "
                "Press Ctrl-Q %d more times to quit.", quit_times);
            quit_times--;
            return;
        }
        exit(0);
        break;
    case CTRL_S:
        editorSave();
        break;
    case CTRL_F:
        editorFind(fd);
        break;
    case BACKSPACE:
    case CTRL_H:
    case DEL_KEY:
        editorDelChar();
        break;
    case PAGE_UP:
    case PAGE_DOWN:
        if (c == PAGE_UP && E.cy != 0)
            E.cy = 0;
        else if (c == PAGE_DOWN && E.cy != E.screenrows-1)
            E.cy = E.screenrows-1;
        {
        int times = E.screenrows;
        while(times--)
            editorMoveCursor(c == PAGE_UP ? ARROW_UP: ARROW_DOWN);
        }
        break;
    case ARROW_UP:
    case ARROW_DOWN:
    case ARROW_LEFT:
    case ARROW_RIGHT:
        editorMoveCursor(c);
        break;
    case CTRL_L:
        /* Just refresht the line as side effect. */
        break;
    case ESC:
        /* Nothing to do for ESC in this mode. */
        break;
    default:
        editorInsertChar(c);
        break;
    }

    quit_times = KILO_QUIT_TIMES; /* Reset it to the original value. */
}

A few things are worth naming explicitly here, because they're easy to read past on a first pass:

Every unlisted key is text. There's no separate “is this printable?” check anywhere in this function. The default case — anything that didn't match one of the explicit control keys above it — is treated as literal content and handed to editorInsertChar(c). The switch isn't a filter for “interesting” keys; it's a short list of exceptions to the rule that a keystroke becomes a character.

Paging reuses arrow movement, one line at a time. PAGE_UP/PAGE_DOWN don't have their own scrolling logic here at all — they snap the cursor to the top or bottom of the visible screen, then call editorMoveCursor() — the exact same function the arrow keys call — E.screenrows times in a row. A page-down is implemented as “press down-arrow once per visible row,” not as a distinct operation. (The actual scrolling of the viewport happens later, inside editorRefreshScreen(), based on where the cursor ends up — Chapter 09 picks that up.)

Where quit_times actually lives

Look again at the top of the function: static int quit_times = KILO_QUIT_TIMES; is declared inside editorProcessKeypress(), as a C static local — not as a field on E. That's a real exception to the pattern Chapter 07 is about to make explicit: almost everything Kilo remembers between calls lives in one global struct, and this three-integer countdown deliberately doesn't. It only needs to persist across calls to this one function, so C's function-local static storage — initialized once, alive for the life of the process — is exactly the right tool, and putting it in E would have made it visible (and mutable) from code that has no business touching it.

Walk the quit logic itself carefully, because its behavior is stricter than it looks. CTRL_Q only exits immediately, via exit(0), if either E.dirty is false (nothing unsaved) or quit_times has already reached zero. Otherwise it prints a warning, decrements the counter, and returns — skipping the very last line of the function. And that last line matters: quit_times = KILO_QUIT_TIMES; runs unconditionally after the switch, for any key that reaches it. Press Ctrl-Q once on a modified file, then press an arrow key by habit, then Ctrl-Q again, and you haven't used up two of your three warnings — the arrow key silently reset the count back to three. Quitting an unsaved file takes three Ctrl-Q presses in a row, with nothing else typed in between, not three presses total over the course of editing.

Where the loop's inputs come from: initEditor()

The loop assumes E.screenrows and E.screencols are already correct the moment editorRefreshScreen() runs for the very first time. That's guaranteed by what happens one step earlier, in main(), before raw mode is even enabled:

void initEditor(void) {
    E.cx = 0;
    E.cy = 0;
    E.rowoff = 0;
    E.coloff = 0;
    E.numrows = 0;
    E.row = NULL;
    E.dirty = 0;
    E.filename = NULL;
    E.syntax = NULL;
    updateWindowSize();
    signal(SIGWINCH, handleSigWinCh);
}

Two lines at the end of an otherwise unremarkable zero-everything initializer do real work. updateWindowSize() queries the terminal's dimensions (Chapter 05) and reserves the two status rows right here, before the loop has run even once. And signal(SIGWINCH, handleSigWinCh) registers a handler for the kernel's “your terminal window was resized” signal — which is the one documented place in kilo.c where a repaint can happen outside the loop's own scheduled call.

The full sequence in main(), in order, is: check argc, initEditor(), pick a syntax highlighter for the given filename, editorOpen() the file, then enableRawMode(), then one status message, then the loop:

int main(int argc, char **argv) {
    if (argc != 2) {
        fprintf(stderr,"Usage: kilo <filename>\n");
        exit(1);
    }

    initEditor();
    editorSelectSyntaxHighlight(argv[1]);
    editorOpen(argv[1]);
    enableRawMode(STDIN_FILENO);
    editorSetStatusMessage(
        "HELP: Ctrl-S = save | Ctrl-Q = quit | Ctrl-F = find");
    while(1) {
        editorRefreshScreen();
        editorProcessKeypress(STDIN_FILENO);
    }
    return 0;
}

Notice the file is opened before the terminal is put into raw mode. That ordering isn't about the refresh cycle directly, but it's a small piece of the same discipline: set up everything that can fail in an ordinary, easy-to-debug mode first (normal terminal, normal stderr), and only flip the terminal into the unusual raw state once there's an editor's worth of state ready to actually run the loop against.

Edge cases: the exceptions worth naming

  • SIGWINCH breaks the “loop is the only place that repaints” rule. handleSigWinCh() calls updateWindowSize() and then editorRefreshScreen() directly, from signal-handler context, whenever the kernel reports the terminal was resized. If you resize your terminal window mid-edit, a repaint happens asynchronously, not because the loop reached its scheduled call — worth remembering before you assume every visible redraw traces back to one iteration of while(1).
  • Ctrl-C does nothing, on purpose, twice over. Chapter 04's raw mode already clears ISIG, so the terminal driver no longer turns Ctrl-C into a SIGINT before Kilo ever sees it — the raw byte (CTRL_C = 3) reaches editorProcessKeypress() like any other key. And once it arrives, the case CTRL_C: branch is an empty break, with a comment explaining the choice: it “can't be so simple to lose the changes to the edited file.” Two separate decisions — disable the signal, then ignore the byte — combine to make the editor's most habitual “get me out of here” keystroke completely harmless.
  • A read() timeout is not a loop iteration. The VMIN=0/VTIME=1 retry inside editorReadKey() happens entirely underneath while(1)'s one call to editorProcessKeypress() — the outer loop doesn't see, and doesn't repaint for, every 100ms timeout while the user is thinking. One trip through while(1) corresponds to exactly one successfully read key, however long the wait for it took.
  • CTRL_L looks like it should do something and doesn't need to. As Chapter 05 flagged, the conventional “redraw the screen” keystroke is an empty case here. It doesn't need a special action because editorRefreshScreen() already runs unconditionally on every single loop iteration — there is no state where the screen is stale and waiting for a manual refresh request to catch up.

So what did we actually gain?

A name for the shape of the whole program: read, react, repaint — paint the current state, block for one key, dispatch it through one switch, loop. No event queue, no scheduler, no threads, and (with one narrow, named exception for terminal resizes) no repaint that isn't the loop's own doing. Everything from Chapter 07 onward — the state you track, the text you store, the moves and edits you make, the files you open and save, the search and highlighting you add — is just more work that gets slotted into one of these two calls. You now know exactly where in the program any given feature has to hook in: either it changes what gets painted, or it's a new case in that switch.

Before you turn the page
  1. Suppose the two calls in while(1) were swapped — editorProcessKeypress() first, then editorRefreshScreen(). Walk through the first three keystrokes of a session and describe exactly what the user would (and wouldn't) see differently, frame by frame.
  2. quit_times resets to KILO_QUIT_TIMES for any key that reaches the bottom of editorProcessKeypress() — including keys whose case is a no-op, like ESC. Trace why a no-op case still reaches that reset line, and what would change if the CTRL_L/ESC cases used return; instead of break;.
  3. handleSigWinCh() calls editorRefreshScreen() directly from a signal handler, while the main loop might be blocked inside editorReadKey()'s own read() call at that exact moment. What has to be true about editorRefreshScreen()'s use of global state (the E struct, the append buffer) for that to be safe rather than a data race?
