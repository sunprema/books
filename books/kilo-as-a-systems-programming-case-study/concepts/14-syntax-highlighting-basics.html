Chapter 14 · Part V — Search & Highlighting

Syntax Highlighting Basics

Every row already carries two versions of itself: chars, what you typed, and render, what tabs turn it into for the screen. Syntax highlighting adds a third: hl, one byte per character of render saying what kind of thing that character is. Not a parse tree. Not a token list you could hand to a compiler. One byte, one character, chosen by a single left-to-right scan that forgets everything the instant it reaches the end of the row — except for one bit of state, an open comment, that it hands to the next row on its way out. This chapter is the machine that fills hl and the smaller machine that turns it into color on screen. What counts as a keyword, a string, or a number is next chapter's problem; this one is about the architecture those rules will plug into.

Motivation: color is a per-character fact, not a per-line one

Think about what a real syntax highlighter has to answer, for every single character on screen: what color is this one? Not "is this line a comment" — a line can open with code, run into a string halfway through, and end inside a comment that continues onto the next line. The honest unit of "what color" is the character, not the line, not even the token. Kilo's answer is almost embarrassingly literal: give every row a second array, the same length as its rendered text, and put one small integer in it per character — a highlight type. Nine possible values, defined as plain #define constants:

#define HL_NORMAL 0
#define HL_NONPRINT 1
#define HL_COMMENT 2   /* Single line comment. */
#define HL_MLCOMMENT 3 /* Multi-line comment. */
#define HL_KEYWORD1 4
#define HL_KEYWORD2 5
#define HL_STRING 6
#define HL_NUMBER 7
#define HL_MATCH 8      /* Search match. */

#define HL_HIGHLIGHT_STRINGS (1<<0)
#define HL_HIGHLIGHT_NUMBERS (1<<1)

You've already met the last one. Chapter 13's editorFind reached into a row's highlight array and stamped HL_MATCH across a search hit, using machinery this chapter is about to build. That's worth sitting with for a second: search didn't need a highlighting feature to borrow the highlighting storage. One array, one byte-per-character convention, and two completely different producers — a parser and a search function — both write into it, and one consumer, the screen-drawing loop, doesn't care which wrote what.

A third parallel buffer

Chapter 08 gave each row two buffers of text: chars (what you typed) and render (what tabs turn it into). hl is the third leg of that stool — not text at all, but a same-length array of small integers riding alongside render, one classification per rendered character. The struct's own comment says it plainly: unsigned char *hl; /* Syntax highlight type for each character in render.*/. Note which buffer it's parallel to — not chars.

Design: a scan with three bits of memory, plus one that survives the row

If hl is going to hold a classification per character, something has to walk the row and decide, character by character, what each one is. Call it a parser if you like, but it's a small one: a single forward pass over render, carrying exactly three pieces of state as it goes.

  • prev_sep — was the character just before this one a token boundary? Needed to tell "the start of a new word" from "the middle of one you're already inside."
  • in_string — either 0, or the exact quote character (" or ') currently open, so the scan knows which quote closes it.
  • in_comment — inside an unterminated /* */ span, whether it opened on this line or was still open when the line began.

That's the entire "parser state." No stack, no lookahead beyond one or two characters, no backtracking. What decides "is this a token boundary" in the first place is one small helper used everywhere the scan needs to answer that question:

int is_separator(int c) {
    return c == '\0' || isspace(c) || strchr(",.()+-/*=~%[];",c) != NULL;
}

Whitespace, the end of the row, or one of a fixed set of punctuation characters — that's what "a token just ended" means to Kilo. Chapter 15's keyword and number rules lean on this same function; it isn't reinvented per rule, it's the one shared notion of "boundary" the whole scanner agrees on.

The fourth piece of state is the interesting one, because it doesn't live in a local variable — it has to outlive the function call. A /* */ comment can span many lines, so when the scan reaches the end of a row still inside one, that fact has to be remembered somewhere the next row's scan can find it. That's erow.hl_oc, "row had open comment at end in last syntax highlight check" — state that belongs to the row, not to any one call of the function that fills it in.

Not quite "does the last character say MLCOMMENT"

Computing "is this row still open" isn't just checking whether the row's last hl byte is HL_MLCOMMENT. A comment can close exactly at end-of-line — and its closing */ is itself colored as part of the comment, so the last byte would still say HL_MLCOMMENT even though nothing is open anymore. editorRowHasOpenComment checks both things: the last hl byte and that the row's actual last two rendered characters aren't the literal closer.

int editorRowHasOpenComment(erow *row) {
    if (row->hl && row->rsize && row->hl[row->rsize-1] == HL_MLCOMMENT &&
        (row->rsize < 2 || (row->render[row->rsize-2] != '*' ||
                            row->render[row->rsize-1] != '/'))) return 1;
    return 0;
}

Implementation: editorUpdateSyntax, one row at a time

editorUpdateSyntax(erow *row) is the whole classifier. It doesn't run on its own schedule or in response to its own event — it's the last line of editorUpdateRow(), the same function Chapter 08 introduced to rebuild render from chars after every edit. Every function that changes a row's text already calls editorUpdateRow(); that means every one of them already triggers a full rehighlight of that row as a side effect, with no separate call site to remember.

There is no incremental highlighting

Type one character, and Kilo does not patch one entry of hl. editorUpdateRow() rebuilds the whole of render, then editorUpdateSyntax() throws away the whole of hl and recomputes it, left to right, from the first character of the row. For a single line, that's cheap enough not to matter. What it buys in return is simplicity: no logic anywhere for "which part of the highlighting is now stale" — the answer is always "all of it, recompute."

Here is the function, with the character-by-character comment/string/number/ keyword body — Chapter 15's subject — folded down to a comment so the shape around it is easy to see:

/* Set every byte of row->hl (that corresponds to every character in the line)
 * to the right syntax highlight type (HL_* defines). */
void editorUpdateSyntax(erow *row) {
    row->hl = realloc(row->hl,row->rsize);
    memset(row->hl,HL_NORMAL,row->rsize);

    if (E.syntax == NULL) return; /* No syntax, everything is HL_NORMAL. */

    int i, prev_sep, in_string, in_comment;
    char *p;
    char **keywords = E.syntax->keywords;
    char *scs = E.syntax->singleline_comment_start;
    char *mcs = E.syntax->multiline_comment_start;
    char *mce = E.syntax->multiline_comment_end;

    /* Point to the first non-space char. */
    p = row->render;
    i = 0; /* Current char offset */
    while(*p && isspace(*p)) { p++; i++; }
    prev_sep = 1; /* Tell the parser if 'i' points to start of word. */
    in_string = 0; /* Are we inside "" or '' ? */
    in_comment = 0; /* Are we inside multi-line comment? */

    /* If the previous line has an open comment, this line starts
     * with an open comment state. */
    if (row->idx > 0 && editorRowHasOpenComment(&E.row[row->idx-1]))
        in_comment = 1;

    /* ... per-character scan: single-line and multi-line comments, strings,
     * non-printable bytes, numbers, keywords — Chapter 15 ... */

    /* Propagate syntax change to the next row if the open comment
     * state changed. This may recursively affect all the following rows
     * in the file. */
    int oc = editorRowHasOpenComment(row);
    if (row->hl_oc != oc && row->idx+1 < E.numrows)
        editorUpdateSyntax(&E.row[row->idx+1]);
    row->hl_oc = oc;
}

Read the shape, not the elided middle. The function resets the row to all-HL_NORMAL and bails immediately if there's no active language (E.syntax == NULL) — a file Kilo doesn't recognize is not a special case anywhere else; it's just this one early return. Otherwise it sets up the three scan variables, seeds in_comment from the previous row's carried-over state, walks the row (Chapter 15), and then — every single time, whether or not anything about this row's own text looks unusual — recomputes whether this row now ends inside an open comment and compares it to what hl_oc said last time.

The recursion: one edit, potentially many rows

That comparison is where this "just rehighlight one row" function stops being quite so local. If the open-comment state changed — a line that used to end inside a comment now doesn't, or vice versa — the very next row's highlighting was computed against the old assumption and is now wrong. So editorUpdateSyntax calls itself on that next row. And if that row's open-comment state also changes as a result, it calls itself again, one row further down. The function's own comment names this plainly: "This may recursively affect all the following rows in the file."

A keystroke that rehighlights the rest of the file

Most edits touch hl_oc not at all — the recursion's base case, row->hl_oc != oc being false, fires almost immediately and the cascade goes nowhere. But type a stray /* near the top of a large C file, with no matching */ below it, and every row underneath flips from "not open" to "open," which means every one of them recurses into the next. One keystroke, one full rehighlight of the rest of the file. It's a narrow, pathological case — not a bug, exactly, more a cost the design accepts in exchange for never having to reason about partial or stale highlight state.

From bytes to color: editorSyntaxToColor and the drawing loop

editorUpdateSyntax never writes an escape sequence. Classification and coloring are two different functions with two different jobs: one decides what a character is, the other decides what that means on screen. The second one is almost insultingly simple — a switch from highlight type to an ANSI foreground color number:

int editorSyntaxToColor(int hl) {
    switch(hl) {
    case HL_COMMENT:
    case HL_MLCOMMENT: return 36;     /* cyan */
    case HL_KEYWORD1: return 33;    /* yellow */
    case HL_KEYWORD2: return 32;    /* green */
    case HL_STRING: return 35;      /* magenta */
    case HL_NUMBER: return 31;      /* red */
    case HL_MATCH: return 34;      /* blu */
    default: return 37;             /* white */
    }
}

Seven cases feeding six colors (comment and multi-line comment share cyan), plus a fallback. Those numbers — 31 through 37 — are the standard ANSI SGR foreground codes from Chapter 05's escape-sequence vocabulary: \x1b[<n>m sets the foreground color to n. Nothing new is invented here; highlighting reuses the exact same escape mechanism that draws the status bar and the cursor.

The actual color only appears when the per-row loop inside editorRefreshScreen draws the visible slice of a row. For each on-screen character it checks the parallel hl byte and picks one of three branches:

int current_color = -1;
...
unsigned char *hl = r->hl+E.coloff;
for (j = 0; j < len; j++) {
    if (hl[j] == HL_NONPRINT) {
        char sym;
        abAppend(&ab,"\x1b[7m",4);
        if (c[j] <= 26) sym = '@'+c[j]; else sym = '?';
        abAppend(&ab,&sym,1);
        abAppend(&ab,"\x1b[0m",4);
    } else if (hl[j] == HL_NORMAL) {
        if (current_color != -1) {
            abAppend(&ab,"\x1b[39m",5);
            current_color = -1;
        }
        abAppend(&ab,c+j,1);
    } else {
        int color = editorSyntaxToColor(hl[j]);
        if (color != current_color) {
            char buf[16];
            int clen = snprintf(buf,sizeof(buf),"\x1b[%dm",color);
            current_color = color;
            abAppend(&ab,buf,clen);
        }
        abAppend(&ab,c+j,1);
    }
}
abAppend(&ab,"\x1b[39m",5);
abAppend(&ab,"\x1b[0K",4);
abAppend(&ab,"\r\n",2);

HL_NONPRINT gets special treatment, not a color at all — a substituted, readable stand-in character (a caret-style letter for low control bytes, ? otherwise) wrapped in reverse video, \x1b[7m ... \x1b[0m, so an unprintable byte in the file is visible and unambiguous rather than silently mangling the terminal. HL_NORMAL resets the foreground to the terminal's own default, \x1b[39m — but only if a color was actually active, via current_color. Everything else asks editorSyntaxToColor for a number and, again, only emits the escape if that number differs from what's already active.

current_color is Chapter 05's batching instinct, reapplied

Without the color != current_color check, every single character of a highlighted word would carry its own color escape — five bytes of overhead per character for no visual difference, since consecutive characters of a keyword are already the same color. Tracking current_color across the loop means one word, one string, one run of digits gets exactly one leading escape. It's the same "accumulate, then act once" instinct the append buffer itself embodies (Chapter 06) — applied here to the content written into that buffer, not just to how it's flushed.

And notice what happens at the very end of every row, unconditionally: \x1b[39m resets the foreground, \x1b[0K erases to end of line, then the ordinary \r\n. Whatever color a row ended in, it never leaks into the next one — the reset isn't left to chance or to whichever branch happened to fire last.

One more thing that happens before any of this: choosing E.syntax

Everything above reads E.syntax as if it's already correctly set. It is, and by the time a single row has been loaded. main() calls editorSelectSyntaxHighlight(argv[1]) — matching the filename against a table of known languages and setting the global E.syntax pointer, or leaving it NULL — before editorOpen() reads a single line of the file. Since editorInsertRow() (called once per line while opening) already ends in editorUpdateRow()/editorUpdateSyntax(), every row is highlighted correctly from the very first paint. There's no separate "now go color the file I already loaded" pass to get wrong or forget to call.

What that table of languages actually contains — how a filename pattern picks a language, how a keyword list is written, where the comment markers come from — is Chapter 16's subject. For now, treat E.syntax as a switch this chapter doesn't build, only reads: either it's set to something with a keyword list and comment markers, or it's NULL and every row in the file is HL_NORMAL, full stop.

Edge cases: what this architecture quietly guarantees (and doesn't)

  • No filetype match means no highlighting, not an error. E.syntax == NULL is a normal, silent state — opening a .txt file or one with no extension just means every row's hl stays HL_NORMAL after the one early return.
  • Highlighting is recomputed, never patched. There is no code path that updates a single byte of hl in place in response to a keystroke — every touched row is fully rebuilt, every time, by the same editorUpdateRow call that already had to rebuild render.
  • An edit's blast radius isn't bounded to one row. The hl_oc propagation means a single keystroke can, in the worst case, trigger a rehighlight cascade through every subsequent row in the file — bounded by file length, not by what was actually typed.
  • hl indexes render, not chars. The same split Chapter 08 drew between on-disk bytes and on-screen bytes applies here: a highlight byte at position i describes render[i]. On a line with tabs before a highlighted span, that index does not line up with the same position in chars — the identical class of gap Chapter 13 found between a search match's offset and E.cx.
  • The color escapes are coalesced, but the classification isn't cached across the pass. current_color in the drawing loop only avoids redundant escapes within a single frame's draw of a single row; it has no memory from one refresh to the next, and no bearing on whether editorUpdateSyntax reruns.
  • HL_MATCH is structurally identical to every other type, but produced from outside. Nothing in editorUpdateSyntax or editorSyntaxToColor treats it specially beyond mapping it to blue — the special case, if there is one, is entirely in who's allowed to write it (Chapter 13's editorFind, and no one else).

So what did we actually gain?

Two shapes, both reused from chapters you've already read. First: highlighting is just a third parallel buffer, the same pattern chars/render already established in Chapter 08 — one more array, same length as its partner, filled by one more function, hung off the same editorUpdateRow hook every edit already runs through. No new event, no new call site to remember. Second: turning data into color is, again, the append-buffer discipline from Chapters 05 and 06 — decide what to write, batch it, emit it once, reset cleanly at the boundary — just aimed at escape sequences instead of screen content. The one genuinely new idea is the hl_oc carry-over and its recursive propagation: proof that "per-row" and "row-local" aren't the same thing the instant a token (a comment) is allowed to span more than one line. Chapter 15 spends its whole length on the scan this chapter elided — what actually makes something a keyword, a string, or a number. The scaffolding it plugs into is everything you just read.

Before you turn the page
  1. Add a tenth highlight type, HL_TODO, for the word TODO appearing inside a comment. Where in editorUpdateSyntax would you have to check for it, and why does it matter that the check has to happen after the comment-detection branches rather than before them?
  2. Trace through what happens if you delete the closing */ of a multi-line comment on line 5 of a 2,000-line file. Which function notices, and roughly how many rows end up rehighlighted before the recursion in editorUpdateSyntax stops?
  3. The drawing loop's current_color optimization only spans one row (it's a local variable, reset to -1 at the top of each row's draw). Would coalescing color escapes across row boundaries be worth doing, given that every row already ends with an unconditional \x1b[39m reset? What would you have to give up to make it work?
