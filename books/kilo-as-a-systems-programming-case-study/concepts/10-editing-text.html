Chapter 10 · Part III — Data & Navigation

Editing Text

Chapter 09 gave the cursor two coordinate systems and a lot of careful arithmetic for turning arrow keys into filerow/filecol pairs — and then never once changed the document. This chapter spends that arithmetic. Typing a character, pressing Enter, pressing Backspace: every one of them starts by computing the exact same filerow/filecol pair navigation already taught you to compute, and then does something navigation never had to — grow an array, split a string in half, or glue two lines into one.

Motivation: three keys, one job

Look at editorProcessKeypress()'s dispatch switch — the same one this book has already shown you for arrows and Page Up/Down — and every key that changes the document funnels into exactly three functions. case ENTER: calls editorInsertNewline(). case BACKSPACE: case CTRL_H: case DEL_KEY: — three different key codes sharing one case block — all call editorDelChar(). And the default: branch, the one that catches anything not otherwise matched, calls editorInsertChar(c). That default branch is worth sitting with: in kilo.c, “insert this character” isn't a special case triggered by looking ahead for printable ASCII — it's simply what happens to any byte the dispatcher didn't already claim for something else. Search the whole file and you won't find a fourth entry point. No word-delete, no line-delete, no undo. Three functions are the entire editing surface of this editor.

Design: one primitive per layer, reused by all three

Chapter 08 built two functions that mutate a single row's bytes — editorRowInsertChar() and editorRowDelChar() — plus editorRowAppendString() for gluing one row onto the end of another. Chapter 08 also gave you the row-array primitives, editorInsertRow() and editorDelRow(), for growing or shrinking the file by a whole line. This chapter doesn't introduce a single new primitive. editorInsertChar(), editorInsertNewline(), and editorDelChar() are all thin dispatchers: each one recomputes filerow/filecol, decides which of those five already-written primitives applies, calls it, and then repeats the exact one-column scroll rule Chapter 09 already taught you for the arrow keys. Editing text, in kilo.c, is navigation plus five function calls you've already read.

Implementation: editorRowInsertChar, the byte-level primitive

Everything that inserts a single character eventually calls this. It has two branches, split on whether at lands inside the existing text or past the end of it:

void editorRowInsertChar(erow *row, int at, int c) {
    if (at > row->size) {
        int padlen = at-row->size;
        row->chars = realloc(row->chars,row->size+padlen+2);
        memset(row->chars+row->size,' ',padlen);
        row->chars[row->size+padlen+1] = '\0';
        row->size += padlen+1;
    } else {
        row->chars = realloc(row->chars,row->size+2);
        memmove(row->chars+at+1,row->chars+at,row->size-at+1);
        row->size++;
    }
    row->chars[at] = c;
    editorUpdateRow(row);
    E.dirty++;
}

The ordinary case (at <= row->size, the else branch) is a textbook insert-in-the-middle-of-a-buffer: grow by two bytes (one for the new character, one because a C string always needs room for its null terminator), then memmove everything from at onward — the null terminator included — one slot to the right, opening a gap. row->chars[at] = c, common to both branches, then overwrites whatever got duplicated into that gap with the real character.

The other branch is the more interesting one, because it answers a question navigation never had to: what happens if you try to insert past the last real character of a line? This if (at > row->size) block is the only place in the file that handles it — not in editorMoveCursor(), not anywhere in Chapter 09. Kilo's answer is to pad: fill the gap between the old end of the line and at with literal ASCII space characters, place the new character right after the padding, and grow row->size by padlen+1. Type a character past the visual end of a short line and Kilo doesn't refuse, doesn't wrap, doesn't beep — it writes real spaces into the file to get there.

Implementation: editorInsertChar wraps the primitive in file-growth and scroll

editorInsertChar(int c) opens exactly like every function in the previous chapter — recompute filerow/filecol from the four cursor fields — and then handles a case that's specific to editing: what if the cursor is sitting on a row that doesn't exist yet? That happens whenever the file is shorter than the cursor's current screen position (an empty file, or a cursor parked past the last line). The fix is a loop, not a single call: while(E.numrows <= filerow) editorInsertRow(E.numrows,"",0); — append blank rows one at a time until a real row exists at filerow, then call editorRowInsertChar() on it. After the character lands, the cursor advance reuses Chapter 09's edge rule verbatim: if E.cx is already sitting on the rightmost visible column, pan E.coloff forward instead of moving the cursor; otherwise, plain E.cx++.

Nothing new under the cursor

Compare this to editorMoveCursor()'s ARROW_RIGHT case from Chapter 09: same edge check, same pan-or-bump choice. Kilo doesn't have a separate “cursor advances because I typed” rule and a “cursor advances because I pressed Right” rule — typing a character and pressing the Right arrow both end by asking the identical question: is the actor at the edge of the frame, or can it just step forward?

Implementation: Enter splits, grows, or both

editorInsertNewline() is the first function in this chapter that genuinely branches three ways, all keyed off where filecol sits once it's been clamped to the row's length (a cursor conceptually past the last character is treated as exactly at it):

void editorInsertNewline(void) {
    int filerow = E.rowoff+E.cy;
    int filecol = E.coloff+E.cx;
    erow *row = (filerow >= E.numrows) ? NULL : &E.row[filerow];

    if (!row) {
        if (filerow == E.numrows) {
            editorInsertRow(filerow,"",0);
            goto fixcursor;
        }
        return;
    }
    if (filecol >= row->size) filecol = row->size;
    if (filecol == 0) {
        editorInsertRow(filerow,"",0);
    } else {
        editorInsertRow(filerow+1,row->chars+filecol,row->size-filecol);
        row = &E.row[filerow];
        row->chars[filecol] = '\0';
        row->size = filecol;
        editorUpdateRow(row);
    }
fixcursor:
    if (E.cy == E.screenrows-1) {
        E.rowoff++;
    } else {
        E.cy++;
    }
    E.cx = 0;
    E.coloff = 0;
}

Past the last row entirely (row == NULL): only handled if the cursor sits exactly one past the last row (filerow == E.numrows) — append one empty row and jump straight to the shared cursor-fixup with goto fixcursor. Any farther out and the function just returns, doing nothing.

At the very start of a line (filecol == 0): editorInsertRow(filerow,"",0) inserts a blank row before the current one. There's no separate step that moves the existing line's text down — editorInsertRow()'s own memmove already shifts every row at or after filerow one slot down, so the original line's content simply ends up one row lower, unchanged. “Insert a blank line above the cursor” and “push my current line down by one” are not two different operations in this code — they're the same memmove, described two ways.

Anywhere else in the line: this is the actual split. editorInsertRow(filerow+1, row->chars+filecol, row->size-filecol) copies everything from the cursor onward into a brand-new row immediately below the current one. Then — and this matters — row is re-fetched from E.row[filerow] rather than reused, because editorInsertRow() just realloc'd the whole E.row array and the old pointer may no longer be valid. Only then does the original row get truncated in place — row->chars[filecol] = '\0'; row->size = filecol; — followed by an explicit editorUpdateRow(row) call. That call is not optional bookkeeping: writing directly into chars/size bypasses every helper function that would normally trigger a render rebuild on its own, so the split's first half needs its own manual refresh even though the second half already got one for free, inside editorInsertRow().

Implementation: Backspace, Delete, and the line-merge

editorDelChar() handles Backspace, Ctrl-H, and the Delete key — all three arrive at the same case block in the dispatcher and all three call this one function:

void editorDelChar(void) {
    int filerow = E.rowoff+E.cy;
    int filecol = E.coloff+E.cx;
    erow *row = (filerow >= E.numrows) ? NULL : &E.row[filerow];

    if (!row || (filecol == 0 && filerow == 0)) return;
    if (filecol == 0) {
        filecol = E.row[filerow-1].size;
        editorRowAppendString(&E.row[filerow-1],row->chars,row->size);
        editorDelRow(filerow);
        row = NULL;
        if (E.cy == 0)
            E.rowoff--;
        else
            E.cy--;
        E.cx = filecol;
        if (E.cx >= E.screencols) {
            int shift = (E.screencols-E.cx)+1;
            E.cx -= shift;
            E.coloff += shift;
        }
    } else {
        editorRowDelChar(row,filecol-1);
        if (E.cx == 0 && E.coloff)
            E.coloff--;
        else
            E.cx--;
    }
    if (row) editorUpdateRow(row);
    E.dirty++;
}

The very first line is a guard: if (!row || (filecol == 0 && filerow == 0)) return;. Backspace at the first character of the file — nothing before the cursor, nowhere to merge into — is a silent no-op. No error, no beep, nothing on the status line. Past that guard there are exactly two paths.

Ordinary case, filecol != 0: editorRowDelChar(row, filecol-1). Notice the -1 — Backspace deletes the character before the cursor position, not the one at it (that's what Delete would do, and this codebase doesn't distinguish the two: all three keys share this one function and this one branch). The scroll-back that follows is, once again, the mirror of a rule you've already seen: give back a column of scroll if there's any to give (E.coloff--), otherwise just step the cursor left.

Line-merge case, filecol == 0: Backspace at the very start of a line joins it to the line above. Three things happen in order. First, filecol is repurposed to hold E.row[filerow-1].size — the previous line's length, which is exactly the seam where the join will happen. Second, editorRowAppendString(&E.row[filerow-1], row->chars, row->size) glues the entire current line onto the end of the previous one. Third, editorDelRow(filerow) removes the now-redundant row and renumbers everything after it. The cursor then moves up (E.rowoff-- if it was already on the screen's top row, else plain E.cy--) and lands at the captured seam — exactly between the old text and the text that was just appended to it, which is where a cursor should be after “typing over” a newline.

A sign error, verified by working the arithmetic

Look again at what happens right after the seam is computed: if (E.cx >= E.screencols) { int shift = (E.screencols-E.cx)+1; E.cx -= shift; E.coloff += shift; }. Work it with numbers, screen width 80 columns. If the seam lands exactly one column past the edge (E.cx == 80): shift = (80-80)+1 = 1, giving E.cx = 79 — correct, the cursor sits on the last visible column. But at E.cx == 81: shift = (80-81)+1 = 0 — E.cx is left completely unchanged, still one column past the valid range. And at E.cx == 200 (backspacing at the start of a short line whose neighbor above is much longer than the screen): shift = (80-200)+1 = -119, so E.cx -= shift adds 119, pushing the cursor to 319 — further off screen, not back onto it — while E.coloff += shift moves the viewport the wrong way at the same time. The formula only happens to work at the single boundary case E.cx == E.screencols; for any larger overflow the sign is backwards. Reproduce it yourself against a fresh clone before repeating it as fact — the same habit this book has asked of you since Chapter 09's Home/End gap.

A quirk you've already met, recurring here

editorRowDelChar(), called by the ordinary mid-line branch above, calls editorUpdateRow(row) one line before decrementing row->size — the opposite order from editorRowInsertChar() and editorRowAppendString(), both of which finish adjusting size before they're done. Chapter 08 already traced through why: the preceding memmove shifts the null terminator one byte left, so for one call editorUpdateRow() reads with the old, larger size and copies that shifted-in null byte into render as if it were content. It's self-correcting by the very next update and Chapter 08 called it harmless in practice — but every Backspace in the middle of a line runs through it, so it's worth remembering exactly where this chapter depends on that earlier finding.

Edge cases: what these three functions quietly guarantee (and don't)

  • Typing past the end of a line writes real spaces, not a refusal. editorRowInsertChar()'s padding branch is the only code path that handles it, and what it does is pad the file with ASCII spaces up to the insertion point.
  • Backspace deletes the character before the cursor, not at it. The filecol-1 passed to editorRowDelChar() is easy to skim past; Delete and Backspace are not distinguished anywhere in this function.
  • “Insert a blank line above” and “split the current line” are the same primitive, called with different arguments. editorInsertNewline() never special-cases the blank-line insertion separately from the mid-line split — both go through editorInsertRow().
  • There is no undo, no word-delete, no line-delete. A full-file search for writes to row->chars or row->size turns up only the five primitives from Chapter 08 plus this chapter's three dispatchers — nothing else in kilo.c ever mutates a row's text. Later chapters that add these features are building them from nothing, not exposing something already half-there.
  • E.dirty is a counter, not a flag. It's incremented — never just set to 1 — at the bottom of every row-mutating primitive and again inside editorInsertChar()/editorDelChar() themselves, so a single keystroke can bump it by more than one. Every place that reads it only checks whether it's nonzero, so the exact count never surfaces — but Chapter 12 will lean on that nonzero-ness to decide whether there's anything to save.

So what did we actually gain?

A second use for arithmetic you'd already written. Every function in this chapter opens with the exact same two lines Chapter 09 opened every navigation function with — recompute filerow/filecol — and then spends that position on a primitive from Chapter 08 instead of a cursor move. Insertion, deletion, splitting a line, merging two lines: none of it required new state, a new coordinate system, or a new scrolling rule. It required reusing the ones you already have. And you've also seen, close up, what that reuse doesn't buy you for free — a re-clamp formula copied from one context can carry a sign error into another, and a call-order quirk found in one chapter keeps mattering every time later code calls the function it lives in. Chapter 11 picks up right where editorInsertRow() already pointed: if you can build a file's rows one line at a time by hand, you can build them the same way from a file on disk.
