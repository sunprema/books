Chapter 09 · Part III — Data & Navigation

Moving Around the File

Chapter 08 gave every line two buffers, chars for truth and render for the screen. This chapter puts the same split to work on a different problem: where is the cursor? It turns out “where” is also two numbers wearing one hat — a position on the screen, and a position in the file — and arrow keys spend most of their time deciding which of the two should actually move.

Motivation: the screen is a window, not the document

A terminal is small. A file is not obligated to be. The moment a document has more lines than E.screenrows or a line wider than E.screencols, “draw the whole thing” stops being an option, and the editor needs a notion of a viewport — a rectangle of the file currently visible — that can slide independently of where the cursor sits inside it. Kilo's answer is to give the cursor two coordinate systems instead of one: E.cx/E.cy, the cursor's position on screen, always somewhere inside [0, E.screencols) × [0, E.screenrows); and E.rowoff/E.coloff, the file row and column sitting in the screen's top-left corner. Add them together and you get the cursor's real position in the document: filerow = E.rowoff+E.cy, filecol = E.coloff+E.cx. That one line of arithmetic, repeated at the top of nearly every function in this chapter, is the whole idea.

Think of it as a camera and an actor. rowoff/coloff are where the camera is pointed at the document; cx/cy are where the actor stands inside the camera's frame. Press an arrow key near the middle of the screen and only the actor moves. Press it at the very edge of the frame and the camera has to pan instead — the actor stays put (or nearly), and the offsets change underneath it. Every branch you're about to read is one of those two decisions, made once per key.

Design: no scroll function, just edge cases inside cursor movement

Search kilo.c for editorScroll and you'll find nothing. There is no separate scrolling subsystem. Scrolling isn't a feature bolted alongside cursor movement — it's a fallthrough case inside it. One function, editorMoveCursor(int key), handles all four arrow keys, and each of its four branches independently decides: are we still inside the visible screen, or did we just walk off the edge of it?

void editorMoveCursor(int key) {
    int filerow = E.rowoff+E.cy;
    int filecol = E.coloff+E.cx;
    int rowlen;
    erow *row = (filerow >= E.numrows) ? NULL : &E.row[filerow];

    switch(key) {
    case ARROW_LEFT:
        if (E.cx == 0) {
            if (E.coloff) {
                E.coloff--;
            } else {
                if (filerow > 0) {
                    E.cy--;
                    E.cx = E.row[filerow-1].size;
                    if (E.cx > E.screencols-1) {
                        E.coloff = E.cx-E.screencols+1;
                        E.cx = E.screencols-1;
                    }
                }
            }
        } else {
            E.cx -= 1;
        }
        break;
    case ARROW_RIGHT:
        if (row && filecol < row->size) {
            if (E.cx == E.screencols-1) {
                E.coloff++;
            } else {
                E.cx += 1;
            }
        } else if (row && filecol == row->size) {
            E.cx = 0;
            E.coloff = 0;
            if (E.cy == E.screenrows-1) {
                E.rowoff++;
            } else {
                E.cy += 1;
            }
        }
        break;
    case ARROW_UP:
        if (E.cy == 0) {
            if (E.rowoff) E.rowoff--;
        } else {
            E.cy -= 1;
        }
        break;
    case ARROW_DOWN:
        if (filerow < E.numrows) {
            if (E.cy == E.screenrows-1) {
                E.rowoff++;
            } else {
                E.cy += 1;
            }
        }
        break;
    }
    /* Fix cx if the current line has not enough chars. */
    filerow = E.rowoff+E.cy;
    filecol = E.coloff+E.cx;
    row = (filerow >= E.numrows) ? NULL : &E.row[filerow];
    rowlen = row ? row->size : 0;
    if (filecol > rowlen) {
        E.cx -= filecol-rowlen;
        if (E.cx < 0) {
            E.coloff += E.cx;
            E.cx = 0;
        }
    }
}

Four cases, then one shared cleanup at the bottom. Read them slowly — the logic is simple arithmetic, but each branch is answering a slightly different question, and it's worth being able to state each one in a sentence before moving on.

Implementation: Left and Right cross line boundaries

ARROW_LEFT asks: is the cursor already at the left edge of the current line's visible slice (E.cx == 0)? If not, it's the easy case — E.cx -= 1 and done. If it is at the left edge, the branch first tries to spend horizontal scroll before spending anything else: if E.coloff is nonzero (the viewport has panned right of column 0), decrement it — the camera pans back, the actor doesn't move at all. Only once coloff is also back to zero, and only if there's a previous line to go to (filerow > 0), does Left actually cross a line boundary: E.cy--, and E.cx jumps to E.row[filerow-1].size — the end of the previous line. Kilo treats the file as one continuous stream of characters with newlines woven in, not a grid of independent rows; pressing Left at column zero is never a no-op once you're past the first line. A clamp right after that jump (if (E.cx > E.screencols-1)) handles the case where that previous line is wider than the screen — it sets coloff to bring the new cursor position on screen rather than leaving E.cx pointing past the visible area.

ARROW_RIGHT is the mirror image, and it only runs at all if a row exists at the cursor's file row (row isn't NULL). Two sub-cases, split on where filecol sits relative to row->size. Still inside the line (filecol < row->size): same edge logic as Left, mirrored — bump E.cx normally, or bump E.coloff instead once E.cx is sitting on the rightmost visible column. Exactly at the end of the line (filecol == row->size — one position past the last character, which is a valid, ordinary cursor position, not an error): Right wraps down to the start of the next line, resetting both E.cx and E.coloff to zero and either bumping E.cy or panning E.rowoff depending on whether the screen's bottom row is already showing.

The camera moves only when the actor hits the frame

Notice the shape repeated in both branches: try the cheap move first (slide within the current screen row), and only touch the offset (rowoff/coloff) once the cursor is already pinned to the edge of what's visible. Every scroll in Kilo happens this way — one line, one edge check, one offset increment. There is no code path that scrolls by more than one row or one column in response to a single arrow key.

Implementation: Up and Down never look sideways

ARROW_UP and ARROW_DOWN are shorter, and deliberately don't do what Left and Right do. Up: if the cursor is already on the top visible screen row (E.cy == 0), pan the camera up by decrementing E.rowoff (only if there's room — if (E.rowoff) guards against panning above line 0); otherwise just move the actor, E.cy -= 1. Down is the same shape flipped: refuse to move at all once filerow would run past E.numrows (there's no line to move onto), otherwise either pan E.rowoff forward or bump E.cy, depending on whether the screen's last row is already in view.

Neither branch touches E.coloff or E.cx at all. Move from a long line to a short one with the Down key, and E.cx is left exactly where it was — possibly now pointing well past the shorter line's last character. That's not a bug left for you to find; it's intentional, and the function's last few lines exist entirely to resolve it.

Implementation: one shared clamp, no memory of where you've been

After the switch, editorMoveCursor() recomputes filerow/filecol one more time (they may have changed inside the branch that just ran), looks up whatever row the cursor now sits on, and applies a single rule: if filecol is past that row's length, pull E.cx back by the overshoot, and if that pulls it negative, absorb the remainder into E.coloff instead and pin E.cx to zero. This is the entire mechanism behind “arrow down from a long line onto a short one snaps the cursor to end-of-line” — and it's worth being precise about what it isn't: there is no preferred_cx field, no remembered “the column I was really trying to stay on.” Every move recomputes and reclamps from scratch. Arrow back up onto the original long line and the cursor lands at whatever column the clamp left it at last time — not necessarily where you started.

A feature many editors have that Kilo doesn't

Editors that do remember a “preferred column” let you arrow down through several short lines and back up through them without losing your original horizontal position. Kilo's clamp has no memory across calls — E.cx is genuinely wherever the last clamp left it, full stop. That's a legitimate, deliberate simplicity trade for a ~1000-line editor, not an oversight; just don't expect the behavior if you've gotten used to a bigger editor's version of vertical movement.

Implementation: Page Up / Page Down are a loop, not a jump

You might expect paging to reassign E.rowoff directly — jump the camera by a whole screen's height in one write. That's not what happens. The PAGE_UP/PAGE_DOWN handler lives in editorProcessKeypress(), not editorMoveCursor(), and it's built entirely out of the single-step logic you just read:

case PAGE_UP:
case PAGE_DOWN:
    if (c == PAGE_UP && E.cy != 0)
        E.cy = 0;
    else if (c == PAGE_DOWN && E.cy != E.screenrows-1)
        E.cy = E.screenrows-1;
    {
    int times = E.screenrows;
    while(times--)
        editorMoveCursor(c == PAGE_UP ? ARROW_UP:
                                        ARROW_DOWN);
    }
    break;

Two steps. First, snap E.cy to whichever edge of the screen the page motion is heading toward — top for Page Up, bottom for Page Down — if it isn't already there. Then run a loop that calls editorMoveCursor(ARROW_UP) or editorMoveCursor(ARROW_DOWN) exactly E.screenrows times in a row. A page motion, in kilo.c, is a screenful of ordinary single-line moves executed back to back, reusing every edge check and every clamp already written for the arrow keys. There's no second scrolling implementation to keep in sync with the first — a deliberate echo of Chapter 08's lesson about not duplicating logic that has to stay consistent.

Edge case: Home and End are parsed, but not wired up

editorReadKey() genuinely recognizes Home and End. Both keys arrive as multi-byte escape sequences — ESC [ H / ESC O H for Home, ESC [ F / ESC O F for End, the two forms different terminal emulators favor — and the parser correctly turns either form into a soft key code, HOME_KEY or END_KEY (both defined at 1000+, alongside the arrow keys, in the same enum):

/* ESC [ sequences (subset). */
switch(seq[1]) {
case 'A': return ARROW_UP;
case 'B': return ARROW_DOWN;
case 'C': return ARROW_RIGHT;
case 'D': return ARROW_LEFT;
case 'H': return HOME_KEY;
case 'F': return END_KEY;
}
/* ESC O sequences. */
else if (seq[0] == 'O') {
    switch(seq[1]) {
    case 'H': return HOME_KEY;
    case 'F': return END_KEY;
    }
}

But walk through editorProcessKeypress()'s dispatch switch — the one this book has already shown you for Page Up/Down and the arrow keys — and there is no case HOME_KEY:, no case END_KEY:, anywhere in it. Both values fall straight through to the switch's default: branch, which calls editorInsertChar(c) on whatever integer c happens to be. editorInsertChar() hands that value down to editorRowInsertChar(), which writes it into a one-byte char slot — row->chars[at] = c; — silently truncating a code like 1005 or 1006 down to whatever byte survives the cast.

Recognized, but never acted on

In the shipped source, pressing Home or End does not move the cursor to the start or end of the current line. The key is correctly identified two layers down in the input pipeline — the escape-sequence parsing is not the problem — and then simply has nowhere to go once it reaches the dispatcher, so it falls into the same bucket as an ordinary typed character and inserts a stray byte into the document instead. This is the same species of finding as Chapter 08's backwards .idx renumbering: a small, verifiable gap in a celebrated, widely-read program, worth noticing rather than assuming away. Confirm it yourself against a fresh clone before repeating it as fact — that's the right instinct, not just for this book.

Edge cases: what the offset arithmetic quietly guarantees

  • E.cx/E.cy are always screen coordinates, never file coordinates. Any code that needs to know “where is the cursor in the document” must compute filerow = E.rowoff+E.cy and filecol = E.coloff+E.cx itself — there is no single field to read that already holds that answer.
  • Scrolling only ever moves one row or one column per key press. Every offset change in editorMoveCursor() is a bare ++ or --. Page Up/Down don't violate this — they just call the single-step function many times.
  • Left and Right cross line boundaries; Up and Down never adjust coloff. The two pairs of keys are not symmetric in what they touch. Left/Right own line-wrapping and horizontal scroll; Up/Down own vertical scroll and rely entirely on the shared post-switch clamp to fix up anything horizontal.
  • There is no remembered preferred column. The post-switch clamp recomputes E.cx fresh from filecol and the current row's length on every single move — nothing here is intended to reproduce “editors that hold your column steady while arrowing through blank lines” behavior.
  • Home and End are a genuine gap between input parsing and input handling. They're recognized correctly and then never dispatched — a reminder that “the terminal sends the right bytes” and “the editor does the right thing with them” are two separate claims, and this codebase only fully backs one of them for these two keys.

So what did we actually gain?

A cursor that never has to know how big the file is. By splitting position into a screen half (cx/cy, always small, always in range) and a file half (rowoff/coloff, unbounded, tracking the document), Kilo turns “scroll a gigantic file smoothly” into “increment or decrement one integer at the right moment” — the same instinct Chapter 08 used to turn “render a tab correctly” into a second buffer. And you've now seen the trade-off up close: this design buys simplicity by giving up any memory of past cursor state, which is exactly why arrowing back up a column you just arrowed down through doesn't always put you back where you started, and exactly why a correctly-parsed key can still do nothing useful if nobody remembered to give it a case in the dispatcher. Chapter 10 builds on this same filerow/filecol pair to do the thing navigation was always in service of: actually changing what's in the file.

Before you turn the page
  1. Trace a cursor sitting at the last column of a long line, then press Down twice in a row onto two progressively shorter lines, then Up twice back to the original line. At each step, write down E.cx, E.coloff, and the row's actual length. Where does the cursor end up relative to where it started, and why?
  2. Write the two missing case labels that would make Home and End actually work — jump to filecol = 0 for Home, filecol = row->size for End — reusing the same rowoff/coloff adjustments editorMoveCursor()'s ARROW_LEFT/RIGHT cases already make when a cursor lands outside the visible screen width.
  3. Page Up/Page Down move by calling editorMoveCursor() E.screenrows times. On a 1,000-line file with a 40-row terminal, roughly how many individual function calls does holding Page Down for ten pages make? Is that a real performance concern for the file sizes Kilo targets, or a cost worth paying for reusing one already-correct function instead of writing a second one?
