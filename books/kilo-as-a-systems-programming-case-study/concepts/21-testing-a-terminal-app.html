Chapter 21 · Part VII — Beyond the Tutorial

Testing a Terminal App

Chapter 20 closed by promising that "verified" would mean more than "I tried it once," and that promise is due now. Chapter 19 already found the uncomfortable fact this chapter has to answer: antirez/kilo ships with zero automated tests and no CI configuration at all — a direct check for every common test/CI path on the repository comes back 404, across the board. That's not a defect unique to Kilo; it's the default state of a program written, by its own README, "in just a few hours." But it means this chapter can't show you Kilo's tests, because there aren't any. It has to design them from a cold start, against a program that was never given a seam to test through.

Motivation: why kilo.c resists testing as written

Two structural facts about kilo.c, both established earlier in this book, are exactly what make testing it hard. First, almost everything hangs off one global struct editorConfig E (Chapter 7) — a function that reads and writes a global can't be run twice, in isolation, with two different states in the same process, the same problem Chapter 19 named when it discussed threading state through as a parameter instead. Second, Kilo talks to the terminal directly: enableRawMode calls tcgetattr/ tcsetattr (Chapter 4), and those calls fail outright with ENOTTY unless the file descriptor genuinely refers to a terminal device. Redirect Kilo's stdin from a plain pipe or a file — the way you'd naively script any other command-line program — and isatty() returns false. Raw mode never engages. You haven't tested the raw-mode path; you've silently tested a different, uninteresting one.

A pipe is not a terminal

This is the trap that swallows a first attempt at testing almost every terminal program, not just Kilo. subprocess.Popen(..., stdin=PIPE) or ordinary shell redirection hands the child process a pipe, and a pipe is not a terminal device — isatty() on it returns false. What you need instead is a pseudo-terminal (a PTY): a kernel-provided pair of devices where one end genuinely presents as a terminal to whatever program is attached to it. Every testing approach in this chapter that drives the real binary end-to-end depends on getting this right.

Design: three layers, three different tools

Not every part of kilo.c has the same relationship to the terminal, and treating them as one testing problem is why testing a program like this feels harder than it needs to be. Split it into three layers, and each one has an obvious tool once you see it that way.

Testing the pure logic: a seam that already exists

Not every function in kilo.c touches the global E. Chapter 10 covered editorRowInsertChar(erow *row, int at, int c) in detail: it takes the row it mutates as an explicit pointer argument, reallocates that row's buffer, and returns. It never reads or writes E at all. Compare that to editorInsertRow(int at, char *s, size_t len), which reallocates E.row directly — a function that can't run without the global existing first. editorRowInsertChar is already, with zero refactoring, callable in isolation: build an erow by hand, call it, check the result.

/* Illustrative sketch — not in antirez's source. */
#include "minunit.h"

MU_TEST(row_insert_char_grows_row) {
    erow row = {0};
    row.chars = malloc(1);
    row.chars[0] = '\0';
    row.size = 0;

    editorRowInsertChar(&row, 0, 'h');
    editorRowInsertChar(&row, 1, 'i');

    mu_assert_int_eq(2, row.size);
    mu_check(strncmp(row.chars, "hi", 2) == 0);
}

minunit is a fitting choice here for a reason beyond convenience: it's a single-header C testing framework built around a handful of macros (MU_TEST, MU_TEST_SUITE, mu_check, mu_assert_int_eq) with no dependency beyond the standard library. A project whose own README brags "does not depend on any library" (Chapter 3) shouldn't reach for a test framework that quietly undoes that. The point generalizes past minunit specifically: general console-app testing practice is to write logic against explicit, passed-in parameters rather than global state or direct I/O calls, precisely so it can run under a fake in a test and under the real thing in production without being duplicated.

Look for functions that already take a pointer

editorDelChar, by contrast, recomputes filerow/ filecol from E.rowoff/E.cy/ E.coloff/E.cx before it does anything (Chapter 10) — it can't run without E populated first. That's not a flaw to fix before writing any tests; it's information. Test what's already shaped for testing first, and let which functions are hard to test tell you where Chapter 19's refactor would pay off most.

Testing the redraw: the append buffer is already a seam

Chapter 5 covered Kilo's struct abuf: before a single byte reaches the terminal, editorRefreshScreen assembles the entire redrawn frame — every escape sequence, every row of text — into one heap-growing append buffer. The source's own comment explains why: batching "avoid[s] flickering effects." That design decision, made for a completely different reason, hands testing exactly what it needs. You don't have to intercept a write() syscall or fake a file descriptor to see what Kilo would draw — the fully-assembled frame already exists, in memory, as an ordinary buffer, for one instant before it's flushed. Capture it there.

Don't assert on the raw bytes

Two escape-sequence strings can render an identical screen while differing byte-for-byte — a cursor-position command reissued with the same coordinates, a color re-set to the color it already was. Asserting directly on the captured bytes makes a test fail on changes that have no visible effect at all. pyte, an in-memory VTXXX-compatible terminal emulator, solves this the way a real terminal would: feed the captured bytes into its Stream, and its Screen maintains the resulting character grid and cursor position — so the test asserts on rendered content (screen.display), the same thing a person looking at their terminal would judge Kilo by.

Testing the whole program: through a real pseudo-terminal

Some questions can only be answered by running the actual binary: does pressing Ctrl-F, typing a query, and pressing Enter really land the cursor on the match (Chapter 13)? Does Ctrl-S really write the file and clear the dirty flag (Chapter 12)? For that, you need the real terminal device the callout above insisted on. pexpect spawns a child process inside an actual PTY and scripts it the way a person would: send bytes as if typed, then expect() blocks until a specific pattern appears in the output, with a timeout. Because the child's stdin really is a terminal, enableRawMode's isatty() check succeeds and the raw-mode path actually runs — the exact thing a plain pipe couldn't give you.

# Illustrative sketch — not from any single source verbatim.
import pexpect, pyte

child = pexpect.spawn("./kilo scratch.txt", dimensions=(24, 80))
screen = pyte.Screen(80, 24)
stream = pyte.Stream(screen)

child.send("hello")
child.expect(pexpect.TIMEOUT, timeout=0.2)   # let the redraw happen
stream.feed(child.before.decode("utf-8", "replace"))

assert "hello" in screen.display[0]

If pulling in Python for this feels like the wrong trade for a project this committed to having no dependencies, there's a shell-native version of the same idea: tmux new-session -d starts the real binary attached to a real pty in a detached session, tmux send-keys delivers keystrokes to it, and tmux capture-pane reads back the rendered pane content as plain text. Same PTY, same principle, no Python interpreter required to drive it.

You don't need a refactor to start

Everything above the append-buffer seam assumes kilo.c stays exactly as it is today — and that's the point. A characterization test (Michael Feathers's term, also called a golden master test) records the current, actual output of existing code for a fixed input and asserts future runs match it, without requiring the code to be decomposed into testable units first. Run the real kilo binary through a PTY with a fixed keystroke script, save the resulting screen as the golden file, and Chapter 19's refactor now has a safety net before a single line moves — solving the chicken-and-egg of "you need tests to refactor safely, but refactoring is what the untested code needs."

Edge cases: where a PTY test quietly lies to you

A PTY makes isatty() pass, but it doesn't make a test deterministic for free. Two specific things bite here, both traceable to earlier chapters.

Kilo's read loop can outrun your test

Chapter 4 set c_cc[VMIN]=0 and c_cc[VTIME]=1, so read() can return before a just-sent keystroke has been fully processed and redrawn. A script that fires off a long chain of keystrokes and only checks the result at the very end can race the program under test and produce results that depend on scheduling, not logic. Send one action, wait for confirmation, then send the next — the same discipline good PTY-testing practice recommends generally.

An unpinned window size breaks reproducibility

Chapter 9's getWindowSize/SIGWINCH handling recomputes E.screenrows/E.screencols from whatever size the terminal reports. A PTY's dimensions are configurable (pexpect's dimensions= argument above pins them); leave them unset and a captured screen dump depends on whatever size your terminal or CI runner happens to default to — the same golden file passing on your machine and failing in CI, for a reason that has nothing to do with a real bug.

Once the pure-logic layer is genuinely isolated from E and from the terminal, it becomes a fuzzing target worth naming even briefly: Kilo's keypress dispatch parses raw bytes, including multi-byte escape sequences (Chapter 4's arrow-key and Page-Up/Page-Down decoding). Feeding that parser truncated or malformed escape byte sequences — the kind a flaky terminal emulator or a half-received network packet could actually produce — is exactly the kind of input space a person writing a keystroke script by hand won't think to cover, but a fuzzer will.

So what did we actually gain?

Not a test suite — kilo.c still doesn't have one, and this chapter didn't write it one. What it gained is a map of where three different kinds of test actually belong, and why reaching for the wrong one is the reason testing a terminal program feels harder than it is. Functions like editorRowInsertChar that already take their state as a parameter need nothing but an ordinary unit test. The append buffer editorRefreshScreen already builds, for one instant, before it's flushed — capture that, and feed it through a terminal emulator like pyte so the assertion is about what a person would actually see. And for the whole program, a pseudo-terminal — not a pipe — is what lets enableRawMode behave as it would for a real user, which is what makes a golden-master test over that PTY session a legitimate safety net for Chapter 19's refactor, starting from exactly the untouched code that exists today. Chapter 22 is where all of this — the terminal handling, the data model, the editing operations, and now the seams for testing them — gets held in your head all at once, as one architecture rather than twenty-one chapters.
