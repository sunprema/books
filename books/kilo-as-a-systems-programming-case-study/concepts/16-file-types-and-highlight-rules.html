Chapter 16 · Part V — Search & Highlighting

File Types and Highlight Rules

Chapter 15 read the entire C/C++ scan body and closed on a promise: that scan is generic over which keywords and delimiters it's given, so a second language shouldn't need a single line of it touched. This chapter is where we cash that promise in. Nothing new gets colored here — instead we go one level up, to the two questions every one of the last two chapters quietly assumed had already been answered: how does Kilo decide which table of keywords and delimiters applies to the file you just opened, and what would it actually take to add a second entry to that table? Both answers turn out to live in about twenty lines of code and one comment block antirez left for exactly this purpose.

Motivation: E.syntax didn't set itself

Go back to the opening line of Chapter 14's editorUpdateSyntax: if (E.syntax == NULL) return;. Every claim in the last two chapters about keywords turning yellow, types turning green, strings surviving an escaped quote — all of it was conditioned on E.syntax already pointing at the right table by the time a single row got scanned. We never asked how it got there. It turns out the answer is deliberately boring: one function, called once, before the file is even read.

The struct that E.syntax points at is one we've been reading through piecemeal since Chapter 15. Here it is whole:

struct editorSyntax {
    char **filematch;
    char **keywords;
    char singleline_comment_start[2];
    char multiline_comment_start[3];
    char multiline_comment_end[3];
    int flags;
};

Five of these six fields you've already used, through E.syntax, without ever seeing the struct declaration itself: keywords drove Chapter 15's whole scan, the three comment-delimiter buffers gated the single- and multi-line comment branches, and flags is where HL_HIGHLIGHT_STRINGS and HL_HIGHLIGHT_NUMBERS live. The one field this chapter is actually about is the first one: char **filematch. It's the only part of the struct that has nothing to do with coloring a row, and everything to do with choosing which struct instance gets used at all.

Design: one array, one instance, one counted by the compiler

Kilo's language table isn't a linked list you register entries into at runtime, and it isn't read from a config file. It's a plain, statically-initialized C array:

struct editorSyntax HLDB[] = {
    {
        /* C / C++ */
        C_HL_extensions,
        C_HL_keywords,
        "//","/*","*/",
        HL_HIGHLIGHT_STRINGS | HL_HIGHLIGHT_NUMBERS
    }
};

#define HLDB_ENTRIES (sizeof(HLDB)/sizeof(HLDB[0]))

One stanza, one language. But look at how HLDB_ENTRIES is defined: not as a hand-written constant like #define HLDB_ENTRIES 1, but as sizeof(HLDB)/sizeof(HLDB[0]) — the array's total byte size divided by one element's byte size, computed by the compiler at build time. That's the detail that makes the whole mechanism additive. Every loop in the file that walks HLDB does so up to HLDB_ENTRIES, so the moment you add a second { ... } stanza inside the braces above, every one of those loops sees two entries instead of one, with nothing else to update. There's no separate "how many languages do we support" counter sitting somewhere else in the file, waiting to drift out of sync with the array it's supposed to describe.

The array shape was always general; only the contents are C-specific

Chapter 15's callout already flagged this: nothing about struct editorSyntax or HLDB's declaration forces exactly one entry. The struct has a char **filematch field, and HLDB is an array, precisely so a second stanza — Python, JSON, Markdown, whatever you need next — can sit right next to the C/C++ one. This chapter is that door, opened.

Implementation: editorSelectSyntaxHighlight, read whole

Here's the function that turns a filename string into a decision about which HLDB stanza — if any — E.syntax should point at:

/* Select the syntax highlight scheme depending on the filename,
 * setting it in the global state E.syntax. */
void editorSelectSyntaxHighlight(char *filename) {
    for (unsigned int j = 0; j < HLDB_ENTRIES; j++) {
        struct editorSyntax *s = HLDB+j;
        unsigned int i = 0;
        while(s->filematch[i]) {
            char *p;
            int patlen = strlen(s->filematch[i]);
            if ((p = strstr(filename,s->filematch[i])) != NULL) {
                if (s->filematch[i][0] != '.' || p[patlen] == '\0') {
                    E.syntax = s;
                    return;
                }
            }
            i++;
        }
    }
}

Two nested loops, and neither one is doing anything exotic. The outer loop walks the HLDB array itself, entry by entry, up to HLDB_ENTRIES. The inner loop walks that entry's filematch array — the same NULL-terminated-array idiom Chapters 3 and 15 already leaned on for keywords — testing each pattern in turn. The instant a pattern matches, the function sets E.syntax = s and returns immediately. There's no scoring, no "check every entry and pick the best match" pass: the first pattern, in the first entry, that matches, wins. With HLDB_ENTRIES == 1 in the shipped source, the outer loop only ever gets one iteration to prove itself — but nothing about how it's written assumes that.

Design: what a filematch pattern actually means

All of the real logic in that function is one if buried in the middle: if (s->filematch[i][0] != '.' || p[patlen] == '\0'). To see why it's there, start with what strstr(filename, pattern) does on its own: it finds the pattern anywhere inside the filename, as a plain substring, and returns a pointer to where the match starts (or NULL). That's exactly right for a pattern like "Makefile" — the one example kilo.c's own doc comment gives for a non-extension match — because you want "Makefile" to match whether the file is named exactly Makefile or something like GNUmakefile.Makefile. But a bare substring match is far too loose for an extension. ".c" is a substring of foo.c — and also of foo.copy, which is not a C file at all.

The dot check is what closes that gap. If the pattern's first character is '.', the match is only accepted when p[patlen] == '\0' — that is, when nothing follows the matched substring inside the filename, which is only true when the match lands at the very end of the string. .c matches main.c (the match starts at index 4, and index 6 — right past it — is the string's terminating '\0'). .c does not match foo.copy, because the match starts at index 3 and index 5 holds 'o', not '\0'. A pattern with no leading dot skips this extra check entirely and keeps strstr's plain "matches anywhere" behavior.

One if, two matching rules

It's easy to read editorSelectSyntaxHighlight as "it checks extensions" and stop there. It actually implements two different matching rules selected by one leading character: a dot-prefixed pattern means "must match at the end of the filename" (an extension), and every other pattern means "match anywhere in the filename" (a full or partial name, like Makefile). Both rules are expressed by the same strstr call — the dot check only adds a constraint, it never changes what gets searched for.

When selection happens, and what "no match" means

editorSelectSyntaxHighlight is called exactly once per run of the program, from main(), and the order matters:

initEditor();
editorSelectSyntaxHighlight(argv[1]);
editorOpen(argv[1]);
enableRawMode(STDIN_FILENO);

Selection happens before editorOpen reads a single line. That ordering is what makes Chapter 14's claim true — that every row is highlighted correctly from the very first paint, with no second pass that goes back and colors an already-loaded file. editorOpen calls editorInsertRow once per line, and editorInsertRow already ends in editorUpdateRow / editorUpdateSyntax. By the time the first of those calls happens, E.syntax is already whatever editorSelectSyntaxHighlight decided it should be — a real HLDB entry, or still NULL.

NULL is not an error case here; it's the ordinary outcome for any filename that matches nothing in HLDB. E.syntax starts life NULL (set in initEditor), and if the inner while loop runs off the end of every entry's filematch array without a match, E.syntax simply never gets reassigned. That's the exact NULL Chapter 14 built its very first branch around: if (E.syntax == NULL) return;, everything HL_NORMAL. Open a .txt file, or a file with no name at all, and you get a perfectly working editor with plain text — the "no highlighting" path was never a separate feature to build, just this function declining to set a pointer.

Implementation: what adding a second language actually costs

kilo.c documents its own extension recipe in a comment block sitting directly above the C/C++ definitions — addressed to whoever reads this code next, typos and all:

/* =========================== Syntax highlights DB =========================
 *
 * In order to add a new syntax, define two arrays with a list of file name
 * matches and keywords. The file name matches are used in order to match
 * a given syntax with a given file name: if a match pattern starts with a
 * dot, it is matched as the last past of the filename, for example ".c".
 * Otherwise the pattern is just searched inside the filenme, like "Makefile").
 *
 * The list of keywords to highlight is just a list of words, however if they
 * a trailing '|' character is added at the end, they are highlighted in
 * a different color, so that you can have two different sets of keywords.
 *
 * Finally add a stanza in the HLDB global variable with two two arrays
 * of strings, and a set of flags in order to enable highlighting of
 * comments and numbers.
 *
 * The characters for single and multi line comments must be exactly two
 * and must be provided as well (see the C language example).
 *
 * There is no support to highlight patterns currently. */

Strip that down to the three steps it's actually describing, and it matches everything this chapter has walked through, in order: define a filematch array (the patterns editorSelectSyntaxHighlight will test), define a keywords array (Chapter 15's trailing-pipe convention still applies — "if" for a keyword, "int|" for a type), and add one more stanza to HLDB combining both arrays with the four comment/flag fields. Nothing else in the file changes. editorSelectSyntaxHighlight's two loops already walk HLDB_ENTRIES entries, whatever that turns out to be; editorUpdateSyntax's scan already reads E.syntax->keywords and the delimiter fields generically, never a C-specific literal. A second language is additive by construction, not by careful editing.

Two real constraints on what you can add

"Additive" doesn't mean "unconstrained." The doc comment names one limit outright — comment delimiters "must be exactly two" characters — and the struct enforces it structurally: singleline_comment_start[2], multiline_comment_start[3], and multiline_comment_end[3] are fixed-size buffers (2 content bytes plus a null terminator), read by Chapter 15's scan as a rigid pair — scs[0]/scs[1], never a strlen() call. A language whose comment marker is one character (#, for Python or shell) doesn't fit this struct as shipped without changing both it and the scan that reads it. And filematch patterns are plain substring/suffix checks, not globs — the doc comment's own last line says so: "There is no support to highlight patterns currently." One pattern per extension, exactly as C_HL_extensions lists .c, .h, .cpp, .hpp, and .cc as five separate strings rather than one clever pattern.

Worked example: sketching a second stanza

None of this needs to stay abstract. Here's what a (hypothetical, not-shipped) Python entry would look like, built from exactly the three pieces the doc comment names:

char *PY_HL_extensions[] = {".py",NULL};
char *PY_HL_keywords[] = {
    "def","return","if","else","elif","for","while",
    "import","class","try","except",NULL,
    /* types, trailing-pipe convention from Chapter 15 */
    "int|","str|","float|","bool|","list|","dict|",NULL
};

struct editorSyntax HLDB[] = {
    { C_HL_extensions, C_HL_keywords, "//","/*","*/",
      HL_HIGHLIGHT_STRINGS | HL_HIGHLIGHT_NUMBERS },
    /* Python -- hypothetical second entry */
    { PY_HL_extensions, PY_HL_keywords, "#","","",
      HL_HIGHLIGHT_STRINGS | HL_HIGHLIGHT_NUMBERS }
};

Follow it through editorSelectSyntaxHighlight for a file named tool.py: HLDB_ENTRIES is now 2. j=0 checks the C/C++ entry's filematch list — .c, .h, and so on — none of which land at the end of tool.py, so the inner loop exhausts without matching. j=1 checks the Python entry: .py matches, and it's the last two characters of the filename, so p[patlen] == '\0' holds. E.syntax is set to the Python stanza, the function returns, and every row of that file now gets highlighted against Python's keyword list instead of C's — without editorUpdateSyntax or the drawing loop knowing or caring that a second language exists at all.

Notice, too, the one real snag this sketch runs straight into: Python's natural single-line comment marker is #, one character — and the constraint from the callout above bites immediately. Writing "#" into a char[2] is legal C (it's just a shorter string in a same-or-larger buffer), but Chapter 15's scan always reads scs[0] and scs[1] as a pair when it tests for a comment start — so scs[1] would end up reading whatever byte happens to sit after the string's terminator, not a second comment character that doesn't exist. Getting a one-character marker working correctly is exactly the kind of small, honest change — to the scan, not just the table — the "additive, no other changes" story above glosses over. Naming it here beats discovering it by surprise later.

So what did we actually gain?

A second look at something we'd been taking on faith since Chapter 14: E.syntax doesn't appear from nowhere, it's set by one function, called once, before the file is even opened, based on a filename test with exactly one piece of real logic in it — a dot at the front of a pattern means "must match at the end," anything else means "match anywhere." HLDB's size is never hand-counted, so growing the table from one language to two is genuinely additive: a new filematch array, a new keywords array, one new stanza, and every function downstream — selection, scanning, drawing — picks it up with zero edits, because all three were already written generically against "whatever E.syntax points at." That's the payoff of Chapter 15's plain, unglamorous design: a scanner with no notion of "C" baked into its control flow is a scanner that's already ready for a second language, the moment someone writes the table for it. The two real limits worth carrying forward — comment delimiters fixed at two characters, and filename matching with no glob support — aren't oversights to route around quietly; they're the honest edges of a table built to hold exactly the one language Kilo shipped with.
