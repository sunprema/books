Chapter 15 · Part V — Search & Highlighting

Highlighting C Code

Last chapter built the scaffolding and elided one block: the actual body of editorUpdateSyntax's while(*p) loop, the part that decides — one character at a time — whether you're looking at a comment, a string, a number, or a keyword. That block is not a lexer in the sense a compiler course would teach you the word. It's about a hundred lines of C, four if chains stacked in a fixed order, three scalars of memory, and a single NULL-terminated array of strings. No tokens are ever allocated. No tree is ever built. And it correctly colors real C and C++ source, keywords and types in different colors, strings that survive an escaped quote, comments that survive a line break. This chapter is that block, read slowly, in the order its author had to write it in: what has to be checked first, and why.

Motivation: one language, one table, one array of strings

Kilo ships exactly one entry in its language table, and it's for C and C++. Everything this chapter walks through lives in two file-scope arrays and a two-field struct, all defined together, above main():

char *C_HL_extensions[] = {".c",".h",".cpp",".hpp",".cc",NULL};
char *C_HL_keywords[] = {
	/* C Keywords */
	"auto","break","case","continue","default","do","else","enum",
	"extern","for","goto","if","register","return","sizeof","static",
	"struct","switch","typedef","union","volatile","while","NULL",

	/* C++ Keywords */
	"alignas","alignof","and", /* ...45 more... */ "xor_eq",

	/* C types */
        "int|","long|","double|","float|","char|","unsigned|","signed|",
        "void|","short|","auto|","const|","bool|",NULL
};

struct editorSyntax HLDB[] = {
    {
        /* C / C++ */
        C_HL_extensions,
        C_HL_keywords,
        "//","/*","*/",
        HL_HIGHLIGHT_STRINGS | HL_HIGHLIGHT_NUMBERS
    }
};

Read C_HL_keywords closely and it's actually three lists taped end to end with no struct boundary between them: plain C keywords, then C++ keywords, then C/C++ type names — int, char, unsigned and friends. The only thing marking that third group as different is a trailing | baked into the string itself: "int|", not "int". That one extra byte is the entire mechanism this chapter is going to spend a section on, because it's the thing that lets int come out green (a type) and return come out yellow (a keyword) from what is, structurally, the exact same matching loop.

One HLDB entry, not a plugin system

HLDB (Chapter 14 read it as E.syntax, already selected) is declared as an array with exactly one element. Nothing about the mechanism you're reading this chapter forces that — the struct has a char **extensions field precisely so a second entry, for Python or JSON or Markdown, could sit right next to this one. Chapter 16 is where that door gets opened. For now, every filename that isn't .c, .h, .cpp, .hpp, or .cc gets E.syntax == NULL and, per Chapter 14, no highlighting at all.

Design: a fixed order of checks, not a grammar

Here's the whole scan body — the block Chapter 14 folded down to a comment. It's one while(*p) loop, and every character of the row passes through it exactly once, landing in exactly one of five branches before the loop's bottom is ever reached:

while(*p) {
    /* Handle // comments. */
    if (prev_sep && *p == scs[0] && *(p+1) == scs[1]) {
        /* From here to end is a comment */
        memset(row->hl+i,HL_COMMENT,row->size-i);
        return;
    }

    /* Handle multi line comments. */
    if (in_comment) {
        row->hl[i] = HL_MLCOMMENT;
        if (*p == mce[0] && *(p+1) == mce[1]) {
            row->hl[i+1] = HL_MLCOMMENT;
            p += 2; i += 2;
            in_comment = 0;
            prev_sep = 1;
            continue;
        } else {
            prev_sep = 0;
            p++; i++;
            continue;
        }
    } else if (*p == mcs[0] && *(p+1) == mcs[1]) {
        row->hl[i] = HL_MLCOMMENT;
        row->hl[i+1] = HL_MLCOMMENT;
        p += 2; i += 2;
        in_comment = 1;
        prev_sep = 0;
        continue;
    }

    /* Handle "" and '' */
    if (in_string) {
        row->hl[i] = HL_STRING;
        if (*p == '\\') {
            row->hl[i+1] = HL_STRING;
            p += 2; i += 2;
            prev_sep = 0;
            continue;
        }
        if (*p == in_string) in_string = 0;
        p++; i++;
        continue;
    } else {
        if (*p == '"' || *p == '\'') {
            in_string = *p;
            row->hl[i] = HL_STRING;
            p++; i++;
            prev_sep = 0;
            continue;
        }
    }

    /* Handle non printable chars. */
    if (!isprint(*p)) {
        row->hl[i] = HL_NONPRINT;
        p++; i++;
        prev_sep = 0;
        continue;
    }

    /* Handle numbers */
    if ((isdigit(*p) && (prev_sep || row->hl[i-1] == HL_NUMBER)) ||
        (*p == '.' && i >0 && row->hl[i-1] == HL_NUMBER)) {
        row->hl[i] = HL_NUMBER;
        p++; i++;
        prev_sep = 0;
        continue;
    }

    /* Handle keywords and lib calls */
    if (prev_sep) {
        int j;
        for (j = 0; keywords[j]; j++) {
            int klen = strlen(keywords[j]);
            int kw2 = keywords[j][klen-1] == '|';
            if (kw2) klen--;

            if (!memcmp(p,keywords[j],klen) &&
                is_separator(*(p+klen)))
            {
                /* Keyword */
                memset(row->hl+i,kw2 ? HL_KEYWORD2 : HL_KEYWORD1,klen);
                p += klen;
                i += klen;
                break;
            }
        }
        if (keywords[j] != NULL) {
            prev_sep = 0;
            continue; /* We had a keyword match */
        }
    }

    /* Not special chars */
    prev_sep = is_separator(*p);
    p++; i++;
}

Notice the order, because the order is the design. Comments are checked before strings, strings before numbers, numbers before keywords, and anything left over falls through to "not special chars" at the very bottom. That's not arbitrary — it's a priority list for a genuinely ambiguous character stream. A " inside a /* */ comment must not open a string; a digit inside a string literal must not get colored as a number; a keyword-shaped substring inside a comment must not get colored as a keyword. Checking comment state first, then string state, then everything else, is what guarantees each of those. Get the order wrong and "don't" would try to open a second string at the apostrophe.

The order is load-bearing, not stylistic

Every branch that fires ends in a continue (or, for the single-line-comment case, a return) — it never falls through to a later check in the same iteration. That's what makes "comments before strings before numbers before keywords" an actual guarantee instead of a suggestion: once in_comment or in_string is true, the loop's very first two ifs intercept the character before the number or keyword branches ever see it. Reorder those blocks and you change what's allowed to nest inside what.

Comments: one line ends the function, one line waits for its closer

Single-line comments get the simplest treatment in the whole scan, because a // answers every remaining question about the row at once: memset(row->hl+i,HL_COMMENT,row->size-i); return;. One call paints everything from here to the end of the row, and the function returns immediately — it never reaches the multi-line-comment propagation logic at the bottom (Chapter 14's hl_oc machinery) for this row, because a row that ends inside a // comment can't also be ending inside an unterminated /* */ comment; the two are mutually exclusive by construction.

row->size, not row->rsize

Look closely at that memset: row->size-i. Every other branch in this function writes into row->hl using offsets measured against row->rsize (the tab-expanded, on-screen length Chapter 08 introduced) — because hl is allocated to rsize, and the scan pointer p walks row->render. This one line reaches for row->size, the raw chars length, instead. On a row with no tabs the two are equal and nothing goes wrong; on one with tabs before a trailing //, size can undercount how far into hl needs painting. It's a real inconsistency in kilo.c's source, not a subtlety to smooth over — worth noticing precisely because the surrounding code is so consistent about the chars vs. render split everywhere else.

Multi-line comments need more state because they can't be resolved in one memset — the loop has to keep scanning to find the closer, or run off the end of the row still open (and hand that fact to the next row via hl_oc, exactly as Chapter 14 described). Three branches cover it. Already inside one (in_comment true): color the current character HL_MLCOMMENT, then check whether it and the next character spell the closer (mce, "*/" for C) — if so, color both characters, advance past both, and clear in_comment; if not, color just this one character and keep going. Not yet inside one: check whether the current pair spells the opener (mcs, "/*"), and if so, color both opener characters immediately and set in_comment = 1.

One asymmetry is easy to miss on a first read: the single-line-comment check is gated on prev_sep, but opening a multi-line comment is not. x/*comment*/, with no space before the /*, still opens a comment correctly — the *p == mcs[0] && *(p+1) == mcs[1] check runs regardless of what came before. A stray // glued onto an identifier, by contrast, only counts as a comment start if the scan is sitting at a token boundary when it sees it.

Strings: a two-state machine that remembers which quote it opened with

in_string isn't a boolean — it's declared int and holds either 0 (not in a string) or the literal quote character currently open, '"' or '\''. That single design choice is what makes "it's fine" highlight correctly: the apostrophe inside the double-quoted string doesn't close anything, because closing only happens when *p == in_string — and in_string is holding '"', not '\''. A single-quoted string started with ' can only be closed by another '; a double-quoted one, only by another ".

Entering a string colors the opening quote and records which one it was: if (*p == '"' || *p == '\'') { in_string = *p; row->hl[i] = HL_STRING; p++; i++; prev_sep = 0; continue; }. Once inside, every character is colored HL_STRING unconditionally — except a backslash, which is special-cased to consume itself and the character right after it as a pair: if (*p == '\\') { row->hl[i+1] = HL_STRING; p += 2; i += 2; prev_sep = 0; continue; }. That's the entire escape mechanism. It doesn't check whether the escaped character is a real C escape sequence — \n, \", and the nonsense \q are all treated identically, colored and skipped as a pair. The highlighter isn't validating your escapes; it's only making sure a backslash-quote can't be mistaken for the string's actual closer.

Closing checks identity, not "a quote"

if (*p == in_string) in_string = 0; — this line is why mismatched quotes can't accidentally close each other. It isn't asking "is this character a quote mark", it's asking "is this character the specific quote mark that opened the string I'm currently in." The backslash branch above it runs first and continues past this line entirely for an escaped character, so an escaped quote never even reaches this identity check.

Numbers: no lookahead, just "was the last thing also a digit"

The number rule is the shortest branch in the scan and the one with the least state: if ((isdigit(*p) && (prev_sep || row->hl[i-1] == HL_NUMBER)) || (*p == '.' && i >0 && row->hl[i-1] == HL_NUMBER)). Read it as two separate permissions. A digit is allowed to become HL_NUMBER if either it's the first character of a new token (prev_sep) or the character right before it was already classified HL_NUMBER — that second clause is what lets a multi-digit run like 1024 continue past its first digit without prev_sep being true the whole way through (after the first 1, prev_sep gets set to 0 by that branch's own last line). A literal . is allowed to become HL_NUMBER only under the second condition — it can extend a number already in progress (3.14), but it can never start one on its own.

There's no separate state variable for "am I inside a number" the way there is for strings and comments — the previous character's own hl byte is the state. That's a genuinely different technique from the other three branches, and it's cheap exactly because a number, unlike a string or comment, can never span more than the current row and never needs anything remembered past the character immediately behind it.

What this rule can't do

Read literally, the number rule has no concept of a number as a unit, only "does this one character extend the previous one." Two consequences worth naming rather than glossing over: a standalone . (a struct member access like p.x) is never colored as a number, because it only qualifies by following a digit already marked HL_NUMBER — correct here, but incidentally so. And a leading minus sign is never part of the number at all: in -5, the - is just punctuation (it satisfies is_separator), so only the 5 is colored. An exponent like 1e10 breaks the rule outright — 1 is colored, e is not a digit so the run stops, and 10 never resumes because prev_sep is false at that point and hl[i-1] (the e's slot) isn't HL_NUMBER. These aren't bugs to patch here — they're the shape of what "one character of lookbehind, no more" buys and costs.

Keywords: a table scan, a trailing pipe, and a boundary check

The keyword branch is the most machinery of the five, and it's worth reading top to bottom. First, it only even runs when prev_sep is true — the scan has to be sitting at the start of a token before it bothers checking the keyword table at all. That's what stops "reprintf" from being flagged partway through as containing "printf": by the time the scan reaches the p in the middle of the word, prev_sep is already false.

Given prev_sep, it walks keywords[] from the front, computing for each candidate:

int klen = strlen(keywords[j]);
int kw2 = keywords[j][klen-1] == '|';
if (kw2) klen--;

if (!memcmp(p,keywords[j],klen) &&
    is_separator(*(p+klen)))
{
    /* Keyword */
    memset(row->hl+i,kw2 ? HL_KEYWORD2 : HL_KEYWORD1,klen);
    p += klen;
    i += klen;
    break;
}

kw2 is read off the last character of the keyword string, not tracked anywhere else — "int|" has kw2 == true, "return" has kw2 == false. When it's true, klen is immediately shortened by one, so the trailing | is never part of what gets compared against the row's text and never counted into how far p and i advance on a match. It exists purely to be read once, at the top of this loop body, and then forgotten.

The match test itself is two conditions, both required. memcmp(p,keywords[j],klen) alone would let "int" match the first three bytes of "internal" or "interval" — it's a raw byte comparison with no notion of word boundaries. is_separator(*(p+klen)) is what rejects that: it demands that the character sitting immediately after the candidate match is itself a separator (whitespace, punctuation, or the row's terminating '\0') — the exact same is_separator helper Chapter 14 introduced for deciding "is this a token boundary" everywhere else in the file. One function, reused as the boundary check on both sides of a keyword: prev_sep gates where a keyword can start, and this call gates where it's allowed to end.

A keyword is painted in one memset, not a per-character loop

Every other branch in this scan advances one character (or one escaped pair) at a time. A matched keyword is different: memset(row->hl+i,kw2 ? HL_KEYWORD2 : HL_KEYWORD1,klen); paints the whole matched span in one call, then p and i both jump forward by klen in one step. int (green, HL_KEYWORD2) and return (yellow, HL_KEYWORD1) are colored by this exact same line — the only thing that differs between them is which list entry matched and what kw2 evaluated to for it.

On a match, the inner for loop breaks immediately — it doesn't keep scanning the rest of the keyword table once it's found one. That's exactly why the line right after the loop works as a match test at all: if (keywords[j] != NULL) { prev_sep = 0; continue; }. If the loop broke early on a match, j is left pointing at the matched entry — not NULL. If the loop ran all the way off the end of the array without matching anything, j lands on the sentinel itself, and keywords[j] is NULL. It's the same NULL-terminated-array idiom is_separator and the loop's own iteration already lean on, reused here to answer "did we match" without a separate boolean flag.

If nothing above matched: the fallthrough

An ordinary character — one that isn't a comment delimiter, a quote, a digit continuing a number, or the start of a matched keyword — falls all the way to the loop's last two lines: prev_sep = is_separator(*p); p++; i++;. Its hl byte is never touched here, so it keeps whatever editorUpdateSyntax's opening memset(row->hl,HL_NORMAL,row->rsize) already set it to (Chapter 14): HL_NORMAL. This is the branch that colors the vast majority of any real C file — identifiers that aren't keywords, operators, braces, whitespace — and it's also the one line every other branch was careful to continue past, so it only ever runs for genuinely plain characters.

Worked example: one line, five decisions

Trace a single realistic line through the scan to see all five branches cooperate: int x = 10; // set default.

int x = 10; // set default
  • int — prev_sep is true at the start of the row, so the keyword branch runs; "int|" matches with kw2 true, and all three characters are painted HL_KEYWORD2 in one memset.
  • The space — falls through every branch (it's not a comment starter, quote, digit, or keyword start) to the bottom two lines; is_separator(' ') is true, so prev_sep becomes true again.
  • x — prev_sep is true, so the keyword table is searched and nothing matches; it falls to the bottom fallthrough and stays HL_NORMAL. is_separator('x') is false, so prev_sep becomes false.
  • = — both separators, both fall through, both set prev_sep true again by the time the scan reaches 1.
  • 10 — 1 is a digit with prev_sep true, so it's colored HL_NUMBER; 0 is a digit whose predecessor is already HL_NUMBER, so it extends the run.
  • ; — falls through, HL_NORMAL, and being in is_separator's punctuation set, sets prev_sep true.
  • The space, then // — the space falls through as before; at /, prev_sep is true and the next character is also /, matching scs, so the comment branch fires: everything from here to end of row is painted HL_COMMENT in one memset, and the function returns without ever looking at the rest of the characters individually.

Seven decisions, five different branches, zero backtracking — the scan never revisits a character or reconsiders a choice it already made. That's the whole story of why this can be "the entire highlighter" and still run comfortably on every keystroke: it's linear in the length of the row, once, no matter how deeply nested the punctuation looks.

So what did we actually gain?

A working C/C++ highlighter that fits in about a hundred lines, built from four techniques you can name: a fixed priority order between comment, string, number, and keyword checks so ambiguous characters resolve unambiguously; a two-state in_string variable that remembers which quote is open, not just that one is; a number rule with no memory beyond "was the last character also a number," cheap precisely because numbers can't span lines; and a keyword table that's really two tables overlaid by a one-byte sigil, matched with a boundary check that reuses the same is_separator helper the rest of the scanner already depends on. None of it is a lexer in the formal sense — there's no grammar, no token stream handed to anything downstream, just hl bytes filled in as a side effect of one pass. That's also exactly why Chapter 16 can swap in a second language by supplying a new keyword list and comment delimiters without touching a single line of the scan you just read: the scan is generic over which keywords and delimiters, it only assumes C-shaped tokens (identifiers, quoted strings, // and /* */ comments) exist at all.
