Chapter 20 · Part VII — Beyond the Tutorial

Adding Features Carefully

Chapter 19 asked what a refactor of kilo.c would touch and, just as carefully, what it would deliberately leave alone. This chapter asks the harder follow-up question: kilo.c's own README doesn't just tolerate extension, it invites it — "a starting point to write other editors." So let's actually grow it. Not by bolting on everything a modern editor has, but by taking five real, frequently-requested features — line numbers, undo/redo, richer search, clipboard support, richer highlighting — and asking of each one the same question: what does this cost, and where does that cost actually land? Some of these features are cheap because kilo.c already built the seam they'd slot into. Others are expensive in ways that don't show up until you measure them. Knowing which is which, before you write a line of code, is what "carefully" means in this chapter's title.

Motivation: an invitation with a budget attached

It's worth reading the invitation precisely. Kilo's README says the program is meant "as a starting point to write other editors or command line interfaces that are more advanced than the usual REPL style CLI." That's not a hedge — antirez built Kilo assuming people would extend it. But every chapter of this book so far has also shown what makes kilo.c worth extending in the first place: one file, zero library dependencies, a size budget tight enough to fit in a name. A feature that quietly triples memory use per keystroke, or that only works if you also link ncurses, wins the feature and loses the thing that made this program worth studying. So the real design question for each feature below isn't "can I add this?" — it's "can I add this without spending the budget every earlier chapter spent eighteen chapters establishing?"

That's also why this chapter reads differently from Chapters 4 through 18. Those chapters explained code that exists. This one proposes code that doesn't — every snippet below is an illustrative sketch, not a line from antirez's source, and each one is sized to show the shape of a change, not to be a finished patch. Where a feature is cheap, we'll show why. Where a feature is expensive, we'll show the actual cost, measured by someone who built it and paid for it.

Design: four questions before the first line of code

Every feature below gets asked the same four questions, in order, because they're the ones that separate a change that stays small from one that quietly isn't:

QuestionWhy it matters here
Does a paved path already exist?Chapter 16 built HLDB precisely so new languages don't touch the scanner. Not every feature gets a seam this generous.
Does it touch shared state or just add to it?A new field on erow is additive. A change to what E.cx means is not — it ripples through every function that reads it.
What's the cost per keystroke, not per feature?A feature that runs once per file open (Chapter 16's syntax selection) has a completely different budget than one that runs on every character typed.
Is there a smaller version that captures most of the value?"Highlight the current match" and "highlight every match" are both "richer search," at very different costs.

With that framework in hand, here are the five features the brief for this chapter names, taken one at a time.

Line numbers

Chapter 9 established that E.cx and E.cy are screen-relative coordinates, clamped into [0, E.screencols) and [0, E.screenrows) — not positions in the file. That single fact is the whole reason line numbers aren't a one-line change. Every column calculation in editorRefreshScreen — the cx-to-render-column loop, the horizontal-scroll check, the status bar's own width math — measures directly against E.screencols with no margin reserved. Printing a number before each line is the easy five percent. Making every one of those existing calculations agree on how many columns are left over is the rest of the feature.

The width isn't a constant

Vim's own gutter width (numberwidth) is a value the user sets, not one Vim recomputes automatically as the file grows. Neovim's own issue tracker carries an open request for a reliable way to query the current gutter width at all, noting you can "do some log10 math and heuristically compute how big the column probably is" from the line count — but that estimate breaks the moment a minimum width is configured. A from-scratch Kilo gutter hits the identical problem: what happens to every cached column offset the moment E.numrows crosses from 9 lines to 10, or 99 to 100, and the gutter needs one more character of width mid-session?

/* Illustrative sketch — not in antirez's source. */
int gutter_width(int numrows) {
    int digits = 1;
    for (int n = numrows; n >= 10; n /= 10) digits++;
    return digits + 1; /* +1 for the space before the text */
}
/* Every place that currently reads E.screencols for on-screen width
   would instead read E.screencols - gutter_width(E.numrows), recomputed
   wherever updateWindowSize() already runs. */

Notice what this sketch does not do: it doesn't touch E.cx's meaning at all. The cursor is still a screen column; the gutter just shrinks what "the screen" means for text. That's the difference between a feature that's expensive because it's structurally invasive and one that's merely tedious because it has several call sites — line numbers are the second kind.

Undo/redo

This is the feature in this chapter with the clearest paper trail on what happens when you skip the "measure first" step. Text-editor undo splits into two designs: a command-based approach that stores the reverse of each edit — "delete these 5 characters at this position" to undo an insert of those same 5 characters — and a state-based approach that snapshots the whole buffer before each edit and restores the snapshot to undo. State-based is far less code. It is also, it turns out, far more expensive to run.

A measured example, not a guess

Developer Matt Duck added undo/redo to his own fork of kilo using periodic full-buffer snapshots, then profiled it with pmap. Total heap grew by 132K and RSS by 184K after only a handful of edits — to a file that was itself under 29K on disk. Each additional snapshot added roughly 192KB, regardless of how small the actual edit was: a single inserted character and a thousand inserted characters cost the same snapshot. For a program whose entire design budget is "less than 1000 lines" and an implicitly frugal memory footprint, that's not a rounding error — it's the concrete price of choosing the design that was easy to write over the one that was cheap to run.

The command-based alternative costs more up front — every editing function (editorInsertChar, editorDelChar, editorInsertNewline, all touched already by Chapter 10) needs a matching "here's how to undo this specific call" record pushed onto a stack, rather than one generic snapshot call inserted everywhere. But the cost scales with what actually changed, not with the size of the buffer it changed in — which is exactly the property a program this size can't afford to give up by accident.

Richer search

Chapter 13 already gave editorFind a specific, narrow limitation worth naming here: it restores at most one previous match's highlighting per keystroke. The FIND_RESTORE_HL macro undoes exactly the single saved_hl/saved_hl_line pair captured on the previous iteration — not a list of them. "Highlight every match in the file, not just the current one" is a real, common request, and it means replacing that single saved slot with an array: one saved-and-restored entry per match, not one.

That gap isn't specific to Kilo, and neither is the fix. Vim's own incremental search didn't originally highlight every match either — patch 8.0.1238, titled "incremental search only shows one match," added the option to highlight all matches on top of the pre-existing single-current-match behavior, years after incremental search itself had shipped. Kilo is, on this specific point, exactly where Vim once was.

The upgrade isn't free

The GNU Emacs manual documents that highlighting a large number of matches can measurably slow the editor down while the search query is still being typed — and exposes a configurable cap on how many matches get highlighted at once specifically to protect responsiveness during the same keystroke-by-keystroke loop Chapter 13 already runs for a single match. An "improved" search that makes every keystroke feel laggy on a large file is not, in the terms this chapter cares about, actually an improvement.

Clipboard support

Chapter 3 established Kilo's zero-dependency constraint, and Chapter 18 named the portability cost of leaning on anything platform-specific. Both rule out the obvious approach: shelling out to pbcopy, xclip, or wl-copy would work, but it would pick a platform, add a spawned process to a program that has never spawned one, and quietly abandon the "any VT100-compatible terminal" story Chapter 5 built the whole rendering layer around. OSC 52 sidesteps all three problems at once: it's a terminal escape sequence, not a system call — ESC ] 52 ; c ; <base64-text> BEL — asking the terminal emulator itself to set the system clipboard, delivered exactly the way Kilo already talks to the terminal for cursor movement and screen clearing.

/* Illustrative sketch — not in antirez's source. */
void osc52CopyToClipboard(const char *text, size_t len) {
    char *b64 = base64_encode(text, len); /* small helper, not in kilo.c */
    printf("\x1b]52;c;%s\x07", b64);
    fflush(stdout);
    free(b64);
}

It has real limits, and they're worth naming before reaching for it rather than after: one documented implementation caps the base64-encoded payload at 74994 bytes before truncating, so this suits a selection or a line, not a whole large file. Inside tmux, the sequence is only forwarded to the outer terminal if set -s set-clipboard on is configured (some setups also need allow-passthrough). And VTE-based terminals — the base of many default Linux desktop terminal emulators — don't support OSC 52 at all, and some will print the raw escape sequence onto the screen instead of silently ignoring it.

Writing and reading are not the same operation

The sketch above only ever sets the clipboard — it's write-only from the editor's side. OSC 52 also defines a query variant for reading the clipboard, and that's a materially more sensitive operation: many terminals refuse it outright or gate it behind a user prompt, because a program silently reading whatever's on your system clipboard is a real information-disclosure risk, not a convenience. "Add clipboard support" quietly means two different features with two very different trust levels, and this chapter is only recommending the first one.

Richer syntax highlighting

This is the one feature in this chapter that Chapters 14 through 16 already did most of the design work for. HLDB is a plain array holding exactly one populated struct editorSyntax, matched by filename in editorSelectSyntaxHighlight, and the source's own doc comment addresses a future maintainer directly: "In order to add a new syntax, define two arrays..." Adding a second language is appending one more HLDB entry — it never touches the character-by-character scanning logic in editorUpdateSyntax at all. Of every feature in this chapter, this is the cheapest, because the paved path was built three chapters ago.

Where the paved path ends

That same doc comment is equally explicit about what isn't supported: "There is no support to highlight patterns currently" — filematch is plain substring/suffix matching only. A genuinely richer version of this feature — glob filename patterns like *.conf, or per-token regex highlight rules — means changing editorSelectSyntaxHighlight's match loop itself, and possibly editorUpdateSyntax's scanner. That's a structurally different, more invasive change than adding a table row, which is exactly why it's the one item on this list worth scoping down or deferring rather than building first.

Edge cases: where features collide

None of these five features live in isolation once more than one of them exists, and the collisions are worth naming rather than discovering later.

Undo meets highlighting

Chapter 14 established that a change to whether a row ends inside an open multi-line comment recurses forward, rehighlighting every subsequent row until the state stops changing. An undo that reinserts a large deleted block — or a redo that removes one — can trigger exactly that recursive rehighlight, once per undo step. A command-based undo (the design this chapter recommends above) makes that cost visible and attributable to one keystroke; a snapshot-based undo would hide it inside the same expensive restore that already made Matt Duck's numbers look the way they did.

Line numbers and search interact too, in a smaller but still real way: Chapter 13's match-repositioning code sets E.cx and E.coloff directly rather than through editorMoveCursor (Chapter 9). Any of those direct writes that assume "column 0 is the left edge of the text" needs the same gutter-width offset the rest of the drawing code would need — a second call site to update, easy to miss precisely because it doesn't look like a drawing function.

So what did we actually gain?

Not five features bolted on at once — a costed list. Line numbers and syntax-highlighting extension are cheap, for two different reasons: one touches many call sites but nothing structural, the other has a seam built for it since Chapter 16. Undo/redo and richer search are exactly as expensive as you choose to make them — a snapshot or highlight-everything implementation is a few hours of typing and a real, measured memory or responsiveness cost; the disciplined version costs more code and less at runtime. Clipboard support is cheap and safe in one direction (write) and a genuinely different, more sensitive feature in the other (read). Kent Beck's engineering aphorism — make the change easy, then make the easy change — is exactly Chapter 19's own method applied here: the small, verified refactor (threading state as a parameter, extracting a dispatch table) comes first, so that each of these features, when it's actually written, is a small, reviewable diff on top of it rather than a change that's hard to untangle from everything else it touched. Chapter 21 is what makes "verified" mean something more than "I tried it once" for any of this.
