What a Language Model Actually Is

Strip away the mystique and a modern LLM does exactly one thing: given some text, it predicts the next token. Everything else — chat, code, reasoning — is that single trick, run in a loop.

Let's build the forward pass from the outside in, because once you can trace a single token through the machine, nothing later in this book will feel like hand-waving. Start with the input. The model does not see characters or words; it sees tokens — subword chunks produced by a tokenizer. The word "caching" might become two tokens, cach and ing; a common word like "the" is one. Each token maps to an integer ID in a fixed vocabulary (tens of thousands of entries).

HowdoLLMswork?

Text → tokens → integer IDs. This is the model's only alphabet.

Each ID indexes a big lookup table — the embedding matrix, shaped [vocab_size × hidden_size]. Out comes a vector of hidden_size numbers (say 4096 of them) that represents that token. Position information is folded in too — Llama and Qwen use rotary embeddings (RoPE), applied inside attention, so the model knows token order.

🔤
Why subwords

Tokens are a compromise between characters (too many steps) and whole words (too big a vocabulary, and no way to spell new words). Subwords let a fixed vocabulary represent any string — including your product names and error codes. This matters later: a fine-tune inherits the base model's tokenizer.

· 1

The stack: attention + feed-forward, ×N

The token vectors now flow up through a stack of identical transformer layers — 32 of them in an 8B model. Each layer has two parts, each wrapped in a residual connection and a normalization step (RMSNorm):

  • Self-attention — each token looks at every earlier token and pulls in what's relevant. It projects each token vector into three roles — a Query, a Key, and a Value — then computes softmax(Q·Kᵀ / √head_dim)·V. Plain English: "how much should I attend to each previous token, and what do I take from it?" Attention is causal — a token may only look backward, never at the future.
  • Feed-forward (MLP) — a small per-token neural network that transforms what attention gathered. This is where a lot of the model's raw "knowledge" lives.
One transformer layer. Stack N of them; the vector gets richer at every floor.

From the top of the stack to a word

After the last layer, the final token's vector passes through one more norm and the LM head — a matrix [hidden_size × vocab_size] (often the embedding matrix reused, "tied"). This produces one logit per vocabulary token: a raw score for "how likely is this the next token?"

A softmax turns those scores into probabilities. Then a sampling step picks the actual next token — greedily (always the top one), or with a little randomness via temperature, top-k, or top-p (nucleus) sampling. That chosen token is appended to the input, and the whole process runs again for the next one.

python · the generation loop, conceptuallyfor _ in range(max_new_tokens):
    logits = model(tokens)          # [seq_len, vocab] — score every token
    next_logits = logits[-1]        # we only need the LAST position
    probs = softmax(next_logits / temperature)
    next_id = sample(probs)          # greedy / top-k / top-p
    tokens.append(next_id)
    if next_id == eos_id: break   # model said "I'm done"
⚠️
The expensive detail hiding in that loop

Notice we feed the whole tokens list back in every step, but only use the last row of logits. Done naively, that recomputes everything about every earlier token, over and over. That waste is exactly what the KV cache (Chapter 4) eliminates — and understanding why is the key that unlocks serving.

· 3

So what are "the weights"?

Every matrix we just met — the embedding table, the Q/K/V and output projections in each attention block, the MLP matrices, the LM head — is a big grid of numbers learned during training. Collectively, those numbers are the model. "Llama-3.1-8B" means roughly 8 billion of them.

🔑
The mental model to keep

A language model is a fixed pile of weights plus a simple loop: embed the tokens, push them up through N layers of attention + MLP, read off a probability for the next token, sample one, repeat. Custom-training (Ch. 6) nudges those weights toward your data. Serving (Ch. 8) runs the loop fast for many users at once.

Two numbers about those weights will govern the rest of the book, so meet them now:

  • Parameter count8B, 70B, …
  • Bytes per parameter2 (fp16) … 0.5 (int4)

Multiply them and you get the model's memory footprint. An 8B model in 16-bit precision is about 16 GB of weights — before it has processed a single token. Chapter 5 is entirely about shrinking that second number.

Acme check-in

Our Support Copilot will be a model exactly like this. Fine-tuning won't change its shape — same layers, same loop. It only changes the values in some of those weight grids so that "next token" leans toward Acme's phrasing and its tag vocabulary.
