Concept 08 · Serving

Inference Servers

You have a model on disk. Calling model.generate() in a loop technically runs it — and wastes almost all your GPU. An inference server is the machinery that makes serving economical.

Remember from Chapter 3 that decode is memory-bound, so a single request leaves the GPU's math units idle. An inference server is a long-running process that loads the weights once and does the hard scheduling so many users share that expensive weight-load. It handles four things a naive loop doesn't:

  • Request scheduling across many concurrent clients.
  • Batching — running many sequences per forward pass to amortize memory traffic.
  • KV-cache management — allocating and freeing that big, dynamic cache.
  • An HTTP API, almost always OpenAI-compatible, so existing client code just works.
🔑
The one you'll reach for: vLLM

vLLM is the leading open-source server, and it's fast because of two ideas: PagedAttention (how it stores the KV cache) and continuous batching (how it schedules work). Learn these two and the rest is configuration.

· 1

PagedAttention: virtual memory for the KV cache

The old way pre-allocated one contiguous KV buffer per request, sized for the longest possible output — so most of it sat empty. Wasteful: prior systems threw away 60–80% of KV memory to fragmentation.

vLLM (Kwon et al., SOSP 2023) borrows the operating system's trick. It cuts the KV cache into fixed-size blocks ("pages") that need not be physically contiguous, and keeps a per-sequence block table mapping token positions to scattered physical blocks — exactly like OS page tables. Fragmentation waste drops to under ~4%, and identical blocks (a shared system prompt!) can be shared across sequences via copy-on-write.

The cache is paged, not pre-booked — so almost no VRAM is wasted.

Continuous batching: never wait for the slowpoke

Static batching forms a batch, runs it until the longest sequence finishes, then starts the next — so finished requests idle, waiting on their neighbours. Continuous batching (a.k.a. iteration-level, or NVIDIA's "in-flight" batching) schedules at token granularity: after every decode step, finished sequences leave and free their blocks, and newly arrived requests join the running batch for the very next step. The GPU stays saturated. This is the single biggest throughput win over a naive loop.

bash · serve your model, OpenAI-compatible$ vllm serve acme-copilot-merged \
     --gpu-memory-utilization 0.9 \
     --max-model-len 8192
# listens on :8000 — PagedAttention is always on (no flag)
⚠️
There is no --enable-paged-attention

PagedAttention is always active in vLLM — don't go looking for a switch. The knob you do turn is --gpu-memory-utilization (default 0.9), which sets how much VRAM becomes the KV-cache pool.

🔌
Serving the adapter without merging

Kept your LoRA adapter instead of merging? Serve it live: vllm serve BASE --enable-lora --lora-modules acme=/path/to/adapter. The adapter's name becomes the model field in requests, and one base can host several adapters at once.

The rest of the field

vLLM / TGI / SGLang

Production GPU serving. TGI is HuggingFace's Rust+Python server (continuous batching, tensor parallelism). SGLang adds RadixAttention — a radix tree that auto-reuses shared prompt prefixes across requests.

Ollama / llama.cpp

Local & dev. llama.cpp is C/C++ inference on GGUF, CPU + optional GPU offload, runs on modest hardware. Ollama wraps it for one-command local runs (ollama run) and its own REST API.

Acme's choice

Acme serves the merged Qwen with vLLM on its own GPU box — private, OpenAI-compatible, high throughput. And remember Chapter 3: support tickets share a lot of boilerplate, so SGLang's prefix reuse is worth a benchmark if that boilerplate is large.
