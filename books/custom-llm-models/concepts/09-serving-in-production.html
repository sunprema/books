Serving in Production

A server that runs is not a server that's tuned. The last mile is spending your GPU deliberately — trading latency for throughput on purpose, and knowing what to do when the model won't fit on one card.

Every production decision comes back to the budget from Chapter 5: weights + KV cache + overhead. Weights are fixed once you've chosen precision. Overhead is small. So the game is the KV-cache pool — and that pool is governed by two settings and one trade-off.

The trade-off: throughput vs. latency

Bigger batches amortize the weight-load across more users, so total tokens/sec climbs. But a bigger batch can also mean any single request waits a touch longer for its turn in each step. That's the whole tension:

  • LatencyTTFT + per-token, one user
  • Throughputtokens/sec, everyone

Continuous batching + PagedAttention are what let modern servers push throughput hard without wrecking latency — together they run roughly 2–4× the throughput of the 2023 baselines the vLLM paper measured, and far more than a naive PyTorch loop.

🔑
Pick your target first

Interactive chat wants low TTFT (tune for latency). A nightly batch job wants maximum tokens/sec (tune for throughput). You can't max both — decide which one the product needs, then turn the knobs toward it.

The two knobs that set concurrency

FlagControls
--max-model-lencontext length (prompt + output) per request — the seq_len in the KV formula
--max-num-seqshow many sequences run in one iteration — the batch
--gpu-memory-utilizationfraction of VRAM given to the KV-cache pool (default 0.9)

These aren't independent — they multiply. The KV formula says cache ∝ seq_len × batch. Give every request 32K of context and you can hold far fewer of them at once than at 8K. Right-size --max-model-len to what your workload actually needs; padding it to a huge number silently slashes concurrency.

⚠️
The classic self-own

Setting --max-model-len 128000 "just in case" reserves cache for 128K-token conversations you never have — and you'll wonder why only two requests fit. Match context length to reality.

🗜️
More seats, same card

Two moves buy concurrency: quantize the weights (Ch. 5) to shrink the first budget term and free VRAM for cache, and quantize the KV cache itself (fp8) to halve dtype_bytes in the formula. Both trade a little quality for a lot of throughput.

When the model won't fit on one GPU

A 70B model in 16-bit is ~140 GB — more than any single card. Two ways to split it across GPUs:

Tensor parallelism

Shard each layer's weight matrices across GPUs; every layer needs cross-GPU communication. Heavy chatter → best within one node over NVLink. --tensor-parallel-size N.

Pipeline parallelism

Split the model by layers into stages on different GPUs. Less communication, but "pipeline bubbles" → good across nodes. --pipeline-parallel-size N.

They compose: tensor-parallel within a node, pipeline-parallel across nodes. Start with tensor parallelism on a single multi-GPU box — it's simpler and usually enough.

🔑
Cheapest fix first

Before you reach for multiple GPUs, ask whether quantization lets the model fit on one. A 70B in 4-bit is ~35 GB — one 48 GB card. Splitting across GPUs adds communication cost and complexity; often a smaller footprint is the better trade.

Calling your endpoint

Because the server is OpenAI-compatible, your application code is boring — which is the point. The official OpenAI client, pointed at localhost:

python · call the Acme Copilotfrom openai import OpenAI

client = OpenAI(base_url="http://localhost:8000/v1",
                api_key="EMPTY")  # ignored unless --api-key set

resp = client.chat.completions.create(
    model="acme-copilot-merged",
    messages=[
      {"role": "system", "content": "You are Acme's support assistant."},
      {"role": "user",   "content": "My invoice looks wrong."},
    ],
)
print(resp.choices[0].message.content)
📊
Watch the right numbers

In production, track TTFT and per-token latency at your peak concurrency, not on an idle box — that's where the KV-cache pool gets tight and requests start queuing. If TTFT spikes under load, you're cache-bound: shrink context, quantize, or add a GPU.
