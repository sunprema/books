Concept 03 · Inference

Inference: Prefill & Decode

Running a model to answer a prompt happens in two phases with opposite personalities. Confuse them and every performance decision you make will be wrong.

When you send a prompt, the model does not start generating immediately in one uniform process. It splits the work in two. First it reads your entire prompt; then it writes the answer one token at a time. These are called prefill and decode, and the whole art of fast serving comes from understanding that they stress completely different parts of the GPU.

Prefill: reading the prompt, all at once

In prefill, the model processes every prompt token in parallel in a single forward pass, and stashes the Key/Value tensors for each (that's the KV cache being born — Chapter 4). Because all the tokens go through together, the GPU does big matrix-by-matrix multiplies where each loaded weight is reused across many tokens.

⚙️
Prefill is compute-bound

High arithmetic intensity — lots of FLOPs per byte read from memory — so prefill saturates the GPU's math units. This is the phase that sets TTFT (time to first token). Longer prompt ⇒ more prefill work ⇒ higher TTFT.

· 1

Decode: writing the answer, one token at a time

Once the prompt is digested, the model enters decode: generate a token, append it, generate the next, and so on. Each step is a forward pass over a single new token (it reuses the cached K/V of everything before it). One token in, one token out, hundreds of times.

🚚
Decode is memory-bound

To produce one token, the GPU must stream all the model's weights and the whole KV cache out of memory — then does comparatively tiny math with them. Low arithmetic intensity: the compute units sit half-idle, waiting on memory bandwidth. This phase sets ITL / TPOT (inter-token latency / time per output token).

Prefill fills the cache in parallel; decode drains memory one token per step.

Why "memory-bound" is the fact that runs your GPU bill

Here is the intuition worth internalizing. During decode with a single request, producing one token multiplies your input against every weight matrix — reading gigabytes of weights to do a modest amount of arithmetic. The weights get reused only as many times as your batch size. Batch of one? Each weight is read, used once, discarded. The GPU's math units starve while the memory bus works flat out.

The fix writes itself: process many requests together. If 32 users' tokens ride through the same weight-load, each weight earns its keep 32× over. Total tokens-per-second climbs dramatically while any single user barely notices. That is the entire economic argument for an inference server — and the reason Chapters 8–9 exist.

🔑
Throughput vs. latency, in one breath

Latency is how fast one answer arrives (TTFT + per-token time). Throughput is how many tokens/sec you serve across everyone. Batching raises throughput, sometimes at the cost of a little latency. Nearly every serving knob is a dial between these two.

The three numbers people actually measure

MetricSet byMeans
TTFTprefillTime to first token — how long until the answer starts. Grows with prompt length.
ITL / TPOTdecodeTime per output token — how fast it streams once started.
ThroughputbatchingTotal tokens/sec across all requests. The utilization number.

NVIDIA defines TPOT as the average time between consecutive output tokens.

🧩
Modern trick: chunked prefill

Because prefill is compute-bound and decode is memory-bound, servers like vLLM can interleave them — slicing a long prefill into chunks and running them alongside decode steps of other requests, so the compute units and the memory bus stay busy at the same time. You don't configure this by hand; it's why the numbers are good out of the box.

🧪
Acme check-in

A support ticket thread is a long prompt (all the back-and-forth) and a short answer (a draft reply + a tag). So Acme's workload is prefill-heavy — TTFT matters, and shared boilerplate across tickets is an opportunity (hold that thought for prefix caching in Ch. 8).
