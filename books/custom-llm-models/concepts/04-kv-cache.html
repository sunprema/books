Concept 04 · The Big Idea

The KV Cache

This is the single most important implementation detail in LLM serving. Understand it and you can predict memory, batch size, and cost. You asked to grasp it — here it is, in full.

Recall the generation loop from Chapter 2: we kept feeding the whole token list back in and using only the last row of output. Let's count the waste. To generate token number n, attention needs the Key and Value vectors of all n−1 tokens before it. Without a cache, we'd recompute every one of them, at every step — that's O(n²) work across the sequence for something we already calculated.

But here's the saving grace: because attention is causal, a past token's Key and Value never change. Token 5's K and V are the same whether we're generating token 6 or token 600. So compute them once, store them, and reuse them. That store is the KV cache.

🔑
What the cache does

It trades compute for memory. Each decode step now computes Q/K/V for only the one new token and attends against the cached K/V of all the old ones — O(n) per step instead of O(n²). Generation goes from quadratic to linear. The price: you must hold every token's K and V in GPU memory, for every layer, for the whole sequence.

The cache is a growing filing cabinet of K/V — one drawer per layer, one folder per token.
· 1

The one formula in this book

How much memory does the cache eat? Exactly this:

kv_bytes = 2 · layers · kv_heads · head_dim · dtype_bytes · seq_len · batch
2one K and one V per position
layerseach transformer layer keeps its own cache
kv_headsnumber of key/value heads (fewer than query heads, thanks to GQA — next page)
head_dimhidden_size ÷ attention_heads
dtype_bytes2 for fp16/bf16, 1 for fp8/int8
seq_lentokens cached so far (prompt + generated)
batchconcurrent sequences
📌
Read the formula like a budget

Two terms are fixed by the model (layers, kv_heads, head_dim). Two you choose at serve time (seq_len = context length, batch = concurrency). One you can shrink with tricks (dtype_bytes, via a quantized KV cache). Serving is the art of spending the leftover VRAM — after weights — on seq_len × batch.

Work it out: Llama-3-8B, by hand

Let's make it real with a model whose config we can look up. Llama-3-8B has 32 layers, hidden size 4096, 32 attention (query) heads, but only 8 KV heads (that's GQA), so head_dim = 4096 / 32 = 128. In fp16, dtype_bytes = 2.

Per token, across all layers, for one sequence:

2 · 32 · 8 · 128 · 2 = 131,072 bytes = 128 KiB / token

Multiply by sequence length:

  • at 4,096 tokens≈ 512 MiB
  • at 8,192 tokens≈ 1 GiB

That's per sequence. Serve 16 concurrent 8K conversations and the cache alone wants 16 GiB — as much as the 16 GB of weights. Now you see why "how many users can I serve?" is really "how much KV cache fits?"

⚠️
If it used full attention…

With 32 KV heads instead of 8, that per-token figure would be 512 KiB and the 8K cache would be 4 GiB per sequence — 4× worse. GQA is doing a lot of quiet work here.

· 3

GQA: the reason the cache isn't ruinous

Standard multi-head attention (MHA) gives every query head its own Key/Value head — maximal cache. The opposite extreme, multi-query attention (MQA), makes all query heads share a single K/V head — tiny cache, but it can hurt quality.

Grouped-query attention (GQA) splits the difference: query heads are divided into groups, and each group shares one K/V head. Llama-3-8B has 32 query heads sharing 8 K/V heads — a group size of 4, so a 4× smaller cache. The GQA paper reports it lands "close to multi-head quality with speed comparable to multi-query."

cache reduction = attention_heads ÷ kv_heads
Llama-3-8B: 32 ÷ 8 = 4×  ·  Llama-3-70B: 64 ÷ 8 = 8×
🔑
The modern default

Llama 3, Qwen2.5, Mistral, Gemma, DeepSeek — all use GQA. When you fine-tune one of these, you inherit its (already efficient) attention structure. You don't design this; you just benefit from it, and now you can predict its memory.

🧪
Acme check-in

Say Acme picks Qwen-8B and budgets 8K context per ticket. Weights ≈ 16 GB (fp16); a 24 GB GPU leaves ~8 GB for cache ≈ 8 concurrent tickets. Want more concurrency on the same card? Two levers: shrink the weights (quantize — Chapter 5) and let the server pack the cache tightly (PagedAttention — Chapter 8).
