Chapter 02 · Paths & Traps

Safer Ways to Open a File

The plain open(2) call has three quieter cousins that exist purely because "open a file by name" is a surprisingly dangerous operation once an attacker — or just an unlucky race — gets to choose what that name points to.

Between the moment you resolve a path and the moment you act on it, the world can change. Someone can swap a symlink under you, another process can be mid-write to the file you're about to read, or the path you meant to check and the path you actually open can turn out to be two different files. This whole family of bugs has a name — TOCTOU, "time of check to time of use" — and three flags on open(2) exist specifically to close the window.

O_PATH — a handle with no read or write power

O_PATH opens a file descriptor that refers to a location in the filesystem without actually opening the file for I/O. You can't read() or write() through it. What you can do is pass it as the dirfd argument to the *at() family — fstatat(), fchownat(), linkat(), readlinkat(), openat() — or reopen it properly through /proc/self/fd/N.

c · resolve once, act many times, safelyint pfd = open("/var/lib/app/data", O_PATH | O_NOFOLLOW);
// pfd now names *this exact inode*, resolved once -- no re-lookup, no race

struct stat st;
fstatat(pfd, "", &st, AT_EMPTY_PATH);      // stat the pfd itself
openat(pfd, "config.json", O_RDONLY);       // relative to that exact directory

Because the *at() calls resolve their relative path starting from the already-pinned pfd, nobody can trick you by renaming or replacing a directory somewhere earlier in the path — you already committed to this inode. If the final component turns out to be a symlink and you passed O_NOFOLLOW, you get an fd referring to the symlink itself, which you can then safely inspect with readlinkat(pfd, "", ...) without ever following it.

Real-world use Container runtimes and systemd use O_PATH extensively to walk untrusted or attacker-influenced paths (bind-mount targets, container root filesystems) one safely-pinned component at a time, refusing to ever silently follow a symlink out of the intended tree.

O_TMPFILE — a file with no name, until you decide it needs one

Normally, creating a file means the moment creat()/open(O_CREAT) returns, a directory entry exists that anyone can open, read, and see mid-write. O_TMPFILE creates a completely unnamed inode — visible in the filesystem only as the caller's fd — that never gets a name unless you explicitly give it one:

c · write the whole file, then publish it atomicallyint fd = open("/var/lib/app", O_TMPFILE | O_WRONLY, 0600);
write(fd, buf, len);                       // nobody else can see this file yet
fsync(fd);

char path[64];
snprintf(path, sizeof path, "/proc/self/fd/%d", fd);
linkat(AT_FDCWD, path, AT_FDCWD, "/var/lib/app/data.json", AT_SYMLINK_FOLLOW);
// only NOW does the fully-written file appear, atomically, with its final name

Readers can never observe a partially-written file, because the file has no name to open until you're done — and the final linkat() is a single atomic filesystem operation. Add O_EXCL to O_TMPFILE and you get the opposite guarantee: a file that can never be linked into the directory tree at all, a true anonymous scratch file that vanishes the instant every fd referencing it is closed — no cleanup code, no crash-leaves-a-tempfile-behind bugs.

A small delight An O_TMPFILE + O_EXCL file is, for all practical purposes, a Schrödinger's file: it fully exists (you can read and write gigabytes to it, it occupies real disk blocks), yet as far as anyone with a plain path is concerned, it was never born and never will be. Close the last fd and it's gone — no obituary required.
Real-world use This "write it all, then link it into place" pattern is the standard, kernel-assisted way to do an atomic file replace — used by systemd's state files, package manager downloads, and any database or config writer that wants to guarantee a reader never sees a half-written file, without the old trick of writing to file.tmp and rename()-ing it (which still leaves a named, visible temp file for a window).

O_NOFOLLOW / AT_SYMLINK_NOFOLLOW — refuse to be redirected

If the last component of the path you're opening is a symbolic link, O_NOFOLLOW makes open() fail with ELOOP instead of quietly following it. That sounds small, but it closes a whole class of classic attacks: a world-writable directory like /tmp lets any local user plant a symlink named exactly what your program is about to create or open — pointing it at /etc/passwd, someone else's home directory, or a device file. Without O_NOFOLLOW, your privileged process happily follows the trap.

c · a lock file that can't be redirectedint fd = open("/tmp/myapp.lock",
               O_CREAT | O_EXCL | O_NOFOLLOW | O_WRONLY, 0600);
if (fd == -1 && errno == ELOOP) {
  // someone planted a symlink where our lock file should be -- bail out
}

AT_SYMLINK_NOFOLLOW is the same idea for the *at() family — pass it to fstatat(), fchownat(), or unlinkat() when you specifically want to operate on the symlink itself, not whatever it points to.

Real-world use Mail delivery agents, cron, sudo, and anything that creates predictably-named files in shared directories use O_NOFOLLOW (often together with O_EXCL) as baseline hardening against symlink-planting attacks from other local users.

O_PATH turns "resolve, then trust the path is still the same thing" into "resolve once, reuse the fd."
