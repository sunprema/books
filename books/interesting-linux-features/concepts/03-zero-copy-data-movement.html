Chapter 03 · Moving Bytes Without Moving Them

Zero-Copy Data Movement

The ordinary way to move data from a file to a socket — read() into a buffer, then write() it back out — copies every byte through your process's memory twice, for no reason except that the API makes you ask for the bytes by name. Four syscalls exist purely to skip that.

A "copy" in the kernel is really a reference to a page of memory. If all you want is for data to travel from a file's page-cache page into a socket buffer, the kernel can just hand out another pointer to the same physical page and bump a reference count — no CPU cycles spent shuffling bytes, no trip through your userspace buffer at all. sendfile, splice, vmsplice, and tee are four different doors into that trick.

sendfile — file straight to socket

The simplest case: you want to serve a static file over a socket, and your process never needs to look at the bytes. sendfile(2) says so directly:

c · the whole web server, one calloff_t offset = 0;
struct stat st; fstat(file_fd, &st);
sendfile(socket_fd, file_fd, &offset, st.st_size);
// bytes travel disk-cache -> socket buffer, entirely inside the kernel

Without it, serving a file means: read() copies disk-cache bytes into your buffer, then write() copies your buffer into the socket's kernel buffer — two copies and two user/kernel context-switch round trips for data your program never actually touches. sendfile collapses that to a single call and, on hardware that supports scatter-gather DMA, the network card can read straight out of the page-cache page itself.

Real-world use This is the mechanism behind nginx's and Apache's sendfile directive for serving static assets — the reason a well-tuned static file server can saturate a network link while barely registering on the CPU.

splice — the general case, via a pipe

sendfile only handles file→socket. splice(2) generalizes the same zero-copy trick to any two file descriptors — as long as at least one of them is a pipe. The kernel implements a pipe internally as a ring of reference-counted page pointers, and splice just moves pointers in and out of that ring instead of copying the bytes they point to:

c · socket to socket, via an anonymous pipe, no copiesint pipefd[2]; pipe(pipefd);
splice(in_socket,  NULL, pipefd[1], NULL, 65536, SPLICE_F_MOVE);
splice(pipefd[0], NULL, out_socket, NULL, 65536, SPLICE_F_MOVE);
// data never enters this process's address space at all

This is exactly how a proxy or load balancer can shovel gigabytes between an inbound and an outbound socket without the proxying process's own memory ever holding a copy of the traffic. SPLICE_F_NONBLOCK makes the pipe operations non-blocking; SPLICE_F_MORE is a hint (like TCP_CORK) that more data is coming right behind this chunk, so the kernel can hold off flushing a partial network packet.

Real-world use HAProxy and other high-throughput L4 proxies use splice-based socket-to-socket forwarding specifically to avoid copying every proxied byte through userspace, which is the difference between a proxy that scales to tens of gigabits and one that becomes the bottleneck.

vmsplice — your own memory, gifted into a pipe

vmsplice(2) goes the other direction: it maps pages of your program's own memory directly into a pipe's buffer, without copying them there first.

c · hand a big buffer to a pipe without copying itstruct iovec iov = { .iov_base = big_buffer, .iov_len = big_buffer_len };
vmsplice(pipefd[1], &iov, 1, 0);
// the pipe now references your pages directly -- don't modify big_buffer
// until you know the reader has consumed it, or use SPLICE_F_GIFT carefully

This is sharp: the pipe now holds live references to your process's actual memory pages, so mutating that buffer before the reader has drained the pipe can corrupt what the reader sees. Used carefully, though, it's the fastest way to get a large in-memory buffer into a pipe (and from there, into splice's zero-copy world) without ever calling write().

tee — duplicate a stream without consuming it

tee(2) copies data from one pipe into another without removing it from the source — the pipe equivalent of a plumbing T-joint. The original data is still there for the next reader to consume normally.

c · fan a stream out to a second consumer, for freetee(source_pipe, monitor_pipe, 65536, 0);
// source_pipe is untouched; monitor_pipe now has a copy of the same pages
splice(source_pipe, NULL, real_destination, NULL, 65536, 0);
Real-world use Stream-fan-out tools that need to forward traffic and mirror it — to a logging pipeline, an intrusion-detection sensor, or a debugging capture — use tee to duplicate a pipe's contents without paying for a second full copy of the data.
A small delight A pipe, in this world, isn't really "a queue of bytes." It's a small ring of pointers to pages somewhere else in memory — which is why tee can "copy" 64 KB of data in microseconds. It's not copying the data. It's copying a handful of pointers to the data, and letting reference counting sort out the rest.

Same page, three destinies: copied twice, moved by reference, or referenced from two places at once.

Sources: splice(2), vmsplice(2), tee(2), sendfile(2) — man7.org Linux man-pages.
