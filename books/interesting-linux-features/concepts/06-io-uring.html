Chapter 06 · Async I/O Without the Syscalls

io_uring

Every traditional syscall — even a fast one — costs a user/kernel context switch. io_uring, introduced by Jens Axboe in Linux 5.1 (2019), asks a genuinely radical question: what if a program could ask the kernel to do I/O without making a syscall for each request?

The answer is two ring buffers, shared between your process and the kernel via a single mmap(). You write a description of an operation — "read this fd," "accept on this socket," "write this buffer" — into a slot on the submission ring. The kernel, working through that ring at its own pace, writes the result into a slot on the completion ring. Your program never has to ask "is it done yet?" through a syscall — it just looks at shared memory.

io_uring_queue_init(256, &ring, 0);

struct io_uring_sqe *sqe = io_uring_get_sqe(&ring);
io_uring_prep_read(sqe, fd, buf, len, offset);
io_uring_submit(&ring);              // io_uring_enter(2) under the hood

struct io_uring_cqe *cqe;
io_uring_wait_cqe(&ring, &cqe);      // blocks until a result appears
ssize_t n = cqe->res;
io_uring_cqe_seen(&ring, cqe);      // mark this slot free again

The raw kernel interface is just two syscalls — io_uring_setup(2) to create the rings, and io_uring_enter(2) to tell the kernel "go look at the submission ring, and optionally wait for at least N completions." A single io_uring_enter() call can submit a whole batch of queued operations and wait for earlier ones to finish, which is already a huge win over one syscall per operation. Most programs use liburing, the reference userspace library, instead of touching the raw ring layout directly.

SQPOLL: the mode with zero syscalls on the hot path

Batching submissions is good. IORING_SETUP_SQPOLL is better: it spins up a dedicated kernel thread that continuously polls the submission ring for new entries. Once that thread is running, your program can add work to the SQ ring by writing directly into shared memory — no syscall at all — and the kernel thread picks it up and executes it.

Why this is a big deal A syscall is typically tens to low hundreds of nanoseconds of pure overhead before any actual work happens — and that overhead multiplies across millions of small I/O operations per second on a busy server. Batched submission plus SQPOLL doesn't just make individual operations faster; it removes the per-operation syscall tax almost entirely from the hottest code path a storage engine has.
Real-world use PostgreSQL has adopted io_uring for asynchronous I/O in its storage layer; high-performance proxies, storage engines, and language runtimes (including parts of the networking stacks in Rust's tokio-uring and Node.js's libuv backends) use it to cut syscall overhead on I/O-bound hot paths. It has also become a significant enough attack surface that some hardened environments (notably some container and mobile platforms) restrict or disable it by default.
A small delight With SQPOLL running, you can genuinely watch your process issue reads and writes to real files and sockets while strace shows almost nothing happening in the traced process itself — the actual work is happening in a kernel thread that was never part of your syscall trace to begin with. It looks, for a moment, like I/O is happening by magic.

Two shared-memory rings replace a syscall-per-operation conversation with a much cheaper "leave a note, check back later."
