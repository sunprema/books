Chapter 04 · A File That Was Never On Disk

Anonymous Memory Files

memfd_create(2) gives you a file descriptor for a file that exists purely in RAM, has no path, no directory entry, and no presence on any disk — yet behaves like a completely ordinary file for every other purpose: you can mmap() it, write() to it, resize it with ftruncate(), and — the interesting part — hand the fd to another, unrelated process.

Before memfd_create, sharing memory between two processes meant either shm_open() (which needs a name in /dev/shm, visible to anyone who can guess it) or old System V shmget() (with its own global, easy-to-collide integer key namespace). memfd_create sidesteps both: it makes a nameless, nowhere-visible chunk of shared memory, and the only way another process gets access is if you deliberately pass it the file descriptor — typically over a Unix domain socket with SCM_RIGHTS.

c · a RAM-only file, shareable only by handing out the fdint fd = memfd_create("shared-buffer", MFD_CLOEXEC | MFD_ALLOW_SEALING);
ftruncate(fd, 4096);
void *p = mmap(NULL, 4096, PROT_READ|PROT_WRITE, MAP_SHARED, fd, 0);
memcpy(p, header, sizeof header);
// send `fd` to another process via a Unix socket + SCM_RIGHTS ...
// ... it mmap()s the *same* physical pages. No copy, no shared filesystem needed.

File sealing: making a shared memfd trustworthy

Handing another process write access to your memory is normally a leap of faith — what stops it from resizing the buffer out from under you, or overwriting data you're still relying on? fcntl(2)'s sealing operations let the creator of a memfd lock down what can be done to it, permanently, before sharing it:

SealPrevents
F_SEAL_SHRINKthe file being made smaller (ftruncate down)
F_SEAL_GROWthe file being made larger
F_SEAL_WRITEany further writes, including through an existing mmap
F_SEAL_SEALadding any more seals — the lock on the lock
c · freeze it, then it's safe to hand to anyonefcntl(fd, F_ADD_SEALS, F_SEAL_SHRINK | F_SEAL_GROW | F_SEAL_WRITE | F_SEAL_SEAL);
// from this point on, this exact fd's contents cannot change, by anyone, ever

Without MFD_ALLOW_SEALING at creation time, a memfd's initial seal set is already F_SEAL_SEAL — sealing is opt-in, so you have to ask for the ability to seal before you can use it.

The whole trick, in one line A memfd plus a full set of seals is a way to say, cryptographically-flavored but entirely in-kernel: "here is a blob of memory, and I promise — enforced by the kernel, not by politeness — that neither I nor anyone I hand this fd to can ever change it again."
Real-world use Wayland compositors and GPU buffer-sharing protocols (like Linux's DMA-BUF ecosystem) use sealed memfds to hand graphics buffers to a compositor without letting a misbehaving client resize or rewrite a buffer the compositor is actively reading from mid-frame. systemd uses memfd to pass structured data (like unit file contents or journal export data) between its processes without touching disk. And container tooling uses memfd to build a binary entirely in memory and fexecve() it — a process that runs from a file that was never written to any filesystem at all.
A small delight Because a memfd is a real fd, you can point /proc/self/fd/N at it and pass that path string to tools that insist on a filename — effectively lying to a program that "this is a file on disk" when it's actually a sealed, nameless chunk of your RAM. It's the file-that-never-existed, wearing a file-shaped costume.

Sealing turns "please don't change this" into a kernel-enforced guarantee, so sharing memory across a trust boundary stops being a leap of faith.

Sources: memfd_create(2), fcntl(2) (file sealing) — man7.org Linux man-pages.
