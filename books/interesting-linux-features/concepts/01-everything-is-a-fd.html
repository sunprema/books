Chapter 01 · The Idea

Everything Is a File Descriptor

Unix's oldest trick — "everything is a file" — turns out to have a second act. The kernel has spent the last twenty years quietly turning things that are not files at all — a counter, a timer, a stream of signals, a whole other process — into file descriptors too, just so you can hand them to epoll and stop thinking about them as special cases.

A file descriptor is really just a small integer that means "ask the kernel about this later." Historically that "this" was a file, a socket, or a pipe. But once you have a single, universal epoll_wait() that can watch a pile of descriptors and tell you which ones are ready, an obvious question appears: why should timers, signals, and events be handled through entirely different, older mechanisms — alarm(), signal handlers, condition variables — when they could just be fds you read a number from?

Why this matters Once a timer, a signal, or "process X has exited" is a readable file descriptor, it can sit in the exact same epoll_wait() loop as your sockets. One thread, one loop, no separate polling mechanisms, no signal-handler footguns. This is the single idea that makes modern C event loops (Redis, Node's libuv, systemd, nginx) as small and fast as they are.

eventfd — a counter you can wait on

eventfd(2) hands back a file descriptor that wraps a single unsigned 64-bit counter, maintained entirely by the kernel:

c · a minimal wakeup fd#include <sys/eventfd.h>

int efd = eventfd(0, EFD_NONBLOCK | EFD_CLOEXEC);
// another thread, or a signal handler-safe context:
uint64_t one = 1;
write(efd, &one, sizeof(one));  // wake the loop

// the event loop:
uint64_t count;
read(efd, &count, sizeof(count)); // blocks until non-zero, then resets to 0

Every write() adds its 8-byte value to the counter; every read() returns the current count and resets it to zero — unless you pass EFD_SEMAPHORE at creation, in which case each read only decrements the counter by one, turning the fd into an actual kernel-backed semaphore. It's the cheapest possible "something happened, go check" signal between threads or processes, and it's entirely epoll-friendly.

Real-world use KVM uses eventfd-backed ioeventfd/irqfd so a guest VM's virtio device writes can notify (or interrupt) the host without a full, expensive VM-exit round trip through QEMU's userspace emulation — the kernel just bumps the counter directly.

timerfd — a timer you can epoll

Before timerfd_create(2), a timer meant alarm() or setitimer() delivering SIGALRM — async-signal-unsafe and impossible to multiplex cleanly. timerfd makes a timer just another fd:

c · a periodic timer, no signals involvedint tfd = timerfd_create(CLOCK_MONOTONIC, 0);
struct itimerspec spec = {
  .it_interval = {1, 0},  // repeat every 1s
  .it_value    = {1, 0},  // first fire in 1s
};
timerfd_settime(tfd, 0, &spec, NULL);

uint64_t expirations;
read(tfd, &expirations, sizeof(expirations)); // how many ticks fired since last read

Reading returns an 8-byte count of how many times the timer has fired since you last read — so a busy loop that fell behind still gets an honest answer instead of a burst of missed signals. TFD_TIMER_ABSTIME lets you arm it against an absolute wall-clock deadline instead of a relative delay, which matters if you don't want drift to accumulate across many short waits.

Real-world use systemd and most modern C/C++ event-loop libraries (libevent, libuv, Boost.Asio on Linux) implement their "sleep until" and periodic-tick primitives on top of timerfd specifically so a timeout can live in the same epoll set as every socket.

signalfd — signals without a signal handler

Signal handlers run at an arbitrary point, on an arbitrary stack, and can only safely call a tiny list of "async-signal-safe" functions. signalfd(2) sidesteps all of that: block the signals you care about, then read them out of a normal fd whenever it's convenient.

c · SIGTERM/SIGINT as data, not an interruptsigset_t mask;
sigemptyset(&mask);
sigaddset(&mask, SIGTERM);
sigaddset(&mask, SIGINT);
sigprocmask(SIG_BLOCK, &mask, NULL); // stop default delivery

int sfd = signalfd(-1, &mask, SFD_NONBLOCK);

struct signalfd_siginfo si;
read(sfd, &si, sizeof(si));
// si.ssi_signo, si.ssi_pid, si.ssi_status ... ordinary struct fields

The signal simply stops arriving asynchronously and instead queues up as a readable struct signalfd_siginfo record on the fd — which means your graceful-shutdown handling on SIGTERM can be plain, ordinary, non-reentrant code that runs in your main loop's turn, not in a signal handler fighting your own data structures.

A small delight Blocking a signal and then reading it back out through a file descriptor means you can literally cat-style pipe your process's own SIGTERM through the same code path you'd use to read a socket. The kernel doesn't care that a signal isn't "really" a stream of bytes — it'll happily pretend.

pidfd — a process, held safely by the hand

The classic way to signal a process, kill(pid, sig), has a nasty race: PIDs get reused. If the process you meant to signal has already exited and the kernel handed its PID to a brand-new, unrelated process, your kill() happily signals the wrong thing. pidfd_open(2) fixes this by giving you a file descriptor that refers to that specific process, not to a PID number that might get recycled:

c · race-free process signalingint pidfd = pidfd_open(child_pid, 0);
// later, maybe much later:
if (pidfd_send_signal(pidfd, SIGTERM, NULL, 0) == -1
    && errno == ESRCH) {
  // the process is really gone -- not "some other process now owns this PID"
}

Because a pidfd is a normal fd, you can also poll() or epoll_wait() on it: it becomes readable the instant the process exits, turning "wait for my child (or any process I have a handle to) to die" into just another epoll event, with zero races and zero SIGCHLD juggling. clone3(2)'s CLONE_PIDFD flag goes one step further and hands you the pidfd atomically at creation time, closing even the tiny window between "process is born" and "I open a handle to it."

Real-world use Container runtimes and init systems (systemd, runc) use pidfds to track child processes without PID-reuse races, and checkpoint/restore tools use pidfd_getfd(2) to duplicate a file descriptor out of a different process by pidfd — handy for live-migrating a running container's open connections.

The picture

Four completely different kernel subsystems, one uniform interface: a readable fd.

Sources: eventfd(2), timerfd_create(2), signalfd(2), pidfd_open(2), pidfd_send_signal(2) — man7.org Linux man-pages.
