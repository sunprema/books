Chapter 05 · A Trap With Excellent Documentation

Epoll's Sharp Edges

epoll(7) is the workhorse behind nearly every high-performance Linux network daemon, and its default mode is friendly and forgiving. Its optional mode is faster, sharper, and will silently stall your server the first time you get it slightly wrong.

Every registered file descriptor in an epoll instance can be watched one of two ways, and the choice changes what "ready" means so fundamentally that code correct under one mode will hang forever under the other.

Level-triggered: "still ready" (the default)

In level-triggered mode — what you get if you don't pass EPOLLET — epoll_wait() reports a file descriptor as ready every single time you call it, for as long as the underlying condition holds. If a socket has 4 KB of unread data and you only read() 1 KB of it, the next epoll_wait() will immediately tell you it's still readable. This is exactly how poll(2) and select(2) already behave — epoll's level-triggered mode is simply a much faster version of the same mental model. You can read a little, do other work, come back, and epoll will keep reminding you there's more.

Edge-triggered: "just changed" (EPOLLET)

Edge-triggered mode notifies you exactly once, at the moment a file descriptor's state transitions — empty buffer becomes non-empty, a socket becomes writable again. It does not keep reminding you. If you read only part of the available data and stop, the rest sits in the kernel buffer, epoll stays silent about it, and unless something else nudges the socket again, that data may never get processed.

The trap The man page's rule is absolute: with EPOLLET, always use non-blocking file descriptors, and always drain a ready fd in a loop — read() or write() repeatedly until you get EAGAIN — before waiting on the next epoll_wait(). Stop one read early and you leave data stranded with no notification ever coming to tell you it's there. 
c · correct edge-triggered drain loopev.events = EPOLLIN | EPOLLET;   // edge-triggered, from man7's own example
epoll_ctl(epfd, EPOLL_CTL_ADD, conn_fd, &ev);

// on notification:
for (;;) {
  ssize_t n = read(conn_fd, buf, sizeof buf);
  if (n > 0)  { handle(buf, n); continue; }
  if (n == -1 && errno == EAGAIN) break;  // truly drained
  if (n == 0) { close_connection(); break; }
}

Why bother with the sharp edge at all? Fewer wasted wakeups. A busy, mostly-idle-between-bursts server spends less time being told "still readable" over and over when it already knows. Under real load, edge-triggered epoll is measurably cheaper — which is why nginx and other high-throughput servers default to it, accepting the drain-loop discipline in exchange.

EPOLLONESHOT: one event, then silence until re-armed

Both modes have a shared problem in a multi-threaded server: if two worker threads share one epoll instance, the same ready fd can be handed to two threads simultaneously — a race where both try to read the same connection at once. EPOLLONESHOT fixes this by disabling further notifications for a file descriptor the instant one event is delivered for it. No other thread will be told about it again until you explicitly re-arm it:

c · hand each ready fd to exactly one workerev.events = EPOLLIN | EPOLLONESHOT;
epoll_ctl(epfd, EPOLL_CTL_ADD, conn_fd, &ev);

// a worker thread handles the event, then re-arms explicitly when done:
ev.events = EPOLLIN | EPOLLONESHOT;
epoll_ctl(epfd, EPOLL_CTL_MOD, conn_fd, &ev);

Only one thread ever "owns" a given connection at a time, and ownership is handed back explicitly — a clean, race-free way to fan an epoll instance's events out across a thread pool.

Real-world use Thread-pool-based servers (many production HTTP and RPC servers) use EPOLLONESHOT exactly for this reason — without it, sharing one epoll fd across worker threads invites double-processing races on the same connection.
A small delight "Edge-triggered" is a genuinely good pun disguised as jargon: it means the notification fires only at the edge of a state change, like a doorbell that only rings the instant someone arrives — and stays completely silent no matter how long they continue standing on your porch afterward.

The same three-event stream, two very different notification counts — and one of them (edge-triggered, undrained) is a live bug.

Sources: epoll(7), epoll_ctl(2) — man7.org Linux man-pages.
