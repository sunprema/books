Chapter 09 · The Kernel Asks You What To Do

userfaultfd

A page fault is normally a conversation your program isn't invited to. You touch an address, the CPU traps, the kernel quietly finds a page — from the file, from swap, from the zero page — maps it in, and resumes you. You never knew it happened. userfaultfd inverts that: it lets a process register a range of its own memory and say when someone touches this and there's nothing there, don't guess — stop the thread and ask me.

What actually happens on a fault

When a thread touches an unmapped-but-valid address, the hardware raises a fault and the kernel takes over. It decides what the page should contain: read it from a file, pull it from swap, hand out a shared zero page, or allocate a fresh anonymous one. The faulting thread is parked for the duration and then resumed as if nothing happened. The policy — where does this page come from? — has always been the kernel's to decide.

userfaultfd hands that decision to a userspace manager. You get a file descriptor. The kernel writes fault notifications to it. The faulting thread stays blocked until you supply the page's contents. The kernel has stopped guessing, because you told it you know better — and, importantly, you might know something the kernel physically cannot: that the page's contents are still on another machine.

The key idea Every other chapter in this book asks the kernel to do something. This one is the kernel asking you. The fd runs backwards: you read requests from it, not data.

Getting the descriptor

There is no glibc wrapper — you call it through syscall(2) directly. It has been in the kernel since Linux 4.3.

c · create the userfault descriptor#define _GNU_SOURCE
#include <fcntl.h>
#include <linux/userfaultfd.h>
#include <sys/ioctl.h>
#include <sys/syscall.h>
#include <unistd.h>

int uffd = syscall(SYS_userfaultfd,
                   O_CLOEXEC | O_NONBLOCK | UFFD_USER_MODE_ONLY);
/* UFFD_USER_MODE_ONLY (Linux 5.11+) is what lets this work unprivileged.
   More on why in a moment. */

The two-step handshake

Before anything else you perform a version handshake with UFFDIO_API. You declare the API version you were built against; the kernel replies with the feature bits and ioctls it actually supports. This is the same defensive pattern as Landlock's ABI probe two chapters from now — announce, don't assume.

c · handshake, then register a rangestruct uffdio_api api = { .api = UFFD_API, .features = 0 };
if (ioctl(uffd, UFFDIO_API, &api) == -1)
    die("UFFDIO_API");
/* api.features and api.ioctls now describe what this kernel can do. */

struct uffdio_register reg = {
    .range = { .start = (unsigned long)addr, .len = len },
    .mode  = UFFDIO_REGISTER_MODE_MISSING,
};
if (ioctl(uffd, UFFDIO_REGISTER, &reg) == -1)
    die("UFFDIO_REGISTER");

Three things you can be notified about

Registration modeFires when…Since
UFFDIO_REGISTER_MODE_MISSINGa page in the range is touched and nothing is mapped there yet4.10
UFFDIO_REGISTER_MODE_WPa write hits a write-protected page in the range5.7
UFFDIO_REGISTER_MODE_MINORa minor fault occurs — the page is in the page cache but not mapped into this process5.13

MODE_MISSING is the one to picture first: it is “this memory is empty and someone wants it.” MODE_WP gives you write-tracking — the foundation for incremental snapshots and concurrent garbage collectors. Write-protection support reached shared memory and hugetlbfs in Linux 5.19.

The handler loop

The descriptor becomes readable when there's a fault to service. You poll it, read a struct uffd_msg, and answer with UFFDIO_COPY (here is the data) or UFFDIO_ZEROPAGE (it's all zeroes). The blocked thread resumes the instant you do.

c · the manager loop — answering a faultstruct pollfd pfd = { .fd = uffd, .events = POLLIN };
poll(&pfd, 1, -1);

struct uffd_msg msg;
read(uffd, &msg, sizeof(msg));

if (msg.event == UFFD_EVENT_PAGEFAULT) {
    unsigned long page = msg.arg.pagefault.address & ~(page_size - 1);

    /* Fetch it from wherever the truth lives — a socket, a
       remote host, a decompressor — into page_buffer, then: */
    struct uffdio_copy copy = {
        .dst  = page,
        .src  = (unsigned long)page_buffer,
        .len  = page_size,
        .mode = 0,
    };
    ioctl(uffd, UFFDIO_COPY, &copy);  /* faulting thread wakes up here */
}

Why anyone would want this

Real-world use QEMU/KVM post-copy live migration is the use case userfaultfd was built for — the initial 4.3 implementation was designed around it. Migrating a VM to another host, you can copy all its RAM first and then switch over (pre-copy), but a busy guest keeps dirtying pages faster than you ship them. Post-copy flips it: switch the guest over to the destination immediately, with almost no memory, and register its entire address space with userfaultfd. The guest runs; every page it actually touches faults; the manager pulls that page over the network on demand. The VM is live in milliseconds and pages itself in over the next few seconds.
Real-world use CRIU (Checkpoint/Restore In Userspace) uses it for lazy restore. Rather than reading a whole checkpoint image back into memory before letting the process run, CRIU registers the lazy regions with userfaultfd and starts the process at once, faulting pages in from the image only as they're needed. The 4.3 interface wasn't enough for this; the “non-cooperative” mode added in Linux 4.11 is what made it workable.

The security story — and a sysctl you'll meet

Here is the uncomfortable part. A userfaultfd manager can hold a faulting thread stopped, indefinitely, at a moment of its choosing. That is a remarkable gift to someone writing a kernel exploit: many kernel bugs are races with windows measured in nanoseconds, and stalling a fault in the middle of a syscall that's copying to or from user memory stretches that window open for as long as the attacker likes. A hard race becomes a reliable one.

So since Linux 5.11 the kernel splits the feature in two, gated by /proc/sys/vm/unprivileged_userfaultfd:

ValueMeaning
0 (default)restrict unprivileged users to handling page faults in user mode only — callers without CAP_SYS_PTRACE must pass UFFD_USER_MODE_ONLY for the call to succeed
1allow unprivileged users to use the userfaultfd syscalls without any restrictions

That's what UFFD_USER_MODE_ONLY in the first snippet was doing. With it set, your manager only ever sees faults raised by user-space code touching the range. If the kernel faults on that memory — a copy_to_user deep inside some syscall — the kernel handles it normally and never asks you. You keep the legitimate use case; you lose the exploit primitive.

If you get EPERM Since Linux 5.2, userfaultfd() fails with EPERM when the caller lacks CAP_SYS_PTRACE in the initial user namespace and the sysctl is 0. The fix is almost never to flip the sysctl to 1 — it's to add UFFD_USER_MODE_ONLY to your flags. Post-copy migration needs the unrestricted form, which is why libvirt had to grow explicit support for granting it to QEMU.
A small delight Virtual memory's founding promise was the lie that your process has a big flat array of bytes all to itself, and the kernel works tirelessly to keep the lie convincing. userfaultfd lets you take over telling the lie. The memory can be on another continent. Your program will never notice — it just does a load, and waits a little longer than usual.
