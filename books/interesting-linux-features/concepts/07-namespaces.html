Chapter 07 · A Different Universe, Same Kernel

Namespaces: Your Own Private Linux

Every "container" you've ever run is a completely ordinary Linux process. What makes it feel like a separate machine isn't virtualization — it's that the kernel can give a process its own private view of shared global state: its own set of visible PIDs, its own filesystem tree, its own network stack. That view is a namespace.

There are several kinds of namespace (network, user, IPC, UTS, cgroup, time), but two of them do most of the work of making a process feel like it's alone on its own machine: the PID namespace, which controls which processes you can even see, and the mount namespace, which controls which filesystem tree you see at all.

PID namespaces: your own process tree

Create a new PID namespace — via clone(CLONE_NEWPID, ...) or unshare(CLONE_NEWPID) followed by a fork — and the very first process born inside it is assigned PID 1, exactly like the real init on a freshly booted machine. That process can only see, signal, and interact with other processes inside the same namespace (and its descendant namespaces); everything outside is invisible to it, as if it simply doesn't exist.

shell · a process that thinks it's PID 1# unshare --pid --fork --mount-proc bash
# ps aux
USER  PID  ...  COMMAND
root    1  ...  bash
root    9  ...  ps aux
# from the host's point of view, this bash has some large, ordinary host PID

PID namespaces nest: each one (other than the very first, the "root" namespace the whole system boots into) has a parent, and a process actually holds one PID per namespace it's a member of — the same task might be PID 4931 as seen from the host and PID 1 as seen from inside its own container, simultaneously true, depending on who's asking.

The PID 1 catch If the namespace's PID 1 dies, the kernel treats it like a real init crashing: every other process left in that namespace is sent SIGKILL. This is why container images so often need a tiny, deliberate init process (tini, dumb-init) instead of running the application directly as PID 1 — a naive app usually doesn't reap zombie children or forward signals the way a real init would.

Mount namespaces: your own filesystem tree

CLONE_NEWNS gives a process its own private copy of the system's mount table. Mounting or unmounting something afterward, inside that namespace, is invisible to every other namespace — and combined with bind mounts (mount --bind, which grafts one part of the filesystem tree onto another path, no new filesystem required), a process can be given a completely custom view of "the filesystem" built entirely out of pieces borrowed from the real one.

shell · a container root, built from bind mounts# mount --bind /var/lib/containers/myapp /mnt/newroot
# mount --bind /var/lib/containers/myapp/etc-resolv /mnt/newroot/etc/resolv.conf
# each bind mount grafts one tree onto another location -- no chroot needed yet

This is more flexible — and much more secure — than the old chroot(2), which only ever changed where / points and famously does nothing to stop a privileged process from escaping. A mount namespace is a first-class kernel object with its own lifecycle, and it's what lets a container runtime assemble an entire root filesystem for a container out of read-only image layers, writable overlays, and selectively bind-mounted host directories — without touching the real host filesystem at all.

Why /proc needs its own mount too A /proc filesystem reflects whichever PID namespace mounted it, not whichever PID namespace is currently looking at it. Which is why creating a new PID namespace almost always goes hand-in-hand with creating a new mount namespace and mounting a fresh /proc inside it — otherwise ps run inside your shiny new "container" would show every process on the whole host.
Real-world use This is, quite literally, what Docker, Podman, and Kubernetes' containerd/runc are built from: a bundle of namespaces (PID, mount, network, UTS, IPC, and usually user) created together via clone()/unshare(), plus cgroups for resource limits. "Container" is a userspace word; the kernel only knows namespaces and cgroups.
A small delight A process inside a fresh PID namespace looking at ps aux and seeing only itself as PID 1 is not being lied to by some virtualization layer — it is getting the kernel's honest, complete answer to "what processes exist," because as far as that namespace's bookkeeping goes, it really is the only process that exists. The isolation isn't a mask over the truth; it is the truth, from where it's standing.

Two "PID 1"s, two entirely different process trees, one kernel keeping honest, separate books for each.

Sources: pid_namespaces(7), mount_namespaces(7), namespaces(7) — man7.org Linux man-pages.
