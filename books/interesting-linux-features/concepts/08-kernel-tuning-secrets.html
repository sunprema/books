Chapter 08 · The Defaults Were Guesses Too

Kernel Tuning Secrets

A handful of knobs under /proc/sys/vm/ and /sys/kernel/mm/ quietly decide how aggressively your kernel swaps, how much it trusts filesystem caches over application memory, and whether it gambles on huge pages at exactly the wrong moment. Their defaults are reasonable for a generic desktop — and frequently wrong for a database, a low-latency service, or anything under real memory pressure.

vm.swappiness — how eager the kernel is to swap

This sysctl (default 60, on a 0–100 scale, though modern kernels accept up to 200) tunes how aggressively the kernel prefers to swap out anonymous memory (your program's heap and stack) versus reclaiming page cache (file-backed pages it can re-read from disk cheaply later). A higher value swaps more readily; a lower value holds onto application memory longer and shrinks the page cache first instead.

shell · check and set it$ cat /proc/sys/vm/swappiness
60
$ sudo sysctl vm.swappiness=10
Real-world use Database and low-latency workloads commonly run with vm.swappiness set to 1 or 10 — they'd rather the OS evict some filesystem cache than pause a hot query while a page of the working set gets swapped back in from disk.

vm.vfs_cache_pressure — how eager the kernel is to forget your files

Every open directory entry (dentry) and inode the kernel has looked at gets cached, so it doesn't have to re-walk the filesystem to find them again. vm.vfs_cache_pressure (default 100) controls how hard the kernel reclaims that cache relative to ordinary page cache. Push it below 100 and the kernel prefers to keep directory/inode metadata cached, sacrificing regular page cache instead; push it toward 0 and it will almost never reclaim metadata — which can, under sustained pressure, contribute to running the system out of memory.

Real-world use Servers with enormous numbers of small files (mail spools, package mirrors, some database layouts) often lower vfs_cache_pressure to keep directory lookups fast, at the cost of a slightly smaller regular file-content cache.

Transparent Huge Pages — a performance feature that sometimes isn't

Modern CPUs can map memory in large 2 MB pages instead of the usual 4 KB, shrinking the number of translation-lookaside-buffer entries needed for a given amount of memory — fewer TLB misses, less page-table walking. Transparent Huge Pages (THP) let the kernel do this automatically, without an application asking for huge pages explicitly, controlled by /sys/kernel/mm/transparent_hugepage/enabled:

ModeBehavior
alwaysthe kernel tries to back every eligible mapping with huge pages
madviseonly for memory a process explicitly requests via madvise(MADV_HUGEPAGE)
neverTHP disabled entirely

The catch: assembling a huge page can require the kernel to synchronously compact memory first — moving pages around to free up a contiguous 2 MB block — and that compaction pause can show up as a sudden, unpredictable latency spike right in the middle of an otherwise fast request.

The gotcha databases keep rediscovering Redis and MongoDB both ship explicit warnings and startup checks against always-mode THP, because that exact compaction stall has caused real, reproducible latency spikes and fork-related slowdowns in production. madvise is the generally-recommended balanced default; latency-critical services frequently disable THP (never) outright.

PSI — the kernel finally admits when it's struggling

Pressure Stall Information, exposed since Linux 4.20 under /proc/pressure/{cpu,memory,io}, answers a question top and free never could: not "how much of a resource is used," but "how much time are tasks actually losing because they're waiting for it."

shell · reading memory pressure directly$ cat /proc/pressure/memory
some avg10=2.35 avg60=1.04 avg300=0.42 total=91923841
full avg10=0.89 avg60=0.31 avg300=0.09 total=30129472

some means at least one task was stalled on this resource for that share of time; full means every non-idle task was stalled simultaneously — a much stronger signal that the whole system, not just one process, is resource-starved. The avg10/avg60/avg300 fields are rolling percentages over the last 10, 60, and 300 seconds.

Real-world use Meta's oomd and similar systems watch PSI thresholds to preemptively kill or throttle lower-priority workloads before the classic Linux OOM killer has to step in — turning "the whole machine just froze for ten seconds" into a clean, early, targeted response instead.
A small delight For decades, "is the system under memory pressure" meant staring at a percentage of RAM used and guessing — a metric that says nothing about whether anyone is actually suffering because of it. PSI is the kernel finally answering the actual question you were asking: not "how full is the room," but "how many people are stuck waiting at the door."

Three rolling windows on the same stall metric — the shape of the trend often matters more than any single number.
