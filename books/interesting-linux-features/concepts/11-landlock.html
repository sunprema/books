Chapter 11 · Locking Your Own Doors

Landlock

Sandboxing on Linux has traditionally required someone else's cooperation. SELinux and AppArmor need an administrator to write and load a policy. Namespaces and chroot want privileges you'd rather not have. seccomp filters syscalls, not paths — it can stop you calling open, but it has no idea what you're opening. Landlock closes that gap: an ordinary unprivileged process can tell the kernel “from this line onward, I may only read /usr and write /tmp/work” — and the kernel will hold it to that, forever.

The idea: voluntary, irreversible, inherited

Landlock is a stackable LSM that, in the kernel's own words, “enables any processes to securely restrict themselves and their future children.” It landed in Linux 5.13. Three properties make it work:

  • No privilege needed. You're giving power up, not taking it. Nothing dangerous about letting anyone do that, so anyone can.
  • Irreversible. “Once a thread is landlocked, there is no way to remove its security policy; only adding more restrictions is allowed.” A one-way ratchet. Compromised code cannot un-sandbox itself, because there is no syscall that would do it.
  • Inherited. The policy “will be enforced on all its subsequently created children as well” — every thread from clone(2) and every process from fork/exec inherits the domain. You cannot escape by spawning a helper.
The key idea Sandboxing is usually something done to a program by an administrator who must guess what it needs. Landlock lets the program — the one thing that actually knows what files it touches — do it to itself, at the exact moment it stops needing anything more.

Always ask the ABI first

Landlock is still growing, and each kernel version knows a different set of access rights. Request a right the running kernel has never heard of and ruleset creation fails outright. So the first thing every real Landlock program does is ask what it's talking to:

c · probe the ABI before anything elseint abi = landlock_create_ruleset(NULL, 0, LANDLOCK_CREATE_RULESET_VERSION);
/* With this flag, attr must be NULL and size must be 0.
   abi < 0  -> no Landlock here (old kernel, or not enabled).
   abi >= 1 -> the highest ABI version this kernel supports. */
if (abi < 0) {
    /* Decide deliberately: degrade quietly, or refuse to run.
       Do not pretend you are sandboxed when you are not. */
}

The rights you can hand back

A ruleset declares which access rights it handles — the complete set it intends to police. Anything handled is denied by default; you then grant it back on specific directory subtrees. Anything not handled is left entirely alone, which is what makes the ABI table above matter: on an older kernel you must drop the rights it doesn't know.

Sandboxing yourself, end to end

As with userfaultfd, glibc provides no wrappers — you write three thin syscall shims:

c · the three syscalls#define _GNU_SOURCE
#include <linux/landlock.h>
#include <sys/prctl.h>
#include <sys/syscall.h>
#include <fcntl.h>
#include <unistd.h>

static int landlock_create_ruleset(const struct landlock_ruleset_attr *attr,
                                   size_t size, __u32 flags) {
    return syscall(__NR_landlock_create_ruleset, attr, size, flags);
}
static int landlock_add_rule(int fd, enum landlock_rule_type type,
                             const void *attr, __u32 flags) {
    return syscall(__NR_landlock_add_rule, fd, type, attr, flags);
}
static int landlock_restrict_self(int fd, __u32 flags) {
    return syscall(__NR_landlock_restrict_self, fd, flags);
}
c · read /usr, write /tmp/work, nothing else — ever againstruct landlock_ruleset_attr attr = {
    .handled_access_fs =
        LANDLOCK_ACCESS_FS_EXECUTE    | LANDLOCK_ACCESS_FS_READ_FILE  |
        LANDLOCK_ACCESS_FS_WRITE_FILE | LANDLOCK_ACCESS_FS_READ_DIR   |
        LANDLOCK_ACCESS_FS_REMOVE_FILE| LANDLOCK_ACCESS_FS_MAKE_REG,
};
int rs = landlock_create_ruleset(&attr, sizeof(attr), 0);

struct landlock_path_beneath_attr pb;

/* Grant: read-only under /usr. Note O_PATH — a descriptor that names a
   place without opening it for I/O. Chapter 02's trick, put to work. */
pb.allowed_access = LANDLOCK_ACCESS_FS_READ_FILE | LANDLOCK_ACCESS_FS_READ_DIR |
                    LANDLOCK_ACCESS_FS_EXECUTE;
pb.parent_fd = open("/usr", O_PATH | O_CLOEXEC);
landlock_add_rule(rs, LANDLOCK_RULE_PATH_BENEATH, &pb, 0);
close(pb.parent_fd);

/* Grant: read-write under /tmp/work. */
pb.allowed_access = LANDLOCK_ACCESS_FS_READ_FILE | LANDLOCK_ACCESS_FS_WRITE_FILE |
                    LANDLOCK_ACCESS_FS_READ_DIR  | LANDLOCK_ACCESS_FS_MAKE_REG |
                    LANDLOCK_ACCESS_FS_REMOVE_FILE;
pb.parent_fd = open("/tmp/work", O_PATH | O_CLOEXEC);
landlock_add_rule(rs, LANDLOCK_RULE_PATH_BENEATH, &pb, 0);
close(pb.parent_fd);

/* Required — see below. */
prctl(PR_SET_NO_NEW_PRIVS, 1, 0, 0, 0);

/* The point of no return. */
landlock_restrict_self(rs, 0);
close(rs);

/* From here: /etc/shadow does not exist as far as this thread is
   concerned. Neither does $HOME. And there is no way back. */
no_new_privs is not optional landlock_restrict_self() refuses to run without PR_SET_NO_NEW_PRIVS first, and the reason is worth understanding. Without it, sandboxed code could exec a set-user-ID binary — sudo, passwd — and gain privileges inside the sandbox, using a trusted program as a crowbar. no_new_privs makes that structurally impossible: nothing this thread execs can ever be more privileged than the thread itself. Only then will the kernel accept the ratchet.

Who's actually using it

Real-world use pacman 7 sandboxes package downloads, so a compromised download can't wander the filesystem with the privileges the package manager runs under. GNOME's localsearch sandboxes its file indexer — a process that parses untrusted documents all day is exactly what you want fenced in. Nix and OpenBSD's rpki-client use it too; rpki-client is a neat case, since it wanted the hardening its home OS gets from unveil(2) and Landlock is the closest Linux equivalent.
Real-world use Suricata makes the argument for self-sandboxing better than any manual could: an intrusion-detection system knows precisely which directories it reads rules from and writes logs to, so it can enforce a tight deny-by-default policy on itself — something no external administrator could have written as accurately.

Tooling is arriving too. Island is a general-purpose Landlock sandboxing wrapper, and systemd has an in-progress LandlockConfig= service setting, which would let you sandbox a unit declaratively without touching its source at all.

A small delight Eleven chapters ago the kernel handed us a file descriptor for a timer, because a number you can poll is more useful than a special case. Here the pattern completes: you build a security policy, get a file descriptor for it, and hand it back. Even the ability to give up your own power arrives as an integer you can close. Everything really is a file.
