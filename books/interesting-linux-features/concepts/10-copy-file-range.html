Chapter 10 · The Best Copy Is No Copy

copy_file_range

Copying a file is the most ordinary thing a program does, and almost every program does it the same naive way: read a chunk into a buffer, write the buffer back out, repeat. The data climbs out of the kernel into your address space and then climbs straight back in, having been changed in no way whatsoever. copy_file_range asks the obvious question — why did it need to leave?

The dance we all learned first

c · the copy loop you have written a dozen timeschar buf[65536];
ssize_t n;
while ((n = read(fd_in, buf, sizeof buf)) > 0)
    write(fd_out, buf, n);
/* every byte: page cache -> user buffer -> page cache. Two crossings,
   two copies, one syscall pair per chunk, all to change nothing. */

For a 4 GB file at 64 KB a chunk that's over 65,000 syscall round-trips and 8 GB of memory traffic to move 4 GB of data. Chapter 03's splice and sendfile already showed the kernel is willing to move bytes without handing them to you. copy_file_range is that idea aimed squarely at the file-to-file case — and it goes considerably further, because when both ends are files the kernel can sometimes skip moving the bytes at all.

The call

c · signature#define _GNU_SOURCE
#define _FILE_OFFSET_BITS 64
#include <unistd.h>

ssize_t copy_file_range(int fd_in,  off_t *_Nullable off_in,
                        int fd_out, off_t *_Nullable off_out,
                        size_t size, unsigned int flags);

The offset arguments carry a small piece of cleverness. Pass NULL and the call behaves like read/write: it starts at the descriptor's current file offset and advances it. Pass a pointer and the descriptor's own offset is left untouched — the kernel reads the starting position out of your variable and updates that instead. So you can copy from the middle of a file that another part of your program is independently reading, without the two fighting over one shared cursor. flags is reserved and must be 0.

c · the whole copyssize_t ret;
do {
    ret = copy_file_range(fd_in, NULL, fd_out, NULL, len, 0);
    if (ret == -1) { perror("copy_file_range"); break; }
    len -= ret;
} while (len > 0 && ret > 0);
Short copies are normal, not an error The return value is the number of bytes copied and “could be less than the size originally requested.” It is not a promise to finish. Loop until you're done — a single call that returns less than size has not failed. A return of 0 means the source offset is at or past end-of-file.

What the kernel does with the freedom

Once the kernel owns the whole copy rather than 65,000 disconnected read/write pairs, it can hand the job to the filesystem, which may know a shortcut. The man page puts it plainly: the call “gives filesystems an opportunity to implement copy acceleration techniques, such as the use of reflinks … or server-side-copy (in the case of NFS).”

Where the files liveWhat can actually happen
Btrfs, XFS with reflink supportA reflink. Both inodes are pointed at the same copy-on-write extents. No data blocks are read, written, or allocated. A 4 GB copy finishes in about the time it takes to update metadata.
Both files on one NFS serverServer-side copy. The client asks the server to do the copy; the bytes never traverse the network at all. If the server's own filesystem does reflinks, the server does the trick above and the total work approaches zero.
An ordinary filesystem with no shortcutThe kernel copies it in-kernel — still no crossings into your address space, and still one syscall instead of thousands.
Real-world use You have already used this without knowing. GNU cp calls copy_file_range, which is why copying a large file on Btrfs can return instantly — --reflink=auto has been the default since coreutils 9. Older versions reached for a Btrfs-specific ioctl; the syscall generalised that trick to every filesystem willing to implement it.

A version history worth knowing

This is a syscall whose behaviour genuinely changed under people's feet, so the kernel version matters more than usual:

VersionWhat changed
Linux 4.5the syscall first appears
glibc 2.27wrapper added, with a user-space fallback for older kernels
glibc 2.30the fallback is removed — the call now genuinely fails on a kernel without it
Linux 5.3major rework; cross-filesystem copies are attempted rather than refused
Linux 5.19cross-filesystem copies properly supported when both filesystems are the same type and implement it; EOPNOTSUPP introduced
The gap you must handle Before Linux 5.3, both files had to be on the same filesystem or you got EXDEV. That's an honest error and easy to handle. The genuinely nasty window is 5.3 through 5.18: cross-filesystem copies were attempted, and on some virtual filesystems the call failed to copy while still reporting success. A copy that silently produces the wrong bytes and returns a cheerful positive number is the worst possible failure mode. Always keep a read/write fallback and take it on EXDEV, EOPNOTSUPP, or EINVAL — this is exactly why coreutils carries one.

The other sharp edges

  • EINVAL — fd_in and fd_out refer to the same file and the ranges overlap. (Non-overlapping ranges in one file are fine.)
  • EBADF — the descriptors aren't open for reading/writing respectively, or fd_out was opened with O_APPEND. Append mode and explicit destination offsets are irreconcilable.
  • EISDIR — one of them is a directory. This moves file contents, not directory entries.
  • Sparse files may get denser. If fd_in is sparse, the call “may expand any holes existing in the requested range” — a 1 GB sparse file that occupies 4 KB on disk can land as a genuine 1 GB of zeroes. Observed on ext4.
A small delight There is a certain honesty to a syscall whose best-case behaviour is to do nothing at all. Ask it to copy four gigabytes on Btrfs and it updates a little metadata, points the second file at the first one's blocks, and returns. The copy is real — write to either file and they part ways instantly. It's just that until someone actually changes something, the cheapest way to have two identical files is to have one.
