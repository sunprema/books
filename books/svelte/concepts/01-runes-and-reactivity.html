Chapter 01 · The Model

Runes & Reactivity

Svelte 5 swapped compiler magic (export let, $:) for runes — explicit, portable reactivity primitives you can spot on sight, grep for, and import into plain .svelte.js files. Get this chapter's model right and the rest of the book is just vocabulary.

counter.svelte 
let count = $state(0);

<button onclick={() => count++}>
  clicked {count} times
</button>

$state isn't a function you import — it's a compiler hint. The Svelte compiler sees it at the top level of a .svelte or .svelte.js file and rewrites count into something the runtime can track. Read it anywhere in a template or an effect and you've created a dependency; write to it and everything depending on it updates. No setState, no subscription boilerplate — it looks and mutates like an ordinary variable.

Deep reactivity, for free

Wrap a plain object or array and $state returns a reactive proxy. Mutating it — todos.push(x), todos[0].done = true — triggers exactly the UI that reads that path, no more, no less. You don't need to reassign to trigger an update, and you don't need immutable-update gymnastics.

let todos = $state([{ text: 'ship it', done: false }]);

function complete(i) {
  todos[i].done = true; // mutate — this is fine, and fast
}

Proxies stop at class instances

The proxy walk is recursive through plain objects and arrays, but it does not proxy class instances or built-ins like Map/Date nested inside state. Put $state on the class's own fields instead (below), or reach for SvelteMap/SvelteDate from svelte/reactivity.

State inside classes

Runes work on class fields too — the compiler turns them into tracked getter/setters. This is how you build a reactive "model" object that isn't tied to any one component.

class Counter {
  count = $state(0);
  increment = () => { this.count++; }; // arrow field — keeps `this` bound
}

Pass counter.increment as a plain method reference (not an arrow field) to an event handler and you'll lose this — the classic JS foot-gun, unchanged by runes.

The reactivity graph

Three rune families cover the whole model, and they compose into one graph: source state flows into derivations and effects, which flow into the DOM.

A write to state ripples right — through every derivation and effect that reads it — to the DOM. Nothing polls; nothing re-runs that didn't depend on what changed.

If you've seen Svelte 4 export let count = 0 and $: doubled = count * 2 still parse in a legacy-mode component, but they're a different, implicit model — reactivity inferred from assignment syntax. Runes make the same intent explicit: let { count = 0 } = $props() and let doubled = $derived(count * 2). This book teaches runes throughout; we only name the old model where the contrast helps.

Try it

Take the counter at the top of this page. Change $state(0) to $state({ count: 0 }) and update the click handler to count.count++. Same behavior — because the reactivity lives on the proxy, not on the variable name.

In fifteen years of framework churn, "the primitive is just a variable" is the single best decision Svelte 5 made. You will forget it's special — that's the point.
