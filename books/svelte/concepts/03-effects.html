Chapter 03 · Side Effects

$effect

Everything that isn't "describe a value from other state" — talking to canvas, wiring a third-party library, logging, a manual fetch — is an effect. It runs after the DOM updates, tracks whatever reactive values it read, and reruns only when one of those changes.

let size = $state(100);

$effect(() => {
  // reads `size` synchronously — that's the one dependency
  canvas.width = canvas.height = size;
  draw(canvas);
});

Only state read synchronously, before any await, counts as a dependency. Read something inside a setTimeout or after an await and the effect won't rerun when it changes — a common "why isn't this updating" trap.

Cleanup, automatically

Return a function from the effect and Svelte calls it as teardown — right before the effect reruns, and again when the component is destroyed. This is how you cancel timers, remove listeners, or close connections without a separate onDestroy.

$effect(() => {
  const id = setInterval(() => tick(), 1000);
  return () => clearInterval(id); // teardown
});

Runs after mount, tears down before every rerun, reruns on the next dependency change — the dashed loop repeats for the effect's whole life.

Effects read state, they shouldn't set it

Writing state inside an effect that also reads that same state is how you build an infinite loop. The docs are blunt about it: "you should not update state inside effects." If two pieces of state genuinely need to stay in sync, prefer a function binding or an oninput handler over a pair of effects watching each other. If you truly can't avoid it, wrap the read in untrack() so it doesn't re-trigger itself.

The other effect runes

  • $effect.pre — same contract, but runs before the DOM updates. Useful for measuring scroll position before new content lands.
  • $effect.root — creates a non-tracked scope with manual cleanup, for effects created outside normal component init (e.g. inside a class constructor).
  • $effect.tracking() — returns whether the current code is running inside a tracked context. Mostly a debugging tool.
Reach for $derived first Before writing an effect, ask: is this computing a value, or doing something the DOM/outside world needs to know about? Computing → $derived. Doing → $effect. This is the single most common Svelte 5 code-review comment.

Try it

Add $effect(() => console.log('count is now', count)) next to the Chapter 01 counter. Click the button, watch the console — then delete the log line and predict what changes (nothing: the effect body still runs, it just does nothing visible).

If your effect's dependency list feels hard to reason about, that's not a useEffect-style skill issue — it's a sign the logic belongs in $derived or an event handler instead. Effects are for the edges of your app, not its center.
