Chapter 07 · The DOM

Bindings & Events

Events are now plain HTML-style attributes, and bind: is how you say "keep this DOM property and this variable in sync" — both directions, no onChange handler required.

Events are attributes, not directives

<button onclick={() => count++}>+1</button>

Svelte 4's on:click directive is gone in runes mode — an event listener is just an attribute that starts with on, and it's case-sensitive (onclick listens for click; there's no such thing as onClick unless the DOM fires a Click event, which it doesn't). Because it's an attribute, the usual shorthands apply: {onclick} when the variable name matches, or spread a whole object of handlers with {...handlers}.

If you've seen Svelte 4 on:click={handler} still works in legacy-mode components, plus modifiers like on:click|preventDefault. In runes mode, call event.preventDefault() yourself inside the handler — one fewer piece of bespoke syntax to remember.

Under the hood, most events are delegated — one listener at the document root handles the actual dispatch. If you ever manually dispatch a delegated event, pass { bubbles: true }, or use on from svelte/events instead of raw addEventListener.

bind:value and friends

let message = $state('hello');
<input bind:value={message} />
<p>{message}</p>

Type in the box, message updates; change message in code, the box updates. One rune, one directive, no event handler in sight. The same pattern covers checkboxes (bind:checked), selects (bind:value), radio/checkbox groups (bind:group), file inputs (bind:files), and media elements (bind:currentTime, bind:paused, …).

bind:this — reaching the raw node

let canvas;

$effect(() => {
  const ctx = canvas.getContext('2d');
  draw(ctx);
});
<canvas bind:this={canvas}></canvas>

canvas is undefined until the component mounts — that's exactly why the read happens inside $effect, which only runs after the DOM exists. Components support bind:this too, giving you the mounted instance to call its exported functions on.

bind:group only works within one component

Radio and checkbox groups synced with bind:group must live in the same component instance — Svelte wires the group by closure, not by a shared name attribute like plain HTML forms. Splitting a radio group across components needs a shared $state value passed down instead (Chapter 09).

Try it

Build a Fahrenheit/Celsius pair: let f = $state(32) and let c = $derived((f - 32) * 5 / 9), two bind:value inputs. Typing in the Fahrenheit box should update Celsius live — because $derived recomputes the moment f changes underneath it.

First instinct when you want two values in sync: reach for bind: or $derived before you reach for an $effect pair watching each other. Bidirectional effects are the easiest way to build a bug that only shows up under fast typing.
