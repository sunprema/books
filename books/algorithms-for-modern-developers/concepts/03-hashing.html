03 Core

Hashing & Hash Maps

If you learn one data structure deeply, make it this one. The hash map is the workhorse of modern software — the dictionary, the object, the cache, the index. It answers "have I seen this? where's its value?" in O(1) time, on average, no sorting required.

The magic is a hash function: feed it any key — a string, a number, a tuple — and it spits out an integer. Take that integer modulo the table size and you get a slot number. To store a key, compute its slot and drop the value there. To look it up, compute the same slot and read it back. You jumped straight to the answer without searching at all.

Key → hash → slot. No scan; you compute where the answer lives.
◆
Collisions — and why O(1) is "amortized"

Two keys can hash to the same slot. Maps handle this by chaining (a little list per slot) or open addressing (probe the next slot). As the table fills it resizes and rehashes everything — a rare O(n) hiccup that averages out. So lookups are amortized O(1): almost always instant, occasionally paying for a resize.

# 3 · INDEX / JOIN — group by a key
by_user = {}
for ev in events:
    by_user.setdefault(ev.user, []).append(ev)

# 4 · MEMO / CACHE — remember past answers
cache = {}
def get(k):
    if k not in cache:
        cache[k] = expensive(k)
    return cache[k]

Notice the pattern: each one replaces a nested loop (a scan-inside-a-scan, the O(n²) trap from chapter 1) with a single pass and O(1) lookups. The "two-sum" interview question, deduping a dataset, joining two tables — all the same move.

●
Two real gotchas

Keys must be immutable & hashable — mutate a key after inserting and you'll never find it again. Hash-flooding: if attackers can pick keys that all collide, an O(1) map degrades to O(n) and becomes a denial-of-service. That's why languages randomize hash seeds (and why you never expose raw hash order as a security boundary).

◎
Where you'll meet it

Everywhere. Database indexes & hash joins, in-memory caches (Redis, memcached), deduplicating a stream, request routing tables, symbol tables in compilers, content addressing (git stores objects by their SHA hash), detecting duplicate uploads, and the dict/object/Map you touch in every program you write.

Hashing gives you instant lookup but no order. When order is the point — ranked results, sorted exports, merge steps — you need sorting, and that's where surprisingly clever engineering hides.
