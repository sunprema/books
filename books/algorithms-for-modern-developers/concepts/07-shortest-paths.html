Shortest Paths: Dijkstra

BFS finds the route with the fewest hops. But the shortest route on a map isn't the one with the fewest turns — it's the one with the least total distance. When edges have weights (miles, milliseconds, dollars), Dijkstra's algorithm finds the cheapest path, and it's beautifully greedy.

The intuition is a flood of water spreading from your start point at constant speed. The first time the flood reaches any node, it must have arrived by the shortest possible route — water always takes the quickest path. Dijkstra simulates that flood: it always expands the closest unvisited node next, locking in its distance as final.

Dijkstra with a priority queue (min-heap)import heapq

def dijkstra(graph, start):
    dist = {start: 0}
    pq = [(0, start)]            # (distance, node)
    while pq:
        d, u = heapq.heappop(pq)  # closest node next
        if d > dist.get(u, inf): continue
        for v, w in graph[u]:    # neighbor, edge weight
            nd = d + w
            if nd < dist.get(v, inf):
                dist[v] = nd          # found a shorter way
                heapq.heappush(pq, (nd, v))
    return dist
★
The engine: a priority queue

"Always expand the closest node" needs a structure that hands you the minimum fast. That's a min-heap (chapter 10). With it, Dijkstra runs in O((V + E) log V) — fast enough for city-sized road networks. The heap is what makes the greedy choice cheap.

Fewest hops (A→B→D) isn't cheapest. Weights change the answer — that's why we need Dijkstra.
●
The one catch: no negative weights

Dijkstra's flood logic assumes reaching a node never gets cheaper later. A negative edge breaks that, so locking distances as "final" becomes wrong. For graphs with negative weights (currency arbitrage, some cost models), use Bellman-Ford instead — slower, but it tolerates negatives and detects negative cycles.

▣
Real maps go further: A*

Production routing rarely runs plain Dijkstra over a continent — it's too broad. A* adds a heuristic ("D is roughly southeast, head that way") to steer the search toward the goal, exploring far fewer nodes. It's Dijkstra plus a sense of direction, and it's what games and map apps actually use.

◎
Where you'll meet it

Navigation: Google/Apple Maps, GPS, ride-sharing ETAs. Networking: OSPF, a core internet routing protocol, runs Dijkstra to pick lowest-cost routes. Games: NPC and unit pathfinding (usually A*). Logistics: delivery routing, supply chains. Anywhere "cheapest path through a weighted network" — even cheapest currency conversion or lowest-latency service mesh hop.

Dijkstra was greedy — always grab the nearest node — and it provably worked. But greed doesn't always win. When local best choices lead to global regret, we need a smarter idea: remembering subproblems. Enter dynamic programming.
