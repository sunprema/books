09 Optimize

Greedy & Intervals

A greedy algorithm makes the choice that looks best right now and never reconsiders. It sounds naive — and often it is — but for a special class of problems, grabbing the local best at every step provably gives the global optimum. When it works, it's faster and simpler than DP.

The catch is proving it works. Greedy fails on plenty of problems (making change with weird coin systems, the knapsack). It succeeds when the problem has a greedy-choice property: a locally optimal pick is always part of some globally optimal solution. The textbook win is interval scheduling.

Activity selection — the canonical greedy

You have a room and many meeting requests, each with a start and end. How do you fit the most non-overlapping meetings? The greedy insight is delightfully counter-intuitive: always pick the meeting that ends earliest. Finishing soonest leaves the most room for everything after.

max non-overlapping intervalsdef max_meetings(intervals):
    intervals.sort(key=lambda iv: iv[1])  # by END time
    count, end = 0, float("-inf")
    for s, e in intervals:
        if s >= end:          # doesn't overlap → take it
            count += 1
            end = e
    return count
★
The shape of a greedy solution

Almost every interval greedy is: sort by the right key (end time, start time, or a ratio), then make a single linear pass taking or skipping each item. The cleverness is entirely in what you sort by. Get that right and the O(n log n) sort dominates; the pass is O(n).

Sorting by finish time and taking the first that fits maximizes the count — provably.

Merging intervals — the everyday cousin

collapse overlapping ranges into onedef merge(intervals):
    intervals.sort()              # by start
    out = [intervals[0]]
    for s, e in intervals[1:]:
        if s <= out[-1][1]:        # overlaps last
            out[-1][1] = max(out[-1][1], e)
        else:
            out.append([s, e])
    return out
◎
Where you'll meet it

Calendars & booking: finding free slots, double-booking detection, meeting-room assignment. Resource scheduling: CPU/job scheduling, ad-slot allocation, CDN cache eviction heuristics. Data: merging date ranges, coalescing log time-spans, IP-range consolidation. Compression: Huffman coding is greedy. Networking: packet scheduling. Finance: simple "buy low, sell high" passes.

●
When greedy lies

Greedy is seductive and often wrong. The classic trap: making change for 30¢ with coins {25, 10, 1} — greedy grabs 25 then five 1s (6 coins), but 10+10+10 (3 coins) is better. If you can't prove the greedy-choice property, fall back to DP. Test greedy against brute force on small inputs before trusting it.

Both greedy scheduling and Dijkstra leaned on a structure that always coughs up the current best item. Time to meet it properly — the heap, and the "top-K" problems it makes easy.
