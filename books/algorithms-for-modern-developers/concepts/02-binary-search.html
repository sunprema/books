02 Search

Binary Search & "Search the Answer"

Think of finding a word in a physical dictionary. You don't start at page one. You flip to the middle, see if your word is before or after, and throw away half the book. Then half of what's left. That's binary search — and it turns a million entries into about twenty steps.

The requirement is one thing: the data must be sorted (or more precisely, monotonic — there's a point where the answer flips from "no" to "yes"). Given that, each comparison eliminates half the remaining candidates, so the cost is O(log n).

binary search — find target in a sorted arraydef search(arr, target):
    lo, hi = 0, len(arr) - 1
    while lo <= hi:
        mid = (lo + hi) // 2   # the middle page
        if arr[mid] == target:
            return mid
        elif arr[mid] < target:
            lo = mid + 1        # answer is to the right
        else:
            hi = mid - 1        # answer is to the left
    return -1
●
Famously easy to get wrong

Off-by-one in the bounds, an infinite loop when lo never advances, integer overflow in (lo+hi) in fixed-width languages (use lo + (hi-lo)/2). Reach for your standard library's version — bisect (Python), Arrays.binarySearch (Java), sort.Search (Go), slice::binary_search (Rust) — unless you have a reason not to.

The everyday win isn't even "find X." It's finding the insertion point — "where would this value go to keep the list sorted?" That powers leaderboards, autocomplete ranges, and time-series queries ("give me all events between 2pm and 3pm").

Each step throws away half. 1,000,000 items → at most 20 comparisons.

The big idea: "binary search the answer"

Here's the trick that elevates this from a lookup to a problem-solving superpower. Binary search doesn't need an array — it needs a yes/no question whose answer is monotonic. If "can I do it with budget X?" is false up to some threshold and true after it, you can binary-search for that threshold even when there's no list at all.

search a value, not an index# "What's the smallest ship capacity that
#  delivers all packages within D days?"
def min_capacity(weights, days):
    lo, hi = max(weights), sum(weights)
    while lo < hi:
        mid = (lo + hi) // 2
        if feasible(weights, days, mid):
            hi = mid            # works → try smaller
        else:
            lo = mid + 1        # too small → go bigger
    return lo
◎
Where you'll meet it

Databases: B-tree indexes are binary search on disk. Version control: git bisect finds the bad commit in log₂(commits) steps. Rate limiting / capacity planning: "smallest server count that holds the load." Time-series & logs: jump to a timestamp in a huge sorted file. Deployment: binary-search a config threshold that flips behavior.
