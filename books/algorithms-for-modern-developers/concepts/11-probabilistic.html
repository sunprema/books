11 Scale

Probabilistic Structures

Here's a radical bargain for the age of big data: give up exactness and get answers in kilobytes that would otherwise need gigabytes. Probabilistic data structures accept a tiny, controllable error rate in exchange for staggering memory savings. At scale, that trade is often a steal.

Bloom filter — "definitely not, or probably yes"

A Bloom filter answers set membership — "have I seen this before?" — using just a bit-array and a few hash functions. To add an item, hash it several ways and set those bits to 1. To check it, hash the same ways and look: if any bit is 0, the item is definitely not present. If all are 1, it's probably present.

A few hashes into a bit array. False positives possible; false negatives impossible.
★
The asymmetry is the point

Zero false negatives. If a Bloom filter says "no," it's the truth — so you can use it as a cheap pre-filter to skip expensive work. At roughly 10 bits per element you get about a 1% false-positive rate, tunable by adding bits and hashes. Storing 1 billion items costs ~1.2 GB exactly, or ~1.2 GB→a few hundred MB as a Bloom filter at a known error rate.

HyperLogLog — counting the uncountable

"How many distinct users hit the site today?" Counting exactly means storing every ID you've seen — gigabytes for a busy service. HyperLogLog estimates the count of unique items with about 2% error using only a couple of kilobytes, whether you're counting thousands or trillions.

◆
The wild intuition

Hash each item to random-looking bits and watch for rare patterns — like a long run of leading zeros. Seeing "10 zeros in a row" is unlikely unless you've hashed roughly 2¹⁰ distinct things. So the rarest pattern observed is a fingerprint of how many uniques flowed past — no list required. Average several such estimators and the ~2% accuracy falls out.

in practice — Redis ships both natively# count unique visitors in ~12 KB, any scale
PFADD visitors:today  user42 user99 user42
PFCOUNT visitors:today        # → ~2 (approx)

# Bloom filter for "seen this URL?"
BF.ADD   seen_urls  "https://..."
BF.EXISTS seen_urls "https://..."  # 0 = definitely no
◎
Where you'll meet it

Bloom filters: databases (Cassandra, Bigtable, RocksDB) skip disk reads for keys that "definitely don't exist"; CDNs & browsers flag malware URLs; "have we emailed this user?" dedup; crawler "seen URL?" sets. HyperLogLog: unique-visitor analytics, COUNT(DISTINCT) estimates in BigQuery/Redshift, network flow cardinality, ad-impression reach. Count-Min Sketch (a cousin) estimates frequencies for trending/heavy-hitters.

●
Know the limits

These are estimates — never use them where exactness is required (billing, balances, "is this exact password seen"). And a classic Bloom filter can't delete items. Use them as fast, cheap pre-filters in front of an exact source of truth, not as the source of truth itself.

That's the toolkit — from Big-O to Bloom filters. None of these is exotic; each is a lens for recognizing a problem you've already met. Flip to the cheatsheet to keep them all on one page.
