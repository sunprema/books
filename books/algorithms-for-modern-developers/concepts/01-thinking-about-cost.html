Thinking About Cost — Big-O

You don't pick an algorithm because it's clever. You pick it because of how its cost grows as the input gets bigger. Big-O is the one notation that lets you predict that growth before you ever run the code.

Here's the whole idea in a sentence: Big-O describes how the work scales as n → ∞, ignoring constants and small terms. We don't care that one loop does 3 operations and another does 5 — we care whether doubling the input doubles the work, quadruples it, or barely moves it. That shape is what survives at scale.

Why throw away the constants? Because at n = 10 everything is fast and it doesn't matter; at n = 10,000,000 the shape is the only thing that matters. An O(n) function on ten million items is a blink; an O(n²) one is your afternoon.

The line that matters

The jump from O(n²) to O(n log n) is the single most common performance win in real code. Most "why is this slow?" bugs are an accidental quadratic — a loop inside a loop over the same data — and most fixes are a hash map or a sort that flattens it.

The curves diverge fast. The flatter your line, the more input you can survive.

Spot the accidental quadratic

the classic O(n²) — and its O(n) cure# ✗ O(n²): a scan inside a scan
for a in items:
    if a in other_list:   # 'in' on a list is O(n)!
        hits.append(a)

# ✓ O(n): build a set once, then O(1) lookups
seen = set(other_list)   # O(n)
hits = [a for a in items if a in seen]  # O(n)
▣
Three flavors: best / average / worst

Quicksort is O(n log n) on average but O(n²) on a pathological input. Real libraries care about the worst case because adversaries and bad luck both happen. We'll see exactly this when we get to sorting.

Don't forget space

Big-O measures time, but the same notation describes memory. Often you trade one for the other: that set above spends O(n) extra memory to buy O(n) time instead of O(n²). Half of practical algorithm design is choosing which resource you can afford to spend.
