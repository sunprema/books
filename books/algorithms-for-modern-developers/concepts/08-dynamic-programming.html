08 Optimize

Dynamic Programming

Dynamic programming has a scary name for a humble idea: don't solve the same subproblem twice — write the answer down. If a big problem breaks into overlapping smaller ones, remember each small answer and an exponential blow-up collapses to something fast.

The classic illustration is Fibonacci. The naive recursion recomputes fib(3) over and over — the call tree explodes to O(2ⁿ). But there are only n distinct answers. Cache them and it's O(n):

memoization — top-down DP, one line of cachingfrom functools import cache

@cache                      # remember every result
def fib(n):
    if n < 2: return n
    return fib(n-1) + fib(n-2)
# 2^n calls → n calls, just by not repeating work
★
The two things DP needs

1 · Overlapping subproblems — the same smaller question comes up many times (so caching pays off). 2 · Optimal substructure — the best answer to the whole is built from best answers to the parts. If both hold, DP applies. If subproblems don't repeat, plain recursion is already fine.

There are two styles, and they're the same idea facing opposite directions:

  • Top-down (memoization) — write the natural recursion, slap a cache on it. Easy; only computes subproblems you actually need.
  • Bottom-up (tabulation) — fill a table from the smallest cases up. No recursion overhead, easy to optimize memory.

The one that ships everywhere: edit distance

"How many single-character insertions, deletions, or substitutions turn kitten into sitting?" This Levenshtein distance is pure DP — build a grid where each cell reuses its neighbors' answers — and it's the engine behind a startling amount of software.

Fill the grid; each cell reuses the cells above, left, and diagonal. The corner is the answer.
◎
Where you'll meet it

Diffs & version control: git diff and merge tools compute longest-common-subsequence (a DP) to show changes. Spell-check & autocomplete & "did you mean?": edit distance ranks suggestions. Bioinformatics: DNA sequence alignment. Fuzzy search & record deduplication. Finance/ops: knapsack-style budgeting. Even some regex engines are DP under the hood.

●
The cost & the trap

DP trades memory for time — that grid is O(n·m) space (often reducible to one row). The hard part isn't coding it; it's spotting the recurrence — defining "what's the subproblem and how does it build on smaller ones?" That framing is the whole skill.

DP wins when greedy local choices would fail. But sometimes greed is exactly right — and far simpler. Knowing which is the art. Next: greedy algorithms.
