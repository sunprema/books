05 Core

Two Pointers & Sliding Window

A whole family of O(n²) brute-force loops collapse to a single O(n) pass with one trick: instead of re-scanning from scratch, keep two indices that crawl through the data, never backing up. It feels like cheating. It isn't.

Two pointers — converge from the ends

On sorted data, put one finger at each end and move them toward each other based on what you see. Looking for a pair that sums to a target? If the sum's too big, the right finger steps left; too small, the left finger steps right. Every step rules out a whole row of the brute-force grid.

pair-sum in O(n) instead of O(n²)def has_pair(arr, target):   # arr is sorted
    lo, hi = 0, len(arr) - 1
    while lo < hi:
        s = arr[lo] + arr[hi]
        if s == target: return True
        if s < target:  lo += 1   # need more → raise floor
        else:           hi -= 1   # too much → lower ceiling
    return False
The fingers walk inward; they collectively touch each element at most once → O(n).

The same shape powers in-place work: reversing an array, partitioning, removing duplicates from a sorted list, merging two sorted streams. One pass, no extra memory.

Sliding window — a stretchy range

When the question is about a contiguous run — "the longest substring without repeats," "the max sum of any 10 consecutive readings," "requests in the last 60 seconds" — keep a window with a left and right edge. Push the right edge to grow; when the window breaks a rule, pull the left edge until it's valid again. Each edge only ever moves forward, so the whole sweep is O(n).

longest substring with all-unique charsdef longest_unique(s):
    seen = {}              # char → last index
    left = best = 0
    for right, c in enumerate(s):
        if c in seen and seen[c] >= left:
            left = seen[c] + 1   # shrink past the repeat
        seen[c] = right
        best = max(best, right - left + 1)
    return best
★
Why it's O(n), not O(n²)

It looks like nested loops, but the left and right edges each advance at most n times total across the whole run — they never reset. Amortized, every element is added once and removed once. Recognizing "I can move a boundary instead of re-scanning" is the entire skill.

◎
Where you'll meet it

Rate limiting: a sliding-window counter — "max 100 requests per 60s" — is literally this pattern over timestamps. Streaming analytics: moving averages, rolling maximums, anomaly windows. Networking: TCP's sliding window for flow control. Text & bioinformatics: substring search, k-mer scanning. Media: buffering N seconds ahead of playback.

▣
When it applies

Two pointers want sorted or two-ended structure; sliding window wants a contiguous range and a rule that's monotonic as the window grows/shrinks. If the subarray can be non-contiguous, or the rule isn't monotonic, you're back to hashing or dynamic programming.

These patterns shine on linear data. But a lot of the world isn't a line — it's a network of connections. To explore that, we need graphs.
