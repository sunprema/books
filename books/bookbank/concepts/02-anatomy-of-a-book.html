The Anatomy of a Book

Let’s open the hood. Because a book is just files, the best way to understand BookBank is to look at one folder. Every book lives under a data root in books/<book-id>/, and the layout is always the same — which is exactly why any book in the library feels familiar the moment you open it.

One folder, predictable insides

books/rust-programming/# the folder is the book
book.json                  # the manifest — the request + the index
index.html                 # cover + table of contents (the landing page)
concepts/
  01-ownership.html        # one page per concept, in order
  02-borrowing.html
cheatsheet.html            # a dense one-page quick reference
assets/
  book.css                 # the book's bespoke stylesheet
  book.js                  # the page-turn pager
  img/                     # images, kept local so the book works offline
cover.png                  # optional: artwork the gallery shows
The id is the folder name. A book’s id (like rust-programming) is its folder, its URL slug, and how the app finds it. Pick it once; it doesn’t change.

book.json: the one file you’ll recognize

If you only learn one file, learn this one. book.json is the manifest — it holds the request (what you asked for) and the index (what got built). The app writes it when you request a book; the plugin reads and updates it as it works. Here’s a trimmed one:

book.json (abridged){
  "id": "rust-programming",
  "title": "Rust: Fearless Systems Programming",
  "topic": "the Rust programming language",
  "persona": "feynman",        // the voice (Ch. 3)
  "theme": "blueprint",        // the look  (Ch. 4)
  "status": "ready",
  "concepts": [
    { "id": "ownership", "title": "Ownership & Moves",
      "file": "concepts/01-ownership.html",
      "status": "ready", "source": "user" }
  ]
}

You mostly don’t hand-edit this — the tools do — but reading it tells you the whole state of a book at a glance: what it’s about, whose voice it’s in, and which chapters exist.

Status: how a book grows up

The status field is a tiny state machine, and it’s worth knowing because it tells you where a book is in its life:

StatusMeans
requestedYou’ve asked for it; nothing is written yet.
readyBuilt. index.html exists and every chapter is written.
revisingYou asked for changes; the tools are improving it in place.

Each chapter has its own status too (requested → ready), which is what lets you add a single new chapter later without rebuilding the whole book. And each chapter records a source: user (you asked for it) or claude (the author judged it essential). Chapters you asked for are never dropped or reordered.

The two-page spread

Open any book and you’ll notice it reads like a physical book: two pages side by side with a center spine, filling the screen, with a page-flip when you turn. Long chapters simply paginate into more spreads. You turn pages with → and ←, and jump home with ↑ — try it right now on this very page.

It adapts On a narrow screen the spread gracefully collapses into a single, normally-scrolling column — so the same book is comfortable on a laptop and on a phone. You don’t do anything; it just happens.

The building blocks on a page

Within that consistent frame, a chapter is built from a small, repeating kit of parts — the same parts this book uses:

  • Prose & leads — the explanation itself, in the persona’s voice.
  • Callouts — key ideas, tips, and warnings, like the boxes on this page.
  • Code blocks — syntax-highlighted, copy-ready, with no external dependencies.
  • Figures — hand-drawn inline diagrams (SVG) for anything explanatory, plus image slots for real artwork (Ch. 7).
  • Tables, challenges, and a cheatsheet — for reference and practice.

The mental model to keep

A book = a folder. book.json is its manifest; index.html is its cover; each concept is one HTML page; the shared look lives in assets/. Once you can picture that, the rest of BookBank is just deciding what goes in — starting with the voice.
