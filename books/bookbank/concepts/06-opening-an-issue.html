Requesting a Book by Issue

Not everyone wants to run a terminal — and you don’t have to. The most approachable way to ask for a book is to open a GitHub Issue on the library repo. You fill in a short form; a maintainer (or an automated routine) picks it up, generates the book, and links a pull request back to your issue for review. It’s the same request from Chapter 5, wearing a friendlier coat.

The request form

The repo ships an Issue Form template called “New book request”. Choosing it gives you labelled fields instead of a blank box — here’s what it looks like:

github.com/sunprema/books · new issue · New book request

What each field maps to

Every field lines up with something you met in Chapter 5. Only Topic and the consent box are required; blank fields become sensible defaults.

FieldBecomes
Topic *the book’s scope (required)
Titleyour wording, or an auto-picked one
Personaa voice — an existing id, or free text
Themea look — an existing id, or free text
Seed conceptsone required chapter per line
Notesaudience, depth, emphasis
Reference materiallinks/images for research & art direction only
About Reference material: those links and images guide research and illustration — they are not published as-is. Your private mood-board never ends up on the public site.

Prefer the command line?

If you live in a terminal, open the same request with the GitHub CLI. Give it the book-request label so the tooling can find it:

open a book request from the terminalgh issue create --repo sunprema/books \
  --label "book-request" \
  --title "[book] PostgreSQL indexing for app developers" \
  --body  "Topic: PostgreSQL indexing for application developers.
Persona: clear-mentor.  Theme: blueprint.
Notes: assume they know SQL but not internals; focus on
B-tree vs GIN and reading EXPLAIN."
The label is the doorbell The book-request label is what tells BookBank’s tooling “here’s a book to make.” The Issue Form adds it for you automatically; if you open a plain issue, add it yourself.

What happens after you submit

  1. Your issue lands in the queue, tagged book-request.
  2. When work starts, it’s marked in-progress so no one duplicates it, and a note is posted: “🏗️ Work started — a PR will be linked here.”
  3. The book is generated and a draft pull request is opened, linked to your issue.
  4. You (and maintainers) review it. When the PR merges, your issue closes automatically and the book publishes.

The friendly front door

An issue is just your Chapter-5 request in a form. Fill in the topic, pick a voice and look if you like, add a note or two, and submit. From there it becomes a tracked, reviewable pull request. Let’s follow one all the way to a live shelf.
