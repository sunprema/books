Make It Yours — Recipes & Gotchas

You now know the whole system. This last chapter is the workbench: a handful of recipes you’ll reach for again and again, and the small gotchas that separate a book that works everywhere from one that quietly breaks. Skim it now; come back to it when you’re building.

Recipe 0: get set up (once)

install the plugin, clone the library# in Claude Code, add the marketplace and install
/plugin marketplace add sunprema/kit
/plugin install bookbank@kit

# then clone the content repo and work from inside it
git clone https://github.com/sunprema/books
cd books

From inside that clone, the skills detect you’re in a content repo and write new books under ./books/<id>/ — no configuration needed.

Recipe 1: write, expand, revise

You want to…Run
Build a new book/write-book <topic>
Build from a filed issue/create-book-from-issue <n>
Add one chapter/write-book expand "<concept>" in <book>
Do all outstanding work/write-book

To revise a book, the app’s “Regenerate” action (or a quick edit to book.json) flips it to revising with a note about what to change; the next run improves it in place rather than rebuilding from scratch. Untouched chapters and the design survive; only what you asked about changes.

Comment right on the page The app’s Draft mode lets you attach a comment to a specific paragraph — “add an example here,” “this is wrong,” “simplify.” The next revise pass applies each one to exactly that block. It’s the fastest way to steer a nearly-there book.

Recipe 2: fill an image slot without the app

Generated a picture for a slot? Drop it in with one command — it converts, renames, and places the file exactly where the slot expects, checking the aspect ratio for you:

place artwork into a declared slotpython3 $CLAUDE_PLUGIN_ROOT/library/place_image.py \
  <book-dir> <slot-id> <path-to-your-image>

In the app it’s even simpler: drag the image onto the placeholder and it’s wired up — no rebuild. A slot left empty isn’t an error; the shelf just shows a gradient cover until the art lands.

The gotchas that actually bite

Stay self-contained A book must render from file:// with no network. That means no CDN links, no hotlinked images, no web fonts that need a connection — every asset lives in the book folder. It’s what makes a book portable and permanent.
Accuracy beats polish Beautiful but wrong is a failure. If a code sample or a number can’t be grounded in research, it doesn’t belong in the book. Persona changes the tone, never the truth.
Check the phone view The two-page spread must collapse cleanly to one scrolling column under ~900px. If you hand-edit layout, narrow the window and confirm nothing clips or overlaps — a book that only works on a wide screen isn’t done.
One strong image per idea, at most. Lean on inline SVG for anything explanatory — it’s crisp, weightless, and matches the theme. Save real image slots for artwork you genuinely can’t draw, like a cover.

The pre-flight checklist

  • Every chapter reads in the chosen voice; the whole book wears the chosen look.
  • Pages turn with ← → and with mouse clicks, through every spread, before crossing chapters.
  • It collapses gracefully on a narrow screen.
  • Nothing reaches for the network; it works offline.
  • Claims and code are grounded; sources noted where useful.
  • book.json is ready, with a one-line summary and any image slots declared.

You can make a book now

You know what BookBank is, how a book is built, how to choose a voice and a look, how to ask well — by prompt or by issue — and how a request travels all the way to a public shelf. The rest is practice. Pick a topic you love, choose a narrator, and press go. The cheatsheet has everything on one page for when you do.

Your first book

  1. Install the plugin, clone the library, and run /write-book on a topic you know well — so you can judge the accuracy.
  2. Regenerate it once with a different persona. Notice how much the voice alone changes the read.
