Rust on the ESP32

This is the chapter that answers your question directly: how do I get a Rust binary running on this chip? The trick is knowing there are two Rusts here — and choosing the right one.

When people say "Rust on ESP," they're being a little loose, because two quite different worlds wear the same name. Understanding the split is 80% of the battle — once you see it, every confusing tutorial suddenly clicks into place. So let's draw the fork in the road clearly before we walk down it.

The two worlds

no_std · bare metal

The crate esp-hal. No operating system underneath — your Rust is the system. Lean, fast, deterministic. Vendor-backed and hit 1.0 in Oct 2025. Async-first (pairs with Embassy).

std · on ESP-IDF

The crates esp-idf-svc / esp-idf-hal. Rust sits on top of Espressif's C framework, so you get std, threads, and a battle-tested Wi-Fi + MQTT stack today.

🔑
How to choose (the short version)

Want the leanest, most "embedded" path and you're happy on the frontier? no_std / esp-hal. Want Wi-Fi and MQTT working this afternoon to feed your Elixir backend? std / esp-idf-svc. For this book's networked fleet, we lean std — its networking is stable today.

The fork in the road

Two HALs, one flashing tool. Pick a lane up front — mixing them mid-project is painful.
📡
Where Wi-Fi lives today

In esp-hal 1.0 the radio (esp-radio, formerly esp-wifi) is still the next stabilization target — it works but its API can shift. In the std world, Wi-Fi & MQTT ride on the mature ESP-IDF C stack. That's why networked projects often start on std.

Setting up the toolchain

The S3 uses Xtensa cores, and — here's a wrinkle — mainline Rust doesn't target Xtensa yet. So you install a small Espressif-maintained fork of the toolchain with a tool called espup. (If you ever use a RISC-V ESP chip like the C3/C6, you can skip this and use plain nightly Rust.)

one-time setup# 1 · install the helpers
$ cargo install espup espflash
$ espup install          # fetches the Xtensa Rust + LLVM
$ source ~/export-esp.sh  # puts the toolchain on PATH

# 2 · scaffold a project (no_std template)
$ cargo install esp-generate
$ esp-generate --chip esp32s3 my-firmware

For the std path you'd instead start from the esp-idf-template (cargo generate esp-rs/esp-idf-template), which wires up the ESP-IDF build for you.

⚠️
The Xtensa gotcha

Forget espup / source export-esp.sh and you'll get baffling "can't find target xtensa-esp32s3-none-elf" errors. Nine times out of ten on the S3, a build failure is a forgotten source ~/export-esp.sh in a fresh terminal.

Your first blink, for real

Here's a complete no_std esp-hal program. Read it top to bottom — every line earns its place:

src/main.rs · esp-hal 1.0#![no_std]
#![no_main]

use esp_hal::{
    clock::CpuClock,
    gpio::{Level, Output, OutputConfig},
    main, delay::Delay,
};

#[main]
fn main() -> ! {
    // hand us the chip's peripherals, once.
    let p = esp_hal::init(
        esp_hal::Config::default()
            .with_cpu_clock(CpuClock::max()),
    );

    let mut led = Output::new(
        p.GPIO2, Level::Low, OutputConfig::default(),
    );
    let delay = Delay::new();

    loop {
        led.toggle();
        delay.delay_millis(500);
    }
}
flash & watch it run$ cargo run --release   # builds, flashes, opens the monitor
🍎
What just happened

No OS booted. The CPU jumped straight into your main, which never returns (-> !) — there's nowhere to return to. esp-hal::init hands you the peripherals exactly once, so two parts of your code can't fight over the same pin. The borrow checker enforced that at compile time.
