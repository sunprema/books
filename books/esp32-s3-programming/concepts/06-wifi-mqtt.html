Wi-Fi, MQTT & the Outside World

A blinking LED is a closed loop. The moment the chip publishes a reading over Wi-Fi, it becomes part of something bigger. Here's how it reaches out.

There are two steps to a device "phoning home." First it must join a Wi-Fi network — get itself an address on the local network, just like your laptop does. Second, having an address, it needs a way of talking that suits thousands of little devices and one server. That second part is where MQTT comes in, and once you understand why it exists, the whole IoT world makes sense.

Why not just HTTP?

You could have each device POST to a web server. It works for a few devices. But HTTP makes every device ask ("is there anything for me?"), connections are heavy, and pushing a command down to a sleepy device is awkward. For fleets, we want something lighter and two-way.

🔑
MQTT in one breath

A tiny publish/subscribe protocol. Devices don't talk to each other or to the server directly — they all connect to a broker and tag messages with a topic. Publish to a topic; anyone subscribed to it gets the message. The broker is the post office; topics are the addresses.

The shape of MQTT

Every device speaks only to the broker. Add a thousand more and the server-side code doesn't change.

Two MQTT ideas worth knowing

  • QoS (0/1/2) — how hard to try to deliver. 0 fire-and-forget, 1 at-least-once, 2 exactly-once. Sensors usually use 0 or 1.
  • Last Will (LWT) — a message the broker sends for a device if it drops off. Perfect for "node went offline" — we'll use it for presence in Chapter 7.

Joining Wi-Fi in Rust

On the std path (esp-idf-svc), connecting looks reassuringly like normal networking code — because underneath, the mature ESP-IDF stack is doing the heavy lifting:

connect to Wi-Fi · esp-idf-svcuse esp_idf_svc::wifi::{BlockingWifi, EspWifi, ClientConfiguration, Configuration};

let mut wifi = BlockingWifi::wrap(
    EspWifi::new(peripherals.modem, sys_loop.clone(), Some(nvs))?,
    sys_loop,
)?;

wifi.set_configuration(&Configuration::Client(
    ClientConfiguration {
        ssid:     "my-wifi".try_into().unwrap(),
        password: "secret".try_into().unwrap(),
        ..Default::default()
    },
))?;

wifi.start()?;
wifi.connect()?;
wifi.wait_netif_up()?;   // we now have an IP address
🔒
Don't bake in secrets

Hard-coding the password is fine for a first run, but for real devices store credentials in NVS (the chip's key-value flash) or provision them over BLE/SoftAP at setup. Firmware in the field can be dumped.

Publishing a reading over MQTT

Now the payoff. With an IP address, open an MQTT client and publish a JSON reading to a per-device topic — this is the message your Elixir backend will receive:

publish · esp-idf-svc MQTTuse esp_idf_svc::mqtt::client::{
    EspMqttClient, MqttClientConfiguration, QoS,
};

let (mut client, mut conn) = EspMqttClient::new(
    "mqtt://broker.local:1883",
    &MqttClientConfiguration {
        client_id: Some("node-7"),
        ..Default::default()
    },
)?;

let topic   = "sensors/node-7/temp";
let payload = format!(r#"{{"c":{:.1}}}"#, temp_c);

client.publish(
    topic,
    QoS::AtLeastOnce,         // QoS 1
    false,                     // not retained
    payload.as_bytes(),
)?;
🍎
Step back and look

That's the whole device side of your system. Read a sensor → wrap it in a few JSON bytes → publish to a topic. Everything else — fan-out, storage, dashboards, commands — happens on the server, where you have Elixir and all the room in the world. Turn the page; we go meet the brain.
