The Elixir Backend

The devices are out in the world, publishing. Now we build the brain that listens — and there's a beautiful reason you chose Elixir for it.

Imagine ten thousand little radios all chirping at once. Most languages handle that by sweating: thread pools, callback spaghetti, careful locking. The BEAM — the virtual machine Elixir runs on — was built for telephone switches in the 1980s, where "tens of thousands of simultaneous conversations that must never drop" was the everyday problem. IoT is the same shape of problem wearing new clothes. That's not a coincidence you're exploiting; it's a perfect fit.

Why the BEAM loves IoT

  • Cheap processes. You can spawn a lightweight process per device — 50,000 of them — each holding that device's state. Not OS threads; BEAM processes, grams not kilograms.
  • Supervision. If one device's handler crashes, a supervisor restarts just that one. The other 49,999 never notice. "Let it crash" is a feature.
  • Soft real-time. The scheduler is preemptive and fair, so one chatty device can't starve the rest.
⚠️
The boundary, made crisp

Elixir runs on your server, not on the ESP32. The chip is too small for the BEAM. (Nerves brings Elixir to Linux boards like a Raspberry Pi — a different class of device.) The S3 reaches Elixir over the network, via the broker.

The backend, drawn

Broker → MQTT client → PubSub fans each reading out to a live dashboard, the database, and alerting.

The flow reads left to right: the broker hands messages to tortoise311 (an Elixir MQTT client), which decodes them and shouts them onto Phoenix.PubSub. Anything that cares — a live chart, a storage writer, an alert rule — subscribes. Adding a new consumer never touches the device code.

Subscribing to the fleet

Here's a tortoise311 connection that subscribes to every device's readings with a wildcard topic, and a handler that re-broadcasts each one over PubSub:

lib/iot/mqtt.exdef start_link(_opts) do
  Tortoise311.Connection.start_link(
    client_id: :backend,
    server: {Tortoise311.Transport.Tcp,
             host: "localhost", port: 1883},
    handler: {Iot.MqttHandler, []},
    subscriptions: [{"sensors/+/temp", 1}]  # + = any node
  )
end
lib/iot/mqtt_handler.exdefmodule Iot.MqttHandler do
  use Tortoise311.Handler

  def handle_message(topic, payload, state) do
    ["sensors", node, "temp"] = topic
    %{"c" => celsius} = Jason.decode!(payload)

    Phoenix.PubSub.broadcast(
      Iot.PubSub, "readings",
      {:reading, node, celsius}
    )
    {:ok, state}
  end
end
🔑
The wildcard is the magic

Subscribing to sensors/+/temp means one piece of code receives readings from every node — present and future. Deploy a thousand more S3s tomorrow and this handler doesn't change a line.

A live dashboard for free

Because the reading is already on PubSub, a Phoenix LiveView page just subscribes and re-renders itself whenever data arrives — no JavaScript, no polling, real-time in the browser:

lib/iot_web/live/dashboard.exdef mount(_p, _s, socket) do
  if connected?(socket),
    do: Phoenix.PubSub.subscribe(Iot.PubSub, "readings")

  {:ok, assign(socket, readings: %{})}
end

def handle_info({:reading, node, c}, socket) do
  # pushes a fresh value into the live page
  {:noreply,
   update(socket, :readings, &Map.put(&1, node, c))}
end
🍎
Feel how little code this is

Three small modules and you have a real-time, multi-device dashboard that scales to thousands of nodes. That's not because we're clever — it's because the BEAM was built for exactly this shape of problem. You picked the right brain.

🪦
Presence, via Last Will

Have each device set an MQTT Last Will on sensors/<id>/status = offline. The broker publishes it automatically if the node drops — your handler flips a dot from green to grey with zero device-side effort.
