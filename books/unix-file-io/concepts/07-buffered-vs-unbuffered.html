Concept 07 · stdio

Buffered vs Unbuffered

printf isn't a system call. It's a layer of paint over write() — and that layer is where a famous, maddening bug lives.

Everything in this book so far — open, read, write, close, lseek — is a system call: a direct request to the kernel, one trip across the wall every time. printf, fprintf, fread, fwrite are something else entirely: ordinary C library functions, running in your process, that eventually call write on your behalf.

🔑
Why bother with a layer at all?

Because a system call is expensive — a trip into the kernel and back. printf("x") a million times, one write each, and you'll spend more time crossing the wall than doing anything useful. stdio's fix: collect your bytes in a buffer in your process's own memory, and only call write when that buffer is full.

Three buffering modes

stdio picks one of three strategies per stream:

modeconstantflushes when…
fully buffered_IOFBFthe buffer fills up
line buffered_IOLBFa newline is written, or the buffer fills
unbuffered_IONBFimmediately, every call

And the default stdio picks for you, before you ask:

Same printf call, different flush timing — depending on what's on the other end of fd 1.

The bug this explains: the vanishing printf

You've probably seen this: a program crashes, and the very last printf before the crash — the one that would have told you exactly where things went wrong — never made it to the screen. It's not lost. It's still sitting in a buffer in your process's memory, and the crash tore that memory down before anything flushed it.

c · this printf may never appearprintf("about to divide by zero...\n");
int x = 1 / y;   /* if y == 0, the process dies right here */
⛔
Why redirecting a program can change its behavior

Run it in a terminal: line buffered, the trailing \n flushes it, you see the message. Pipe it — ./prog | tee log — and stdout silently switches to fully buffered, because the far end is no longer a terminal. Now that same printf waits in a bigger buffer that a crash never empties. The exact same source code, two different outcomes.

· 27

Forcing the issue: fflush and setvbuf

You control both ends of this. To push whatever's waiting out now, call fflush. To pick a stream's mode yourself instead of taking the default, call setvbuf before you do any I/O on it:

c · flush explicitly, or opt out of buffering entirelyprintf("about to divide by zero...\n");
fflush(stdout);                       /* force it out, right now */
int x = 1 / y;

setvbuf(stdout, NULL, _IONBF, 0);  /* or: never buffer this stream at all */
🧭
The layering, stated plainly

write(2) hands bytes to the kernel — the moment it returns, the data is out of your process, safe from your process crashing (though not yet necessarily on disk; that's a different guarantee, fsync's). printf hands bytes to a buffer in your own memory first, and only calls write later. A crash empties your memory, not the kernel's.

On normal program exit, the C runtime flushes every open stdio stream for you automatically — that's why this bug only bites on crashes (`abort`, a segfault, `_exit` skipping cleanup), never on a clean `return` from `main`.

Sources: setvbuf(3) / man 3 stdio — _IOFBF/_IOLBF/_IONBF, stdout line-buffered on a terminal else fully buffered, stderr always unbuffered.
