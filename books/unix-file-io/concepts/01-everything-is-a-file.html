Concept 01 · The Idea

Everything Is a File

There's exactly one big idea in Unix I/O, and everything else is a footnote to it. Get this and the rest is just details.

Let's start with a question that sounds too simple to be interesting. When your program wants to save some text to disk, what does it actually do? And here's the sneaky follow-up: when it wants to print that same text to your screen instead, how much of the code has to change?

In most of the world's operating systems the honest answer used to be "a lot" — a file was one kind of thing, the screen another, the keyboard a third, the network a fourth, and each had its own special incantations. Unix looked at that pile of special cases and made a wonderfully stubborn decision: refuse to have special cases. Declare that all of them are the same kind of thing — a file — and give that one thing a tiny, universal set of verbs.

🔑
The whole idea in one line

A file is just a stream of bytes you can open, read, write, and close. Make everything — disk files, the terminal, pipes, network sockets, even devices — answer to those same four verbs, and any tool that speaks "file" automatically speaks to all of them.

That's it. That's the concept the whole book unpacks. Notice what it buys you: a program that knows how to copy bytes from one file to another already knows how to copy from the keyboard to the network, because as far as it can tell those are also just files. The generality isn't a feature someone added — it falls out for free the moment you refuse to treat them differently.

What "a stream of bytes" really means

Be careful here, because this is where intuition can mislead you. A Unix file has no built-in idea of lines, or records, or paragraphs, or rows. It doesn't know it's a JPEG or a poem. It is a flat sequence of bytes — byte 0, byte 1, byte 2, on and on — and a length. Nothing more.

Any structure you think a file has, you put there by agreement. "Lines" are just the convention that a byte with value 10 (we call it '\n') means "a line ended here." The operating system neither knows nor cares. This is a feature: because the kernel imposes no format, the same machinery carries text, images, audio, and machine code without ever needing to understand any of them.

No lines, no records — just numbered bytes. Structure is something you agree to, not something the file has.

So when we say "read a file," we precisely mean: copy some run of those numbered bytes into my program's memory. When we say "write," we mean the reverse. Keep that picture — a numbered row of bytes — and the system calls in the next chapters will feel obvious.

The four verbs, and nothing more

Here is the entire vocabulary of Unix I/O. Four verbs. You will spend the rest of this book just learning each one properly, but you can see the whole shape right now:

verbwhat it does
openName a file; get back a small ticket (a descriptor) to refer to it.
readCopy up to N bytes from the file into your memory.
writeCopy up to N bytes from your memory into the file.
closeHand the ticket back; you're done.

Watch these four verbs do something they have no right to be able to do. This little C program copies its input to its output, one buffer at a time — and it never once mentions what its input or output actually is:

c · a universal copy loop#include <unistd.h>

int main(void) {
    char buf[4096];
    ssize_t n;
    /* fd 0 = standard input, fd 1 = standard output */
    while ((n = read(0, buf, sizeof buf)) > 0)
        write(1, buf, n);
    return 0;
}

Compile it, and every one of these just works, with zero changes to the program:

shell · the same binary, four different worlds$ ./copy < poem.txt > copy.txt   # file → file
$ ./copy                        # keyboard → screen
$ cat log | ./copy | grep err   # pipe → pipe
$ ./copy < /dev/urandom | ...    # a device → wherever

But why is that so powerful?

Because the program's author didn't have to anticipate any of those uses. She wrote "read bytes, write bytes." The user — at the shell, later, from outside — decides whether byte-source number 0 is a file, a keyboard, or another program's output. The decision moved out of the code and into composition. That inversion is why Unix tools snap together like Lego, and it's entirely a consequence of "everything is a file."

The famous slogan, stated honestly

You'll hear "in Unix, everything is a file." It's a slogan, so let's be precise about what's true. Not every single thing in the system is literally a file. What's true is that an enormous range of things are made to present the file interface — to answer to read/write/close — so that programs can treat them uniformly:

  • Regular files on disk — the obvious case.
  • Directories — themselves a kind of file the kernel manages.
  • Terminals & devices — your keyboard and screen live under /dev and are read and written like files.
  • Pipes — a byte conduit between two programs, with no file on disk at all.
  • Network sockets — once connected, you read and write them like anything else.
🧭
Where we go next

Every verb above started with that mysterious small number — 0, 1, the thing open hands back. That number is the file descriptor, the handle the whole system turns on. It's the entire next chapter, and it's smaller and simpler than you'd guess.
