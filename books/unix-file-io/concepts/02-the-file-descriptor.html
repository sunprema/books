The File Descriptor

The most important object in Unix I/O is also the least impressive-looking: a small non-negative integer. Let's see why that little number is doing so much work.

Imagine a coat check at a theatre. You hand over your coat and get back a numbered brass tag — say, 3. That tag is not your coat. It's tiny, you can put it in your pocket, and by itself it's meaningless. But hand it back at the counter and the attendant instantly produces exactly your coat, because behind the counter tag 3 points at a shelf that holds all the real, bulky details.

A file descriptor is that brass tag. When you open a file, the kernel does all the heavy lifting — finds the file on disk, checks your permissions, sets up bookkeeping — and hands you back a small integer. From then on, you just wave the number. "Read from 3." "Write to 3." "Close 3." You never touch the real machinery; the kernel keeps that safely behind its counter.

🔑
Definition

A file descriptor (fd) is a small non-negative integer that a process uses to refer to an open file. It's an index into a per-process table the kernel keeps for you. The number is only meaningful inside your process — your fd 3 and my fd 3 are unrelated.

Three tags you're born holding: 0, 1, 2

Here's a lovely convention. Every process, the instant it starts, is handed three descriptors already open. You didn't ask for them; the shell set them up before your program's first line ran:

fdnameC constantusually
0standard inputSTDIN_FILENOyour keyboard
1standard outputSTDOUT_FILENOyour screen
2standard errorSTDERR_FILENOyour screen too

This is why the copy program in Chapter 1 could say read(0, …) and write(1, …) with no setup — those tags were already in its pocket. And it's why error messages go to 2, kept separate from real output on 1: so you can redirect a program's results into a file while its complaints still reach your eyes.

💡
Why two outputs, 1 and 2?

Because "the answer" and "something went wrong" are different streams that deserve different destinations. Run find / -name x > hits.txt and the matches (fd 1) pour into the file while the "permission denied" gripes (fd 2) still scroll past on screen. One stream, you couldn't separate them. Two streams — and remember, they're just files — and the shell can point each one wherever you like.

The rule that quietly makes redirection work

There's one more rule about descriptors, and it looks like a boring implementation detail until you see the trick it enables. When you open something new, the kernel always gives you the lowest-numbered descriptor not currently in use.

0,1,2 are taken at birth, so your first open() reliably returns 3. Nothing is random.

Because 0, 1, and 2 are always taken when your program starts, your first open almost always returns 3. Predictable. And here is the sleight of hand it powers: if you first close descriptor 1, then immediately open a file, that file gets slot 1 — the lowest free one. Now every printf, which faithfully writes to fd 1, lands in the file instead of the screen. You just built output redirection with two system calls, and the program doing the printing has no idea. We'll wire this up for real in Chapter 8.

· 7

Descriptors are a resource — spend them wisely

Your process can't hold infinitely many tags at once; the kernel caps how many descriptors a process may have open. On a typical Linux box the default soft limit is 1024 (you can see it with ulimit -n). POSIX only promises a rather stingy minimum, so don't assume a huge budget:

shell · how many tags may I hold?$ ulimit -n
1024
⚠️
The classic bug: descriptor leaks

Open a file in a loop and forget to close it, and you slowly consume tags you never give back. Eventually open fails with EMFILE ("too many open files") and your program falls over — often long after the real mistake, which makes it maddening to debug. The discipline is simple: every open is a promise to close. Close as soon as you're done, not "eventually."

One reassurance: when a process exits, the kernel automatically closes every descriptor it still held. So short-lived programs get away with sloppiness. It's the long-running ones — servers, daemons, that loop over ten thousand files — where a leaked descriptor is a time bomb.
