Concept 03 · The Syscall

open(): Getting a Handle

One system call turns a name on disk into a descriptor in your pocket. Its two little arguments hide more subtlety than any other call in this book.

You have a path — a string like "notes.txt". You want a descriptor. open is the bridge. You tell it what to open and how you intend to use it, and it hands back the brass tag — or tells you why it couldn't.

c · the signature#include <fcntl.h>

int open(const char *path, int flags, ... /* mode_t mode */);

The return value is the descriptor — a non-negative integer — or -1 on failure. That ... is real C: a third argument, mode, that you supply only when you're creating a new file. We'll get to it.

🔑
The shape to memorize

int fd = open(path, flags, mode); — flags say how you'll use the file (and whether to create it); mode gives the permission bits for a file you're creating. Success → a descriptor ≥ 0. Failure → -1, and the reason is left in errno.

The one access mode — and the trap

Every open must declare exactly one access mode: are you reading, writing, or both?

  • O_RDONLY — read only
  • O_WRONLY — write only
  • O_RDWR — read and write
· 9
⛔
The trap almost everyone hits once

These three are not independent bits you OR together. Their values are literally 0, 1, and 2 — occupying the low two bits together. So O_RDONLY | O_WRONLY does not mean "read and write." It's a logical error and is nothing like O_RDWR. Pick one access mode. It's the other flags that you OR onto it.

The other flags: OR these on

After the one access mode, you bitwise-OR in any extra behaviors you want. These are real independent bits. The workhorses:

flagmeaning
O_CREATCreate the file if it doesn't exist (this is what needs mode).
O_TRUNCIf it exists, chop it to zero length — start fresh.
O_APPENDEvery write jumps to the end first — safe appending.
O_EXCLWith O_CREAT: fail if the file already exists.
O_CLOEXECAuto-close this fd if the process execs a new program.
O_NONBLOCKDon't let reads/writes block; return instead.

Two idioms cover most of daily life. To read an existing file: O_RDONLY. To create-or-overwrite a fresh output file: O_WRONLY | O_CREAT | O_TRUNC. That second combination is exactly what the shell's > redirection does under the hood.

mode & umask: the permissions twist

When you create a file with O_CREAT, you must say what permissions it should have — the read/write/execute bits for owner, group, and others. You pass them as mode, usually written in octal:

c · create a file, 0644 requestedint fd = open("out.txt",
              O_WRONLY | O_CREAT | O_TRUNC,
              0644);   /* rw-r--r-- : owner writes, all read */
if (fd == -1) { perror("open"); return 1; }

But here's the twist that surprises people: the file you get may not have exactly the bits you asked for. Every process carries a umask — a set of permission bits to strip away from every file it creates. The kernel computes the final mode as:

💡
The formula

final = mode & ~umask. If your umask is the common 022, it clears the group-write and other-write bits. So requesting 0666 actually yields 0644; requesting 0777 yields 0755. The umask is a safety net set by your shell so you don't accidentally create world-writable files. It subtracts — it can never grant a permission you didn't ask for.

Instead of raw octal you can OR named constants — S_IRUSR (owner read), S_IWUSR (owner write), S_IRGRP, S_IROTH, and so on — which some people find clearer than counting octal digits. And when O_CREAT is absent, you simply omit mode entirely; it's meaningless when you're only opening something that already exists.

When it fails: check −1, read errno

System calls do not throw exceptions. They tell you they failed by returning a value — for open, that's -1 — and they leave a numeric reason code in the global errno. Ignore the return value and you'll happily read from descriptor -1, which fails, and the bug surfaces three functions later wearing a disguise.

c · the non-negotiable checkint fd = open("config.ini", O_RDONLY);
if (fd == -1) {
    perror("open config.ini");  /* prints: "open config.ini: No such file or directory" */
    return 1;
}

perror translates the current errno into a human sentence and prints it to stderr. Common values you'll meet:

errnomeans
ENOENTNo such file or directory.
EACCESPermission denied.
EEXISTFile exists (you used O_CREAT|O_EXCL).
EMFILEThis process has too many open descriptors.
EISDIRYou tried to open a directory for writing.
🧭
The habit that pays for itself

Check every system call's return value, right where you make it. It feels tedious for about a week, and then it saves you for the rest of your career — because down here, a failure that you don't notice doesn't stop your program. It just quietly corrupts what happens next.

Sources: Linux man-pages open(2) (access modes are not a bitmask; mode & ~umask), umask(2), errno(3), perror(3); W. R. Stevens, APUE, ch. 3–4.
