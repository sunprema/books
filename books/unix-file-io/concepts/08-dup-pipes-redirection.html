dup2, Pipes & Redirection

Every piece is on the table. Now watch the shell build > file and a | b out of nothing but a descriptor table and one small function.

dup2 is the last piece of machinery: it lets you say, precisely, "make descriptor number N mean the same open file description as descriptor M."

c · the signature#include <unistd.h>

int dup2(int oldfd, int newfd);
🔑
What dup2 actually does, step by step

1. If newfd was already open, close it first (silently). 2. Make newfd refer to the very same open file description as oldfd — sharing its offset, per Chapter 5. 3. Return newfd. Both steps happen atomically — no window where newfd is briefly closed and some other thread could grab that number first.

Redirection, in six lines

This is the whole trick behind your shell's > out.txt. Open the real destination on some free descriptor, then dup2 it onto 1 — the number every printf already writes to — and close the original:

c · what "./prog > out.txt" does under the hoodint fd = open("out.txt", O_WRONLY|O_CREAT|O_TRUNC, 0644);
dup2(fd, STDOUT_FILENO);   /* fd 1 now means out.txt */
close(fd);                /* the extra ticket; fd 1 alone is enough */
execvp("prog", argv);      /* prog's printf(...) never even knows */

After this, descriptor 1 is out.txt — every write to stdout, in every library, in code that has no idea a file is even involved, lands in the file. This is precisely the "no special cases" idea from Chapter 1, cashed in for real.

Pipes: a file with no name and two ends

pipe(2) creates a small, one-directional byte conduit that lives only in the kernel — no path, no disk. It hands back two descriptors: fds[0] to read from, fds[1] to write into. Whatever you write into 1 comes back out of 0, in order, like a length of garden hose.

c · a | b, built by handint p[2]; pipe(p);            /* p[0]=read end, p[1]=write end */
if (fork() == 0) {              /* child = the LEFT side, "a" */
    dup2(p[1], STDOUT_FILENO);  /* a's stdout → pipe's write end */
    close(p[0]); close(p[1]);
    execlp("a", "a", NULL);
}
dup2(p[0], STDIN_FILENO);    /* parent = "b": stdin ← pipe's read end */
close(p[0]); close(p[1]);
execlp("b", "b", NULL);
· 31

Same three tools every time: pipe makes the conduit, fork gives both ends a process to live in, dup2 wires each end onto stdin or stdout so neither program has to know a pipe is involved at all. Nothing here is special-cased for pipes — it is exactly the redirection recipe from the last page, aimed at a pipe's fd instead of a file's.

⚠️
Two edge cases worth knowing

Reading with no writers left: once every write end is closed, read on the pipe returns 0 — ordinary EOF, not an error. Writing with no readers left: the kernel sends your process SIGPIPE (which kills it by default); if you catch or ignore that signal, write instead fails with EPIPE.

💡
The atomicity guarantee that makes pipes safe to share

POSIX guarantees that a write of up to PIPE_BUF bytes (4096 on Linux) lands in the pipe as one uninterrupted chunk — never torn apart or interleaved with another writer's bytes. That's why several processes can share one pipe and append short messages without a lock: below PIPE_BUF, the kernel itself keeps each write intact.

Sources: Linux man-pages dup(2) (atomic close+reassign, shared offset), pipe(7) (read end/write end, EOF on no writers, SIGPIPE/EPIPE on no readers, PIPE_BUF = 4096 on Linux); W. R. Stevens, APUE, ch. 15.
