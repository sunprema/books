read() & write(): Moving Bytes

You have a descriptor. Now the two verbs that actually move bytes through it — and a promise each one makes that is smaller than you'd guess.

Once open hands you a descriptor, the real work is just two calls. Both take the same shape: a descriptor, a buffer in your memory, and a count. One copies bytes in, the other copies bytes out.

c · the signatures#include <unistd.h>

ssize_t read(int fd, void *buf, size_t count);
ssize_t write(int fd, const void *buf, size_t count);

read asks for up to count bytes and copies them into buf. write takes up to count bytes from buf and copies them to the file. Both return -1 on error (check errno), or the number of bytes actually moved. That word actually is the whole chapter.

🔑
The promise these calls make — and don't

Neither call promises to move all count bytes. A short read or short write — moving fewer bytes than you asked for — is not an error. It's normal, expected behavior you must code for.

The offset moves itself

Here's something neither call mentions in its signature, yet both depend on: every open file has a hidden current offset — where the next byte will come from or go to. You never pass it in. read and write both start at the current offset and, on success, advance it by exactly the number of bytes moved. Call read twice in a row and the second call picks up exactly where the first left off, automatically.

The kernel keeps the offset for you, per open file description — see Chapter 5.

Why a read comes back short

A short read isn't a bug — it's the world being honest with you. It happens because fewer bytes are available right now: you're reading from a pipe and the writer paused, from a terminal where a line hasn't been typed yet, or from a regular file where you simply asked for more than remains.

💡
The one true end-of-file signal

read returning 0 means, unambiguously, "the offset is at or past the end — there is nothing left." That's the only reliable EOF test. A short read of, say, 3 bytes when you asked for 10 is not EOF — it just means try again for more, or accept what you got.

So a correct reader loops, shrinking its remaining count each time, until it either fills its buffer or sees a 0:

c · read exactly n bytes, or fewer at true EOFssize_t read_full(int fd, void *buf, size_t n) {
    size_t got = 0;
    while (got < n) {
        ssize_t r = read(fd, (char *)buf + got, n - got);
        if (r == -1) return -1;      /* real error */
        if (r == 0)  break;             /* true EOF */
        got += (size_t)r;
    }
    return (ssize_t)got;
}

write has the exact same trap, and it's meaner

write can hand back fewer bytes written than you asked for too — a full disk, a pipe whose buffer is momentarily full, a signal arriving mid-call. The bug this causes is nastier than a short read, because the symptom is silent data loss, not a crash.

⛔
The single most common I/O bug in C

write(fd, buf, n); with the return value ignored. It compiles, it usually works in testing on a fast local disk, and it quietly truncates output the day it runs against a pipe, a slow network filesystem, or a full disk.

c · the write loop you must get rightssize_t write_full(int fd, const void *buf, size_t n) {
    size_t sent = 0;
    while (sent < n) {
        ssize_t w = write(fd, (const char *)buf + sent, n - sent);
        if (w == -1) {
            if (errno == EINTR) continue;   /* a signal, not a failure — retry */
            return -1;
        }
        sent += (size_t)w;
    }
    return (ssize_t)sent;
}

Both loops also handle errno == EINTR — "interrupted by a signal before any data was transferred." That's not damage, just a call that got politely interrupted; the fix is simply to try again.

Sources: Linux man-pages read(2), write(2) — short reads/writes are explicitly not errors; read returning 0 as end-of-file; EINTR retry semantics.
