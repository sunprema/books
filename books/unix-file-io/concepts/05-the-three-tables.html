The Three Tables

One picture answers half the confusing questions in Unix I/O at once: why two opens of the same file don't share an offset, why fork does, and why dup is different from both.

We've been sloppy on purpose, saying "the descriptor points at the file." It's more interesting than that — there isn't one indirection, there are three, stacked like a coat check with two separate ledgers behind the counter.

🔑
The three layers, inside out

1. Descriptor table — per process: small integers (0, 1, 2, 3…) → pointers. 2. Open file description table — per open: holds the current offset and the access mode/status flags. 3. Inode (v-node) table — per file on disk: the actual data, size, permissions, owner.

Now the case that surprises people: fork()

Here's the picture that makes fork click. When a process calls fork, the child gets its own copy of the descriptor table — but that copy still points at the same open file descriptions as the parent's. Nothing new is opened; no new offset is created.

This is why a shell pipeline's two ends of a pipe stay in sync, and why two processes writing the same shared, inherited fd don't clobber each other's position.
· 19

And where dup fits

dup(fd) does to a single process exactly what fork does across two: it hands back a new descriptor number that points at the same open file description as the original. Read through either number and both advance together — because underneath, there's only one offset to advance. Chapter 8 is built entirely on this fact.

One picture, three answers

"Why don't two opens share a position?" Different open file descriptions. "Why does a child process keep reading where the parent left off?" Same open file description. "Why does redirecting fd 1 with dup2 not need a new offset?" Same reason — no new open file description gets created.

Sources: W. R. Stevens & S. Rago, Advanced Programming in the UNIX Environment, ch. 3 (the classic three-table figure); POSIX.1-2017 rationale for open, dup, and fork file-descriptor semantics.
