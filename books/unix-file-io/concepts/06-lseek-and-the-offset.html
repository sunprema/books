Concept 06 · Position

lseek() & the Offset

read and write always act at the current offset. lseek is the one call that lets you move it — anywhere, including places that don't exist yet.

So far the offset has only moved itself, one byte at a time, as a side effect of reading and writing. lseek breaks that habit: it repositions the offset directly, without moving a single byte of data.

c · the signature#include <unistd.h>

off_t lseek(int fd, off_t offset, int whence);

whence says what offset is measured from — three named constants, three little clock faces:

whencenew offset =
SEEK_SETexactly offset, counted from byte 0
SEEK_CURcurrent position + offset
SEEK_ENDend of file + offset

🔑
This is what "random access" means

lseek(fd, 0, SEEK_SET) — rewind to the start. lseek(fd, 0, SEEK_CUR) — a trick to read the current offset without moving it (its return value is the new position). lseek(fd, -10, SEEK_END) — jump to 10 bytes before the end, to grab a trailer. None of these touch the file's bytes; they only move where the next read/write will happen.

c · read the last 10 bytes of a filechar tail[10];
lseek(fd, -10, SEEK_END);
read(fd, tail, sizeof tail);

Return value: the new offset from the start of the file, or -1 on error. Get the arithmetic wrong (seek before byte 0) and that's exactly what you get.

The strangest trick: seeking past the end

lseek will happily move the offset beyond the current end of file — that's not an error either. Nothing happens yet. But write a few bytes there, and the file's size jumps to cover the gap, and the gap in between reads back as null bytes ('\0'). This is a hole — sometimes the filesystem doesn't even allocate real disk blocks for it.

A "sparse file." Reading a hole costs a memory copy of zeros; it never touches disk, because there's nothing there to touch.
· 23
⚠️
Not every descriptor can seek

Pipes, sockets, and terminals have no notion of "position" — there's nothing to rewind to, the bytes are gone the moment they're read. Call lseek on one of these and you get -1 with errno == ESPIPE ("illegal seek"). Only regular files (and a few disk-backed special files) support it.

Newer Linux kernels add two more whence values for sparse files specifically: SEEK_HOLE finds the next hole at or after a position, and SEEK_DATA finds the next real data. Tools that copy huge sparse files efficiently (without materializing every zero) use exactly this pair to skip the holes instead of copying them.

🧭
Where we go next

Everything so far has been the raw kernel calls. But you rarely write read/write by hand in C — you call printf, fread, fwrite. Those sit on top of these syscalls, and the gap between them explains a famous mystery: why a crashed program's last printf sometimes never appears.

Sources: Linux man-pages lseek(2) — SEEK_SET/CUR/END, holes read as null bytes, ESPIPE for pipes/sockets/FIFOs/terminals, SEEK_HOLE/SEEK_DATA (Linux 3.1+).
