The Six Engines, Up Close

The architecture from the last chapter wasn't a metaphor. It's six real stages, each with one job, wired into a single pipeline that turns raw events into responsible action.

The best thing about a pipeline is that you can stop worrying about all of it at once. The six engines — Event, Context, Memory, Priority, Reasoning, Execution — hand work to each other in a straight line, and each stage has exactly one responsibility you can reason about in isolation. An event arrives; it gets enriched with context; that context is deepened by memory; the result is ranked by priority; reasoning forms a recommendation; execution makes it real. Read it once, left to right, and you've understood the whole machine.

This chapter walks each engine in turn — what goes in, what comes out, and the one idea that makes it earn its place. None of these are things you build by hand. They are things you declare, which is the whole point we'll land on at the end.

1 · The Event Engine

Intelligence has to start with noticing. The Event Engine observes domain events as they happen and treats each one as an input worth thinking about. These aren't synthetic AI events — they're the ordinary heartbeat of your app:

  • Task Created · Ticket Updated · Customer Escalated
  • Invoice Paid · Meeting Scheduled · Deployment Failed

Each event is a trigger that says "something changed; it might mean something." The Event Engine's job is not to interpret — only to catch the signal and pass it forward so the later stages can decide whether it matters.

2 · The Context Builder

A raw event — ticket_updated(id: 482) — is almost meaningless on its own. The Context Builder turns it into a structured contextual snapshot: a clean, typed picture of the world around that event, assembled from your resources at the moment it fired.

Crucially, the language model never reaches into your database. It receives this snapshot, already gathered and shaped. That single rule is what keeps the whole system inspectable and safe.

elixir · a context snapshot%{
  current_user: "Priya Anand",
  projects: ["Atlas", "Renewals Q3"],
  goals: ["cut churn 15%"],
  deadlines: ["Atlas GA — Jul 12"],
  relationships: ["reports to: D. Okafor"],
  recent_activity: ["closed #471", "merged PR 88"],
  upcoming_meetings: ["Renewal sync — Wed 10:00"]
}
🔑
The model sees context, never SQL

The LLM is handed a finished, structured snapshot — never a database connection and never a query. Context is built for it, not fetched by it. That boundary is what lets your policies and tenancy stay in force around every intelligent feature.

3 · The Memory Engine

Context is the present; memory is everything that makes the present legible. The Memory Engine maintains persistent memory in three tiers, so a recommendation can draw on who you are, what your team is chasing, and what the organization has already decided.

TierWhat it remembersExample
UserPreferences, habits, role, communication style, working hours"Prefers terse briefs before 9am"
TeamCurrent goals, active projects, dependencies"Atlas blocks the Renewals launch"
OrgDecisions, policies, architecture records, historical events"We standardized on Postgres in 2024"

4 · The Priority Engine

Now the system knows what happened, what surrounds it, and what it should remember. The Priority Engine answers the next human question: of everything on the table, what actually matters most right now? It ranks work by weighing several factors at once:

  • Business Impact · Urgency · Deadlines
  • Dependencies · Executive Requests
  • Customer Risk · Historical Behavior

What makes this engine trustworthy is its output shape. It never returns a bare number you have to take on faith. It returns a triple — a score, a human-readable reason, and a confidence — so a ranking can always be questioned and explained:

elixir · what priority produces%{
  priority_score: 0.87,
  reason: "Enterprise customer 12 days from
           renewal hit a blocker",
  confidence: 0.9
}

A score of 0.87 means nothing without the sentence beside it. Together they let a person — or the next engine — act on the ranking and audit it.

5 · The Reasoning Engine

This is where it all converges. The Reasoning Engine consumes four streams at once — the Context snapshot, the three tiers of Memory, the Priority ranking, and your Domain Rules — and produces judgments a colleague might:

  • Recommendations and Daily Briefs
  • Risk Detection and Opportunity Detection
  • Meeting Preparation and Executive Summaries

Everything before this point existed to make this stage's input good. Reasoning is only as sharp as the context, memory, and priorities feeding it — which is exactly why those have their own engines.

· 3

6 · The Execution Engine

A recommendation that never leaves the screen is just advice. The Execution Engine turns reasoning into real actions in the systems you already run:

  • Create Task · Draft Email · Schedule Meeting
  • Update CRM · Notify Team · Generate Report

Some of these are harmless to do automatically. Others touch the outside world, and those should never fire unsupervised. Whether an action runs straight away or waits for a human depends on policy.

⚠️
Gate the actions that touch the world

Drafting an email is reversible; sending it is not. Execution that crosses your app's boundary — mail, CRM writes, deployments — should be gated by an approval policy, not trusted blindly. In Chapter 9 we make that gate explicit as a state machine: proposed → approved → executed.

Six stages, one straight line: each engine refines the last engine's output, and only the final stage touches the world.

That's the whole machine. And here's the part worth remembering: you never solder these six together by hand. You declare them in an intelligence block, and the framework builds the pipeline — observers, snapshots, memory tiers, ranking, prompts, and gated actions — for you.
