Concept 07 · The DSL

A Spark DSL for Intelligence

Six chapters of theory pointed at one artifact: a vocabulary you write inside a domain to declare what your application understands and decides.

Here's the language we've been building toward. Everything so far — declarative intelligence, the meaning gap, Spark's section-and-entity machinery, the six engines — was preparation for a handful of words you'd actually type. This chapter lays them out. None of them are magic; each is an ordinary Spark entity living in an ordinary Spark section, and each maps to a real piece of generated behavior.

Recall the shape of an Ash extension from chapter 3: an Extension bundles Sections, each Section holds Entities, and every Entity has a schema that says what arguments and options it accepts. AshIntelligence is just such an extension. The top-level words — intelligence, connector, brief, memory, priorities, agent, recommendation — are sections and entities. The nested lines inside them are entities with their own little schemas. That's the whole trick.

Let's start at the root. The intelligence block is the one section that anchors the extension; it carries the global settings every other piece reads from.

elixir · the root sectionintelligence do
  provider :openai
  memory :persistent
  priority_engine :default
end

Three single-line entities. provider takes an atom and decides which LLM client gets called — swap :openai for :anthropic and nothing else changes. memory :persistent selects the memory backend (versus :ephemeral), and priority_engine names the ranking strategy. To Spark these are entities whose schema is "exactly one atom, from this allowed set" — so a typo like :opennai is a compile error, not a 3 a.m. runtime surprise.

· 1

Connectors: where context comes from

An intelligence layer is only as good as what it can see. Connectors declare the systems AshIntelligence is allowed to read and act through. Each connector is one entity taking a single atom; you list as many as you need.

elixir · data sourcesconnector :slack
connector :github
connector :google_calendar

Behind each atom sits an adapter that knows how to authenticate and fetch — Slack threads, GitHub PRs, calendar events. From the DSL's point of view they're interchangeable nouns; the framework owns the messy "how."

Briefs: scheduled understanding

A brief is a Spark entity that also opens a block — it has nested entities of its own. This is the section-within-section pattern: the brief carries a name, a schedule, and a list of include lines saying what the digest should contain.

elixir · a daily digestbrief :daily do
  schedule "0 8 * * *"
  include :priorities
  include :meetings
  include :risks
  include :recommendations
end

That schedule string is a cron expression — at 08:00, every day. The framework reads it and registers an AshOban scheduled trigger, so the brief runs itself without you hand-writing a worker. (We build exactly this in the next chapter.) Each include is an entity taking one atom that names an engine's output to fold into the digest.

🧩
Why this isn't magic

Every word here is an ordinary Spark entity in an ordinary Spark section. There's no metaprogramming sleight of hand — the same machinery that powers attributes do … end in a resource powers intelligence do … end here. If you can read one Ash DSL, you can read this one.

· 2

Memory and priorities: the tunable parts

Two sections let you shape how the layer thinks. memory declares what's worth remembering across sessions; priorities declares how competing demands get ranked.

elixir · what to remembermemory do
  remember :user_preferences
  remember :working_style
  remember :recent_projects
  expire_after :90_days
end

Each remember is an entity naming a memory category; expire_after sets a retention window. The priorities block is the interesting one — it's a tiny scoring model expressed as data:

elixir · how to rankpriorities do
  weight :deadline, 0.4
  weight :business_value, 0.3
  weight :customer_risk, 0.2
  weight :executive_request, 0.1
end

Each weight is an entity whose schema is an atom plus a float — a named factor and its share of the score. Spark validates the shape at compile time: forget the float and it won't compile. And because the declaration is plain data, you can do something better still.

📐
A weight-sum verifier

Spark lets you attach a Verifier that runs after the DSL is parsed but before the module compiles. A tiny verifier could sum every weight and emit a warning if they don't add to ~1.0 — turning "these numbers are nonsense" from a silent runtime bug into a friendly compile-time nudge. That's declarative intelligence checking itself.

Every line of DSL is data: a Section holding two :weight entities. Spark parses the words into this tree, and a verifier can inspect it.
· 3

Agents and recommendations: acting

The last two words turn understanding into motion. An agent declares an autonomous worker scoped to a connector; a recommendation names a suggested action and the module that runs it.

elixir · workers and actionsagent :planner
agent :calendar
agent :github
agent :salesforce
agent :reviewer

recommendation :review_pr do
  execute ReviewPullRequest
end

Each agent is one entity taking an atom. The recommendation entity opens a block whose execute line points at a module implementing the work — a normal Elixir module, called when the recommendation is accepted. Notice the division of labor: you've described what can happen; the framework owns when and how.

Put it all together and the vocabulary reads like a description of a thoughtful chief of staff: know my providers, remember my preferences, watch these systems, rank by these factors, brief me each morning, and here are the agents and actions you may take.

elixir · the capstone — a full domaindefmodule MyApp.Intelligence do
  use Ash.Domain, extensions: [AshIntelligence]

  intelligence do
    provider :anthropic
    memory :persistent

    brief :daily do
      include :priorities
      include :meetings
      include :risks
      include :recommendations
    end

    priorities do
      weight :deadline, 0.4
      weight :customer_risk, 0.3
      weight :executive_request, 0.2
      weight :team_blocker, 0.1
    end

    connector :github
    connector :slack
    connector :calendar

    agent :planner
    agent :reviewer
  end
end

Forty lines, and there's not a prompt, a job queue, or an HTTP call in sight. The weights sum to 1.0; the verifier stays quiet. This is the destination — and notice it's the same domain you'd already write in Ash, with one extension added.

🔑
You write WHAT; the framework writes HOW

You declare the provider, what to remember, how to weight, what to brief on, which agents may act. The framework generates the prompts, the scheduled jobs, the context snapshots, the LLM calls, and the stored results. Intent in; orchestration out — exactly the Ash bargain, now applied to intelligence.

But does the engine behind these words actually exist? Mostly — yes. Almost every "HOW" above is already shipping in real Elixir packages today. In the next chapter we wire them together and build this for real.
