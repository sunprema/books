How You'd Actually Build It Today

AshIntelligence-the-named-package is a proposal — but you don't have to wait for it. About 80% of it can be assembled right now from real, shipping libraries.

Good news — you don't have to wait for AshIntelligence to ship. Here's the parts bin. Almost everything the proposal promises already exists in the Ash ecosystem as separate, production packages; what's missing is mostly the unifying DSL that snaps them together. This chapter is the credibility chapter: every API below is real, and you could mix deps.get them this afternoon.

The trick to seeing it is to stop thinking of "AI" as one monolithic feature and instead break it into the capabilities the PRD actually asked for: structured reasoning, tool use, memory, scheduling, and lifecycle. Each of those already has a home.

The headline

You can build ~80% of AshIntelligence today by composing ash_ai + ash_oban + ash_state_machine behind a Spark DSL. The missing piece is mostly the unifying DSL and the memory/priority engines.

· 1

Reasoning, declared: prompt-backed actions

The cornerstone is ash_ai (the ash-project/ash_ai repo) — the official LLM toolbox for Ash. It gives you five things: prompt-backed actions with structured outputs, tool calling, vectorization/embeddings, a chat generator (mix ash_ai.gen.chat), and an MCP server. Under the hood it calls models through req_llm.

A "prompt-backed action" is the headline trick: an ordinary Ash action whose run is an LLM call, but whose return type is a real Ash type with constraints. The framework forces the model's reply to fit. Here's the real syntax:

elixir · ash_ai · prompt-backed actionaction :analyze_sentiment, :atom do
  constraints one_of: [:positive, :negative]
  description "Analyzes the sentiment of text."

  argument :text, :string do
    allow_nil? false
  end

  run prompt("openai:gpt-4o", tools: true)
end

The prompt/2 helper takes more than a model string. tools: can be false, true, or a list of tool-name atoms; max_iterations: bounds the tool loop; prompt: supplies a custom EEx template; req_llm: injects a client or mock for tests; and verbose?: true traces the exchange. Swap the first string to "anthropic:claude-..." and you've changed providers — nothing else moves.

If you want pure Ecto-schema structured outputs without the action wrapper, InstructorLite does exactly that (tested against OpenAI, Anthropic, and Gemini). But inside Ash, the action form is what makes intelligence declarative: it's just another action your policies and audits already understand.

🔌
Provider-agnostic, for free

Because ash_ai calls models via req_llm, the provider is a string. ReqLLM ships six providers — OpenAI, Anthropic, Google, Groq, xAI, OpenRouter — behind one consistent API. Switching is a config/string change, not a rewrite.

· 2

Tools, memory, and the daily brief

Reasoning is only half of intelligence; the other half is doing. In ash_ai you expose existing Ash actions to the model as tools — at the domain level — and they run under the same authorization as everything else:

elixir · ash_ai · exposing actions as toolstools do
  tool :read_posts, MyApp.Blog.Post, :read
  tool :create_post, MyApp.Blog.Post, :create
end

Only public attributes (public?: true) are usable by the model for filtering and sorting; use load: to pull in relationships. And when you run the loop yourself, you pass an actor and tenant, so every tool call is checked:

elixir · ash_ai · authorized tool loopAshAi.ToolLoop.run(messages,
  actor: current_user,
  tenant: org,
  on_tool_start: fn _ -> :ok end,
  on_tool_end:   fn _ -> :ok end
)

Memory is the same story. ash_ai's vectorize block embeds records into pgvector so you can recall them semantically:

elixir · ash_ai · vectorizationvectorize do
  full_text do
    text(fn record ->
      "Name: #{record.name}\nBio: #{record.biography}"
    end)
    used_attributes [:name, :biography]
  end

  strategy :after_action   # or :ash_oban (async), or :manual
  embedding_model MyApp.OpenAiEmbeddingModel
end

The embedding model is a small module — use AshAi.EmbeddingModel with def dimensions(_), do: 3072 and a def generate(texts, _opts) returning {:ok, [vectors]}. Search then filters inside a read action with vector_cosine_distance(full_text_vector, ^search_vector). That's semantic recall — the "memory" of Concept 04 — in real, running code.

· 3

Scheduling & the codegen seam

The brief :daily from Concept 01 is just a cron job. ash_oban — Ash's binding over the Postgres-backed Oban queue — gives you scheduled triggers that fire an action on a cron (and, conveniently, it's also the :ash_oban strategy for async vectorization above):

elixir · ash_oban · scheduled trigger (plausible)oban do
  triggers do
    trigger :daily_brief do
      action :daily_brief
      scheduler_cron "0 7 * * *"   # 07:00 every day
      queue :intelligence
    end
  end
end

So how would the single intelligence do … end block of the proposal produce all of this? Through Spark transformers. Spark is the DSL engine under every Ash extension. At compile time, a transformer reads your DSL as data and injects modules — exactly the "Generated Components" the PRD lists: Oban workers, prompt templates, context builders, LiveView components, telemetry, audit logs. Conceptually, for each brief you declare, a transformer emits one Oban worker:

elixir · PSEUDOCODE · Spark transformer sketchdefmodule AshIntelligence.Transformers.BuildBriefs do
  use Spark.Dsl.Transformer

  def transform(dsl_state) do
    briefs = Transformer.get_entities(dsl_state, [:intelligence])

    # for each declared brief, inject an Oban worker
    Enum.each(briefs, fn brief ->
      inject_oban_worker!(brief.name, brief.schedule)
      register_scheduled_trigger!(brief.name)
    end)

    {:ok, dsl_state}   # hand the DSL back, enriched
  end
end

That returned {:ok, dsl_state} is the whole magic: the transformer hands the DSL back, enriched with code you never wrote. ash_state_machine (Concept 09) handles the recommendation lifecycle the same declarative way. Compose these five packages behind one Spark DSL and you have most of AshIntelligence — running, today.

⚠️
Keep the model on a leash

Expose intelligence only through authorized actions and tools — never hand an LLM raw repo or database access. The actor:/tenant: you pass to the tool loop is what keeps the model inside your policies. We'll harden this in Concept 10.

The parts bin: AshIntelligence isn't one library to invent — it's a composition of packages that already ship.
