Concept 10 · Safety

Safe by Construction

An intelligence layer is dangerous precisely because it's powerful — so the whole game is to make it inherit your app's guarantees rather than route around them.

There are two versions of this layer, and they look identical in a demo. The scary one reads every row in your database and acts with no leash: it can surface a salary it was never meant to see, write to a record the user could never touch, and leak one tenant's data into another's brief. The safe one can only ever do what the current user could do by hand — no more, no less. The entire discipline of this chapter is making sure you build the second one without noticing you had a choice.

The good news is that you already wrote the hard part. Your Ash app has policies, authorizers, tenant scoping, and data classification. The intelligence layer's job is not to re-implement any of that — it's to stand behind it. Every recommendation must respect your policies, your authorization rules, your tenant boundaries, and your data classification. The engine must never become a side door.

Why this is structurally true, not just a promise

Recall from the tools chapter: the model never touches your database. It acts only through ash_ai tools, and its context is built from authorized reads. Crucially, you pass actor: and tenant: into the tool loop, and those flow into every action the model can invoke.

Ash policies and authorizers run on actions. So when the model calls a tool, that tool runs an action as the actor, under the same authorizer that guards your UI. The model physically cannot read rows the actor isn't allowed to see, or write what they can't write. Authorization is inherited, not duplicated.

🔑
Authorization is inherited

You don't write new permission logic for the AI. It rides on the permissions you already have. The same policies do … end that protects your controllers protects every tool call — because both go through the same actions.

· 1

The shape of the gate

Picture a request heading toward the LLM. Before it ever reaches the Context Builder, it passes through the policy/authorizer gate and the tenant scope. Unauthorized rows are filtered out — the model only ever receives a snapshot of what this actor, in this tenant, was allowed to load.

The model only sees what the actor is allowed to see.

Because the gate sits in front of the Context Builder, there is no snapshot of forbidden data sitting around for a clever prompt to coax out. The data was never loaded.

⛔
Never hand it the keys

Never give the model raw database or repo access, and never a blanket "run any SQL" tool. Always go through authorized actions and tools. That one rule prevents the classic data-leak — because a free-SQL tool runs as no actor, under no policy, in no tenant.

· 2

What the code actually looks like

The policy lives on the resource, exactly where it already lives. Nothing about it is "AI-aware" — that's the point.

elixir · policies guard the action, not the callerdefmodule MyApp.CRM.Deal do
  use Ash.Resource,
    authorizers: [Ash.Policy.Authorizer],
    extensions: [AshAi]

  policies do
    policy action_type(:read) do
      authorize_if relates_to_actor_via(:owner)
      authorize_if actor_attribute_equals(:role, :admin)
    end

    policy action_type([:create, :update]) do
      authorize_if relates_to_actor_via(:owner)
    end
  end
end

And the tool loop simply passes the current actor and tenant through. Every action the model triggers is now authorized as that person, in that tenant:

elixir · the model acts as the actor# the same actor/tenant your controller would use
MyApp.Intelligence.run_brief(prompt,
  actor: conn.assigns.current_user,
  tenant: conn.assigns.current_org
)

# under the hood, every ash_ai tool call becomes:
Deal
|> Ash.Query.for_read(:read)
|> Ash.read!(actor: actor, tenant: tenant)
# → policies + tenant scope applied. No bypass exists.

Tenancy in Ash is first-class — attribute-based or context-based — and the intelligence layer runs inside that same scope. There is no separate "AI tenant." There is just the tenant the request arrived in.

Data classification

You can mark attributes public or private. Private attributes aren't exposed to tools unless you explicitly load them. So even within rows the actor can see, a salary or a token stays out of the model's context until you make a deliberate choice to include it.

🔀
Provider-agnostic is a safety feature

Supporting multiple providers (OpenAI, Anthropic, Gemini, Ollama) via req_llm isn't just convenience. It lets you route by cost and latency — and fall back to a local model (Ollama) for sensitive data that should never leave your network. Swapping providers changes config, not business logic.

· 3

If you can't see it, you can't trust it

Safety isn't only about what the model can't do; it's about being able to watch what it actually does. Elixir's standard instrumentation library is :telemetry, and both Ash and Oban already emit telemetry events. You attach handlers and turn the intelligence layer from a black box into a dashboard.

Provider abstraction makes provider and cost a real, measurable, swappable dimension — you can compare two models on the same workload instead of guessing. Here are the signals worth wiring up:

SignalWhy you watch it
Prompt latencyThe user feels every second. Spikes flag a slow provider, a bloated context, or a retry storm.
Provider costSpend scales with usage and can surprise you. Track it per feature so a runaway loop shows up as dollars, not a postmortem.
Token usageThe early-warning version of cost and latency. Rising tokens mean your context is growing — often a sign of an unbounded include.
Recommendation qualityThe whole point. Without a quality signal you can't tell an upgrade from a regression when you swap models or prompts.
User feedbackThumbs up/down and overrides are ground truth. They're the labels that tell you whether quality is real or imagined.
Execution successDid the action the model proposed actually complete? Silent failures erode trust fastest.
Agent healthQueue depth, error rate, restarts. The operational pulse — is the layer up, behind, or thrashing?

Wire these into the same observability stack you already run. Because Ash and Oban emit the events, you're attaching handlers, not inventing a metrics pipeline.

🔑
The whole chapter in one line

The intelligence layer is safe by construction when it can only do what the actor could do by hand, in their tenant, through authorized actions — and when every move it makes is observable. Inherit your guarantees; don't reinvent them.

With the leash firmly attached and the dashboards lit, we've earned the fun part. Next: the exciting things this layer lets you build once you trust it.
