Concept 11 · The Payoff

Exciting Use Cases

Ten things this layer can do on Monday morning — each one a small declaration, all of them the same six engines underneath.

Here is the part that should feel almost unfair. Everything you are about to read — the morning brief, the priority radar, the risk alerts, the PR reviewer, the escalation handler — is produced by the same six engines you have already met: context, memory, reasoning, agents, connectors, and the live surfaces. The magic isn't any single feature. It's that you don't build ten systems; you declare ten differences against one. Change the event, change the weights, change the action — reuse everything else.

So read these as recipes, not products. Each is a real moment in someone's workday, what AshIntelligence does in that moment, and — for a few — the tiny bit of DSL that expresses it.

1 · The Daily Brief

8am, before standup. One screen: top priorities, today's meetings, emerging risks, recommended actions. Generated overnight by a brief :daily on a cron, waiting when you sit down.

2 · Priority Radar

Your work, ranked by score — each item carrying a human-readable reason and a confidence. No more staring at a flat backlog guessing what to touch first.

3 · Risk Detection

An enterprise account twelve days from renewal just filed an angry ticket. A deploy failed three times tonight. A deadline is quietly slipping. The layer surfaces these before they blow up.

4 · Opportunity Detection

A stalled deal lights up with buying signals. An upsell window opens. A long-quiet account suddenly gets active. The same scan that finds risk finds upside.

Notice the symmetry already: risk and opportunity are the same reasoning pass with different weights. That is the whole trick, and it repeats.

· 1

From watching to helping

The first four use cases only read. The next ones start to do — and that is exactly where the connectors and agents earn their keep.

5 · Meeting Preparation

Fifteen minutes before a calendar event, a prep pack assembles itself: who's attending, your history with them, open items, last decisions. Pulled via the google_calendar connector.

6 · PR Review Agent

A pull request lands. The :reviewer agent reads the diff through the github connector, drafts a review, and — after you approve — posts it. A senior eye that never sleeps.

7 · Customer Escalation

A ticket gets escalated. The layer drafts a response, pings the owner in Slack, and creates a follow-up task — every action gated behind an approval click.

8 · Executive Summary

A weekly roll-up of team memory and activity, distilled into a crisp brief for leadership. The same brief machinery as #1, aimed up the org chart.

9 · Sales / CRM Automation

A call ends. The :planner and :salesforce agents update the CRM, log the outcome, and draft the follow-up email for you to send.

10 · The Live Surfaces

Memory Viewer, Agent Console, Activity Timeline — LiveView components so a human can see what the layer believes and steer what it's doing, in real time.

📋
Adopt in this order

Start with the read-only use cases — briefs, dashboards, the risk radar. They're genuinely valuable and completely safe: nothing happens to the outside world. Once you trust the reasoning, graduate to acting — and even then, behind an approval gate.

· 2

One worked example, twice

Let's make two of these concrete with the DSL from Concept 7. First, the PR-review recommendation — an event-triggered piece of reasoning that executes a real action through an agent and a connector:

elixir · recommendation → actionuse AshIntelligence

connector :github
agent :reviewer

recommendation :review_pr do
  on :pull_request_opened
  agent :reviewer
  execute ReviewPullRequest
  approve :before_posting
end

Read it as English: "When a PR opens, hand it to the reviewer agent, have it run ReviewPullRequest, and wait for my approval before anything is posted." The diff-fetch, the LLM call, the GitHub write — all generated. You declared the difference: this event, this agent, this action.

And the daily brief from #1 — a scheduled read-only roll-up — is just as small:

elixir · the morning briefbrief :daily do
  cron "0 7 * * *"
  include :priorities
  include :meetings
  include :risks
  include :recommendations
end

Different shape, identical idea. One block acts and one block reports, but both are a description handed to the same runtime. The executive summary (#8) is this same brief with a weekly cron and a team-scoped memory query. You are not learning ten APIs; you are filling in ten sets of blanks.

🔑
It's always the same pipeline

Every use case here is: declare the event, declare the priority weights, declare the action — and reuse context, memory, reasoning, approval, and the live surfaces unchanged. Ten features, one pipeline, parameterized.

Why Elixir & OTP fit this perfectly

These are long-running, concurrent, per-user, event-driven workloads — thousands of small supervised processes, each watching its own stream, surviving crashes, reacting in real time. That is precisely the shape the BEAM was built for. The runtime you'd otherwise hand-build is the one OTP already gives you.

Where this goes

If a single layer can brief you, rank your work, catch your risks, and act with your permission — all from declarations — what does an application become? That's Concept 12: the long-term vision.
