The Intelligence Layer: Architecture

Before we open up any single part, here is the whole machine on one page — eight stages, one direction of flow, and one rule that holds it all together.

Every intelligent feature in this framework rides the same assembly line. A thing happens in your domain, and it travels down through a fixed sequence of stages, each one adding something, until it arrives at a person as a recommendation they can act on. Learn this one pipeline and you've learned how all of AshIntelligence works; the rest of the book is just zooming in on individual stations.

Follow a single event the whole way down. A support ticket escalates — a status field flips in one of your Ash resources. That's the spark.

  • Event Collector. The extension notices the change and captures it as a typed event — not a vague webhook, but "ticket #4012 escalated."
  • Context Builder. It assembles a structured snapshot around that event: who the customer is, what the ticket says, related records, recent activity.
  • Memory Engine. It merges in persistent memory — this user's preferences, the team's standing goals — so the moment has history.
  • Priority Engine. It ranks the event against everything else in flight, so the model spends attention where it matters.
  • Reasoning Engine. The LLM turns context + memory + priority + your domain rules into recommendations, each with its reasoning attached.
  • Execution Engine. Approved recommendations become real Ash actions — a reply drafted, an owner reassigned, a follow-up scheduled.
  • User Interfaces. LiveView surfaces it all: the daily brief, dashboards, the agent console.
🔑
The one rule

The LLM never touches your database. It only ever sees a structured snapshot the framework assembled from authorized actions. Sandbox the model behind context you built, and a whole class of safety and accuracy problems simply cannot occur.

Two directions of flow

Read the diagram as a loop with two halves. Events flow down: raw domain changes descend through collection, context, memory, and priority until they reach the model as a single well-formed question. Actions and views flow out: the model's answers leave through the Execution Engine as real Ash actions, and through the interfaces as briefs and dashboards a person reads. Down is "help me understand"; out is "here's what I did and what you should know."

⏳
It runs per-event, mostly async

This pipeline fires once per event, and most of it happens off the request path. Later chapters wire it to Oban jobs and supervised OTP processes — so an escalating ticket never blocks a web request, and a slow LLM call never takes the system down.

That's the map. Eight stages, one rule, two directions — now let's open up each engine and see what happens inside.
