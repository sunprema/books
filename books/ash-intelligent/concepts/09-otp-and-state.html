Concept 09 · Runtime

OTP Architecture & Recommendation Lifecycle

Intelligence is long-lived and personal. That single fact decides how it should run — and how its recommendations should move through the world.

A web request is a mayfly: it is born, it does one small thing, and it dies. That model is perfect for a stateless API and exactly wrong for intelligence. Your understanding of a user is not a thing you reassemble from scratch on every click. It is long-lived and per-person — it accumulates, it remembers yesterday, it forms opinions over weeks. Something that lives like that wants a process of its own, not a request handler that vanishes the moment it answers.

Elixir, by happy coincidence, is built almost entirely out of long-lived processes. OTP — the runtime and patterns underneath the language — gives us supervision trees: a supervisor starts child processes and restarts them when they fall over. The cultural motto is "let it crash." Don't defensively wrap every line in error handling; instead isolate failures so that when one thing breaks, a supervisor quietly restarts just that thing and the rest of the system never notices.

That is the property we want for intelligence. One user's reasoning going sideways — a bad model reply, a parsing error, a runaway computation — should be a contained event, not a global outage. So we give each user their own little tree of processes.

🔑
Why a process per user

Per-user process isolation buys two things at once: fault tolerance (a crash restarts one subtree, not the system) and a natural home for that user's live memory — context and state simply live in the process, in RAM, ready, instead of being rebuilt from the database on every request.

· 1

The supervision tree

Picture it top-down. A Company Supervisor owns the tenant. Under it, one User Supervisor per active user. And under each user supervisor sits that user's private intelligence: five processes — Memory, Context, Priority, Reasoning, and Execution — each a small server holding its own state.

One subtree per user. The dashed frames are walls: a fault inside User A's reasoning restarts only A's branch.

Because each user's memory and context live in their own server's state, one user's reasoning literally cannot reach into another's. Isolation isn't a policy you enforce — it's the shape of the runtime.

· 2

Spawning a user's tree on demand

You don't want to start a thousand supervisors at boot for users who may never log in. The idiomatic answer is a DynamicSupervisor — it spawns child trees on demand — paired with a Registry so you can find a running process again by {company_id, user_id} instead of hoarding raw PIDs.

elixir · per-user intelligence subtreedefmodule AshIntelligence.UserSupervisor do
  # one supervised tree per user: Memory, Context,
  # Priority, Reasoning, Execution
  use Supervisor

  def start_link(%{company_id: c, user_id: u} = arg) do
    name = {:via, Registry, {AshIntelligence.Registry, {c, u}}}
    Supervisor.start_link(__MODULE__, arg, name: name)
  end

  def init(arg) do
    children = [
      {AshIntelligence.Memory, arg},
      {AshIntelligence.Context, arg},
      {AshIntelligence.Priority, arg},
      {AshIntelligence.Reasoning, arg},
      {AshIntelligence.Execution, arg}
    ]

    Supervisor.init(children, strategy: :one_for_one)
  end
end

# start (or look up) a user's intelligence on demand
defmodule AshIntelligence.Users do
  def ensure_started(company_id, user_id) do
    spec = {AshIntelligence.UserSupervisor,
            %{company_id: company_id, user_id: user_id}}

    DynamicSupervisor.start_child(AshIntelligence.Sup, spec)
  end
end

With :one_for_one, if a user's Reasoning process crashes, the supervisor restarts that one child and leaves Memory and Context untouched — the user keeps their accumulated state and loses only the work in flight.

The recommendation lifecycle

Processes are where intelligence lives. The other half of this chapter is what it produces and how that artifact moves toward acting on the world. A recommendation isn't a one-shot value; it has a life. It is detected, drafted, held for a human, approved, executed, verified, and finally filed away. That is a state machine.

Seven states, left to right. The amber step — Awaiting Approval → Approved — is the human gate: nothing side-effecting happens before it.

Ash has a real extension for exactly this: ash_state_machine. You declare the states and the legal transitions on the resource itself, and each transition becomes an action — which means it inherits everything actions already give you.

elixir · ash_state_machine resourcedefmodule AshIntelligence.Recommendation do
  use Ash.Resource,
    extensions: [AshStateMachine]

  state_machine do
    initial_states [:detected]
    default_initial_state :detected

    transitions do
      transition :draft,            from: :detected,           to: :drafted
      transition :request_approval, from: :drafted,            to: :awaiting_approval
      transition :approve,          from: :awaiting_approval,  to: :approved
      transition :execute,          from: :approved,           to: :executed
      transition :verify,           from: :executed,           to: :verified
      transition :archive,          from: :verified,           to: :archived
    end
  end
end
⚠️
The gate is not decoration

Awaiting Approval is the moment a human says yes before anything touches the outside world. Any side-effecting transition — execute above all — should require it by policy, so the machine cannot legally act on a recommendation no person ever approved.

📐
Transitions are just actions

Because each transition is an Ash action, it inherits your policies and emits telemetry for free. Approval rules, tenancy, and audit logging aren't bolted onto the lifecycle — they ride along with it. We pull on exactly that thread next.

Put the two halves together and the architecture reads cleanly: per-user processes hold the living intelligence, and a state machine governs the artifacts that intelligence wants to act on — with a human standing at the one gate that matters. Chapter 10 makes those guardrails explicit: the policies and tenancy that keep all of this safe.
