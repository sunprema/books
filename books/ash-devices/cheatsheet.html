The whole philosophy on one page.

1 · The Mental Model

Plumbing (avoid)Domain (prefer)
GPIO.write(17,1)Door.unlock()
mqtt publishRobot.move(:bay_3)
write_registerIncubator.set_temperature(37)

Hardware is an implementation detail. Resources are the product.

2 · The Layers

Application
   ↓
Resource   # the API
   ↓
Adapter    # the contract
   ↓
Hardware | Simulator

Apps never depend on a specific device. The adapter is the only seam.

3 · Resource Skeleton

use Ash.Resource,
  domain: Lab,
  data_layer: Ash.DataLayer.Ets

attributes do
  uuid_primary_key :id
  attribute :name, :string
  attribute :target_celsius, :float
  attribute :reported_celsius, :float
end

actions do
  defaults [:read, :create, :update]
  update :set_temperature do
    accept [:target_celsius]
  end
end

target vs reported = the twin.

4 · Capability proposed

device do
  capability :motion
  capability :vision
end

Reusable bundles: actions + events + sim + adapter contract. Built on real Spark extensions.

5 · Adapter Contract

@behaviour Door.Adapter

@callback unlock(config) ::
  :ok | {:error, term}

Swap impls by config. Real protocol libs: Circuits.GPIO/I2C/SPI/UART Tortoise311 (MQTT) modbux opex62541 (OPC-UA) CAN

6 · Circuits.GPIO 2.x

{:ok, g} =
  Circuits.GPIO.open(
    {"gpiochip0", 17},
    :output)
Circuits.GPIO.write(g, 1)

2.x uses gpio_specs (tuple / label), not bare pin ints.

7 · State Machine AshStateMachine

state_machine do
  initial_states [:locked]
  default_initial_state :locked
  transitions do
    transition :unlock,
      from: :locked, to: :closed
  end
end

# inside the action:
change transition_state(:closed)

8 · Events

Descriptive names: DoorOpened, RobotMoved, TemperatureChanged — never generic "updated".

pub_sub do
  module LabWeb.Endpoint
  publish :unlock, ["door", :id]
end
  • Ash notifiers — post-commit callback
  • Ash.Notifier.PubSub — fan-out
  • AshEvents — durable log + replay

9 · Digital Twin

Two values per device: desired vs reported. A reconcile loop drives actual → desired. The durable desired state survives disconnection.

Same shape as AWS Device Shadows / k8s reconciliation.

10 · Simulation-First

The Simulator is the default runtime — same contract as hardware.

Simulator → Resource →
  action → assert

Prefer sim over mocks. If it can't be simulated, it can't be in the framework.

11 · Good vs Bad

Good

Robot · Incubator · Experiment

{:error,:offline}

Bad

HardwareManager · DeviceHelper · CommonUtils

{:error,"Something went wrong"}

12 · The 10-Question Checklist

  1. Belongs in a Resource?
  2. Is it a Capability?
  3. Needs a new Adapter?
  4. Is it simulatable?
  5. Is state explicit?
  6. Does it emit events?
  7. Testable without hardware?
  8. Is the API declarative?
  9. Is it composable?
  10. Works with different hardware?

Multiple “No” → reconsider the design.
