The Craft: APIs, Names & Errors

A framework is only as humane as the code people write against it. This chapter is the house style — the small, stubborn rules that keep AshDevices readable by the people who actually live with the machines.

Here is a test I want you to carry around in your pocket for the rest of this book. Write a line of your API. Now imagine the person who owns the problem — not a programmer, but the scientist hunched over the microscope, or the line engineer with grease on her sleeves — standing behind you. Read the line out loud to her. Does she nod? Or does she squint and ask, "what's a GPIO?" That nod, or that squint, is the whole of API design. Everything else is decoration.

APIs Read Like English

Public APIs should read like English. They should describe what the user intends, not how the silicon happens to do it. A good call is a sentence; a bad call is a confession that you leaked your wiring diagram into the namespace.

elixir · intent, not mechanism# Reads like a sentence a human would say out loud:
Robot.move(robot, :loading_station)
Microscope.capture(scope)
Door.unlock(door)

# Avoid: the implementation bleeding through the front door
Robot.write_register(robot, 0x12, 3)
Door.toggle_gpio(door, 17, 1)
🔑
The aloud test

The test of an API is whether you can read it aloud to the person who owns the problem — a scientist, a line engineer — and they nod. If you have to first explain what a register is, the name is wrong, not the listener.

Notice what each good call hides: Robot.move/2 might publish MQTT, drive a Modbus servo, or step a simulator — the caller neither knows nor cares. The verb is the contract. The wire is an accident of this particular Tuesday.

Names Are Real-World Concepts

Names should represent real-world concepts — the things a person in the room would point at. A robot is a Robot. An incubator is an Incubator. The moment you reach for Manager, Helper, or Util, stop: those are the words we use when we've stopped understanding what the thing actually is.

Good · real things

Names you could point to on a bench:

  • Robot
  • Incubator
  • Experiment
  • Camera

Each one is a noun a scientist already owns.

Avoid · fog

Names that point at nothing:

  • HardwareManager
  • DeviceHelper
  • CommonUtils
  • GenericController

What does a HardwareManager manage? Everything. So, nothing.

🪒
The naming razor

"If a name cannot be explained to a scientist or automation engineer, reconsider it."

Why do Manager, Helper, and Util names rot? Because they're a coat-closet. The first time you can't decide where a function lives, you toss it in Utils — and now Utils is the place where meaning goes to die, a junk drawer that grows forever and explains nothing. A real name is a fence: it tells you what belongs inside and, more usefully, what does not. Incubator will never tempt you to hide a date-formatting function inside it. CommonUtils begs you to.

So the discipline is almost embarrassingly simple: name the device, name the capability, name the event — and when no good name comes, treat that as a signal that the concept isn't clear yet, not that you need a vaguer word.

Errors You Can Act On

Never return ambiguous errors. An error is not a place to be polite or coy — it is a message from the machine to a tired human at 3 a.m., and that human needs to know two things: what is wrong and what to do about it. "Something went wrong" tells her neither. It just confirms that the machine knows a secret and won't share.

Prefer · domain error

elixir{:error, :offline}

Tells you exactly which world fell over — and you can branch on it.

Over · a shrug

elixir{:error, "Something went wrong"}

A string you can log but never act on. The pager goes off; nobody knows why.

Domain-specific errors are preferred because they're data, not prose. Code can match on an atom; it cannot match on an apology. Here is the vocabulary of a system that respects the human on call:

Domain errorWhat it tells the human
:offlineThe device isn't answering — check power/link, not your code.
:limit_exceededA safe bound was hit; back off, don't push harder.
:unsafe_stateRefused on purpose — the machine protected itself.
:door_obstructedSomething physical is in the way. Go look.
:calibration_requiredReadings are untrustworthy until you recalibrate.
🧩
Composes with Ash

You're not inventing this alone. Ash already has structured errors — typed, class-tagged, aggregatable. Your domain atoms ride inside that machinery rather than fighting it: an :offline becomes a proper invalid-action error that the framework can collect, render, and return over any API without losing its meaning.

Modules, Extension Points & Restraint

Prefer small modules. A module should be a thing you can hold in your head whole. The instant a file becomes a place where unrelated functions are parked because they had nowhere else to go, you've grown a HardwareManager in disguise. The layout below is the whole framework, and every name is a real concept:

text · layoutAshDevices/
  Resource/      # the device as a declarative resource
  Capability/    # what a device can be asked to do
  Adapter/       # the bridge to a real protocol
  Simulator/     # the faithful fake
  Runtime/       # processes that keep devices alive
  Event/         # things the machine reports
  StateMachine/  # lifecycles & transitions
  Telemetry/     # measurements & signals
  Studio/        # the live control surface

Prefer extension points over conditionals

When a new kind of device shows up, you should be able to add it without surgery on the core. That means a contract others implement and register — not an ever-growing if that the framework's maintainers must keep editing forever.

Good · a contract

elixirdefmodule MyMqttAdapter do
  @behaviour AshDevices.Adapter
  # implement, then register
end

Behaviour → Plugin → Registration. New adapters need minimal framework changes.

Avoid · a growing fork

elixirif adapter == :mqtt do
  …
else if adapter == :gpio do
  …  # :nerves next? edit core again
end

Every new device reopens the same file. The core can never close.

This is exactly the seam the adapters chapter opened, and it's what Spark extensions are for: register, don't branch.

⚖️
Three rules in the margin

Performance: Correctness > Simplicity > Performance. Make it right, make it clear, and only then make it fast — simulation correctness matters more than raw speed, and premature optimization is just guessing with extra steps.

Dependencies: Before adding one, ask — can OTP do this? Does Ash already provide it? Is it actively maintained? Does it fit the philosophy? Prefer small dependency trees.

Documentation: Every public module explains what problem it solves, why it exists, when to use it, and how it talks to the framework — and examples are required, not optional.

Where we're headed

Good names and English-shaped APIs make the system readable in the code — when you're sitting still, reading. But a machine's truest test is while it's running. Next, the Studio: the live surface that makes the whole cyber-physical system readable while it actually moves.
