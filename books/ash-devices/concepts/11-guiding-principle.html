Concept 11 · Closing

The Decision Checklist

Every chapter handed you one principle. Here, at the end, we squeeze all of them into something small enough to carry in your head — a list you run before you write a single line.

We've come a long way together, you and I. We started with a cheeky question — what if a robot could be declarative? — and we kept pulling on that thread until a whole architecture fell out: resources and capabilities, adapters and simulators, explicit state, events, tests that never touch a wire. That's a lot of ideas. And the trouble with a lot of ideas is that, in the heat of building a feature, you forget them. So let me give you the trick I use myself.

Before you write a line, ask the machine ten questions. Out loud, if you like — I won't tell anyone. Each one is just a chapter of this book, folded down small. If the answers come back mostly yes, you're building with the grain of the framework. If they come back no, the framework is trying to tell you something, and it's usually right.

  1. Does this belong in a Resource?if it has state, it's a resource
  2. Can this be expressed as a Capability?verbs are capabilities, not code
  3. Does this require a new Adapter?the wire lives behind the adapter
  4. Can it be simulated?the simulator is the default runtime
  5. Is state explicit?no hidden machine inside a driver
  6. Does it emit events?the world announces what it did
  7. Is it testable without hardware?CI never plugs in a cable
  8. Is the API declarative?intent, never a pin number
  9. Is the implementation composable?small parts that snap together
  10. Would this still work with different hardware?the adapter makes hardware swappable
✓
The rule

If multiple answers are "No," reconsider the design. One no is a tradeoff; a handful of them is the framework telling you you've wandered back into writing wires.

Future-proofing: keep it domain-agnostic

Now here's the part I find genuinely beautiful, and I want you to notice it. Look back at those ten questions. Read them again. Not one of them mentions a door, a robot, a microscope, or a greenhouse. That's not an accident — that's the whole game.

Assume the framework will eventually be asked to serve places we haven't even drawn yet:

  • laboratories — pipettes, incubators, microscopes
  • robotics — arms, grippers, mobile bases
  • factories — conveyors, welders, presses
  • agriculture — vents, pumps, soil sensors
  • smart buildings — locks, HVAC, lighting
  • autonomous vehicles — drives, lidar, brakes
  • warehouses — shuttles, scanners, sorters

So avoid assumptions specific to one domain. The core abstractions should remain domain-agnostic. And the payoff is exactly this: the same ten questions hold whether you're moving a welding robot or nudging a greenhouse vent shut against the cold. The checklist doesn't know the difference — and that is the property that lets one framework quietly serve every domain we paraded through the use-cases chapter.

The guiding principle

If you forget everything else — the events, the adapters, the testing — keep this one sentence. It's the gravity the whole book orbits.

🔑
The guiding principle

Every line of code should reinforce the central idea of AshDevices: Developers should build physical systems using the same declarative, composable, and testable principles they already use to build modern Ash applications. When in doubt, prefer the solution that makes hardware feel more like software.

Think about what we're really after here. We want a robot to be as ordinary to program as a database row — no ceremony, no priesthood of cables, just a description of what's true and a machine that makes it so. The gap between software and the physical world has always felt like a law of nature. It isn't. It's a choice we keep making, one GPIO.write at a time, and a framework like this is a bet that we don't have to keep making it.

That's the send-off, my friend. Go build something that moves. And when you want the whole book on one page — the ten questions, the principle, the shapes of the code — turn to the cheatsheet. It's all of this, distilled to something you can pin above your desk.
