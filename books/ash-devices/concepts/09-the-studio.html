Concept 09 · Tooling

The Studio: Seeing the Runtime

A dashboard tells you a number. An environment lets you reach in, turn the knob, step the machine, and replay yesterday. The Studio should be the second kind of thing.

Let me tell you the mistake almost everybody makes when they build a tool for a running system. They build a dashboard. A wall of dials and sparklines and a big green "OK" light. And it is genuinely useful — for about a minute — because a dashboard answers exactly one kind of question: "what is the number right now?" Temperature is 37.2. Door is closed. Robot is idle. Lovely. But the moment you actually want to do something — nudge the robot, force the door open to test the interlock, rewind to the instant the incubator spiked — the dashboard just stares back at you. It has no knobs. It was never meant to.

So here is the design rule we are going to insist on, and it changes everything downstream:

🔑
The Studio is an environment

The Studio should be a development environment, not merely a dashboard. It is a workbench for cyber-physical systems — a place where you reach into the running machine, turn a knob, step it forward, and replay what already happened.

Once you commit to "environment, not dashboard," you immediately need a second rule to keep it honest — because the easiest way to make a workbench look impressive is to cover it in pretty things that don't actually do anything.

The rule that keeps it honest

Here is the discipline. It sounds almost severe, but it is the whole secret:

⚠
No decorative gauges

Every visual element should correspond to a runtime concept. Avoid visualizations that have no runtime meaning. If a needle on the screen doesn't map to a real thing the framework models, it is a lie with good production values — and you should delete it.

Why so strict? Because a tool that shows you things that aren't really there teaches you a system that doesn't exist. When the Studio sweats, you should be able to point at any pixel and say "that one is the resource; that one is the live event." Nothing decorative. Nothing invented.

Why this is almost free here

Now, why can we be so confident the Studio reads the truth and not a hand-maintained copy of it? Because in this stack the description and the runtime are the same artifact.

An Ash resource defined with a Spark DSL isn't just code that runs — it is data you can introspect. The framework reads those declarations to do its job; the Studio reads the very same declarations to draw its panels. There is no parallel "model of the system" to drift out of date, because there is only one model, and everyone — runtime and tool — is looking at it.

🔍
One source of truth

Because resources and DSLs are introspectable data, the Studio needs no second description of your devices. It renders the declarations the runtime already obeys. The map and the territory are the same file.

The Timeline panel is the sharpest example. The AshEvents log from Concept 06 is a timeline already — an append-only record of what happened, in order. So "scrub and replay yesterday" isn't a feature you bolt on; it's just reading that log backward and forward. The Studio scrubs the event log; the event log was always there.

And what's the natural way to build a live, interactive tool like this on the BEAM? A Phoenix LiveView. It already holds live server state, pushes updates as events stream in, and takes a click and turns it into an action — exactly the shape of a workbench. That's the natural fit, not a new API we're inventing.

Who is this for?

One more design decision settles a hundred smaller ones. Who is the human at the bench?

🔑
Built for developers

The framework — and so the Studio — is primarily for developers. That single choice tells us exactly what to optimize for.

And when developers are the audience, the priorities line up in order:

  • Discoverability — you can find the action you need without a manual.
  • Consistency — every device behaves the same shape, so nothing surprises you.
  • Composability — pieces snap together; a new device reuses what you know.
  • Excellent error messages — the failure tells you what to do next.
  • Great documentation — the path from curious to productive is short.
🧭
Why uniformity is the UX

Discoverability and consistency aren't accidents here — they fall straight out of the declarative core. Because everything is a resource with the same shape, learning one device teaches you all of them. That uniformity isn't a side effect of the design; it is the user experience.

💡
The punchline

Developer experience is a feature. Not a nicety you add at the end — a property you design for from the resource up, exactly like throughput or safety.

Point it at the simulator

Here is where the Studio gets genuinely thrilling. Aim it not at a rack of steel but at the simulator from Concept 07. Now you can drive a virtual lab — open doors that don't exist, watch their state machines step, see the event stream pour down the timeline — and train an operator before a single bolt has been bought. Same panels, same truth, no hardware required.

Where we're headed

And that's the machine. We have resources, capabilities, adapters, state machines, events, a simulator, and now a workbench to watch the whole thing breathe. We've built it. Next comes the fun part — what you can actually do with all of it. The stunning use cases.
