Concept 07 · Simulation

Simulation First

If the last six chapters were the skeleton, this one is the heart. Everything AshDevices is for beats here: build the whole lab on a laptop, on a plane, with the wheels up — and have it be true.

Let me tell you about a plane ride. You're at 35,000 feet with no wifi, a dead phone, and a half-built incubator controller. On the bench at the office there's a real incubator — a heavy steel box that costs more than your car and lives behind a key-card door. You can't touch it. And yet, by the time the seatbelt sign goes off, you've written the control loop, demoed it to the person in 14C, and watched it recover from a heater fault — twice. How? Because in AshDevices the real machine is the special case. The simulator is home base.

This inverts the instinct most of us carry. We tend to think the hardware is the "real" thing and the simulator is a sad cardboard stand-in we tolerate until the real one is free. AshDevices says: no. Flip it.

🔑
The core rule

Every feature must be implementable in the Simulator. Never design an API that requires physical hardware to run. Simulation is the default runtime — the place where code is born, demoed, and tested. Hardware is the deployment target, not the development environment.

Read that last line twice, because it's load-bearing for the entire framework. You don't simulate because you can't reach the device. You simulate because simulation is where the work belongs. The device is just where it eventually runs.

A simulator is not a stub

Here's the trap, and it's a seductive one. You need Incubator.set_temperature/2 to work without an incubator, so you write the obvious thing:

elixir · the lie that feels like progress# DON'T: a stub that just agrees with you
def set_temperature(_device, _celsius), do: :ok
def read_temperature(_device), do: {:ok, 37.0}

That isn't a simulator. That's a yes-man. It returns :ok for everything, reaches 37 °C the instant you ask, never overshoots, never lags, never fails. It tells you exactly what you hoped to hear — which is precisely why it's worthless.

✈️
The flight-sim test

A flight simulator that never stalls is a toy, not a trainer. The entire value of a flight simulator is that it stalls like the real plane — same warning shudder, same nose drop, same recovery. A simulator earns its keep by reproducing the failures, not by hiding them.

· 1

So in AshDevices a simulator is a real model: it has timing (heat doesn't teleport), limits (the element maxes out at 42 °C), and failure modes (the door can be left ajar, the sensor can drift). It earns the same respect as production code.

🧪
First-class, not fake

Simulation is a first-class runtime. Never implement "fake" behavior. The simulator should behave as closely as possible to real hardware — and therefore a simulation bug is a framework bug, filed and fixed with the same seriousness as a crash on the bench.

Hardware-in-the-Loop

Now the move that makes this an engineering discipline rather than a nice idea. Recall the adapter seam from Chapter 5: a resource never talks to a wire, it talks to an adapter that satisfies a @behaviour. Hardware-in-the-Loop (HIL) cashes that seam in.

HIL means: run your real control software against a simulated plant that exposes the identical interface. The robot's brain doesn't know — and must not be able to find out — whether the joint it just commanded is steel or arithmetic. That's the whole trick: exercise the dangerous edges (faults, limits, thermal runaway, a jammed gripper) safely, repeatably, and before the hardware exists or at a scale no lab could afford.

🔌
The HIL contract

The Simulator and the physical device implement the same behaviour. Applications must never detect which one is active. Because both satisfy the contract from Chapter 5, switching between them is a config change — not a rewrite, not a flag in your logic, not an if Mix.env().

The same call, the same resource, the same action — two plugs that click into one socket:

one resource, one socket, two interchangeable plugs

The payoff: one call, two worlds

Your application code is written exactly once and never branches on environment. Robot.move/2 looks identical whether it's pushing electrons through a sim or amps through a motor:

elixir · application code — environment-blind# This line is the same in dev, in CI, and on the factory floor.
Robot.move(robot, :loading_station)

What changes is a single block of config — the plug you snap into the socket:

elixir · config/dev.exs & config/test.exs# Dev + CI: the simulated plant. No hardware in the building.
config :my_lab, Robot,
  adapter: Robot.Adapters.Sim,
  sim: [latency_ms: 120, jam_after: :never]
elixir · config/prod.exs# Prod: the real arm, same behaviour, different plug.
config :my_lab, Robot,
  adapter: Robot.Adapters.Modbus,
  modbus: [host: "10.0.0.7", unit_id: 3]

No if in your logic. No "is this a test?" sniffing. The resource asks its configured adapter to do the work, and the adapter — sim or steel — honors the same contract. That is HIL in three lines of config.

Testing: simulators argue, mocks agree

This reframes how you test. The default isn't a mock — it's the simulator itself, driven through the real resource and the real action:

🔁
The preferred pipeline

Simulator → Resource → Action → Assertion. Start the simulated device, call a real Ash action against it, let time advance, then assert on the state and events it actually produced. Avoid mocking adapters unless you are specifically testing adapter-internal behavior.

Why does this matter so much? Because of a subtle difference in what the two approaches even test:

A mock tests your guess about the device. You wrote the mock, so the mock can only ever confirm what you already believed. A good simulator tests the device's actual modelled behavior. The distinction is sharp: mocks lie back to you; simulators argue with you. When your control loop is wrong, the mock nods along and the bug ships. The simulator overshoots, oscillates, or trips a fault — and tells you, today, on the plane.

· 3

Mock-based test

Asserts your assumptions. Brittle: it breaks when you rename a function, not when the device misbehaves. Green even if your control logic is nonsense — the mock was told to say 37.

Simulation-based test

Exercises real device logic — timing, limits, failure modes. Goes red when the behavior is wrong. Survives refactors. Doubles as a demo and a regression net for the physics.

Here's the shape of it — an incubator test with no hardware anywhere in sight. We start the device on the Sim adapter, command a setpoint, let simulated time pass, and assert that physics and events came out right:

elixir · test/incubator_test.exsdefmodule IncubatorTest do
  use ExUnit.Case, async: true
  import Devices.SimHelpers

  test "converges to setpoint and emits an event" do
    # Simulator → Resource: a virtual incubator, no steel
    incubator =
      Incubator
      |> Ash.Changeset.for_create(:start, %{adapter: :sim})
      |> Ash.create!()

    # Action: a plain intent the lab would recognise
    incubator =
      incubator
      |> Ash.Changeset.for_update(:set_temperature, %{celsius: 37.0})
      |> Ash.update!()

    # Advance simulated time — heat doesn't teleport
    advance(incubator, minutes(20))

    # Assertion: physics converged…
    incubator = Ash.reload!(incubator)
    assert_in_delta incubator.reported_celsius, 37.0, 0.3

    # …and the device announced what happened
    assert_received {:device_event,
      %{name: :temperature_changed, celsius: 37.0}}
  end
end
⚠️
Illustrative API

The sim helpers (advance/2, minutes/1) are proposed ergonomics; the Ash calls — Ash.create!, Ash.update!, changesets, actions — are the real, idiomatic shape. The point isn't the exact names; it's that the test never touches a wire and still pins down behavior.

The constraint that keeps us honest

🔑
The thesis, restated

If it can't be simulated, it can't be in the framework. Simulation isn't a testing nicety bolted on at the end — it's the design constraint that keeps the whole architecture honest. A feature that secretly needs real hardware is a feature with a hidden wire, and hidden wires are exactly what Chapter 2 told us to stop building. Chapter 11's review checklist makes it literal: "Can it be simulated?"

With the architecture now sound — declarative resources, an adapter seam, and a simulator that argues back — what's left is craft. Next chapter: making the APIs, names, and errors good enough that the humans on the other end actually enjoy driving these machines.
