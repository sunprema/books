Intent, State & Events

Four habits separate a device you can reason about from a tangle of pokes and flags: actions are intent, state is explicit, events are first-class, and the twin remembers what you wanted.

Let me ask you a plain question. When a scientist walks up to a robot arm and says "pick up the sample," what did they just do? They didn't set pin 17 high. They didn't write 0x12 into a register. They expressed an intent — a verb that means something in the world of the lab. And here is the thing I want you to hold onto: the closer your code stays to that sentence, the easier your whole system is to understand, to test, and to trust.

So the first habit is simple to state and surprisingly hard to keep: actions describe business operations, not register pokes. An action on an AshDevices resource is a named action a domain expert would say out loud.

✓ Intent — a verb you can say

elixir · goodRobot.pick(robot)
Camera.capture(camera)
Experiment.start(run)

✗ Pokes — a wire and a prayer

elixir · badset_pin(17)
write_register(0x12)
publish_mqtt(topic, payload)

The left column reads like a sentence in the domain; the right column reads like a comment on someone's monitor. Both might toggle the same wire underneath — but only the left one survives a vendor swap, a code review, or a new teammate. These are custom named Ash actions on the resource, so they get arguments, validations, authorization, and notifications for free.

🔑
The test for an action

If a domain expert — a roboticist, a chemist, a building manager — would say it out loud, it's an action. If only an electrician would understand it, it's an implementation detail hiding in the wrong place.

State is explicit

The second habit: if a device has a lifecycle, model it. A door is closed, or open, or locked. An experiment is queued, running, or finished. Don't let that lifecycle live as a private flag tucked inside a process where nobody can see it.

Here's the picture I keep in my head. Hidden state in a GenServer is like a machine whose dial you can't see. While everything runs, you don't miss the dial. But the day it jams, you walk up and you cannot tell which state it's stuck in — closed? mid-open? locked? You're reduced to poking it and guessing. So make the dial visible: declare the states, and the framework will refuse the moves that don't make sense.

For this, reach for AshStateMachine — a real Ash 3.x extension. Here's a minimal door with three states:

elixir · lib/lab/devices/door.exdefmodule Lab.Devices.Door do
  use Ash.Resource,
    domain: Lab.Devices,
    data_layer: Ash.DataLayer.Ets,
    extensions: [AshStateMachine]

  state_machine do
    initial_states [:locked]
    default_initial_state :locked

    transitions do
      transition :unlock, from: :locked, to: :closed
      transition :open,   from: :closed, to: :open
      transition :close,  from: :open,    to: :closed
      transition :lock,   from: :closed, to: :locked
    end
  end

  actions do
    update :unlock do change transition_state(:closed) end
    update :open   do change transition_state(:open)   end
  end
end

Every piece here is a confirmed API: the state_machine do…end block, initial_states, default_initial_state, the transitions block with transition :name, from:, to:, and the transition_state/1 change inside an action. from: accepts a single atom or a list, and :* means any state.

🧩
Why this is the win

Try to open a :locked door and the framework rejects the transition — there is no :locked → :open edge. The rule lives in one declared place, not scattered across a dozen if-statements that someday disagree with each other.

Events are first-class

The third habit: every meaningful operation should emit an event — and the event should say what happened in plain words. A device that only mutates rows is a device with amnesia. A device that emits DoorOpened can be watched, audited, charted, and replayed.

✓ Descriptive — past-tense facts

eventsRobotMoved
CameraCaptured
TemperatureChanged
DoorOpened

✗ Generic — tells you nothing

events:updated
:changed
:event

Ash gives you two distinct mechanisms, and it matters that you don't conflate them. First, notifiers: an Ash.Notifier module receives an %Ash.Notifier.Notification{} after the transaction commits — with fields resource, action, data, changeset, and actor. You attach them via use Ash.Resource, notifiers: [...]. The built-in Ash.Notifier.PubSub broadcasts those over Phoenix.PubSub:

elixir · in the resourceuse Ash.Resource,
  notifiers: [Ash.Notifier.PubSub]

pub_sub do
  module LabWeb.Endpoint
  prefix "door"
  publish :open, ["opened", :id]
end

Second, AshEvents — a real event-sourcing and audit extension that persists create/update/destroy operations as events into an event-log resource and supports replay. That's a durable store, not a transient callback.

⚠️
Don't mistake a callback for a log

Notifiers fire only after commit and are transient — miss one and it's gone. They are perfect for live UI updates, useless as history. When you need a durable record you can audit or replay, reach for AshEvents. Both are pre-1.0, so verify the DSL against current docs.

Digital twins

The fourth habit closes the loop. A resource should distinguish between desired state and actual state — and never let them blur together. A digital twin is a live virtual representation of a device, kept in sync with the physical thing, and synchronization should always be observable.

You've seen this pattern before even if you didn't name it: AWS IoT Device Shadows, Kubernetes-style reconciliation. You store the desired state durably; the device reports its actual state; the system computes the delta and drives actual toward desired, tolerating disconnection along the way.

desired state is durable · the loop drives actual → desired

Map it onto AshDevices and it's just two groups of attributes: the desired side (target_celsius) and the reported side (reported_celsius), with a reconciliation loop closing the gap. Because desired state is durable, a device that drops offline catches up when it reconnects — the intent was waiting for it the whole time.

🌡️
The thermostat in your hallway

You set the dial — that's desired. The room temperature chases it — that's reported. The twin is the thermostat's memory of the setting, held even while the furnace is off. Cut the power and restore it, and the dial still knows what you wanted.

Where we're headed

Actions, explicit state, first-class events, faithful twins — every one of these is only trustworthy if you can exercise it without real hardware. You can't keep a robot on your desk to run a test suite. So the next idea is the one that makes all of this provable: simulation-first.
