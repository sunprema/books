Concept 05 · Adapters

Domain Before Protocol

A door knows how to be locked. It should not know it's a wire on pin 17. The instant your Door learns about GPIO, you've welded your application to a circuit board.

Here's a scene. A scientist walks up to your lab software and says "lock the freezer." She does not say "set the general-purpose I/O line at offset seventeen on gpiochip zero to logic high." She'd look at you very strangely if you asked her to. And yet that second sentence is exactly what most device code is — the protocol leaking all the way up into the parts of the program that should only ever speak human.

So the rule, stated bluntly, before we soften it with any nuance:

🔑
The rule

Never expose protocol-specific APIs. No part of your application that expresses intent should ever name a wire, a register, a topic, or a baud rate. Adapters translate domain intent into protocol operations — that is their entire job, and nobody else's.

Avoid — the wire leaks up

elixir · protocol talkGPIO.write(17, 1)
Serial.send(port, "M114\n")
MQTT.publish("dev/3/cmd", b)

Prefer — intent a human reads

elixir · domain talkDoor.lock(door)
Robot.move(robot, :dock)
Light.turn_on(light)

Same physical effect; worlds apart in what they commit you to. The left married you to a chip; the right married you to a vocabulary — and vocabularies are cheap to keep while you swap everything underneath.

The layering, stated exactly

If you remember one thing from this chapter, make it this. There is a correct stack and an incorrect one, and they differ by a single box — but that box is the whole game.

Correct

Application → Resource → Adapter → Hardware

Incorrect

Application → GPIO → Hardware

📐
The invariant

Applications must never depend on specific hardware. Changing hardware should never require application changes. If swapping a vendor forces you to touch a controller, a view, or a business rule — the layering has been violated upstream.

intent flows down; only the amber adapter ever touches a protocol

Resources are pure domain models

This is the load-bearing wall, so let me be concrete about what it forbids. Your Door resource does not talk to GPIO. It does not open a serial port. It does not publish to MQTT. It has never heard of USB. A resource that does any of those things has stopped being a domain model and become a device driver wearing a domain model's clothes.

Resources DEFINE intent

What's true, what's allowed, what may be requested: lock, unlock, the states a door can be in.

Adapters EXECUTE intent

How a request becomes a real effect: toggle a pin, write a register, record it in memory.

Keep that boundary religiously and the payoff is enormous: every resource becomes testable without any hardware in the room, because intent is just data until an adapter cashes it in.

An adapter is a contract

So what is an adapter, mechanically? In Elixir the answer is delightfully ordinary: it's a @behaviour — a named set of callbacks — plus any module that promises to implement them. The resource depends on the contract, never on a particular module. Swapping implementations is swapping which module answers the call.

elixir · the contractdefmodule AshDevices.Adapter.Door do
  @callback lock(config :: map) :: :ok | {:error, term}
  @callback unlock(config :: map) :: :ok | {:error, term}
end

Two characters in the same play, both bound by that script. First, the one that touches copper:

elixir · real adapter (circuits.gpio 2.x)defmodule AshDevices.Adapter.Door.Gpio do
  @behaviour AshDevices.Adapter.Door

  @impl true
  def lock(config) do
    # 2.x opens a gpio_spec: a {chip, line} tuple or a
    # label string, NOT a bare pin integer.
    {:ok, gpio} = Circuits.GPIO.open({"gpiochip0", 17}, :output)
    Circuits.GPIO.write(gpio, 1)
  end

  @impl true
  def unlock(config) do
    {:ok, gpio} = Circuits.GPIO.open("GPIO17", :output)
    Circuits.GPIO.write(gpio, 0)
  end
end

And second, its understudy — same contract, no copper at all, just memory:

elixir · sim adapterdefmodule AshDevices.Adapter.Door.Sim do
  @behaviour AshDevices.Adapter.Door

  @impl true
  def lock(config),
    do: Agent.update(config.store, &Map.put(&1, :state, :locked))

  @impl true
  def unlock(config),
    do: Agent.update(config.store, &Map.put(&1, :state, :unlocked))
end

The whole point in one line: the Door resource calls unlock. Which module runs is config. The resource cannot tell, and must not care, whether it just moved real steel or a number in a test.

What the adapter layer can wrap

Because the adapter is the only thing allowed to know a protocol, it's also where the BEAM's real, shipping hardware libraries live. None of these are invented:

ProtocolWhat it's forElixir lib

Protocols are an implementation detail of one layer. tortoise311 is the maintained MQTT 3.1.1 client; modbux speaks both Modbus TCP and RTU; the Circuits family is your direct-I/O kit on Nerves. Your Door sees none of these names — that's the design working.

⚠️
Honest footing

Some of these — opex62541, and a few older Modbus bindings — are lightly maintained; pin and verify versions before leaning on them. And to be square: AshDevices' adapter DSL is a proposal. The protocol libraries above are real and installable today. We're inventing the seam, not the wires.

Changing hardware = changing a config line

Now the payoff. Because the resource binds to the contract and the concrete module is config, swapping the entire physical reality of a door is a one-line edit. No controller changes. No rule changes. A line.

elixir · config — flip the world# config/dev.exs — simulate
config :lab, Lab.Devices.Door,
  adapter: AshDevices.Adapter.Door.Sim

# config/prod.exs — real steel
config :lab, Lab.Devices.Door,
  adapter: AshDevices.Adapter.Door.Gpio
🔑
The seam

That single adapter: line is the seam that makes simulation-first development possible. Build the whole app, test every rule, demo it to the scientist — with not one wire connected. Then flip the config and the same code drives copper. We build out that story in Chapter 7.

Where we're headed

Intent now flows down to an adapter that makes it physical. That raises three sharp questions: what are the discrete units of intent you may request (actions)? How do we track where a device actually is right now (state)? And when something happens out there, how does the device tell everyone (events)? Next chapter, we give intent a shape.
