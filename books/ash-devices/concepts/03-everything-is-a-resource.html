Everything Is a Resource

A robot is a resource. A door is a resource. A whole laboratory is a resource. Once you really believe that, the architecture practically writes itself.

Suppose I hand you an incubator — a box that keeps cells warm at body temperature — and I ask you to put it in your software. Your first instinct, trained by twenty years of bad habits, is to reach for the thing that talks to it: the serial cable, the driver, the little daemon. You start modelling the wire. Don't. Step back and ask the only question that matters: what is this thing, as a thing? It has a name. It has a temperature you want, and a temperature it actually has. You can ask it to warm up. It lives in a lab. That description — that little portrait of a noun and its verbs — is the whole device. And in AshDevices, the place where nouns and their verbs live is called a Resource.

So here is the design principle, and it is almost aggressively simple. Every physical object you care about — a Robot, a Camera, a Microscope, a Door, a Sensor — becomes an Ash Resource. And not only the gadgets that have plugs. A Laboratory is a resource. An Experiment — a plan, a thing with no wires at all — is a resource too. If it has identity, state, and things you can do to it, it is a resource.

🔑
The rule

Avoid introducing parallel abstractions. If something behaves like a Resource, it should probably be one. Don't invent a special "DeviceManager" or a bespoke struct on the side — reach for the resource you already know how to build.

Now, a resource is a pure domain model, and I want to be very strict about what that means, because it's the part everybody gets wrong. A resource describes the device. It does not talk to GPIO. It does not open a serial port. It does not publish MQTT. It has never heard of USB. The resource defines intent — "I would like to be 37 degrees" — and somebody else, the adapter, goes and makes the heat happen. We'll meet adapters properly in Chapter 5; for now, just hold the wall in your mind: descriptions on this side, wires on the far side.

What an Ash Resource actually is

Let's build one from first principles, because the word "resource" is doing a lot of work and I don't want it to stay a mystery. An Ash Resource is just an Elixir module that says use Ash.Resource. That one line wakes up a little factory which reads four kinds of declarations you write by hand:

  • attributes — the typed fields. Plain types like :string, :integer, :float, :atom, :utc_datetime, plus helpers uuid_primary_key and timestamps(). This is what the device is.
  • actions — the verbs. You get defaults [:read, :create, :update, :destroy] for free, and you add named ones — a create/read/update/destroy action that can accept fields, take an argument, run a change, or enforce a validate. This is what you can do.
  • relationships — belongs_to, has_many, has_one, many_to_many. How devices hang together.
  • calculations (read-only values from expr(...)) and aggregates (count/sum/avg/min/max/first over related data). Derived truth, computed on demand.

Here is a real, idiomatic one. Read it like a sentence:

elixir · the incubator, describeddefmodule Lab.Devices.Incubator do
  use Ash.Resource,
    domain: Lab.Devices,
    data_layer: Ash.DataLayer.Ets

  actions do
    defaults [:read, :destroy, create: :*, update: :*]

    update :set_temperature do
      accept [:target_celsius]
    end
  end

  attributes do
    uuid_primary_key :id
    attribute :name, :string, allow_nil?: false, public?: true
    attribute :target_celsius, :float, public?: true
    attribute :reported_celsius, :float, public?: true
  end

  relationships do
    belongs_to :laboratory, Lab.Devices.Laboratory
  end
end

Notice target_celsius sitting right next to reported_celsius. One is what you want; the other is what's true. That tiny pair is the seed of the digital twin — desired state shadowing actual state — and we'll grow it into a whole chapter (Ch. 6). It's already here, for free, just from describing the device honestly.

🔌
Find the wire

Go back and look at that module. There is no GPIO pin. No serial baud rate. No MQTT topic. That's the point. Getting the box to actually reach 37 °C is the adapter's job. The resource only ever says "I should be 37."

Resources live in a Domain

One device alone is lonely. Resources are grouped under a Domain — a module that does use Ash.Domain and simply lists its members. The domain is the address book; relationships are the streets between houses.

elixir · the lab as a domaindefmodule Lab.Devices do
  use Ash.Domain

  resources do
    resource Lab.Devices.Laboratory
    resource Lab.Devices.Incubator
    resource Lab.Devices.Experiment
  end
end

A Laboratory has_many :devices; an Experiment relates the devices it orchestrates. And you drive any of it the way you drive every Ash app — Ash.create/2, Ash.read/2, Ash.update/2, Ash.destroy/2, or a tidy code interface like Incubator.set_temperature(box, 37.0). Same verbs for a database row and for a warm steel box. That symmetry is the entire trick.

Where this is going

If a Robot and a Door are both resources, they'll share behaviour — both move, both emit events. The lazy answer is inheritance. Ash doesn't do inheritance, and it's right not to. Next chapter: how to share behaviour by composition — with Capabilities.
