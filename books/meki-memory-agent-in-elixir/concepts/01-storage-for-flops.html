Concept 01 · The Why

Storage for FLOPs

On a data-center GPU you scale a model by adding parameters. On a phone, that same move makes it unusably slow. MeKi changes which resource you spend.

Here is the awkward fact about putting a language model on a phone. The whole industry scales models the same way: add parameters, do more arithmetic. That works beautifully when you have a rack of GPUs and gigabytes of fast memory. It falls apart on a Qualcomm Snapdragon, where every extra parameter must be hauled across a narrow bus from RAM into the NPU before it can be multiplied — and that hauling, not the math, is what you wait for.

The MeKi paper measures this directly. On a Snapdragon 8 Elite running a small Qwen3 model with a 10K KV cache, roughly 38% of inference time is spent just moving FFN weights from RAM to the NPU (the DMA cost), and only about 32% doing the FFN's actual computation. Make the model bigger and that movement cost grows fastest. Mixture-of-Experts makes it worse, not better: sparse routing means constantly streaming large, disjoint expert weights across the same narrow bus.

📟
The hardware reality

On a mobile SoC the scarce resources are FLOPs (NPU compute) and RAM (working memory + bandwidth to the NPU). The abundant, cheap, and mostly-idle resource is ROM — flash storage. UFS 4.0 delivers about 4.2 GB/s of read bandwidth that the model barely touches during generation.

The central question

Can we scale up the model capacity by using the storage space, without increasing the latency and FLOPs during inference?

MeKi — Memory-based Expert Knowledge Injection — answers yes. The trick, in one sentence: give every transformer layer a giant table of pre-computed knowledge vectors indexed by token id, kept in ROM; at each step read out one tiny vector with a cheap lookup, nudge it with a small gate, and add it to the layer alongside the FFN. No routing. No big matmul. Capacity grows with the size of the table — which lives in storage you weren't using.

MeKi moves the scaling burden from the two scarce columns to the abundant one.

Why build it in Elixir?

You would not train a 50-billion-token run on the BEAM — that is Megatron-LM's job. But MeKi's inference shape is a wonderful fit for Elixir's numerical stack:

  • Nx gives us tensors and defn (compiled numerical functions), with the EXLA backend for fused CPU/GPU kernels.
  • Axon lets us express the layer — including a fully custom MeKi block — declaratively.
  • Nx.Serving turns a model into a supervised, auto-batching inference server that drops straight into a Phoenix app.

And MeKi's headline operation — “read a vector from a big table by id” — is exactly the kind of cheap, bandwidth-bound lookup the BEAM is happy to coordinate around. Over the next four chapters we build the architecture; then we write it.
