Token-Level Memory Experts

An “expert” in MeKi is not a network you route to. It is a vector you look up — one per token id, per layer. Two such vectors, fused.

Mixture-of-Experts picks, for each token, a few sub-networks to run. MeKi does something simpler and cheaper: for each token it reads a single dedicated knowledge vector straight out of a table. The token id is the address. No router, no learned gating over experts, no branching — just memory.

Two complementary tables produce that vector. The first is static memory: a plain, trainable embedding table for layer l,

Ml ∈ ℝ|V|×dmem,   mstatic(xt) = Ml[xt]

where |V| is the vocabulary size and dmem is a small memory dimension — 128 or 256, far below the model width dmodel. This table learns per-token, per-layer priors directly in its own little space.

🗂️
Why "expert"?

Because each row carries knowledge specialized to one token at one depth. The vocabulary is the set of experts, and the lookup activates exactly one of them — the sparsest routing imaginable, with no routing cost.

5 ·

The second table is dynamic memory. During training only, MeKi feeds the model's global word embedding Eglobal through a small non-linear projection Gl (a SwiGLU), synthesizing a layer-specific feature from the shared semantic space:

mdyn(xt) = Gl( Eglobal[xt] )

The final expert vector fuses the two, stabilized by RMSNorm and two learnable scalars:

elt = αl · RMSNorm( mstatic(xt) + βl · mdyn(xt) )

Read the knobs: βl sets the per-layer blend between learned static priors and projected dynamic features; αl sets how strongly this layer injects knowledge at all. Both are scalars, both are learned.

In Nx, the lookups are one call each — gather rows by id with Nx.take/2:

elixir · iex# M_l :: {vocab, d_mem}   token_ids :: {batch, seq}
iex> m_static = Nx.take(m_l, token_ids)
#=> tensor shape {batch, seq, d_mem}

# dynamic side (training): project the global embedding rows
iex> e_rows  = Nx.take(e_global, token_ids)  # {b, s, d_model}
iex> m_dyn   = MeKi.SwiGLU.forward(e_rows, g_params)

Do both tables really earn their keep? The ablation says yes. On the 0.6B model, static-only lifts the average benchmark score from 53.0 to 54.8, dynamic-only lands in the same neighborhood, and combining them beats each by a further +0.7 / +0.8 — evidence the two carry complementary information. We will see in Chapter 5 that the dynamic side costs nothing at inference, because it gets folded away entirely.

🧠
The shape that matters

Everything so far lives in the cramped dmem space (e.g. 256), not the wide dmodel space (e.g. 2048). That is what keeps both the table and the upcoming gate cheap.
