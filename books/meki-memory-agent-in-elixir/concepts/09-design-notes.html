Design Notes & Trade-offs

Where memory scaling belongs in the landscape, when to reach for it, and an honest accounting of what this book did and did not do.

MeKi is one answer to a family question — how do I add knowledge to a model without adding compute? — and it helps to see it next to its neighbours, because each makes a different trade.

ApproachAdds capacity viaInference cost
Dense scalingmore parameters (RAM)FLOPs + DMA ↑↑
MoErouted expert FFNsrouting + weight streaming
RAGretrieved documentsretrieval + longer context
PLEtoken-id embedding depthcheap, but bottlenecks
MeKitoken-keyed ROM tablelookup + thin gate ≈ 0

The distinctive MeKi bet: capacity that is indexed by the token itself (no learned router, unlike MoE), kept in cheap storage (not RAM), and integrated in parallel as a width-expander (not stacked in series, unlike PLE's depth growth). It is closest in spirit to a per-layer, per-token key–value store baked into the weights.

When to reach for it

Good fit

  • On-device / edge LLMs where ROM is plentiful but FLOPs and RAM bandwidth hurt.
  • Knowledge-heavy tasks (factual recall, science QA).
  • You control pre-training and can afford extra training compute.

Poor fit

  • You only have a fine-tune budget — MeKi is a from-scratch architecture change.
  • Storage is the scarce resource, not compute.
  • You need knowledge to update without retraining — that is RAG's job.

“Meki, a memory agent”?

This book's title frames MeKi as a memory agent. Be precise about what that means: MeKi is not an autonomous agent in the tool-using sense. It is a memory mechanism — each layer consults a private table of token-keyed knowledge and injects what it finds. If you are building an actual agent in Elixir (say with Jido or an Ash-based tool layer), MeKi is the kind of component that would make its underlying on-device model punch above its parameter count. The “agent” is the system you build around it; MeKi is the memory it thinks with.

🧰
An Elixir deployment blueprint

Train (in your framework of choice) → export params → reparameterize_layer/2 per layer to bake each M̃ˡ → store the tables as a memory-mapped binary (the BEAM's :binary + Nx.from_binary/2 read them lazily) → load the active weights into an Axon inference model → wrap in Nx.Serving in your supervision tree → call batched_run/2 from Phoenix.

🧪
Honest scope, one more time

We implemented MeKi's architecture and re-parameterization faithfully in Nx/Axon, with API-correct code. We did not pre-train it on 50B tokens or reproduce the benchmark numbers — those are quoted from the paper. Treat the code as a correct skeleton to train and measure, not a finished checkpoint.

🔑
The one-sentence takeaway

MeKi scales a model by spending storage instead of arithmetic: train with a rich projection, fold it into a static ROM table, and infer with a lookup plus a thin gate — capacity without latency.
