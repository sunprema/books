Concept 10 · GO

Go — Concurrent GC & Escape Analysis

Go is garbage-collected like Java, but it made a sharp, opinionated choice: above all else, keep the pauses tiny. It would rather do a little extra collecting work than ever freeze your goroutines for long. The result is a collector tuned for servers juggling thousands of concurrent connections.

Go's collector is a concurrent, tri-color mark-and-sweep. "Tri-color" is just the bookkeeping from Concept 05 made precise — every object is white (unvisited), gray (reached, but its children not yet scanned), or black (fully scanned):

Objects flow white → gray → black as marking proceeds. When no gray remain, every white object is garbage.

The clever part: nearly all of this marking happens while your program keeps running. To stay correct when your code mutates pointers mid-collection, Go uses a hybrid write barrier — a tiny bit of code on pointer writes that keeps the tri-color invariant intact. The stop-the-world windows are reduced to sub-millisecond bookends, not the whole collection.

★
The design priority

Go deliberately is not generational (historically) and not the highest-throughput collector. It trades some efficiency for consistently low latency — short, predictable pauses — because that's what network services need most.

Escape analysis — the compiler keeps GC off the hook

Here's Go's quiet brilliance, and it ties straight back to Concepts 02 and 03. Before anything reaches the collector, the compiler asks of every value: does a pointer to this escape the function that made it? If not — if it dies when the function returns — put it on the stack, free and automatic, never troubling the GC at all. Only values that genuinely outlive their function "escape" to the heap.

Go — see what escapes// go build -gcflags='-m'
func stays() int {
    x := 42      // stays on the stack
    return x
}
func escapes() *int {
    x := 42      // "moved to heap: x"
    return &x     // pointer outlives the frame
}
◆
Why this matters

The cheapest garbage is the kind that's never created. By proving values can live on the stack, the compiler keeps huge amounts of short-lived data out of the heap entirely — so the GC has dramatically less to chase. Remember broken() from Concept 02? In Go that's safe: returning &x just makes x escape to the heap automatically.

One knob: GOGC

Go keeps tuning refreshingly simple. The main dial is GOGC (default 100): collect once the heap has grown by that percentage since the last collection. Raise it to trade memory for fewer collections; lower it to trade CPU for a smaller footprint. (GOMEMLIMIT adds a soft ceiling.)

▣
As of 2026

Go 1.24 added a standard weak package for weak references (caches that don't pin memory). Go 1.26 introduced "Green Tea" GC, a redesigned, more memory-aware collector — the project's biggest GC change in years, improving locality and scalability on modern hardware.

What you gain

GC safety with tiny pauses, plus a compiler that quietly keeps most allocations off the heap. Great fit for high-concurrency servers.

What you pay

Lower peak throughput than a throughput-tuned collector, and higher memory use than manual/Rust — you give the heap slack so collections stay rare.
