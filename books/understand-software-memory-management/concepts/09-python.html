Concept 09 · PYTHON

Python — Counting References (+ Cycles)

Python (the standard CPython) took the refcounting road from Concept 05 — and then, because refcounting alone leaks cycles, bolted a tracing collector on top to sweep them up. Two collectors, working as a team. It's a wonderful case study in why no single technique is enough.

Every Python object carries a hidden field: ob_refcnt, its reference count. Bind it to a name, pass it to a function, stick it in a list — the count goes up. Let a name go out of scope or del it — the count goes down. Hit zero and the object is freed immediately, right then, no waiting.

Python — watch the countimport sys
x = [1, 2, 3]
sys.getrefcount(x)   # 2 (x, plus the arg itself)
y = x
sys.getrefcount(x)   # 3
del y
sys.getrefcount(x)   # 2 — back down
★
The upside: prompt & predictable

Because objects die the instant their count hits zero, cleanup is immediate and incremental — no big pause to scan the whole heap. A file handle closes the moment the last reference drops. Memory usage stays tight. This is refcounting's gift.

The cycle problem, made concrete

Python — a cycle refcounting can't freea = {}
b = {}
a['b'] = b   # a → b
b['a'] = a   # b → a  (a cycle!)
del a, b      # outside refs gone…
# …but each still holds the other at count 1.
# Refcount alone: leaked forever.

This is the exact blind spot we predicted. Their counts never reach zero because they prop each other up. So CPython runs a second collector.

The cycle detector: tracing, generational

Layered on top of refcounting is a tracing garbage collector (the gc module) whose only job is to find unreachable cycles. It periodically examines container objects, computes whether a group is reachable from outside, and if a cluster only points at itself, reclaims it. And — you guessed it — it's generational, with three generations, collecting the youngest most often.

Refcounting does the bulk, promptly; the tracing collector mops up what cycles leave behind.

The GIL — and its slow-motion exit

◆
Why the Global Interpreter Lock existed

Those refcounts are touched constantly. If two threads bumped the same count at once, it would corrupt. The cheap historical fix: a single GIL letting only one thread run Python bytecode at a time. Simple and safe — but it meant pure-Python code couldn't use multiple cores.

▣
As of 2026 (CPython 3.13 → 3.14)

The free-threaded build (PEP 703) finally makes the GIL optional. To keep refcounts safe without it, 3.13 introduced biased reference counting (the owning thread uses fast non-atomic updates); 3.14 added a thread-safe incremental GC and cut the single-thread penalty to ~5–10%. It's maturing fast but still stabilizing for production.

What you gain

Effortless productivity, prompt cleanup, and a hands-off memory model great for scripting and glue.

What you pay

Per-object overhead (every object carries a count + header), refcount churn, and — historically — the GIL capping CPU parallelism.

Python proves the thesis of this book: cleanly combine two imperfect collectors and you get a practical whole. Now the last language — one that's GC'd like Java but tuned for a very different goal: keeping pauses tiny while running thousands of concurrent things. Go.
