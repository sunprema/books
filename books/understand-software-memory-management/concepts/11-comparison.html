Concept 11 · The Payoff

Five Languages, One Table

Every language in this book solves the same problem — when is a heap object dead, and who reclaims it? — and each picks a different point on the triangle of safety, speed/predictability, and ease. You can't max all three; you choose which two matter for the job.

LanguageStrategyWho frees itCyclesPausesMemory safe?
C manual malloc/free you, by hand n/a (your problem) none ✗ no
RUST ownership + RAII, compile-time compiler inserts drop leak via Rc; use Weak none (deterministic) ✓ at compile time
ZIG manual + explicit allocators you, with defer n/a (your problem) none ~ runtime-checked (debug)

Need max control / no runtime

Kernels, embedded, allocators, real-time. C / Zig for raw manual control; Rust when you also want the compiler to prove you got it right.

Need productivity at scale

Business apps, big services. Java (mature, tunable, pick your collector) or Go (tiny pauses, great concurrency, simple knob).

Need speed of thought

Scripts, data, glue, prototypes. Python — hands-off memory, instant feedback; reach for native extensions when the CPU bill comes due.

★
The whole book in three sentences

The stack frees itself for free; the heap is the part someone must manage. Someone is either you (C, Zig), the compiler reasoning at build time (Rust), or a collector reasoning at runtime by reachability (Java, Go) — often paired with refcounting (Python). And the entire history of GC is one long campaign to shrink the stop-the-world pause toward zero.

▣
There is no "best" — only fits

A GC isn't a crutch and manual memory isn't machismo. They're different tools for different costs. The engineer's job isn't to find the one true answer; it's to know which bill you're willing to pay — latency, memory, CPU, developer time, or a fight with the borrow checker — and pick accordingly. Now you can read that bill.
