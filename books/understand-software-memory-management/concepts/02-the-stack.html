The Stack — Automatic & Fast

Imagine a stack of plates in a cafeteria. You only ever add a plate to the top, and you only ever take one off the top. That's it. That single rule is the most efficient memory idea ever invented.

When your program calls a function, the machine shoves a fresh plate onto the stack — a stack frame. On that plate sits everything the function needs that's local to it: its parameters, its local variables, and a little note saying "when you're done, jump back to here." When the function returns, the plate is lifted off. Gone. Instantly.

And here's the beautiful part — how much does it cost to free a stack frame? Nothing. Truly nothing. The "stack pointer" is just a number that says where the top is. Allocating means subtracting from it; freeing means adding back. One CPU instruction. No bookkeeping, no searching, no list of free blocks. It's so cheap it barely deserves the word "allocation."

★
Why the stack is free

Allocation = move the stack pointer down by N bytes. Deallocation = move it back up. The data isn't even erased — it's just abandoned, ready to be overwritten by the next call. That's why locals hold garbage until you initialize them.

C — frames stack up and pop offint square(int x) {
    int r = x * x;   // 'r' lives on square's plate
    return r;        // plate popped; r is gone
}
int main(void) {
    int a = square(5); // push frame, run, pop
    return 0;
}

Because frames stack in call order and pop in exactly the reverse order, you get a perfect record of "who called whom." That record is the call stack you see in every debugger and every crash report. The stack isn't just storage — it's the program's memory of how it got here.

square's frame sits on top of main's. Return pops it; SP jumps back up.

The catch: it must be small and short-lived

The stack's superpower — automatic cleanup tied to scope — is also its cage. Two hard limits:

It's tiny

A thread's stack is typically 1–8 MB, fixed. Allocate a giant array as a local, or recurse too deep, and you blow past the end — a stack overflow, and the program dies on the spot.

It can't outlive the call

The frame is gone when the function returns. So you can't return a pointer to a local — the plate it lived on is already off the stack. That pointer now aims at garbage.

C — the classic dangling-pointer mistakeint* broken(void) {
    int n = 42;
    return &n;   // ✗ address of a local —
}              //   the frame dies here!
▲
Use-after-return

The caller now holds a pointer into a frame that's been popped. The next function call reuses that exact space — so the value silently changes underneath you. No crash, no warning, just wrong answers. This is the bug that pushes data onto the heap.

So the stack is perfect for "I know how big this is and I'll be done with it before I return." For everything else — data that must grow, or live longer than the function that made it — we need a region with no such rules. We need the heap.
