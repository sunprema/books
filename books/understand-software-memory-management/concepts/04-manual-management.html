Doing It By Hand — and the Bugs It Breeds

For decades the deal was simple: you call malloc, you call free, and you'd better match them up perfectly. It sounds easy. It is, for ten lines. Across a million lines and twenty programmers, it has caused more security holes than any other single idea in computing.

The trouble is that "when is this object dead?" is genuinely hard to answer. A pointer gets copied, passed into a function, stored in a list, handed to another thread. Now five places might be looking at the same shelf. Who frees it? If everyone assumes someone else will, it leaks. If two of them free it, the warehouse's records get corrupted. There's no local rule — you have to reason about the whole program at once.

Let's meet the four horsemen. These exact bugs are why Rust, Java, Python, and Go exist.

① The leak — forgot to free

C — memory you can never get backvoid handle(void) {
    char* buf = malloc(1 << 20);  // 1 MB
    if (early_error()) return;     // ✗ buf leaked!
    use(buf);
    free(buf);
}

One forgotten path and a megabyte vanishes every call. A server doing this slowly swells until the OS kills it. Not a crash you can catch — a slow, creeping death.

② Use-after-free — freed, then touched

C — the shelf was given awayfree(p);
printf("%d", p->value);  // ✗ reading a returned shelf

The instant you free(p), the allocator may hand that exact space to the next request. Your p still points there — but now it's someone else's data. This is the bug attackers love: control what lands on the reused shelf and you can hijack the program.

③ Double-free — freed twice

C — corrupting the allocator's booksfree(p);
// … later, a different code path …
free(p);   // ✗ second free scrambles the heap

The allocator keeps its free-list inside the heap itself. Free the same block twice and you tangle that list — a corruption that often crashes much later, somewhere innocent, making it maddening to trace.

④ Dangling & the null deref

A pointer that outlives what it points to is dangling. Reading through one — or through an uninitialized or NULL pointer — is undefined behavior: maybe a crash, maybe silent garbage, maybe a security hole.

After free, p dangles — aimed at a shelf now owned by other data.
▲
This is not a small problem

Microsoft and Google have each reported that roughly 70% of their serious security vulnerabilities are memory-safety bugs — overwhelmingly these four. That number is the entire reason the industry has spent decades inventing ways to take the job away from the programmer.

So what do we do about it?

Three escapes from manual misery, and the rest of this book is a tour of each:

  • Make the compiler prove it. Track ownership at compile time and insert the free for you, exactly once, exactly on time. Rust C++ RAII
  • Keep it manual but make the bugs loud. Pass allocators explicitly, and ship a debug allocator that catches leaks and double-frees in testing. Zig
  • Hire a robot janitor. A garbage collector runs alongside your program, finds memory nothing can reach, and reclaims it. Java Python Go

That robot janitor sounds like cheating — how can a program tell, while running, which memory is truly dead? It's one of the prettiest ideas in computer science. Let's go see how it works.
