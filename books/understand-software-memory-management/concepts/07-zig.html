Concept 07 · ZIG

Zig — Explicit Allocators, Manual but Honest

Zig looks at Rust's borrow checker and the GC's janitor and politely declines both. It keeps memory manual — like C — but fixes C's real sin: hidden, silent allocation. In Zig, memory never moves without you seeing it. Nothing allocates behind your back.

The single most distinctive decision in the whole language: there is no global allocator. A function that needs heap memory must be handed an allocator as a parameter. You can see, right in the signature, exactly who can allocate and where the memory comes from.

Zig — allocate, then defer the freefn readThing(allocator: std.mem.Allocator) ![]u8 {
    const buf = try allocator.alloc(u8, 1024);
    defer allocator.free(buf);  // runs on EVERY exit path
    // … use buf …
    return result;
}
★
defer — the antidote to the leak

defer schedules cleanup to run when the scope exits, no matter how — normal return, early return, or an error. You write the free on the line right after the alloc, so the pair is impossible to miss. It's manual management with the forgetting designed out.

Compare that to C's leak from Concept 04, where an early return skipped the free. In Zig, defer fires on the early return too. The bug pattern simply can't form when you follow the idiom.

The allocator is a choice, not a default

Because allocators are values you pass around, you pick the strategy that fits the job — and swap it without touching the code that allocates:

AllocatorBest for
DebugAllocatordevelopment — catches leaks, double-free, use-after-free
ArenaAllocatorallocate many, free all at once (per request/frame)
FixedBufferAllocatora fixed stack/static buffer — zero heap, no failure
page_allocatorstraight from the OS, big chunks
c_allocatorplain libc malloc/free
▣
Version note (Zig 0.14)

The old GeneralPurposeAllocator was renamed: its safety-checking debug build is now DebugAllocator. Use it in tests and it will fail your test if anything leaks — turning "did I free everything?" from a hope into a checked fact.

The arena is the secret weapon Zig makes easy. When you have a burst of allocations that all die together — everything you touch handling one web request, say — you give them an arena, and at the end you call deinit once. All of it, freed in a single stroke. No per-object bookkeeping, no leaks possible, and it's blisteringly fast.

Zig — an arena: free everything at oncevar arena = std.heap.ArenaAllocator.init(base);
defer arena.deinit();          // frees ALL of it
const a = arena.allocator();
const x = try a.alloc(u8, 64); // no individual free needed
const y = try a.alloc(u8, 64);

What you gain

Total control and total visibility. No hidden costs, tiny runtime, predictable performance — and a debug allocator that makes the classic bugs loud during testing.

What you pay

It's still you who must reason about lifetimes. The compiler won't reject a use-after-free at build time the way Rust does — it trusts you, and verifies at runtime in debug builds.

So Zig sits exactly between C and Rust: more honest and safer than C, more manual and flexible than Rust. Now we cross the river to the other philosophy entirely — hand the whole problem to a runtime. Enter the JVM.
