Garbage Collection: How the Magic Works

Here's the trick, and it's gorgeous. The collector never tries to figure out what's dead. That's too hard. Instead it figures out what's alive — and declares everything else garbage. Reframe the question and it becomes answerable.

What does "alive" mean precisely? An object is alive if the program can still reach it. Start from the things the program obviously holds right now — the roots: every local variable on every thread's stack, plus the global variables. Anything a root points to is alive. Anything those point to is alive. Follow every pointer, like ink spreading through water. When you can't reach any further, you've found every live object. Everything you didn't touch? Unreachable. The program has no way to ever name it again. It is, by definition, garbage.

★
The core insight

Liveness = reachability from the roots. The collector doesn't read your mind about intentions; it reads your pointers. If nothing points at it, you can't use it, so it's safe to reclaim. This is provably correct — you can never collect something still in use.

Tracing: mark, then sweep

The classic algorithm, "mark-and-sweep," is two passes over the object graph:

  • Mark. Start at the roots. Walk every reachable pointer, painting each object you reach as "in use."
  • Sweep. Scan the heap. Anything still unpainted gets returned to the free list. Wipe the paint and wait for next time.
A, B, C are reachable from root r1 → kept. D and E point only at each other → collected.

The other family: reference counting

There's a second, simpler idea. Give every object a little tally: how many pointers currently aim at me? Make a new reference, add one. Drop a reference, subtract one. The instant the tally hits zero, nobody's looking — free it immediately.

refcounting, in spiritx = obj      # obj.count = 1
y = obj      # obj.count = 2
del x        # obj.count = 1
del y        # obj.count = 0 → freed now

It's wonderfully prompt — memory dies the moment it's done, no pause. But it has a fatal blind spot: cycles. If A points to B and B points back to A, each keeps the other's tally at 1 forever, even when the program has dropped all outside references. Their counts never reach zero. They leak.

●
No single method wins

Tracing handles cycles effortlessly but runs in bursts (and historically paused your program). Refcounting is prompt and incremental but leaks cycles and adds overhead to every pointer assignment. So real runtimes combine them — and you'll see exactly that in Python.

★
The weak generational hypothesis

Most objects die young. The temporary string, the loop variable, the request object — born and abandoned in milliseconds. A few objects (caches, config, long-lived state) live nearly forever.

So split the heap by age. Keep newborns in a small young generation and collect it often and fast — it's mostly garbage, so each pass reclaims a lot for little work. Objects that survive a few rounds get promoted to the old generation, which you scan rarely. You spend your effort where the garbage actually is. This one idea makes modern GCs practical.

The price of a janitor

What you gain

No leaks (except cycles in pure refcounting), no use-after-free, no double-free. Whole bug classes simply gone. You think about logic, not lifetimes.

What you pay

CPU and memory overhead, less predictable timing, and the classic stop-the-world pause — moments when the collector freezes your threads to work safely.

Decades of GC research is one long war on that pause — shrinking it from seconds, to milliseconds, to (in today's ZGC) well under a millisecond. Now let's watch five languages each pick a very different point on this map, starting with the one that refuses to hire a janitor at all: Rust.
