Rust — Ownership, No GC Needed

Rust pulls off something that sounds impossible: the safety of a garbage collector with the speed of hand-written C, and no janitor at all. The trick is to move the whole argument from runtime to compile time. The compiler proves where every free goes — and writes it for you.

It rests on three rules, the ownership model:

  • Every value has exactly one owner — one variable responsible for it.
  • When the owner goes out of scope, the value is dropped (freed) — automatically, deterministically, exactly once.
  • Ownership can move to a new owner, but only one owner exists at a time.

That's RAII — the C++ idea that a resource's life is tied to a variable's scope — but here the compiler enforces it rather than trusting you. Watch a move invalidate the old name:

Rust — a move, checked at compile timelet s1 = String::from("hi"); // s1 owns the heap buffer
let s2 = s1;                // ownership MOVES to s2
println!("{}", s1);        // ✗ compile error:
//   value borrowed here after move

No double-free is even expressible: after the move, s1 isn't a second owner you might accidentally free — it's gone. The compiler refuses to build the program. The bug dies before it's born.

◆
Where's the free()?

You never write one. The compiler knows s2 dies at the end of its block, so it inserts the drop (which frees the buffer) right there. Deterministic — the exact same spot every run. No collector, no pause, no tally. The cleanup is baked into the machine code.

Borrowing — use without owning

Moving everywhere would be painful, so Rust lets you borrow a reference. The borrow checker enforces one famous rule:

★
Aliasing XOR mutation

At any moment you may have either many readers (&T, shared) or exactly one writer (&mut T, exclusive) — never both. That single rule statically kills data races and use-after-free, because a reference can't outlive the value it points to.

Rust — the borrow checker stops a danglefn dangle() -> &String {   // ✗ won't compile
    let s = String::from("x");
    &s                       // s is dropped here;
}                            // returning &s would dangle

When one owner isn't enough

Sometimes data genuinely needs many owners (a node in a graph). Rust makes you opt in to reference counting, explicitly:

TypeMeaning
Box<T>one owner, heap-allocated
Rc<T>shared owners, single-thread refcount
Arc<T>shared owners, atomic (thread-safe)
Weak<T>non-owning ref — breaks cycles

So Rust does have refcounting — but only where you asked for it, with a visible cost, not as a blanket tax on every object. And cycles? They can still leak with Rc; the answer is Weak, a deliberate choice rather than a silent collector cleaning up after you.

▣
The bargain

You pay up front, in learning the borrow checker and occasionally restructuring code to satisfy it. In return: C-class speed, no runtime, predictable cleanup, and memory bugs caught at compile time. The cost moved from "3am production crash" to "this morning's compiler error." For systems work, that's the trade of the decade.
