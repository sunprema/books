How a Running Program Sees Memory

Before we argue about stacks, heaps, and garbage collectors, let's stand back and ask the dumb, wonderful question: when your program is running, where does it actually keep its stuff?

Here's a fact that surprised me the first time I really looked at it. Your program does not see the actual chips of RAM soldered to the board. The operating system tells it a beautiful lie. Every process gets handed its own private, enormous, perfectly tidy ribbon of numbered boxes — address 0, address 1, address 2, all the way up to some astronomical number — as if it owned the whole machine. This is the virtual address space, and it's a lie that makes everything else possible.

◆
Why the lie?

If every program saw the same real memory, they'd scribble over each other constantly. Give each one its own imaginary map and the hardware (the MMU) quietly translates "your address 5000" into "real chip location 0x7F3A…" — different for each program. Isolation for free.

Now, that long ribbon isn't used uniformly. The program lays out its furniture in a few distinct neighborhoods, and they almost never change their floor plan. From the bottom of the address space climbing up:

  • The code (text) segment — the actual machine instructions of your program. Read-only, fixed size, loaded once. Nobody writes here; it's the sheet music.
  • Globals & statics — variables that live for the whole run. Split into initialized data (you gave them a value) and the BSS (zero by default). Fixed size, known at link time.
  • The heap — the wild, growable region. Grows upward as you ask for memory at runtime. This is where "give me space for a million things I'll size later" lives.
  • The stack — grows downward from the top. Function calls, local variables, return addresses. Fast, automatic, tiny.

The heap climbs up, the stack falls down, and the big empty gap in the middle is the breathing room they share. Run out of that gap — the two collide — and you're out of memory.

One process's virtual address space. Stack falls, heap climbs, the gap is shared.

Two questions decide everything

For any piece of data your program creates, the language (or you) must answer two questions. Almost every idea in this book is just a different answer to them:

How big & how long?

Size known at compile time and lifetime tied to a function call? → the stack can handle it automatically.

Who frees it?

Size or lifetime unknown until runtime? → the heap. And now someone must remember to give it back.

That second question — who frees the heap? — is the whole war. You can do it by hand (C, and carefully in Zig). The compiler can do it for you by tracking ownership (Rust). Or a garbage collector can find the abandoned memory while your program runs (Java, Python, Go). Same problem, radically different bargains.

★
The one idea to carry forward

"Memory management" is really just lifetime management. The hard part was never getting memory — it's knowing the exact moment a piece is dead so the space can be reused without anyone still looking at it.

So let's start with the easy half — the memory that frees itself, the stack — and earn our way up to the hard half.
