Chapter 09

Downstream: The Sidecar & Replay Harness

Your process commits a .llog artifact and moves on — its job is done in microseconds. Everything valuable happens after that, out of your process, in a separate container. You don't build any of it. But understanding it tells you why the capture is shaped the way it is, and pins down the one thing you control that turns a checkpoint into a runnable failing test: the replay_boundary flag.

Where the checkpoint goes

The committed artifact lands in /dev/shm/llog/events/. From there a Go sidecar — the llog-agent daemon — takes over. It runs as its own process, typically a separate container in the same pod, deliberately decoupled from your application: your app never blocks on it, and it never reaches back into your app.

The sidecar watches the events directory, then runs each checkpoint through a fixed pipeline:

Two of those stages connect straight back to earlier chapters. Sanitize is the host-side half of the privacy story from Chapter 6: key filtering, pattern detection, and entropy detection run here, and raw execution data never leaves the host — the sidecar-to-platform link is mTLS. Fingerprint is the downstream half of storm containment from Chapter 7: the sidecar dedupes by failure signature before any expensive AI analysis, and caps AI/PR/ticket actions per fingerprint per window.

You can inspect a checkpoint yourself

The pipeline stages are composable filters — each reads a checkpoint and writes one back, over a JSON envelope — and they're exposed as CLI verbs in llog-ctl. When you're integrating and want to see exactly what your instrumentation produced, reach for these:

$ llog-ctl inspect events/3f9c.llog   # human-readable: the graph, intent, error, frames
$ llog-ctl verify  events/3f9c.llog   # checksum + classification (COMPLETE / TRUNCATED / ...)
$ llog-ctl cat     events/3f9c.llog   # the checkpoint as a JSON envelope

# the stages are pipeable — the sidecar just runs them in-process:
$ llog-ctl cat events/3f9c.llog | llog-ctl sanitize | llog-ctl fingerprint

The payoff: a checkpoint becomes a failing test

Here is where the flag you set in Chapter 3 earns its keep. The Replay Harness Generator turns a Failure Checkpoint into a runnable failing test plus sanitized fixtures. The scopes you marked replay_boundary=True become mock seams: the generated test stubs each one with the snapshot captured at that boundary, so the reproduction runs offline and deterministically, without ever touching the real external service.

The discipline that makes this correct: you stub a completed seam; you never stub the scope the error died in. Stubbing the failing scope would erase the very failure under test. Look at the refund flow from Chapter 3:

demo/refund.py — the seam vs. the bugwith llog.scope("ChargeGateway", replay_boundary=True):
    llog.snapshot("gateway_request", {"amount": amount, "retries": divisor})
    charge = charge_gateway(amount)            # external call — the SEAM
    llog.snapshot("gateway_response", charge)   # ← becomes the stub's return value

with llog.scope("SettleLedger"):
    fee = amount / divisor                    # the BUG — replays for real → ZeroDivisionError
    return charge["net"] - fee

On replay, ChargeGateway is not executed — it's patched to return the captured gateway_response, so the test needs no network. But SettleLedger runs for real, and reproduces the exact ZeroDivisionError that a gateway retries count of 0 caused. Stubbing the completed seam is what lets the test run offline; running the failing scope for real is what reproduces the bug. That generated test then runs in CI as a regression guard, and is the validation gate for any AI-proposed fix — diagnosis ships first, and PR generation is gated behind sandbox validation.

Your one job here

Put replay_boundary=True on the scopes that wrap your external-call seams — payment gateways, HTTP clients, third-party APIs, database calls you'd want stubbed — and snapshot their responses. That single flag is what lets a production failure come back to you as an offline, deterministic, reproducing test. Everything else downstream is automatic.

The whole arc, in one line

A production failure happens → LinkLog captures the execution context → the sidecar sanitizes it → an analyzer explains the root cause → and you get a failing test that reproduces it. That is the entire point of the four chapters of capture you just wired up.

Two reassurances

The sidecar is offline-first: it completes its whole job — capture through dispatch — even with the control channel down, which is exactly when it matters most. And that channel is outbound-only, agent-initiated, and default-off: no downstream system can name a file on your host or reach into your app. You are shipping a recorder, not a remote-control surface.

That's the full integration path, from init() to a reproducing test. Flip to the Cheatsheet for the whole API, every config knob, and the traps — on one page you can keep open while you wire it in.
