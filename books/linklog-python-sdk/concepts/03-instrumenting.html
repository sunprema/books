Chapter 03

Instrumenting Code: intent, scope, snapshot

Chapter 2 got a .llog on disk. Now let's earn the detail inside it. Three primitives — a decorator, a context manager, and a one-liner — draw the causality graph the reader reconstructs downstream. You don't build the tree by hand; you write ordinary Python and the tree falls out of your call structure.

Here is the whole canonical example first, then we'll take it apart line by line. This is demo/refund.py — the flow every other chapter reuses. Read it once as plain code; every LinkLog call is designed to disappear into how you'd structure a refund anyway.

demo/refund.py — the canonical flowimport llog


@llog.intent("ProcessRefund")
def _process_refund(amount: float, divisor: int) -> float:
    llog.snapshot("input", {"amount": amount, "divisor": divisor})

    with llog.scope("ValidateCard"):
        llog.snapshot("request", {"amount": amount, "currency": "USD"})
        if amount <= 0:
            raise ValueError("refund amount must be positive")

    with llog.scope("ChargeGateway", replay_boundary=True):
        llog.snapshot("gateway_request", {"amount": amount, "retries": divisor})
        charge = charge_gateway(amount)
        # the captured response — becomes the stub's return value on replay
        llog.snapshot("gateway_response", charge)

    with llog.scope("SettleLedger"):
        # divisor=0 → ZeroDivisionError, two scopes down from the intent
        fee = amount / divisor
        return charge["net"] - fee

Run it with divisor=0 and it dies in SettleLedger. The checkpoint won't say "ZeroDivisionError at refund.py:69" — it'll say the goal was ProcessRefund; validation passed; the gateway returned this; and settlement divided by a zero retry-count. Same three primitives produced all of that. Let's meet them.

intent — the root of the graph

@llog.intent("ProcessRefund") is a decorator. It names the business purpose, opens the in-memory buffer when the function is entered, and decides the buffer's fate on the way out: discard on success, commit on failure. It is the one root event — every other event in the checkpoint descends from it.

Works on sync AND async

The decorator inspects the function with inspect.iscoroutinefunction and wraps it either way. That matters: FastAPI handlers are async, and they're the primary integration. You put @llog.intent(...) on an async def route exactly as you would on a plain worker function — no separate API, no await gymnastics.

Under the hood the wrapper is almost boringly symmetrical — which is the point. On the happy path the buffer is discarded and nothing touches disk; on any BaseException the forensic capture runs and re-raises, so your exception still propagates unchanged:

llog/tracker.py — the async wrapper (abridged)@functools.wraps(fn)
async def awrapper(*a, **kw):
    s, ts, tp = _begin(name)          # open buffer, emit T_INTENT (the root)
    if s is None:
        return await fn(*a, **kw)     # SDK disabled → zero overhead
    try:
        r = await fn(*a, **kw)
    except BaseException as exc:
        _fail(s, exc)               # ERROR → FRAMEs → TRAILER → commit
        raise                       # your exception is never swallowed
    else:
        s.buffer.discard()           # success → close(fd), gone
        return r
    finally:
        _end(ts, tp)
What "root" means, precisely

_begin emits a T_INTENT event with parent NO_PARENT (the sentinel 0). Every event carries its own eid; every event except the intent carries a pid pointing at its parent. The intent is the only event with no parent — so it is the single root the reader starts its one-pass walk from.

scope — a step, and a branch

with llog.scope("ValidateCard"): marks a logical step. A scope is a context manager, and here's the mechanism that makes the graph a tree: on entry it emits a SCOPE_START whose parent is the current parent, then sets itself as the new current parent for everything inside the with block. Nest a scope inside another and it points at the outer one — that's how branches form.

llog/tracker.py — scope() sets the parenteid = s.emit(P.T_SCOPE_START, _parent.get(), a)
token = _parent.set(eid)   # this scope is now the parent for everything inside
try:
    yield
except BaseException:
    if s.error_pid is None:
        s.error_pid = eid    # innermost scope wins — remember WHERE, then re-raise
    raise                    # NO SPAN_END — the scope never completed
else:
    s.emit(P.T_SPAN_END, eid, {P.A_DURATION: ...})   # clean exit only
finally:
    _parent.reset(token)   # restore the parent for the next sibling

Two extras worth knowing. You may pass arbitrary **attrs keywords — they ride along on the SCOPE_START as structured attributes. And replay_boundary=True tags a scope as an external-call seam: on ChargeGateway it marks the point the replay generator will later stub with the captured snapshots instead of hitting a real gateway. That's a preview — the full replay story is Chapter 9. Here it's just a flag on the event.

Gotcha — no manual open/close

There is no scope.open() / scope.close() in the public API, and that's deliberate. Internally every scope is a lifecycle-managed handle with a parent pointer, but that's architecture, not interface. Hand-balancing open/close would invite unbalanced lifetimes on exactly the failure path where correctness matters most — so the SDK uses the decorator and the with block, and lets Python guarantee the balance for you.

snapshot — a labelled value on the current parent

llog.snapshot("request", value) captures one labelled value and attaches it to whatever the current parent is — a scope if you're inside one, the intent if you're not. It's how state gets into the checkpoint: the arguments you came in with, the gateway's response, a computed total. Cheap, structured, and it never raises.

Trap — snapshot outside an intent is a no-op

Call snapshot() when no intent is active and it's a counted no-op — it increments stats.dropped_no_buffer and returns. There's no trace to attach to, and guessing an implicit intent would invent a business purpose nobody declared. If a library-level snapshot() seems to vanish, check that it actually ran inside an @llog.intent call stack.

Naming is an API

Scope names ("SettleLedger") and snapshot keys ("gateway_response") become the vocabulary of every checkpoint this code ever writes — they're what you'll grep, group, and read at 3 a.m. Keep them stable and meaningful. Renaming a scope changes the shape of your forensic history.

The graph the refund draws

Put the three together and the refund flow produces this tree. Notice where the error lands: not on the intent, but on SettleLedger — the innermost scope the exception passed through — and that SettleLedger scope is left OPEN, with no SPAN_END. That dangling SCOPE_START is the truthful record: it marks exactly where execution was standing when it failed.

The whole product claim, in one line

The error's pid is SettleLedger's eid — not the intent's. The SDK records error_pid in the innermost scope's except, before any finally unwinds the parent stack. So the checkpoint says "failed in SettleLedger," preserving the exact execution path — the difference between a symptom and a story.

Read as a sequence, the events land in exactly execution order, so the reader replays your run top to bottom:
