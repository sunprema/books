Chapter 08

The Guarantees: Zero-Panic, Bounded, Fast

You are about to wrap production code paths in a recording library and leave it on forever. That is only reasonable if the library makes a few hard promises: it will never crash your app, never leak state back into it, never grow without bound, and never cost enough to notice. LinkLog is built around exactly those promises — here they are, and here is what each one means for how you integrate and verify.

1. Zero-panic: it cannot take your app down

This is the load-bearing guarantee (invariant #7). An observability tool that crashes the thing it observes is worse than no tool at all. So every public entry point in the SDK is wrapped: on any internal error — /dev/shm full, a serialization failure, a syscall that returns the wrong thing — LinkLog fails silently and stops recording rather than raising into your code. If there is no active session, every call is a counted no-op.

tracker.py — the shape of every write pathdef snapshot(name, value):
    s = _session.get()
    if s is None:
        stats.dropped_no_buffer += 1   # no trace to attach to → counted no-op
        return
    try:
        s.emit(...)
    except Exception:
        pass                        # never raises into your app
The habit this forces on you

Because the SDK swallows its own errors, a broken capture looks identical to a working one from the outside — there is no exception and no falsy return to tip you off. So when you verify your instrumentation, assert the artifact exists on disk (/dev/shm/llog/events/), never trust a return value or the absence of an exception. This is the single most important debugging habit with LinkLog. The project's own reference PoC returned False from a broken commit for weeks and no test noticed — because nothing was checking the disk.

2. Write-only: nothing flows back

Recapping invariant #6 from Chapter 6: the SDK is a one-way street. Snapshots go into the buffer and there is no public API that hands captured execution state back to your application. Your instrumentation can never become an accidental read-back or exfiltration surface inside your own process.

3. Bounded snapshots: capture anything safely

You should be able to snapshot() almost any value without first checking whether it's huge, cyclic, or serializable. LinkLog makes that safe by bounding every snapshot on three axes — depth (about 3 levels), byte size, and item count — inside the walk itself. A deeply nested or self-referential object simply stops being descended; it can't blow up your memory or hang the walk.

Values it can't serialize don't throw — they're recorded as a marker so the shape survives:

what LinkLog records for the awkward casesllog.snapshot("ctx", {
    "order_id": "ord_88",        # → captured normally
    "db": connection,             # → "<Unserializable: Connection at 0x10f3a20>"
    "blob": huge_bytes,            # → truncated at the byte cap, with a marker
})

An unserializable object becomes <Unserializable: TypeName at 0x...>; an over-long str/bytes is truncated with a trailing .... The rule of thumb: capture is defensive, so you don't have to be.

4. Fast: the success path is effectively free

"Record everything and discard on success" only works if recording is cheap enough to always leave on. The budget is under 1ms per request (under 0.5ms for a 20-step path), and the measured reality on Linux is far under it:

OperationMeasured (Linux, p50)
10-snapshot request, end to end~0.040 ms
O_TMPFILE discard on success~5 µs
6,400 successful requests0 artifacts left on disk

The point isn't the exact numbers — it's the ratio. The overhead sits an order of magnitude below the budget, which is what makes generous recording affordable. You are not trading latency for context here.

Benchmark only on Linux

These figures come from the real mechanism (O_TMPFILE in /dev/shm). On macOS/Windows the SDK runs its hidden-temp-file fallback, so any number you measure there describes the fallback, not the product. If you're going to quote a success-path cost, measure it on Linux.

5. Resource-safe: it won't fill your pod

Buffers live in tmpfs, and tmpfs pages count against your container's cgroup memory limit — so an unbounded recorder could OOM the very pod it's protecting. LinkLog caps the buffer size and event count per request, and caps concurrent buffers and total buffer bytes across the process. It will stop recording before it fills /dev/shm, not after.

6. Introspection: check that it's actually working

Because failures are swallowed, you'll want a way to confirm the SDK is doing what you expect. llog.stats is a live set of counters you can read at any time (for a health endpoint, a test assertion, or a quick REPL check):

a health check on your instrumentationimport llog

# ... after some traffic ...
print(llog.stats.buffers_created, llog.stats.commits, llog.stats.discards)
assert llog.stats.write_errors == 0      # capture is healthy
CounterWhat it tells you
buffers_createdSessions opened — one per instrumented request/intent.
commitsCheckpoints written to disk (i.e. failures kept).
discardsBuffers thrown away on success — should dwarf commits.
write_errorsInternal write failures. Non-zero means capture is degraded.
commit_errorsFailures while committing an artifact. Investigate if non-zero.
cap_truncationsBuffers that hit the size cap and stopped recording early.
dropped_no_bufferCalls made with no active session (e.g. snapshot() outside an intent).
suppressedCommits dropped by the storm rate limiter (Chapter 7).

Alongside the counters, llog.config reflects the effective configuration and llog.current_session() returns the active session (or None) if you need to check whether you're inside a captured context.
