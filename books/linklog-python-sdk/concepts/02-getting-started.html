Chapter 02

Getting Started: Your First Checkpoint

Enough theory. In the next few pages you'll install the SDK, initialise it once, wrap a single function, and watch a .llog artifact appear on disk the instant that function fails — and stay absent when it succeeds. One file, one failure, one checkpoint. That's the whole loop, and everything later is a refinement of it.

Install it

The package is llog. It needs Python 3.9 or newer and pulls in two small dependencies — msgspec>=0.18 for MessagePack encoding and xxhash>=3.4 for the artifact checksum. It's Apache-2.0 licensed. Install it straight from the SDK source tree:

# from the linklog repo root — installs the `llog` package
$ pip install ./sdk/python

# confirm it imported
$ python -c "import llog; print(llog.__version__)"
0.1.0
One import, one namespace

Everything you touch — init, intent, scope, snapshot, the excepthook — hangs off the top-level llog module. There's nothing to sub-import and no client object to thread through your code. import llog is the whole surface.

Initialise once, at process start

Call llog.init() exactly once when your process boots — next to where you'd configure logging. The only required argument is a service name; it's stamped into every checkpoint's header so the sidecar knows who committed it.

import llog

llog.init(service="payment-api")

Here's the part that changes how you write instrumentation: before init() runs, config.enabled is False, and every single llog call is a silent, counted no-op. A @llog.intent decorator just calls your function. A with llog.scope(...) just runs its body. A snapshot() increments a dropped-counter and returns. Nothing captures, nothing raises.

So you can sprinkle instrumentation through a library or a service and ship it dormant. It costs effectively nothing and does nothing until some deployment opts in by calling init(). You're never forced to choose between "instrumented" and "safe to import."

A complete, runnable example

Let's get one checkpoint on disk. Here's a minimal script — modelled on the SDK's own refund_cli.py demo — that initialises, installs the excepthook, wraps one function with an intent, and then deliberately raises. Nothing is wrapped in try/except: an unhandled exception is exactly the point.

# first_checkpoint.py
import llog

llog.init(service="refund-cli")
llog.install_excepthook()          # the CLI / worker fallback path

@llog.intent("ProcessRefund")          # declares the business purpose
def process_refund(amount, divisor):
    return amount / divisor           # divisor=0 → ZeroDivisionError

if __name__ == "__main__":
    # unhandled on purpose: it propagates out to the excepthook
    print(process_refund(42.00, 0))

Run it and you get the traceback you'd expect on stderr, and the process still exits non-zero — LinkLog doesn't swallow the failure, it just records it on the way out:

$ python first_checkpoint.py
Traceback (most recent call last):
  ...
ZeroDivisionError: float division by zero

Why did that commit a checkpoint? The @llog.intent decorator opens an in-memory buffer when process_refund is called. Because the exception is unhandled, it propagates all the way out of the program, and the install_excepthook() hook you installed catches it there, commits the buffer as a Failure Checkpoint, and re-raises so Python prints its traceback as normal. (Chapter 4 covers the full detection story — hooks, framework middleware, watchdogs. Here we just need one artifact on disk.)

Why an excepthook here

A CLI or worker has no framework between your code and the interpreter, so an unhandled exception really does reach sys.excepthook. Web frameworks are different — they catch the exception and return a 500, so the global hook never fires. That's why the primary integration is framework middleware (llog.instrument_fastapi(app)), with the excepthook as the fallback. More in Chapter 4.

Where the artifact lands

On failure, a committed checkpoint appears as a {uuid}.llog file under /dev/shm/llog/events/ — that's where the sidecar watches for work. On success, that directory stays empty. Nothing is written, sampled, or cleaned up, because nothing was ever kept.

# after the failing run above
$ ls /dev/shm/llog/events/
7b2f...e91.llog

# now change divisor to a non-zero value and re-run —
# the buffer is discarded, and the directory is empty again
$ ls /dev/shm/llog/events/
$

The one debugging habit to build now

Remember the design promise from Chapter 1: the SDK is zero-panic. By contract it never raises into your application — on any internal error it fails silently and stops recording. That's exactly what you want in production, and it's a trap while you're learning. A capture that silently broke looks identical, from your code's side, to a capture that worked. There's no exception, and — since the SDK is write-only — no return value to inspect either.

Verify on disk, never by return value

The only trustworthy proof that a checkpoint was captured is that the artifact exists in /dev/shm/llog/events/. Never conclude "it worked" from the absence of an exception or from any value the SDK handed back — it hands nothing back on purpose. When something looks wrong, ls the events directory (or assert the file is there in a test). This one habit will save you hours.

The Linux reality

Everything above — /dev/shm, the anonymous O_TMPFILE buffer, the metadata-only hard link that rescues it — is Linux-only. Those are kernel features that don't exist on macOS or Windows.

You can still develop locally. On a non-Linux host the SDK automatically falls back to a hidden temp file with the same lifecycle: delete on success, rename into place on failure. So the loop you just ran works on your Mac, and the API is identical. But the fallback is a convenience, not the mechanism — the real capture path, and any benchmark worth trusting, only exists on Linux.

A green test on macOS proves nothing about the mechanism

If a test suite "passes" on macOS, it exercised the fallback — not O_TMPFILE, not the /dev/shm hard-link rescue, not the anonymous-inode detection boundary. It proved the API shape, and nothing about the thing the product is built on. The SDK's own test suite refuses to collect on macOS for exactly this reason. The demo scripts are Linux-only too; the repo ships a linux-test harness (a podman VM) so you can run the real path from a Mac. Reach for it before you trust any capture, lifecycle, or performance result.

You've got a checkpoint on disk — but it only records "ProcessRefund raised." Next we make it tell a story: Chapter 3 introduces the three primitives — intent, scope, and snapshot — that turn a bare failure into the goal, the steps, and the exact state that broke it.
