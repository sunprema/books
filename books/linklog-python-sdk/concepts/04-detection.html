Chapter 04

Catching Failures: Middleware & the Excepthook

Here is the part that surprises people. LinkLog's primary detection path is not the global exception hook you'd reach for first — it's a piece of ASGI middleware. The global sys.excepthook is only the fallback. That ordering looks backwards until you see what a web framework does to your exceptions, so let's start there.

Why the hook you'd reach for first is the wrong one

Reach for sys.excepthook and your instinct is sound for a script: an unhandled exception unwinds all the way to the interpreter, the hook fires, you capture it. But a web framework never lets an exception get that far. When your handler raises, FastAPI (or any ASGI framework) catches it inside its own error-handling layer and converts it into a 500 response. The interpreter never sees an unhandled exception, so sys.excepthook never fires.

Sit with what that means. A tool built only on the global hook would be structurally blind to nearly every production web failure — the exact case LinkLog exists to capture. So the design inverts the obvious priority: middleware is primary, the hook is the fallback for the places a framework isn't standing in the way.

The primary path: one line of middleware

Install it on the app object once, at startup. There's a convenience wrapper and the explicit form — they do the same thing:

# demo/refund_api.py — the PRIMARY detection path
import llog
from fastapi import FastAPI
from refund import process_refund

llog.init(service="refund-api", mode="anonymous")

app = FastAPI()
llog.instrument_fastapi(app)   # convenience wrapper

# …or the explicit form, identical effect:
# app.add_middleware(llog.LinkLogMiddleware, capture_5xx=True)

@app.get("/refund")
def refund():
    # divisor=0 → ZeroDivisionError deep inside SettleLedger.
    # FastAPI turns the unhandled exception into a 500;
    # the middleware is what notices and commits.
    result = process_refund(amount=42.00, divisor=0)
    return {"refunded": result}   # never reached

Two things about this middleware are worth understanding, because they explain why it's built the way it is.

It's pure ASGI, not BaseHTTPMiddleware. That means it works with FastAPI and any other ASGI framework — Starlette, Quart, your own — and it deliberately avoids FastAPI's HTTP-middleware base class, which wraps responses in a way that breaks streaming. The rule LinkLog holds itself to is never to change the behaviour of the app it observes.

It must sit OUTSIDE the framework's exception handlers. This is the whole reason it can be primary. If it sat inside, the framework would have already swallowed the exception before the middleware ran, and there'd be nothing to see. From its position on the outside, it opens a session automatically for every HTTP request, named "{METHOD} {path}" — so GET /refund becomes the intent, with no decorator from you.

The two shapes it catches

From outside the framework, a request can fail in two visibly different ways, and the middleware handles both. Here is its core, lightly trimmed:

# llog/hooks.py — inside LinkLogMiddleware.__call__
status = {"code": 200}

async def send_wrapper(message):
    if message["type"] == "http.response.start":
        status["code"] = message.get("status", 200)
    await send(message)

try:
    await self.app(scope, receive, send_wrapper)
except BaseException as exc:
    tracker._fail(s, exc)          # SHAPE A: it propagated → capture
    raise                         # …then re-raise, unchanged
else:
    if self.capture_5xx and status["code"] >= 500:
        # SHAPE B: the framework caught it and returned a 500.
        # No exception out here — the status IS the failure signal.
        tracker._capture_and_commit(
            s, RuntimeError(f"HTTP {status['code']} returned by {name}")
        )
    else:
        s.buffer.discard()          # success → buffer vanishes
  • Shape A — the handler raises and it propagates. The middleware catches it, commits the checkpoint, and re-raises so the framework's own error handling still runs exactly as before. You don't lose the 500 response your client expects.
  • Shape B — the framework swallows it and emits a 5xx. No exception ever reaches the middleware. All it sees is status >= 500, and from out here that status code is the only evidence a failure happened. So a synthetic RuntimeError carries the checkpoint. This is the common case in production.
The capture_5xx switch

Pass capture_5xx=False to capture only exceptions that propagate (Shape A) and ignore swallowed 500s (Shape B). Useful if another layer already returns intentional 500s you don't consider failures. The default, True, is what you want in most services — the swallowed 500 is precisely the failure you can't see any other way.

Gotcha — don't open a second buffer

The middleware owns the request's session. Do not also decorate your top-level route handler with @llog.intent — that opens a second buffer competing with the one the middleware already opened. With middleware in place, let the request be the intent, and reach only for llog.scope() and llog.snapshot() inside your handlers and service code. Save @llog.intent for code paths outside an HTTP request — CLIs, workers, background jobs, sub-tasks.

The fallback: the global excepthook

Now the other half. A CLI or a worker has no framework standing between your exception and the interpreter — so when something goes genuinely unhandled, sys.excepthook does fire. That's where install_excepthook() earns its keep:

# demo/refund_cli.py — the FALLBACK detection path
import llog
from refund import process_refund

llog.init(service="refund-cli", mode="anonymous")
llog.install_excepthook()

if __name__ == "__main__":
    # NOT wrapped in try/except — the whole point of the fallback
    # is an UNHANDLED exception reaching sys.excepthook.
    print("refund-cli: processing a refund with a bad gateway retry count...")
    result = process_refund(amount=42.00, divisor=0)
    print(f"refund-cli: refunded {result}")   # never reached

The important behaviour here is what it doesn't change. install_excepthook() chains to the previous hook: LinkLog commits the checkpoint first, then calls the hook that was already installed — so the traceback still prints to stderr and the process still exits non-zero, exactly as it did before you added a single line. You never swallow a traceback the developer expected to see. Call llog.uninstall_excepthook() to put the original hook back.

The key guarantee

Neither path alters your app's behaviour. Shape A re-raises; the excepthook chains and still exits non-zero. LinkLog observes; it never intervenes. That's the invariant that lets you install it in production without holding your breath.

When nothing raises: manual commit

Some failures never throw. You computed a result that is perfectly valid Python and completely wrong — a refund of the wrong amount, a total that doesn't reconcile. No exception, no 500, so neither detection path fires. For that, commit the current session's checkpoint deliberately:

if reconciled_total != expected_total:
    llog.snapshot("reconciled_total", reconciled_total)
    path = llog.commit()   # keep this context — a failure that never raised
    if path:
        log.warning("ledger drift; checkpoint at %s", path)

llog.commit() returns the artifact path, or None if there was nothing to commit or the commit was rate-limited. And that's the quiet unifying detail: every failure path — the @llog.intent decorator, the excepthook, the 5xx middleware, and this manual commit — routes through the same rate-limited capture (_capture_and_commit). One code path decides whether an artifact is written, so a failure storm is contained identically no matter which door it came through. We'll pull that thread in Chapter 7.

What neither path can cover

An OOM-kill, a SIGKILL, a native segfault — these run no Python at all, so no hook, middleware, or manual commit can fire, and the in-memory buffer dies with the process. That entire class is out of reach of any handler by definition. Catching it needs a different mechanism, where the buffer outlives the process — which is exactly what Chapter 5's STAGING mode is for.
