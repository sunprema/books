Chapter 05

Buffer Modes: Anonymous vs Staging

Here is the one config choice that decides whether you can see a whole class of failures. Both modes produce byte-identical artifacts and differ in exactly one respect: whether the buffer can survive its own process. Get this right and an OOM-kill becomes a readable checkpoint; get it wrong and it vanishes without a trace.

Start with the switch

Everything you built in Chapters 1–4 ran in the default mode. To change the survival property, you change one keyword argument at init():

# The default — what you have been running so far.
import llog
llog.init(service="payment-api")                    # mode="anonymous"

# Opt in to a buffer that outlives a hard crash.
llog.init(service="payment-api", mode="staging")   # named buffer + watchdog

That single argument moves the buffer from an anonymous inode to a named file, and — as a side effect worth understanding — starts a background heartbeat thread. Before you decide which to use, let's see why the difference matters at all.

Anonymous: zero trace, one blind spot

In mode="anonymous" (the default) the buffer is an O_TMPFILE — a real file with no name, an unnamed inode living in /dev/shm. Its two fates are the ones you already know:

  • Success is close(fd). The kernel reclaims the pages and the inode is gone. There was never a directory entry, so there is nothing to clean up.
  • Ordinary failure is a metadata-only hard link from /proc/self/fd/{fd} into events/{uuid}.llog. No bytes are copied to keep a failure.

The property this buys you is precious: a successful request leaves zero trace on disk, ever. Nothing to sample, rotate, or bill for. But that same anonymity has a hard edge.

The detection boundary

If the process is killed uncatchably — OOM-kill, SIGKILL, a native segfault — no Python code runs on the way out. The O_TMPFILE inode dies with the process's last descriptor, before anything can react. There is no checkpoint, not even a partial one. This is a property of anonymous inodes, not a bug to fix later.

Sit with why that matters in production. In Kubernetes, an OOM-kill is one of the most valuable failure classes to capture — a pod hits its memory limit and the kernel reaps it with no stack trace, no 500, no warning. And it is exactly the class the default mode cannot see. The excepthook never fires; the middleware never runs; the buffer is already gone.

Staging: a name is what survives

In mode="staging" the buffer is a named file at /dev/shm/llog/staging/{pid}-{uuid}.llog. The lifecycle mirrors anonymous mode but runs one step in reverse:

  • Success is unlink() then close() — remove the name first, then release the fd.
  • Ordinary failure is a rename() of the named file into events/.

Because the file has a name, it is no longer tied to the life of any file descriptor. On a hard crash the process dies but the named file remains on tmpfs — an orphan. The sidecar can then walk in, find it, and rescue it. That is the whole trick: a name is a handle the kernel keeps even after your process is gone.

Nothing is free. Because a successful request's buffer briefly exists under a real name before unlink() runs, staging opens a small exposure window: for that instant, raw request state is a readable file on tmpfs. That is the price of survivability, and it is why the SDK refuses to enable staging into a world-readable directory.

Byte-identical, remember

The artifacts these two modes commit are the same bytes. Downstream — the sidecar, the reader, the checkpoint you read — cannot tell which mode wrote a given .llog. The mode changes only which failures can produce an artifact at all, never what the artifact looks like.

Why staging needs a heartbeat

When you pass mode="staging", init() automatically starts a watchdog thread. You do not wire it up — it is a consequence of the mode:

llog.init(service="payment-api", mode="staging")
# └─ starts one daemon thread that touches, every ~5s:
#      /dev/shm/llog/live/{pid}

That thread does one job: touch a liveness heartbeat file at /dev/shm/llog/live/{pid}. Here is the WHY, and it is the subtle part. The sidecar's reaper does not key on pid liveness — it cannot. It runs in a separate container, and Kubernetes pods do not share a PID namespace by default, so from the sidecar's vantage point your app's pids are invisible. A "is this pid alive?" check would silently reap nothing. Instead, the reaper keys on the staleness of the heartbeat file: when live/{pid} stops being touched, the process is presumed dead and its staging orphans are fair game.

Without the heartbeat, tmpfs fills

The heartbeat is not optional dressing — it is what makes reaping possible. If it never runs (or the live/ directory is unwritable), the reaper can never conclude a process is dead, orphans are never reaped, and staging files accumulate on tmpfs until /dev/shm fills — which then breaks the very SDK that was meant to capture the failure. The thread is a daemon and never raises, so a missed beat degrades reaping; it does not crash your app.

What a reaped orphan looks like

A buffer rescued by the reaper was never finished by your process — the crash cut it off mid-flight. So it has no TRAILER. The reader classifies a trailer-less artifact as TRUNCATED, and — this is important — that is a normal outcome, not corruption. Truncated means "the recording was cut short," which is precisely what an OOM-kill does.

The payoff is in the graph. The checkpoint shows the scope that was still OPEN when the process died — the SCOPE_START that never got its SPAN_END. That open scope is where the kill struck:

# inspecting a reaped orphan
ProcessRefund
├─ ValidateCard          # completed
└─ ChargeGateway [OPEN]    ← still open: killed here

In anonymous mode that same OOM-kill produces an empty events/ directory and a shrug. In staging mode it produces a checkpoint that points a finger at ChargeGateway. Same failure, wildly different amount of knowledge.

See it with the hang demo

The repo ships demo/refund_hang.py to make this concrete. It wedges a request forever inside ChargeGateway — no exception is ever raised, so nothing but a kill ends it. Run it in staging mode and SIGKILL it:

$ LLOG_MODE=staging python demo/refund_hang.py &
# refund-hang[staging]: pid 4127 is now wedged inside ChargeGateway
$ kill -9 %1
# liveness file goes stale → agent reaps the orphan →
# TRUNCATED checkpoint showing ChargeGateway [OPEN]

Now run the same script in the default anonymous mode and kill it the same way:

$ LLOG_MODE=anonymous python demo/refund_hang.py &
$ kill -9 %1
# events/ is empty. You get NOTHING.
# The O_TMPFILE inode died with the process. Correct, not a bug.
LLOG_MODE is a demo convention, not an SDK feature

LLOG_MODE is not a built-in environment variable the SDK reads. The demo reads os.environ itself and passes the result to init(mode=...). In your own app you pass mode="staging" to init() directly (or wire your own env plumbing if you want one). Don't go looking for an LLOG_MODE knob inside llog — there isn't one.

One hard constraint: same tmpfs

The buffer and the events/ directory must live on the same filesystem. Hard links and rename() cannot cross filesystems — attempt it and you get EXDEV, and the commit silently fails. Never point root at /tmp (often a different mount); keep it on /dev/shm. The sidecar is what moves artifacts to durable storage after pickup — that copy is allowed to cross filesystems; your commit is not.

Which one should you pick?

Default to anonymous. Reach for staging only when the failures you must capture include the ones anonymous mode is blind to.

Choose…When…Because
Anonymous default the common case — request failures, exceptions, 5xx zero success-path trace, no exposure window, no extra thread. Simplest thing that works.
Staging OOM-kills and hard crashes are failures you must capture memory-limited K8s pods, workers prone to being killed. The only way to see the class anonymous can't.

A useful way to frame it: anonymous mode captures failures your code can observe; staging mode also captures the failures that kill your code before it can observe anything. If your pods get OOM-killed and you have never seen a checkpoint for it, that is not the absence of a problem — it is the detection boundary, and staging is how you cross it.
