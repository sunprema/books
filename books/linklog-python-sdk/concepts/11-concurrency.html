Chapter 11

Concurrency: Hundreds of Requests at Once

Every chapter so far followed a single request through capture. But your service doesn't serve one request — it serves hundreds, interleaved, and the obvious question is "if hundreds fail at once, how does llog keep them straight?" The one-line answer is reassuring; the consequences are worth spelling out, because one of them is the difference between a correct artifact and a privacy leak that only ever appears under load.

The one line

Every in-flight request gets its own Session → its own Buffer → its own file descriptor → its own causality graph. There is no process-wide buffer that requests write into together, so "hundreds at once" is just hundreds of independent objects. The only shared state is read-mostly config plus two deliberately-locked coordination points: the storm limiter and one-time directory setup.

What keeps each request's state separate depends on your execution model, and the two cases differ enough to matter. Get the second one wrong and you don't get a crash — you get two users' data in one file.

Case one: thread-per-request (WSGI, sync workers)

Picture 300 requests on 300 threads. When @llog.intent fires it calls _begin(), which builds a fresh Session and stashes it in a ContextVar — not a plain global. Each thread reads its own value, so there's no cross-talk:

llog/tracker.py — per-request isolation lives in two ContextVarsfrom contextvars import ContextVar

_session: ContextVar[Session | None] = ContextVar("llog_session", default=None)
_parent:  ContextVar[str | None]     = ContextVar("llog_parent",  default=None)

def _begin(intent_name):
    s = Session(intent_name)     # fresh Session → fresh Buffer → fresh fd
    _session.set(s)              # each thread reads *its own* copy
    return s

A second ContextVar, _parent, holds the current parent event id, so nested with llog.scope(...) blocks build the correct parent→child graph per thread. And each Buffer holds one O_TMPFILE fd whose writes go straight to the kernel via os.write — no application-memory buffering (that's invariant #2 from Chapter 8) — so two concurrent threads never touch the same descriptor. Threads are the easy case: the OS already keeps them apart, and a ContextVar rides along.

Case two: async / FastAPI — the case the design is built around

Async is where a naive design quietly breaks. FastAPI interleaves many requests on a single event-loop thread — they take turns, hopping in and out at every await. Now ask what would happen if the tracker had stored its session in a threading.local or a module global:

The bug that only exists under concurrency

A threading.local is shared across coroutines that interleave on the same thread. A module global is shared across everything. Either one would splice two concurrently-interleaved requests' execution graphs into one artifact — a correctness bug and a privacy leak (invariant #5, Chapter 6). Request A's snapshot lands in Request B's checkpoint. It passes every single-request test and only surfaces when two requests overlap in time.

That is exactly why the tracker uses contextvars and not a thread-local. contextvars give each async task its own copy of _session and _parent, so the graph stays correct with hundreds of coroutines interleaved on one thread. You don't configure this and you can't get it wrong — but it's worth knowing it's load-bearing, because it's the reason you install the ASGI middleware (Chapter 4) rather than reaching for a global hook:

main.py — middleware is the primary path, and it's concurrency-correct by constructionfrom fastapi import FastAPI
import llog

app = FastAPI()
app.add_middleware(llog.ASGIMiddleware)   # each request → its own contextvar-scoped Session

The one place two threads share a buffer

There is exactly one write path where two threads legitimately touch the same buffer: the request thread, and the watchdog thread (Chapter 4) that commits a deadlocked request which never returns (invariant #9). Two threads appending into one MessagePack stream would interleave two objects and corrupt the artifact — so every Buffer carries its own threading.Lock, taken by write / discard / commit, plus a closed-state flag that turns post-commit writes into no-ops.

Why this lock never shows up in a flame graph

Because the lock is per buffer, it's never a contention point between requests — a request only ever contends with its own watchdog, and only if it's hung. Uncontended acquisition is ~20–25ns and lands on the success path, so it's benchmarked with the watchdog live, not assumed (DR-001; still unverified until the deadline registry is wired). One request's lock has nothing to say to another's.

The real stress case isn't concurrency — it's correlated failure

Here's the reframe that matters. Hundreds of requests running at once is not the scary scenario — they're independent objects and the machinery above keeps them apart. The scary scenario is correlated failure (invariant #8, and the whole of Chapter 7): a shared dependency dies, all N requests fail together, and all N buffers try to commit at the same instant — filling /dev/shm and handing the sidecar N identical artifacts for one bug. The tool would invert under exactly the load it exists to survive.

This is the one point where every failing thread converges on a single shared object — the per-signature storm limiter — and correctness under that contention is its entire job:

AspectBehavior under a storm
Keya cheap local signature — exc_type | innermost_frame | intent — not the sidecar's authoritative fingerprint (DR-011)
Limitper-signature token bucket, default 10 commits/min (max_commits_per_min)

That's the same containment you met in Chapter 7, now seen from the concurrency angle: it's the deliberate exception to "every request is independent," and it's globally locked precisely because its correctness under a thundering herd is the property you're relying on.

Resource ceilings that bound "hundreds → thousands"

Concurrency turns small per-request costs into budgets. Three are managed explicitly so LinkLog can never become the reason your app fell over:

  • One fd per in-flight buffer. File-descriptor count is a real budget at high concurrency (HLD §9). So the root fd for linkat and the directory setup are opened once per process (_get_root_fd, _ensure_dirs), never per request — precisely to not double the fd cost on the hot path.
  • tmpfs pages count against your cgroup. Every buffer lives in /dev/shm, whose pages count against the container's memory limit — so each buffer is capped (MAX_BUFFER_BYTES, MAX_EVENTS). An unbounded buffer could OOM-kill the very app it observes (invariant #7).
  • Zero panic throughout. A full /dev/shm, an fd closed underneath it, a limiter bug — every path degrades to "stop recording" or "allow the commit," never a raise into your application code (Chapter 8).
Verify concurrency on Linux, or you've verified nothing

O_TMPFILE, /dev/shm, and the linkat commit exist only on Linux; macOS and Windows silently use the named-file fallback, and the test suite refuses to collect there. So every claim on this page is verified on real Linux via the linux-test harness — not on a developer laptop. A concurrency stress test that "passes" on your Mac proved nothing about the mechanism the product is built on. The next chapter makes that discipline a command you can run.

So: hundreds of requests are hundreds of independent objects, async correctness rides on contextvars, one per-buffer lock guards the watchdog seam, and the storm limiter is the single deliberate convergence point. All of which raises a fair question — how do you know your integration actually did all this? That's the final chapter.
