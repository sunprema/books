Chapter 12

Verifying Your Integration

You've wired the whole thing: middleware, intents, buffer mode, allowlist. It compiles, your test throws and gets caught, everything looks green. Here's the uncomfortable truth this chapter exists for — with LinkLog, "looks like it works" is exactly the failure mode. Every common integration mistake is invisible when you read the code and obvious only when you run it. So LinkLog ships a way to force the running and read the verdict off disk.

Three ways an integration lies to you

These aren't hypotheticals — they're the specific mistakes a person (or, increasingly, a coding agent) makes wiring LinkLog in, each of which produces a false green:

The false greens

1. The pre-empted hook. Reaching for a global sys.excepthook that your web framework's exception handling silently pre-empts (invariant #9) — it rarely fires, but nothing errors. 2. The Mac false green. "Verifying" on a macOS laptop, where the core capture path doesn't even exist, so a test passes having exercised nothing (Chapter 5). 3. The wrong mode. Picking ANONYMOUS mode and never capturing the OOM-kills that are the whole point in Kubernetes (invariant #10). All three read as working. Only execution on the real platform tells the truth.

And there's a fourth, subtler one that undoes naive testing entirely: because LinkLog is zero-panic (Chapter 8), its internal errors are swallowed. So a caught exception in your handler is not proof anything was captured. The exception firing and the checkpoint landing are two independent events — and only one of them is the thing you care about.

The rule that redefines "done"

Assert the artifact, not the return value. Integration is complete not when the code compiles or the exception is caught, but when a forced, real failure produces a .llog on disk — on Linux. LinkLog gives you two commands that make exactly that the definition of done.

Step one: llog rules sync — teach the integrator the non-obvious path

Most people won't hand-integrate an SDK anymore — their coding agent will. Left unaided, an agent reading the SDK source gets the LinkLog-specific parts wrong in precisely the invisible ways above. So the SDK ships its own integration rules and writes them where the agent will read them:

terminal$ llog rules sync
  # reads the rules bundled *inside* the installed SDK (offline, no network)
  # merges them into ./AGENTS.md between version-stamped markers:
  wrote <!-- BEGIN llog-python v1 --> … <!-- END llog-python -->

$ llog rules sync # idempotent: run it again → byte-identical, no duplicate block
$ llog rules sync --also CLAUDE.md   # optional second target

The block is marker-scoped, so a polyglot repo gets one clean block per SDK and syncing one never disturbs another's. It carries the SDK version and warns if your installed SDK is newer than the block on disk. And what it writes into AGENTS.md is the non-obvious correct path, stated as rules:

  • Framework middleware is primary; sys.excepthook is only a fallback (frameworks swallow exceptions — Chapter 4).
  • The ANONYMOUS-vs-STAGING choice, tied to its consequence: anonymous can't see an OOM-kill (Chapter 5).
  • Allowlist capture as the default for regulated data (Chapter 6).
  • The write-only boundary, and: zero-panic means a caught exception is not proof of capture — assert the artifact.
  • And it redefines done: integration is complete when llog verify-integration forces a failure and sees a .llog land on Linux.
Why the rules can be trusted (DR-018)

These rules are prose that makes falsifiable claims about how to wire capture — exactly the kind of thing this whole product exists because prose drifts from truth. So they're not hand-maintained per SDK. They're composed from one canonical source (a language-neutral core.md plus a per-language overlay) into a committed bundle, and CI holds the bundle to reality: it regenerates the bundle and fails on any diff, and it executes the integration snippet embedded in the rules on the Linux harness, asserting a checkpoint lands. Rotted rules go red. The rules can't document a path that doesn't run.

Step two: llog verify-integration — force a failure, assert the artifact

This is the executable done-gate. It's a self-contained smoke test of the mechanism: it instruments a tiny function, forces a raise, and then asserts a .llog actually appears in /dev/shm/llog/events/. If a sidecar happens to be watching, it confirms pickup too — but "no sidecar" is still a normal PASS, because it's asserting the SDK-side commit, which needs no watcher.

terminal — the same command, three honest outcomes# On Linux, correctly wired:
$ llog verify-integration
  forcing a failure through your instrumentation…
  PASS  artifact observed: /dev/shm/llog/events/3f9c.llog     # exit 0

# On Linux, capture is broken (e.g. a parent dir the agent can't write):
$ llog verify-integration
  forcing a failure through your instrumentation…
  FAIL  no artifact appeared — capture is not working              # exit 1

# On macOS:
$ llog verify-integration
  REFUSED  no O_TMPFILE / no /dev/shm here — a green would be false  # exit 3

Three distinct exit codes, so a caller — a CI job, a Makefile, your agent — can tell "broken" apart from "not applicable here":

That macOS REFUSED is the most important design choice in the command. It does not report success on a platform where the capture mechanism doesn't exist — it explains, in plain language, why a green there would be a lie, and exits non-zero. The refusal is load-bearing, not decorative: it's mutation-tested by stubbing the platform guard to "allow" and confirming it would false-green on the fallback. A tool that told you "yes, it works" on your laptop would be teaching you the exact trap the product exists to eliminate.

It is not a phone-home

verify-integration produces a local artifact and inspects it locally — it sends nothing off-host. And it reads the checkpoint it just wrote through the reader path, as a diagnostic tool, never back through the SDK: the write-only wall from Chapter 6 stays intact even in the tool that verifies capture.

Wire it into CI as your done-gate

The payoff is a one-line gate that fails the build when capture silently breaks — a dependency bump, a middleware reorder, a mode flipped in a refactor. Run it where the mechanism is real:

.github/workflows/ci.yml — runs on a Linux runner- name: Verify LinkLog capture still fires
  run: llog verify-integration      # exit 0 required; 1 fails the build, 3 shouldn't occur on Linux CI

On a non-root container — and production is never root — it still must pass; a root-only green would hide a CAP_DAC_READ_SEARCH gap. That's the whole discipline of this book made mechanical: don't trust that capture works because the code looks right; force a failure and watch the checkpoint land.

That closes the loop. You've built capture (Chapters 1–8), watched it survive load (Chapter 11), followed the checkpoint downstream and out to the fleet (Chapters 9–10), and now you can prove the whole chain fires on every commit. Flip to the Cheatsheet for the whole API, every config knob, and the traps — on one page you can keep open while you wire it in.
