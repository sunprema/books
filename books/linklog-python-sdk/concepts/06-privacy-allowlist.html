Chapter 06

Privacy: Allowlist Capture & the Write-Only Wall

An instrument that records your program's execution state is, by definition, standing next to your most sensitive data. LinkLog gives you two guarantees that keep it from becoming a liability: it never hands that state back to your own code, and — when you ask it to — it only ever captures the fields you have named as safe. Let's earn both.

Guarantee A — the write-only wall

Start with the smallest possible surprise. Call snapshot() and look at what you get back:

# snapshot() records state — it does not return it
result = llog.snapshot("amount", 42.0)   # -> None. Always.

# There is no llog.read(), no llog.dump(), no session.events[...].
# Whatever you capture goes INTO the buffer and never comes back out.

That is the whole of invariant 6, and it is deliberately boring. Nothing in the public API — not snapshot, not scope, not intent — returns captured execution state to application code. Snapshots flow one way, into the in-memory buffer, and the only thing that ever reads them back is the SDK's own checksum routine, reading from its own file descriptor (invariant 2 forbids keeping a heap copy at all).

Why insist on this? Because the failure mode it forecloses is subtle. If instrumentation could return the values it captured, then any code path in your process — a plugin, a dependency, a compromised handler — could ask the observability layer to replay a customer's card number back into memory it controls. The instrument would have quietly become a data-exfiltration surface inside your own process. A write-only wall means there is no such API to abuse: the values are inert the moment they land.

The key idea

Instrumentation you cannot query is instrumentation nobody can turn into a leak. snapshot() returns None on purpose — the absence of a getter is the feature.

Guarantee B — why scrub-after-capture isn't enough

The write-only wall stops the data leaving through your process. The next question is what the SDK captures in the first place. The tempting answer is "capture everything, then scrub the sensitive bits later in the sidecar." For regulated data, that answer quietly fails.

Pattern-based scrubbing catches things that look like secrets — a 16-digit run, an SSN-shaped string, a high-entropy token. It cannot catch contextual PII in free text. A customer's name sitting in a note field has no pattern. An address in a prose comment has no pattern. Scrub-after-capture will sail right past them, because there is nothing to match. So for regulated deployments LinkLog inverts the default: capture only the fields you have declared, and never capture the rest at all.

import llog

llog.init(
    service="payment-api",
    capture_mode="allowlist",                 # default is "full"
    allow_fields={"amount", "currency", "order_id"},
    allow_config="llog.allow.json",           # optional {"allow_fields": [...]}
)

The ceiling on what may ever be captured is the union of three sources — the init argument, the JSON config file, and an environment variable. All three feed one frozen set:

# 1. the init argument — set literal
allow_fields={"amount", "currency", "order_id"}

# 2. llog.allow.json — ops/security-ownable, auditable as one file
{ "allow_fields": ["shipping_country", "line_count"] }

# 3. the LLOG_ALLOW_FIELDS env var — comma-separated
export LLOG_ALLOW_FIELDS="amount,currency,order_id"

# ceiling  =  argument  ∪  config file  ∪  env var
Why three sources

They map to who owns the decision. Engineers declare fields inline; a security team owns llog.allow.json as a single reviewable artifact; ops sets LLOG_ALLOW_FIELDS per environment. The SDK unions whatever loads — but as you'll see next, "union" here can only ever narrow the surface, never widen it past intent.

The three rules that make it a real control

An allowlist is only worth trusting if its edges are sharp. Three rules from snapshot.py and DR-010 turn it from a control-shaped suggestion into an actual control.

1 — Gate every leaf, keep the structure

The walker always traverses containers, so the shape of your data stays visible: a reader of the checkpoint can see the request had an ssn field. But a scalar value is captured only if its own key is declared. Structure comes free; values are earned. Crucially, declaring a container does not expose the secrets inside it — the allow decision is made on each leaf's own key, not on the subtree it lives in:

# from snapshot.py — the leaf gate
if isinstance(v, _ATOMIC):
    if allow is not None and key not in allow:
        return _withheld(v)      # undeclared leaf -> type + shape only
    return v                    # declared leaf -> kept

2 — Withheld = type + shape, never value

An undeclared value is not dropped to null — that would erase the fact it existed. It becomes a placeholder that records what was there without what it held. Length is the "shape" for strings and bytes (enough to know a 16-char string sat here); for anything else the type name alone is the shape:

def _withheld(v):
    t = type(v).__name__
    if isinstance(v, (str, bytes)):
        return f"<withheld: {t} len={len(v)}>"
    return f"<withheld: {t}>"

3 — The allowlist is a ceiling, and errors fail closed

Here is the mental flip that matters. Capture runs the opposite direction from redaction: more declaration is less safe. A redaction list is a floor you add to; an allowlist is a ceiling you narrow from. So less-trusted sources may only ever narrow the set — no code path exists to widen it per call, because a silent widen is exactly the leak. And a malformed config file fails closed: it yields a narrower set, never a crash of init (invariant 7), never a wider capture surface.

# from _load_allow_fields — a broken config narrows, never crashes, never widens
if config_path:
    try:
        with open(config_path) as fh:
            doc = json.load(fh)
        for f in doc.get("allow_fields", []):
            fields.add(str(f))
    except Exception:
        pass              # fail closed: no field added, init still succeeds
Gotcha

Declare leaf field names, not container names. Adding "user" to allow_fields does not expose user.ssn — the container is traversed either way, and each leaf inside is still gated by its own key. To keep a value, name the exact leaf key you consider safe (e.g. "amount", "currency"), never the object that wraps it.

A concrete before & after

Put the three rules together on one snapshot. We declare a single safe leaf — amount — and hand the walker a dict with a card number and a nested user object:

llog.init(service="payment-api",
          capture_mode="allowlist",
          allow_fields={"amount"})

llog.snapshot("payment", {
    "amount": 42.0,
    "card_number": "4111111111111111",
    "user": {"ssn": "123-45-6789", "email": "ana@example.com"},
})

The amount leaf is declared, so it survives. Every other leaf is withheld down to type-and-shape — including the two inside user, even though user itself was traversed so the structure remains legible. This is what actually lands in the buffer:

{
  "amount": 42.0,
  "card_number": "<withheld: str len=16>",
  "user": {
    "ssn": "<withheld: str len=11>",
    "email": "<withheld: str len=15>"
  }
}

When "full" is the right call

Allowlist mode is a recommendation, not a mandate. The default is capture_mode="full", and it is byte-identical to pre-allowlist behavior — no filtering, every value captured within the usual depth and size bounds. The filtering is a single set lookup per leaf inside the walk the SDK already runs, so there is no measurable success-path cost either way; the choice is purely about what the data is.

capture_modeWhat it capturesReach for it when
"full" defaultEvery value, bounded by depth/size onlyNon-sensitive services, internal tools, local dev
"allowlist"Only declared leaf values; the rest → type + shapeRegulated data — payments, PII, health, anything audited

The bigger picture — two barriers, not one

Allowlist capture is the capture-side barrier: it decides what raw data is ever recorded. It is not the only barrier. Invariant 5 draws a hard security boundary around the host — raw execution data never leaves it. All sanitization (key filtering, pattern detection, entropy detection) happens in the Go sidecar, on the same machine, before any external transmission; and the sidecar-to-platform link is mTLS.

So think of it as defense in depth. Allowlist capture keeps the contextual PII a scrubber can't catch from ever entering the buffer; sidecar scrubbing catches the pattern-shaped secrets on the way out. Either alone is a barrier; together they are the reason instrumentation this deep is safe to run against regulated traffic at all.

Defense in depth

Allowlist capture is the first barrier, not the sole one. Even in "full" mode the sidecar still scrubs before anything leaves the host — but for regulated data you want both: declare your safe leaves, and let the sidecar be the backstop.
